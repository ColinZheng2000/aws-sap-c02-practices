A company is building a serverless application that runs on an AWS Lambda function that is attached to a VPC. The company needs to integrate the application with a new service from an external provider. The external provider supports only requests that come from public IPv4 addresses that are in an allow list.  
  
The company must provide a single public IP address to the external provider before the application can start using the new service.  
  
Which solution will give the application the ability to access the new service?

- A. Deploy a NAT gateway. Associate an Elastic IP address with the NAT gateway. Configure the VPC to use the NAT gateway.
- B. Deploy an egress-only internet gateway. Associate an Elastic IP address with the egress-only internet gateway. Configure the elastic network interface on the Lambda function to use the egress-only internet gateway.
- C. Deploy an internet gateway. Associate an Elastic IP address with the internet gateway. Configure the Lambda function to use the internet gateway.
- D. Deploy an internet gateway. Associate an Elastic IP address with the internet gateway. Configure the default route in the public VPC route table to use the internet gateway.


---

### Analysis for Student Learning

#### Understanding the Problem

The core requirement is to enable an **AWS Lambda function** (which is running inside a **VPC**) to access an **external service** that restricts access to a **single, specific, and static public IPv4 address** (an **allow list**).

- **Lambda in a VPC:** This is a crucial detail. When a Lambda function is configured to run inside a VPC, its network traffic is routed through the VPC's networking infrastructure. The function's Elastic Network Interface (ENI) is placed in a subnet. For it to access the internet, a specific outbound routing mechanism is required.
    
- **Static Public IP Address (Allow List):** This is the key constraint. The solution must provide a **fixed, single public IPv4 address** for all outbound traffic from the Lambda function to the external provider.
    

---

#### Services Explanation

1. **AWS Lambda in a VPC:** When Lambda is configured to access resources within a VPC, it creates an **Elastic Network Interface (ENI)** in the specified subnets. Because the ENI is within a VPC, outbound traffic to the internet does not automatically use public IP addresses for NAT (Network Address Translation) and needs an explicit route to an Internet Gateway (IGW) or a **NAT Gateway (NAT GW)**. Lambda functions in **private subnets** (which is a common and secure configuration) cannot directly access the internet.
    
2. **NAT Gateway (NAT GW):** A **Network Address Translation (NAT) Gateway** is an AWS-managed service that allows instances in a **private subnet** to connect to the internet or other AWS services, but prevents the internet from initiating a connection to those instances.
    
    - It requires an **Elastic IP address (EIP)** to be associated with it. This EIP is the **single, static public IP address** used for all outbound traffic originating from the private subnet(s) routed through the NAT Gateway. This perfectly aligns with the 'single public IP address for an allow list' requirement.
        
    - It must be deployed in a **public subnet**.
        
3. **Internet Gateway (IGW):** An **Internet Gateway** is a horizontally scaled, redundant, and highly available VPC component that allows communication between instances in your VPC and the internet.
    
    - It enables instances in **public subnets** (those with a route table pointing to the IGW) to directly communicate with the internet.
        
    - It does _not_ provide a single, static public IP address for outbound traffic from instances that don't already have one (like Lambda ENIs in private subnets, which are most common). In a public subnet, an instance with a public IP address (or an attached EIP) communicates directly; instances without one cannot. The IGW itself is not associated with an EIP to act as a single NAT point for multiple internal resources.
        
4. **Egress-Only Internet Gateway (EIGW):** This gateway serves a similar purpose to a NAT GW, but is specifically for **IPv6 traffic**. It allows instances in a VPC's private subnets to initiate outbound connections to the internet over IPv6 but prevents inbound connections from the internet.
    
    - It is **only used for IPv6 traffic** and is irrelevant for the stated requirement of a **public IPv4 address**.
        
5. **Elastic IP Address (EIP):** A static, public IPv4 address that you can associate with a running instance, network interface, or, crucially, a **NAT Gateway**. By associating it with a NAT Gateway, it provides the required fixed public IP address for all internal resources using that gateway.
    

---

#### Correct Option(s) Explanation

**A. Deploy a NAT gateway. Associate an Elastic IP address with the NAT gateway. Configure the VPC to use the NAT gateway.**

- **NAT Gateway (NAT GW)** is the correct service to use to enable instances in a private subnet (where the secure Lambda function ENI is typically placed) to connect to the internet.
    
- The **Elastic IP (EIP)** associated with the NAT GW provides the necessary **single, static public IPv4 address**. This is the IP address that the company can provide to the external provider for their allow list.
    
- Configuring the VPC (specifically the **route table** for the Lambda's private subnet) to route internet-bound traffic ($0.0.0.0/0$) to the NAT GW ensures all outbound traffic from the Lambda function will appear to originate from the NAT Gateway's EIP.
    

> _Reference: AWS Documentation - NAT Gateways: "A NAT gateway uses a static IP address. When you create a NAT gateway, you associate an Elastic IP address with it. You cannot associate multiple Elastic IP addresses with a NAT gateway."_

---

#### Incorrect Options Explanation

**B. Deploy an egress-only internet gateway. Associate an Elastic IP address with the egress-only internet gateway. Configure the elastic network interface on the Lambda function to use the egress-only internet gateway.**

- **Egress-Only Internet Gateway (EIGW)** is for **IPv6 traffic only**. The requirement is for a public **IPv4** address.
    
- EIGWs do not have Elastic IP addresses associated with them for the purpose of a single-point IPv4 NAT.
    

**C. Deploy an internet gateway. Associate an Elastic IP address with the internet gateway. Configure the Lambda function to use the internet gateway.**

- An **Internet Gateway (IGW)** is a component for routing traffic to and from the internet; it is not a NAT device.
    
- You **cannot associate an Elastic IP address directly with an IGW** to act as a single NAT IP for multiple resources. This is the function of a NAT Gateway.
    
- Lambda functions are configured via VPC settings (subnets, security groups) and their outbound traffic is controlled by the **route table** of the subnet they are in, not by directly "configuring the Lambda function" to use the IGW.
    

**D. Deploy an internet gateway. Associate an Elastic IP address with the internet gateway. Configure the default route in the public VPC route table to use the internet gateway.**

- Similar to option C, you **cannot associate an Elastic IP address directly with an IGW** to provide a single, static public IP for outbound NAT.
    
- Configuring the default route in the _public_ VPC route table to use the IGW is standard practice for public subnets, but it doesn't solve the problem of providing a _single, static_ outbound IP for a Lambda function typically running in a **private subnet** for security. If the Lambda were in a public subnet, its ENI wouldn't have a public IP by default, and it would still need a NAT device (or an EIP attached to its ENI, which is not scalable or easily manageable for Lambda).
