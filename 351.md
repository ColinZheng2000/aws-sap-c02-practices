A company is running its solution on AWS in a manually created VPC. The company is using AWS CloudFormation to provision other parts of the infrastructure. According to a new requirement, the company must manage all infrastructure in an automatic way.

What should the company do to meet this new requirement with the LEAST effort?

A. Create a new AWS Cloud Development Kit (AWS CDK) stack that strictly provisions the existing VPC resources and configuration. Use AWS CDK to import the VPC into the stack and to manage the VPC.
B. Create a CloudFormation stack set that creates the VPC. Use the stack set to import the VPC into the stack.
C. Create a new CloudFormation template that strictly provisions the existing VPC resources and configuration. From the CloudFormation console, create a new stack by importing the Existing resources.
D. Create a new CloudFormation template that creates the VPC. Use the AWS Serverless Application Model (AWS SAM) CLI to import the VPC.


# AWS SAP-C02 Question Analysis — Q351

## 📋 Analysis

### 1. Understanding the Problem

The environment already contains a manually created production VPC. The company has partially adopted Infrastructure as Code (IaC) by using AWS CloudFormation for other infrastructure components, but now wants all infrastructure to be managed automatically.

The key requirement is:
- Bring existing manually created infrastructure under IaC management
- Use the least operational effort
- Avoid rebuilding or migrating the production network

This is fundamentally an AWS CloudFormation resource import scenario.

Core AWS knowledge tested:
- CloudFormation resource import
- Existing resource adoption into IaC
- Difference between CloudFormation, CDK, SAM, and StackSets
- Operational risk reduction in enterprise environments

---

### 2. Options Analysis

#### ✅ Option C (CORRECT): Import existing VPC resources into a CloudFormation stack

This is the correct solution because AWS CloudFormation natively supports importing existing resources into a stack.

Why this fully satisfies the requirement:
- The company already uses CloudFormation
- Existing VPC resources can be adopted without recreation
- Infrastructure becomes centrally managed through IaC
- Minimal migration effort
- Lowest operational risk

Correct implementation flow:
1. Create a CloudFormation template that accurately describes the existing VPC configuration
2. Use CloudFormation resource import
3. CloudFormation begins managing the existing resources

This is the AWS-recommended approach for:
- Legacy infrastructure modernization
- Gradual IaC adoption
- Governance standardization
- Drift detection enablement

Official AWS documentation:
- AWS CloudFormation resource import
- https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/resource-import.html

Important enterprise insight:
Recreating a production VPC is usually avoided because:
- VPC IDs are referenced by many systems
- Subnet IDs are embedded in deployments
- Security groups may be integrated with databases and partner allowlists
- Route tables, peering, Transit Gateway attachments, and Direct Connect integrations may already exist

Therefore, importing existing resources is significantly safer than rebuilding.

---

#### ❌ Option A (INCORRECT): Use AWS CDK import

AWS CDK is powerful for modern IaC development, but this is not the least-effort solution.

Why it is incorrect:
- CDK ultimately synthesizes CloudFormation templates underneath
- CDK import workflows are more operationally involved
- Introducing CDK adds additional tooling and deployment processes
- The company already uses CloudFormation directly

Key SAP-C02 lesson:
When the requirement emphasizes “least effort,” avoid introducing unnecessary abstraction layers.

Additional enterprise consideration:
Many enterprises standardize first on raw CloudFormation before later evolving toward CDK-based platform engineering.

---

#### ❌ Option B (INCORRECT): Use CloudFormation StackSets

StackSets are designed for:
- Multi-account deployments
- Multi-region deployments
- AWS Organizations governance

Why this does not fit:
- The problem is not centralized deployment at scale
- StackSets are not intended for importing standalone existing infrastructure
- This adds unnecessary architectural complexity

Common exam trap:
AWS often includes StackSets whenever CloudFormation appears, but StackSets only solve organization-wide deployment scenarios.

---

#### ❌ Option D (INCORRECT): Use AWS SAM CLI

AWS SAM focuses on:
- AWS Lambda
- API Gateway
- Event-driven serverless applications

Why this is incorrect:
- SAM is not intended for VPC infrastructure lifecycle management
- SAM does not provide the proper workflow for importing existing VPC resources
- The scenario is unrelated to serverless workloads

SAP-C02 exam pattern:
If the workload is not serverless-focused, SAM is frequently a distractor option.

---

### 3. Hands-On Knowledge

In enterprise environments, importing existing infrastructure into CloudFormation requires careful planning.

Typical production migration process:

#### Step 1 — Discover Existing Resources

Inventory:
- VPC IDs
- Subnets
- Route tables
- NAT gateways
- Internet gateways
- Security groups
- NACLs
- DHCP option sets

Useful CLI example:
```bash
aws ec2 describe-vpcs
```

---

#### Step 2 — Build Matching CloudFormation Template

The template must match the existing configuration exactly.

Example:
```yaml
Resources:
  ProdVPC:
    Type: AWS::EC2::VPC
    Properties:
      CidrBlock: 10.0.0.0/16
```

Mismatch risks:
- CIDR mismatch
- Missing tags
- Incorrect DNS settings
- Dependency inconsistency

These can cause import failure.

---

#### Step 3 — Import Resources

Typical workflow:
```bash
aws cloudformation create-change-set \
  --stack-name network-stack \
  --change-set-name import-vpc \
  --change-set-type IMPORT
```

CloudFormation then associates the existing physical resource with the logical stack resource.

---

#### Step 4 — Run Drift Detection

After import:
```bash
aws cloudformation detect-stack-drift \
  --stack-name network-stack
```

This helps identify:
- Manual configuration drift
- Unauthorized changes
- Compliance deviations

---

#### Real-World Enterprise Gotchas

1. Not all resources support import
Always verify supported resource types:
https://docs.aws.amazon.com/AWSCloudFormation/latest/UserGuide/resource-import-supported-resources.html

2. Imported resources are not recreated automatically
If someone deletes the resource manually afterward, stack recovery may require additional remediation.

3. Networking imports should be phased
Import foundational components first:
- VPC
- Subnets
- Route tables
- Security groups

Then import dependent workloads later.

4. Use Change Sets before execution
This reduces accidental replacement risk.

---

#### Enterprise Architecture Best Practice

A common modernization path:

```text
Manual Infrastructure
        ↓
CloudFormation Import
        ↓
Drift Detection
        ↓
CI/CD Automation
        ↓
Governed IaC Platform
```

This approach is heavily used in:
- Pharmaceutical companies
- Manufacturing enterprises
- Financial institutions
- Healthcare environments

because production networking changes are highly sensitive.

**面试官可能追问 (Possible Follow-up Questions)**：

1. "CloudFormation resource import有哪些限制？"
   → 回答要点 (Key Answer Points)：  
   不是所有AWS资源都支持Import；模板属性必须与现有资源严格一致；Import不会自动创建依赖关系；部分资源属性不可修改，否则会触发Replacement。

2. "为什么生产环境通常避免重建VPC？"
   → 回答要点 (Key Answer Points)：  
   生产VPC通常已经与大量资源耦合，包括Subnet ID、Security Group、Transit Gateway、Peering、Direct Connect、数据库白名单等；重建会导致大规模依赖变更和高停机风险。
