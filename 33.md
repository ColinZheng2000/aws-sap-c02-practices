A solutions architect needs to advise a company on how to migrate its on-premises data processing application to the AWS Cloud. Currently, users upload input files through a web portal. The web server then stores the uploaded files on NAS and messages the processing server over a message queue. Each media file can take up to 1 hour to process. The company has determined that the number of media files awaiting processing is significantly higher during business hours, with the number of files rapidly declining after business hours.  
  
What is the MOST cost-effective migration recommendation?

- A. Create a queue using Amazon SQS. Configure the existing web server to publish to the new queue. When there are messages in the queue, invoke an AWS Lambda function to pull requests from the queue and process the files. Store the processed files in an Amazon S3 bucket.
- B. Create a queue using Amazon MQ. Configure the existing web server to publish to the new queue. When there are messages in the queue, create a new Amazon EC2 instance to pull requests from the queue and process the files. Store the processed files in Amazon EFS. Shut down the EC2 instance after the task is complete.
- C. Create a queue using Amazon MQ. Configure the existing web server to publish to the new queue. When there are messages in the queue, invoke an AWS Lambda function to pull requests from the queue and process the files. Store the processed files in Amazon EFS.
- D. Create a queue using Amazon SQS. Configure the existing web server to publish to the new queue. Use Amazon EC2 instances in an EC2 Auto Scaling group to pull requests from the queue and process the files. Scale the EC2 instances based on the SQS queue length. Store the processed files in an Amazon S3 bucket.
This is a classic AWS Solutions Architect - Professional level question that focuses on choosing the most **cost-effective** and **scalable** architecture for a variable-load, asynchronous data processing workload.


## Analysis

### Understanding the Problem

The company needs to migrate an on-premises data processing application to AWS.

- **Current Architecture:** Web server $\rightarrow$ Stores file on **NAS** $\rightarrow$ Messages processing server over a **message queue**.
    
- **Workload Characteristics:**
    
    - **Asynchronous:** File upload and processing are decoupled by a message queue.
        
    - **Long Processing Time:** Each file takes up to $\mathbf{1}$ **hour** to process.
        
    - **Variable/Spiky Load:** Significantly higher during business hours and rapidly declining afterward.
        
- **Primary Goal:** Determine the **MOST cost-effective** migration recommendation that can handle the variable load.
    

The key to cost-effectiveness here is **elasticity** (scaling out during peak and scaling in/down during off-peak) and selecting the services with the most favorable pricing model for a $\mathbf{1}$-**hour long, variable-load task.**

### Services Explanation

|**Service**|**Category**|**Use Case Relevance**|**Key Feature for Cost-Effectiveness**|
|---|---|---|---|
|**Amazon SQS (Simple Queue Service)**|Message Queue|Highly scalable, fully managed message queuing service.|**Decoupling** components, **pay-per-use** for messages, virtually **unlimited scale**. Ideal for variable load.|
|**Amazon MQ**|Message Queue|Managed message broker service for compatibility with existing standards (ActiveMQ, RabbitMQ).|Good for lift-and-shift, but **less cost-effective** than SQS because you pay for the broker instance size/runtime, regardless of use.|
|**AWS Lambda**|Compute|Serverless compute service; runs code in response to events.|**Pay-per-use** (duration and memory), **automatic scaling** to huge volumes. **Cost-effective** for _short-duration_ tasks.|
|**Amazon EC2**|Compute|Virtual machines (VMs) for running applications.|Provides flexible, customizable compute. Can scale with Auto Scaling, but **you pay for running instance time**.|
|**EC2 Auto Scaling**|Scaling|Automatically adjusts the number of EC2 instances based on demand (e.g., SQS queue length).|Ideal for variable load on EC2, ensuring resources match demand.|
|**Amazon S3 (Simple Storage Service)**|Object Storage|Highly durable, scalable, and low-cost object storage.|**Cost-effective** for large volumes of static data; ideal for **processed file storage**.|
|**Amazon EFS (Elastic File System)**|File Storage|Scalable, elastic, POSIX-compliant file storage.|**Expensive** compared to S3; better suited for use cases requiring a shared file system (like NAS replacement for EC2 fleets).|

### Correct Option(s) Explanation

The correct option is **D. Create a queue using Amazon SQS. Configure the existing web server to publish to the new queue. Use Amazon EC2 instances in an EC2 Auto Scaling group to pull requests from the queue and process the files. Scale the EC2 instances based on the SQS queue length. Store the processed files in an Amazon S3 bucket.**

1. **Amazon SQS for the Queue:** SQS is the **most cost-effective** queueing service on AWS compared to Amazon MQ, as it's fully consumption-based and scales to virtually any load without pre-provisioning brokers.
    
2. **Amazon EC2 with Auto Scaling for Processing:**
    
    - The processing time is **up to 1 hour**. This is **well beyond the current Lambda maximum timeout of 15 minutes** (as of October 2025). This immediately rules out any solution using AWS Lambda (Options A and C).
        
    - **EC2 Auto Scaling** is the correct compute choice for long-running, variable-load tasks. **Scaling based on SQS queue length** is the best practice for cost-effectiveness, ensuring that the number of running EC2 instances directly tracks the _actual_ amount of work waiting to be done. The Auto Scaling group will automatically terminate idle instances when the queue is empty (off-peak), directly addressing the variable load and fulfilling the **MOST cost-effective** requirement.
        
    - _Reference:_ AWS documentation on [Scaling Your Auto Scaling Group with Amazon SQS](https://docs.aws.amazon.com/autoscaling/ec2/userguide/as-using-sqs-queue.html) highlights this as a best practice for decoupled architectures.
        
3. **Amazon S3 for Storage:** S3 is the most scalable, durable, and **cost-effective** option for storing the final processed media files.
    

### Incorrect Options Explanation

- **A. Create a queue using Amazon SQS. Configure the existing web server to publish to the new queue. When there are messages in the queue, invoke an AWS Lambda function to pull requests from the queue and process the files. Store the processed files in an Amazon S3 bucket.**
    
    - **Incorrect:** AWS Lambda has a **maximum execution time limit of 15 minutes**. Since the processing takes up to **1 hour**, Lambda cannot be used, making this option technically unviable.
        
- **B. Create a queue using Amazon MQ. Configure the existing web server to publish to the new queue. When there are messages in the queue, create a new Amazon EC2 instance to pull requests from the queue and process the files. Store the processed files in Amazon EFS. Shut down the EC2 instance after the task is complete.**
    
    - **Incorrect:**
        
        1. **Amazon MQ:** This is less cost-effective than SQS because you pay for the underlying broker instance even during low-use periods, violating the "MOST cost-effective" requirement.
            
        2. **Creating a "new" EC2 instance for _each_ message:** This is an operationally complex and highly inefficient pattern. Launching a new EC2 instance takes several minutes, adding significant delay and incurring substantial costs compared to using an existing Auto Scaling group that scales out pre-configured instances.
            
        3. **Amazon EFS:** EFS is significantly more expensive than S3 for static processed file storage.
            
- **C. Create a queue using Amazon MQ. Configure the existing web server to publish to the new queue. When there are messages in the queue, invoke an AWS Lambda function to pull requests from the queue and process the files. Store the processed files in Amazon EFS.**
    
    - **Incorrect:**
        
        1. **AWS Lambda:** Ruled out because the $\mathbf{1}$-**hour processing time** exceeds the 15-minute limit (same reason as Option A).
            
        2. **Amazon MQ:** Less cost-effective than SQS (same reason as Option B).
            
        3. **Amazon EFS:** Less cost-effective than S3 (same reason as Option B).
