A company is planning a one-time migration of an on-premises MySQL database to Amazon Aurora MySQL in the us-east-1 Region. The company's current internet connection has limited bandwidth. The on-premises MySQL database is 60 TB in size. The company estimates that it will take a month to transfer the data to AWS over the current internet connection. The company needs a migration solution that will migrate the database more quickly.

Which solution will migrate the database in the LEAST amount of time?

A. Request a 1 Gbps AWS Direct Connect connection between the on-premises data center and AWS. Use AWS Database Migration Service (AWS DMS) to migrate the on-premises MySQL database to Aurora MySQL.
B. Use AWS DataSync with the current internet connection to accelerate the data transfer between the on-premises data center and AWS. Use AWS Application Migration Service to migrate the on-premises MySQL database to Aurora MySQL.
C. Order an AWS Snowball Edge device. Load the data into an Amazon S3 bucket by using the S3 interface. Use AWS Database Migration Service (AWS DMS) to migrate the data from Amazon S3 to Aurora MySQL.
D. Order an AWS Snowball device. Load the data into an Amazon S3 bucket by using the S3 Adapter for Snowball. Use AWS Application Migration Service to migrate the data from Amazon S3 to Aurora MySQL.

Suggest Answer: C

---

## Analysis

### Understanding the Problem

A company needs to perform a **one-time migration** of a **60 TB on-premises MySQL database** to **Amazon Aurora MySQL** in `us-east-1`. The key constraints:
- **Limited internet bandwidth** at the on-premises data center
- Current estimate: **1 month** to transfer 60 TB over the existing internet connection
- **Goal: minimize total migration time** (not cost, not effort — pure speed)

The bottleneck is the **physical data transfer** of 60 TB from on-premises to AWS. This is a classic "large dataset, limited bandwidth" scenario where **offline data transfer appliances** (AWS Snow Family) are the optimal solution.

**Scale check — 60 TB over limited internet:**
- If "limited bandwidth" = 100 Mbps (12.5 MB/s theoretical), 60 TB ≈ 60,000,000 MB → 60,000,000 / 12.5 = 4,800,000 seconds ≈ **55 days** at 100% utilization
- Even at 500 Mbps, it's 11+ days. Real-world throughput is lower due to TCP overhead, packet loss, and shared bandwidth.

### Correct Option: C

**Order an AWS Snowball Edge device → Load data into S3 via the S3 interface → Use AWS DMS to migrate from S3 to Aurora MySQL.**

- **Snowball Edge solves the bandwidth problem**: Snowball Edge provides up to **100 TB** of usable storage in a ruggedized physical appliance. The 60 TB database fits comfortably within a single device. Data transfer happens at **local network speeds** (typically 10 Gbps or higher via RJ45/SFP+), not over the internet. Once shipped back to AWS, the data is loaded into S3 within days — far faster than a month over the internet.
- **S3 interface (native)**: Snowball Edge exposes an **Amazon S3-compatible endpoint** locally. You can use standard S3 tools (AWS CLI with `--endpoint-url`, S3 SDKs) to copy data onto the device. No special adapters needed — the S3 interface is a built-in feature of Snowball Edge.
- **AWS DMS with S3 as source**: AWS Database Migration Service supports **Amazon S3 as a source endpoint**. You export your MySQL data to CSV, Parquet, or other formats into S3 (via Snowball Edge), then DMS reads from S3 and migrates to Aurora MySQL. DMS handles schema conversion (if needed with Schema Conversion Tool), ongoing replication, and validation.
- **End-to-end workflow is proven and documented**: Snowball Edge → S3 → DMS → Aurora is a well-established AWS migration pattern for large databases.
- **One-time migration appropriate**: Since this is a one-time migration (not ongoing sync), the offline transfer model is perfectly suited. There's no need for a permanent high-bandwidth connection like Direct Connect.

**Estimated time comparison:**
| Approach | Approximate Time |
|---|---|
| Current internet (limited) | ~30 days |
| 1 Gbps Direct Connect | ~6-8 days (pure transfer) + weeks for cross-connect setup |
| Snowball Edge + DMS | 3-5 days (data copy to device) + 2-3 days (shipping) + 1-2 days (S3→Aurora via DMS) = **~7-10 days total** |

### Incorrect Options

**A: 1 Gbps Direct Connect + AWS DMS**
- **Direct Connect setup takes weeks, not days**: Ordering, installing, and provisioning a Direct Connect connection involves working with AWS and the colocation provider. The cross-connect alone can take 2-4 weeks or longer. This defeats the purpose of "fastest migration."
- **1 Gbps is still limited for 60 TB**: At theoretical maximum 1 Gbps (125 MB/s), 60 TB = 60,000,000 MB → 480,000 seconds ≈ **5.5 days** of pure transfer at 100% utilization. Real-world throughput is typically 70-80% due to TCP overhead and DMS processing. Plus setup time, this is NOT faster than Snowball Edge.
- **Ongoing cost**: Direct Connect has monthly port charges. For a one-time migration, this is cost-inefficient.
- **DMS over the network for 60 TB**: DMS performs best with stable, high-bandwidth connections. Running a full load of 60 TB over a network connection (even Direct Connect) risks interruptions, network fluctuations, and requires careful task monitoring.

**B: AWS DataSync over current internet + AWS Application Migration Service**
- **DataSync does not solve the bandwidth problem**: DataSync accelerates transfers using parallel streams, compression, and WAN optimization, but it still operates over the same internet link. It cannot make a 100 Mbps link behave like a 10 Gbps link. For a 60 TB dataset, DataSync over limited internet would still take weeks.
- **Application Migration Service is for SERVER migration, NOT database migration**: AWS Application Migration Service (formerly CloudEndure Migration) is designed for **block-level replication of servers** (lift-and-shift of VMs). It is not a database migration tool. Using it to migrate a MySQL database implies migrating the entire server, not converting to Aurora's managed database engine. This introduces unnecessary complexity and misses the benefits of Aurora.
- **Wrong tool combination**: DataSync + Application Migration Service addresses neither the bandwidth constraint nor the database migration requirement correctly.

**D: AWS Snowball (classic) + S3 Adapter + AWS Application Migration Service**
- **Snowball (classic, non-Edge) is limited to 50 TB**: The original Snowball device has **50 TB** of usable storage (80 TB raw). A 60 TB database **cannot fit** on a single Snowball device. You would need to order and manage multiple devices, increasing complexity and migration time.
- **S3 Adapter is legacy technology**: The S3 Adapter for Snowball was a software component for older Snowball devices. Snowball Edge natively supports the S3 interface without requiring a separate adapter. Using the S3 Adapter adds unnecessary configuration and compatibility overhead.
- **Application Migration Service is wrong for database migration** (same issue as option B): It migrates servers at the block level, not databases to managed services like Aurora.
- **Snowball Edge was specifically designed to replace classic Snowball** for large-scale data transfer use cases (up to 100 TB, with compute capabilities).

### Hands-On Enterprise Knowledge

**Large Database Migration to Aurora using Snowball Edge + DMS — Production Workflow:**

**Phase 1: Preparation (On-Premises)**
1. **Export MySQL data to S3-compatible format** (CSV or Parquet):
   ```bash
   # Use mysqldump or SELECT INTO OUTFILE for each table
   mysqldump --tab=/export/path --fields-terminated-by=',' \
     --fields-enclosed-by='"' --lines-terminated-by='\n' \
     -h localhost -u admin -p mydatabase
   ```
2. **Order Snowball Edge** via AWS Console (Snow Family → Order). Select **Snowball Edge Storage Optimized** (up to 100 TB usable).
3. **Receive the device** (typically 2-3 business days after ordering).

**Phase 2: Data Copy to Snowball Edge**
1. Connect Snowball Edge to the local network (RJ45, SFP+, or QSFP+ interfaces).
2. Unlock the device using the AWS OpsHub GUI or CLI.
3. Use the **S3 interface** (natively exposed by Snowball Edge) to copy data:
   ```bash
   aws s3 cp /export/path/ s3://my-migration-bucket/ \
     --endpoint-url https://<snowball-ip>:8443 \
     --recursive
   ```
4. Verify data integrity (use `s3api head-object` or MD5 checksums).

**Phase 3: AWS-Side Migration**
1. Ship the device back to AWS. AWS loads the data into the designated S3 bucket (typically 1-2 business days after receipt).
2. **Create an Aurora MySQL target cluster** in us-east-1 with appropriate instance class (e.g., `db.r6g.4xlarge` for production workloads).
3. **Configure DMS**:
   - **Source endpoint**: Amazon S3 (specify bucket, prefix, and file format)
   - **Target endpoint**: Aurora MySQL
   - **Replication instance**: Choose a `dms.c5.4xlarge` or larger for 60 TB throughput
   - **Task**: Full load with validation enabled
4. Execute the DMS task and monitor via CloudWatch.

**Phase 4: Cutover**
1. After full load completes, perform data validation (row counts, checksums).
2. If needed, enable DMS ongoing replication (CDC from S3 is not possible — for CDC, you would set up a temporary DMS task from on-premises MySQL over the internet with just the change data, which is much smaller than 60 TB).
3. Update application connection strings to point to Aurora endpoint.

**Production Considerations:**
- **Encryption**: Snowball Edge supports 256-bit encryption by default. Ensure the S3 bucket also has server-side encryption enabled.
- **Compliance/Chain of Custody**: Snowball Edge provides an E Ink shipping label, GPS tracking, and tamper-evident enclosure. Ideal for medical/industrial companies with data sovereignty concerns.
- **Schema Conversion**: If the on-premises database uses MySQL-specific features not supported by Aurora MySQL (rare, as Aurora is MySQL-compatible), use the AWS Schema Conversion Tool (SCT) before DMS migration.
- **Cost**: Snowball Edge has a service fee per job (+ shipping). For 60 TB one-time migration, this is significantly cheaper than a month of Direct Connect port charges.
