An AWS customer has a web application that runs on premises. The web application fetches data from a third-party API that is behind a firewall. The third party accepts only one public CIDR block in each client's allow list.  
The customer wants to migrate their web application to the AWS Cloud. The application will be hosted on a set of Amazon EC2 instances behind an Application  
Load Balancer (ALB) in a VPC. The ALB is located in public subnets. The EC2 instances are located in private subnets. NAT gateways provide internet access to the private subnets.  
How should a solutions architect ensure that the web application can continue to call the third-party API after the migration?  

- A. Associate a block of customer-owned public IP addresses to the VPC. Enable public IP addressing for public subnets in the VPC.
- B. Register a block of customer-owned public IP addresses in the AWS account. Create Elastic IP addresses from the address block and assign them to the NAT gateways in the VPC.
- C. Create Elastic IP addresses from the block of customer-owned IP addresses. Assign the static Elastic IP addresses to the ALB.
- D. Register a block of customer-owned public IP addresses in the AWS account. Set up AWS Global Accelerator to use Elastic IP addresses from the address block. Set the ALB as the accelerator endpoint.

---

### Understanding the Problem

The customer is migrating a web application from on-premises to AWS.

- **Current Setup:** On-premises web application fetches data from a **third-party API** that is behind a firewall.
    
- **Key Constraint:** The third-party API firewall **only allows one public CIDR block** from the client (the web application) in its allow list.
    
- **AWS Target Architecture:**
    
    - Web application runs on **Amazon EC2 instances** in **private subnets**.
        
    - **Application Load Balancer (ALB)** in **public subnets**.
        
    - **NAT gateways** provide outbound internet access for the EC2 instances in private subnets.
        
- **Goal:** Ensure the web application can continue to call the third-party API using a **single, predictable public CIDR block** after migration.
    

Since the EC2 instances are in private subnets, their outbound traffic to the internet (which includes the third-party API) must pass through the **NAT gateway**. Therefore, the solution must assign a **static, predictable public IP address or CIDR block** to the NAT gateway's outbound interface.

### Services Explanation

|**NAT Gateway**|A Network Address Translation (NAT) service that enables instances in a private subnet to connect to the internet or other AWS services, but prevents the internet from initiating a connection with those instances. A NAT Gateway must be in a public subnet and **must be associated with an Elastic IP address** for its outbound traffic.|**Crucial.** It dictates the public IP address(es) that the web application's outbound traffic will use to reach the third-party API.|
|**Elastic IP Address (EIP)**|A static, public IPv4 address that you can associate with an instance or network interface (like a NAT Gateway). An EIP remains static even if the associated resource changes.|**Essential.** Standard EIPs are managed by AWS. They provide the necessary static IP for the NAT Gateway.|

This is a complex way to achieve a static outbound IP, but aligns with the wording of option B.|
|**Application Load Balancer (ALB)**|A service that automatically distributes incoming application traffic across multiple targets, such as EC2 instances.|**Not relevant for outbound traffic.** The ALB handles _inbound_ traffic from clients to the EC2 instances. It can provide static IPs for _inbound_ traffic but is overkill and incorrect for _outbound_ traffic from an EC2 instance behind a NAT.|

### Correct Option(s) Explanation

**B. Register a block of customer-owned public IP addresses in the AWS account. Create Elastic IP addresses from the address block and assign them to the NAT gateways in the VPC.**

- **Mechanism:** When an EC2 instance in a private subnet makes an outbound connection to the internet, its traffic is routed through the **NAT Gateway** in the public subnet. The NAT Gateway's public-facing IP address becomes the source IP address for this outbound traffic.
    
- **Solution Alignment:** By assigning a **static Elastic IP address (EIP)** to the NAT Gateway, the web application's outbound traffic will always appear to originate from that single EIP. The third-party API only needs to allow-list the CIDR block containing this EIP (which, if it's a single EIP, is a /32 block).
    
- **Customer-Owned IP (CoIP) Context:** The prompt specifically mentions a _customer-owned public IP address block_. The process of **registering** (bringing in) a CoIP block (using a service like IPAM, as referenced in the AWS documentation for **Bring Your Own IP (BYOIP)**) and then provisioning **EIPs** from that block to assign to the NAT Gateway is the correct, albeit slightly more complex, professional solution to meet the requirement of using a single, customer-specified public CIDR block for outbound traffic.
    

_Reference: **AWS Documentation on NAT Gateway and Elastic IP Addresses** - A NAT gateway requires an Elastic IP address. The EIP is used to send traffic to the internet._

---

### Incorrect Options Explanation

**A. Associate a block of customer-owned public IP addresses to the VPC. Enable public IP addressing for public subnets in the VPC.**

- **Flaw:** The EC2 instances are in **private subnets**, so enabling public IP addressing on **public subnets** is irrelevant to their outbound traffic. Public IP addressing on the EC2 instances themselves would only matter if they were in public subnets, which they are not. Furthermore, simply "associating a block" to a VPC doesn't automatically make it the source IP for NAT Gateway traffic; a specific EIP must be assigned to the NAT Gateway.
    

**C. Create Elastic IP addresses from the block of customer-owned IP addresses. Assign the static Elastic IP addresses to the ALB.**

- **Flaw:** The **Application Load Balancer (ALB)** is an **inbound** traffic service. It handles traffic _from_ clients _to_ the EC2 instances. It has **no role** in the **outbound** traffic flow _from_ the EC2 instances _to_ the third-party API. The outbound traffic is routed through the NAT Gateway.
    

**D. Register a block of customer-owned public IP addresses in the AWS account. Set up AWS Global Accelerator to use Elastic IP addresses from the address block. Set the ALB as the accelerator endpoint.**

- **Flaw:** **AWS Global Accelerator** is an **inbound** service. While it provides static, anycast IP addresses, these are meant for clients on the internet to connect **to** the application. It is completely irrelevant for controlling the **outbound** source IP address of the EC2 instances when they connect to an external API.
