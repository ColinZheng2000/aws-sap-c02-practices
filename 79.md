A company has developed a web application. The company is hosting the application on a group of Amazon EC2 instances behind an Application Load Balancer.  
The company wants to improve the security posture of the application and plans to use AWS WAF web ACLs. The solution must not adversely affect legitimate traffic to the application.  
How should a solutions architect configure the web ACLs to meet these requirements?  

- A. Set the action of the web ACL rules to Count. Enable AWS WAF logging. Analyze the requests for false positives. Modify the rules to avoid any false positive. Over time, change the action of the web ACL rules from Count to Block.
- B. Use only rate-based rules in the web ACLs, and set the throttle limit as high as possible. Temporarily block all requests that exceed the limit. Define nested rules to narrow the scope of the rate tracking.
- C. Set the action of the web ACL rules to Block. Use only AWS managed rule groups in the web ACLs. Evaluate the rule groups by using Amazon CloudWatch metrics with AWS WAF sampled requests or AWS WAF logs.
- D. Use only custom rule groups in the web ACLs, and set the action to Allow. Enable AWS WAF logging. Analyze the requests for false positives. Modify the rules to avoid any false positive. Over time, change the action of the web ACL rules from Allow to Block.

# AWS WAF Deployment Strategy for Production Traffic
---

## Analysis

### 1. Understanding the Problem
The scenario presents a classic operational challenge in security engineering: **False Positives vs. Security Enforcement**.
* **The Goal:** Implement AWS WAF to protect an application running on EC2 behind an ALB.
* **The Constraint:** The solution must **not** adversely affect legitimate traffic (Availability).

If you immediately deploy WAF rules (especially Managed Rules) with the action set to `Block`, there is a high probability that legitimate traffic patterns will match a generic attack signature (a false positive), causing valid users to face 403 Forbidden errors. The question requires a deployment strategy that validates the rules against live traffic without dropping packets.

### 2. Services Explanation

* **AWS WAF (Web Application Firewall):** A web application firewall that helps protect your web applications or APIs against common web exploits and bots that may affect availability, compromise security, or consume excessive resources.
* **Web ACL (Web Access Control List):** The core resource in AWS WAF. It contains the rules (managed or custom) that define how to inspect web requests.
* **Rule Action - Count:** This is a non-terminating action (in most contexts) used for testing. When a request matches a rule with the `Count` action, WAF counts the request in CloudWatch metrics and adds an entry to the WAF logs, but it **does not block** the request. It allows the request to pass through to the backend (or the next rule).
* **AWS WAF Logging:** Captures detailed information about traffic analyzed by your Web ACL, including the timestamp, header details, and which rule was matched. This is crucial for forensic analysis and rule tuning.



### 3. Correct Option Explanation

**Option A is correct** because it follows the AWS Best Practice for deploying WAF in a production environment.

1.  **Baseline (Count Mode):** By setting rules to `Count`, the WAF inspects traffic and reports matches (potential attacks) without interfering with the user experience. This satisfies the requirement to "not adversely affect legitimate traffic."
2.  **Analysis:** By enabling WAF logging, the Solutions Architect can review which requests triggered the rules. If legitimate traffic triggered a rule (a false positive), the Architect can refine the rule (e.g., adding an exclusion for a specific URI or header).
3.  **Enforcement (Block Mode):** Once the logs show that only malicious traffic is being flagged and legitimate traffic is safe, the action is switched to `Block` to actually secure the application.

### 4. Incorrect Options Explanation

* **Option B is incorrect.**
    * **Limited Scope:** Rate-based rules primarily protect against volumetric attacks (DDoS, brute force). They do not protect against SQL Injection, Cross-Site Scripting (XSS), or other OWASP Top 10 vulnerabilities. Using *only* rate-based rules fails to significantly "improve the security posture" regarding application-layer exploits.
    * **Configuration:** Setting the limit "as high as possible" defeats the purpose of rate limiting, as it would likely allow malicious floods to pass through.

* **Option C is incorrect.**
    * **Violation of Requirements:** Setting the action immediately to `Block` is the most common cause of production outages during WAF implementation. Managed Rule Groups are generic; they often flag legitimate proprietary API calls or complex form submissions as malicious. This would likely block legitimate traffic, violating the prompt's core requirement.

* **Option D is incorrect.**
    * **Operational Overhead:** Using *only* custom rule groups implies the company cannot use AWS Managed Rules (which are updated by AWS security researchers). This increases operational burden significantly.
    * **Logic Flaw:** While the "Allow" logic could theoretically work for testing (if the default action is Block), the industry standard and AWS-native feature designed specifically for this testing phase is the **Count** action, not "Allow."
