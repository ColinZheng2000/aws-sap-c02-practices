A company is building a software-as-a-service (SaaS) solution on AWS. The company has deployed an Amazon API Gateway REST API with AWS Lambda integration in multiple AWS Regions and in the same production account.  
The company offers tiered pricing that gives customers the ability to pay for the capacity to make a certain number of API calls per second. The premium tier offers up to 3,000 calls per second, and customers are identified by a unique API key. Several premium tier customers in various Regions report that they receive error responses of 429 Too Many Requests from multiple API methods during peak usage hours. Logs indicate that the Lambda function is never invoked.  
What could be the cause of the error messages for these customers?  

- A. The Lambda function reached its concurrency limit.
- B. The Lambda function its Region limit for concurrency.
- C. The company reached its API Gateway account limit for calls per second.
- D. The company reached its API Gateway default per-method limit for calls per second.

# AWS SAP-C02 Exam Analysis: API Gateway Throttling & Limits

---

## Analysis

### 1. Understanding the Problem
The core of this scenario lies in identifying where the "throttle" is occurring in the request flow.
* **Architecture:** API Gateway $\rightarrow$ Lambda.
* **Symptom:** Clients receive `429 Too Many Requests`.
* **Critical Evidence:** The logs show the **Lambda function is never invoked**.
* **Context:** Multi-tenant SaaS, single account, multiple regions, high-volume (Premium tier = 3,000 RPS).

The HTTP `429` error indicates throttling. However, throttling can occur at the API Gateway layer (Token Bucket algorithm) or the Lambda layer (Concurrency limits). The fact that the Lambda function is "never invoked" implies the request was rejected at the API Gateway entry point before it could even attempt to trigger the backend service.

### 2. Services Explanation

* **Amazon API Gateway Throttling:**
    API Gateway applies throttling at multiple levels using the Token Bucket Algorithm.
    1.  **Account-Level Throttling:** By default, there is a hard limit of **10,000 requests per second (RPS)** (with a burst of 5,000) **per Region** per AWS Account. This limit is shared across *all* APIs in that specific Region and Account.
    2.  **Usage Plan Throttling:** Limits applied to specific API Keys (e.g., the 3,000 RPS Premium tier mentioned).
    3.  **Method-Level Throttling:** Granular limits set on specific API resource methods.

* **AWS Lambda Concurrency:**
    * **Reserved Concurrency:** Guarantees a specific number of concurrent instances for a function.
    * **Provisioned Concurrency:** Pre-initializes execution environments.
    * If Lambda throttles, it returns a 429 to the caller (API Gateway). However, typically, this is logged as an integration failure or latency in API Gateway logs, whereas API Gateway throttling stops the request at the "front door."

### 3. Correct Option Explanation

**C. The company reached its API Gateway account limit for calls per second.**

* **Logic:** API Gateway has a default account-level throttle limit of 10,000 RPS per Region.
* **The "Noisy Neighbor" Effect:** The company hosts this SaaS in a **single production account**. Even though individual Premium customers are limited to 3,000 RPS via Usage Plans, if you have 4 Premium customers active in the same Region simultaneously, the total demand could be 12,000 RPS (4 * 3,000).
* **Result:** The aggregate traffic exceeds the 10,000 RPS Account Limit. When the account limit is reached, API Gateway rejects excess requests with a `429 Too Many Requests` error immediately.
* **Evidence Match:** Because the rejection happens at the Account Throttling layer of API Gateway, the request is dropped before the integration request is sent to Lambda. This perfectly explains why "the Lambda function is never invoked."

> **Ref:** [Amazon API Gateway Quotas and Important Notes](https://docs.aws.amazon.com/apigateway/latest/developerguide/limits.html)

### 4. Incorrect Options Explanation

**A. The Lambda function reached its concurrency limit.**
* **Why Incorrect:** If Lambda reaches concurrency limits, the API Gateway *Integration* attempts to call Lambda, gets rejected, and then returns an error. While this often results in a 429 to the client, the prompt specifically states "Logs indicate that the Lambda function is never invoked." In an exam context, this phrasing is designed to point you toward the layer *before* Lambda (API Gateway).

**B. The Lambda function reached its Region limit for concurrency.**
* **Why Incorrect:** Similar to Option A, this refers to the backend computation layer. If the Region limit for Lambda (default 1,000 concurrent executions) was hit, the rejection would come from the Lambda service. The API Gateway would have attempted the invocation. The "never invoked" clue rules this out.

**D. The company reached its API Gateway default per-method limit for calls per second.**
* **Why Incorrect:** While API Gateway allows for method-level throttling, the "default" usually mirrors the higher-level limits unless explicitly constricted. In a SaaS scenario with "Unique API Keys" and "Tiered Pricing," throttling is primarily managed via **Usage Plans**, not ad-hoc per-method default settings.
* Furthermore, Option C (Account Limit) is the more scientifically accurate reason for a system-wide failure across "multiple premium customers" in a shared account. The aggregate volume is the root cause, hitting the hard ceiling of the account quota.
