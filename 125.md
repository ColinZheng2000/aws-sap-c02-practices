A company has a website that runs on Amazon EC2 instances behind an Application Load Balancer (ALB). The instances are in an Auto Scaling group. The ALB is associated with an AWS WAF web ACL.  
  
The website often encounters attacks in the application layer. The attacks produce sudden and significant increases in traffic on the application server. The access logs show that each attack originates from different IP addresses. A solutions architect needs to implement a solution to mitigate these attacks.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Create an Amazon CloudWatch alarm that monitors server access. Set a threshold based on access by IP address. Configure an alarm action that adds the IP address to the web ACL’s deny list.
- B. Deploy AWS Shield Advanced in addition to AWS WAF. Add the ALB as a protected resource.
- C. Create an Amazon CloudWatch alarm that monitors user IP addresses. Set a threshold based on access by IP address. Configure the alarm to invoke an AWS Lambda function to add a deny rule in the application server’s subnet route table for any IP addresses that activate the alarm.
- D. Inspect access logs to find a pattern of IP addresses that launched the attacks. Use an Amazon Route 53 geolocation routing policy to deny traffic from the countries that host those IP addresses.

# Analysis: Mitigating Application Layer Attacks with Least Operational Overhead

## 1. Understanding the Problem

The scenario presents a classic distributed denial-of-service (DDoS) challenge targeting the **Application Layer (Layer 7)**.

* **Architecture:** The workload is standard: EC2 instances in an Auto Scaling Group behind an Application Load Balancer (ALB). An AWS WAF Web ACL is already attached to the ALB.
* **The Threat:** Random, sudden traffic spikes originating from **rotating/different IP addresses**. This indicates a sophisticated attack where simple static IP blocking will fail because the source IPs constantly change (e.g., a botnet).
* **The Constraint:** The primary goal is to mitigate these attacks with the **LEAST operational overhead**.

**Key Takeaway:** We need a solution that automates the detection and mitigation of dynamic threats without requiring the team to manually analyze logs or write complex custom scripts for every new attack wave.

---

## 2. Services Explanation & Hands-on Context

### **AWS Shield (Standard vs. Advanced)**
* **Shield Standard:** Automatically included for all AWS customers at no extra cost. It protects against common infrastructure layer (Layer 3/4) attacks (e.g., SYN floods, UDP floods). It does *not* offer specialized protection for Layer 7 (Application) attacks.
* **Shield Advanced:** A paid service (requires a 1-year commitment and a monthly fee, typically ~$3,000 USD). It provides:
    * **Layer 7 Mitigation:** When paired with AWS WAF, it can automatically detect and mitigate application-layer attacks.
    * **SRT Support:** Access to the 24/7 AWS Shield Response Team.
    * **Cost Protection:** covers scaling costs resulting from DDoS spikes.
    * **Advanced Reporting:** Real-time visibility into attacks.

### **AWS WAF (Web Application Firewall)**
* **Function:** protects web applications from common web exploits (SQL injection, XSS) and bots.
* **Hands-on Note:** You attach WAF to CloudFront, ALBs, or API Gateways. While powerful, WAF rules usually need to be defined *before* an attack (static) or updated programmatically (dynamic). Doing this manually during a live attack is high operational overhead.

### **Amazon CloudWatch & Lambda**
* **Function:** CloudWatch monitors metrics; Lambda runs code.
* **Operational Reality:** While you *can* glue these together to build a custom "IPS" (Intrusion Prevention System), it requires significant maintenance. You must write the logic to differentiate legitimate traffic spikes from attacks, manage IP lists, and handle race conditions.

---

## 3. Correct Option Explanation

**Option: Deploy AWS Shield Advanced in addition to AWS WAF. Add the ALB as a protected resource.**

This is the correct answer because it addresses the "Least Operational Overhead" requirement through **managed automation**.

1.  **Automatic Application Layer DDoS Mitigation:** When you enable Shield Advanced on an ALB and associate a WAF Web ACL, you can enable a feature specifically designed for this scenario. Shield Advanced establishes a baseline of your application's normal traffic. When it detects a deviation (the attack), it automatically generates WAF rules to mitigate the attack (e.g., by creating rate-based rules or blocking specific signatures).
2.  **No Custom Scripting:** Unlike the CloudWatch/Lambda solutions, this does not require the Solutions Architect to build, maintain, or debug custom mitigation logic.
3.  **Handling Rotating IPs:** Shield Advanced utilizes AWS's global threat intelligence to identify bad actors much faster than a single user analyzing their local logs could.

> **Hands-on Tip:** In the AWS Console > WAF & Shield > Global Settings, ensuring the **"Automatic application layer DDoS mitigation"** is toggled on is the critical step here. This allows Shield to write rules to your Web ACL automatically.



---

## 4. Incorrect Options Explanation

**Option A: Create a CloudWatch alarm... monitor access... add IP to WAF deny list.**
* **Why it fails:** This is high operational overhead. CloudWatch does not natively trigger alarms based on *specific* IP addresses extracted from ALB logs without complex setup (like CloudWatch Logs Insights or Metric Filters). Furthermore, since the IPs are *different* for each attack (rotating IPs), blocking one IP is ineffective. The attack will just shift to a new IP, and your WAF IP set limit might fill up quickly.

**Option C: CloudWatch alarm ... Lambda ... add deny rule in subnet route table.**
* **Why it fails:**
    1.  **Wrong Tool:** VPC Route Tables are for routing traffic, not filtering it. You generally cannot "deny" an IP in a Route Table (you can only blackhole traffic, which is clumsy).
    2.  **NACL Limitations:** If the option meant Network ACLs (NACLs), they have a strict limit on the number of rules (default 20, max 40 recommended for performance). A botnet attack involves thousands of IPs; you cannot block them all in a NACL.
    3.  **Overhead:** Building and maintaining the Lambda logic to update network infrastructure involves high risk and overhead.

**Option D: Inspect access logs... Route 53 geolocation routing policy to deny traffic.**
* **Why it fails:**
    1.  **Manual Intervention:** "Inspect access logs" implies a human or a custom log analyzer must find the pattern. This is too slow for a sudden spike.
    2.  **Ineffective (Blunt Instrument):** Geo-blocking is a "sledgehammer" approach. If the attack comes from IP addresses in the United States, and your customers are also in the United States, blocking the country takes your business offline. Attacks often utilize botnets distributed across many "valid" countries.
