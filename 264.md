A company is migrating an application to the AWS Cloud. The application runs in an on-premises data center and writes thousands of images into a mounted NFS file system each night. After the company migrates the application, the company will host the application on an Amazon EC2 instance with a mounted Amazon Elastic File System (Amazon EFS) file system.

The company has established an AWS Direct Connect connection to AWS. Before the migration cutover, a solutions architect must build a process that will replicate the newly created on-premises images to the EFS file system.

What is the MOST operationally efficient way to replicate the images?

A. Configure a periodic process to run the aws s3 sync command from the on-premises file system to Amazon S3. Configure an AWS Lambda function to process event notifications from Amazon S3 and copy the images from Amazon S3 to the EFS file system.
B. Deploy an AWS Storage Gateway file gateway with an NFS mount point. Mount the file gateway file system on the on-premises server. Configure a process to periodically copy the images to the mount point.
C. Deploy an AWS DataSync agent to an on-premises server that has access to the NFS file system. Send data over the Direct Connect connection to an S3 bucket by using a public VIF. Configure an AWS Lambda function to process event notifications from Amazon S3 and copy the images from Amazon S3 to the EFS file system.
D. Deploy an AWS DataSync agent to an on-premises server that has access to the NFS file system. Send data over the Direct Connect connection to an AWS PrivateLink interface VPC endpoint for Amazon EFS by using a private VIF. Configure a DataSync scheduled task to send the images to the EFS file system every 24 hours.


# Replicating On-Premises NFS Images to Amazon EFS

## Understanding the Problem

### Current Environment

```text
On-Premises Application
        │
        └── Writes thousands of images nightly
                    │
                    └── NFS File System
```

### Target AWS Environment

```text
Amazon EC2
        │
        └── Amazon EFS mounted
```

### Requirement

Before migration cutover:

- newly created on-premises images must replicate to Amazon EFS
- use existing AWS Direct Connect
- solution must be MOST operationally efficient

| Option | Result | Key Reason |
|---|---|---|
| A | ❌ Incorrect | Too many custom components |
| B | ❌ Incorrect | Storage Gateway not ideal for replication workflow |

# Correct Option

# D. AWS DataSync → Amazon EFS

## Architecture Flow

```text
On-Prem NFS
      │
      └── DataSync Agent
                │
                └── Direct Connect (private VIF)
                          │
                          └── Amazon EFS
```

---

## Why This Is Correct

AWS DataSync is purpose-built for:

- NFS migrations
- EFS replication
- scheduled synchronization
- high-scale file transfer

### Native Integrations

DataSync directly supports:
- NFS
- SMB
- Amazon EFS
- Amazon S3
- FSx

No custom scripting required.

---

## Why This Is MOST Operationally Efficient

### Minimal Components

Only requires:
- DataSync agent
- scheduled task

No need for:
- custom scripts
- Lambda orchestration
- S3 staging buckets
- event processing pipelines

# Incorrect Options

# A. S3 Sync + Lambda + EFS

## Architecture

```text
On-Prem
   │
aws3 sync
   │
   └── S3
         │
         └── Lambda
                │
                └── EFS
```

## Why Incorrect

Too operationally complex.

Problems:
- custom sync process
- Lambda copy logic
- S3 event handling
- additional failure points

Also:
- indirect path to EFS
- unnecessary staging layer

### SAP-C02 Lesson

Avoid:
- custom glue architectures

Prefer:
- managed transfer services

---

# B. AWS Storage Gateway File Gateway

## Why Incorrect

Storage Gateway File Gateway primarily provides:
- cached access to S3

It is NOT optimized for:
- scheduled NFS-to-EFS replication workflows

### Major Problem

This does not directly solve:
- efficient synchronization to EFS

Operationally inferior compared to DataSync.

---

# C. DataSync → S3 → Lambda → EFS

## Why Incorrect

This option partially uses the correct service (DataSync), but still introduces:

- S3 staging
- Lambda processing
- extra data movement

### Inefficient Data Route

```text
On-Prem
   │
DataSync
   │
   └── S3
         │
         └── Lambda
                │
                └── EFS
```

Instead of direct:

```text
On-Prem → DataSync → EFS
```

---

## Additional Technical Issue

The option mentions:
- public VIF

This is less ideal for enterprise private transfers compared to:
- private VIF
