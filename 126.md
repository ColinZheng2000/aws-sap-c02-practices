A telecommunications company is running an application on AWS. The company has set up an AWS Direct Connect connection between the company's on-premises data center and AWS. The company deployed the application on Amazon EC2 instances in multiple Availability Zones behind an internal Application Load Balancer (ALB). The company's clients connect from the on-premises network by using HTTPS. The TLS terminates in the ALB. The company has multiple target groups and uses path-based routing to forward requests based on the URL path.  
  
The company is planning to deploy an on-premises firewall appliance with an allow list that is based on IP address. A solutions architect must develop a solution to allow traffic flow to AWS from the on-premises network so that the clients can continue to access the application.  
  
Which solution will meet these requirements?

- A. Configure the existing ALB to use static IP addresses. Assign IP addresses in multiple Availability Zones to the ALB. Add the ALB IP addresses to the firewall appliance.
- B. Create a Network Load Balancer (NLB). Associate the NLB with one static IP addresses in multiple Availability Zones. Create an ALB-type target group for the NLB and add the existing ALB. Add the NLB IP addresses to the firewall appliance. Update the clients to connect to the NLB.
- C. Create a Network Load Balancer (NLB). Associate the LNB with one static IP addresses in multiple Availability Zones. Add the existing target groups to the NLB. Update the clients to connect to the NLB. Delete the ALB Add the NLB IP addresses to the firewall appliance.
- D. Create a Gateway Load Balancer (GWLB). Assign static IP addresses to the GWLB in multiple Availability Zones. Create an ALB-type target group for the GWLB and add the existing ALB. Add the GWLB IP addresses to the firewall appliance. Update the clients to connect to the GWLB.

## 1. Understanding the Problem

**Scenario Overview:**
A telecommunications company operates a hybrid architecture using AWS Direct Connect. They have an application running on EC2 instances behind an **Application Load Balancer (ALB)**. The application relies on **path-based routing** (a Layer 7 feature) to direct traffic to different target groups based on the URL.

**The Conflict:**
The security team wants to install an on-premises firewall that allows traffic based strictly on an **allow list of IP addresses**. 

**The Core Challenge:**
* **ALB Limitation:** Application Load Balancers do not have fixed static IP addresses. Their IP addresses change dynamically as AWS scales the load balancer nodes up or down based on traffic. Consequently, you cannot configure a firewall allow list with ALB IP addresses reliably.
* **Requirement:** You need a solution that provides a **static IP** entry point while **preserving Layer 7 capabilities** (TLS termination and path-based routing).

---

## 2. Services Explanation & Hands-on Context

To solve this, you must understand the interaction between Layer 4 and Layer 7 load balancers.

### Application Load Balancer (ALB)
* **Layer:** Layer 7 (HTTP/HTTPS).
* **IP Behavior:** Dynamic. The DNS name resolves to multiple changing IPs.
* **Key Features:** Path-based routing (`/api` vs `/images`), Host-based routing, TLS termination.
* **Hands-on Note:** You cannot attach an Elastic IP (EIP) to an ALB.

### Network Load Balancer (NLB)
* **Layer:** Layer 4 (TCP/UDP/TLS).
* **IP Behavior:** Static. You can assign one Elastic IP (EIP) per Availability Zone (AZ).
* **Performance:** Capable of handling millions of requests per second with ultra-low latency.
* **Hands-on Note:** NLBs do not understand HTTP paths. They only forward TCP/UDP traffic.

### ALB-type Target Group for NLB
This is the "glue" feature. AWS allows you to register an Application Load Balancer as a target within a Network Load Balancer's target group.
* **Traffic Flow:** Client $\rightarrow$ NLB (Static IP) $\rightarrow$ ALB (L7 Routing) $\rightarrow$ EC2.
* **Benefits:** You get the static IP of the NLB for your firewall, and the advanced routing of the ALB for your application.

## 3. Correct Option Explanation

**Option B: Create a Network Load Balancer (NLB). Associate the NLB with one static IP address in multiple Availability Zones. Create an ALB-type target group for the NLB and add the existing ALB. Add the NLB IP addresses to the firewall appliance. Update the clients to connect to the NLB.**

This is the standard architectural pattern for exposing an ALB with static IP addresses.

* **Static IPs:** By creating an NLB, you can assign static Elastic IP addresses to it. This satisfies the on-premises firewall requirement.
* **Preserving Logic:** By setting the ALB as the target of the NLB, the SSL/TLS termination and path-based routing logic remain on the ALB. The NLB simply passes the TCP traffic through to the ALB.
* **Implementation:**
    1.  Create an NLB with an internet-facing or internal scheme (internal in this case, as it's via Direct Connect).
    2.  Allocate Elastic IPs (or private static IPs) for the NLB subnets.
    3.  Create a Target Group with target type **"Application Load Balancer"**.
    4.  Register the existing ALB ARN into this Target Group.
    5.  Configure the Listener on the NLB (e.g., TCP 443) to forward to this Target Group.

**Why this works for Medical/Industrial sectors:** These industries often have legacy hardware firewalls or strict regulatory compliance (HIPAA/GxP) requiring explicit IP whitelisting. This "NLB-fronting-ALB" pattern is the standard bridge between modern cloud elasticity and rigid legacy network requirements.

## 4. Incorrect Options Explanation

**A. Configure the existing ALB to use static IP addresses...**
* **Why it's wrong:** This is technically impossible. ALBs do not support static IP assignment. Their IP addresses are managed by AWS and change as the service scales. Attempting to whitelist current ALB IPs will result in a service outage when the IPs inevitably rotate.

**C. Create a Network Load Balancer (NLB)... Add the existing target groups to the NLB... Delete the ALB...**
* **Why it's wrong:** This option deletes the ALB. The question states the application uses **path-based routing** (e.g., routing `example.com/app1` to Target Group A and `example.com/app2` to Target Group B). 
* **Technical Gap:** The NLB operates at Layer 4 (Transport). It cannot inspect the URL path inside the HTTP packet. If you replace the ALB with an NLB, you lose the ability to perform path-based routing, breaking the application logic.

**D. Create a Gateway Load Balancer (GWLB)...**
* **Why it's wrong:** The Gateway Load Balancer is designed primarily for deploying and scaling third-party virtual appliances (like firewalls, intrusion detection systems, and deep packet inspection systems) using the GENEVE protocol.
* **Complexity:** While GWLB is powerful, it is not the correct tool for simply providing a static IP entry point for a web application. It introduces unnecessary complexity and encapsulation overhead that the client application is likely not configured to handle.
