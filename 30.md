A company is using AWS Organizations to manage multiple AWS accounts. For security purposes, the company requires the creation of an Amazon Simple Notification Service (Amazon SNS) topic that enables integration with a third-party alerting system in all the Organizations member accounts.  
A solutions architect used an AWS CloudFormation template to create the SNS topic and stack sets to automate the deployment of CloudFormation stacks. Trusted access has been enabled in Organizations.  
What should the solutions architect do to deploy the CloudFormation StackSets in all AWS accounts?

- A. Create a stack set in the Organizations member accounts. Use service-managed permissions. Set deployment options to deploy to an organization. Use CloudFormation StackSets drift detection.
- B. Create stacks in the Organizations member accounts. Use self-service permissions. Set deployment options to deploy to an organization. Enable the CloudFormation StackSets automatic deployment.
- C. Create a stack set in the Organizations management account. Use service-managed permissions. Set deployment options to deploy to the organization. Enable CloudFormation StackSets automatic deployment.
- D. Create stacks in the Organizations management account. Use service-managed permissions. Set deployment options to deploy to the organization. Enable CloudFormation StackSets drift detection.

# AWS Certified Solutions Architect -Professional (SAP-C02) Exam Question Analysis

## Understanding the Problem

The requirement is to deploy a specific resource, an **Amazon SNS topic** integrated with a third-party system, into **all member accounts** within an **AWS Organization**. The solution must leverage **AWS CloudFormation** for defining the resource and **CloudFormation StackSets** for automated, large-scale deployment. **Trusted access** has already been enabled between CloudFormation StackSets and AWS Organizations. The task is to identify the correct configuration for the CloudFormation StackSet deployment.

Key constraints and goals:
1.  **Target:** All member accounts in the AWS Organization.
2.  **Tool:** AWS CloudFormation StackSets.
3.  **Deployment Model:** Automated and centralized.
4.  **Prerequisite:** Trusted access is enabled.

### 1. Stack: The Unit of Deployment

In the context of the question, a **Stack** is the actual result of running the CloudFormation template in a specific member account.

- **What it is:** A stack is the fundamental _unit_ of deployment for CloudFormation.
    
- **Location:** It resides within a **single AWS account** in a single AWS Region.
    
- **Relationship:** Every member account targeted by the solution will have its own independent **stack** that contains the Amazon SNS topic resource.
    

### 2. StackSet: The Multi-Account Orchestrator

A **StackSet** is the control plane used to manage the deployment of the exact same stack definition across the entire organization.

- **What it is:** A StackSet is a **management resource** that defines the source template and a list of target accounts and Regions for deployment.
    
- **Location:** It is created and managed **centrally** in the AWS Organizations management account (or a delegated administrator account), as identified by the correct option (C).
    
- **Operation:** When you create or update a StackSet, it automatically initiates the creation, update, or deletion of individual **Stack Instances** (which are the stacks themselves) in all the target member accounts and Regions. The StackSet abstracts the tedious, repetitive work of logging into each account to launch a stack.
***

## Correct Option(s) Explanation

### **C. Create a stack set in the Organizations management account. Use service-managed permissions. Set deployment options to deploy to the organization. Enable CloudFormation StackSets automatic deployment.**

This option describes the optimal and prescribed method for central resource deployment across an entire AWS Organization using StackSets, especially when trusted access is enabled.

* **Create a stack set in the Organizations management account:** StackSets are fundamentally managed from a central administrator/management account to deploy resources to member accounts.
* **Use service-managed permissions:** This is the most efficient permission model when deploying to an Organization. Since **trusted access has been enabled** (a prerequisite), StackSets can assume a service-linked role in the management account and automatically provision the necessary execution roles in the member accounts, simplifying the process and eliminating manual IAM setup.
* **Set deployment options to deploy to the organization:** This is the correct target scope. Instead of selecting individual accounts, you specify the entire organization or specific Organizational Units (OUs) as the target.
* **Enable CloudFormation StackSets automatic deployment (Auto-Deployment):** This crucial feature ensures that the StackSet automatically deploys the SNS topic to *new accounts* that are subsequently added to the organization (or the targeted OUs). This satisfies the need for continuous coverage ("in all the Organizations member accounts") without manual intervention for new accounts. (Refer to **AWS Documentation: Enable automatic deployment**).

***

## Incorrect Options Explanation

### **A. Create a stack set in the Organizations member accounts. Use service-managed permissions. Set deployment options to deploy to an organization. Use CloudFormation StackSets drift detection.**

* **Incorrect Location:** StackSets are created and managed centrally from the **management account**, not the member accounts.
* **Incorrect Feature:** While drift detection is a useful feature, it focuses on verifying configuration consistency *after* deployment. It's not a deployment mechanism and doesn't address the requirement of deploying to *new* accounts automatically, which is the role of auto-deployment.

### **B. Create stacks in the Organizations member accounts. Use self-service permissions. Set deployment options to deploy to an organization. Enable the CloudFormation StackSets automatic deployment.**

* **Incorrect Concept:** You create **StackSets** centrally, not individual **stacks** in each member account. StackSets is the orchestration layer that creates the individual stacks.
* **Incorrect Permission Model:** **Self-service permissions** require manual setup of the execution role and trust relationships in *every* member account, which is cumbersome and defeats the purpose of centralized management with Organizations. **Service-managed permissions** are the correct choice when deploying to an Organization with trusted access enabled.

### **D. Create stacks in the Organizations management account. Use service-managed permissions. Set deployment options to deploy to the organization. Enable CloudFormation StackSets drift detection.**

* **Incorrect Concept:** As with B, the central object is a **StackSet**, not individual stacks. You only have one single StackSet in the management account, which provisions multiple **Stack Instances** (stacks) in the member accounts.
* **Incorrect Feature:** Drift detection is an operational feature, not the mechanism for automatic deployment to new accounts (which is **auto-deployment**). Deploying to a stack in the management account only creates the resource in the management account, not the member accounts.
