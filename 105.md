A company is creating a sequel for a popular online game. A large number of users from all over the world will play the game within the first week after launch.  
Currently, the game consists of the following components deployed in a single AWS Region:  
✑ Amazon S3 bucket that stores game assets  
✑ Amazon DynamoDB table that stores player scores  
A solutions architect needs to design a multi-Region solution that will reduce latency, improve reliability, and require the least effort to implement.  
What should the solutions architect do to meet these requirements?  

- A. Create an Amazon CloudFront distribution to serve assets from the S3 bucket. Configure S3 Cross-Region Replication. Create a new DynamoDB table in a new Region. Use the new table as a replica target for DynamoDB global tables.
- B. Create an Amazon CloudFront distribution to serve assets from the S3 bucket. Configure S3 Same-Region Replication. Create a new DynamoDB table in a new Region. Configure asynchronous replication between the DynamoDB tables by using AWS Database Migration Service (AWS DMS) with change data capture (CDC).
- C. Create another S3 bucket in a new Region, and configure S3 Cross-Region Replication between the buckets. Create an Amazon CloudFront distribution and configure origin failover with two origins accessing the S3 buckets in each Region. Configure DynamoDB global tables by enabling Amazon DynamoDB Streams, and add a replica table in a new Region.
- D. Create another S3 bucket in the sine Region, and configure S3 Same-Region Replication between the buckets. Create an Amazon CloudFront distribution and configure origin failover with two origins accessing the S3 buckets. Create a new DynamoDB table in a new Region. Use the new table as a replica target for DynamoDB global tables.

# Analysis of Multi-Region Game Architecture Question
## Understanding the Problem
The core requirements and current state are:
- Current setup: Single AWS Region with Amazon S3 (game assets) and Amazon DynamoDB (player scores) for an online game sequel.
- Key requirements:
  1. Multi-Region deployment (support global users).
  2. Reduce latency (critical for game performance).
  3. Improve reliability (resilience against Region outages).
  4. Least implementation effort (avoid complex custom solutions).

## Services Explanation
### Key Services Involved
1. **Amazon CloudFront**: Global CDN that caches content at 400+ edge locations to reduce latency for global users. Supports **origin failover** (automatically switches to a secondary origin if the primary fails) for high reliability. [CloudFront Origin Failover Docs](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/high_availability_origin_failover.html)
2. **S3 Replication**:
   - **Cross-Region Replication (CRR)**: Replicates S3 objects between buckets in different AWS Regions (enables multi-Region redundancy for assets). [S3 CRR Docs](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html#crr)
   - **Same-Region Replication (SRR)**: Replicates objects within the same Region (used for compliance/backup, not multi-Region requirements). [S3 SRR Docs](https://docs.aws.amazon.com/AmazonS3/latest/userguide/replication.html#srr)
3. **Amazon DynamoDB Global Tables**: Managed multi-Region replication for DynamoDB. Uses **DynamoDB Streams** to capture table changes and replicate asynchronously across replica tables in different Regions. Provides low-latency global access and automatic failover—no custom replication logic needed. [DynamoDB Global Tables Docs](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/GlobalTables.html)
4. **AWS Database Migration Service (DMS)**: Used for migrating or replicating data between databases. Requires manual setup of replication instances, tasks, and change data capture (CDC)—more complex than DynamoDB Global Tables for DynamoDB-to-DynamoDB replication. [AWS DMS Docs](https://docs.aws.amazon.com/dms/latest/userguide/CHAP_Introduction.html)
5. **DynamoDB Streams**: Captures sequential changes to a DynamoDB table (creates, updates, deletes). Required for DynamoDB Global Tables to replicate data across Regions. [DynamoDB Streams Docs](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Streams.html)

## Correct Option Explanation
### Option C
This option meets all requirements with minimal effort, aligning with AWS best practices:
1. **Multi-Region Asset Delivery**: 
   - Creating a second S3 bucket in a new Region + S3 CRR ensures game assets are replicated across Regions (multi-Region redundancy).
   - CloudFront with **origin failover** (two S3 origins in separate Regions) automatically switches to the secondary bucket if the primary Region’s S3 is unavailable—maximizing reliability.
2. **Low Latency**: CloudFront caches assets at edge locations globally, reducing latency for users worldwide (critical for game performance).
3. **Reliable Score Storage**: 
   - Enabling DynamoDB Streams + adding a replica table in a new Region configures DynamoDB Global Tables. This provides managed multi-Region replication, automatic failover, and low-latency access to player scores—no custom code or complex setup.
4. **Least Effort**: All components (S3 CRR, CloudFront origin failover, DynamoDB Global Tables) are managed AWS services. No manual replication logic (e.g., DMS) or infrastructure management is required.

## Incorrect Options Explanation
### Option A
- **Reliability Gap**: CloudFront is configured to serve assets from a single S3 bucket (no origin failover). If the primary Region’s S3 bucket is unavailable, CloudFront cannot fail over to the replicated bucket—violating the "improve reliability" requirement.
- **Incomplete DynamoDB Setup**: DynamoDB Global Tables require DynamoDB Streams to replicate data, but this option does not mention enabling Streams (a mandatory step for Global Tables). The setup would fail to replicate player scores reliably.

### Option B
- **Invalid S3 Replication**: S3 Same-Region Replication (SRR) only replicates assets within the original Region—failing the "multi-Region" requirement. Global users would still experience latency from a single Region’s S3 bucket.
- **Overly Complex DynamoDB Replication**: AWS DMS with CDC requires manual setup (replication instances, tasks, CDC configuration) and ongoing maintenance—contradicts the "least effort" requirement. DynamoDB Global Tables are purpose-built for this use case and require far less effort.

### Option D
- **Invalid S3 Replication**: S3 SRR does not support multi-Region deployment. The second S3 bucket is in the **same Region** as the original, so assets are not replicated across Regions—failing the multi-Region and low-latency requirements for global users.
- **Reliability Gap**: CloudFront origin failover uses two buckets in the same Region, so a Region-wide outage would take both buckets offline—no true multi-Region resilience.
