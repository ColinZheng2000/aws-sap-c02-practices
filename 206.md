A company has migrated a legacy application to the AWS Cloud. The application runs on three Amazon EC2 instances that are spread across three Availability Zones. One EC2 instance is in each Availability Zone. The EC2 instances are running in three private subnets of the VPC and are set up as targets for an Application Load Balancer (ALB) that is associated with three public subnets.

The application needs to communicate with on-premises systems. Only traffic from IP addresses in the company's IP address range are allowed to access the on-premises systems. The company's security team is bringing only one IP address from its internal IP address range to the cloud. The company has added this IP address to the allow list for the company firewall. The company also has created an Elastic IP address for this IP address.

A solutions architect needs to create a solution that gives the application the ability to communicate with the on-premises systems. The solution also must be able to mitigate failures automatically.

Which solution will meet these requirements?

A. Deploy three NAT gateways, one in each public subnet. Assign the Elastic IP address to the NAT gateways. Turn on health checks for the NAT gateways. If a NAT gateway fails a health check, recreate the NAT gateway and assign the Elastic IP address to the new NAT gateway.
B. Replace the ALB with a Network Load Balancer (NLB). Assign the Elastic IP address to the NLB Turn on health checks for the NLIn the case of a failed health check, redeploy the NLB in different subnets.
C. Deploy a single NAT gateway in a public subnet. Assign the Elastic IP address to the NAT gateway. Use Amazon CloudWatch with a custom metric to monitor the NAT gateway. If the NAT gateway is unhealthy, invoke an AWS Lambda function to create a new NAT gateway in a different subnet. Assign the Elastic IP address to the new NAT gateway.
D. Assign the Elastic IP address to the ALB. Create an Amazon Route 53 simple record with the Elastic IP address as the value. Create a Route 53 health check. In the case of a failed health check, recreate the ALB in different subnets.

## Analysis

### Understanding the Problem

A company has migrated a legacy application to AWS running on three EC2 instances in private subnets across three Availability Zones (one per AZ), fronted by an Application Load Balancer (ALB) in public subnets. The application needs **outbound connectivity** to on-premises systems. The critical constraint is that the on-premises firewall only allows traffic from **a single IP address** — the company has provisioned exactly one Elastic IP (EIP) for this purpose. The solution must also **automatically mitigate failures** (high availability for the outbound path). The challenge is: private EC2 instances need a NAT mechanism with a static, single public IP that can fail over to another Availability Zone if the NAT component fails.

### Correct Option: C — Single NAT Gateway + CloudWatch + Lambda Failover

**Why C is correct:**

- **Single NAT Gateway in a public subnet**: A NAT Gateway provides outbound internet connectivity for EC2 instances in private subnets. It has a fixed Elastic IP address, satisfying the on-premises firewall allow-list requirement (a single, static source IP).
- **Elastic IP on the NAT Gateway**: The EIP is the public source IP for all outbound traffic from the private EC2 instances to on-premises. This matches exactly what the on-premises firewall expects.
- **CloudWatch monitoring**: NAT Gateway publishes metrics (e.g., `ErrorPortAllocation`, `PacketsDropCount`, `BytesOutToDestination`) to CloudWatch. A CloudWatch alarm can detect when the NAT Gateway becomes unhealthy or unreachable.
- **Lambda-based automated failover**: When the alarm triggers, an AWS Lambda function creates a new NAT Gateway in a **different public subnet** (in a different AZ) and **reassigns the Elastic IP** from the failed NAT Gateway to the new one. This preserves the single IP address that the on-premises firewall allows while achieving cross-AZ high availability.
- **Route table update**: The Lambda function can also update the private subnet route tables to point `0.0.0.0/0` to the new NAT Gateway, ensuring traffic flows through the healthy NAT instance.

**Key architecture points**:
- A NAT Gateway is an AZ-resilient service (within its AZ). If the AZ fails, the NAT Gateway becomes unavailable. Lambda-based failover to another AZ addresses this.
- The EIP is the critical resource that must be preserved (since the on-premises firewall only allows one IP). By reassigning the EIP to the new NAT Gateway, the source IP remains unchanged.

**Official reference**: [NAT Gateway Basics](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-nat-gateway.html) — "Each NAT gateway is created in a specific Availability Zone and implemented with redundancy in that zone." For multi-AZ resilience, you need a NAT Gateway in each AZ or a custom failover mechanism.

### Incorrect Options

**A — Three NAT gateways (one per AZ) sharing a single EIP**

- **Technically impossible**: An Elastic IP address can be associated with only **one** NAT Gateway (or one EC2 instance) at a time. You cannot assign the same EIP to three different NAT Gateways simultaneously. AWS enforces a 1:1 mapping between an EIP and a network interface/resource.
- Even if you could, health-check-based recreation of a NAT Gateway introduces downtime during the detection and provisioning window (NAT Gateway creation takes 1-2 minutes).
- Three NAT Gateways × 24/7 = significantly higher cost ($0.045/hour per NAT Gateway × 3 = ~$97/month just in NAT Gateway hourly charges, plus data processing fees).

**B — Replace ALB with NLB, assign EIP to NLB**

- **Fundamental misunderstanding of traffic direction**: The NLB is an **inbound** load balancer — it distributes incoming client traffic to targets. The requirement is for **outbound** traffic from EC2 instances to on-premises systems. An NLB does not provide outbound NAT capabilities for backend instances.
- While an NLB can have static IP addresses (EIPs), those are the **destination** IPs that on-premises systems would connect to (if on-premises were initiating connections). But the application EC2 instances need to initiate outbound connections to on-premises — that's a NAT use case, not a load balancing use case.
- Redeploying the NLB to different subnets on health check failure doesn't solve the outbound NAT problem.

**D — Assign EIP to ALB, Route 53 simple record with EIP**

- **ALB does not support Elastic IP addresses**: Application Load Balancers use dynamic IP addresses that change over time. Only Network Load Balancers (NLB) support static IP/EIP assignment. You cannot assign an EIP to an ALB — this is a hard technical limitation.
- **Same inbound vs. outbound confusion as Option B**: The ALB handles incoming web traffic to the application. It does not provide outbound NAT for EC2 instances to reach on-premises systems.
- Route 53 health checks and DNS failover are designed for directing client traffic to healthy endpoints — they do not address the outbound connectivity requirement.

### Hands-On Enterprise Knowledge

1. **NAT Gateway high availability pattern**: A single NAT Gateway per AZ is the standard HA approach (one NAT GW in each AZ, each with its own EIP, and each AZ's private route table points to its local NAT GW). However, this only works when the destination (on-premises firewall) can accept traffic from multiple source IPs. When you're restricted to a **single IP** (as in this question), you must implement custom failover with EIP reassignment. This is a well-known enterprise pattern documented in AWS blogs.

2. **Lambda-based failover implementation details**: The Lambda function for NAT Gateway failover typically:
   - Detaches the EIP from the failed NAT Gateway
   - Creates a new NAT Gateway in a pre-selected backup public subnet (different AZ)
   - Associates the EIP with the new NAT Gateway
   - Updates the private subnet route tables (`0.0.0.0/0` → new NAT Gateway ID)
   - Sends an SNS notification to the operations team
   
   This can be orchestrated via an **AWS Systems Manager Automation runbook** or a Step Functions state machine for more complex workflows.

3. **NAT Gateway vs. NAT Instance**: In enterprise environments, NAT Gateways are preferred over NAT Instances (EC2-based NAT) because:
   - NAT Gateways are fully managed (no patching, no maintenance)
   - NAT Gateways provide higher bandwidth (up to 100 Gbps)
   - NAT Gateways are highly available within an AZ (redundant hardware)
   - However, for the single-EIP cross-AZ failover scenario, a NAT Instance (EC2) offers more flexibility (you can script EIP reassignment more easily with an EC2 instance). The SAP-C02 exam may test this tradeoff.

4. **Why not Transit Gateway + VPN?**: An alternative enterprise pattern is using **AWS Transit Gateway** with a **VPN connection** to on-premises, and routing all outbound VPC traffic through a centralized egress VPC. The Transit Gateway VPN attachment uses a pair of tunnels, each with its own public IP. For a single-IP constraint, you could use **AWS Direct Connect** (which can provide a single public IP for the on-premises side of the connection), but the question doesn't mention Direct Connect and it may not be cost-appropriate.

5. **Cost consideration**: A single NAT Gateway costs $0.045/hour (~$32.40/month) plus $0.045 per GB processed. Three NAT Gateways (Option A) would triple the hourly cost. The Lambda-based failover minimizes steady-state cost while achieving the required resilience.
