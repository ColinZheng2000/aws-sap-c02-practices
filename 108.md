A company runs a content management application on a single Windows Amazon EC2 instance in a development environment. The application reads and writes static content to a 2 TB Amazon Elastic Block Store (Amazon EBS) volume that is attached to the instance as the root device. The company plans to deploy this application in production as a highly available and fault-tolerant solution that runs on at least three EC2 instances across multiple Availability Zones.  
  
A solutions architect must design a solution that joins all the instances that run the application to an Active Directory domain. The solution also must implement Windows ACLs to control access to file contents. The application always must maintain exactly the same content on all running instances at any given point in time.  
  
Which solution will meet these requirements with the LEAST management overhead?

- A. Create an Amazon Elastic File System (Amazon EFS) file share. Create an Auto Scaling group that extends across three Availability Zones and maintains a minimum size of three instances. Implement a user data script to install the application, join the instance to the AD domain, and mount the EFS file share.
- B. Create a new AMI from the current EC2 Instance that is running. Create an Amazon FSx for Lustre file system. Create an Auto Scaling group that extends across three Availability Zones and maintains a minimum size of three instances. Implement a user data script to join the instance to the AD domain and mount the FSx for Lustre file system.
- C. Create an Amazon FSx for Windows File Server file system. Create an Auto Scaling group that extends across three Availability Zones and maintains a minimum size of three instances. Implement a user data script to install the application and mount the FSx for Windows File Server file system. Perform a seamless domain join to join the instance to the AD domain.
- D. Create a new AMI from the current EC2 instance that is running. Create an Amazon Elastic File System (Amazon EFS) file system. Create an Auto Scaling group that extends across three Availability Zones and maintains a minimum size of three Instances. Perform a seamless domain join to join the instance to the AD domain.

# Analysis: Windows HA Shared File System with AD Integration
tags: #AWS-SAP-C02 #EC2 #FSx #AutoScaling #ActiveDirectory #Windows #HighAvailability #ACLs

## 1. Understanding the Problem
The core requirements are:
- **High Availability (HA)**: 3+ Windows EC2 instances across multiple AZs.
- **Shared Content**: All instances access identical static content (shared file system).
- **Windows Compatibility**: Support for Windows OS, Active Directory (AD) domain join, and Windows ACLs (file access control).
- **Least Management Overhead**: Minimize manual configuration/scripting; use AWS-managed services.

## 2. Services Explanation
### Key Services Involved
| Service | Purpose & Critical Features |
|---------|------------------------------|
| **Amazon FSx for Windows File Server** | Managed Windows file system (SMB protocol, native to Windows). Supports Windows ACLs, seamless AD integration, and AWS-managed maintenance (patching/backups). Purpose-built for Windows workloads. |
| **Amazon EFS** | Elastic file system for Linux (NFS protocol). Not natively compatible with Windows (requires third-party mount helpers) and does NOT support Windows ACLs (only NFS-based ACLs). Adds management overhead for Windows. |
| **Amazon FSx for Lustre** | High-performance file system for Linux HPC workloads. Incompatible with Windows (Lustre is Linux-only) and lacks Windows ACL/AD support. |
| **Auto Scaling Group (ASG)** | Multi-AZ deployment to maintain 3+ instances. Automates instance provisioning/replacement, reducing manual management. |
| **Seamless Domain Join** | AWS feature to automatically join Windows EC2 instances to AD without complex scripts. Lowers overhead vs. user-data-driven AD join. |
| **AMI (Amazon Machine Image)** | Captures pre-configured instances (e.g., with application installed). Reduces app installation overhead but is irrelevant if the file system/AD requirements are unmet. |

### Official AWS Documentation References
- FSx for Windows: "Fully managed Windows file system with SMB support, Windows ACLs, and AD integration" (AWS FSx for Windows User Guide).
- EFS Windows Compatibility: "EFS uses NFS; Windows requires the EFS Windows Mount Helper (extra overhead) and does not support Windows ACLs" (Amazon EFS User Guide).
- Seamless Domain Join: "Automatically joins EC2 instances to AD without scripts" (AWS Directory Service User Guide).
- FSx for Lustre: "Optimized for Linux HPC workloads; not compatible with Windows" (Amazon FSx for Lustre User Guide).

## 3. Correct Option Explanation (Option C)
This option meets all requirements with minimal management overhead:
- **FSx for Windows File Server**: Natively supports Windows SMB, Windows ACLs, and AD integration—addresses the core Windows/access control needs. AWS manages the file system (no manual patching/backups).
- **Multi-AZ Auto Scaling Group**: Ensures 3+ instances across AZs for HA/fault tolerance; AWS automates scaling/replacement.
- **Seamless Domain Join**: Eliminates complex user-data scripts for AD join, reducing management overhead.
- **User Data for App/Mount**: Mounting FSx for Windows is simple (native SMB), and the app installation script is low-effort. The solution avoids unmanaged workarounds (e.g., EFS mount helpers) and leverages AWS-managed services.

## 4. Incorrect Options Explanation
### Option A (EFS + User Data AD Join)
- **Critical Flaw**: EFS is NFS-based—Windows instances require the EFS Windows Mount Helper (extra overhead) and EFS does NOT support Windows ACLs (fails access control requirement).
- **Secondary Flaw**: AD join via user data is more complex than seamless domain join, increasing management overhead.

### Option B (FSx for Lustre + AMI)
- **Critical Flaw**: FSx for Lustre is Linux-only—Windows instances cannot natively mount it, and it lacks Windows ACL/AD support.
- **Irrelevant Strength**: AMI pre-installs the app, but the file system incompatibility makes this option invalid.

### Option D (EFS + Seamless Domain Join)
- **Critical Flaw**: EFS still requires Windows mount helpers (overhead) and does not support Windows ACLs—fails core Windows/access control requirements.
- **Irrelevant Strength**: Seamless domain join and AMI simplify deployment, but the EFS incompatibility renders the solution non-compliant.
