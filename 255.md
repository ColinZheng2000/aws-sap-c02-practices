A company is migrating mobile banking applications to run on Amazon EC2 instances in a VPC. Backend service applications run in an on-premises data center.
The data center has an AWS Direct Connect connection into AWS. The applications that run in the VPC need to resolve DNS requests to an on-premises Active
Directory domain that runs in the data center.
Which solution will meet these requirements with the LEAST administrative overhead?

A. Provision a set of EC2 instances across two Availability Zones in the VPC as caching DNS servers to resolve DNS queries from the application servers within the VPC.
B. Provision an Amazon Route 53 private hosted zone. Configure NS records that point to on-premises DNS servers.
C. Create DNS endpoints by using Amazon Route 53 Resolver Add conditional forwarding rules to resolve DNS namespaces between the on-premises data center and the VPC.
D. Provision a new Active Directory domain controller in the VPC with a bidirectional trust between this new domain and the on-premises Active Directory domain.


Suggest Answer: C

---

## 📋 Analysis

### Understanding the Problem

A company is migrating mobile banking applications to EC2 in a VPC, while backend services remain on-premises connected via AWS Direct Connect. The core challenge is **hybrid DNS resolution**: applications running in the VPC must be able to resolve DNS names in the on-premises Active Directory domain (e.g., `backend.internal.corp`). The solution must minimize **administrative overhead** — meaning no self-managed DNS servers, no complex Active Directory deployments, and no ongoing maintenance burden.

### ✅ Correct Option: C

**Amazon Route 53 Resolver with conditional forwarding rules**

This is the correct answer because it provides a fully managed, AWS-native hybrid DNS solution:

1. **Route 53 Resolver Endpoints**: Two types of endpoints are created:
   - **Outbound Endpoint**: Forwards DNS queries from the VPC to on-premises DNS servers. Deployed across multiple AZs for high availability, it provides elastic network interfaces (ENIs) that serve as the source IPs for forwarded queries.
   - **Inbound Endpoint** (optional but complementary): Allows on-premises DNS servers to resolve records in the Route 53 private hosted zone — enabling bidirectional name resolution.

2. **Conditional Forwarding Rules**: You define rules that match specific domain names (e.g., `corp.internal`) and specify the IP addresses of the on-premises DNS servers as targets. When a query matches a rule, it is forwarded across the Direct Connect link to on-premises DNS servers. Queries that don't match any rule are resolved normally using the VPC's default DNS (Route 53 Resolver, also known as the `.2` resolver at `VPC-base + 2`).

3. **Least Administrative Overhead**: This is a fully managed service — there are no EC2 instances to patch, no DNS software to configure (no BIND, no Windows DNS Server), no high availability to engineer manually. Route 53 Resolver automatically scales and is natively highly available across the AZs you select.

4. **Direct Connect Integration**: Since Direct Connect already provides private network connectivity between the VPC and on-premises, DNS queries flow over this low-latency, consistent path.

### ❌ Incorrect Options

**Option A — Self-Managed EC2 DNS Caching Servers**

- This requires deploying and managing EC2 instances running DNS software (e.g., BIND, dnsmasq, or Windows DNS Server) across two AZs. Administrative overhead includes:
  - OS patching and hardening
  - DNS software configuration and updates
  - Auto Scaling and failover logic
  - Monitoring DNS query performance and health
  - Managing security groups for DNS (port 53)
- This is the **highest** overhead option — exactly what the question asks you to avoid.

**Option B — Route 53 Private Hosted Zone with NS Records**

- A private hosted zone is for **authoritative DNS hosting** (i.e., "I own this domain and define its records"). Creating NS records that point to on-premises DNS servers is a misunderstanding of how DNS delegation works.
- NS records delegate authority for a subdomain to a set of name servers. But VPC DNS (Route 53 Resolver) does not automatically forward queries based on NS delegation records — NS records are responses to queries, not forwarding instructions. The VPC resolver would not know to forward queries to the on-premises NS servers.
- This approach simply would not work for enabling VPC resources to resolve on-premises domain names.

**Option D — New AD Domain Controller in VPC with Bidirectional Trust**

- Deploying and managing Active Directory domain controllers in AWS is a heavyweight operation:
  - Domain controllers must be patched, monitored, and backed up
  - Bidirectional trust configuration requires careful coordination between on-premises and cloud AD teams
  - Network connectivity must be reliable for trust operations and replication
  - This introduces authentication dependencies and Kerberos complexities when the requirement is **only** DNS resolution
- While this would enable DNS resolution (AD domain controllers run DNS Server), it is vastly more complex than necessary. This is appropriate for full hybrid identity scenarios, not for simple DNS forwarding.

### 🔧 Hands-On Knowledge

- **Route 53 Resolver Architecture**: The VPC's default DNS resolver is at `VPC.2` (e.g., `10.0.0.2`). Route 53 Resolver endpoints are deployed as managed ENIs in your subnets. An outbound endpoint needs at least two IP addresses in different AZs for high availability.
- **Conditional Forwarding Rule Structure**: Each rule has:
  - A **domain name** to match (e.g., `corp.internal` or `ad.company.com`)
  - **Target IP addresses** (the on-premises DNS servers, reachable via Direct Connect)
  - An optional **rule type** (`Forward` or `System` — Forward sends to specified targets)
  - Rules can be shared across VPCs using AWS Resource Access Manager (RAM)
- **Direct Connect Considerations**: Ensure the on-premises DNS server IPs are reachable via the Direct Connect virtual interface. The VPC route tables must have appropriate routes (usually via a Virtual Private Gateway attached to the Direct Connect).
- **Monitoring**: Use Route 53 Resolver query logging to send DNS query logs to CloudWatch Logs, S3, or Kinesis Data Firehose. This helps troubleshoot resolution failures and provides security auditing of DNS activity.
