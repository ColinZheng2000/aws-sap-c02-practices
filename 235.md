A company has a complex web application that leverages Amazon CloudFront for global scalability and performance. Over time, users report that the web application is slowing down.
The company's operations team reports that the CloudFront cache hit ratio has been dropping steadily. The cache metrics report indicates that query strings on some URLs are inconsistently ordered and are specified sometimes in mixed-case letters and sometimes in lowercase letters.
Which set of actions should the solutions architect take to increase the cache hit ratio as quickly possible?

A. Deploy a Lambda@Edge function to sort parameters by name and force them to be lowercase. Select the CloudFront viewer request trigger to invoke the function.
B. Update the CloudFront distribution to disable caching based on query string parameters.
C. Deploy a reverse proxy after the load balancer to post process the emitted URLs in the application to force the URL strings to be lowercase.
D. Update the CloudFront distribution to specify case-insensitive query string processing.


Suggest Answer: A

---

## Analysis

### Understanding the Problem

A complex web application uses Amazon CloudFront for global scalability and performance. Over time, the **cache hit ratio** has been steadily dropping — meaning fewer requests are being served from CloudFront edge caches and more are being forwarded to the origin, increasing latency and load.

The operations team's cache metrics analysis reveals two specific issues with query strings in request URLs:
1. **Inconsistent ordering** of query parameters — e.g., `?name=foo&type=bar` vs. `?type=bar&name=foo` are treated as **different cache keys** by CloudFront, even though they're functionally identical.
2. **Mixed case** — e.g., `?Name=Foo` vs. `?name=foo` are also treated as different cache keys because query string matching is case-sensitive by default.

Together, these issues cause **cache fragmentation**: many cache entries for what is effectively the same content, dramatically reducing cache hit ratio. The solution must increase the cache hit ratio **as quickly as possible** — implying minimal latency impact on the request path.

### Correct Option(s) Explanation

**Option A (Correct):** Deploy a **Lambda@Edge** function that **sorts query parameters alphabetically by name** and **forces them to lowercase**. Associate this function with the CloudFront **viewer request** trigger.

**Why this works:**

- **Lambda@Edge** runs at CloudFront edge locations, allowing you to modify incoming requests before they hit the CloudFront cache. The **viewer request** trigger is the earliest point in the request lifecycle — it fires for every incoming request before cache lookup.
- The function performs **query string normalization**:
  - **Sorting:** Parameters are alphabetically sorted by key name, so `?b=2&a=1` becomes `?a=1&b=2`
  - **Lowercasing:** All parameter names and values are converted to lowercase, so `?Name=Foo` becomes `?name=foo`
- By normalizing the query string at the edge, **all functionally equivalent requests produce the same cache key**, maximizing cache hits.
- **Speed:** Lambda@Edge at viewer request is the fastest approach — normalization happens before CloudFront checks its cache, so the very next normalized request results in a cache hit. No backend changes are needed, and no infrastructure needs to be deployed.
- This solution works **immediately** upon deployment — as soon as the Lambda@Edge function propagates to all edge locations (typically under 5 minutes).

- **Reference:** [Lambda@Edge Viewer Request Trigger](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/lambda-at-the-edge.html) | [CloudFront Cache Key and Origin Requests](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/CacheKeyAndOriginRequests.html)

### Incorrect Options Explanation

**Option B:** Update the CloudFront distribution to **disable caching based on query string parameters**.

- ❌ This means CloudFront would forward **all requests** to the origin (or cache without considering query strings at all). If query strings are critical to serving different content (e.g., `?page=2` vs `?page=3`), disabling query-string-based caching would serve **incorrect cached content** to users. This is not a viable solution for a "complex web application" that presumably uses query parameters to differentiate content. It also does not increase cache hit ratio if query strings are simply ignored — cached content may be wrong.

**Option C:** Deploy a **reverse proxy** after the load balancer to post-process emitted URLs and force them to lowercase.

- ❌ Multiple critical issues:
  1. **Wrong placement:** A reverse proxy deployed *after* the load balancer operates at the **origin**, not at the edge. This means CloudFront still sees unnormalized URLs from clients and continues to have poor cache hit ratio.
  2. **Only addresses case, not ordering:** The description mentions only lowercasing, not sorting query parameters. The inconsistent ordering problem remains unresolved.
  3. **Slower to deploy:** Setting up a reverse proxy (e.g., NGINX, HAProxy) on EC2 instances or in containers behind the load balancer requires infrastructure provisioning, configuration, and testing — much slower than deploying a Lambda@Edge function.
  4. **Does not help caching:** Even if the origin serves normalized URLs, CloudFront's cache key is based on the **incoming viewer request URL**, not the origin's response URL. This solution fundamentally misunderstands how CloudFront caching works.

**Option D:** Update CloudFront distribution to specify **case-insensitive query string processing**.

- ❌ While CloudFront supports case-insensitive query string processing as a native cache policy setting, this feature only solves **half the problem** (case sensitivity). The query parameter **ordering issue** (`?a=1&b=2` vs `?b=2&a=1`) is **not addressed** by this setting — CloudFront still treats differently ordered parameters as distinct cache keys. The cache hit ratio would improve partially but not fully, whereas Lambda@Edge solves both problems comprehensively. Additionally, at the time this exam question was written, case-insensitive query string processing may not have been a generally available CloudFront feature.

### Enterprise Hands-on Knowledge

1. **Lambda@Edge Normalization Pattern:** A production-grade query string normalization function looks like:
   ```javascript
   exports.handler = (event, context, callback) => {
     const request = event.Records[0].cf.request;
     const params = new URLSearchParams(request.querystring);
     
     // Sort params alphabetically and lowercase all values
     const sorted = new URLSearchParams();
     [...params.entries()]
       .sort(([a], [b]) => a.toLowerCase().localeCompare(b.toLowerCase()))
       .forEach(([key, value]) => {
         sorted.append(key.toLowerCase(), value.toLowerCase());
       });
     
     request.querystring = sorted.toString();
     callback(null, request);
   };
   ```

2. **Lambda@Edge Best Practices:**
   - Use **Node.js runtime** for fastest cold start at the edge
   - Keep function size **under 1 MB** (Lambda@Edge has stricter size limits)
   - Set **conservative timeout** (viewer request max is 5 seconds, but aim for sub-100ms execution)
   - Publish function versions to `us-east-1` — Lambda@Edge functions must be created in this Region
   - Use CloudFront **cache policy** with query strings included in the cache key to complement normalization

3. **Cache Hit Ratio Monitoring:** Before and after deploying normalization:
   - Monitor `CacheHitRate` and `OriginLatency` CloudWatch metrics for the CloudFront distribution
   - Enable **CloudFront standard logs** to audit which query strings are still causing cache misses
   - Use **CloudFront real-time logs** for near-instant visibility into cache behavior

4. **Alternative Long-Term Fix:** While Lambda@Edge solves the problem quickly, the **root cause** is the application generating inconsistently formatted URLs. Work with the development team to standardize URL generation in the application code — enforce consistent parameter ordering and lowercase formatting at the source. This is the true long-term fix.

5. **Common SAP-C02 Trap:** When "as quickly as possible" appears in a question about CloudFront, Lambda@Edge is frequently the answer because you can deploy a function globally in minutes without provisioning servers, load balancers, or changing application code.
