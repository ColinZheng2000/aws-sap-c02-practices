A solutions architect is investigating an issue in which a company cannot establish new sessions in Amazon Workspaces. An initial analysis indicates that the issue involves user profiles. The Amazon Workspaces environment is configured to use Amazon FSx for Windows File Server as the profile share storage. The FSx for Windows File Server file system is configured with 10 TB of storage.  
  
The solutions architect discovers that the file system has reached Its maximum capacity. The solutions architect must ensure that users can regain access. The solution also must prevent the problem from occurring again.  
  
Which solution will meet these requirements?

- A. Remove old user profiles to create space. Migrate the user profiles to an Amazon FSx for Lustre file system.
- B. Increase capacity by using the update-file-system command. Implement an Amazon CloudWatch metric that monitors free space. Use Amazon EventBridge to invoke an AWS Lambda function to increase capacity as required.
- C. Monitor the file system by using the FreeStorageCapacity metric in Amazon CloudWatch. Use AWS Step Functions to increase the capacity as required.
- D. Remove old user profiles to create space. Create an additional FSx for Windows File Server file system. Update the user profile redirection for 50% of the users to use the new file system.

# Analysis of Amazon Workspaces FSx Capacity Issue Solution
## Understanding the Problem
The core requirements are:
1. **Immediate resolution**: Restore Amazon Workspaces session creation by addressing the full Amazon FSx for Windows File Server (used as profile share storage).
2. **Long-term prevention**: Avoid future capacity exhaustion to prevent recurring access issues.
3. **Alignment with workload**: Maintain compatibility with Windows user profiles (FSLogix) since FSx for Windows is the designated storage for this use case.

Current state: The FSx for Windows file system (10 TB) is at maximum capacity, blocking new user profile creation (required for Workspaces sessions). The solution must resolve the immediate capacity crunch and implement proactive safeguards.

## Services Explanation
### Key AWS Services Involved
1. **Amazon FSx for Windows File Server** (AWS Docs: [Scaling Capacity for FSx for Windows](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/Scaling-capacity.html)):
   - Purpose-built for Windows workloads (e.g., user profile shares via FSLogix), supporting seamless storage scaling (up to 64 TB) without downtime.
   - Allows capacity increases via the `update-file-system` API, AWS CLI, or console—even when the file system is at full capacity (as long as scaling to a larger size).
   - Integrates with CloudWatch for capacity monitoring via native metrics.
2. **Amazon CloudWatch Metrics** (AWS Docs: [FSx for Windows CloudWatch Metrics](https://docs.aws.amazon.com/fsx/latest/WindowsGuide/monitoring-cloudwatch.html)):
   - `FreeStorageCapacity`: Native metric tracking available space on the FSx file system (critical for detecting capacity thresholds).
3. **Amazon EventBridge** (AWS Docs: [EventBridge Event Sources](https://docs.aws.amazon.com/eventbridge/latest/userguide/eb-event-sources.html)):
   - Triggers actions (e.g., Lambda functions) in response to CloudWatch alarms (e.g., when `FreeStorageCapacity` drops below a threshold).
4. **AWS Lambda** (AWS Docs: [Lambda with FSx APIs](https://docs.aws.amazon.com/lambda/latest/dg/services-fsx.html)):
   - Executes serverless code to call AWS APIs (e.g., `fsx:UpdateFileSystem`) and automate capacity scaling—no managed infrastructure required.
5. **Other Relevant Services (Incorrect Options)** :
   - **FSx for Lustre**: Optimized for high-performance computing (HPC) workloads (not Windows user profiles). Incompatible with FSLogix profile shares (AWS Docs: [FSx for Lustre Use Cases](https://docs.aws.amazon.com/fsx/latest/LustreGuide/what-is.html)).
   - **AWS Step Functions**: Orchestrates complex workflows but is overkill for simple "monitor → scale" logic. Adds unnecessary complexity vs. Lambda + EventBridge.
   - **Additional FSx for Windows File System**: Requires manual reconfiguration of user profile redirection (high effort) and does not address proactive scaling.

## Correct Option Explanation (Option B)
This solution addresses both immediate and long-term requirements with minimal effort and alignment with AWS best practices:
1. **Immediate Capacity Relief**: Using the `update-file-system` command directly increases the FSx for Windows storage capacity (e.g., from 10 TB to 20 TB). AWS docs confirm this operation is non-disruptive and works even when the file system is at full capacity—immediately freeing space for new user profiles and restoring Workspaces sessions.
2. **Long-Term Prevention**: 
   - CloudWatch’s `FreeStorageCapacity` metric continuously monitors available space.
   - EventBridge triggers a Lambda function when the metric crosses a threshold (e.g., <10% free space).
   - Lambda automates capacity increases via the `update-file-system` API, eliminating manual intervention and preventing future outages.
3. **Minimal Effort**: Leverages AWS-native integrations (CloudWatch → EventBridge → Lambda) and retains the existing FSx for Windows setup (no migration or reconfiguration of user profiles).

## Incorrect Options Explanation
### Option A (Remove Old Profiles + Migrate to FSx for Lustre)
- **Why Incorrect**: 
  1. FSx for Lustre is incompatible with Windows user profile shares (FSLogix)—it is designed for HPC, not Windows workloads (violates workload alignment).
  2. Removing old profiles is a manual, temporary fix (no long-term prevention).
  3. Migration requires reconfiguring Workspaces profile redirection (high effort) and abandons the purpose-built FSx for Windows environment.

### Option C (CloudWatch + Step Functions to Scale)
- **Why Incorrect**: 
  1. No immediate resolution: The solution only implements monitoring and Step Functions for scaling but fails to address the current full capacity (users remain unable to create sessions).
  2. Overengineering: Step Functions is unnecessary for simple "alarm → scale" workflows. Lambda + EventBridge is more efficient and requires less configuration (violates "least effort").

### Option D (Remove Old Profiles + Additional FSx File System)
- **Why Incorrect**: 
  1. Manual effort: Removing old profiles is a temporary, labor-intensive fix (not scalable for ongoing maintenance).
  2. Complex reconfiguration: Redirecting 50% of users to a new FSx file system requires updating Workspaces group policies or FSLogix settings (high operational overhead).
  3. No long-term prevention: The new file system will eventually hit capacity unless proactive scaling is implemented (the solution lacks auto-scaling).
