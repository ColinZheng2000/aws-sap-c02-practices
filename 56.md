A company has an environment that has a single AWS account. A solutions architect is reviewing the environment to recommend what the company could improve specifically in terms of access to the AWS Management Console. The company’s IT support workers currently access the console for administrative tasks, authenticating with named IAM users that have been mapped to their job role.  
  
The IT support workers no longer want to maintain both their Active Directory and IAM user accounts. They want to be able to access the console by using their existing Active Directory credentials. The solutions architect is using AWS IAM Identity Center (AWS Single Sign-On) to implement this functionality.  
  
Which solution will meet these requirements MOST cost-effectively?

- A. Create an organization in AWS Organizations. Turn on the IAM Identity Center feature in Organizations. Create and configure a directory in AWS Directory Service for Microsoft Active Directory (AWS Managed Microsoft AD) with a two-way trust to the company’s on-premises Active Directory. Configure IAM Identity Center and set the AWS Managed Microsoft AD directory as the identity source. Create permission sets and map them to the existing groups within the AWS Managed Microsoft AD directory.
- B. Create an organization in AWS Organizations. Turn on the IAM Identity Center feature in Organizations. Create and configure an AD Connector to connect to the company’s on-premises Active Directory. Configure IAM Identity Center and select the AD Connector as the identity source. Create permission sets and map them to the existing groups within the company’s Active Directory.
- C. Create an organization in AWS Organizations. Turn on all features for the organization. Create and configure a directory in AWS Directory Service for Microsoft Active Directory (AWS Managed Microsoft AD) with a two-way trust to the company’s on-premises Active Directory. Configure IAM Identity Center and select the AWS Managed Microsoft AD directory as the identity source. Create permission sets and map them to the existing groups within the AWS Managed Microsoft AD directory.
- D. Create an organization in AWS Organizations. Turn on all features for the organization. Create and configure an AD Connector to connect to the company’s on-premises Active Directory. Configure IAM Identity Center and set the AD Connector as the identity source. Create permission sets and map them to the existing groups within the company’s Active Directory.

## 1. Understanding the Problem

The company has a single AWS account and needs to solve two core challenges:

- Eliminate the need for IT staff to maintain both on-premises Active Directory (AD) and AWS IAM user accounts.
- Enable IT staff to access the AWS Management Console using their **existing on-prem AD credentials** via AWS IAM Identity Center.
- The solution must be the **most cost-effective** while meeting the above requirements.

## 2. Services Explanation

To solve this problem, we need to understand four key AWS services and their roles:

### 2.1 AWS Organizations

A service to centrally manage multiple AWS accounts. IAM Identity Center **requires an AWS Organization with "All Features" enabled** (not just basic "Consolidated Billing" features). This is because IAM Identity Center relies on Organization-level capabilities like service control policies (SCPs) and account aggregation._Reference_: [IAM Identity Center Prerequisites - AWS Organizations](https://docs.aws.amazon.com/singlesignon/latest/userguide/prereqs.html#prereqs-org)

### 2.2 AWS IAM Identity Center (AWS SSO)

A centralized identity management service that enables single sign-on (SSO) access to AWS accounts and cloud applications. It supports three identity sources:

- AWS IAM Identity Center (built-in directory, not relevant here).
- AWS Directory Service (AWS Managed Microsoft AD or AD Connector).
- External identity providers (e.g., Okta, not needed here).

### 2.3 AD Connector

A free, proxy-based service that connects AWS services (like IAM Identity Center) to **on-premises AD** without storing credentials. It forwards authentication requests to the on-prem AD, eliminating the need for a separate managed directory or trust relationships._Key Advantage_: No additional cost (unlike AWS Managed Microsoft AD) and minimal setup._Reference_: [What Is AD Connector?](https://docs.aws.amazon.com/directoryservice/latest/admin-guide/directory_microsoft_ad_connector.html)

### 2.4 AWS Managed Microsoft AD

A managed directory service in AWS that acts as a standalone AD domain. To integrate with on-prem AD, it requires a **two-way trust relationship** (complex setup) and incurs hourly pricing (not cost-effective for this use case)._Reference_: [AWS Managed Microsoft AD Pricing](https://aws.amazon.com/directoryservice/pricing/)

## 3. Correct Option Explanation: Option D

Option D meets all requirements and is the most cost-effective. Here’s why:

1. **AWS Organizations Setup**: Creates an Organization and enables "All Features"—this is a mandatory prerequisite for IAM Identity Center (per AWS docs).
2. **Cost-Effective Identity Source**: Uses AD Connector instead of AWS Managed Microsoft AD. AD Connector is free, avoids hourly charges, and directly leverages existing on-prem AD (no trust setup needed).
3. **IAM Identity Center Configuration**: Selects AD Connector as the identity source, so IT staff authenticate with their existing on-prem AD credentials.
4. **Permission Mapping**: Maps permission sets to existing on-prem AD groups—aligns with the company’s goal of reusing existing AD structures.

## 4. Incorrect Options Explanation

### 4.1 Option A

- **Cost Issue**: Uses AWS Managed Microsoft AD (hourly cost), which is more expensive than AD Connector.
- **Organizations Misconfiguration**: Only "turns on the IAM Identity Center feature" instead of enabling "All Features"—violates IAM Identity Center prerequisites.
- **Complexity**: Requires a two-way trust with on-prem AD (unnecessary for this use case).

### 4.2 Option B

- **Organizations Misconfiguration**: Fails to enable "All Features" for the Organization (only enables the IAM Identity Center feature), which is required for IAM Identity Center to function.
- **Other Flaws**: While AD Connector is the right identity source, the missing "All Features" step makes this solution invalid.

### 4.3 Option C

- **Cost Issue**: Uses AWS Managed Microsoft AD (hourly cost), which is not cost-effective compared to AD Connector.
- **Complexity**: Requires a two-way trust with on-prem AD (unnecessary overhead for proxying auth requests).
