A company has implemented a new security requirement. According to the new requirement, the company must scan all traffic from corporate AWS instances in the company's VPC for violations of the company's security policies. As a result of these scans, the company can block access to and from specific IP addresses.
To meet the new requirement, the company deploys a set of Amazon EC2 instances in private subnets to serve as transparent proxies. The company installs approved proxy server software on these EC2 instances. The company modifies the route tables on all subnets to use the corresponding EC2 instances with proxy software as the default route. The company also creates security groups that are compliant with the security policies and assigns these security groups to the EC2 instances.
Despite these configurations, the traffic of the EC2 instances in their private subnets is not being properly forwarded to the internet.
What should a solutions architect do to resolve this issue?

A. Disable source/destination checks on the EC2 instances that run the proxy software.
B. Add a rule to the security group that is assigned to the proxy EC2 instances to allow all traffic between instances that have this security group. Assign this security group to all EC2 instances in the VPC.
C. Change the VPC's DHCP options set. Set the DNS server options to point to the addresses of the proxy EC2 instances.
D. Assign one additional elastic network interface to each proxy EC2 instance. Ensure that one of these network interfaces has a route to the private subnets. Ensure that the other network interface has a route to the internet.



# AWS SAP-C02 Question Analysis — Q350

## 📋 Analysis

### 1. Understanding the Problem

The company deployed EC2 instances as transparent proxy/security appliances inside private subnets.  
All subnet route tables were modified so traffic from workloads is redirected through these EC2 instances before reaching the internet.

The routing design itself is mostly correct, but the forwarding behavior is failing.

The critical AWS networking concept is:
- an EC2 instance normally only accepts traffic where it is the source or destination
- a proxy/router/firewall forwards transit traffic on behalf of other systems

Therefore, when EC2 acts like a network appliance, AWS requires additional configuration to allow packet forwarding.

- --

### 2. Options Analysis

#### ✅ Option A (CORRECT): Disable source/destination checks on the EC2 instances that run the proxy software.

This is the correct answer.

By default, EC2 instances enforce source/destination checks. AWS validates that:
- incoming packets are destined for the instance itself
- outgoing packets originate from the instance itself

For transparent proxies, NAT instances, routers, or firewall appliances:
- packets belong to other instances
- the EC2 instance only forwards the traffic

As a result:
- AWS drops the packets unless source/destination checks are disabled

This is a mandatory configuration for:
- NAT instances
- EC2-based firewalls
- IDS/IPS appliances
- custom routing appliances
- transparent proxies

This behavior is heavily tested in SAP-C02 because it represents a core VPC networking concept.

Official AWS guidance explicitly states:
- disable source/destination checks when an EC2 instance performs routing or NAT functions

Relevant AWS official references:
- Amazon EC2 User Guide → Source/destination checks
- Amazon VPC User Guide → NAT instances

- --

#### ❌ Option B (INCORRECT): Modify security groups to allow all intra-group traffic

This does not address the actual issue.

Security groups only control:
- allowed inbound traffic
- allowed outbound traffic

The problem is not packet filtering.  
The packets are rejected because AWS networking detects the EC2 instance forwarding third-party traffic.

Even if all traffic is allowed in the security group:
- source/destination checks still block forwarding

Additionally, assigning one permissive security group to all EC2 instances:
- weakens segmentation
- violates least-privilege design principles

This is a common exam trap:
- confusing routing behavior with security group behavior

- --

#### ❌ Option C (INCORRECT): Change DHCP options set to use proxy instances as DNS servers

DHCP option sets control:
- DNS server assignment
- domain-name settings
- NTP configuration

They do not affect:
- packet forwarding
- routing
- internet gateway traffic flow

The architecture already redirects traffic through route tables.  
DNS configuration changes cannot enable EC2 transit forwarding.

This option incorrectly mixes:
- DNS resolution
with
- network routing

- --

#### ❌ Option D (INCORRECT): Add additional ENIs for separate routing paths

Additional ENIs are not required to solve this issue.

Some advanced firewall architectures use:
- separate trust/untrust interfaces
- dedicated management interfaces

However, the root problem still remains:
- source/destination checks are enabled

Even with multiple ENIs:
- the EC2 instance still cannot forward traffic properly until source/destination checks are disabled

This option introduces unnecessary complexity without resolving the core issue.

### 3. Hands-On Knowledge

In enterprise AWS environments, EC2-based security appliances are commonly used for:
- deep packet inspection
- outbound filtering
- IDS/IPS
- compliance controls
- SSL/TLS inspection

Typical architecture pattern:
1. Private subnet route tables send `0.0.0.0/0` traffic to inspection appliances
2. Security appliance inspects traffic
3. Appliance forwards traffic to NAT Gateway or Internet Gateway
4. Source/destination checks are disabled

AWS CLI example:

```bash
aws ec2 modify-instance-attribute \
  - -instance-id i-1234567890abcdef0 \
  - -no-source-dest-check
```

Verification command:

```bash
aws ec2 describe-instance-attribute \
  - -instance-id i-1234567890abcdef0 \
  - -attribute sourceDestCheck
```

Important production considerations:
- deploy appliances across multiple AZs
- avoid asymmetric routing
- monitor connection tracking limits
- use autoscaling where possible
- ensure route table failover automation

Modern AWS recommendation:
- use Gateway Load Balancer (GWLB) for scalable traffic inspection

GWLB advantages:
- transparent appliance insertion
- automatic scaling
- high availability
- integration with third-party firewall vendors

However, SAP-C02 still frequently tests classic EC2 forwarding behavior because many enterprises continue using EC2-based inspection appliances.

Relevant AWS official documentation:
- https://docs.aws.amazon.com/vpc/latest/userguide/VPC_NAT_Instance.html
- https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/using-eni.html#change_source_dest_check
- https://docs.aws.amazon.com/vpc/latest/userguide/gateway-load-balancer.html

* *面试官可能追问 (Possible Follow-up Questions)**：

1. "What is source/destination check in AWS?"
   → 回答要点 (Key Answer Points)：By default, EC2 validates that traffic belongs to the instance itself. When an EC2 instance acts as a router, NAT device, firewall, or proxy, this check must be disabled so the instance can forward transit traffic.

2. "When should Gateway Load Balancer be used instead of standalone EC2 firewalls?"
   → 回答要点 (Key Answer Points)：GWLB is preferred for scalable inspection architectures requiring high availability and automatic appliance scaling. It simplifies traffic insertion and operational management compared to manually managed EC2 proxy appliances.
