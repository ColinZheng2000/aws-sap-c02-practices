An online retail company hosts its stateful web-based application and MySQL database in an on-premises data center on a single server. The company wants to increase its customer base by conducting more marketing campaigns and promotions. In preparation, the company wants to migrate its application and database to AWS to increase the reliability of its architecture.
Which solution should provide the HIGHEST level of reliability?

A. Migrate the database to an Amazon RDS MySQL Multi-AZ DB instance. Deploy the application in an Auto Scaling group on Amazon EC2 instances behind an Application Load Balancer Store sessions in Amazon Neptune.
B. Migrate the database to Amazon Aurora MySQL. Deploy the application in an Auto Scaling group on Amazon EC2 instances behind an Application Load Balancer. Store sessions in an Amazon ElastiCache for Redis replication group.
C. Migrate the database to Amazon DocumentDB (with MongoDB compatibility). Deploy the application in an Auto Scaling group on Amazon EC2 instances behind a Network Load Balancer. Store sessions in Amazon Kinesis Data Firehose.
D. Migrate the database to an Amazon RDS MariaDB Multi-AZ DB instance. Deploy the application in an Auto Scaling group on Amazon EC2 instances behind an Application Load Balancer. Store sessions in Amazon ElastiCache for Memcached.


Suggest Answer: B

---

## 📋 Analysis

### Understanding the Problem

A retail company runs a stateful web application with a MySQL database on a single on-premises server — a precarious single point of failure. They plan marketing campaigns that will drive more traffic, so they need to migrate to AWS with the **highest possible reliability**. The solution must preserve session state (the application is stateful) and ensure the database tier can survive infrastructure failures.

### ✅ Correct Option: B

**Aurora MySQL + EC2 Auto Scaling behind ALB + ElastiCache for Redis (replication group)**

This combination delivers the highest reliability:

1. **Amazon Aurora MySQL** provides enterprise-grade database reliability:
   - Data is replicated 6 copies across 3 Availability Zones by default
   - Automatic failover to a read replica (typically < 30 seconds)
   - Continuous backup to Amazon S3 with point-in-time recovery
   - Storage is self-healing; corrupted blocks are automatically repaired
   - Aurora is designed for 99.99% availability (can be extended to 99.999% with Aurora Global Database)

2. **ElastiCache for Redis replication group** is the gold standard for session management:
   - Redis replication group with Multi-AZ provides automatic failover for the session store
   - Redis supports data persistence (snapshots + append-only file), so sessions survive node failures
   - Sub-millisecond latency for session lookups keeps the application responsive
   - Redis `AUTH` token and encryption in-transit/at-rest meet security requirements

3. **EC2 Auto Scaling + ALB** provides compute resilience:
   - ALB distributes traffic across healthy instances in multiple AZs
   - Auto Scaling replaces failed instances automatically
   - Health checks detect unhealthy instances and route traffic away

### ❌ Incorrect Options

**Option A — RDS MySQL Multi-AZ + Neptune for sessions**

- **Amazon Neptune is a graph database** designed for highly connected datasets (social networks, recommendation engines, fraud detection). It is entirely unsuitable for session storage — sessions are simple key-value lookups, not graph traversals. Using Neptune for sessions would add enormous cost and latency for zero benefit.
- RDS MySQL Multi-AZ is reliable but less durable than Aurora (data is replicated to only 2 copies — a primary and a standby).

**Option C — DocumentDB + NLB + Kinesis Data Firehose for sessions**

- **Amazon DocumentDB** is MongoDB-compatible; migrating from MySQL would require a complete schema and query rewrite — significant risk and effort.
- **Kinesis Data Firehose** is a streaming ingestion service for delivering data to S3, Redshift, Splunk, etc. It is not a session store. Sessions require low-latency **reads and writes**; Firehose only handles data ingestion (one direction).
- NLB operates at Layer 4 (TCP/TLS) and lacks the Layer 7 routing and sticky session capabilities of ALB, making it harder to maintain session affinity.

**Option D — RDS MariaDB Multi-AZ + ElastiCache for Memcached**

- **ElastiCache Memcached** does NOT support replication, persistence, or automatic failover. If a Memcached node fails, all cached sessions on that node are permanently lost. For a stateful application, this is a reliability regression.
- Memcached is a pure cache — it's designed for data that can be regenerated, not for session data that must survive failures.
- RDS MariaDB Multi-AZ is adequate but not the *highest* reliability option compared to Aurora.

### 🔧 Hands-On Knowledge

- **Reliability Hierarchy for MySQL on AWS**: Aurora MySQL > RDS MySQL Multi-AZ > RDS MySQL Single-AZ > Self-managed MySQL on EC2. Aurora wins because of its distributed, self-healing storage architecture.
- **Session Store Comparison**: ElastiCache Redis (replication group) > DynamoDB (with DAX) > ElastiCache Memcached (no persistence) > Sticky sessions on ALB (instance-bound, lost on scale-in). For truly critical sessions, some enterprises combine ElastiCache Redis with DynamoDB as a durable fallback.
- **ALB Sticky Sessions**: Even with an external session store, enabling ALB sticky sessions (`stickiness.enabled=true`) can reduce session store lookups and improve performance.
- **Aurora Serverless v2**: For variable workloads (like campaign-driven traffic spikes), Aurora Serverless v2 can automatically scale capacity up and down without downtime — ideal for retail use cases with promotional peaks.
