A company is building an application on AWS. The application sends logs to an Amazon Elasticsearch Service (Amazon ES) cluster for analysis. All data must be stored within a VPC.
Some of the company's developers work from home. Other developers work from three different company office locations. The developers need to access
Amazon ES to analyze and visualize logs directly from their local development machines.
Which solution will meet these requirements?

A. Configure and set up an AWS Client VPN endpoint. Associate the Client VPN endpoint with a subnet in the VPC. Configure a Client VPN self-service portal. Instruct the developers to connect by using the client for Client VPN.
B. Create a transit gateway, and connect it to the VPC. Create an AWS Site-to-Site VPN. Create an attachment to the transit gateway. Instruct the developers to connect by using an OpenVPN client.
C. Create a transit gateway, and connect it to the VPC. Order an AWS Direct Connect connection. Set up a public VIF on the Direct Connect connection. Associate the public VIF with the transit gateway. Instruct the developers to connect to the Direct Connect connection
D. Create and configure a bastion host in a public subnet of the VPC. Configure the bastion host security group to allow SSH access from the company CIDR ranges. Instruct the developers to connect by using SSH.



# AWS SAP-C02 Question Analysis — Q316

## 📋 Analysis

### 1. Understanding the Problem

Key requirements:

- Amazon ES/OpenSearch cluster must remain inside a VPC
- Developers need direct access from:
  - Home networks
  - Three office locations
- Access must be secure
- Developers connect from local development machines
- Need practical remote connectivity into AWS VPC resources

This is fundamentally a:
> Secure remote user access to private VPC resources problem.

The critical clue:
- "developers work from home"

This immediately suggests:
- Remote user VPN access
- Client-based connectivity
- Not just site-to-site networking

---

## 2. Option Analysis

---

### ✅ Option A (CORRECT): AWS Client VPN

This is the correct solution.

Why:

- AWS Client VPN is purpose-built for:
  - Remote employee access
  - Secure user-level VPN connectivity
  - Access from home or office locations

Architecture:

```text
Developer Laptop
      ↓
AWS Client VPN
      ↓
VPC Subnet Association
      ↓
Amazon ES/OpenSearch
```

Key advantages:

- Fully managed VPN service
- Supports remote users anywhere
- Works for both:
  - Home developers
  - Office developers
- Secure encrypted access into VPC
- No need to expose OpenSearch publicly
- Scales automatically
- Easy client configuration

Important detail:
- Associating the Client VPN endpoint with VPC subnets enables access to VPC resources like OpenSearch.

The self-service portal:
- Simplifies onboarding
- Allows developers to download VPN configuration profiles

This is the standard AWS-native answer.

Official reference:
https://docs.aws.amazon.com/vpn/latest/clientvpn-admin/what-is.html

---

### ❌ Option B (INCORRECT): Transit Gateway + Site-to-Site VPN

This is designed for network-to-network connectivity, not individual developers.

Problems:

- Site-to-Site VPN connects:
  - Corporate networks
  - Data centers
  - Routers/firewalls
- It does NOT support:
  - Individual remote laptops
  - Home-user connectivity

Major flaw:
- Home developers cannot practically use Site-to-Site VPN.

Also:
- OpenVPN clients are unrelated to AWS Site-to-Site VPN architecture.

Transit Gateway is unnecessary complexity here.

This option confuses:
- Hybrid network connectivity
with
- Remote user access

---

### ❌ Option C (INCORRECT): Direct Connect

This is entirely inappropriate for this use case.

Problems:

- Direct Connect is for:
  - Dedicated private enterprise connectivity
  - Large-scale hybrid infrastructure
- Extremely expensive and operationally heavy
- Does not help home users
- Public VIF is incorrect for private VPC OpenSearch access
- Developers do not "connect to Direct Connect"

Additional issue:
- Transit Gateway + public VIF combination is architecturally incorrect here.

This is massive overengineering.

---

### ❌ Option D (INCORRECT): Bastion Host

This partially solves remote access but fails the broader requirement.

Problems:

- Bastion hosts are mainly for:
  - Administrative SSH access
  - Managing EC2 instances
- Not ideal for application/service access

Major limitations:

- Home developers are not covered unless public internet access is opened
- SSH tunneling for OpenSearch access is operationally awkward
- Not scalable
- Security management burden
- Single point of failure unless highly engineered

Most importantly:
- This is not the AWS-recommended modern approach for remote workforce access.

AWS Client VPN replaced many historical bastion-style patterns.

## 3. Core AWS Concept Tested

This question tests whether you can distinguish between:

| Connectivity Type | Correct AWS Service |
|---|---|
| Remote users | AWS Client VPN |
| Office/data center networks | Site-to-Site VPN |
| Dedicated enterprise private circuits | Direct Connect |
| Administrative jump access | Bastion Host |

## 5. Why Client VPN Fits Perfectly

AWS Client VPN provides:

- TLS-encrypted connectivity
- User-based authentication
- Centralized access management
- Native VPC integration
- Support for roaming/home users
- Minimal infrastructure management

Compared with bastion hosts:

| Feature | Client VPN | Bastion Host |
|---|---|---|
| Remote user access | Excellent | Limited |
| Scalability | High | Manual |
| Security posture | Strong | Weaker |
| Operational overhead | Low | Higher |
| AWS-native solution | Yes | Legacy pattern |

## 7. Exam Trap Recognition

Common SAP-C02 traps:

### Trap 1:
Using Site-to-Site VPN for individual users

Remember:
- Site-to-Site VPN = network ↔ network
- Client VPN = user ↔ VPC

---

### Trap 2:
Using Direct Connect for remote employees

Direct Connect:
- Is not a user remote-access solution
- Is enterprise infrastructure connectivity

---

### Trap 3:
Using bastion hosts as universal remote access

Bastions:
- Primarily administrative
- Not modern workforce networking solutions

## 💼 面试扩展 (Interview Extension)

> 面试频率：🎤🎤🎤

### 面试官可能追问

#### 1. “Client VPN 和 Site-to-Site VPN 的区别是什么？”

回答要点：

- Client VPN：
  - 用户级访问
  - 适合远程员工
  - 基于客户端连接

- Site-to-Site VPN：
  - 网络级连接
  - 适合办公室/数据中心互联
  - 基于路由器设备

---

#### 2. “为什么不直接把 OpenSearch 暴露到公网？”

回答要点：

- OpenSearch 常包含敏感日志
- VPC-only 模式更安全
- 避免公网攻击面
- 满足企业合规要求
- 通过 VPN 提供受控访问更符合最佳实践

---

#### 3. “什么时候还会使用 bastion host？”

回答要点：

适用于：

- SSH/RDP 管理
- 临时运维访问
- Legacy 环境

但现代 AWS 架构更推荐：

- SSM Session Manager
- Client VPN
- Zero Trust 访问模式
