A company has a multi-tier web application that runs on a fleet of Amazon EC2 instances behind an Application Load Balancer (ALB). The instances are in an Auto Scaling group. The ALB and the Auto Scaling group are replicated in a backup AWS Region. The minimum value and the maximum value for the Auto Scaling group are set to zero. An Amazon RDS Multi-AZ DB instance stores the application's data. The DB instance has a read replica in the backup Region. The application presents an endpoint to end users by using an Amazon Route 53 record.  
  
The company needs to reduce its RTO to less than 15 minutes by giving the application the ability to automatically fail over to the backup Region. The company does not have a large enough budget for an active-active strategy.  
  
What should a solutions architect recommend to meet these requirements?

- A. Reconfigure the application's Route 53 record with a latency-based routing policy that load balances traffic between the two ALBs. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Create an Amazon CloudWatch alarm that is based on the HTTPCode_Target_5XX_Count metric for the ALB in the primary Region. Configure the CloudWatch alarm to invoke the Lambda function.
- B. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Configure Route 53 with a health check that monitors the web application and sends an Amazon Simple Notification Service (Amazon SNS) notification to the Lambda function when the health check status is unhealthy. Update the application's Route 53 record with a failover policy that routes traffic to the ALB in the backup Region when a health check failure occurs.
- C. Configure the Auto Scaling group in the backup Region to have the same values as the Auto Scaling group in the primary Region. Reconfigure the application's Route 53 record with a latency-based routing policy that load balances traffic between the two ALBs. Remove the read replica. Replace the read replica with a standalone RDS DB instance. Configure Cross-Region Replication between the RDS DB instances by using snapshots and Amazon S3.
- D. Configure an endpoint in AWS Global Accelerator with the two ALBs as equal weighted targets. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Create an Amazon CloudWatch alarm that is based on the HTTPCode_Target_5XX_Count metric for the ALB in the primary Region. Configure the CloudWatch alarm to invoke the Lambda function.

## Analysis

---

### Understanding the Problem

The company has a multi-tier web application using **Amazon EC2** instances in an **Auto Scaling group** behind an **Application Load Balancer (ALB)** in a primary AWS Region. A secondary (backup) Region has an identical setup, but the Auto Scaling group has a minimum and maximum capacity of **zero** (meaning no instances are running). Data is stored in an **Amazon RDS Multi-AZ DB instance** with a **read replica** in the backup Region. **Amazon Route 53** presents the endpoint to users.

The primary goal is to achieve an **Recovery Time Objective (RTO)** of **less than 15 minutes** by implementing an **automatic failover** to the backup Region. The solution must be **cost-effective** (ruling out an active-active strategy, which would require the backup Auto Scaling group to be running). This scenario is known as a **Pilot Light** disaster recovery strategy.

Key requirements:

1. **Automatic Failover:** Transition to the backup Region without manual intervention.
    
2. **RTO < 15 minutes:** The switchover process must be fast.
    
3. **Cost-Effective:** The backup environment should be minimally running (Pilot Light).
    
4. **Complete Switch:** The process must:
    
    - Route traffic to the backup ALB.
        
    - Scale up the EC2 fleet in the backup Region (modify Auto Scaling group).
        
    - Make the backup database operational (promote the RDS read replica).

### Correct Option(s) Explanation

**B. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Configure Route 53 with a health check that monitors the web application and sends an Amazon Simple Notification Service (Amazon SNS) notification to the Lambda function when the health check status is unhealthy. Update the application's Route 53 record with a failover policy that routes traffic to the ALB in the backup Region when a health check failure occurs.**

This solution uses the appropriate services and workflow for a **Pilot Light** DR strategy with a low RTO goal.

1. **Traffic Failover:**
    
    - **Route 53 Health Check:** Continuously monitors the health of the primary Region's application endpoint (via the ALB).
        
    - **Route 53 Failover Routing Policy:** The primary (weighted, typically 1) record points to the primary ALB and is associated with the health check. The secondary (weighted, typically 1) record points to the backup ALB and is set to become active only when the health check for the primary fails. This is the **standard, fastest, and most cost-effective way to achieve DNS failover** in AWS, directly addressing the requirement to route traffic to the backup ALB.
        
2. **Operational Readiness (The "Pilot Light" Activation):**
    
    - **Lambda Function:** A single Lambda function in the backup Region performs the two necessary scaling/promotion actions:
        
        - **Promotes the RDS read replica** to a standalone primary DB instance.
            
        - **Modifies the backup Auto Scaling group's desired/min/max capacity** from 0 to N (the required running count), quickly launching the EC2 fleet.
            
3. **Automatic Trigger:**
    
    - **Route 53 Health Check → SNS → Lambda:** The Route 53 Health Check can be configured to send an **Amazon SNS** notification upon a state change (Healthy to Unhealthy). The Lambda function in the backup Region can be subscribed to this SNS topic. This links the traffic failure detection directly to the DR environment activation, satisfying the **automatic failover** requirement and contributing to an RTO of under 15 minutes. (Refer to the _AWS Disaster Recovery whitepaper_ for the Pilot Light pattern).

### Incorrect Options Explanation

**A. Reconfigure the application's Route 53 record with a latency-based routing policy that load balances traffic between the two ALBs. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Create an Amazon CloudWatch alarm that is based on the HTTPCode_Target_5XX_Count metric for the ALB in the primary Region. Configure the CloudWatch alarm to invoke the Lambda function.**

- **Incorrect Routing:** Latency-based routing is for performance and directs traffic to the Region with the lowest latency. It is not a DR policy. For this to work as DR, the primary ALB would need to be completely removed from DNS when it fails, which the CloudWatch alarm mechanism does not do directly. The Failover Routing Policy is the correct choice.
    
- **Suboptimal Trigger:** Using a CloudWatch alarm on the primary ALB's 5XX count is a valid way to detect failure, but it is less direct for DNS failover than the Route 53 Health Check that is inherently tied to the Failover Routing Policy. The failover and activation steps are separate, whereas Option B links them via the Route 53 Health Check mechanism itself.
    

**C. Configure the Auto Scaling group in the backup Region to have the same values as the Auto Scaling group in the primary Region. Reconfigure the application's Route 53 record with a latency-based routing policy that load balances traffic between the two ALBs. Remove the read replica. Replace the read replica with a standalone RDS DB instance. Configure Cross-Region Replication between the RDS DB instances by using snapshots and Amazon S3.**

- **Violates Cost Constraint:** Setting the ASG in the backup Region to the same values as the primary means the EC2 instances are running 24/7. This is an **active-passive** (Warm Standby) or **active-active** approach, which **violates the budget constraint** against an active-active strategy and is much more expensive than the Pilot Light approach specified (min=max=0).
    
- **Incorrect Routing:** Latency-based routing is incorrect for DR failover.
    
- **Incorrect Data Replication:** Removing the Read Replica and replacing it with manual snapshot/S3 replication is a **much slower** process for recovery than promoting a Read Replica. Snapshot replication is a slower DR method, typically for **Backup and Restore**, not Pilot Light with an RTO<15 minutes.
    

**D. Configure an endpoint in AWS Global Accelerator with the two ALBs as equal weighted targets. Create an AWS Lambda function in the backup Region to promote the read replica and modify the Auto Scaling group values. Create an Amazon CloudWatch alarm that is based on the HTTPCode_Target_5XX_Count metric for the ALB in the primary Region. Configure the CloudWatch alarm to invoke the Lambda function.**

- **Incorrect Routing/Weighting:** Global Accelerator's health check will automatically direct traffic away from an unhealthy primary ALB and to the secondary ALB. However, the targets are set as "equal weighted." With a Pilot Light strategy, the backup ALB has no healthy targets (ASG is min=max=0) and is likely to be marked unhealthy itself, or it will direct traffic to zero instances, which fails. For this to work with Pilot Light, the weight of the primary must be high (e.g., 255) and the secondary's weight low (e.g., 0). Furthermore, the CloudWatch alarm trigger is a complex, separate mechanism. Route 53 Failover Routing is simpler and more appropriate for a Pilot Light DR goal.
    
- **Suboptimal Trigger:** Similar to option A, using a CloudWatch alarm is an indirect method for triggering the activation steps, separate from the primary traffic routing mechanism.
