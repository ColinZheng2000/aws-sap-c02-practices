A company has an application that stores data in a single Amazon S3 bucket. The company must keep all data for 1 year. The company’s security team is concerned that an attacker could gain access to the AWS account through leaked long-term credentials.

Which solution will ensure that existing and future objects in the S3 bucket are protected?

A. Create a new AWS account that is accessible only to the security team through an assumed role. Create an S3 bucket in the new account. Enable S3 Versioning and S3 Object Lock. Configure a default retention period of 1 year. Set up replication from the existing S3 bucket to the new S3 bucket. Create an S3 Batch Replication job to copy all existing data.
B. Use the s3-bucket-versioning-enabled AWS Config managed rule. Configure an automatic remediation action that uses an AWS Lambda function to enable S3 Versioning and MFA Delete on noncompliant resources. Add an S3 Lifecycle rule to delete objects after 1 year.
C. Explicitly deny bucket creation from all users and roles except for an AWS Service Catalog launch constraint role. Define a Service Catalog product for the creation of the S3 bucket to force S3 Versioning and MFA Delete to be enabled. Authorize users to launch the product when they need to create an S3 bucket.
D. Enable Amazon GuardDuty with the S3 protection feature for the account and the AWS Region. Add an S3 Lifecycle rule to delete objects after 1 year.



# Understanding the Problem

The key requirement is not just retention, but protection against an attacker who gains access to the AWS account using leaked long-term credentials.

The solution must:
- Protect both existing and future objects
- Prevent deletion or modification for 1 year
- Remain secure even if the primary account is compromised
- Provide immutable storage behavior

This is fundamentally an S3 Object Lock + isolated account architecture question.

Important signals:
- “attacker could gain access to the AWS account”
- “existing and future objects”
- “must keep all data for 1 year”

Normal IAM permissions, lifecycle rules, or versioning alone are insufficient because a compromised admin could still delete objects or disable protections.

# Correct Answer

✅ A. Create a separate AWS account controlled by the security team, enable S3 Versioning and S3 Object Lock with a 1-year retention period, and replicate all existing and future objects.

# Why A is Correct

This option combines all required protections:

## 1. S3 Object Lock Provides WORM Protection

S3 Object Lock enables:
- Write Once Read Many (WORM)
- Retention enforcement
- Protection against deletion/modification

Even administrators cannot delete protected objects before retention expires.

This directly addresses:
- ransomware
- malicious insiders
- compromised credentials

## 2. Versioning Is Required

S3 Object Lock requires:
- S3 Versioning enabled

This protects object versions and preserves immutable copies.

## 3. Separate AWS Account Improves Security Isolation

The question explicitly mentions concern about account compromise.

Using a separate account:
- isolates immutable backups
- limits blast radius
- prevents attackers in the primary account from deleting protected copies

This is a major enterprise security best practice.

## 4. Replication Protects Future Objects

S3 replication continuously copies:
- new uploads
- updates

Thus future objects are protected automatically.

## 5. Batch Replication Protects Existing Objects

The question explicitly requires:
- existing objects protected

S3 Batch Replication handles historical objects already in the bucket.

# Why the Other Options Are Incorrect

## ❌ B. AWS Config + MFA Delete + Lifecycle

This sounds security-focused but fails multiple requirements.

### Problems

#### MFA Delete is insufficient
If attackers obtain privileged credentials and session access, MFA protections may still be bypassed operationally.

More importantly:
- MFA Delete is not immutable storage
- It does not provide WORM compliance guarantees

#### No protection against admin compromise
Attackers with sufficient privileges could:
- alter lifecycle policies
- disable versioning
- delete versions eventually

#### Lifecycle expiration deletes data
The requirement is retention and protection, not automated deletion as the primary control.

Most critically:
- no Object Lock

Therefore objects are not immutable.

## ❌ C. Service Catalog enforced bucket creation

This option focuses only on:
- future bucket creation governance

But the problem concerns:
- an existing bucket
- existing objects
- protection from compromised credentials

### Problems

- Does not protect existing data
- No cross-account isolation
- MFA Delete still not sufficient
- No immutable backup architecture

This is governance, not durable ransomware-resistant retention.

## ❌ D. GuardDuty + Lifecycle

GuardDuty only:
- detects suspicious activity

It does not:
- prevent deletion
- create immutable storage
- enforce retention

Lifecycle rules actually automate deletion after 1 year and provide no protection against malicious deletion before that.

This completely misses the core requirement.

# Enterprise Hands-On Knowledge

In real enterprise environments, ransomware-resistant AWS backup designs commonly include:

## Immutable Backup Account Pattern

Organizations often use:
- separate backup accounts
- tightly restricted access
- Object Lock in Compliance mode

This mirrors enterprise backup vault architectures.

## Why Object Lock Matters

Without Object Lock:
- attackers can delete snapshots
- attackers can purge versions
- attackers can destroy backups

Versioning alone is not enough.

## Compliance vs Governance Mode

S3 Object Lock supports:
- Governance mode
- Compliance mode

Compliance mode is strongest because:
- even root users cannot remove retention

SAP-C02 frequently tests understanding that Object Lock is the true immutable storage mechanism.

# Interview Extension

A common follow-up interview question is:

“How do you protect backups from ransomware in AWS?”

Strong answer:
- separate backup account
- least privilege
- immutable storage
- S3 Object Lock
- cross-account replication

When you see:

Immediately think:
- S3 Object Lock
- immutable storage

Versioning alone is usually a distractor.
