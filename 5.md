A company uses a service to collect metadata from applications that the company hosts on premises. Consumer devices such as TVs and internet radios access the applications. Many older devices do not support certain HTTP headers and exhibit errors when these headers are present in responses. The company has configured an on-premises load balancer to remove the unsupported headers from responses sent to older devices, which the company identified by the User-Agent headers.  
The company wants to migrate the service to AWS, adopt serverless technologies, and retain the ability to support the older devices. The company has already migrated the applications into a set of AWS Lambda functions.  
Which solution will meet these requirements?

- A. Create an Amazon CloudFront distribution for the metadata service. Create an Application Load Balancer (ALB). Configure the CloudFront distribution to forward requests to the ALB. Configure the ALB to invoke the correct Lambda function for each type of request. Create a CloudFront function to remove the problematic headers based on the value of the User-Agent header.
- B. Create an Amazon API Gateway REST API for the metadata service. Configure API Gateway to invoke the correct Lambda function for each type of request. Modify the default gateway responses to remove the problematic headers based on the value of the User-Agent header.
- C. Create an Amazon API Gateway HTTP API for the metadata service. Configure API Gateway to invoke the correct Lambda function for each type of request. Create a response mapping template to remove the problematic headers based on the value of the User-Agent. Associate the response data mapping with the HTTP API.
- D. Create an Amazon CloudFront distribution for the metadata service. Create an Application Load Balancer (ALB). Configure the CloudFront distribution to forward requests to the ALB. Configure the ALB to invoke the correct Lambda function for each type of request. Create a Lambda@Edge function that will remove the problematic headers in response to viewer requests based on the value of the User-Agent header.

## Understanding the Problem

The company is migrating an on-premises metadata service to **AWS serverless technologies**, specifically **AWS Lambda functions**. The core requirement is to **retain the ability to remove specific unsupported HTTP headers** from responses for older consumer devices. This header removal logic is conditional, depending on the **User-Agent header** in the request, and previously handled by an on-premises load balancer. The final architecture must be serverless and support this conditional response modification.

**Key Requirements:**
1.  Migrate to AWS, using **serverless technologies** (Lambda).
2.  Maintain the ability to serve older devices.
3.  Conditionally **remove unsupported HTTP headers from the response** (response modification).
4.  The condition for removal is based on the **User-Agent header** in the request.

## Correct Option(s) Explanation

The correct solution is **D. Create an Amazon CloudFront distribution for the metadata service. Create an Application Load Balancer (ALB). Configure the CloudFront distribution to forward requests to the ALB. Configure the ALB to invoke the correct Lambda function for each type of request. Create a Lambda@Edge function that will remove the problematic headers in response to viewer requests based on the value of the User-Agent header.**

* **CloudFront + Lambda@Edge:** This is the most effective and standard AWS pattern for performing conditional, custom HTTP request and response manipulation at scale and with low latency.
    * **Lambda@Edge** is capable of running custom logic (written in Node.js or Python) that can **inspect the User-Agent header** in the original *Viewer Request* or *Origin Request* and, crucially, **modify the response headers** in the *Origin Response* or *Viewer Response* events.
    * The requirement is to remove headers from the **response** based on a condition in the **request (User-Agent)**. A **Lambda@Edge function attached to the *Viewer Response* event** (or *Origin Response*) allows the custom logic to check the `User-Agent` (which is accessible if configured to be forwarded) and remove the problematic headers from the response before it reaches the end user.
* **Serverless Compatibility:** The architecture uses Lambda for compute and CloudFront/Lambda@Edge for modification, maintaining a serverless-friendly approach for the application's core logic and custom requirements.
* **Flexibility:** While an API Gateway could also invoke Lambda, using CloudFront and ALB provides a robust, scalable CDN layer. The ALB's role is to route traffic to the Lambda functions, which is a supported feature.

**(Reference: AWS Lambda@Edge Documentation - Event Structure and Execution Phases)**

## Incorrect Options Explanation

* **A. Create an Amazon CloudFront distribution for the metadata service. Create an Application Load Balancer (ALB). Configure the CloudFront distribution to forward requests to the ALB. Configure the ALB to invoke the correct Lambda function for each type of request. Create a CloudFront function to remove the problematic headers based on the value of the User-Agent header.**
    * **Incorrect Service Choice:** **CloudFront Functions** are designed for lightweight, simple logic (like header manipulation and URL rewrites). While they can modify headers, their execution model is primarily for simple request/response modification **at the viewer request/response stage**. While a CloudFront function *can* remove response headers (in the Viewer Response event), Lambda@Edge provides a more robust environment, especially for complex logic involving inspecting the User-Agent from the request and conditionally modifying the response, which is a common and primary use case for **Lambda@Edge**. Given the complexity involved in maintaining the logic for different older devices and the greater capabilities of Lambda@Edge, CloudFront Functions are often considered a less suitable/more restrictive choice for this type of conditional response manipulation than Lambda@Edge. **Lambda@Edge is the standard, official AWS best practice for this type of complex, conditional header modification.**

* **B. Create an Amazon API Gateway REST API for the metadata service. Configure API Gateway to invoke the correct Lambda function for each type of request. Modify the default gateway responses to remove the problematic headers based on the value of the User-Agent header.**
    * **Incorrect API Gateway Feature:** **Gateway Responses** in API Gateway (REST API) are primarily used to customize **error responses** (e.g., 400s, 500s) generated by the API Gateway service itself, not successful $\text{(200)}$ responses from the integrated Lambda function. It's not the mechanism to conditionally remove headers from a standard, successful application response based on the incoming request's User-Agent.

* **C. Create an Amazon API Gateway HTTP API for the metadata service. Configure API Gateway to invoke the correct Lambda function for each type of request. Create a response mapping template to remove the problematic headers based on the value of the User-Agent. Associate the response data mapping with the HTTP API.**
    * **Incorrect Feature/Inadequate Service:** HTTP APIs are simpler and don't natively support the complex **VTL (Velocity Template Language)** mapping capabilities of REST APIs for conditional logic within the response flow. While **REST APIs** do support mapping templates, the mechanism is usually for transforming the payload, and while header modification is possible, implementing **conditional logic** within the VTL to check the original request's User-Agent and then modify the response headers is generally **complex, error-prone, and not the intended best practice** compared to using an edge compute service like Lambda@Edge, which is designed for this exact purpose. HTTP APIs are even more limited in this regard.
