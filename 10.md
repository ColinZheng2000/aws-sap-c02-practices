A company is running an application on several Amazon EC2 instances in an Auto Scaling group behind an Application Load Balancer. The load on the application varies throughout the day, and EC2 instances are scaled in and out on a regular basis. Log files from the EC2 instances are copied to a central  
Amazon S3 bucket every 15 minutes. The security team discovers that log files are missing from some of the terminated EC2 instances.  
Which set of actions will ensure that log files are copied to the central S3 bucket from the terminated EC2 instances?  

- A. Create a script to copy log files to Amazon S3, and store the script in a file on the EC2 instance. Create an Auto Scaling lifecycle hook and an Amazon EventBridge (Amazon CloudWatch Events) rule to detect lifecycle events from the Auto Scaling group. Invoke an AWS Lambda function on the autoscaling:EC2_INSTANCE_TERMINATING transition to send ABANDON to the Auto Scaling group to prevent termination, run the script to copy the log files, and terminate the instance using the AWS SDK.
- B. Create an AWS Systems Manager document with a script to copy log files to Amazon S3. Create an Auto Scaling lifecycle hook and an Amazon EventBridge (Amazon CloudWatch Events) rule to detect lifecycle events from the Auto Scaling group. Invoke an AWS Lambda function on the autoscaling:EC2_INSTANCE_TERMINATING transition to call the AWS Systems Manager API SendCommand operation to run the document to copy the log files and send CONTINUE to the Auto Scaling group to terminate the instance.
- C. Change the log delivery rate to every 5 minutes. Create a script to copy log files to Amazon S3, and add the script to EC2 instance user data. Create an Amazon EventBridge (Amazon CloudWatch Events) rule to detect EC2 instance termination. Invoke an AWS Lambda function from the EventBridge (CloudWatch Events) rule that uses the AWS CLI to run the user-data script to copy the log files and terminate the instance.
- D. Create an AWS Systems Manager document with a script to copy log files to Amazon S3. Create an Auto Scaling lifecycle hook that publishes a message to an Amazon Simple Notification Service (Amazon SNS) topic. From the SNS notification, call the AWS Systems Manager API SendCommand operation to run the document to copy the log files and send ABANDON to the Auto Scaling group to terminate the instance.

## Understanding the Problem

The core issue is that log files from **Amazon EC2 instances** are being lost when the instances are **terminated** by an **Auto Scaling group** (ASG). The application has a varying load, causing the ASG to regularly scale in and out. Logs are currently copied to **Amazon S3** only every 15 minutes. When an instance is terminated between these 15-minute intervals, the most recent logs are lost. The company needs a solution to reliably ensure that the logs are copied to the central S3 bucket **before** any terminating EC2 instance is completely shut down and removed by the Auto Scaling group. This requires a mechanism to **pause** the termination process, execute a log-copy script, and then allow the termination to complete.

## Correct Option(s) Explanation

The correct option is **B**.

**B. Create an AWS Systems Manager document with a script to copy log files to Amazon S3. Create an Auto Scaling lifecycle hook and an Amazon EventBridge (Amazon CloudWatch Events) rule to detect lifecycle events from the Auto Scaling group. Invoke an AWS Lambda function on the autoscaling:EC2_INSTANCE_TERMINATING transition to call the AWS Systems Manager API SendCommand operation to run the document to copy the log files and send CONTINUE to the Auto Scaling group to terminate the instance.**

This option provides the most robust, scalable, and standard AWS approach to this problem:

1. **Preparation (SSM Document):** An **AWS Systems Manager Document** is used to store the script for copying logs to S3. This is the **best practice** for running scripts remotely on managed EC2 instances, as it's secure, auditable, and doesn't require storing the script directly on the instance (unlike option A/C).
    
2. **Pausing Termination (Lifecycle Hook):** An **Auto Scaling lifecycle hook** is configured for the `autoscaling:EC2_INSTANCE_TERMINATING` state. This pauses the EC2 instance's termination, putting it into a `WaiT` state, ensuring the instance is available for the log-copy operation.
    
3. **Event Detection (EventBridge):** **Amazon EventBridge** is configured to detect the event generated by the lifecycle hook (the instance entering the `WaiT` state).
    
4. **Action Execution (Lambda & SSM):** The EventBridge event triggers an **AWS Lambda function**. The Lambda function's job is two-fold:
    
    - It uses the **AWS Systems Manager API's `SendCommand` operation** to execute the log-copy SSM Document on the paused EC2 instance. This runs the script to copy the final logs to S3.
        
    - Once the `SendCommand` completes (ensuring the logs are copied), the Lambda function sends a **`CONTINUE` signal** to the Auto Scaling group using the `CompleteLifecycleAction` API call. This signals the ASG to proceed with the termination.
        

This solution is serverless (Lambda/EventBridge) for the control plane and leverages **SSM Run Command** for reliable execution on the instance, aligning perfectly with AWS best practices for automation during ASG lifecycle events.

## Incorrect Options Explanation

|Option|Reason for Incorrectness|
|---|---|
|**A**|**Incorrect Control Flow:** It suggests sending an **`ABANDON`** signal to prevent termination, then running the script, and finally using the **AWS SDK to terminate the instance**. The `ABANDON` signal _cancels_ the scaling activity, leaving the instance running, which is undesirable for a scale-in event. Furthermore, manually terminating the instance via the SDK is not the correct way to complete a scale-in activity initiated by the ASG; the ASG should be allowed to complete its own scaling action via `CONTINUE` or `ABANDON` after the lifecycle hook action is complete.|
|**C**|**Timing and Reliability:** Relying on an **EventBridge rule to detect EC2 instance termination** (not the ASG lifecycle hook) is too late. The EC2 instance termination event fires _after_ the ASG has told the OS to shut down, meaning the script may not have enough time to run successfully before the OS or network connectivity is lost. The **lifecycle hook** (as used in B, A, and D) is the necessary component to _pause_ the termination before the shutdown process begins. Also, running a script from **user data** via CLI is an unreliable, unmanaged, and insecure approach compared to using SSM Run Command.|
|**D**|**Incorrect Control Flow and Architecture:** This option uses **Amazon SNS** instead of EventBridge/Lambda, which is a possible but less common pattern for complex logic. The major flaw is the reliance on calling the **AWS Systems Manager API `SendCommand` operation** _directly_ from the SNS notification, which would require a complex setup (e.g., an SNS subscription to a Lambda function to call the API). More critically, it sends the **`ABANDON`** signal to the ASG, which, as in option A, **cancels the scaling activity**, leaving the instance running instead of terminating it.|
