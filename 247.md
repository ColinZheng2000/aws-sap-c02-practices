A company runs its application on Amazon EC2 instances and AWS Lambda functions. The EC2 instances experience a continuous and stable load. The Lambda functions experience a varied and unpredictable load. The application includes a caching layer that uses an Amazon MemoryDB for Redis cluster.

A solutions architect must recommend a solution to minimize the company's overall monthly costs.

Which solution will meet these requirements?

A. Purchase an EC2 instance Savings Plan to cover the EC2 instances. Purchase a Compute Savings Plan for Lambda to cover the minimum expected consumption of the Lambda functions. Purchase reserved nodes to cover the MemoryDB cache nodes.
B. Purchase a Compute Savings Plan to cover the EC2 instances. Purchase Lambda reserved concurrency to cover the expected Lambda usage. Purchase reserved nodes to cover the MemoryDB cache nodes.
C. Purchase a Compute Savings Plan to cover the entire expected cost of the EC2 instances, Lambda functions, and MemoryDB cache nodes.
D. Purchase a Compute Savings Plan to cover the EC2 instances and the MemoryDB cache nodes. Purchase Lambda reserved concurrency to cover the expected Lambda usage.


Suggest Answer: A

---

## 📊 Analysis

### 🔍 Understanding the Problem

A company runs an application across **EC2 instances** (with **continuous, stable load**) and **AWS Lambda functions** (with **varied, unpredictable load**). There is a caching layer using **Amazon MemoryDB for Redis**. The solutions architect must recommend the optimal **cost-saving purchasing strategy** to minimize overall monthly costs.

Key workload characteristics:
- **EC2**: Continuous, stable, predictable → best suited for long-term commitment discounts
- **Lambda**: Varied and unpredictable → needs flexibility in discount model
- **MemoryDB for Redis**: Database/caching service → needs its own specific reservation model
- Goal: Minimize overall monthly cost

### ✅ Correct Option: A — EC2 Instance Savings Plan + Compute Savings Plan (Lambda) + Reserved Nodes (MemoryDB)

**Why it's correct:**

This is the optimal cost-optimization strategy because each service's discount model is matched to its workload pattern:

1. **EC2 Instance Savings Plan for EC2 (Stable Load)**:
   - Provides the **deepest discount** for EC2 usage — up to **72%** off On-Demand pricing
   - Applies to a specific **instance family in a specific region** (e.g., m6i in us-east-1)
   - Ideal for **continuous, stable, predictable** workloads where you know exactly what instance families you'll run
   - More restrictive than Compute Savings Plan, but the trade-off is a **deeper discount**
   - Since EC2 load is "continuous and stable," the company can confidently commit to specific instance families without risking underutilization

2. **Compute Savings Plan for Lambda (Varied/Unpredictable Load)**:
   - Provides **maximum flexibility** — applies across EC2 (any family, any region), Lambda, and Fargate
   - Discount for Lambda is approximately **12–17%** off On-Demand (varies by commitment term)
   - Since Lambda usage is "varied and unpredictable," the flexibility of Compute Savings Plan prevents over-commitment waste
   - Unlike reserved concurrency (which is a **capacity reservation, not a cost-saving mechanism**), Compute Savings Plan directly reduces the Lambda per-invocation cost
   - Covers the **minimum expected consumption** — any spikes beyond the commitment just bill at On-Demand rates

3. **Reserved Nodes for MemoryDB**:
   - MemoryDB is a **database/caching service**, NOT a compute service
   - Neither EC2 Instance Savings Plan nor Compute Savings Plan covers MemoryDB
   - **Reserved nodes** are the correct cost-saving mechanism for MemoryDB, ElastiCache, RDS, Redshift, and other database services
   - Reserved nodes provide up to **50–60% discount** vs. On-Demand pricing for 1-year or 3-year terms

4. **Why This Combination Is Optimal**:
   - Each cost-saving instrument is targeted at the right resource type
   - Stable workloads get the deepest discounts (EC2 Instance SP)
   - Variable workloads get flexible discounts (Compute SP)
   - Database services get their native reservation model (Reserved Nodes)

**Official AWS Documentation**: 
- [AWS Savings Plans Types](https://docs.aws.amazon.com/savingsplans/latest/userguide/what-is-savings-plans.html)
- [MemoryDB Reserved Nodes](https://docs.aws.amazon.com/memorydb/latest/devguide/reserved-nodes.html)

### ❌ Incorrect Options Explanation

**B. Compute Savings Plan (EC2) + Lambda Reserved Concurrency + Reserved Nodes (MemoryDB)**

- **Why incorrect — two critical errors**:
  1. **Compute Savings Plan for EC2 stable load is suboptimal**: Compute Savings Plan offers less discount (typically ~66% max) compared to EC2 Instance Savings Plan (up to 72%). For a known, stable EC2 workload, you're leaving money on the table by choosing the more flexible but less discounted plan.
  2. **Lambda Reserved Concurrency does NOT provide cost savings**: This is a fundamental exam trap. Reserved concurrency:
     - **Guarantees capacity** (ensures N concurrent executions are always available)
     - **Sets an upper bound** (prevents a function from exceeding N concurrent executions)
     - Does **NOT** reduce the per-invocation cost
     - It's a **throttling/reservation mechanism**, not a **billing discount mechanism**
     - Provisioned Concurrency (a separate concept) actually **increases** cost because you're paying for pre-warmed execution environments

**C. Single Compute Savings Plan to cover EC2, Lambda, AND MemoryDB**

- **Why incorrect**: Compute Savings Plan applies **only to compute services**: EC2, Lambda, and Fargate. It does **NOT** cover:
  - Amazon MemoryDB for Redis (database/caching service)
  - Amazon RDS (database service)
  - Amazon ElastiCache (caching service)
  - Amazon Redshift (data warehouse)
  - Any other non-compute AWS service
  - Trying to use Compute Savings Plan for MemoryDB would simply not work — those costs would remain at On-Demand rates. This option indicates a fundamental misunderstanding of Savings Plan scope.

**D. Compute Savings Plan (EC2 + MemoryDB) + Lambda Reserved Concurrency**

- **Why incorrect — compounds multiple errors from B and C**:
  1. Compute Savings Plan does **not** cover MemoryDB (same error as C)
  2. Lambda Reserved Concurrency does **not** provide cost savings (same error as B)
  3. For EC2 with stable load, Compute Savings Plan is less discounted than EC2 Instance Savings Plan
  4. This option saves the least money while implying coverage that doesn't exist

### 🏭 Enterprise Hands-On Knowledge

1. **Savings Plans vs. Reserved Instances — Modern Recommendation**:

   | Feature | Compute Savings Plan | EC2 Instance Savings Plan | Reserved Instances |
   |---|---|---|---|
   | **Applies to** | EC2, Lambda, Fargate | EC2 only | EC2 only |
   | **Flexibility** | Any family, size, region, OS | Specific family, region | Specific instance type, AZ |
   | **Max Discount** | ~66% | ~72% | ~72% |
   | **Can sell on marketplace** | ❌ No | ❌ No | ✅ Yes (third-party) |
   | **Capacity reservation** | ❌ No | ❌ No | ✅ Yes (zonal RI) |
   | **Best for** | Mixed/variable compute | Known, stable EC2 family | Specific AZ capacity guarantee |

   **SAP-C02 Exam Tip**: If the question asks for the **deepest discount** and the workload is **stable/predictable EC2** → EC2 Instance Savings Plan. If **flexibility** or **cross-service coverage** is needed → Compute Savings Plan. If **capacity reservation in a specific AZ** is needed → Zonal Reserved Instances.

2. **MemoryDB vs. ElastiCache — Cost Optimization Context**:
   - Both support **reserved nodes** for cost savings
   - MemoryDB is designed for **durable** in-memory workloads (Redis with multi-AZ transaction log) — it's inherently more expensive than ElastiCache but provides data durability
   - If the question specified ElastiCache instead of MemoryDB, the same reserved node logic applies
   - **Important**: MemoryDB reserved nodes commit to a specific node type in a specific region (similar to RDS reserved instances)

3. **Lambda Cost Components**:
   - **Invocation cost**: Per-request charge (Compute Savings Plan discounts this)
   - **Duration cost**: GB-seconds of memory × execution time (Compute Savings Plan discounts this)
   - **Provisioned Concurrency**: Additional charge for pre-warmed environments (NOT discounted by Savings Plans)
   - **Reserved Concurrency**: FREE — it's just a throttling/guarantee mechanism with no billing impact

4. **Real-World Cost Optimization Strategy for Mixed Workloads**:
   ```
   Step 1: Identify stable compute → EC2 Instance Savings Plan (3-year, all upfront = max discount)
   Step 2: Identify variable compute → Compute Savings Plan (1-year, partial upfront = balanced)
   Step 3: Database services → Reserved Instances/Nodes per service (RDS RI, MemoryDB Reserved Nodes, etc.)
   Step 4: Storage → S3 Lifecycle policies, EBS volume type optimization, CloudFront for egress reduction
   Step 5: Review regularly → AWS Cost Explorer + AWS Budgets + AWS Compute Optimizer
   ```

5. **Calculating Savings Plan Commitment Amount**:
   - Analyze 30-day usage via AWS Cost Explorer "Savings Plans Recommendations"
   - For Lambda: Look at total monthly Lambda On-Demand cost → commit to ~80% of that as Compute Savings Plan (let spikes float at On-Demand)
   - For EC2 stable: Commit to 100% of hourly usage at the instance family level
   - **Medical/Industrial compliance note**: Savings Plans are monetary commitments, not capacity guarantees. If you need guaranteed capacity (e.g., for critical patient systems), zonal Reserved Instances with capacity reservation are still required.

6. **Cost Allocation Tags for Enterprise**: Tag EC2 instances, Lambda functions, and MemoryDB clusters with cost allocation tags (e.g., `CostCenter`, `Application`, `Environment`). AWS Cost Explorer can then break down Savings Plan coverage and utilization by tag, enabling chargeback to departments — critical for large organizations with multiple cost centers.
