A company's compliance audit reveals that some Amazon Elastic Block Store (Amazon EBS) volumes that were created in an AWS account were not encrypted. A solutions architect must implement a solution to encrypt all new EBS volumes at rest.

Which solution will meet this requirement with the LEAST effort?

A. Create an Amazon EventBridge rule to detect the creation of unencrypted EBS volumes. Invoke an AWS Lambda function to delete noncompliant volumes.
B. Use AWS Audit Manager with data encryption.
C. Create an AWS Config rule to detect the creation of a new EBS volume. Encrypt the volume by using AWS Systems Manager Automation.
D. Turn on EBS encryption by default in all AWS Regions.



Suggest Answer: D

---

## 📋 Analysis

### Understanding the Problem

A compliance audit flagged that some EBS volumes in the AWS account are not encrypted at rest. The solutions architect must implement a solution to ensure **all new EBS volumes** are encrypted going forward, with the **least possible effort**. This is fundamentally a prevention problem — the best solution stops unencrypted volumes from ever being created rather than detecting and remediating them after the fact.

### ✅ Correct Option: D

**Turn on EBS encryption by default in all AWS Regions**

This is the correct answer because it requires zero code, zero automation, and zero ongoing maintenance:

1. **Account-Level Setting**: EBS encryption by default is a per-Region account attribute. When enabled, every new EBS volume created in that Region is automatically encrypted at rest — including volumes created via EC2 launch, CloudFormation, Auto Scaling, or the AWS CLI/SDK. There are no exceptions to manage.

2. **Truly "Least Effort"**: This is a one-time configuration change per Region (can be done via AWS Console, CLI `aws ec2 enable-ebs-encryption-by-default`, or CloudFormation). No Lambda functions, no EventBridge rules, no Config rules to maintain. It takes less than 5 minutes to activate across all Regions.

3. **Transparent to Workloads**: Encryption is performed on the underlying storage hardware using AES-256. The encryption/decryption is transparent to EC2 instances — there is no performance penalty (modern EC2 instance types have hardware-accelerated encryption).

4. **Automatic KMS Integration**: By default, volumes are encrypted with the AWS-managed KMS key (`aws/ebs`). You can optionally specify a customer-managed CMK for more control over key rotation policies and access auditing.

5. **Snapshots Inherit Encryption**: Any snapshot created from an encrypted volume is automatically encrypted. Any volume created from an encrypted snapshot is automatically encrypted — maintaining the encryption chain without any additional configuration.

### ❌ Incorrect Options

**Option A — EventBridge Rule + Lambda to Delete Noncompliant Volumes**

- This is a **reactive, destructive** approach. Volumes are created unencrypted, then detected and deleted — causing application disruption and potential data loss.
- Deleting a volume that an instance is actively using would crash the instance.
- Requires writing and maintaining Lambda code, IAM roles, and EventBridge rule patterns.
- Does not prevent the creation of unencrypted volumes — it only cleans up afterward (and poorly, at that).

**Option B — AWS Audit Manager**

- AWS Audit Manager is a **compliance assessment** tool, not an enforcement tool. It helps you gather evidence for audits by automatically collecting reports and logs. It cannot prevent or remediate unencrypted volumes.
- Using Audit Manager for this requirement is like using a security camera to stop a theft — it documents the problem but doesn't prevent it.
- Audit Manager is useful for ongoing compliance reporting once encryption is already enforced, but it is not the solution to the enforcement itself.

**Option C — AWS Config Rule + Systems Manager Automation**

- This is a commonly used **remediation pattern** but requires more effort than Option D:
  - You must enable AWS Config (costs for configuration item recording).
  - You must configure a managed or custom Config rule (e.g., `ec2-ebs-encryption-by-default`).
  - You must set up Systems Manager Automation runbooks and link them as remediation actions.
  - Encrypting an existing unencrypted volume requires creating an encrypted snapshot and recreating the volume — causing downtime.
- While this approach works, it is a reactive remediation, not proactive prevention. It also requires ongoing maintenance and incurs Config + Automation costs.

### 🔧 Hands-On Knowledge

- **How to Enable**: From AWS CLI — `aws ec2 enable-ebs-encryption-by-default --region <region>`. Check status with `aws ec2 get-ebs-encryption-by-default --region <region>`.
- **What Gets Encrypted**: Boot volumes (root), data volumes, and instance store-backed AMIs (though instance store volumes themselves are ephemeral and not encryptable — they inherit encryption from the AMI snapshot).
- **KMS Key Considerations**: Using the AWS-managed `aws/ebs` key is free and sufficient for compliance. Use a customer-managed CMK only when you need key rotation control, cross-account sharing of encrypted snapshots, or detailed KMS API audit logs in CloudTrail.
- **Existing Volumes**: Enabling default encryption does NOT encrypt existing unencrypted volumes. You must handle those separately (create encrypted snapshots → recreate volumes, or use Data Lifecycle Manager to automate the process).
- **Organizational Enforcement**: Use a Service Control Policy (SCP) to deny `ec2:CreateVolume` unless the `encrypted` parameter is `true`, and combine it with a Config rule for detective control at the organizational level.
- **Reference**: [EBS Encryption by Default](https://docs.aws.amazon.com/ebs/latest/userguide/ebs-encryption-by-default.html) | [AWS KMS Developer Guide](https://docs.aws.amazon.com/kms/latest/developerguide/overview.html) | [SCP for EBS Encryption](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps_examples.html#example-ebs-encryption)
