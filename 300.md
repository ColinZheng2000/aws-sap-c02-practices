A company has a web application that securely uploads pictures and videos to an Amazon S3 bucket. The company requires that only authenticated users are allowed to post content. The application generates a presigned URL that is used to upload objects through a browser interface. Most users are reporting slow upload times for objects larger than 100 MB.
What can a Solutions Architect do to improve the performance of these uploads while ensuring only authenticated users are allowed to post content?

A. Set up an Amazon API Gateway with an edge-optimized API endpoint that has a resource as an S3 service proxy. Configure the PUT method for this resource to expose the S3 PutObject operation. Secure the API Gateway using a COGNITO_USER_POOLS authorizer. Have the browser interface use API Gateway instead of the presigned URL to upload objects.
B. Set up an Amazon API Gateway with a regional API endpoint that has a resource as an S3 service proxy. Configure the PUT method for this resource to expose the S3 PutObject operation. Secure the API Gateway using an AWS Lambda authorizer. Have the browser interface use API Gateway instead of the presigned URL to upload API objects.
C. Enable an S3 Transfer Acceleration endpoint on the S3 bucket. Use the endpoint when generating the presigned URL. Have the browser interface upload the objects to this URL using the S3 multipart upload API.
D. Configure an Amazon CloudFront distribution for the destination S3 bucket. Enable PUT and POST methods for the CloudFront cache behavior. Update the CloudFront origin to use an origin access identity (OAI). Give the OAI user s3:PutObject permissions in the bucket policy. Have the browser interface upload objects using the CloudFront distribution.


# AWS SAP-C02 Analysis

## Understanding the Problem

Current architecture:

```text
Authenticated User
      |
Presigned URL
      |
Amazon S3
```

Problem:
- Uploads larger than 100 MB are slow

Requirements:
- Maintain secure authenticated uploads
- Improve upload performance
- Browser-based uploads
- Continue using controlled upload access

Key technical clues:
- Large uploads
- Global internet upload latency likely involved
- Presigned URLs already exist
- Need optimized transfer path

The best AWS-native optimization for large S3 uploads is:
- S3 Transfer Acceleration
- Multipart upload

---

# Correct Option

## C. Enable S3 Transfer Acceleration and use multipart upload with presigned URLs

### Why this is correct

This is the AWS-recommended solution for:
- Large uploads
- Global users
- Browser uploads
- Long-distance network optimization

Architecture:

```text
User
   |
S3 Transfer Acceleration Edge Location
   |
AWS Global Network
   |
S3 Bucket
```

S3 Transfer Acceleration:
- Uses AWS edge locations
- Uses optimized AWS backbone networking
- Reduces internet latency
- Improves upload throughput globally

---

## Multipart upload benefits

Multipart upload:
- Splits large files into chunks
- Uploads parts in parallel
- Retries only failed parts
- Improves resiliency and speed

Very important for:
- Files >100 MB
- Videos
- Large images
- Mobile/international users

---

## Security remains intact

Presigned URLs:
- Continue enforcing authenticated access
- Are time-limited
- Can be generated only for authenticated users

This preserves the existing secure design.

---

## Why this is the best answer

It:
- Directly solves upload performance
- Uses native S3 optimization
- Scales automatically
- Minimizes operational complexity
- Maintains presigned URL security model
- Is highly cost-effective

# Incorrect Options

## A. API Gateway edge-optimized endpoint with Cognito

### Why incorrect

This introduces API Gateway into the upload path.

Problems:
- API Gateway is not optimized for large binary uploads
- Payload size limits exist
- Additional latency introduced
- More expensive
- Less scalable for large media uploads

Most importantly:
API Gateway is not intended to replace direct high-volume S3 uploads.

---

## Key SAP-C02 lesson

For large object uploads:
- Direct-to-S3 architectures are preferred

Avoid:
```text
Client -> API Gateway -> S3
```

Prefer:
```text
Client -> S3
```

---

## B. Regional API Gateway + Lambda authorizer

### Why incorrect

Same fundamental issue as A:
- API Gateway is not ideal for large uploads

Additionally:
- Regional endpoint provides less optimization than edge-optimized
- Lambda authorizer adds unnecessary complexity and latency

Still inferior to direct S3 upload acceleration.

---

## D. CloudFront with PUT/POST uploads to S3

### Why incorrect

CloudFront is primarily optimized for:
- Content delivery
- Downloads
- Caching

Not large upload acceleration.

---

## Important technical issue

CloudFront upload support:
- Exists
- But is not the recommended solution for high-performance S3 uploads

S3 Transfer Acceleration specifically exists for this exact use case.

---

## Additional issue: OAI misuse

Origin Access Identity (OAI):
- Primarily used to restrict GET access from CloudFront to S3
- Not commonly used for authenticated upload workflows

This option mixes concepts incorrectly.

# Key SAP-C02 Concepts Tested

## 1. S3 Transfer Acceleration

Purpose:
- Faster uploads/downloads over long distances

Uses:
- CloudFront edge locations internally
- AWS global backbone

Endpoint format:

```text
bucketname.s3-accelerate.amazonaws.com
```

---

## 2. Multipart upload

Benefits:
- Parallel uploads
- Better throughput
- Retry
- Improved reliability

Recommended for:
- Files larger than 100 MB

Critical SAP-C02 concept.

---

## 3. Presigned URLs

Best practice for browser uploads:
- Temporary delegated upload permissions
- Avoid exposing AWS credentials
- Fine-grained upload control

Very common enterprise architecture.

---

## 4. Direct-to-S3 architecture

Enterprise best practice:

```text
Application Authenticates User
        |
Generate Presigned URL
        |
Browser Uploads Directly to S3
```

Benefits:
- Reduced backend load
- Lower cost
- Better scalability
- Improved throughput

---

# Enterprise Hands-On Knowledge

## Real-world large media upload architecture

Common architecture:

```text
Frontend App
    |
Auth Service
    |
Generate Multipart Presigned URLs
    |
Browser Multipart Upload
    |
S3 Transfer Acceleration
```

Used heavily in:
- Media companies
- Healthcare imaging
- Manufacturing inspection systems
- Video platforms

---

## Multipart upload operational considerations

### Parallelism tuning

Typical optimal:
- 8–32 MB chunks
- 5–10 parallel uploads

Too many parts:
- Increases overhead

Too few:
- Reduces throughput gains

---

## Security best practices

### Restrict presigned URLs

Use:
- Short expiration times
- Content-length restrictions
- MIME type validation

---

## Monitoring uploads

Monitor:
- S3 request metrics
- Transfer acceleration usage
- Multipart upload failures

Useful CloudWatch metrics:
- 4xxErrors
- FirstByteLatency
- BytesUploaded

---

## Cost consideration

Transfer Acceleration:
- Adds additional cost

But justified when:
- Global users
- Large uploads
- High latency regions
- Enterprise UX requirements
