A company has a website that runs on four Amazon EC2 instances that are behind an Application Load Balancer (ALB). When the ALB detects that an EC2 instance is no longer available, an Amazon CloudWatch alarm enters the ALARM state. A member of the company's operations team then manually adds a new EC2 instance behind the ALB.

A solutions architect needs to design a highly available solution that automatically handles the replacement of EC2 instances. The company needs to minimize downtime during the switch to the new solution.

Which set of steps should the solutions architect take to meet these requirements?

A. Delete the existing ALB. Create an Auto Scaling group that is configured to handle the web application traffic. Attach a new launch template to the Auto Scaling group. Create a new ALB. Attach the Auto Scaling group to the new ALB. Attach the existing EC2 instances to the Auto Scaling group.
B. Create an Auto Scaling group that is configured to handle the web application traffic. Attach a new launch template to the Auto Scaling group. Attach the Auto Scaling group to the existing ALAttach the existing EC2 instances to the Auto Scaling group.
C. Delete the existing ALB and the EC2 instances. Create an Auto Scaling group that is configured to handle the web application traffic. Attach a new launch template to the Auto Scaling group. Create a new ALB. Attach the Auto Scaling group to the new ALB. Wait for the Auto Scaling group to launch the minimum number of EC2 instances.
D. Create an Auto Scaling group that is configured to handle the web application traffic. Attach a new launch template to the Auto Scaling group. Attach the Auto Scaling group to the existing ALB. Wait for the existing ALB to register the existing EC2 instances with the Auto Scaling group.

## Analysis

### Understanding the Problem

A company operates a website on four EC2 instances behind an Application Load Balancer (ALB). Currently, when the ALB health check detects an unhealthy instance, a CloudWatch alarm triggers, and an operations team member **manually** adds a replacement EC2 instance behind the ALB. The goal is to **automate the replacement** of failed EC2 instances while **minimizing downtime** during the transition from the manual process to the automated solution. This means the solution must preserve the existing infrastructure (ALB, running EC2 instances) and augment it with automation — not tear everything down and rebuild. The natural fit is Amazon EC2 Auto Scaling, which automatically replaces unhealthy instances and maintains desired capacity.

### Correct Option: B — Create ASG + Attach to Existing ALB + Attach Existing EC2 Instances

**Why B is correct:**

- **Preserves the existing ALB**: No downtime from deleting and recreating the load balancer. The ALB's DNS name, SSL certificates, listener rules, and target group configuration remain intact.
- **Creates an Auto Scaling group** configured to handle the web application traffic. The ASG provides automatic instance replacement: if an EC2 instance fails the ALB health check (or the built-in EC2 health check), the ASG automatically terminates it and launches a replacement.
- **Attaches the existing (running) EC2 instances to the ASG**: This is critical for zero-downtime transition. The existing four EC2 instances become part of the ASG. The ASG's desired capacity is set to match the current instance count (4), so no new instances are unnecessarily launched.
- **Attaches the ASG to the existing ALB** (via the ALB's target group): New instances launched by the ASG are automatically registered with the ALB target group. Unhealthy instances are automatically deregistered.
- **Launch template** ensures all new instances have consistent configuration (AMI, instance type, security groups, user data).
- **Minimal downtime**: The entire transition can be performed without any service interruption. The existing instances continue serving traffic while the ASG is created and associated.

**Official reference**: [Attach EC2 instances to an Auto Scaling group](https://docs.aws.amazon.com/autoscaling/ec2/userguide/attach-instance-asg.html) — "You can attach an existing EC2 instance to an existing Auto Scaling group, or you can attach an existing EC2 instance to a new Auto Scaling group as you create it."

### Incorrect Options

**A — Delete existing ALB → Create ASG → Create new ALB → Attach existing EC2 instances**

- **Deleting the existing ALB causes downtime**: The ALB has a DNS name that clients use to reach the website. Deleting and recreating it means the new ALB gets a different DNS name, requiring a DNS update (which takes time to propagate). During this window, users cannot access the application.
- While attaching existing EC2 instances to the new ASG preserves the compute layer, the unnecessary ALB deletion violates the "minimize downtime" requirement.
- This approach creates a risky change window: DNS propagation can take anywhere from a few seconds to 48 hours depending on TTL settings and client DNS caching behavior.

**C — Delete existing ALB and EC2 instances → Create everything from scratch → Wait for ASG to launch new instances**

- **Complete downtime**: This is the worst option. Deleting all EC2 instances AND the ALB simultaneously means the application is completely unavailable until:
  - The new ASG launches and initializes the minimum number of instances (several minutes for EC2 boot + application startup)
  - The new ALB provisions and its DNS name propagates
- This is a "rip and replace" approach that guarantees extended downtime — the exact opposite of what's required.
- Even if the ASG launches instances quickly, there's a period of zero capacity with no running application.

**D — Create ASG → Attach to existing ALB → Wait for ALB to register existing EC2 instances with ASG**

- **Architectural misunderstanding**: The ALB does **not** register EC2 instances with the ASG. The relationship works in the opposite direction: the ASG registers its instances with the ALB's target group. There is no mechanism for an ALB to "register" instances with an Auto Scaling group.
- This option passive-aggressively avoids the explicit action of attaching the existing EC2 instances to the ASG. The existing instances would remain outside the ASG's management, meaning they wouldn't benefit from automatic replacement if they become unhealthy.
- The existing four instances would continue to rely on the manual replacement process, defeating the purpose of the automation.

### Hands-On Enterprise Knowledge

1. **Transitioning from static EC2 to ASG**: This is a common enterprise migration scenario. The step-by-step process in practice:
   1. Create a launch template from one of the existing healthy EC2 instances (or build a golden AMI)
   2. Create an Auto Scaling group with desired capacity = current instance count (4)
   3. Associate the ASG with the existing ALB's target group
   4. Attach the existing EC2 instances to the ASG using the `AttachInstances` API or Console action
   5. Verify all instances are healthy and serving traffic
   6. (Optional) Enable ASG health checks to replace any instances that don't match the launch template

2. **ASG health checks**: An Auto Scaling group can use both **EC2 health checks** (instance reachability) and **ELB health checks** (ALB target group health). When ELB health checks are enabled, the ASG considers an instance unhealthy if it fails the ALB health check, and automatically replaces it. This is exactly the automation needed to replace the manual ops team intervention.

3. **Lifecycle hooks**: When attaching existing instances to an ASG, consider using lifecycle hooks to run custom scripts (via Systems Manager or Lambda) before an instance enters `InService` state. This can validate that the instance's application is fully initialized before the ALB routes traffic to it.

4. **Cooldown and scaling policies**: When first setting up the ASG, ensure that scaling policies (if any) have appropriate cooldown periods. Without careful configuration, attaching four existing instances might trigger a scale-in event if the ASG's target tracking policy interprets the sudden increase in capacity as over-provisioning.

5. **CloudWatch alarm retention**: The existing CloudWatch alarm (ALARM state when instance unavailable) can be retired after the ASG is in place, as the ASG automatically handles replacement. However, it's good practice to retain a CloudWatch alarm on `GroupTotalInstances < 4` as a safety net to alert if the ASG itself experiences issues.

6. **Rollback strategy**: Before implementing, take a manual snapshot (AMI) of each existing EC2 instance. If the ASG transition causes unexpected issues, you can manually re-launch the instances from the AMIs and revert to the manual process while troubleshooting.
