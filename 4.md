- D. Enable Aurora Scaling for Aurora writers. Use a Network Load Balancer with the least outstanding requests routing algorithm and sticky sessions enabled.

## AWS SAP-C02 Exam Question Analysis

### Understanding the Problem

The company is migrating a **two-tier web-based application** from an on-premises data center to AWS. The application is **stateful** and consists of a single server. It uses a **PostgreSQL database**. The goal is to design a scalable solution using **Amazon Aurora PostgreSQL**, **Amazon EC2 Auto Scaling**, and **Elastic Load Balancing** that provides a **consistent user experience** as the user base grows.

* **Application is stateful:** This is the critical constraint. A stateful application means that a user's session data is stored on the specific server they initially connected to. If subsequent requests from the same user are routed to a different server, the user's session will be lost, resulting in an inconsistent and frustrating experience.
* **Need for Consistency:** This directly mandates the use of **sticky sessions** (or session affinity) on the load balancer to ensure a user is always returned to the same application server.
* **Need for Scaling:** Both the application and database tiers must scale.
    * **Application Tier:** **EC2 Auto Scaling** is already planned, meaning multiple EC2 instances will run the application. An **Elastic Load Balancer (ELB)** will distribute traffic to these instances.
    * **Database Tier:** **Amazon Aurora PostgreSQL** is the service, and it needs to be configured for scaling. Read capacity is usually scaled using **Aurora Replicas**.

***

### Services Explanation

| AWS Service | Description | Relevance to Solution |
| :--- | :--- | :--- |
| **Amazon Aurora PostgreSQL** | A relational database service compatible with PostgreSQL, offering high performance, availability, and scalability. It separates compute and storage. | The database solution, which needs to scale read capacity. |
| **Aurora Auto Scaling (for Replicas)** | Automatically adds or removes **Aurora Replicas** (read-only copies of data) to handle fluctuations in read traffic. | Provides automated read-scaling for the database tier. |
| **Amazon EC2 Auto Scaling** | Automatically adds or removes **Amazon EC2 instances** (application servers) based on demand. | Provides automated scaling for the application tier. |
| **Elastic Load Balancing (ELB)** | Distributes incoming application traffic across multiple targets, such as EC2 instances. There are three main types: Application Load Balancer (ALB), Network Load Balancer (NLB), and Gateway Load Balancer (GLB). | Distributes traffic to the application tier and, critically, must support **sticky sessions**. |
| **Application Load Balancer (ALB)** | Best suited for load balancing of HTTP and HTTPS traffic (Layer 7). It provides advanced request routing and supports **sticky sessions** based on a cookie. | The ideal choice for a web-based, stateful application. |
| **Network Load Balancer (NLB)** | Best suited for extreme performance and static IP addresses (Layer 4 - TCP/UDP). It supports **sticky sessions** based on client IP addresses. | A less flexible option for HTTP/web traffic; IP-based stickiness can be unreliable if users are behind a shared NAT or proxy. |
| **Sticky Sessions (Session Affinity)** | A load balancer feature that binds a user's session to a specific EC2 instance, ensuring all requests from that user during the session are sent to the same server. **Essential for stateful applications.** | The mechanism to ensure a "consistent user experience." |
| **Round Robin Routing** | A simple load balancing algorithm that distributes requests sequentially among the available instances. | A common and simple distribution method. |

***

### Correct Option(s) Explanation

The correct option is **C. Enable Aurora Auto Scaling for Aurora Replicas. Use an Application Load Balancer with the round robin routing and sticky sessions enabled.**

1.  **Application Tier (Consistent User Experience/Scaling):**
    * **Application Load Balancer (ALB):** Since this is a "two-tier **web-based** application," an ALB (Layer 7) is the appropriate choice. ALB is explicitly designed for HTTP/HTTPS traffic and provides rich routing features.
    * **Sticky Sessions Enabled:** The application is **stateful**, so **sticky sessions** *must* be enabled on the load balancer to ensure the user is routed to the same application server throughout their session. ALB's cookie-based sticky sessions are the most robust method for HTTP traffic.
    * **Round Robin Routing:** This is a common and acceptable algorithm for initial request distribution when sticky sessions are enabled.

2.  **Database Tier (Scaling):**
    * **Enable Aurora Auto Scaling for Aurora Replicas:** Aurora Replicas handle the **read** load. As the user base grows, read operations (user browsing/querying) will increase significantly. Auto Scaling Aurora Replicas is the AWS-recommended way to automatically scale the database **read capacity** to handle this growth (as per the [Amazon Aurora User Guide](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Overview.Replicas.html)).

***

### Incorrect Options Explanation

* **A. Enable Aurora Auto Scaling for Aurora Replicas. Use a Network Load Balancer with the least outstanding requests routing algorithm and sticky sessions enabled.**
    * **Incorrect Load Balancer:** While NLB supports stickiness (based on client IP), ALB is the standard, more robust choice for a web-based, stateful application because of its cookie-based sticky sessions. Also, the "least outstanding requests" routing algorithm is more common for ALBs than NLBs, as NLB's primary routing is a flow hash.

* **B. Enable Aurora Auto Scaling for Aurora writers. Use an Application Load Balancer with the round robin routing algorithm and sticky sessions enabled.**
    * **Incorrect Aurora Component:** **Aurora writers** are the primary instance that handles all **write** operations. Scaling the *writer* would mean scaling the compute for the primary instance, which is generally done via instance class upgrade (vertical scaling) or promotion of a replica, not through Aurora Auto Scaling for horizontal scaling. **Aurora Auto Scaling is designed to scale *Replicas* (read capacity)** to handle a growing user base, which generates high read traffic.

* **D. Enable Aurora Scaling for Aurora writers. Use a Network Load Balancer with the least outstanding requests routing algorithm and sticky sessions enabled.**
    * **Incorrect Aurora Component:** As explained for option B, Aurora Auto Scaling should target **Replicas** for read-scaling, not the writer.
    * **Incorrect Load Balancer:** As explained for option A, NLB is less suitable than ALB for a web-based, stateful application requiring sticky sessions.
