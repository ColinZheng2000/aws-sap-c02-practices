A company has set up its entire infrastructure on AWS. The company uses Amazon EC2 instances to host its ecommerce website and uses Amazon S3 to store static data. Three engineers at the company handle the cloud administration and development through one AWS account. Occasionally, an engineer alters an EC2 security group configuration of another engineer and causes noncompliance issues in the environment.

A solutions architect must set up a system that tracks changes that the engineers make. The system must send alerts when the engineers make noncompliant changes to the security settings for the EC2 instances.

What is the FASTEST way for the solutions architect to meet these requirements?

A. Set up AWS Organizations for the company. Apply SCPs to govern and track noncompliant security group changes that are made to the AWS account.
B. Enable AWS CloudTrail to capture the changes to EC2 security groups. Enable Amazon CloudWatch rules to provide alerts when noncompliant security settings are detected.
C. Enable SCPs on the AWS account to provide alerts when noncompliant security group changes are made to the environment.
D. Enable AWS Config on the EC2 security groups to track any noncompliant changes. Send the changes as alerts through an Amazon Simple Notification Service (Amazon SNS) topic.

Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company with three engineers sharing one AWS account needs to **track changes** to EC2 security groups and receive **alerts** when noncompliant modifications occur. The keyword is **FASTEST** — the solution must minimize setup time while delivering compliance tracking and notification capabilities.

### Correct Option: D

**Enable AWS Config on the EC2 security groups to track any noncompliant changes. Send the changes as alerts through an Amazon Simple Notification Service (Amazon SNS) topic.**

- **AWS Config** is a fully managed service that continuously monitors and records AWS resource configurations. It provides a timeline of configuration changes (who made the change, when, and what changed) out of the box.
- **AWS Config Rules** include pre-built managed rules specifically for EC2 security groups, such as `restricted-ssh`, `restricted-common-ports`, and `ec2-security-group-attached-to-eni`. These rules instantly evaluate whether a security group configuration is compliant.
- When AWS Config detects a noncompliant change, it can trigger an **Amazon SNS** notification natively — no custom Lambda function or complex EventBridge pattern is required.
- AWS Config can be enabled in a single AWS account with just a few clicks in the console, making it the **fastest** path to operational compliance monitoring.
- The service automatically records configuration history, so you have an audit trail of every change made by each engineer.

### Incorrect Options

**A: Set up AWS Organizations + SCPs** — AWS Organizations requires creating a new organization structure, inviting accounts, and configuring Service Control Policies (SCPs). SCPs are **preventive** controls (guardrails), not detective controls — they cannot send alerts. They also don't track changes. Setting up Organizations from scratch is the opposite of "fastest."

**B: CloudTrail + CloudWatch rules** — While CloudTrail captures API calls (including `AuthorizeSecurityGroupIngress`), CloudTrail itself has **no concept of compliance**. To detect "noncompliant" changes, you would need to write custom CloudWatch Events/EventBridge rules with Lambda functions that evaluate each change against your compliance standards. This requires development effort, testing, and maintenance — far slower than enabling AWS Config with pre-built rules.

**C: SCPs on the AWS account** — SCPs require AWS Organizations, which the company does not have. Furthermore, SCPs are preventive (they block actions) and do **not provide alerting capabilities**. They also cannot track what changes were made — they only prevent specific API calls from being executed.

### Hands-On Knowledge

1. **AWS Config setup in practice**: Go to AWS Config console → Settings → Choose "Record all resources supported in this region" → Set up an S3 bucket for configuration history → Create an SNS topic for notifications. This takes under 5 minutes.
2. **Managed rules for security groups**: Key rules include `restricted-ssh` (checks if SSH is open to 0.0.0.0/0), `ec2-security-group-attached-to-eni-periodic` (checks unused groups), `vpc-default-security-group-closed` (checks default SG rules).
3. **AWS Config vs CloudTrail mindset**: AWS Config tells you *what changed and is it compliant?* CloudTrail tells you *who did what API call?* They complement each other — use both in production.
4. **SCP limitations**: SCPs work at the organization/OU/account level and only restrict actions; they never grant permissions. They require AWS Organizations and affect all principals in the scope.
5. **Real-world pattern**: Production environments typically combine AWS Config (compliance detection) + SNS (alerting) + Lambda (auto-remediation) for a complete security posture.
