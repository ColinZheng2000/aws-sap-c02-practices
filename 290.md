A company is managing many AWS accounts by using an organization in AWS Organizations. Different business units in the company run applications on Amazon EC2 instances. All the EC2 instances must have a BusinessUnit tag so that the company can track the cost for each business unit.

A recent audit revealed that some instances were missing this tag. The company manually added the missing tag to the instances.

What should a solutions architect do to enforce the tagging requirement in the future?

A. Enable tag policies in the organization. Create a tag policy for the BusinessUnit tag. Ensure that compliance with tag key capitalization is turned off. Implement the tag policy for the ec2:instance resource type. Attach the tag policy to the root of the organization.
B. Enable tag policies in the organization. Create a tag policy for the BusinessUnit tag. Ensure that compliance with tag key capitalization is turned on. Implement the tag policy for the ec2:instance resource type. Attach the tag policy to the organization's management account.
C. Create an SCP and attach the SCP to the root of the organization. Include the following statement in the SCP:
```json
{
  "Sid": "DenyEC2Creation",
  "Effect": "Deny",
  "Action": [
    "ec2:RunInstances"
  ],
  "Resource": [
    "arn:aws:ec2:*:*:instance/*"
  ],
 "Condition": {
    "Null": {
      "aws:RequestTag/BusinessUnit": "true"
    }
  }
}
```

D. Create an SCP and attach the SCP to the organization's management account. Include the following statement in the SCP:
```json

{
  "Sid": "DenyEC2Creation",
  "Effect": "Deny",
  "Action": [
    "e2:RunInstances"
  ],
  "Resource": [
    "arn:aws:ec2:*:*:instance/*"
  ],
  "Condition": {
    "Null": {
      "aws:RequestTag/BusinessUnit": "false"
    }
  }
}
```


# AWS SAP-C02 Analysis - Enforcing Mandatory EC2 Tags Across AWS Organizations

---

# Understanding the Problem

The company:

- Uses AWS Organizations with many AWS accounts
- Needs all EC2 instances to contain a mandatory `BusinessUnit` tag
- Wants enforcement going forward, not just reporting
- Needs centralized governance across the organization

Key distinction:

- Detecting missing tags ≠ enforcing tags
- Tag Policies help standardize and report compliance
 SCPs can actively prevent resource creation without required tags

The requirement is:
> Prevent EC2 instances from being launched unless the `BusinessUnit` tag is provided.

---

# Correct Option

## ✅ C — Use an SCP to deny `ec2:RunInstances` if the `BusinessUnit` tag is missing

### Why this is correct

This option uses:

- AWS Organizations Service Control Policies (SCPs)
- Explicit deny logic
- `aws:RequestTag/BusinessUnit`
- Condition operator `Null`

This enforces governance at the organization level.

### Important logic

```json
"Condition": {
  "Null": {
    "aws:RequestTag/BusinessUnit": "true"
  }
}
```

Meaning:

- If the request does NOT contain the `BusinessUnit` tag
- The condition evaluates to TRUE
- The action is denied

Therefore:

- Launch succeeds only when the tag is supplied
- Preventive control is implemented
- Works across all member accounts attached to the SCP scope

### Why attach to the root?

Attaching the SCP to the root ensures:

- All accounts inherit the policy
- Organization-wide governance
- Consistent enforcement

This is the standard enterprise governance model.

---

# Incorrect Options

---

## ❌ A — Tag Policy attached to root

### Why incorrect

Tag Policies:

- Standardize tags
- Help with compliance visibility
- Can enforce capitalization and allowed values

BUT:

- They do NOT block resource creation
- They are not preventive controls

Even with a tag policy:

- Users can still launch EC2 instances without tags
- Noncompliance is only reported

### Additional issue

The option disables capitalization compliance:

```text
Ensure that compliance with tag key capitalization is turned off
```

This weakens governance consistency.

### Key exam point

SAP-C02 frequently tests:

| Feature | Preventive? |
|---|---|
| Tag Policies | ❌ No |
| SCPs | ✅ Yes |

---

## ❌ B — Tag Policy attached to management account

### Multiple problems

### 1. Tag Policies still do not enforce creation requirements

Same issue as Option A.

### 2. Wrong attachment target

Tag policies should be attached to:

- Root
- OU
- Member

Not specifically to only the management account if organization-wide enforcement is desired.

### 3. Capitalization compliance alone is insufficient

Even if capitalization compliance is enabled:

- Missing tags are still possible
- EC2 creation is not blocked

---

## ❌ D — SCP attached to management account with wrong condition

### Problem 1 — Wrong attachment scope

Attaching to only the management account does not protect member accounts.

The organization root or OUs should be targeted.

### Problem 2 — Incorrect Null condition

```json
"Null": {
  "aws:RequestTag/BusinessUnit": "false"
```

This means:

- Deny when the tag EXISTS

Which is opposite of the requirement.

Correct logic:

```json
"Null": {
  "aws:RequestTag/BusinessUnit": "true"
}
```

Meaning deny when the tag is absent.

# Key AWS Concepts

## AWS Tag Policies

### Purpose

- Standardize tagging
- Detect noncompliance
- Enforce conventions
- Enforce capitalization
- Restrict allowed values

### Limitation

Cannot block API calls.

### Common enterprise use

- Governance dashboards
- Cost allocation consistency
- Reporting compliance

---

## Service Control Policies (SCPs)

### Purpose

Organization-wide permission guardrails.

### Important characteristics

- Apply to accounts/OUs/root
- Restrict maximum permissions
- Explicit deny overrides allow
- Ideal for preventive governance

### Common enterprise usage

- Require encryption
- Restrict regions
- Prevent disabling logging
- Enforce mandatory tags

# Hands-on Enterprise Knowledge

## Real-world tagging strategy

Large enterprises commonly require:

| Tag | Purpose |
|---|---|
| BusinessUnit | Cost allocation |
| Environment | Dev/Test/Prod |
| Owner | Accountability |
| Application | Service mapping |
| Compliance | Regulatory classification |

---

## Recommended architecture

### Combine multiple governance services

| Service | Role |
|---|---|
| SCP | Prevent missing tags |
| Tag Policies | Standardize values |
| AWS Config | Detect drift |
| Cost Explorer | Allocate costs |
| Control Tower | Baseline governance |

This layered governance model is common in regulated industries such as:

- Healthcare
- Manufacturing
- Pharmaceuticals
- Financial services

---

## Example production SCP pattern

Many enterprises expand conditions like:

```json
"Condition": {
  "Null": {
    "aws:RequestTag/BusinessUnit": "true",
    "aws:RequestTag/Environment": "true",
    "aws:RequestTag/Owner": "true"
  }
}
```

This ensures all mandatory tags exist during provisioning.

---

## Operational caveat

Some AWS services launch EC2 instances indirectly:

- Auto Scaling
- EKS
- EMR
- ECS Capacity Providers

You must ensure:

- Launch templates include required tags
- Automation pipelines pass tags correctly
- IaC tools (CloudFormation/Terraform) enforce tagging

Otherwise SCPs can unexpectedly block deployments.

# Exam Tips (SAP-C02)

## Remember this distinction

### Tag Policies
- Compliance visibility
- Standardization
- Reporting

### SCPs
- Preventive enforcement
- Permission guardrails

If the exam says:

- "Prevent"
- "Block"
- "Enforce"

The answer is usually SCP, IAM condition keys, or preventive controls.
