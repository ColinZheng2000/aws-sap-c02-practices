A company uses an AWS CodeCommit repository. The company must store a backup copy of the data that is in the repository in a second AWS Region.

Which solution will meet these requirements?

A. Configure AWS Elastic Disaster Recovery to replicate the CodeCommit repository data to the second Region.
B. Use AWS Backup to back up the CodeCommit repository on an hourly schedule. Create a cross-Region copy in the second Region.
C. Create an Amazon EventBridge rule to invoke AWS CodeBuild when the company pushes code to the repository. Use CodeBuild to clone the repository. Create a .zip file of the content. Copy the file to an S3 bucket in the second Region.
D. Create an AWS Step Functions workflow on an hourly schedule to take a snapshot of the CodeCommit repository. Configure the workflow to copy the snapshot to an S3 bucket in the second Region

Suggested Answer: C

---

## 📋 Analysis

### 1. Understanding the Problem

The core requirement is to **maintain a backup copy of an AWS CodeCommit repository in a second AWS Region**. CodeCommit is a fully managed source control service that hosts Git repositories. Unlike databases or block storage volumes, CodeCommit does not offer native, built-in cross-region replication or snapshot capabilities. Therefore, the solution must be architected using a combination of AWS services to clone the repository content and transfer it to another Region.

**Key considerations:**
- CodeCommit repositories are Git-based; the backup strategy must preserve the full repository history (all branches, tags, commits).
- The solution must operate across Regions, meaning data transfer and cross-region API calls are involved.
- The backup should be automated and triggered upon code changes or on a schedule.

---

### 2. Options Analysis

#### ✅ **Option C (CORRECT): EventBridge + CodeBuild → Clone → Zip → S3 (Cross-Region)**

This option leverages an **event-driven, serverless architecture**:

1. **Amazon EventBridge** detects a CodeCommit event (e.g., `referenceCreated` or `referenceUpdated` when code is pushed) and triggers an AWS CodeBuild project.
2. **AWS CodeBuild** clones the entire CodeCommit repository (including all branches and history via `git clone --mirror`), creates a compressed `.zip` archive of the repository content, and uses the AWS CLI/SDK to copy the artifact to an **Amazon S3 bucket in the second Region**.
3. S3 provides durable, cross-region storage for the backup artifact. Optionally, S3 Cross-Region Replication (CRR) can be added for further redundancy.

**Why this works:**
- CodeCommit natively integrates with EventBridge, emitting events for repository state changes.
- CodeBuild is a fully managed build service that can run Git commands and shell scripts — capable of cloning a repo and producing a ZIP artifact.
- Cross-region S3 copy is straightforward using `aws s3 cp` with the target bucket in another Region.
- The solution is event-driven (near real-time) and serverless (no EC2 instances to manage).

#### ❌ **Option A (INCORRECT): AWS Elastic Disaster Recovery (DRS)**

- **AWS Elastic Disaster Recovery (DRS)** is designed for **block-level replication of entire servers** (physical, virtual, or cloud-based) to AWS for disaster recovery purposes. It replicates operating systems, applications, and data at the volume level.
- DRS does **not** support replicating CodeCommit repositories. CodeCommit is a managed Git service, not a server or disk volume. DRS operates at the infrastructure layer (EC2/EBS), not the application layer (Git repos).
- **Misconception alert:** DRS is often confused with services that replicate data; always verify that the source resource type is supported by the replication service.

#### ❌ **Option B (INCORRECT): AWS Backup with Cross-Region Copy**

- **AWS Backup** is a centralized backup service that supports a wide range of AWS services — **but CodeCommit is NOT among the supported resources**.
- As of this writing, AWS Backup supports: EC2, EBS, RDS, Aurora, DynamoDB, EFS, S3, DocumentDB, Neptune, FSx, Storage Gateway, and a few others. CodeCommit repositories are not backup-able via AWS Backup.
- **Exam tip:** The SAP-C02 exam frequently tests knowledge of *which services integrate with AWS Backup*. CodeCommit, CloudFormation, and API Gateway are common "trap" answers — they are NOT supported by AWS Backup.

#### ❌ **Option D (INCORRECT): Step Functions Hourly Schedule → "Snapshot" CodeCommit → Copy to S3**

- **CodeCommit does not have a "snapshot" API or feature.** Unlike EBS volumes, RDS databases, or Redshift clusters that provide native snapshot capabilities, CodeCommit repositories are Git repositories — they do not offer snapshot operations.
- Step Functions cannot directly "take a snapshot" of a CodeCommit repo because no such API exists. You would need to invoke a Lambda function or CodeBuild project to perform the actual clone-and-archive logic.
- While Step Functions can orchestrate such a workflow on a schedule (via EventBridge Scheduler), the phrasing "take a snapshot of the CodeCommit repository" reveals the option's invalidity — there is no snapshot mechanism for CodeCommit.

### 3. Hands-On Knowledge

#### How to implement the correct solution (Option C) in practice:

**Step 1: Create an S3 bucket in the target Region**
```bash
aws s3 mb s3://codecommit-backup-bucket --region us-west-2
```

**Step 2: Create a CodeBuild project (`buildspec.yml`)**
```yaml
version: 0.2
phases:
  build:
    commands:
      - echo "Cloning CodeCommit repository..."
      - git clone --mirror https://git-codecommit.$AWS_REGION.amazonaws.com/v1/repos/$REPO_NAME /tmp/repo
      - cd /tmp/repo
      - TIMESTAMP=$(date +%Y%m%d%H%M%S)
      - zip -r /tmp/repo-backup-$TIMESTAMP.zip .
      - aws s3 cp /tmp/repo-backup-$TIMESTAMP.zip s3://codecommit-backup-bucket/ --region us-west-2
```
> **Note:** Ensure the CodeBuild service role has permissions for `codecommit:GitPull` and `s3:PutObject` on the target bucket.

**Step 3: Create an EventBridge rule**
- **Event source:** AWS CodeCommit
- **Event type:** `CodeCommit Repository State Change`
- **Detail type:** `referenceCreated` or `referenceUpdated`
- **Target:** The CodeBuild project created above

**Step 4: IAM Permissions Required**
| Service | Action | Resource |
|---------|--------|----------|
| CodeCommit | `codecommit:GitPull` | Repository ARN |
| S3 | `s3:PutObject` | Backup bucket ARN |
| CodeBuild | `codebuild:StartBuild` | CodeBuild project ARN |
| EventBridge | `events:PutRule`, `events:PutTargets` | Event bus |

**Alternative approach – Using AWS CodePipeline:**
If you prefer a managed CI/CD approach, CodePipeline can also be configured with CodeCommit as the source, and a custom action to zip and push to S3 in another Region. However, EventBridge + CodeBuild is more lightweight and cost-effective for a pure backup use case.

**Important Gotcha:**
- CodeCommit is **not multi-Region**. A repository exists in a single Region. There is no automatic cross-region replication for CodeCommit repositories. You must build a custom solution.
- `git clone --mirror` is preferred over `git clone` because it preserves all branches, tags, and refs — essential for a true backup.
