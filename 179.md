A company wants to containerize a multi-tier web application and move the application from an on-premises data center to AWS. The application includes web. application, and database tiers. The company needs to make the application fault tolerant and scalable. Some frequently accessed data must always be available across application servers. Frontend web servers need session persistence and must scale to meet increases in traffic.

Which solution will meet these requirements with the LEAST ongoing operational overhead?

A. Run the application on Amazon Elastic Container Service (Amazon ECS) on AWS Fargate. Use Amazon Elastic File System (Amazon EFS) for data that is frequently accessed between the web and application tiers. Store the frontend web server session data in Amazon Simple Queue Service (Amazon SQS).
B. Run the application on Amazon Elastic Container Service (Amazon ECS) on Amazon EC2. Use Amazon ElastiCache for Redis to cache frontend web server session data. Use Amazon Elastic Block Store (Amazon EBS) with Multi-Attach on EC2 instances that are distributed across multiple Availability Zones.
C. Run the application on Amazon Elastic Kubernetes Service (Amazon EKS). Configure Amazon EKS to use managed node groups. Use ReplicaSets to run the web servers and applications. Create an Amazon Elastic File System (Amazon EFS) file system. Mount the EFS file system across all EKS pods to store frontend web server session data.
D. Deploy the application on Amazon Elastic Kubernetes Service (Amazon EKS). Configure Amazon EKS to use managed node groups. Run the web servers and application as Kubernetes deployments in the EKS cluster. Store the frontend web server session data in an Amazon DynamoDB table. Create an Amazon Elastic File System (Amazon EFS) volume that all applications will mount at the time of deployment.


Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company is migrating an on-premises multi-tier web application (web, application, database) to AWS using containers. The architecture must be **fault tolerant** and **scalable**. Two specific technical requirements stand out: (1) frequently accessed data must be **always available across application servers** (shared storage), and (2) frontend web servers need **session persistence** (sticky sessions) while scaling. The overarching constraint is **LEAST ongoing operational overhead** — favor managed services.

### Correct Option: D

**Deploy the application on Amazon Elastic Kubernetes Service (Amazon EKS). Configure Amazon EKS to use managed node groups. Run the web servers and application as Kubernetes deployments in the EKS cluster. Store the frontend web server session data in an Amazon DynamoDB table. Create an Amazon Elastic File System (Amazon EFS) volume that all applications will mount at the time of deployment.**

- **Amazon DynamoDB for session state**: DynamoDB is a fully managed, serverless NoSQL key-value database with single-digit millisecond latency. It automatically replicates across multiple Availability Zones, providing high availability for session data. Using DynamoDB for session storage is a well-established pattern (AWS provides official session handlers for PHP, Java, etc. that use DynamoDB as the backing store). Sessions are key-value by nature — a perfect fit.
- **Amazon EFS for shared data**: EFS provides a POSIX-compliant, fully managed NFS file system that can be mounted concurrently by thousands of EC2 instances or EKS pods across multiple Availability Zones. This satisfies "frequently accessed data always available across application servers." EFS automatically scales storage capacity up and down as files are added or removed.
- **EKS managed node groups**: AWS handles the provisioning, lifecycle management, and patching of worker nodes. This reduces operational overhead compared to self-managed node groups or ECS on EC2.
- **Kubernetes Deployments**: Provide declarative rolling updates, self-healing (auto-restart failed pods), and horizontal pod autoscaling (HPA) — all key for scalability and fault tolerance.

### Incorrect Options

**A: ECS Fargate + EFS + SQS for sessions** — Fargate excels at minimizing operational overhead (serverless compute), which is conceptually aligned with the question. However, **Amazon SQS is a message queue**, not a session store. SQS messages are designed for asynchronous decoupling, not for fast key-value lookups with session semantics (TTL, concurrent reads, etc.). This architectural mismatch makes the option invalid regardless of Fargate's operational benefits.

**B: ECS EC2 + ElastiCache Redis + EBS Multi-Attach** — Amazon EBS Multi-Attach is limited to a **single Availability Zone** and supports only Provisioned IOPS (io1/io2) volumes. The question requires fault tolerance, which implies multi-AZ deployment. EBS Multi-Attach cannot span AZs. Furthermore, ECS on EC2 requires managing the underlying EC2 instances (patching, scaling, AMI management), which incurs higher operational overhead than managed options.

**C: EKS + ReplicaSets + EFS for sessions** — Using ReplicaSets directly (rather than Deployments) is a Kubernetes anti-pattern for stateless web workloads; Deployments provide rolling updates and revision history. Using EFS for session data, while technically possible, has significant drawbacks: filesystem-based session storage introduces file locking contention, lacks TTL-based expiration, and is orders of magnitude slower than DynamoDB for key-value lookups.

### Hands-On Knowledge

1. **DynamoDB as Session Store**: AWS provides pre-built session handlers (e.g., `dynamodb-session-handler` for Tomcat, `connect-dynamodb` for Express.js). Key design: use the session ID as the partition key, set DynamoDB TTL on items for automatic session expiry, and enable DAX if you need microsecond latency.
2. **EFS CSI Driver for EKS**: Install the Amazon EFS CSI Driver via EKS add-ons. Create a `StorageClass` with `provisioner: efs.csi.aws.com`, then a `PersistentVolumeClaim`. Pods reference the PVC in their volume mounts. EFS supports `ReadWriteMany` access mode — critical for shared data across pods.
3. **Session stickiness alternative**: If using Application Load Balancer (ALB) with EKS, enable `stickiness.enabled: true` on the target group via the AWS Load Balancer Controller. However, server-side session storage (DynamoDB) is preferred because it survives pod restarts and avoids the "lost session" problem when a pod is replaced.
4. **EKS operational overhead**: Managed node groups handle AMI updates and node termination, but you still need to configure cluster autoscaler, manage IAM roles for service accounts (IRSA), and apply security patches. For truly minimal overhead, consider ECS Fargate or EKS Fargate.
5. **Cost optimization**: EFS costs approximately $0.30/GB-month (Standard) plus throughput charges. DynamoDB on-demand pricing charges per read/write request unit. For high-traffic session workloads, consider DynamoDB provisioned capacity with auto scaling to control costs.
