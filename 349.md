A company uses a mobile app on AWS to run online contests. The company selects a winner at random at the end of each contest. The contests run for variable lengths of time. The company does not need to retain any data from a contest after the contest is finished.

The company uses custom code that is hosted on Amazon EC2 instances to process the contest data and select a winner. The EC2 instances run behind an Application Load Balancer and store contest entries on Amazon RDS DB instances. The company must design a new architecture to reduce the cost of running the contests.

Which solution will meet these requirements MOST cost-effectively?

A. Migrate storage of the contest entries to Amazon DynamoDB. Create a DynamoDB Accelerator (DAX) cluster. Rewrite the code to run as Amazon Elastic Container Service (Amazon ECS) containers that use the Fargate launch type. At the end of the contest, delete the DynamoDB table.
B. Migrate the storage of the contest entries to Amazon Redshift. Rewrite the code as AWS Lambda functions. At the end of the contest, delete the Redshift cluster.
C. Add an Amazon ElastiCache for Redis cluster in front of the RDS DB instances to cache the contest entries. Rewrite the code to run as Amazon Elastic Container Service (Amazon ECS) containers that use the Fargate launch type. Set the ElastiCache TTL attribute on each entry to expire each entry at the end of the contest.
D. Migrate the storage of the contest entries to Amazon DynamoDB. Rewrite the code as AWS Lambda functions. Set the DynamoDB TTL attribute on each entry to expire each entry at the end of the contest.



# AWS SAP-C02 Question Analysis — Q349

## 📋 Analysis

### 1. Understanding the Problem

The workload is an online contest platform with these characteristics:

- Contest duration is unpredictable
- Traffic can fluctuate significantly
- Contest data becomes useless after the contest ends
- Existing architecture uses:
  - EC2 for compute
  - ALB for traffic distribution
  - RDS for storage
- The company wants maximum cost reduction

The key AWS architecture principle here is:
> Avoid always-running infrastructure for temporary and bursty workloads.

Important requirement clues:

- “Does not need to retain data”
  → Strong indicator for TTL-based deletion

- “Variable lengths of time”
  → Indicates unpredictable scaling demand

- “Reduce cost”
  → Strong preference toward serverless and pay-per-use services

This is fundamentally a migration from:
- Persistent infrastructure
to
- Event-driven serverless architecture

- --

### 2. Options Analysis

#### ✅ Option D (CORRECT): DynamoDB + Lambda + DynamoDB TTL

This is the most cost-effective and operationally optimized solution.

Why this architecture is correct:

##### Serverless Compute Minimizes Idle Cost

AWS Lambda only charges:
- Per invocation
- Per execution duration

There are:
- No idle EC2 costs
- No container runtime costs
- No ALB costs
- No server management overhead

This perfectly fits intermittent contest workloads.

##### DynamoDB Fits Bursty Mobile Traffic

DynamoDB provides:
- Automatic scaling
- Extremely high write throughput
- Low operational overhead
- Pay-per-request pricing

Mobile contest systems often generate:
- Sudden spikes
- Massive concurrent writes
- Short-lived datasets

DynamoDB is purpose-built for this access pattern.

##### TTL Automatically Removes Contest Data

DynamoDB TTL:
- Automatically expires records
- Eliminates cleanup jobs
- Reduces storage costs
- Matches the temporary-data requirement exactly

This is one of the strongest requirement alignments in the question.

##### Operational Simplicity

The new architecture removes:
- EC2 fleet management
- Patch management
- Auto Scaling Groups
- RDS maintenance
- ALB operational cost

Enterprise impact:
- Lower TCO
- Smaller operations team burden
- Better scaling characteristics

#### ❌ Option A (INCORRECT): DynamoDB + DAX + ECS Fargate

This solution improves scalability but is not the MOST cost-effective.

##### DAX Is Unnecessary

DynamoDB Accelerator (DAX):
- Is designed for microsecond latency reads
- Benefits read-heavy workloads

Contest entry systems are usually:
- Write-heavy
- Append-oriented
- Not latency-sensitive at microsecond level

Adding DAX introduces:
- Additional always-running nodes
- More cost
- More operational complexity

##### ECS Fargate Costs More Than Lambda

Fargate still charges while containers run.

Compared to Lambda:
- Less cost-efficient for intermittent workloads
- Requires task lifecycle management
- Typically higher baseline cost

##### Overengineering Problem

This option introduces:
- Containers
- DAX caching layer
- Additional orchestration complexity

without solving a real business need.

SAP-C02 exam lesson:
> If serverless Lambda can handle the workload, it is usually more cost-efficient than Fargate for bursty short-lived processing.

- ECS Fargate pricing:

#### ❌ Option B (INCORRECT): Redshift + Lambda

This option is architecturally inappropriate.

##### Redshift Is an Analytics Warehouse

Amazon Redshift is optimized for:
- OLAP analytics
- Complex aggregations
- Petabyte-scale reporting

Contest entries are:
- Transactional data
- Short-lived operational records
- High-ingest mobile transactions

This is not a Redshift use case.

##### High Cost and Poor Fit

Redshift clusters:
- Are expensive
- Require cluster lifecycle management
- Are not optimized for ephemeral transactional workloads

Even if deleted afterward:
- Provisioning overhead remains
- Cost efficiency is poor

##### Mismatch Between Service and Workload

This violates a core SAP-C02 principle:
> Choose services aligned with workload access patterns.

Relevant AWS documentation:
- Redshift concepts:
  https://docs.aws.amazon.com/redshift/latest/dg/c_redshift-and-dw-concepts.html

#### ❌ Option C (INCORRECT): ElastiCache + RDS + ECS Fargate

This option reduces some database pressure but fails the cost optimization requirement.

##### RDS Remains Always-On

The largest issue:
- RDS instances continue running continuously

This means:
- Persistent compute cost
- Storage cost
- Backup cost
- Maintenance overhead

##### Redis TTL Does Not Solve Persistence Cost

Redis expiration:
- Only removes cache entries
- Does not eliminate backend infrastructure

The architecture still maintains:
- RDS
- Redis cluster
- ECS infrastructure

##### Fargate Still More Expensive Than Lambda

For temporary workloads:
- Lambda is usually cheaper
- Fargate introduces runtime baseline costs

##### Complexity Increases

This option actually adds:
- Cache invalidation considerations
- Redis management
- More networking complexity

without eliminating the expensive persistent components.

Relevant AWS documentation:
- ElastiCache Redis TTL:

#### DynamoDB TTL Example

Enable TTL:

```bash
aws dynamodb update-time-to-live \
  - -table-name ContestEntries \
  - -time-to-live-specification \
  "Enabled=true, AttributeName=expirationTime"
```

Sample item:

```json
{
  "contestId": "contest-001",
  "userId": "user-100",
  "expirationTime": 1782847200
}
```

After expiration:
- DynamoDB deletes items automatically
- No scheduled cleanup tasks required

- --

#### Why Lambda Is Often Cheaper Than Containers

For sporadic workloads:

Lambda:
- Charges only during execution

Fargate:
- Charges while tasks are allocated CPU/memory

EC2:
- Charges continuously

Real-world enterprise observation:
- Many companies reduce 60–90% operational cost by replacing low-utilization EC2 fleets with Lambda.

- --

#### Common SAP-C02 Trap

Many exam questions intentionally tempt candidates with:
- Extra caching layers
- Containers
- Analytics databases

But the best answer is often:
> Simplest fully managed serverless architecture that satisfies requirements.

Always evaluate:
1. Can infrastructure be eliminated?
2. Can scaling become automatic?
3. Can cleanup become automatic?
4. Can idle cost approach zero?

If yes, serverless is usually preferred.

* *面试官可能追问 (Possible Follow-up Questions)**：

1. "Why is DynamoDB better than RDS for unpredictable mobile traffic?"
   → 回答要点 (Key Answer Points)：DynamoDB scales automatically without connection bottlenecks and supports massive concurrent writes. RDS requires instance sizing and scaling planning, making it less suitable for sudden traffic spikes.

2. "When should you choose Fargate instead of Lambda?"
   → 回答要点 (Key Answer Points)：Fargate is better for long-running processes, container portability, custom runtimes, or workloads exceeding Lambda limits. Lambda is ideal for event-driven short-lived executions.
