A company has multiple business units that each have separate accounts on AWS. Each business unit manages its own network with several VPCs that have CIDR ranges that overlap. The company’s marketing team has created a new internal application and wants to make the application accessible to all the other business units. The solution must use private IP addresses only.

Which solution will meet these requirements with the LEAST operational overhead?

A. Instruct each business unit to add a unique secondary CIDR range to the business unit's VPC. Peer the VPCs and use a private NAT gateway in the secondary range to route traffic to the marketing team.
B. Create an Amazon EC2 instance to serve as a virtual appliance in the marketing account's VPC. Create an AWS Site-to-Site VPN connection between the marketing team and each business unit's VPC. Perform NAT where necessary.
C. Create an AWS PrivateLink endpoint service to share the marketing application. Grant permission to specific AWS accounts to connect to the service. Create interface VPC endpoints in other accounts to access the application by using private IP addresses.
D. Create a Network Load Balancer (NLB) in front of the marketing application in a private subnet. Create an API Gateway API. Use the Amazon API Gateway private integration to connect the API to the NLB. Activate IAM authorization for the API. Grant access to the accounts of the other business units.

Suggested Answer: C

---

## 📋 Analysis

### 1. Understanding the Problem

The company has a **multi-account, multi-VPC architecture** with a critical constraint: **overlapping CIDR ranges** across business units. The marketing team needs to share an internal application with all other business units using **private IP addresses only** (no public internet exposure). The solution must minimize operational overhead.

**Key challenge:** Overlapping CIDRs make traditional VPC peering problematic because VPC peering does not support routing between VPCs with overlapping CIDR blocks. Additionally, the solution must work across separate AWS accounts.

---

### 2. Options Analysis

#### ✅ **Option C (CORRECT): AWS PrivateLink Endpoint Service**

This option uses **AWS PrivateLink**, which is purpose-built for sharing services across VPCs and accounts using private IP addresses:

1. The **marketing account** creates an **endpoint service** (backed by a Network Load Balancer or Gateway Load Balancer) that exposes the application.
2. The marketing account **grants permission** to specific AWS accounts (the other business units) to connect to the endpoint service.
3. Each **consumer account** creates an **interface VPC endpoint** in their VPC. This endpoint appears as an elastic network interface (ENI) with a private IP from their VPC's CIDR range.
4. Traffic flows entirely over the AWS backbone — never traverses the public internet.

**Why this is the best solution (LEAST operational overhead):**
- PrivateLink **natively handles overlapping CIDRs** — each consumer VPC uses its own private IPs for the endpoint ENI. There is no CIDR conflict because no VPC peering or transitive routing is involved.
- **No NAT, no VPN appliances, no CIDR re-engineering** — eliminating the major operational burdens of other options.
- Cross-account access is managed via IAM-style permission grants on the endpoint service.
- Fully managed, highly available, and scales automatically.

#### ❌ **Option A (INCORRECT): Secondary CIDR + VPC Peering + Private NAT Gateway**

- **VPC peering does not support overlapping CIDRs.** Adding a secondary CIDR range does not solve the fundamental problem — if both VPCs have overlapping primary CIDRs, peering can't distinguish between them.
- Private NAT gateway adds significant operational overhead (managing routing tables, NAT instances/gateways, high availability).
- Even with non-overlapping secondary CIDRs, orchestrating this across multiple accounts and VPCs creates a complex, fragile mesh of peerings.
- **Operational overhead is HIGH.**

#### ❌ **Option B (INCORRECT): EC2 Virtual Appliance + Site-to-Site VPN**

- Deploying an EC2 instance as a virtual appliance and establishing **Site-to-Site VPN connections to each business unit** creates enormous operational overhead:
  - Managing VPN tunnels (IKE/IPsec configuration, tunnel maintenance, monitoring)
  - NAT configuration on the virtual appliance
  - EC2 instance high availability and patching
  - Bandwidth limitations of VPN tunnels (1.25 Gbps per tunnel)
- This is essentially building a hub-and-spoke VPN architecture manually — **maximum operational overhead**.

#### ❌ **Option D (INCORRECT): NLB + API Gateway Private Integration**

- While technically feasible, this adds unnecessary layers:
  - API Gateway introduces additional cost, latency, and configuration complexity
  - API Gateway private integration with an NLB requires VPC Endpoints for API Gateway in each consumer VPC
  - The API Gateway layer is redundant — PrivateLink alone (Option C) already provides private connectivity without the API Gateway overhead
- **Higher operational overhead** than Option C with no added benefit for this use case.

### 3. Hands-On Knowledge

#### AWS PrivateLink Architecture for Cross-Account Service Sharing

```
┌─────────────────────────────┐    ┌──────────────────────────────┐
│   Marketing Account (Provider)│    │  Business Unit Account (Consumer)│
│                               │    │                                │
│  ┌───────────┐  ┌──────────┐ │    │  ┌────────────┐  ┌───────────┐ │
│  │   App     │→ │   NLB    │─│────│→│   Interface │→ │  Private  │ │
│  │ (Target)  │  │(Private) │ │    │  │   VPC      │  │   App     │ │
│  └───────────┘  └────┬─────┘ │    │  │  Endpoint  │  │ Consumer  │ │
│                      │        │    │  └────────────┘  └───────────┘ │
│               ┌──────▼──────┐ │    │                                │
│               │  Endpoint   │ │    │  Overlapping CIDRs:            │
│               │  Service    │ │    │  Marketing: 10.0.0.0/16        │
│               └─────────────┘ │    │  Business Unit A: 10.0.0.0/16  │
└─────────────────────────────┘    └──────────────────────────────┘
         AWS Backbone (Private Connectivity — No CIDR Conflict)
```

#### Key Operational Steps (Provider Side — Marketing Account):
```bash
# 1. Create NLB in private subnets, target group pointing to the application
aws elbv2 create-load-balancer --name marketing-nlb --type network \
  --subnets subnet-xxx subnet-yyy --scheme internal

# 2. Create endpoint service from the NLB
aws ec2 create-vpc-endpoint-service-configuration \
  --network-load-balancer-arns arn:aws:elasticloadbalancing:...:loadbalancer/net/marketing-nlb/... \
  --acceptance-required

# 3. Grant permission to consumer accounts
aws ec2 modify-vpc-endpoint-service-permissions \
  --service-id vpce-svc-xxx \
  --add-allowed-principals arn:aws:iam::111122223333:root
```

#### Key Operational Steps (Consumer Side — Business Unit Account):
```bash
# Create interface VPC endpoint to connect to the endpoint service
aws ec2 create-vpc-endpoint \
  --vpc-id vpc-consumer \
  --service-name com.amazonaws.vpce.region.vpce-svc-xxx \
  --vpc-endpoint-type Interface \
  --subnet-ids subnet-aaa subnet-bbb
```

**Exam-Ready PrivateLink Facts:**
| Feature | Detail |
|---------|--------|
| Traffic Path | AWS backbone only, never public internet |
| Overlapping CIDRs | ✅ Supported (no routing conflict) |
| Cross-Region | ❌ Must be same Region |
| Cross-Account | ✅ Supported via allow/deny principals |
| IPv6 | ✅ Supported |
| Bandwidth | Up to 100 Gbps per endpoint |
| Encryption | Traffic encrypted at the VPC level by default |

**Pro tip:** When the exam mentions "overlapping CIDRs," "multiple VPCs," "multiple accounts," and "private connectivity," PrivateLink is almost always the correct answer. It is the AWS-recommended pattern for service sharing across VPCs.
