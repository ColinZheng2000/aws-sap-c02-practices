A research company is running daily simulations in the AWS Cloud to meet high demand. The simulations run on several hundred Amazon EC2 instances that are based on Amazon Linux 2. Occasionally, a simulation gets stuck and requires a cloud operations engineer to solve the problem by connecting to an EC2 instance through SSH.
Company policy states that no EC2 instance can use the same SSH key and that all connections must be logged in AWS CloudTrail.
How can a solutions architect meet these requirements?

A. Launch new EC2 instances, and generate an individual SSH key for each instance. Store the SSH key in AWS Secrets Manager. Create a new IAM policy, and attach it to the engineers' IAM role with an Allow statement for the GetSecretValue action. Instruct the engineers to fetch the SSH key from Secrets Manager when they connect through any SSH client.
B. Create an AWS Systems Manager document to run commands on EC2 instances to set a new unique SSH key. Create a new IAM policy, and attach it to the engineers' IAM role with an Allow statement to run Systems Manager documents. Instruct the engineers to run the document to set an SSH key and to connect through any SSH client.
C. Launch new EC2 instances without setting up any SSH key for the instances. Set up EC2 Instance Connect on each instance. Create a new IAM policy, and attach it to the engineers' IAM role with an Allow statement for the SendSSHPublicKey action. Instruct the engineers to connect to the instance by using a browser-based SSH client from the EC2 console.
D. Set up AWS Secrets Manager to store the EC2 SSH key. Create a new AWS Lambda function to create a new SSH key and to call AWS Systems Manager Session Manager to set the SSH key on the EC2 instance. Configure Secrets Manager to use the Lambda function for automatic rotation once daily. Instruct the engineers to fetch the SSH key from Secrets Manager when they connect through any SSH client.


Suggest Answer: C

## 📋 Analysis

### Understanding the Problem

A research company runs hundreds of EC2 instances for daily simulations. Occasionally, an instance gets stuck and an engineer must SSH in to troubleshoot. Two hard security requirements exist: (1) **no EC2 instance can use the same SSH key** — meaning no shared or persistent keys allowed, and (2) **all connections must be logged in AWS CloudTrail** for auditability. The solution must balance security compliance with operational simplicity for the engineers.

### ✅ Correct Option: C

**Launch instances without SSH keys → Enable EC2 Instance Connect → IAM policy granting `SendSSHPublicKey` → Browser-based SSH from EC2 console**

This solution elegantly meets both requirements:

1. **No Shared or Persistent SSH Keys**: EC2 Instance Connect generates a **temporary, one-time SSH key pair** for each connection attempt. The public key is pushed to the instance metadata service (IMDS) and is valid for only 60 seconds. After the 60-second window expires, the public key is automatically removed from the instance. This means every connection uses a unique key — there is no key reuse whatsoever.

2. **CloudTrail Auditability**: The `SendSSHPublicKey` API call is logged in AWS CloudTrail with details including:
   - The IAM principal (which engineer made the request)
   - The target EC2 instance ID
   - The timestamp
   - The source IP address
   
   This provides a complete, immutable audit trail of who accessed which instance and when — satisfying the logging requirement without any additional infrastructure.

3. **Operational Simplicity**: Engineers can connect directly from the EC2 console in their browser — no SSH client installation required, no private key files to manage, no `.pem` files to lose or accidentally share.

4. **IAM Integration**: Access control is managed through IAM policies. You can restrict which IAM principals can connect to which instances using resource-level permissions and condition keys (e.g., `ec2:osuser` to limit which OS user they can connect as).

### ❌ Incorrect Options

**Option A — Individual SSH keys stored in Secrets Manager**

- While each instance gets a unique SSH key, the key is **persistent** — any engineer with `GetSecretValue` permission can retrieve the same private key at any time and use it repeatedly. This violates the intent of "no shared SSH key" because the key becomes a standing credential, not a per-session credential.
- Secrets Manager access is logged, but the log shows key retrieval, not actual SSH connections. An engineer could retrieve the key once and connect many times without additional CloudTrail entries.
- Managing hundreds of secrets (one per instance) in Secrets Manager introduces operational overhead and cost ($0.40 per secret per month).

**Option B — SSM Document to set SSH key on demand**

- The SSM document sets a persistent SSH key on the instance — it doesn't rotate or expire. Once set, that key remains valid indefinitely.
- The `RunCommand` or `RunDocument` API calls are logged, but the actual SSH connections using the set key are not captured in CloudTrail.
- Engineers must manage private keys on their local machines after the document runs — reintroducing the risk of key sharing or loss.

**Option D — Lambda + Session Manager + Secrets Manager rotation**

- This is an over-engineered solution. Daily key rotation means a key could be shared for up to 24 hours before rotation — not acceptable if "no shared key" means per-session uniqueness.
- Using Session Manager (which is designed for agent-based access without SSH keys) to *set SSH keys* conflates two different access mechanisms.
- Lambda functions and Secrets Manager automatic rotation add cost, complexity, and failure points without delivering the per-session uniqueness that EC2 Instance Connect provides natively.

### 🔧 Hands-On Knowledge

- **EC2 Instance Connect Prerequisites**: Amazon Linux 2 comes with EC2 Instance Connect pre-installed. For other AMIs, you must install the `ec2-instance-connect` package. The instance must also have a public IP or be accessible via a VPN/Direct Connect.
- **IAM Policy Example** for EC2 Instance Connect:
  ```json
  {
      "Version": "2012-10-17",
      "Statement": [
          {
              "Effect": "Allow",
              "Action": "ec2-instance-connect:SendSSHPublicKey",
              "Resource": "arn:aws:ec2:*:*:instance/*",
              "Condition": {
                  "StringEquals": {
                      "ec2:osuser": "ec2-user"
                  }
              }
          },
          {
              "Effect": "Allow",
              "Action": "ec2:DescribeInstances",
              "Resource": "*"
          }
      ]
  }
  ```
- **EC2 Instance Connect vs. Systems Manager Session Manager**:
  | Feature | EC2 Instance Connect | Session Manager |
  |---|---|---|
  | Protocol | SSH only | Shell access (no SSH) |
  | Key management | Ephemeral per-session | No keys (IAM-based) |
  | CloudTrail logging | `SendSSHPublicKey` API | `StartSession` API |
  | Browser access | EC2 console SSH | Fleet Manager |
  | Port requirements | Port 22 to instance | Outbound HTTPS to SSM endpoints |
- **For even stronger security**: Consider Systems Manager Session Manager instead of SSH entirely. It eliminates SSH keys, doesn't require port 22 open, and provides full session logging to S3/CloudWatch Logs with session-level command auditing.
- **Reference**: [EC2 Instance Connect](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/Connect-using-EC2-Instance-Connect.html) | [Session Manager Auditing](https://docs.aws.amazon.com/systems-manager/latest/userguide/session-manager-logging.html)
