A solutions architect is designing a solution to process events. The solution must have the ability to scale in and out based on the number of events that the solution receives. If a processing error occurs, the event must move into a separate queue for review.

Which solution will meet these requirements?

A. Send event details to an Amazon Simple Notification Service (Amazon SNS) topic. Configure an AWS Lambda function as a subscriber to the SNS topic to process the events. Add an on-failure destination to the function. Set an Amazon Simple Queue Service (Amazon SQS) queue as the target.
B. Publish events to an Amazon Simple Queue Service (Amazon SQS) queue. Create an Amazon EC2 Auto Scaling group. Configure the Auto Scaling group to scale in and out based on the ApproximateAgeOfOldestMessage metric of the queue. Configure the application to write failed messages to a dead-letter queue.
C. Write events to an Amazon DynamoDB table. Configure a DynamoDB stream for the table. Configure the stream to invoke an AWS Lambda function. Configure the Lambda function to process the events.
D. Publish events to an Amazon EventBndge event bus. Create and run an application on an Amazon EC2 instance with an Auto Scaling group that is behind an Application Load Balancer (ALB). Set the ALB as the event bus target. Configure the event bus to retry events. Write messages to a dead-letter queue if the application cannot process the messages.

Suggested Answer: A

---

## рҹ“Ӣ Analysis

### 1. Understanding the Problem

The company needs an **event processing pipeline** with three specific requirements:

1. **Auto-scaling:** The solution must scale in and out elastically based on the number of incoming events.
2. **Error handling:** If a processing error occurs, the failed event must be moved into a **separate queue for review** (a dead-letter queue / DLQ pattern).
3. **Operational simplicity:** The architect should prefer managed, serverless services to reduce operational burden.

The core design challenge is choosing the right combination of messaging/event services, compute, and error-handling mechanisms that together satisfy all three requirements.

---

### 2. Options Analysis

#### вң… **Option A (CORRECT): SNS вҶ’ Lambda (Async) вҶ’ On-Failure Destination (SQS)**

This is a **fully serverless, event-driven architecture**:

1. **Amazon SNS** receives event details and fans them out to subscribers. SNS is a fully managed pub/sub service with virtually unlimited scale.
2. **AWS Lambda** is configured as a subscriber to the SNS topic. Lambda **automatically scales** вҖ” as the event rate increases, Lambda provisions more concurrent executions; as it decreases, unused concurrency is released. No manual scaling configuration needed.
3. **On-failure destination** вҖ” when Lambda is invoked asynchronously by SNS, failed invocations (after exhausting the retry policy, default: 2 retries) are automatically sent to a configured **SQS queue** for review. This is the native "DLQ pattern" for Lambda.

**Why this is the optimal solution:**
- **Serverless auto-scaling:** Lambda concurrency scales seamlessly with event volume вҖ” no threshold tuning, no cool-down periods, no over-provisioning.
- **Built-in dead-letter pattern:** Lambda's on-failure destination (or async DLQ) sends failed events to SQS for manual review without any custom code.
- **Zero infrastructure management:** No EC2 instances, no Auto Scaling groups, no load balancers.
- **Pay-per-use:** Costs are proportional to event volume, including zero cost when idle.

#### вқҢ **Option B (INCORRECT): SQS + EC2 Auto Scaling (ApproximateAgeOfOldestMessage)**

- While SQS вҶ’ EC2 is a valid processing pattern, it has significant drawbacks compared to Option A:
  - **Scaling latency:** EC2 Auto Scaling reacts to metric changes with a delay (metric collection interval + alarm evaluation + instance launch/bootstrap time). During sudden bursts, the `ApproximateAgeOfOldestMessage` will spike before new instances come online вҖ” causing processing lag.
  - **Over-provisioning risk:** To handle unpredictable bursts, you'd need to either over-provision (wasteful) or accept latency during scale-out.
  - **Operational overhead:** Managing EC2 AMIs, patching, capacity planning, and Auto Scaling group tuning.
- The SQS DLQ itself is fine вҖ” but the compute layer (EC2) introduces unnecessary operational complexity when Lambda can do the same job serverlessly.

#### вқҢ **Option C (INCORRECT): DynamoDB Streams + Lambda**

- DynamoDB Streams is a change-data-capture mechanism for DynamoDB tables вҖ” it's not a general-purpose event ingestion mechanism.
- **No DLQ concept:** DynamoDB Streams + Lambda has no built-in mechanism to move failed processing attempts to a separate queue. If the Lambda function fails, the stream record is retried until it succeeds or expires (24 hours). There is no "review queue" for failed events.
- Writing events to DynamoDB first adds write latency, throughput provisioning concerns, and unnecessary cost versus direct event ingestion via SNS/SQS.

#### вқҢ **Option D (INCORRECT): EventBridge + EC2 Behind ALB**

- EventBridge rules can target **API destinations** (HTTP endpoints), but configuring an ALB with EC2 as the event bus target is architecturally awkward:
  - The ALB must be internet-facing or accessed via a configured API destination, adding network complexity.
  - EC2 Auto Scaling behind an ALB requires management of both the Auto Scaling group and the ALB.
  - EventBridge retries and DLQ are built-in, which is good вҖ” but the compute architecture is unnecessarily heavy.
- Compare to Option A's simplicity: SNS вҶ’ Lambda вҶ’ SQS DLQ is all managed, no servers, no load balancers.

#### Implementing Option A (AWS CLI)

```bash
# 1. Create SNS topic
aws sns create-topic --name event-processing-topic

# 2. Create SQS queue for failed events (on-failure destination)
aws sqs create-queue --queue-name event-processing-dlq

# 3. Create Lambda function with on-failure destination
aws lambda create-function \
  --function-name event-processor \
  --runtime python3.11 \
  --handler index.handler \
  --role arn:aws:iam::account:role/lambda-sns-sqs-role

# 4. Configure async event invoke config with on-failure destination
aws lambda put-function-event-invoke-config \
  --function-name event-processor \
  --maximum-retry-attempts 2 \
  --destination-config '{
    "OnFailure": {
      "Destination": "arn:aws:sqs:region:account:event-processing-dlq"
    }
  }'

# 5. Subscribe Lambda to SNS topic
aws sns subscribe \
  --topic-arn arn:aws:sns:region:account:event-processing-topic \
  --protocol lambda \
  --notification-endpoint arn:aws:lambda:region:account:function:event-processor
```

**Lambda Async Invocation Flow (for understanding DLQ behavior):**
```
Event вҶ’ Lambda Invocation (Attempt 1)
  в”ңв”Җ Success вҶ’ Done
  в””в”Җ Failure вҶ’ Retry (Attempt 2)
       в”ңв”Җ Success вҶ’ Done
       в””в”Җ Failure вҶ’ Retry (Attempt 3, if configured)
            в”ңв”Җ Success вҶ’ Done
            в””в”Җ Failure вҶ’ On-Failure Destination (SQS DLQ)
```

**Key Differences Between Lambda DLQ vs. SQS DLQ:**

| Aspect | Lambda On-Failure Destination | SQS DLQ (Redrive Policy) |
|--------|------------------------------|--------------------------|
| Triggered by | Async invocation failure after retries | Exceeding maxReceiveCount |
| Supported dest | SQS, SNS, Lambda, EventBridge | SQS only |
| Granularity | Per-function configuration | Per-queue configuration |
| Retry control | `maximum-retry-attempts` (0-2) | `maxReceiveCount` (1-1000) |

**Important Exam Gotcha вҖ” Synchronous vs. Asynchronous Lambda Invocation:**
- SNS invokes Lambda **asynchronously** вҖ” the Lambda service queues the event and processes it. This means on-failure destinations and DLQs work.
- If the question said "SQS вҶ’ Lambda" instead, the invocation would be synchronous (poll-based), and on-failure destinations would NOT apply. Instead, you'd use SQS DLQ at the queue level.
- Always verify that the invocation model matches the error-handling mechanism.
