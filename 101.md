An external audit of a company’s serverless application reveals IAM policies that grant too many permissions. These policies are attached to the company's AWS Lambda execution roles. Hundreds of the company's Lambda functions have broad access permissions such as full access to Amazon S3 buckets and Amazon DynamoDB tables. The company wants each function to have only the minimum permissions that the function needs to complete its task.

A solutions architect must determine which permissions each Lambda function needs.

What should the solutions architect do to meet this requirement with the LEAST amount of effort?

A. Set up Amazon CodeGuru to profile the Lambda functions and search for AWS API calls. Create an inventory of the required API calls and resources for each Lambda function. Create new IAM access policies for each Lambda function. Review the new policies to ensure that they meet the company's business requirements.
B. Turn on AWS CloudTrail logging for the AWS account. Use AWS Identity and Access Management Access Analyzer to generate IAM access policies based on the activity recorded in the CloudTrail log. Review the generated policies to ensure that they meet the company's business requirements.
C. Turn on AWS CloudTrail logging for the AWS account. Create a script to parse the CloudTrail log, search for AWS API calls by Lambda execution role, and create a summary report. Review the report. Create IAM access policies that provide more restrictive permissions for each Lambda function.
D. Turn on AWS CloudTrail logging for the AWS account. Export the CloudTrail logs to Amazon S3. Use Amazon EMR to process the CloudTrail logs in Amazon S3 and produce a report of API calls and resources used by each execution role. Create a new IAM access policy for each role. Export the generated roles to an S3 bucket. Review the generated policies to ensure that they meet the company’s business requirements.

# Q: Rightsizing IAM permissions for many Lambda functions

## Understanding the Problem

- Hundreds of Lambda functions currently run with overly broad IAM permissions (e.g., full access to S3 and DynamoDB).
- Goal: **principle of least privilege** – each Lambda function should have only the minimal permissions it actually uses.
- Constraint: Need to determine required permissions for each function with the **least amount of effort**.
- We are looking for an approach that:
  - Analyzes **actual usage** of AWS APIs by Lambda functions.
  - Automates the generation of IAM **policies based on real activity**.
  - Minimizes manual scripting, parsing, or custom analytics.

- --

## Services Explanation

### AWS Identity and Access Management (IAM) & Lambda Execution Role
- Lambda functions assume an **IAM execution role**; permissions for AWS API calls come from this role’s attached policies.
- Over-permissive policies violate best practices:
  - [Security best practices in IAM](https://docs.aws.amazon.com/IAM/latest/UserGuide/best-practices.html)
  - [Security best practices for AWS Lambda](https://docs.aws.amazon.com/lambda/latest/dg/best-practices.html#lambda-security)

### AWS CloudTrail
- CloudTrail records **AWS API calls** and related activity in your AWS account.
- It logs who called which API, on which resource, from where, and when.
- For this use case:
  - Needed as a **data source of historical activity** to understand:
    - Which Lambda execution role made which API calls.
    - Which resources were accessed.

Docs:  
- [What is AWS CloudTrail?](https://docs.aws.amazon.com/awscloudtrail/latest/userguide/cloudtrail-user-guide.html)

### IAM Access Analyzer – Policy Generation from CloudTrail
- **IAM Access Analyzer** can:
  - **Generate IAM policies** from recorded CloudTrail activity.
  - Suggest **least-privilege policies** by looking at the **past API calls** made by a principal (e.g., a role).
- Workflow:
  1. Enable CloudTrail (if not already).
  2. In IAM Access Analyzer, use **“Generate policies based on CloudTrail activity”**.
  3. It analyzes CloudTrail logs for a given role.
  4. It outputs a **policy template** with actions and resources actually used.
  5. Review and refine these policies.

Docs:  
- [IAM Access Analyzer policy generation](https://docs.aws.amazon.com/IAM/latest/UserGuide/access-analyzer-policy-generation.html)  
- [IAM Access Analyzer in IAM](https://docs.aws.amazon.com/IAM/latest/UserGuide/what-is-access-analyzer.html)

### Other Services Mentioned

#### Amazon CodeGuru
- **CodeGuru Profiler**: runtime performance analysis.
- **CodeGuru Reviewer**: static code analysis for Java/Python (code quality, security issues).
- Not designed to **inventory AWS API calls per function in a way that directly produces IAM policies**.

Docs:  
- [What is Amazon CodeGuru?](https://docs.aws.amazon.com/codeguru/latest/userguide/welcome.html)

#### Amazon EMR
- Managed Hadoop/Spark cluster for large-scale data processing.
- Could parse and analyze CloudTrail logs, but that requires heavy **custom ETL and scripting**.
- Not a purpose-built IAM policy tool; more effort than needed here.

Docs:  
- [What is Amazon EMR?](https://docs.aws.amazon.com/emr/latest/ManagementGuide/emr-what-is-emr.html)

## Correct Option and Insightful Explanation

### ✅ Correct Answer: **B**

> Turn on AWS CloudTrail logging for the AWS account. Use AWS Identity and Access Management Access Analyzer to generate IAM access policies based on the activity recorded in the CloudTrail log. Review the generated policies to ensure that they meet the company's business requirements.

**Why B is best (least effort):**

1. **Purpose-built for this use case**  
   - IAM Access Analyzer **policy generation** is explicitly designed for:
     - Taking **CloudTrail logs** as input.
     - Producing **least-privilege policy suggestions** for IAM roles and users based on real historical usage.
   - This maps directly to:
     - “Determine which permissions each Lambda function needs”
     - “Least amount of effort.”

2. **High degree of automation**
   - No need to write log parsing scripts (as in C).
   - No need to manage data processing clusters (as in D).
   - No need to manually inspect code or profiler output (as in A).

3. **Data-driven least-privilege**
   - Looks at **actual API calls** the Lambda execution roles made.
   - Generates **granular actions** and **resource ARNs**.
   - Helps systematically remove unused permissions from overly broad S3 and DynamoDB access.

4. **Review step (still required)**
   - AWS docs emphasize that generated policies are **recommendations**:
     - Must be **reviewed and adjusted** for:
       - Future needs not yet observed in CloudTrail.
       - Edge cases, infrequent operations, or new features.
   - This is reflected in the option: “Review the generated policies to ensure that they meet the company’s business requirements.”

This approach therefore aligns with AWS best practices and minimizes manual and custom work.

## Incorrect Options and Insightful Explanation

### ❌ Option A

> Set up Amazon CodeGuru to profile the Lambda functions and search for AWS API calls. Create an inventory of the required API calls and resources for each Lambda function. Create new IAM access policies for each Lambda function. Review the new policies...

**Why it’s incorrect / higher effort:**

- **Misuse of CodeGuru:**
  - CodeGuru Profiler is for performance/efficiency, not for **systematic IAM policy generation**.
  - CodeGuru Reviewer can detect API misuse patterns, but not a full mapping of every action/resource per role with policy generation.
- **Manual inventory and mapping**:
  - “Search for AWS API calls” + “create an inventory” + “create new IAM policies” = mostly **manual work**.
- **Not leveraging CloudTrail**:
  - Ignores the native AWS mechanism designed for this type of usage analysis.

Compared to Access Analyzer’s built-in **policy generation from CloudTrail**, this is clearly more work and less targeted.

- --

### ❌ Option C

> Turn on AWS CloudTrail logging for the AWS account. Create a script to parse the CloudTrail log, search for AWS API calls by Lambda execution role, and create a summary report. Review the report. Create IAM access policies...

**Why it’s incorrect / higher effort:**

- **Reinventing IAM Access Analyzer**:
  - This is essentially building your own **log analysis and policy inference** engine.
  - Requires:
    - Writing and maintaining parsing scripts.
    - Defining logic for action/resource aggregation.
    - Manual translation of reports into IAM policies.
- **Manual policy creation**:
  - Even after the report, a human still must **manually write** the IAM policies.

Access Analyzer **already does this** natively and reduces the need for custom code, so C is strictly more effort than B.

- --

### ❌ Option D

> Turn on AWS CloudTrail logging for the AWS account. Export the CloudTrail logs to Amazon S3. Use Amazon EMR to process the CloudTrail logs in Amazon S3 and produce a report of API calls and resources used by each execution role. Create a new IAM access policy for each role. Export the generated roles to an S3 bucket. Review the generated policies...

**Why it’s incorrect / highest effort:**

- **Over-engineered solution**:
  - Spinning up EMR, writing Spark/Hadoop jobs, and building a pipeline just to analyze CloudTrail logs is heavy.
  - EMR makes sense for **big data analytics**, not for per-role IAM policy generation when a **managed IAM feature (Access Analyzer)** exists.
- **Custom implementation & maintenance**:
  - Must:
    - Design data schema and transformation.
    - Implement job logic for deriving actions/resources per execution role.
    - Implement policy generation logic.
  - Ongoing maintenance and operational overhead.

This is far from “least amount of effort” compared with simply using CloudTrail + IAM Access Analyzer’s built-in policy generation.

## Key Takeaways for the Exam

- When you see:  
  - “Determine required IAM permissions based on actual usage”  
  - “Least privilege” + “minimize effort”  
  - Think: **CloudTrail + IAM Access Analyzer policy generation**.
- Avoid custom log parsing or large-scale data processing when AWS already provides a **managed, security-focused service**.
- CodeGuru is for **code and performance analysis**, not for IAM policy derivation.
- EMR and custom scripts indicate **higher operational and development overhead**, often not aligned with “least effort” choices.

**Final Answer:** **B**
