A company has a web application that allows users to upload short videos. The videos are stored on Amazon EBS volumes and analyzed by custom recognition software for categorization.  
The website contains static content that has variable traffic with peaks in certain months. The architecture consists of Amazon EC2 instances running in an Auto  
Scaling group for the web application and EC2 instances running in an Auto Scaling group to process an Amazon SQS-queue. The company wants to re-architect the application to reduce operational overhead using AWS managed services where possible and remove dependencies on third-party software.  
Which solution meets these requirements?  

- A. Use Amazon ECS containers for the web application and Spot instances for the Scaling group that processes the SQS queue. Replace the custom software with Amazon Rekognition to categorize the videos.
- B. Store the uploaded videos in Amazon EFS and mount the file system to the EC2 instances for the web application. Process the SQS queue with an AWS Lambda function that calls the Amazon Rekognition API to categorize the videos.
- C. Host the web application in Amazon S3. Store the uploaded videos in Amazon S3. Use S3 event notification to publish events to the SQS queue. Process the SQS queue with an AWS Lambda function that call the Amazon Rekognition API to categorize the videos.
- D. Use AWS Elastic Beanstalk to launch EC2 instances in an Auto Scaling group for the application and launch a worker environment to process the SQS queue. Replace the custom software with Amazon Rekognition to categorize the videos.

# AWS SAP-C02 Practice Question Analysis

## Understanding the Problem

The company's goal is to **re-architect** their web application to **reduce operational overhead** by utilizing **AWS managed services** wherever possible and **removing dependencies on third-party software**.

**Current Architecture:**
1.  **Web Application & Static Content:** EC2 instances in an Auto Scaling Group (ASG) for the web app, variable traffic (peaks).
2.  **Video Storage:** Amazon EBS volumes (which are tied to specific EC2 instances/Availability Zones and require management).
3.  **Video Processing:** EC2 instances in an ASG processing an **Amazon SQS queue** with **custom recognition software** (third-party dependency).

**Key Requirements for New Architecture:**
1.  **Reduce Operational Overhead:** Move away from managing EC2 instances/ASGs/EBS for core tasks. $\rightarrow$ Use Serverless or fully managed services.
2.  **AWS Managed Services:** Maximize their use.
3.  **Replace Custom Software:** Eliminate the third-party dependency. $\rightarrow$ Use Amazon Rekognition.

***

## Services Explanation

| Service                                    | Category                  | Use Case & Operational Overhead Impact                                                                                                                          |
| :----------------------------------------- | :------------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Amazon S3** (Simple Storage Service)     | Storage/Serverless        | Object storage for static content hosting, video storage, and generating event notifications. **Low operational overhead** (no servers to manage).              |
| **Amazon Rekognition**                     | AI/ML Managed Service     | Deep learning-based image and video analysis. **Low operational overhead** (fully managed API service). Meets the requirement to replace custom software.       |
| **AWS Lambda**                             | Compute/Serverless        | Run code without provisioning or managing servers. Ideal for event-driven, intermittent workloads. **Lowest operational overhead** for compute.                 |
| **Amazon SQS** (Simple Queue Service)      | Messaging/Managed Service | Decouples microservices. **Low operational overhead** (fully managed queue service).                                                                            |
| **Amazon EC2** (Elastic Compute Cloud)     | Compute/IaaS              | Virtual servers. **High operational overhead** (OS patching, scaling, hardware maintenance).                                                                    |
| **Amazon EBS** (Elastic Block Store)       | Storage/Volume            | Block storage for EC2. **Moderate operational overhead** (snapshot/backup management, AZ dependency).                                                           |
| **Amazon EFS** (Elastic File System)       | Storage/Managed Service   | Scalable file storage for EC2. **Moderate operational overhead** (still needs EC2 to mount to, file-system management).                                         |
| **Amazon ECS** (Elastic Container Service) | Compute/Managed Service   | Container orchestration. **Lower operational overhead than EC2/ASG**, but still requires managing the underlying EC2 instances (or Fargate for lower overhead). |
| **AWS Elastic Beanstalk**                  | Compute/PaaS              | Orchestrates EC2/ASG/Load Balancers. **Lower operational overhead than raw EC2/ASG**, but still manages servers behind the scenes.                              |

## Correct Option(s) Explanation

**C. Host the web application in Amazon S3. Store the uploaded videos in Amazon S3. Use S3 event notification to publish events to the SQS queue. Process the SQS queue with an AWS Lambda function that call the Amazon Rekognition API to categorize the videos.**

This solution best meets all the requirements by adopting the maximum number of serverless and fully managed services:

1.  **Reduce Operational Overhead (Web Application):** Hosting the static content and web application files in **Amazon S3** completely eliminates the need to manage EC2 instances, an Auto Scaling Group, and load balancing for the application layer. This is the **lowest overhead** solution for static/semi-static web hosting, addressing the variable traffic requirement efficiently (Reference: *Hosting a Static Website on Amazon S3*).
2.  **Reduce Operational Overhead (Storage):** Storing uploaded videos in **Amazon S3** replaces Amazon EBS, offering highly durable, scalable, and readily available storage without the need for volume management, snapshot management, or AZ dependency.
3.  **Decoupling/Event-Driven Architecture:** Using **S3 event notification** to publish events directly to the **SQS queue** is the most efficient, managed, and decoupled way to trigger the processing workflow whenever a new video is uploaded.
4.  **Reduce Operational Overhead (Processing):** Using an **AWS Lambda function** to process the SQS queue is the **lowest overhead compute option**. Lambda automatically scales, has built-in SQS polling, and you only pay for compute time used.
5.  **Remove Third-Party Dependency:** The Lambda function calls the fully managed **Amazon Rekognition API** for video categorization, successfully replacing the custom, self-managed software.

This architecture is the most serverless option, resulting in the **lowest possible operational overhead**.

***

## Incorrect Options Explanation

**A. Use Amazon ECS containers for the web application and Spot instances for the Scaling group that processes the SQS queue. Replace the custom software with Amazon Rekognition to categorize the videos.**

* **Operational Overhead:** While **Amazon ECS** is better than raw EC2, it still requires managing an underlying compute layer (EC2 or Fargate). The requirement is to *reduce* overhead using *managed services*, and hosting static content on S3 (Option C) is a lower-overhead solution.
* **Storage:** The option doesn't address the current use of EBS for video storage, which needs to be replaced.
* **Spot Instances:** Using Spot instances for the SQS worker is good for cost but doesn't reduce the operational overhead of managing the EC2 instances, their ASG, and the SQS polling mechanism (which Lambda handles automatically).

**B. Store the uploaded videos in Amazon EFS and mount the file system to the EC2 instances for the web application. Process the SQS queue with an AWS Lambda function that calls the Amazon Rekognition API to categorize the videos.**

* **Operational Overhead (Web Application):** The web application still runs on **EC2 instances** (implied by mounting EFS to them), which is high overhead. The goal is to eliminate EC2 management for the web layer.
* **Operational Overhead (Storage):** **Amazon EFS** is a managed service but is typically used for shared file systems that must be mounted to EC2 instances. S3 is the superior, lower-overhead choice for large-scale, durable *object storage* (videos) that can directly trigger events.
* **Missing Component:** This option misses the critical step of how the SQS queue is populated with video upload events.

**D. Use AWS Elastic Beanstalk to launch EC2 instances in an Auto Scaling group for the application and launch a worker environment to process the SQS queue. Replace the custom software with Amazon Rekognition to categorize the videos.**

* **Operational Overhead:** **Elastic Beanstalk** is a Platform-as-a-Service (PaaS) that abstracts some management, but it still runs on and manages **EC2 instances** and an Auto Scaling Group. This approach still involves server management, which is exactly what the company wants to minimize.
* **Better Alternative Exists:** Options C's combination of S3/Lambda/Rekognition offers a far greater reduction in operational overhead (a fully serverless approach) compared to Beanstalk, which is merely a layer of abstraction over EC2.
* **Storage:** The option doesn't address the current use of EBS for video storage, which needs to be replaced.
