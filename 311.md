A car rental company has built a serverless REST API to provide data to its mobile app. The app consists of an Amazon API Gateway API with a Regional endpoint, AWS Lambda functions, and an Amazon Aurora MySQL Serverless DB cluster. The company recently opened the API to mobile apps of partners. A significant increase in the number of requests resulted, causing sporadic database memory errors. Analysis of the API traffic indicates that clients are making multiple HTTP GET requests for the same queries in a short period of time. Traffic is concentrated during business hours, with spikes around holidays and other events.
The company needs to improve its ability to support the additional usage while minimizing the increase in costs associated with the solution.
Which strategy meets these requirements?

A. Convert the API Gateway Regional endpoint to an edge-optimized endpoint. Enable caching in the production stage.
B. Implement an Amazon ElastiCache for Redis cache to store the results of the database calls. Modify the Lambda functions to use the cache.
C. Modify the Aurora Serverless DB cluster configuration to increase the maximum amount of available memory.
D. Enable throttling in the API Gateway production stage. Set the rate and burst values to limit the incoming calls.



# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

Current architecture:

```text
Mobile Apps
    ↓
API Gateway (Regional)
    ↓
Lambda
    ↓
Aurora MySQL Serverless
```

Symptoms:
- significant increase in API traffic,
- repeated HTTP GET requests,
- same queries requested many times,
- Aurora memory errors,
- predictable traffic spikes,
- requirement to minimize additional costs.

This is a classic:
> repeated-read workload problem.

The key clue is:

> "clients are making multiple HTTP GET requests for the same queries in a short period of time"

That strongly indicates:
- cacheable responses,
- API-layer caching opportunity.

- --

## ✅ Correct Answer: A

> Convert the API Gateway Regional endpoint to an edge-optimized endpoint. Enable caching in the production stage.

- --

## 2. Why Option A Is Correct

This option provides:
- request offloading,
- reduced Lambda invocations,
- reduced database queries,
- improved scalability,
- low operational overhead.

- --

### ✅ API Gateway caching directly solves repeated GET requests

API Gateway supports:
- built-in response caching,
- stage-level cache configuration,
- TTL-based expiration.

For repeated GET requests:

```text
First request:
Client → API Gateway → Lambda → Aurora

Subsequent requests:
Client → API Gateway Cache HIT
```

Benefits:
- Lambda is bypassed,
- Aurora is bypassed,
- database memory pressure drops significantly.

- --

### ✅ Ideal for serverless architectures

The architecture is already fully serverless:
- API Gateway,
- Lambda,
- Aurora Serverless.

Using native API Gateway caching:
- preserves serverless simplicity,
- avoids introducing new infrastructure,
- minimizes operational burden.

SAP-C02 strongly favors:
> managed native features before adding complexity.

- --

### ✅ Edge-optimized endpoint improves partner access

The company now supports:
- external partner mobile applications,
- potentially global users.

Edge-optimized endpoints:
- leverage CloudFront edge locations,
- reduce latency,
- improve user experience geographically.

While the primary fix is caching,
this adds additional scalability and performance benefits.

- --

### ✅ Cost-efficient solution

Compared with Redis or database scaling:
- no dedicated cluster,
- no extra VPC infrastructure,
- no application code rewrite,
- lower operational costs.

This directly aligns with:
> "minimizing the increase in costs"

- --

## ❌ Why Option B Is Incorrect

> Use ElastiCache for Redis and modify Lambda functions.

This can work technically, but it is not the best answer.

- --

### ❌ Higher complexity

Requires:
- Redis cluster management,
- Lambda cache integration,
- cache invalidation strategy,
- VPC networking,
- application modifications.

For a simple repeated GET pattern,
API Gateway caching is much simpler.

- --

### ❌ More expensive

Redis clusters:
- run continuously,
- incur infrastructure costs,
- add maintenance overhead.

API Gateway caching is typically cheaper for:
- straightforward API response caching.

- --

### ❌ Overengineered for this use case

The duplicated requests occur:
- before Lambda execution.

Therefore:
> cache at the API layer first.

Earlier caching reduces more downstream load.

- --

## ❌ Why Option C Is Incorrect

> Increase Aurora memory.

This is scaling the database instead of reducing unnecessary load.

- --

### ❌ Does not eliminate duplicated queries

Repeated requests still:
- invoke Lambda,
- hit Aurora,
- consume resources.

- --

### ❌ Poor cost optimization

Increasing Aurora capacity:
- increases recurring costs,
- without addressing inefficient request patterns.

Caching is much more efficient.

- --

### ❌ Less scalable

As traffic grows:
- database scaling alone becomes increasingly expensive.

Caching scales better for read-heavy workloads.

- --

## ❌ Why Option D Is Incorrect

> Enable API throttling.

Throttling is a protection mechanism, not a scalability optimization.

- --

### ❌ Restricts legitimate traffic

Partner applications may receive:
- HTTP 429 errors,
- degraded user experience.

The business goal is:
> support more usage

—not reject requests.

- --

### ❌ Does not solve duplicate reads

Even throttled traffic still:
- repeatedly hits Lambda and Aurora.

- --

### ❌ Wrong optimization target

Throttling is intended for:
- abuse prevention,
- DDoS mitigation,
- fair usage enforcement.

Not:
- repeated-query optimization.

- --

## 4. Important Architectural Principle

For repeated read requests:

```text
Cache as close to the client as possible
```

Preferred caching hierarchy:

```text
Client
 ↓
CDN / Edge Cache
 ↓
API Cache
 ↓
Application Cache
 ↓
Database
```

Earlier cache placement means:
- fewer downstream requests,
- lower infrastructure costs,
- better scalability.

   → 回答要点 (Key Answer Points)：Edge-optimized 使用 CloudFront 边缘节点优化全球访问；Regional endpoint 更适合本地区域内部访问。
