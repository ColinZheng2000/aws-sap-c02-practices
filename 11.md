## Question Solved: Scalable and Cost-Efficient Web Fleet for Video Content

A company used **Amazon EC2 instances** to deploy a web fleet to host a blog site. The EC2 instances are behind an **Application Load Balancer (ALB)** and are configured in an **Auto Scaling group**. The web application stores all blog content on an **Amazon EFS volume**. The company recently added a feature for bloggers to add video to their posts, attracting 10 times the previous user traffic. At peak times of day, users report **buffering and timeout issues** while attempting to reach the site or watch videos.

Which is the **MOST cost-efficient and scalable deployment** that will resolve the issues for users?

A. Reconfigure Amazon EFS to enable maximum I/O.
B. Update the blog site to use instance store volumes for storage. Copy the site contents to the volumes at launch and to Amazon S3 at shutdown.
C. Configure an **Amazon CloudFront distribution**. Point the distribution to an **S3 bucket**, and migrate the videos from EFS to **Amazon S3**.
D. Set up an **Amazon CloudFront distribution** for all site contents, and point the distribution at the **ALB**.

---

## Analysis

### Understanding the Problem

The core issue is **performance degradation** (buffering and timeouts) under a **10x increase in user traffic**, especially when serving **video content**. The existing architecture uses **Amazon EFS** for all content, which is a network file system ideal for shared file access across multiple EC2 instances, but its performance (throughput/IOPS) is often tied to the amount of data stored or can be limited by the network path for extremely high-demand media files like video. The massive traffic surge, particularly for video streaming, is likely **overloading the EFS volume, the EC2 instances, or both**.

The solution must be **MOST cost-efficient** and **scalable**, which points toward leveraging **global content delivery** and **optimized media storage**.

---

### Services Explanation

| Service | Description | Use Case Relevance |
| :--- | :--- | :--- |
| **Amazon EFS (Elastic File System)** | A scalable, elastic, cloud-native NFS file system for AWS Cloud services and on-premises resources. It provides shared file storage for multiple EC2 instances. | **Current Content Storage.** Good for shared web files, but less ideal and potentially costly for *very high-throughput* video streaming to *global* users. |
| **Amazon S3 (Simple Storage Service)** | Object storage with industry-leading scalability, data availability, security, and performance. Excellent for static content, backups, data lakes, and media/video files. | **Highly scalable, durable, and low-cost storage for media files.** Perfect origin for video content accessed globally via a CDN. |
| **Amazon CloudFront** | A fast content delivery network (CDN) service that securely delivers data, videos, applications, and APIs globally with low latency and high transfer speeds. | **Critical for reducing latency (buffering) and scaling capacity** by caching content closer to users and offloading the burden from the origin (EC2/ALB/EFS). |
| **Application Load Balancer (ALB)** | Distributes incoming application traffic across multiple targets, such as EC2 instances, in multiple Availability Zones. | **Existing component (Origin).** Can be used as a CloudFront origin for *dynamic* or *non-cacheable* content. |
| **EC2 Instance Store** | Temporary block-level storage for your EC2 instance. Data persists only for the life of the instance. | **Not a durable or shared solution.** Unsuitable for primary blog or video content storage. |

---

### Correct Option(s) Explanation

**C. Configure an Amazon CloudFront distribution. Point the distribution to an S3 bucket, and migrate the videos from EFS to Amazon S3.** ✅

* **Scalability & Performance:** Videos are **static content** and are accessed by a massive number of users. The combination of **Amazon S3** (for highly durable, high-throughput media storage) and **Amazon CloudFront** (for global caching) is the **standard, most scalable, and cost-efficient pattern** for delivering video/media content. CloudFront caches the videos at edge locations globally, drastically reducing latency (solving **buffering** issues) and the load on the origin (EFS/EC2/ALB), thus solving **timeout** issues.
* **Cost Efficiency:** S3 storage is significantly cheaper per GB than EFS. Furthermore, using CloudFront dramatically reduces data transfer costs from EFS/EC2 to the internet, as CloudFront transfer rates are generally more favorable and the content is served from the edge.

---

### Incorrect Options Explanation

**A. Reconfigure Amazon EFS to enable maximum I/O.** ❌

* While changing the throughput mode (e.g., to **Max I/O**) can increase aggregate throughput, it often comes with a slight increase in latency and may still be insufficient or overly expensive for a **10x traffic increase** on global video streaming. It doesn't solve the core issue of content being served from a single region far from the users (high latency/buffering), nor is it the **MOST cost-efficient** solution compared to a CDN/S3 combination.

**B. Update the blog site to use instance store volumes for storage. Copy the site contents to the volumes at launch and to Amazon S3 at shutdown.** ❌

* **Instance store** is **temporary** and not suitable for primary application content like a blog or video library, as the data is lost when the instance stops or terminates. It is also **not a shared file system**, meaning each EC2 instance would need a separate copy, which complicates content updates and is not a scalable, durable design. It does not address the need for a global content delivery network (CDN) to reduce latency.

**D. Set up an Amazon CloudFront distribution for all site contents, and point the distribution at the ALB.** ❌

* This is a valid approach for serving **dynamic content** (content that changes based on user interaction) from the ALB/EC2 fleet. However, it still uses the ALB/EC2/EFS complex as the origin for the **videos**. Videos are **static content**; storing and serving them directly from **Amazon S3** via CloudFront (Option C) is vastly more scalable, performant, and cost-efficient because S3 is purpose-built for high-throughput static media and scales virtually infinitely, whereas the ALB/EC2/EFS origin can become a bottleneck under extreme load. CloudFront pointing to an ALB is better for the *dynamic* parts of the site (e.g., login, comments), but S3 is superior for the *static* videos.
