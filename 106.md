A company has an on-premises website application that provides real estate information for potential renters and buyers. The website uses a Java backend and a NoSQL MongoDB database to store subscriber data.  
  
The company needs to migrate the entire application to AWS with a similar structure. The application must be deployed for high availability, and the company cannot make changes to the application.  
  
Which solution will meet these requirements?

- A. Use an Amazon Aurora DB cluster as the database for the subscriber data. Deploy Amazon EC2 instances in an Auto Scaling group across multiple Availability Zones for the Java backend application.
- B. Use MongoDB on Amazon EC2 instances as the database for the subscriber data. Deploy EC2 instances in an Auto Scaling group in a single Availability Zone for the Java backend application.
- C. Configure Amazon DocumentDB (with MongoDB compatibility) with appropriately sized instances in multiple Availability Zones as the database for the subscriber data. Deploy Amazon EC2 instances in an Auto Scaling group across multiple Availability Zones for the Java backend application.
- D. Configure Amazon DocumentDB (with MongoDB compatibility) in on-demand capacity mode in multiple Availability Zones as the database for the subscriber data. Deploy Amazon EC2 instances in an Auto Scaling group across multiple Availability Zones for the Java backend application.

# Analysis of AWS Migration for Java + MongoDB Application Question
## Understanding the Problem
The core requirements and current state are:
- Current setup: On-premises application with **Java backend** and **NoSQL MongoDB database** (stores subscriber data).
- Migration goal: Move to AWS with 3 key constraints:
  1. Maintain **similar structure** (no changes to the application).
  2. Ensure **high availability (HA)** for both backend and database.
  3. No application modifications (requires database compatibility with MongoDB).

## Services Explanation
### Key Services Involved
1. **Amazon Aurora**: Managed relational database (MySQL/PostgreSQL-compatible) designed for relational workloads. It is NOT a NoSQL database and cannot replace MongoDB—critical incompatibility for the application’s NoSQL dependency. [Aurora Overview](https://docs.aws.amazon.com/rds/latest/AuroraUserGuide/CHAP_AuroraOverview.html)
2. **MongoDB on Amazon EC2**: Self-managed MongoDB deployment on EC2 instances. While it matches the database type, it requires manual management of HA (e.g., setting up replica sets across AZs), patching, backups, and scaling—adding operational overhead. It does not leverage AWS-managed database benefits.
3. **Amazon DocumentDB (with MongoDB Compatibility)**: Managed NoSQL database service that is drop-in compatible with MongoDB. It eliminates the need for application changes (uses the same MongoDB drivers/APIs) and natively supports **multi-AZ deployments** for HA (primary instance + replicas across AZs, automatic failover). Two capacity modes:
   - **Provisioned Capacity**: Fixed-size instances (aligns with on-prem MongoDB’s "fixed server" structure).
   - **On-Demand Capacity**: Auto-scales compute resources based on workload (variable structure, not aligned with traditional on-prem deployments). [DocumentDB Docs](https://docs.aws.amazon.com/documentdb/latest/developerguide/what-is.html)
4. **EC2 Auto Scaling Groups (ASG)**: Automatically scales EC2 instances for the Java backend. HA is only achieved if instances are deployed **across multiple Availability Zones** (single AZ deployments create a single point of failure). [ASG Multi-AZ Docs](https://docs.aws.amazon.com/autoscaling/ec2/userguide/asg-placement-groups.html#asg-multi-az)

## Correct Option Explanation
### Option C
This solution meets all requirements by aligning with the application’s structure, ensuring HA, and avoiding modifications:
1. **Database Compatibility & Structure**: Amazon DocumentDB is MongoDB-compatible—no application changes are needed (uses the same drivers/queries as on-prem MongoDB). The "appropriately sized instances" refers to **provisioned capacity mode**, which matches the on-prem MongoDB’s fixed-server structure (similar deployment model).
2. **Database High Availability**: DocumentDB is deployed across multiple AZs natively (primary + replica instances), providing automatic failover and resilience against AZ outages—no manual HA setup required.
3. **Backend High Availability**: EC2 instances in an Auto Scaling Group across multiple AZs ensure the Java backend remains available if one AZ fails.
4. **Minimal Operational Overhead**: DocumentDB is a managed service (AWS handles patching, backups, scaling), reducing effort compared to self-managed MongoDB on EC2.

## Incorrect Options Explanation
### Option A
- **Incompatible Database**: Amazon Aurora is a relational database, but the application uses NoSQL MongoDB. Migrating to Aurora would require significant application changes (rewriting NoSQL queries to relational SQL)—violating the "no changes to the application" requirement.
- **NoSQL Mismatch**: Aurora cannot replace MongoDB’s NoSQL functionality, making this solution structurally incompatible.

### Option B
- **Backend HA Failure**: The Java backend is deployed in an Auto Scaling Group in a **single Availability Zone**. A single AZ outage would take the backend offline—failing the high availability requirement.
- **Self-Managed Database Overhead**: Running MongoDB on EC2 requires manual setup of HA (e.g., replica sets), patching, and backups. This increases operational effort and risks reliability compared to managed DocumentDB.

### Option D
- **Structure Mismatch**: DocumentDB’s **on-demand capacity mode** auto-scales compute resources based on workload, which does not match the on-prem MongoDB’s fixed-server structure (violates the "similar structure" requirement). On-prem MongoDB uses statically provisioned servers, not pay-as-you-go auto-scaled compute.
- **Unnecessary Capacity Mode**: While on-demand is compatible with the application, it deviates from the original deployment structure. The requirement for "similar structure" favors provisioned instances (Option C) over on-demand.
