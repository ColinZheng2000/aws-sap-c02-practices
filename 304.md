A company provides a software as a service (SaaS) application that runs in the AWS Cloud. The application runs on Amazon EC2 instances behind a Network Load Balancer (NLB). The instances are in an Auto Scaling group and are distributed across three Availability Zones in a single AWS Region.

The company is deploying the application into additional Regions. The company must provide static IP addresses for the application to customers so that the customers can add the IP addresses to allow lists. The solution must automatically route customers to the Region that is geographically closest to them.

Which solution will meet these requirements?

A. Create an Amazon CloudFront distribution. Create a CloudFront origin group. Add the NLB for each additional Region to the origin group. Provide customers with the IP address ranges of the distribution’s edge locations.
B. Create an AWS Global Accelerator standard accelerator. Create a standard accelerator endpoint for the NLB in each additional Region. Provide customers with the Global Accelerator IP address.
C. Create an Amazon CloudFront distribution. Create a custom origin for the NLB in each additional Region. Provide customers with the IP address ranges of the distribution’s edge locations.
D. Create an AWS Global Accelerator custom routing accelerator. Create a listener for the custom routing accelerator. Add the IP address and ports for the NLB in each additional Region. Provide customers with the Global Accelerator IP address.


# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

The company already operates a highly available application architecture:
- EC2 instances in Auto Scaling groups
- Network Load Balancer (NLB)
- Multi-AZ deployment

The new requirement introduces global expansion with two critical networking constraints:

1. Customers need static IP addresses for firewall allow lists.
2. End users should automatically connect to the geographically closest AWS Region.

This immediately narrows the design space because:
- DNS-only solutions do not provide fixed IP addresses.
- CloudFront focuses mainly on HTTP/HTTPS acceleration and CDN behavior.
- The solution must support Layer 4 workloads behind NLB.

The key AWS service designed for this exact scenario is AWS Global Accelerator.

---

### 2. Options Analysis

#### ✅ Option B (CORRECT): Use AWS Global Accelerator standard accelerator with Regional NLB endpoints

This is the correct solution.

AWS Global Accelerator Standard Accelerator provides:
- two static Anycast IP addresses,
- intelligent global traffic routing,
- automatic routing to the nearest healthy Region,
- seamless integration with NLB.

Why this fully satisfies the requirements:

##### Static IP addresses for allow lists
Global Accelerator provides fixed static Anycast IPs.

This is extremely important in enterprise environments because:
- customer firewalls can permanently allow-list the IPs,
- backend infrastructure can change without affecting clients,
- there is no dependency on changing AWS public IP pools.

##### Geographic routing
Global Accelerator routes users through the AWS global backbone network to:
- the closest healthy endpoint,
- based on latency and endpoint health.

This is superior to traditional DNS routing because:
- traffic enters AWS infrastructure earlier,
- failover is faster,
- routing decisions are more deterministic.

##### Native support for NLB
Each Region can expose:
- one NLB,
- connected to Auto Scaling EC2 instances.

Global Accelerator endpoint groups can point directly to these Regional NLBs.

##### Enterprise operational advantages
This architecture is widely used in:
- healthcare SaaS,
- industrial platforms,
- B2B enterprise systems,
- regulated environments requiring firewall governance.

It minimizes operational complexity while improving:
- latency,
- resiliency,
- failover behavior,
- compliance alignment.

Relevant AWS official references:
- AWS Global Accelerator Developer Guide
- “How AWS Global Accelerator works”
- “Endpoint groups in AWS Global Accelerator”

---

#### ❌ Option A (INCORRECT): CloudFront origin group with NLB origins

This option is incorrect because CloudFront does not provide stable static IP addresses suitable for enterprise allow-listing.

Key problems:

##### CloudFront IP ranges are dynamic
CloudFront uses:
- many edge locations,
- large IP pools,
- periodically updated address ranges.

Customers would need to:
- continuously maintain AWS IP ranges,
- update firewall rules regularly.

This directly violates the operational simplicity requirement.

##### Origin groups are mainly for failover
CloudFront origin groups are designed for:
- primary/secondary origin failover,
- CDN redundancy.

They are not intended for:
- TCP/UDP application acceleration,
- Layer 4 global routing,
- static ingress IP use cases.

##### Architectural mismatch
The backend uses NLB:
- indicating TCP/UDP or Layer 4 workloads,
- which aligns better with Global Accelerator.

CloudFront primarily optimizes:
- HTTP,
- HTTPS,
- cached web content.

---

#### ❌ Option C (INCORRECT): CloudFront custom origins for Regional NLBs

This option has similar issues to Option A.

Although CloudFront can distribute traffic globally:
- it still does not provide fixed customer-facing static IP addresses.

Major limitations:

##### No static Anycast IPs
Customers cannot reliably allow-list:
- all CloudFront edge IP ranges,
- without operational overhead.

##### Wrong service purpose
CloudFront is a CDN service optimized for:
- caching,
- content acceleration,
- web delivery.

The requirement is:
- application ingress optimization,
- global traffic routing,
- static IP exposure.

That is the responsibility of Global Accelerator.

##### Common SAP-C02 exam trap
AWS frequently tests understanding of:
- CloudFront vs Global Accelerator.

Key distinction:

| Service | Primary Purpose |
|---|---|
| CloudFront | HTTP/HTTPS content delivery |
| Global Accelerator | Global TCP/UDP application routing with static IPs |

---

#### ❌ Option D (INCORRECT): Global Accelerator custom routing accelerator

This option is incorrect because Custom Routing Accelerator is intended for specialized use cases.

Custom routing accelerators are typically used for:
- gaming,
- VoIP,
- real-time communications,
- direct client-to-instance routing.

Why it does not fit:

##### Wrong accelerator type
The requirement only needs:
- Regional routing,
- static IPs,
- NLB integration.

This is exactly what Standard Accelerator already provides.

##### Custom routing complexity
Custom Routing Accelerator:
- maps users to specific EC2 instances and ports,
- requires more advanced routing logic,
- is unnecessary overhead here.

##### NLB integration mismatch
The question architecture is:
- load balancer centric,
- not direct instance routing centric.

For SAP-C02:
- unless the scenario explicitly requires deterministic port-to-instance routing,
- choose Standard Accelerator instead of Custom Routing Accelerator.

### 3. Hands-On Knowledge

Typical enterprise architecture:

```text
Users
   ↓
AWS Global Accelerator
   ↓
Regional Endpoint Groups
   ↓
Network Load Balancers
   ↓
EC2 Auto Scaling Groups
```

Important real-world benefits:

##### Faster failover than DNS
Global Accelerator performs health-based routing quickly because:
- it does not rely heavily on DNS TTL expiration,
- traffic steering occurs within AWS edge infrastructure.

##### Better performance
Traffic enters the AWS backbone network as early as possible:
- reducing internet path variability,
- improving packet loss and latency.

##### Stable customer integrations
Enterprise customers strongly prefer:
- fixed ingress IPs,
- predictable network policies,
- reduced firewall change management.

This is especially common in:
- hospitals,
- manufacturing systems,
- financial services,
- B2B SaaS integrations.

Example AWS CLI workflow:

Create accelerator:
```bash
aws globalaccelerator create-accelerator \
  --name prod-global-app
```

Create listener:
```bash
aws globalaccelerator create-listener \
  --accelerator-arn <accelerator-arn> \
  --protocol TCP \
  --port-ranges FromPort=443,ToPort=443
```

Create endpoint group:
```bash
aws globalaccelerator create-endpoint-group \
  --listener-arn <listener-arn> \
  --endpoint-group-region us-east-1
```

Add NLB endpoint:
```bash
aws globalaccelerator add-endpoints \
  --endpoint-group-arn <group-arn> \
  --endpoint-configurations EndpointId=<nlb-arn>,Weight=100
```

Operational considerations:
- Global Accelerator pricing includes:
  - hourly accelerator cost,
  - data transfer acceleration charges.
- Commonly paired with:
  - AWS Shield Standard,
  - WAF (through ALB architectures),
  - Route 53 health-aware DNS.

Common exam comparison:

| Feature | CloudFront | Global Accelerator |
|---|---|---|
| Static IP | No | Yes |
| Anycast IP | No | Yes |
| TCP/UDP acceleration | Limited | Yes |
| CDN caching | Yes | No |
| NLB support | Indirect | Native |
| Enterprise allow-list friendly | Poor | Excellent |

**面试官可能追问 (Possible Follow-up Questions)**：

1. "Global Accelerator 和 Route 53 latency routing 的区别是什么？"
   → 回答要点 (Key Answer Points)：Route 53 latency routing 是 DNS 解析层面的调度，受 DNS TTL 和客户端缓存影响；Global Accelerator 使用 Anycast 静态 IP 和 AWS 全球骨干网络，故障切换更快、性能更稳定。

2. "为什么企业 allow-list 场景通常选择 Global Accelerator？"
   → 回答要点 (Key Answer Points)：Global Accelerator 提供固定静态 IP，适合长期防火墙规则管理；CloudFront IP 范围较大且会变化，企业运维复杂度高。
