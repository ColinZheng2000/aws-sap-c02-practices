An application is deployed on Amazon EC2 instances that run in an Auto Scaling group. The Auto Scaling group configuration uses only one type of instance.

CPU and memory utilization metrics show that the instances are underutilized. A solutions architect needs to implement a solution to permanently reduce the EC2 cost and increase the utilization.

Which solution will meet these requirements with the LEAST number of configuration changes in the future?

A. List instance types that have properties that are similar to the properties that the current instances have. Modify the Auto Scaling group's launch template configuration to use multiple instance types from the list.
B. Use the information about the application's CPU and memory utilization to select an instance type that matches the requirements. Modify the Auto Scaling group's configuration by adding the new instance type. Remove the current instance type from the configuration.
C. Use the information about the application's CPU and memory utilization to specify CPU and memory requirements in a new revision of the Auto Scaling group's launch template. Remove the current instance type from the configuration.
D. Create a script that selects the appropriate instance types from the AWS Price List Bulk API. Use the selected instance types to create a new revision of the Auto Scaling group's launch template.


Suggest Answer:  C

---

## Analysis

### Understanding the Problem

An application runs on Amazon EC2 instances within an Auto Scaling group (ASG) that currently uses only **one instance type**. Monitoring data reveals that both **CPU and memory utilization are low** — the instances are significantly over-provisioned. The solutions architect must permanently reduce EC2 costs and increase resource utilization. Critically, the solution must also require the **least number of configuration changes in the future**, meaning it should adapt automatically as AWS releases new instance types or deprecates old ones.

### Correct Option(s) Explanation

**Option C (Correct):** Use CPU and memory utilization data to specify **CPU and memory requirements** (vCPUs and memory size) directly in a new revision of the Auto Scaling group's launch template, and remove the explicitly specified instance type.

**Why this works:**

- This is **attribute-based instance type selection** (ABIS), a feature of EC2 Auto Scaling that lets you define compute requirements as a set of instance attributes (vCPU count, memory, architecture, etc.) instead of hard-coding specific instance types.
- When you configure ABIS, EC2 Auto Scaling **automatically identifies and uses any instance type** — including newly released or previous-generation types — that matches your defined attributes. This eliminates the need to manually update the launch template when:
  - AWS introduces new instance families (e.g., from m6i to m7i)
  - Certain instance types become unavailable in an Availability Zone
  - Better price-performance options emerge
- By specifying only the CPU and memory that the application actually needs (rather than the oversized instance type currently in use), you achieve both **cost reduction** (smaller, right-sized instances) and **higher utilization**.
- This approach requires **zero future configuration changes** because the attribute-based rules automatically include all compatible instance types, present and future.

- **Reference:** [AWS EC2 Auto Scaling Attribute-Based Instance Type Selection](https://docs.aws.amazon.com/autoscaling/ec2/userguide/create-mixed-instances-group-attribute-based-instance-type-selection.html)

### Incorrect Options Explanation

**Option A:** List instance types with similar properties and modify the launch template to use **multiple instance types** from that list.

- ❌ This is the traditional "mixed instances policy" approach where you manually enumerate a list of acceptable instance types. While this provides some flexibility for spot/diversification, it still requires a **hard-coded list**. When AWS introduces new instance families (e.g., m7i, c7i), you must **manually update the list** to take advantage of them. If an instance type in the list is retired, the ASG may fail to launch instances. This violates the "least configuration changes in the future" requirement.

**Option B:** Select a single matching instance type based on CPU/memory, and replace the current type with it.

- ❌ This merely swaps one hard-coded instance type for another. It addresses the immediate cost issue but provides **zero future flexibility**. When better instance types become available or the chosen type faces capacity constraints, the team must make manual configuration changes again. This is the **opposite** of "least number of configuration changes in the future."

**Option D:** Create a script using the AWS Price List Bulk API to select instance types and update the launch template.

- ❌ While this automates part of the process, it introduces **custom scripting complexity** that must be maintained, monitored, and updated over time. The Price List API returns pricing data, not instance compatibility or availability. You would need to build complex logic for filtering, availability zone mapping, and launch template updates. This is a **DIY approach** when a native, fully-managed AWS feature (ABIS) already exists to solve the problem. The operational overhead of maintaining custom scripts contradicts the "least effort" principle.

### Enterprise Hands-on Knowledge

1. **ABIS Configuration Example:** In a CloudFormation or launch template, instead of specifying `InstanceType: m5.xlarge`, you define:
   ```json
   {
     "InstanceRequirements": {
       "VCpuCount": {"Min": 2, "Max": 4},
       "MemoryMiB": {"Min": 4096, "Max": 8192},
       "CpuManufacturers": ["intel", "amd"],
       "InstanceGenerations": ["current"]
     }
   }
   ```
   Auto Scaling will then select from all eligible instance types.

2. **Cost Optimization Strategy:** ABIS combined with a **mixed instances policy** and **purchase options** (On-Demand + Spot) provides the most robust cost optimization. Define attributes for the base compute requirements, then let EC2 Auto Scaling pick the cheapest available option that meets those attributes across multiple purchase models.

3. **Right-Sizing Workflow:** Before implementing ABIS:
   - Use **AWS Compute Optimizer** to get instance type recommendations based on CloudWatch metrics
   - Review **CloudWatch Agent** memory metrics (not just CPU, as standard EC2 metrics don't include memory)
   - Test with a **canary deployment** or blue/green strategy to validate the new, smaller instance types under production load

4. **Common SAP-C02 Trap:** The exam frequently tests whether candidates understand the difference between specifying instance types explicitly (Options A, B) versus using attribute-based selection (Option C). The key phrase is "least number of configuration changes in the future" — this always points to attribute-based instance type selection.

5. **Architecture Consideration:** For applications that are truly underutilized, also evaluate whether the compute layer can be replaced with **AWS Lambda** or **Amazon ECS on Fargate** for even greater cost efficiency, though this would require application refactoring and is beyond the scope of this specific question.
