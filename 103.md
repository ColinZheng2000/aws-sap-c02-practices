A company uses AWS Organizations for a multi-account setup in the AWS Cloud. The company uses AWS Control Tower for governance and uses AWS Transit Gateway for VPC connectivity across accounts.

In an AWS application account, the company's application team has deployed a web application that uses AWS Lambda and Amazon RDS. The company's database administrators have a separate DBA account and use the account to centrally manage all the databases across the organization. The database administrators use an Amazon EC2 instance that is deployed in the DBA account to access an RDS database that is deployed in the application account.

The application team has stored the database credentials as secrets in AWS Secrets Manager in the application account. The application team is manually sharing the secrets with the database administrators. The secrets are encrypted by the default AWS managed key for Secrets Manager in the application account. A solutions architect needs to implement a solution that gives the database administrators access to the database and eliminates the need to manually share the secrets.

Which solution will meet these requirements?

Grant the role the required permissions to access the shared secrets. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.
B. In the application account, create an IAM role that is named DBA-Secret. Grant the role the required permissions to access the secrets. In the DBA account, create an IAM role that is named DBA-Admin. Grant the DBA-Admin role the required permissions to assume the DBA-Secret role in the application account. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.
C. In the DBA account, create an IAM role that is named DBA-Admin. Grant the role the required permissions to access the secrets and the default AWS managed key in the application account. In the application account, attach resource-based policies to the key to allow access from the DBA account. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.

# Cross-account access to Secrets Manager secrets (DBA ↔ Application account)

## Understanding the Problem

- Organization:
  - Multi-account setup with **AWS Organizations** and **AWS Control Tower**.
  - **Application account**:
    - Web app using **Lambda** + **Amazon RDS**.
    - DB credentials stored as **AWS Secrets Manager** secrets.
    - Secrets encrypted with **default AWS managed KMS key for Secrets Manager** in that account.
  - **DBA account**:
    - DBAs centrally manage all DBs.
    - Use an **EC2 instance** to access RDS databases in other accounts (here: the application account).

- Current issue:
  - Application team **manually shares** DB secrets with DBAs.
  - Need a solution to:
    - Let DBAs on the EC2 instance access the RDS database in the application account.
    - **Eliminate manual secret sharing**.
    - Use existing setup: secrets are in the application account.

- Core requirement:
  - Implement a **secure cross-account access pattern** so that:
    - DBA account IAM role (on EC2) can retrieve **Secrets Manager secrets** that live in the application account.
    - Proper handling of **KMS encryption** for those secrets.

## Services Explanation

### AWS Secrets Manager
- Stores, manages, rotates secrets (e.g., DB credentials).
- Secrets are encrypted with a **KMS key** (default AWS-managed key or customer-managed CMK).
- Supports **resource-based policies** for cross-account access, *but* only when using a **customer-managed KMS key** that also allows cross-account use.
- Docs:  
  - Secrets Manager concepts: https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html  
  - Cross-account secrets: https://docs.aws.amazon.com/secretsmanager/latest/userguide/auth-and-access_identity-based-policies.html#permissions_grant-get-secret-value-to-one-account

### AWS Key Management Service (AWS KMS)
- Manages keys that encrypt Secrets Manager data.
- **AWS managed keys** (e.g., `aws/secretsmanager`) are controlled by AWS and **cannot be shared cross-account** or modified to allow other accounts.
- **Customer managed keys (CMKs)** can have **key policies** that grant cross-account access.
- Docs:  
  - Key types: https://docs.aws.amazon.com/kms/latest/developerguide/concepts.html#key_mgmt  
  - Cross-account access with KMS: https://docs.aws.amazon.com/kms/latest/developerguide/key-policy-modifying-external-accounts.html  

### AWS Identity and Access Management (IAM)
- **Identity-based policies**: attached to roles/users, defining what they *can* do.
- **Role assumption (STS AssumeRole)**: principal in one account assumes a role in another account (classic multi-account pattern).
- For cross-account secrets:
  - One role in **application account** with `secretsmanager:GetSecretValue`.
  - A role in **DBA account** allowed to **assume** that role.

- Docs:  
  - Cross-account role access: https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_common-scenarios.html#iam-roles-common-scenarios-third-party  
  - Example for using roles with Secrets Manager: still based on standard IAM patterns.

### AWS Resource Access Manager (AWS RAM)
- Shares **supported resource types** (e.g., subnets, transit gateways, Route 53 Resolver rules) across AWS accounts.
- **Secrets Manager secrets are *not* shareable via AWS RAM**.
- Docs (resource types): https://docs.aws.amazon.com/ram/latest/userguide/shareable.html  

### AWS Organizations & Service Control Policies (SCPs)
- SCPs define **guardrails**: what actions are allowed/denied in member accounts.
- SCPs **do not grant permissions**; they only set *maximum possible* permissions.
- To access Secrets Manager, you still need:
  - IAM permissions (identity policies or role assumption) and
  - KMS permission (if encryption key is involved).
- Docs:  
  - SCPs: https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps.html

## Correct Option: **B**

> B. In the application account, create an IAM role that is named DBA-Secret. Grant the role the required permissions to access the secrets. In the DBA account, create an IAM role that is named DBA-Admin. Grant the DBA-Admin role the required permissions to assume the DBA-Secret role in the application account. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.

### Why this works

1. **Standard cross-account role pattern**  
   - A role in the **application account** (`DBA-Secret`) that:
     - Has an IAM policy allowing `secretsmanager:GetSecretValue` on the secret(s).
     - Can be assumed by a principal (role) from the DBA account (trust policy).
   - A role in the **DBA account** (`DBA-Admin`) that:
     - Has permission `sts:AssumeRole` on the application-account role ARN.
     - Is attached to the DBA EC2 instance (either via instance profile or using STS on the instance).

2. **KMS key compatibility**
   - The secret is encrypted with the **default AWS-managed key** for Secrets Manager in the application account.
   - That key is automatically usable by principals in the same account based on IAM permissions.  
   - If the DBA uses `AssumeRole` to assume `DBA-Secret` in the **application account**, they become an IAM principal **inside the application account**.
   - Thus, the `aws/secretsmanager` key can be used transparently; no cross-account KMS policy is required.
   - AWS-managed keys: “You cannot change their key policies or share them with other AWS accounts” — hence you must bring the caller into the account via role assumption.  
     - Ref: https://docs.aws.amazon.com/kms/latest/developerguide/concepts.html#aws-managed-cmk

3. **No manual sharing of secrets**
   - DBAs retrieve secrets at runtime through the assumed role, removing any out-of-band/manual secret exchange.

4. **Security & least privilege**
   - You tightly control:
     - Which secrets are accessible,
     - Which role/account can assume `DBA-Secret`,
     - And from which EC2 instances (via IAM role attachment and possibly Conditions).

## Insightful Explanation of Incorrect Options

### Option A

> A. Use AWS Resource Access Manager (AWS RAM) to share the secrets ...

**Why it’s wrong:**

- **Secrets Manager secrets are not shareable via AWS RAM.**
  - AWS RAM supports specific resource types (e.g., subnets, transit gateways, etc.).
  - Secrets **are not in the supported list**.
  - Doc: https://docs.aws.amazon.com/ram/latest/userguide/shareable.html
- Even if RAM could share the secret resource, the underlying key is a **default AWS-managed key**, which cannot be configured for cross-account access.  
- Therefore, this option is technically invalid.

**Services involved:**
- AWS RAM – not applicable to Secrets Manager here.
- IAM – attaching role to EC2 is fine, but fundamental secret sharing mechanism is unsupported.

### Option C

> C. In the DBA account, create an IAM role that is named DBA-Admin. Grant the role the required permissions to access the secrets and the default AWS managed key in the application account. In the application account, attach resource-based policies to the key to allow access from the DBA account. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.

**Why it’s wrong:**

- This suggests:
  - Giving the DBA-Admin role permissions to use the **default AWS-managed key** in the application account.
  - Adding **resource-based policies** to that KMS key for cross-account access.
- Problem:
  - **Default AWS-managed KMS keys** (like the Secrets Manager AWS-managed key) **do not have editable key policies** and **cannot be shared cross-account**.
  - You **cannot** attach resource-based policies to AWS-managed keys in the way described.
  - To share KMS across accounts, you must use a **customer-managed CMK**, not an AWS-managed key.
  - Docs:  
    - “You cannot change the key policy or delete AWS managed keys” and they are not for cross-account sharing.  
    - https://docs.aws.amazon.com/kms/latest/developerguide/concepts.html#aws-managed-cmk

**Additional nuance:**
- If the question said the secrets used a **customer-managed CMK**, then a solution like this could be crafted (with a correct policy design). But the question explicitly states the **default AWS managed key**, so this route is blocked.

- --

### Option D

> D. In the DBA account, create an IAM role that is named DBA-Admin. Grant the role the required permissions to access the secrets in the application account. Attach an SCP to the application account to allow access to the secrets from the DBA account. Attach the DBA-Admin role to the EC2 instance for access to the cross-account secrets.

**Why it’s wrong:**

1. **Misuse of SCPs**
   - **Service Control Policies (SCPs) do not grant permissions**.  
   - SCPs only define what is **allowed or denied at the organization level**. The actual permissions still must be granted using IAM identity-based and resource-based policies.
   - Simply “attaching an SCP to allow access” does not make the DBA role able to read secrets.
   - Docs: https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps.html

2. **Cross-account IAM logic missing**
   - For a principal in DBA account to access a secret in application account:
     - You either:  
       - Use a resource-based policy on the secret + cross-account KMS config (which is impossible with AWS-managed key), or  
       - Use **AssumeRole** into the application account (as in option B).
   - D only mentions an IAM role in the DBA account and an SCP but no proper cross-account trust or KMS handling.

3. **KMS issue remains**
   - Even if somehow IAM allowed `secretsmanager:GetSecretValue` directly from the DBA account, the underlying **AWS-managed key** still blocks cross-account decryption.
   - SCPs don’t change KMS key policies; they just constrain allowed actions.

## Key Takeaways for the SAP-C02 Exam

1. **Default AWS-managed keys** (e.g., `aws/secretsmanager`):
   - Cannot be customized or shared cross-account.
   - If you need cross-account access to Secrets Manager, **assume a role into the account where the secret lives**.

2. **Cross-account IAM best practice**:
   - Use **role assumption (STS AssumeRole)**, not direct cross-account IAM principal access where KMS is involved with AWS-managed keys.

3. **SCPs**:
   - Are **guardrails**, not permission grants.

4. **AWS RAM**:
   - Only supports specific resource types. Always check the supported list before assuming a resource can be shared that way.

5. For cross-account secrets:
   - Option A (RAM) – not supported.
   - Option C (resource-based KMS policy on AWS-managed key) – not possible.
   - Option D (SCP) – misunderstanding of SCPs and no proper KMS handling.
   - **Option B** – correct and aligns with IAM, KMS, and Secrets Manager behavior.
