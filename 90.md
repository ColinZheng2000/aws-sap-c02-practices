A large company is running a popular web application. The application runs on several Amazon EC2 Linux instances in an Auto Scaling group in a private subnet.  
An Application Load Balancer is targeting the instances in the Auto Scaling group in the private subnet. AWS Systems Manager Session Manager is configured, and AWS Systems Manager Agent is running on all the EC2 instances.  
The company recently released a new version of the application. Some EC2 instances are now being marked as unhealthy and are being terminated. As a result, the application is running at reduced capacity. A solutions architect tries to determine the root cause by analyzing Amazon CloudWatch logs that are collected from the application, but the logs are inconclusive.  
How should the solutions architect gain access to an EC2 instance to troubleshoot the issue?  

- A. Suspend the Auto Scaling group's HealthCheck scaling process. Use Session Manager to log in to an instance that is marked as unhealthy.
- B. Enable EC2 instance termination protection. Use Session Manager to log in to an instance that is marked as unhealthy.
- C. Set the termination policy to OldestInstance on the Auto Scaling group. Use Session Manager to log in to an instance that is marked an unhealthy.
- D. Suspend the Auto Scaling group's Terminate process. Use Session Manager to log in to an instance that is marked as unhealthy.

## 💡 Understanding the Problem

A **popular web application** runs on **Amazon EC2 Linux instances** within an **Auto Scaling group (ASG)** in a **private subnet**, using an **Application Load Balancer (ALB)**. The ASG is configured for health checks, and instances are being **marked as unhealthy and terminated** after a new application version deployment, leading to reduced capacity. Logs are inconclusive, and the Solutions Architect needs to **access an unhealthy, running instance for troubleshooting** without having it terminated by the ASG. **AWS Systems Manager Session Manager** is the mandated access method.

The core challenge is finding the right mechanism to **prevent the ASG from terminating the specific unhealthy instance** before the architect can connect and diagnose the root cause of the application failure.

## ✅ Correct Option Explanation

### D. Suspend the Auto Scaling group's Terminate process. Use Session Manager to log in to an instance that is marked as unhealthy.

* **Suspend the ASG's Terminate process:** The **`Terminate`** process is the specific ASG action responsible for *removing* instances marked as unhealthy (or scaling down). By suspending this process, the ASG will **stop terminating** any instances, including the unhealthy ones, but will continue to monitor their health. This preserves the unhealthy instance for troubleshooting.
* **Use Session Manager to log in:** Since the instances are in a private subnet and Session Manager is configured, this is the **secure and intended method** for accessing the instance to inspect application logs, configuration, and perform real-time diagnosis.

This approach is detailed in the **AWS documentation for Auto Scaling lifecycle hooks and troubleshooting**. Suspending the `Terminate` process is the standard, least-disruptive way to stop ASG termination actions temporarily for diagnosis.

## ❌ Incorrect Options Explanation

### A. Suspend the Auto Scaling group's HealthCheck scaling process. Use Session Manager to log in to an instance that is marked as unhealthy.

* **Incorrect:** Suspending the **`HealthCheck`** process would prevent the ASG from *marking an instance as unhealthy* in the future. However, the problem states that instances are *already* being marked unhealthy and terminated. Crucially, the ASG still relies on the **`Terminate`** process to remove instances that are *already* marked unhealthy or that are needed for scaling down. This action is unlikely to stop the termination of an already-unhealthy instance and might hide future issues.

### B. Enable EC2 instance termination protection. Use Session Manager to log in to an instance that is marked as unhealthy.

* **Incorrect:** **EC2 instance termination protection** prevents a user from *manually* terminating an EC2 instance using the EC2 console, API, or CLI. It **does not prevent Auto Scaling from terminating an instance**. The ASG is designed to override this protection when it needs to terminate an instance as part of its scaling or health management processes.
    * *Reference:* AWS documentation explicitly states that ASG can terminate instances regardless of the `DisableApiTermination` setting.

### C. Set the termination policy to OldestInstance on the Auto Scaling group. Use Session Manager to log in to an instance that is marked an unhealthy.

* **Incorrect:** The **termination policy** dictates *which* instance is terminated when a scale-in event (reduction in capacity) occurs or when multiple instances are marked unhealthy. Setting it to `OldestInstance` might change *which* healthy instance is chosen for scale-in, but it does **not prevent the ASG from terminating instances marked as unhealthy** due to failed health checks. The primary termination trigger in this scenario is the unhealthy state, not a general scale-in event.
