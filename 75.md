A video processing company wants to build a machine learning (ML) model by using 600 TB of compressed data that is stored as thousands of files in the company's on-premises network attached storage system. The company does not have the necessary compute resources on premises for ML experiments and wants to use AWS.  
The company needs to complete the data transfer to AWS within 3 weeks. The data transfer will be a one-time transfer. The data must be encrypted in transit. The measured upload speed of the company's internet connection is 100 Mbps, and multiple departments share the connection.  
Which solution will meet these requirements MOST cost-effectively?  

- A. Order several AWS Snowball Edge Storage Optimized devices by using the AWS Management Console. Configure the devices with a destination S3 bucket. Copy the data to the devices. Ship the devices back to AWS.
- B. Set up a 10 Gbps AWS Direct Connect connection between the company location and the nearest AWS Region. Transfer the data over a VPN connection into the Region to store the data in Amazon S3.
- C. Create a VPN connection between the on-premises network storage and the nearest AWS Region. Transfer the data over the VPN connection.
- D. Deploy an AWS Storage Gateway file gateway on premises. Configure the file gateway with a destination S3 bucket. Copy the data to the file gateway.

## 💡 Analysis

### Understanding the Problem

The core of this question is a classic AWS data migration scenario with three critical constraints:
1.  **Large Data Volume:** 600 TB
2.  **Tight Timeline:** Within 3 weeks (21 days)
3.  **Slow Network:** 100 Mbps, shared

The solution must also be **cost-effective** and **encrypted in transit**.

The first step is to calculate the feasibility of any network-based transfer.

**Data Transfer Calculation:**
* **Data Size:** 600 TB
* **Network Speed:** 100 Mbps (Megabits per second)

Let's convert 600 TB to Megabits:
* 600 TB = 600 * 1,024 GB = 614,400 GB
* 614,400 GB = 614,400 * 1,024 MB = 629,145,600 MB
* 629,145,600 MB * 8 bits/byte = 5,033,164,800 Megabits (Mb)

Now, let's find the time to transfer this over a 100 Mbps line (assuming 100% efficiency, which is unrealistic):
* Time (seconds) = 5,033,164,800 Mb / 100 Mbps = 50,331,648 seconds
* Time (days) = 50,331,648 / (60 sec * 60 min * 24 hr) = **~582.5 days**

This simple calculation shows that it would take **over 1.5 years** to transfer the data over the existing internet connection. The 3-week deadline makes any solution relying on this 100 Mbps connection (Options C and D) impossible.

We must find a solution that either bypasses this network (offline transfer) or establishes a much faster one.

---

### Services Explanation

* **AWS Snowball Edge Storage Optimized:** These are physical, rugged devices shipped to your data center. You connect them to your local network, transfer data at high speed (e.g., 10 GbE), and ship them back to AWS. AWS then uploads the data from the device directly into your S3 bucket. This is an **offline transfer** method. The Storage Optimized models provide 80 TB of usable S3-compatible storage. Data is encrypted on the device (256-bit encryption), and the device itself is secure, providing encryption "in transit."
* **AWS Direct Connect (DX):** This service establishes a dedicated, private network connection from your on-premises data center to an AWS Direct Connect location. It bypasses the public internet and provides high, consistent bandwidth (1 Gbps, 10 Gbps, 100 Gbps). It's excellent for hybrid workloads but is expensive and takes **weeks or months to provision**, as it involves physical cross-connects with an AWS partner.
* **AWS Site-to-Site VPN:** This service creates an encrypted tunnel (IPsec) between your on-premises network and your AWS VPC. However, this tunnel runs **over your existing internet connection**, so its maximum speed is limited by your 100 Mbps line.
* **AWS Storage Gateway (File Gateway):** This is a hybrid-cloud service. A virtual or hardware appliance on-premises provides a standard file interface (NFS/SMB) that stores data in Amazon S3. While it's a great tool for migrating file servers or for hybrid access, it **uses your internet connection** to upload data.

---

### Correct Option(s) Explanation

**A. Order several AWS Snowball Edge Storage Optimized devices...**
* **Timeline:** This is the *only* solution that meets the 3-week deadline. The transfer time is limited only by your local network speed and the shipping time. A typical Snowball job (including shipping, local data copy, and AWS ingestion) can be completed in 1-2 weeks.
* **Volume:** 600 TB / 80 TB (per device) = 7.5. Ordering "several" (e.g., 8) devices is the correct approach.
* **Cost:** For a **one-time** transfer, the per-job service fee for Snowball devices is far more cost-effective than provisioning a 10 Gbps Direct Connect circuit, which has high monthly port fees and requires a carrier circuit.
* **Encryption:** Snowball provides end-to-end encryption and a secure, tamper-evident appliance, meeting the "encrypted in transit" requirement.

As stated in the AWS Snowball documentation, "For large-scale data migrations, AWS Snowball is a good choice. It addresses common challenges with large-scale data transfers including high network costs, long transfer times, and security concerns."

---

### Incorrect Options Explanation

* **B. Set up a 10 Gbps AWS Direct Connect connection...**
    * **Fails on Timeline:** Provisioning a new 10 Gbps dedicated Direct Connect connection is a complex process that *cannot* be completed within 3 weeks. It often takes 4-12 weeks or more.
    * **Fails on Cost:** This is the *least* cost-effective solution for a *one-time* transfer. It is a very expensive, long-term-contract service.

* **C. Create a VPN connection...**
    * **Fails on Timeline:** A VPN uses the existing 100 Mbps connection. As calculated, this would take **~582.5 days**, massively missing the 3-week deadline.

* **D. Deploy an AWS Storage Gateway file gateway...**
    * **Fails on Timeline:** A File Gateway *also* uses the existing 100 Mbps connection to upload the data. This solution would face the same bottleneck as the VPN and take **~582.5 days**.
