A company is using an on-premises Active Directory service for user authentication. The company wants to use the same authentication service to sign in to the company’s AWS accounts, which are using AWS Organizations. AWS Site-to-Site VPN connectivity already exists between the on-premises environment and all the company’s AWS accounts.  
The company’s security policy requires conditional access to the accounts based on user groups and roles. User identities must be managed in a single location.  
Which solution will meet these requirements?

- A. Configure AWS IAM Identity Center (AWS Single Sign-On) to connect to Active Directory by using SAML 2.0. Enable automatic provisioning by using the System for Cross-domain Identity Management (SCIM) v2.0 protocol. Grant access to the AWS accounts by using attribute-based access controls (ABACs).
- B. Configure AWS IAM Identity Center (AWS Single Sign-On) by using IAM Identity Center as an identity source. Enable automatic provisioning by using the System for Cross-domain Identity Management (SCIM) v2.0 protocol. Grant access to the AWS accounts by using IAM Identity Center permission sets.
- C. In one of the company’s AWS accounts, configure AWS Identity and Access Management (IAM) to use a SAML 2.0 identity provider. Provision IAM users that are mapped to the federated users. Grant access that corresponds to appropriate groups in Active Directory. Grant access to the required AWS accounts by using cross-account IAM users.
- D. In one of the company’s AWS accounts, configure AWS Identity and Access Management (IAM) to use an OpenID Connect (OIDC) identity provider. Provision IAM roles that grant access to the AWS account for the federated users that correspond to appropriate groups in Active Directory. Grant access to the required AWS accounts by using cross-account IAM roles.

## Understanding the Problem

The company needs a solution to **federate** an **on-premises Active Directory (AD)** for user authentication into **multiple AWS accounts** managed by **AWS Organizations**.

Key requirements are:
1.  Use the **on-premises AD** for **authentication**.
2.  The solution must work across **all AWS accounts** (suggesting a centralized, multi-account approach).
3.  Conditional access based on **user groups and roles** (requires attribute-based access control or similar fine-grained control).
4.  **Single location** for user identity management (centralized identity source).
5.  **Site-to-Site VPN** connectivity already exists (though this is secondary, it confirms network access for the solution).

The best solution must be central, integrate with AD for authentication, support conditional access, and be scalable across an AWS Organization.

## Correct Option(s) Explanation

### **A. Configure AWS IAM Identity Center (AWS Single Sign-On) to connect to Active Directory by using SAML 2.0. Enable automatic provisioning by using the System for Cross-domain Identity Management (SCIM) v2.0 protocol. Grant access to the AWS accounts by using attribute-based access controls (ABACs).**

This is the **correct and best solution** as it meets all requirements:
* **Centralized and Multi-Account:** **AWS IAM Identity Center** is the native, central service designed for managing access to multiple AWS accounts within an Organization.
* **Authentication:** Connecting to **Active Directory by using SAML 2.0** handles the authentication requirement.
* **Single Location for Identity Management:** **SCIM v2.0** enables **automatic provisioning** of users and groups from AD to IAM Identity Center, ensuring that user identities are managed in a single location (AD, with synchronization to IAM Identity Center).
* **Conditional Access/Groups and Roles:** **ABAC** allows granting access based on attributes (like AD group membership, synchronized via SCIM), directly fulfilling the requirement for **conditional access based on user groups and roles**.

This combination of services is the **official AWS best practice** for federating an external identity store to an AWS Organization for centralized, scalable access management. (Refer to the AWS IAM Identity Center User Guide - "Connect your external identity provider to IAM Identity Center").

## Incorrect Options Explanation

### **B. Configure AWS IAM Identity Center (AWS Single Sign-On) by using IAM Identity Center as an identity source. Enable automatic provisioning by using the System for Cross-domain Identity Management (SCIM) v2.0 protocol. Grant access to the AWS accounts by using IAM Identity Center permission sets.**

* **Incorrect Identity Source:** This option suggests using **IAM Identity Center as an identity source**, which means the users would be managed *within* IAM Identity Center's own directory, *not* the on-premises Active Directory. The requirement is to use the **on-premises Active Directory service** for user authentication.

### **C. In one of the company’s AWS accounts, configure AWS Identity and Access Management (IAM) to use a SAML 2.0 identity provider. Provision IAM users that are mapped to the federated users. Grant access that corresponds to appropriate groups in Active Directory. Grant access to the required AWS accounts by using cross-account IAM users.**

* **Not Centralized/Scalable:** This approach is **account-specific** (using IAM in one account). It also involves **provisioning IAM users**, which is an older, non-preferred method for federation. The modern, scalable method for Organizations is to use IAM Identity Center, which creates temporary session credentials rather than permanent IAM users.
* **Incorrect Cross-Account Method:** The concept of "cross-account IAM users" is generally not how federation works across accounts; cross-account *roles* are used for this in traditional IAM federation, and IAM Identity Center handles it implicitly.

### **D. In one of the company’s AWS accounts, configure AWS Identity and Access Management (IAM) to use an OpenID Connect (OIDC) identity provider. Provision IAM roles that grant access to the AWS account for the federated users that correspond to appropriate groups in Active Directory. Grant access to the required AWS accounts by using cross-account IAM roles.**

* **Not Centralized/Scalable:** Similar to C, this is an **account-specific** (using IAM in one account) solution, not the centralized, Organization-wide solution required. While creating IAM roles for federation is better than provisioning users, doing this *individually in every single account* is not scalable or centrally managed.
* **Protocol Mismatch:** While OIDC *can* be used, SAML 2.0 is the standard and best fit for integrating with Active Directory for this enterprise-level scenario, which IAM Identity Center is specifically designed to handle natively. The overall architecture is the primary flaw.
