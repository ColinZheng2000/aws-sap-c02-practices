A company runs an application in an on-premises data center. The application gives users the ability to upload media files. The files persist in a file server. The web application has many users. The application server is overutilized, which causes data uploads to fail occasionally. The company frequently adds new storage to the file server. The company wants to resolve these challenges by migrating the application to AWS.

Users from across the United States and Canada access the application. Only authenticated users should have the ability to access the application to upload files. The company will consider a solution that refactors the application, and the company needs to accelerate application development.

Which solution will meet these requirements with the LEAST operational overhead?

A. Use AWS Application Migration Service to migrate the application server to Amazon EC2 instances. Create an Auto Scaling group for the EC2 instances. Use an Application Load Balancer to distribute the requests. Modify the application to use Amazon S3 to persist the files. Use Amazon Cognito to authenticate users.
B. Use AWS Application Migration Service to migrate the application server to Amazon EC2 instances. Create an Auto Scaling group for the EC2 instances. Use an Application Load Balancer to distribute the requests. Set up AWS IAM Identity Center (AWS Single Sign-On) to give users the ability to sign in to the application. Modify the application to use Amazon S3 to persist the files.
C. Create a static website for uploads of media files. Store the static assets in Amazon S3. Use AWS AppSync to create an API. Use AWS Lambda resolvers to upload the media files to Amazon S3. Use Amazon Cognito to authenticate users.
D. Use AWS Amplify to create a static website for uploads of media files. Use Amplify Hosting to serve the website through Amazon CloudFront. Use Amazon S3 to store the uploaded media files. Use Amazon Cognito to authenticate users.


Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company has an on-premises web application that handles media file uploads from many users. The current architecture suffers from three pain points: (1) the application server is overutilized causing upload failures, (2) frequent storage expansion on the file server is operationally burdensome, and (3) the company wants to **refactor** (not just lift-and-shift) while **accelerating development** with the **least operational overhead**. Users span across the US and Canada, and only authenticated users may upload files. The core ask is a migration strategy that maximizes agility and minimizes infrastructure management — a strong signal for serverless/managed services.

### Correct Option: D — AWS Amplify + Cognito + S3 + CloudFront

**Why D is correct:**

- **AWS Amplify** is a fully managed development platform that provides a complete CI/CD pipeline, hosting, authentication, and storage integration out of the box. It directly addresses the "accelerate application development" requirement.
- **Amplify Hosting** serves the static website through **Amazon CloudFront**, a global CDN with edge locations across North America, ensuring low-latency access for users in the US and Canada.
- **Amazon Cognito** provides fully managed user authentication, user pools, and federation — directly meeting the "only authenticated users should upload" requirement without managing any auth infrastructure.
- **Amazon S3** serves as the durable, infinitely scalable storage backend for uploaded media files, eliminating the need to provision or manage file servers (solving the "frequently adds new storage" pain point).
- **Least operational overhead**: Amplify is serverless — no EC2 instances to patch, no Auto Scaling groups to tune, no load balancers to configure. The entire stack is managed by AWS.
- Amplify's built-in CI/CD (connected to a Git repository) automatically builds and deploys the application on every code commit, aligning with the need to accelerate development.

**Official reference**: [AWS Amplify Documentation](https://docs.aws.amazon.com/amplify/latest/userguide/welcome.html) — "AWS Amplify is a complete solution that lets frontend web and mobile developers easily build, ship, and host full-stack applications on AWS."

### Incorrect Options

**A — Application Migration Service (MGN) → EC2 + ASG + ALB + S3 + Cognito**

- **AWS Application Migration Service (MGN)** is designed for **lift-and-shift (rehost)** migrations, not refactoring. The question explicitly states the company "will consider a solution that refactors the application."
- EC2 + ASG + ALB introduces significant operational overhead: patching AMIs, managing scaling policies, configuring security groups, maintaining the ALB, etc. This contradicts "LEAST operational overhead."
- While Cognito and S3 are correct service choices, the overall architecture is too infrastructure-heavy for a refactor scenario that prioritizes speed and low ops burden.

**B — Application Migration Service → EC2 + ASG + ALB + IAM Identity Center + S3**

- Same MGN lift-and-shift problem as Option A.
- **AWS IAM Identity Center (formerly AWS SSO)** is designed for **workforce identity** — managing employee access to AWS accounts and business applications. It is **not** an application end-user authentication service. For authenticating external application users, Amazon Cognito is the appropriate service.
- IAM Identity Center integrates with corporate directories (Microsoft AD, Okta, etc.) for employee SSO, not for customer-facing authentication workflows like sign-up, sign-in, and MFA for a public web application.

**C — Static website on S3 + AppSync + Lambda + Cognito**

- **AWS AppSync** is a managed GraphQL service. While it can technically handle file uploads via GraphQL mutations backed by Lambda resolvers → S3, this architecture introduces unnecessary complexity. AppSync is optimized for real-time data synchronization and complex GraphQL queries, not for simple file upload workflows.
- Requires manually wiring together S3 static hosting, AppSync API, Lambda resolvers, and Cognito — this is more operational overhead and slower to develop than Amplify, which provides these integrations pre-configured.
- S3 static website hosting alone (without CloudFront) does not provide the same level of global edge caching and HTTPS performance as Amplify Hosting + CloudFront. For a user base across US and Canada, CDN acceleration matters.

### Hands-On Enterprise Knowledge

1. **Amplify vs. manually assembled serverless**: In enterprise settings, AWS Amplify dramatically reduces the "undifferentiated heavy lifting" of connecting Cognito, S3, API Gateway/Lambda, and CloudFront. A single `amplify add auth` and `amplify add storage` CLI command configures IAM roles, Cognito user pools, S3 bucket policies, and CloudFront distributions in minutes — work that would otherwise take days of writing CloudFormation/Terraform.

2. **Refactor (re-architect) vs. rehost**: The SAP-C02 exam frequently tests your ability to distinguish migration strategies (the 6 R's). When the question says "refactors the application" and "least operational overhead," **avoid EC2-based solutions**. Serverless architectures (Amplify, Lambda, API Gateway, DynamoDB, S3) are the canonical answer pattern.

3. **Cognito vs. IAM Identity Center**: Memorize this distinction:
   - **Cognito** = external customer/application end-user identity (user pools, social logins, OIDC/SAML for app users)
   - **IAM Identity Center** = internal workforce identity (employee SSO to AWS accounts and cloud applications)

4. **S3 for media storage**: When migrating from on-premises file servers to AWS for user-generated content, S3 is almost always the answer. It provides 11 9's of durability, infinite scalability, lifecycle policies (e.g., transition to Glacier for old files), pre-signed URLs for secure uploads, and event notifications to trigger downstream processing (Lambda, SQS, SNS).

5. **Cost and performance**: For a geographically distributed user base (US + Canada), CloudFront reduces latency and data transfer costs. S3 transfer acceleration is an alternative but Amplify Hosting includes CloudFront by default, making it the simpler choice.
