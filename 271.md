A solutions architect wants to make sure that only AWS users or roles with suitable permissions can access a new Amazon API Gateway endpoint. The solutions architect wants an end-to-end view of each request to analyze the latency of the request and create service maps.

How can the solutions architect design the API Gateway access control and perform request inspections?

A. For the API Gateway method, set the authorization to AWS_IAM. Then, give the IAM user or role execute-api:Invoke permission on the REST API resource. Enable the API caller to sign requests with AWS Signature when accessing the endpoint. Use AWS X-Ray to trace and analyze user requests to API Gateway.
B. For the API Gateway resource, set CORS to enabled and only return the company's domain in Access-Control-Allow-Origin headers. Then, give the IAM user or role execute-api:Invoke permission on the REST API resource. Use Amazon CloudWatch to trace and analyze user requests to API Gateway.
C. Create an AWS Lambda function as the custom authorizer, ask the API client to pass the key and secret when making the call, and then use Lambda to validate the key/secret pair against the IAM system. Use AWS X-Ray to trace and analyze user requests to API Gateway.
D. Create a client certificate for API Gateway. Distribute the certificate to the AWS users and roles that need to access the endpoint. Enable the API caller to pass the client certificate when accessing the endpoint. Use Amazon CloudWatch to trace and analyze user requests to API Gateway.

## Understanding the Problem
Two separate requirements must be met:

1. **Access control**: Only AWS users/roles with appropriate permissions can invoke the API Gateway endpoint.
   - The most direct and AWS-native way is **IAM authorization** on API Gateway, enforced by `execute-api:Invoke` permissions and **SigV4** request signing.

2. **End-to-end request view with latency breakdowns and service maps**
   - This is specifically what **AWS X-Ray** provides (distributed tracing, segment/subsegment timings, and **service map** visualization).
   - CloudWatch alone provides logs/metrics but does **not** provide X-Ray style end-to-end traces and service maps.

So the design should be **API Gateway IAM auth + SigV4** and **X-Ray tracing**.

## Correct Option Explanation

### ✅Option A
* *Access control (correct)**
- Set the API Gateway method authorization to **`AWSAM`**.
- Grant principals (IAM users/roles) the permission **`execute-api:Invoke`** scoped to the API/stage/method/resource ARN.
- Require callers to sign requests using **AWS Signature Version 4 (SigV4)**. This ties access to valid AWS credentials and IAM policy evaluation.

* *Request inspection / end-to-end (correct)**
- Enable and use **AWS X-Ray** with API Gateway to trace requests and analyze latency.
- X-Ray provides:
  - Trace timeline (end-to-end)
  - Latency per hop (API Gateway → Lambda/ECS/HTTP backends, etc.)
  - **Service maps**

## Incorrect Options Explanation

### ❌Option B
- **CORS is not an authorization mechanism**. It is a browser enforcement feature and does not prevent non-browser clients (curl, scripts, other services) from calling your API directly.
- Even with `execute-api:Invoke`, without using IAM authorization (`AWS_IAM`) at the method, you not actually enforcing IAM auth at the API Gateway front door.
- **CloudWatch** (metrics/logs) does not provide **service maps** or true distributed tracing like X-Ray.

### ❌Option C
- A Lambda authorizer is valid for custom auth, but this option proposes passing “key and secret” and validating against.
  - That’s not how IAM credentials should be handled; you should not build a custom system where clients pass secrets to an authorizer to be validated “against IAM.”
  - It increases operational and security risk (credential handling, rotation, leakage) and duplicates what SigV4 + IAM already provides.
- X-Ray part is fine, but the access control design is unnecessarily complex and not best practice for “only AWS users/roles”.

### ❌Option D
- API **client certificates** are used for **mutual TLS** on *custom domains* (mTLS) or for certain backend integrations, but they are **not mechanism to distribute to “IAM users and roles”** (roles are not entities that hold client certs).
- Certificates do not integrate with IAM authorization in the way the requirement states (“only AWS users/roles with suitable permissions”).
- CloudWatch again does not satisfy the **service maps** requirement (X-Ray does).

 Hands-on Knowledge (what to do in real deployments)
- **Use least-privilege `execute-api:Invoke` policies**:
  - Scope by stage/method/resource; avoid `execute-api:/*` unless necessary.
  - Consider separate roles for CI/CD, internal services, and human admins.

- **Prefer IAM auth for-to-service calls**:
  - For workloads running on AWS (ECS tasks, Lambda, EC2), SigV4 signing is straightforward using SDKs and instance/task roles.
  - If external customers must call the API, IAM auth may be awkward; you’d typically use **Cognito/JWT authorizers** or a custom IdP—different than this question.

- **Enable X-Ray sampling thoughtfully**:
  - Default sampling may miss bursts or low-volume paths. Adjust sampling rules for critical endpoints.
  - Ensure downstream services (Lambda, ECS app via X-Ray SDK/daemon) propagate the trace header so you actually get end-to-end traces.

- **Operational debugging combo**:
  - X-Ray for latency + dependency map
  - CloudWatch Logs (access logs/execution logs) for request details and failures
  - CloudWatch Metrics/Alarms for SLO monitoring
