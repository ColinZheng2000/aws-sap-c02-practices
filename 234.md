A company implements a containerized application by using Amazon Elastic Container Service (Amazon ECS) and Amazon API Gateway. The application data is stored in Amazon Aurora databases and Amazon DynamoDB databases. The company automates infrastructure provisioning by using AWS CloudFormation. The company automates application deployment by using AWS CodePipeline.

A solutions architect needs to implement a disaster recovery (DR) strategy that meets an RPO of 2 hours and an RTO of 4 hours.

Which solution will meet these requirements MOST cost-effectively?

A. Set up an Aurora global database and DynamoDB global tables to replicate the databases to a secondary AWS Region. In the primary Region and in the secondary Region, configure an API Gateway API with a Regional endpoint. Implement Amazon CloudFront with origin failover to route traffic to the secondary Region during a DR scenario.
B. Use AWS Database Migration Service (AWS DMS), Amazon EventBridge (Amazon CloudWatch Events), and AWS Lambda to replicate the Aurora databases to a secondary AWS Region. Use DynamoDB Streams, EventBridge (CloudWatch Events), and Lambda to replicate the DynamoDB databases to the secondary Region. In the primary Region and in the secondary Region, configure an API Gateway API with a Regional endpoint. Implement Amazon Route 53 failover routing to switch traffic from the primary Region to the secondary Region.
C. Use AWS Backup to create backups of the Aurora databases and the DynamoDB databases in a secondary AWS Region. In the primary Region and in the secondary Region, configure an API Gateway API with a Regional endpoint. Implement Amazon Route 53 failover routing to switch traffic from the primary Region to the secondary Region.
D. Set up an Aurora global database and DynamoDB global tables to replicate the databases to a secondary AWS Region. In the primary Region and in the secondary Region, configure an API Gateway API with a Regional endpoint. Implement Amazon Route 53 failover routing to switch traffic from the primary Region to the secondary Region.



Suggest Answer: C

## Analysis

### Understanding the Problem

A company hosts a containerized application on Amazon ECS with API Gateway as the frontend, backed by both Amazon Aurora (relational) and Amazon DynamoDB (NoSQL) databases. Infrastructure is provisioned via CloudFormation, and CI/CD uses CodePipeline. The business requires a **disaster recovery (DR) strategy** with:

- **RPO (Recovery Point Objective): 2 hours** — maximum acceptable data loss is 2 hours of transactions.
- **RTO (Recovery Time Objective): 4 hours** — the application must be fully operational in the secondary Region within 4 hours of a disaster declaration.

The solution must be the **MOST cost-effective** option that meets both objectives. The key tension here is that the RPO/RTO values are moderate (not zero, not days), which means real-time replication solutions (Aurora Global Database, DynamoDB Global Tables) may be **overkill and overpriced** for the requirements.

### Correct Option(s) Explanation

**Option C (Correct):** Use **AWS Backup** to create backups of both Aurora and DynamoDB databases to a **secondary AWS Region**. Configure API Gateway with Regional endpoints in both Regions, and use **Amazon Route 53 failover routing** to switch traffic during a DR event.

**Why this works:**

- **AWS Backup** is a fully managed, central backup service that supports both Aurora and DynamoDB. You can create a **backup plan** with a schedule (e.g., every 2 hours) and set the destination to a secondary Region for cross-region copy.
- **RPO of 2 hours is satisfied:** AWS Backup supports hourly backup schedules. With a 2-hour backup frequency, the maximum data loss is at most 2 hours of transactions. For DynamoDB, AWS Backup provides continuous backup with point-in-time recovery (PITR), enabling restores to any point within the backup window.
- **RTO of 4 hours is satisfied:** Restoring from an AWS Backup snapshot for Aurora (which is based on RDS snapshots) and DynamoDB (which restores to a new table) can typically be completed well within 4 hours. The infrastructure (ECS, API Gateway) is already pre-deployed in the secondary Region via CloudFormation, so only data restoration is needed during DR.
- **Most cost-effective:** AWS Backup costs are based on backup storage consumed and data transferred across Regions. Compared to:
  - Aurora Global Database (charged per replicated write I/O, plus the cost of a full secondary cluster)
  - DynamoDB Global Tables (charged per replicated write unit plus storage in each Region)
  - AWS DMS with a running replication instance (24×7 instance cost + data transfer)
  
  **Backup-based DR has the lowest ongoing cost** when RPO tolerates hours rather than seconds. You pay only for backup storage and periodic cross-region data transfer, not continuous replication.

- **Route 53 failover routing** provides health-check-based DNS failover to the secondary Region's API Gateway endpoint.

- **Reference:** [AWS Backup Cross-Region Backup](https://docs.aws.amazon.com/aws-backup/latest/devguide/cross-region-backup.html) | [AWS Backup for DynamoDB](https://docs.aws.amazon.com/aws-backup/latest/devguide/dynamodb-backups.html) | [Route 53 DNS Failover](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/dns-failover.html)

### Incorrect Options Explanation

**Option A:** Aurora Global Database + DynamoDB Global Tables + CloudFront with origin failover.

- ❌ **RPO/RTO overkill:** Aurora Global Database provides sub-second replication (RPO near zero) and DynamoDB Global Tables offer multi-active (also near-zero RPO). These are designed for mission-critical, low-RTO workloads. Using them for a 2-hour RPO is **excessively expensive** — you pay for a full secondary Aurora cluster (compute + storage) and write replication units for DynamoDB continuously. This violates the "most cost-effective" requirement.
- ❌ **CloudFront origin failover** is designed for static/dynamic content failover at the CDN level, not for API Gateway Regional endpoints. This adds unnecessary architectural complexity.

**Option B:** AWS DMS + EventBridge + Lambda for Aurora replication, DynamoDB Streams + EventBridge + Lambda for DynamoDB replication, Route 53 failover.

- ❌ **High operational complexity:** This custom-built replication pipeline requires:
  - A **DMS replication instance** running 24×7 (significant cost)
  - Lambda functions to orchestrate replication
  - EventBridge rules to trigger processes
  - Monitoring and error handling for the entire custom pipeline
- ❌ **Cost:** The DMS replication instance alone (typically a `dms.c5.xlarge` or larger) represents a significant fixed monthly cost, even at modest throughput. This is more expensive than periodic AWS Backup snapshots.
- ❌ **Reliability risk:** A self-managed replication pipeline has more failure modes (Lambda timeouts, DMS task failures, schema mapping issues) compared to the fully managed AWS Backup service.

**Option D:** Aurora Global Database + DynamoDB Global Tables + Route 53 failover.

- ❌ Similar to Option A, this uses **premium replication technologies** (Global Database + Global Tables) for a modest RPO/RTO requirement. While Route 53 failover is appropriate, the database replication choice is disproportionately expensive. Aurora Global Database requires a full secondary DB cluster (compute instances + storage) running at all times, and DynamoDB Global Tables charge for every replicated write across Regions. For a 2-hour RPO, periodic backups are **far more economical**.

### Enterprise Hands-on Knowledge

1. **RPO/RTO Cost Spectrum:** As a rule of thumb for the SAP-C02 exam and real-world architecture:
   - **RPO < 1 minute, RTO < 15 minutes:** Aurora Global Database / DynamoDB Global Tables / Multi-Region active-active
   - **RPO 1–12 hours, RTO 1–8 hours:** AWS Backup with cross-region copy / scheduled RDS snapshots
   - **RPO > 12 hours, RTO > 8 hours:** Manual snapshots, pilot light strategy

2. **AWS Backup Plan Design for DR:** A well-designed cross-region backup plan includes:
   - **Backup rule** with cron schedule (e.g., `cron(0 */2 * * ? *)` for every 2 hours)
   - **Copy action** to the DR Region with a retention period aligned to compliance requirements
   - **Vault lock** in the DR Region to prevent backup deletion (ransomware protection)

3. **DR Runbook Essentials:** For a 4-hour RTO, you need a pre-tested runbook covering:
   - Trigger: Who declares DR and how (CloudWatch alarm, manual decision)?
   - Restore sequence: Aurora restore → DynamoDB restore → DNS failover with Route 53
   - Validation: Smoke tests against the secondary Region endpoints
   - **Pre-deploy ECS infrastructure** in the secondary Region via CloudFormation (infrastructure as code) so only data restoration is needed during an actual DR event.

4. **Common SAP-C02 Exam Insight:** The exam consistently pairs moderate RPO/RTO values with backup-based solutions and tight RPO/RTO values with replication-based solutions. If you see RPO measured in hours (not minutes), **AWS Backup** is typically the correct answer when "most cost-effective" is specified.

5. **Cost Comparison Reality:** At the time of writing, DynamoDB Global Tables charge approximately $0.02 per 1 million replicated write request units. For a write-heavy workload, this can exceed thousands of dollars per month. AWS Backup for DynamoDB charges per GB of backup storage (approximately $0.10/GB/month), making it orders of magnitude cheaper for workloads tolerant of hours-level RPO.
