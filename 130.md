A company is migrating a document processing workload to AWS. The company has updated many applications to natively use the Amazon S3 API to store, retrieve, and modify documents that a processing server generates at a rate of approximately 5 documents every second. After the document processing is finished, customers can download the documents directly from Amazon S3.  
  
During the migration, the company discovered that it could not immediately update the processing server that generates many documents to support the S3 API. The server runs on Linux and requires fast local access to the files that the server generates and modifies. When the server finishes processing, the files must be available to the public for download within 30 minutes.  
  
Which solution will meet these requirements with the LEAST amount of effort?

- A. Migrate the application to an AWS Lambda function. Use the AWS SDK for Java to generate, modify, and access the files that the company stores directly in Amazon S3.
- B. Set up an Amazon S3 File Gateway and configure a file share that is linked to the document store. Mount the file share on an Amazon EC2 instance by using NFS. When changes occur in Amazon S3, initiate a RefreshCache API call to update the S3 File Gateway.
- C. Configure Amazon FSx for Lustre with an import and export policy. Link the new file system to an S3 bucket. Install the Lustre client and mount the document store to an Amazon EC2 instance by using NFS.
- D. Configure AWS DataSync to connect to an Amazon EC2 instance. Configure a task to synchronize the generated files to and from Amazon S3.

# AWS SAP-C02 Question Analysis: Document Processing Workload Migration
## Understanding the Problem
The company is migrating a document processing workload to AWS with key constraints:
1. Most applications use the S3 API, but one Linux-based processing server **cannot be updated to support S3 API** (no code changes allowed).
2. The server requires **fast local access** to files it generates/modifies (5 documents/second).
3. Post-processing, files must be publicly available in S3 within 30 minutes (flexible sync window).
4. Solution must require the **least operational effort** (no app rewrites, minimal configuration).

## Services Explanation
### Core Services Involved
1. **Amazon S3**: Object storage for final document hosting (public downloads). Not natively compatible with file system access (requires gateways for POSIX/NFS support).
   - Official Reference: https://docs.aws.amazon.com/s3/latest/userguide/Welcome.html
2. **S3 File Gateway (Storage Gateway)**: Provides NFS/SMB file shares mapped to S3 buckets. Caches frequently accessed files locally for low-latency access, auto-syncs changes to S3. Enables legacy apps (without S3 API support) to use S3 as a file system.
   - Official Reference: https://docs.aws.amazon.com/storagegateway/latest/file/file-gateway.html
3. **Amazon FSx for Lustre**: Managed high-performance file system optimized for HPC, machine learning, and media processing. Supports NFS and S3 import/export but is designed for sub-millisecond latency and extreme throughput (overkill for standard document processing).
   - Official Reference: https://docs.aws.amazon.com/fsx/latest/LustreGuide/what-is.html
4. **AWS DataSync**: Service for high-speed data transfer between storage systems (e.g., EC2 EBS ↔ S3). Requires manual sync task configuration and does not provide a file system interface (apps must write to local storage first).
   - Official Reference: https://docs.aws.amazon.com/datasync/latest/userguide/what-is-datasync.html
5. **AWS Lambda**: Serverless compute for event-driven tasks. Has temporary storage limits (/tmp: 512MB–10GB) and requires app rewrites (incompatible with "no server updates" constraint).
   - Official Reference: https://docs.aws.amazon.com/lambda/latest/dg/configuration-function-common.html#configuration-tmp-space
6. **EC2 + NFS**: Linux servers can mount NFS shares (from S3 File Gateway/FSx) for file system access—no OS-level changes needed.

## Correct Option Explanation
### Option B (S3 File Gateway + EC2 NFS Mount)
This solution meets all requirements with minimal effort:
1. **No app modifications**: The Linux server uses standard NFS to mount the S3 File Gateway share (no S3 API integration required—matches the "cannot update server" constraint).
2. **Fast local access**: S3 File Gateway caches frequently accessed files locally (on the gateway instance or EC2), providing low-latency I/O for the server’s 5 documents/second generation rate.
3. **Auto-sync to S3**: Files written/modified via the NFS share are automatically synced to the linked S3 bucket (the `RefreshCache API` call is optional for force-updating cached data, but the gateway’s native sync handles the 30-minute window easily).
4. **Least effort**: No code rewrites (unlike Lambda), no overcomplicated HPC-focused setup (unlike FSx), and no manual sync task management (unlike DataSync).

AWS’s official guidance emphasizes S3 File Gateway for "legacy applications that require file system access to S3" (https://docs.aws.amazon.com/storagegateway/latest/file/use-cases.html), making it the optimal choice here.

## Incorrect Options Explanation
### Option A (Lambda + S3 SDK)
- **Flaw 1: High effort**: Migrating the processing server to Lambda requires full app rewrites (contradicts "cannot update the server" constraint) and S3 SDK integration (additional code changes).
- **Flaw 2: Storage limitation**: Lambda’s `/tmp` directory has size limits (max 10GB) and is ephemeral—unsuitable for generating/modifying documents at 5 per second (risk of storage exhaustion).
- **Conclusion**: Violates "least effort" and "no server updates" requirements.

### Option C (FSx for Lustre)
- **Flaw 1: Overkill**: FSx for Lustre is optimized for HPC workloads (sub-millisecond latency, 100GB/s throughput) — the server only needs "fast local access" (solved by S3 File Gateway’s cache) without extreme performance requirements.
- **Flaw 2: Higher effort**: Requires configuring Lustre file systems, import/export policies, and client installations—more complex than S3 File Gateway’s plug-and-play NFS setup.
- **Conclusion**: Unnecessary complexity; does not meet "least effort" requirement.

### Option D (DataSync + EC2)
- **Flaw 1: No file system access**: DataSync is a transfer tool, not a file system interface. The server would need to write files to EC2 EBS first (local storage) before DataSync syncs to S3—adding an extra layer of management.
- **Flaw 2: Sync overhead**: Requires scheduling/cron jobs for ongoing syncs (5 documents/second) and monitoring—more operational effort than S3 File Gateway’s auto-sync.
- **Conclusion**: Less efficient and higher effort than Option B.
