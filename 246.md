A company wants to use Amazon S3 to back up its on-premises file storage solution. The company's on-premises file storage solution supports NFS, and the company wants its new solution to support NFS. The company wants to archive the backup files after 5 days. If the company needs archived files for disaster recovery, the company is willing to wait a few days for the retrieval of those files.
Which solution meets these requirements MOST cost-effectively?

A. Deploy an AWS Storage Gateway files gateway that is associated with an S3 bucket. Move the files from the on-premises file storage solution to the file gateway. Create an S3 Lifecycle rule to move the file to S3 Standard-Infrequent Access (S3 Standard-IA) after 5 days.
B. Deploy an AWS Storage Gateway volume gateway that is associated with an S3 bucket. Move the files from the on-premises file storage solution to the volume gateway. Create an S3 Lifecycle rule to move the files to S3 Glacier Deep Archive after 5 days.
C. Deploy an AWS Storage Gateway tape gateway that is associated with an S3 bucket. Move the files from the on-premises file storage solution to the tape gateway. Create an S3 Lifecycle rule to move the files to S3 Standard-Infrequent Access (S3 Standard-IA) after 5 days.
D. Deploy an AWS Storage Gateway file gateway that is associated with an S3 bucket. Move the files from the on-premises file storage solution to the tape gateway. Create an S3 Lifecycle rule to move the files to S3 Standard-Infrequent Access (S3 Standard-IA) after 5 days.
E. Deploy an AWS Storage Gateway file gateway that is associated with an S3 bucket. Move the files from the on-premises file storage solution to the file gateway. Create an S3 Lifecycle rule to move the files to S3 Glacier Deep Archive after 5 days.


Suggest Answer: E

---

## 📊 Analysis

### 🔍 Understanding the Problem

A company has an **on-premises file storage solution** that supports **NFS** (Network File System). They want to migrate/backup this storage to **Amazon S3** while **preserving NFS compatibility** for the new solution. After **5 days**, files should be **archived**. For disaster recovery purposes, the company is **willing to wait a few days** for retrieval of archived files — this is the critical clue for selecting the appropriate S3 storage class.

Key requirements:
- NFS protocol support (both on-premises and the new AWS solution)
- Backup to Amazon S3 as the backing store
- Archive files after 5 days
- Retrieval time of "a few days" is acceptable for archived data
- Most cost-effective solution

### ✅ Correct Option: E — File Gateway → S3 → S3 Glacier Deep Archive after 5 days

**Why it's correct:**

Let's break down each component:

1. **AWS Storage Gateway — File Gateway**: 
   - Provides **NFS v3 and v4.1** (as well as SMB) protocol access to S3 data
   - Presents S3 buckets as locally mountable NFS shares to on-premises applications
   - Caches frequently accessed data locally for low-latency access
   - Data is durably stored in S3; the file gateway acts as a transparent bridge between NFS clients and S3
   - This is the **only Storage Gateway type that supports NFS natively**

2. **Amazon S3 Lifecycle Rule to S3 Glacier Deep Archive after 5 days**:
   - **S3 Glacier Deep Archive** is the **lowest-cost** S3 storage class, designed for long-term retention of data that is rarely accessed
   - Standard retrieval time: **12 to 48 hours** — perfectly aligns with the company's willingness to "wait a few days"
   - Cost: As low as **$0.00099 per GB/month** (as of 2025), making it the most cost-effective archival option
   - Durability: 99.999999999% (11 nines) — same as S3 Standard

3. **End-to-End Flow**: 
   - On-prem NFS clients → mount File Gateway (NFS) → data stored in S3 Standard
   - Day 5: S3 Lifecycle rule transitions objects to S3 Glacier Deep Archive
   - DR scenario: Initiate restore request → wait 12–48 hours → data available

- [S3 Glacier Deep Archive](https://docs.aws.amazon.com/AmazonS3/latest/userguide/glacier-storage-classes.html)

### ❌ Incorrect Options Explanation

**A. File Gateway → S3 Standard-IA after 5 days**

- **Why incorrect**: S3 Standard-Infrequent Access (Standard-IA) is designed for data accessed **infrequently but with rapid retrieval** (milliseconds). It is **not the most cost-effective** for archival data that the company is willing to wait days to retrieve. Standard-IA costs approximately **$0.0125/GB/month** — about **12x more expensive** than Glacier Deep Archive. The company explicitly says they can wait "a few days" for retrieval, so they should leverage the cheapest storage class available.

**B. Volume Gateway → S3 Glacier Deep Archive after 5 days**

- **Why incorrect**: **Volume Gateway** presents **iSCSI block storage** volumes (not NFS). While the backend storage (S3 + Glacier Deep Archive) is appropriate, the front-end protocol is wrong:
  - Volume Gateway appears as iSCSI targets to on-premises servers
  - On-premises NFS-based file storage cannot natively connect to iSCSI block volumes
  - Would require complete redesign of the on-premises solution — the company's existing solution "supports NFS" and they want the new solution to support NFS

**C. Tape Gateway → S3 Standard-IA after 5 days**

- **Why incorrect — two problems**:
  1. **Tape Gateway** presents a **Virtual Tape Library (VTL)** interface using iSCSI — it emulates physical tape drives and media changers. It does **not** support NFS. On-premises applications would need to speak VTL/iSCSI, which is incompatible with the existing NFS-based file storage.
  2. **S3 Standard-IA** is not cost-optimal given the "willing to wait a few days" retrieval requirement, as explained in option A.

**D. File Gateway (NFS frontend) but move files to Tape Gateway → S3 Standard-IA**

- **Why incorrect**: This option is internally inconsistent — it describes using File Gateway for NFS access but then references moving files to Tape Gateway. These are separate Storage Gateway types and cannot be chained in this manner. It's a mix-up of two different gateway types with conflicting architectures. Additionally, S3 Standard-IA is not the most cost-effective archival tier.

### 🏭 Enterprise Hands-On Knowledge

1. **Storage Gateway Family — Know Your Protocols**:

   | Gateway Type | Protocol | Interface | Primary Use Case |
   |---|---|---|---|
   | **File Gateway** | NFS v3/v4.1, SMB | File shares | On-premises file storage backed by S3; hybrid cloud file shares |
   | **Volume Gateway — Cached** | iSCSI | Block volumes | Low-latency access to frequently accessed data; full volume stored in S3 |
   | **Volume Gateway — Stored** | iSCSI | Block volumes | Full local copy with async backup to S3; low-latency + DR |
   | **Tape Gateway** | iSCSI VTL | Virtual tapes | Replacement for physical tape backup infrastructure; integrates with existing backup software (NetBackup, Commvault, etc.) |

   **SAP-C02 Exam Tip**: If the question mentions NFS → File Gateway. If iSCSI → Volume or Tape Gateway. If "existing backup software" or "tape" → Tape Gateway.

2. **S3 Storage Class Selection Decision Tree**:
   - Data accessed **frequently** → S3 Standard
   - Data accessed **infrequently, needs fast retrieval** → S3 Standard-IA or S3 Intelligent-Tiering
   - Data accessed **rarely (<1x/month), needs fast retrieval** → S3 One Zone-IA (non-critical) or S3 Glacier Instant Retrieval
   - Data accessed **rarely, can wait minutes-hours** → S3 Glacier Flexible Retrieval (formerly just "Glacier")
   - Data accessed **almost never, can wait 12-48 hours** → S3 Glacier Deep Archive ⬅️ *This scenario*

3. **File Gateway Caching Strategy**: File Gateway maintains a local cache of recently accessed files. For a backup/archive use case, configure the cache size appropriately — frequently accessed recent files (≤5 days old) remain hot in cache; archived files are retrieved from S3 Glacier Deep Archive on-demand.

4. **S3 Lifecycle Transition Gotcha**: Objects **must remain in S3 Standard for a minimum of 30 days** before transitioning to S3 Standard-IA or S3 One Zone-IA. However, you can transition **directly from S3 Standard to S3 Glacier Deep Archive** after 1 day (no minimum for Glacier Flexible Retrieval or Glacier Deep Archive). A lifecycle rule transitioning at day 5 is valid.

5. **Restore Costs & Planning**: S3 Glacier Deep Archive retrieval pricing includes:
   - **Standard retrieval**: Free (12–48 hours) for up to a certain amount per month
   - **Bulk retrieval**: Even cheaper but 48+ hours
   - For DR planning, document the restore process: Initiate `RestoreObject` API call → Wait for restore completion → Access objects via File Gateway → Copy to recovery environment

6. **Real-World DR Architecture**: A more complete enterprise solution might pair File Gateway with:
   - **AWS Backup** for automated, policy-driven backup scheduling
   - **S3 Cross-Region Replication (CRR)** to replicate to a DR region
   - **S3 Object Lock** for WORM (Write Once Read Many) compliance (important for medical/industrial regulatory requirements like FDA 21 CFR Part 11)
