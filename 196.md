A company is creating a REST API to share information with six of its partners based in the United States. The company has created an Amazon API Gateway
Regional endpoint. Each of the six partners will access the API once per day to post daily sales figures.
After initial deployment, the company observes 1,000 requests per second originating from 500 different IP addresses around the world. The company believes this traffic is originating from a botnet and wants to secure its API while minimizing cost.
Which approach should the company take to secure its API?

A. Create an Amazon CloudFront distribution with the API as the origin. Create an AWS WAF web ACL with a rule to block clients that submit more than five requests per day. Associate the web ACL with the CloudFront distribution. Configure CloudFront with an origin access identity (OAI) and associate it with the distribution. Configure API Gateway to ensure only the OAI can run the POST method.
B. Create an Amazon CloudFront distribution with the API as the origin. Create an AWS WAF web ACL with a rule to block clients that submit more than five requests per day. Associate the web ACL with the CloudFront distribution. Add a custom header to the CloudFront distribution populated with an API key. Configure the API to require an API key on the POST method.
C. Create an AWS WAF web ACL with a rule to allow access to the IP addresses used by the six partners. Associate the web ACL with the API. Create a resource policy with a request limit and associate it with the API. Configure the API to require an API key on the POST method.
D. Create an AWS WAF web ACL with a rule to allow access to the IP addresses used by the six partners. Associate the web ACL with the API. Create a usage plan with a request limit and associate it with the API. Create an API key and add it to the usage plan.


Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company exposed a REST API via Amazon API Gateway (Regional endpoint) for **six known US-based partners**, each posting daily sales figures (once per day). Unexpectedly, the API is receiving **1,000 requests/second from 500 different IPs worldwide** — a botnet attack. The solution must **secure the API** while **minimizing cost**. The critical insight: the six partners are **known** entities with **known IP addresses**, so an IP allowlist is the most targeted and cost-effective defense.

### Correct Option: D

**Create an AWS WAF web ACL with a rule to allow access to the IP addresses used by the six partners. Associate the web ACL with the API. Create a usage plan with a request limit and associate it with the API. Create an API key and add it to the usage plan.**

This solution implements **defense in depth** with three layers, each addressing a specific threat:

- **Layer 1 — IP Allowlist (AWS WAF)**: An IP set containing only the six partners' IP addresses is attached to a WAF web ACL rule with `Allow` action. All requests from non-whitelisted IPs are blocked with a `Block` default action. Crucially, WAF evaluates at the edge before the request reaches API Gateway's backend processing, so **blocked requests are not billed** against your API Gateway request quota. WAF charges $0.60/million requests evaluated — significantly cheaper than processing malicious requests through API Gateway ($3.50/million).

- **Layer 2 — Usage Plan (Rate Limiting)**: A usage plan defines throttling limits (rate and burst) appropriate for six partners posting once daily — e.g., 10 requests per second with a burst of 20. This protects against abuse even from whitelisted IPs (e.g., a compromised partner system) and prevents accidental or malicious overuse.

- **Layer 3 — API Key (Authentication)**: The API key, associated with the usage plan, is distributed to each partner. API Gateway validates the `x-api-key` header on each request. This ensures that even if an attacker spoofs a partner's IP, they still need the API key to invoke the POST method.

### Incorrect Options

**A: CloudFront + WAF rate-based rule (5/day) + OAI** — Multiple problems: (1) **Origin Access Identity (OAI)** is a CloudFront feature for **S3 origins only** — it cannot be applied to API Gateway origins. API Gateway uses **resource policies** for origin-level access control, not OAI. (2) AWS WAF **rate-based rules** operate on a 5-minute evaluation window with a minimum threshold of **100 requests per 5 minutes**. A "5 requests per day" rule is technically impossible — WAF rate-based rules cannot span a 24-hour window. (3) Adding CloudFront introduces additional latency and cost (CloudFront data transfer + request charges) without adding proportionate security value when the partners are known and few.

**B: CloudFront + WAF rate-based rule (5/day) + custom header API key** — Same WAF rate-based rule limitation as Option A (5 requests/day is not achievable). While adding a custom header with an API key at the CloudFront level and validating it at API Gateway is technically possible, it introduces unnecessary architectural complexity (CloudFront Functions or Lambda@Edge to inject the header, API Gateway resource policy or Lambda authorizer to validate it). The CloudFront layer adds cost without commensurate benefit given the simple use case.

**C: WAF IP allowlist + resource policy with request limit + API key** — API Gateway **resource policies** are IAM-based JSON documents that control who can invoke an API (by AWS account, source VPC, IP range). They do **not** provide request rate limiting or throttling. A "request limit" in a resource policy is not a feature — resource policies grant or deny access, they don't meter usage. Rate limiting is the domain of **usage plans**, not resource policies.

### Hands-On Knowledge

1. **WAF IP Set Setup**:
   ```bash
   aws wafv2 create-ip-set \
     --name "PartnerIPs" --scope REGIONAL \
     --addresses "203.0.113.0/24" "198.51.100.1/32" \
     --ip-address-version IPV4
   ```
   Then create a web ACL rule with a `Allow` action matching the IP set, and set the default web ACL action to `Block`.

2. **API Gateway Usage Plan Configuration**:
   - Create a usage plan: throttle rate = 10 req/sec, burst = 20 req/sec, quota = 50 req/day (covers six partners with buffer).
   - Create an API key for each partner (or one shared key) and associate it with the usage plan.
   - Associate the usage plan with the API stage (e.g., `prod`).
   - Enable API key requirement on the POST method: set `API Key Required = true` in the method request settings.

3. **Cost Optimization Analysis**:
   - Without WAF: 1,000 req/s × 86,400 sec/day × 30 days = ~2.6 billion requests/month × $3.50/million = **~$9,100/month** in API Gateway charges alone.
   - With WAF IP allowlist: Blocked requests are evaluated by WAF ($0.60/million) but never reach API Gateway. ~2.6 billion × $0.60/million = **~$1,560/month** for WAF + minimal API Gateway charges for legitimate traffic ($0.35/month for 180 requests). **Savings: ~83%.**

4. **WAF Association with API Gateway Regional**: WAF can be associated directly with a Regional API Gateway endpoint. In the WAF console, choose "Regional resources" → select the API Gateway REST API and stage. Note: WAF cannot be associated with Edge-optimized API Gateway endpoints without CloudFront.

5. **Geo-Restriction Alternative**: If the partners' IPs are not static, consider CloudFront geo-restriction (whitelist US only) as a pre-filter before WAF processing. This blocks non-US traffic at the CloudFront edge (no charge for blocked requests) and reduces WAF evaluation volume.
