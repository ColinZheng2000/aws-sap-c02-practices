A data analytics company has an Amazon Redshift cluster that consists of several reserved nodes. The cluster is experiencing unexpected bursts of usage because a team of employees is compiling a deep audit analysis report. The queries to generate the report are complex read queries and are CPU intensive.

Business requirements dictate that the cluster must be able to service read and write queries at all times. A solutions architect must devise a solution that accommodates the bursts of usage.

Which solution meets these requirements MOST cost-effectively?

A. Provision an Amazon EMR cluster Offload the complex data processing tasks.
B. Deploy an AWS Lambda function to add capacity to the Amazon Redshift cluster by using a classic resize operation when the cluster’s CPU metrics in Amazon CloudWatch reach 80%.
C. Deploy an AWS Lambda function to add capacity to the Amazon Redshift cluster by using an elastic resize operation when the cluster’s CPU metrics in Amazon CloudWatch reach 80%.
D. Turn on the Concurrency Scaling feature for the Amazon Redshift cluster.



Suggest Answer: D

---

## 📊 Analysis

### 🔍 Understanding the Problem

A company operates an **Amazon Redshift cluster** provisioned with **reserved nodes** (indicating a long-term, steady-state workload commitment). A team is running complex, **CPU-intensive read queries** for a deep audit analysis, causing **unexpected usage bursts**. The critical business requirement is that the cluster **must continue to service both read and write queries at all times** — downtime is unacceptable. The solution must be **cost-effective** while accommodating these intermittent query spikes.

Key constraints:
- Reserved nodes are already purchased (long-term commitment)
- Bursts are temporary and driven by a specific audit project
- Read and write availability must be preserved
- Cost-effectiveness is a priority

### ✅ Correct Option: D — Turn on the Concurrency Scaling feature for the Amazon Redshift cluster

**Why it's correct:**

**Amazon Redshift Concurrency Scaling** is a feature purpose-built for exactly this scenario. When enabled, Redshift automatically adds **transient, additional cluster capacity** to handle spikes in concurrent read queries (and certain write operations). Key benefits:

1. **Automatic & Transparent**: Concurrency Scaling kicks in automatically when the main cluster's query queue depth exceeds a threshold. No manual intervention, no Lambda functions, no CloudWatch alarms needed.

2. **No Downtime**: The main cluster continues to serve all workloads uninterrupted. Concurrency Scaling adds separate, transient clusters that process queued queries — the primary cluster never pauses or becomes unavailable.

3. **Cost-Effective for Bursts**: AWS provides **1 hour of Concurrency Scaling credits per day** (accumulated) at no additional charge for each 24 hours of reserved node usage. Beyond that, you pay per second for transient cluster usage — aligning costs directly with burst needs rather than provisioning permanent capacity.

4. **Read-Query Optimized**: Concurrency Scaling is specifically designed for read-heavy, CPU-intensive workloads like the audit queries described. Writes continue to be served by the primary cluster seamlessly.

**Official AWS Documentation Reference**: [Amazon Redshift Concurrency Scaling](https://docs.aws.amazon.com/redshift/latest/dg/concurrency-scaling.html) — "Concurrency Scaling provides consistently fast query performance even as concurrent queries increase, at a predictable cost."

### ❌ Incorrect Options Explanation

**A. Provision an Amazon EMR cluster. Offload the complex data processing tasks.**

- **Why incorrect**: Amazon EMR is a **big data processing platform** (Hadoop/Spark/Hive) designed for distributed data processing and analytics on massive datasets. It is **not a substitute** for Redshift query processing. Offloading Redshift queries to EMR would require:
  - Exporting data from Redshift to S3 (data movement costs and latency)
  - Rewriting all audit queries in Spark/Hive SQL (compatibility issues)
  - Building custom orchestration between two very different services
  - Fundamentally, EMR does not serve as a "query offload" for Redshift — they are complementary tools for different purposes.

**B. Deploy an AWS Lambda function to add capacity via classic resize when CPU reaches 80%**

- **Why incorrect**: Classic resize involves **redistributing data across new nodes**, which causes the cluster to become **read-only for hours** during the operation. This directly violates the requirement that the cluster must service both read **and write** queries at all times. Additionally:
  - Reactive approach (CloudWatch alarm → Lambda → resize) introduces latency — bursts are already hurting performance before scaling occurs
  - Classic resize is slow and disruptive; it's meant for permanent capacity changes, not temporary bursts
  - Once resized, you're paying for the larger cluster permanently until you resize back down

**C. Deploy an AWS Lambda function to add capacity via elastic resize when CPU reaches 80%**

- **Why incorrect**: While elastic resize is faster than classic resize (typically 10–15 minutes vs. hours), it still:
  - May cause the cluster to be **unavailable for a short period** during the resize operation
  - Is a **reactive** approach — the burst is already degrading performance before the Lambda triggers
  - Does not scale back down automatically; requires another resize operation to return to original size
  - Incurs unnecessary costs for full-time node capacity when only temporary burst handling is needed
  - Elastic resize is designed for permanent cluster resizing, not for dynamic, unpredictable workload spikes

### 🏭 Enterprise Hands-On Knowledge

1. **Concurrency Scaling vs. Resize — Know When to Use Each**:
   - **Concurrency Scaling**: Transient, temporary query spikes (minutes to hours). Ideal for reporting bursts, month-end processing, ad-hoc analytics.
   - **Elastic Resize**: Planned, permanent capacity changes (e.g., company growth, new workloads). Faster than classic, minimal downtime.
   - **Classic Resize**: Major node type or version changes, or when elastic resize isn't supported for the target configuration.
   - **Snapshot & Restore**: Used when you need to change cluster topology drastically (e.g., DS2 → RA3 instances).

2. **Concurrency Scaling Credits Calculation**: For every 24 hours of reserved node usage, you accumulate 1 hour of Concurrency Scaling credit. Credits expire after 24 hours. If your burst patterns are predictable (e.g., daily report runs), you may never pay extra.

3. **RA3 Nodes and AQUA**: Modern Redshift RA3 instances with managed storage and AQUA (Advanced Query Accelerator) provide additional query performance for CPU-intensive workloads. If the underlying issue is chronic, upgrading node types should be evaluated alongside Concurrency Scaling.

4. **Workload Management (WLM)**: For enterprise deployments, configure WLM queues to route different query types appropriately. Critical write queries can be assigned to a high-priority queue, while the audit read queries can leverage Concurrency Scaling through a separate queue with appropriate concurrency limits.

5. **CloudWatch Metrics to Monitor**: `QueryQueueDuration` and `DatabaseConnections` are better indicators of concurrency pressure than CPU alone. Set up dashboards tracking these alongside `CPUUtilization`.

6. **Cost Optimization Pro Tip**: If bursts are truly temporary (weeks-long audit project), Concurrency Scaling is ideal. If the increased load becomes permanent, re-evaluate and consider elastic resize to add permanent nodes, using reserved instances for the new baseline.
