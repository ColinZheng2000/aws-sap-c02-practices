A company wants to manage the costs associated with a group of 20 applications that are infrequently used, but are still business-critical, by migrating to AWS.
The applications are a mix of Java and Node.js spread across different instance clusters. The company wants to minimize costs while standardizing by using a single deployment methodology. Most of the applications are part of month-end processing routines with a small number of concurrent users, but they are occasionally run at other times. Average application memory consumption is less than 1 GB, though some applications use as much as 2.5 GB of memory during peak processing. The most important application in the group is a billing report written in Java that accesses multiple data sources and often for several hours.
Which is the MOST cost-effective solution?

A. Deploy a separate AWS Lambda function for each application. Use AWS CloudTrail logs and Amazon CloudWatch alarms to verify completion of critical jobs.
B. Deploy Amazon ECS containers on Amazon EC2 with Auto Scaling configured for memory utilization of 75%. Deploy an ECS task for each application being migrated with ECS task scaling. Monitor services and hosts by using Amazon CloudWatch.
C. Deploy AWS Elastic Beanstalk for each application with Auto Scaling to ensure that all requests have sufficient resources. Monitor each AWS Elastic Beanstalk deployment by using CloudWatch alarms.
D. Deploy a new Amazon EC2 instance cluster that co-hosts all applications by using EC2 Auto Scaling and Application Load Balancers. Scale cluster size based on a custom metric set on instance memory utilization. Purchase 3-year Reserved Instance reservations equal to the GroupMaxSize parameter of the Auto Scaling group.


Suggest Answer: B

---

## Analysis

### Understanding the Problem

A company has approximately 20 business-critical but infrequently used applications (Java + Node.js) spread across different instance clusters on-premises. The core requirements are: (1) **minimize costs**, (2) **standardize on a single deployment methodology** across all applications, (3) handle variable memory footprints (most <1 GB, peaks up to 2.5 GB), and (4) support a long-running Java billing report that runs for **several hours**. The "several hours" runtime is a critical constraint — it immediately rules out AWS Lambda (15-minute maximum execution timeout). The apps are used for month-end processing with sporadic usage at other times, meaning a solution that scales down (or to zero) during idle periods is ideal.

### Correct Option: B — Amazon ECS on EC2 + Auto Scaling + ECS Task Scaling

**Why B is correct:**

- **Amazon ECS on EC2** provides container orchestration with full control over the underlying EC2 instances. Containers standardize the deployment methodology across both Java and Node.js applications — a single deployment paradigm (Docker containers on ECS).
- **No timeout limitation**: Unlike AWS Lambda (15-minute hard limit), ECS tasks can run indefinitely, supporting the Java billing report that runs "for several hours" and accesses "multiple data sources."
- **Memory fit**: EC2 instances in the ECS cluster can be right-sized to accommodate the 2.5 GB peak memory requirement. ECS task-level memory reservations ensure each container gets its required memory allocation.
- **Cost optimization**: Auto Scaling based on memory utilization (75%) ensures the cluster only expands when needed. Since applications are used infrequently, the cluster can scale down during long idle periods. ECS task scaling allows individual applications to scale independently.
- **Amazon CloudWatch** provides unified monitoring for both the ECS service metrics and the host-level EC2 metrics.
- **Single methodology**: All 20 applications run as ECS tasks, regardless of language (Java or Node.js), meeting the standardization requirement.

**Official reference**: [Amazon ECS Best Practices Guide](https://docs.aws.amazon.com/AmazonECS/latest/bestpracticesguide/intro.html) — ECS supports both long-running services and batch workloads with customizable task definitions, memory/CPU reservations, and Auto Scaling integration.

### Incorrect Options

**A — AWS Lambda for each application**

- **Fatal flaw: Lambda's 15-minute maximum execution timeout**. The Java billing report "often runs for several hours" — Lambda cannot support this. This alone disqualifies the option.
- Lambda's maximum memory is 10,240 MB (10 GB), so the 2.5 GB peak memory is not a problem technically, but the timeout is a hard blocker.
- Even for shorter-running applications, Lambda's ephemeral nature (stateless, cold starts) may not be ideal for month-end batch processing jobs that require persistent connections to multiple data sources.
- Using CloudTrail logs (audit logs) to "verify completion of critical jobs" is an inappropriate use — CloudTrail records API calls, not application-level job completion status. CloudWatch Logs or custom metrics would be the correct choice.

**C — AWS Elastic Beanstalk for each application**

- Deploying **20 separate Elastic Beanstalk environments** (one per application) creates significant operational sprawl. Each EB environment includes its own load balancer, Auto Scaling group, and EC2 instances. This is far from cost-effective.
- Elastic Beanstalk is designed for web applications with request-based Auto Scaling. The apps described are batch processing workloads, not request-driven web services. Scaling based on request counts (the typical EB auto-scaling metric) doesn't align with CPU/memory-intensive batch jobs.
- CloudWatch alarms alone don't address the core need for a cost-effective, standardized deployment approach.

**D — Single EC2 cluster co-hosting all 20 apps + 3-year Reserved Instances**

- Co-hosting 20 heterogeneous applications (Java + Node.js, different memory profiles) on a single EC2 cluster is an **anti-pattern**. Resource contention, dependency conflicts (different Node.js versions, different JVM versions), and lack of workload isolation create operational risk.
- "Scale cluster size based on a custom metric set on instance memory utilization" — custom metrics add operational complexity. More importantly, co-hosted apps can't scale independently; one memory-hungry app could trigger a scale-out that unnecessarily replicates all 20 apps.
- **3-year Reserved Instance commitment** contradicts the "infrequently used" nature of the applications. RI purchases are cost-effective for steady-state, predictable workloads (e.g., 24/7 baseline). For sporadic month-end processing, you'd be paying for resources you rarely use. A savings plan or Spot/On-Demand mix would be more appropriate.
- Setting RI equal to `GroupMaxSize` is financially wasteful — you'd pay for maximum capacity 24/7/365 even though peak usage occurs only a few days per month.

### Hands-On Enterprise Knowledge

1. **Lambda timeout is non-negotiable**: In the SAP-C02 exam, whenever a question mentions workloads running for "hours," Lambda is automatically ruled out. The 15-minute execution timeout (configurable up to 15 minutes, default 3 seconds) is a hard limit that cannot be extended. For long-running batch jobs, ECS, AWS Batch, or EC2 are the appropriate compute choices.

2. **ECS task vs. service**: For the month-end processing scenario, consider using **ECS RunTask** API calls (triggered by EventBridge scheduled rules) rather than long-running ECS services. This way, tasks only run when needed. For the sporadic ad-hoc usage, on-demand RunTask invocations keep costs near zero during idle periods.

3. **ECS on EC2 vs. Fargate**: While AWS Fargate (serverless ECS) would also work and further reduce operational overhead, it is generally more expensive per vCPU-hour for long-running workloads. The SAP-C02 exam expects you to weigh cost vs. operational overhead. For tasks running several hours daily (billing report), EC2 launch type with Savings Plans or Spot Instances is more cost-effective than Fargate.

4. **Container standardization in enterprises**: A common enterprise pattern is using ECS/EKS as the "golden deployment platform" — every application team containerizes their app (regardless of language) and deploys via the same CI/CD pipeline. This aligns perfectly with the "standardize by using a single deployment methodology" requirement.

5. **Memory-based Auto Scaling**: ECS cluster Auto Scaling can use the `MemoryReservation` CloudWatch metric. Setting a target of 75% memory reservation means the cluster will scale out before tasks experience memory pressure. This is more precise than CPU-based scaling for memory-bound Java applications.
