A finance company hosts a data lake in Amazon S3. The company receives financial data records over SFTP each night from several third parties. The company runs its own SFTP server on an Amazon EC2 instance in a public subnet of a VPC. After the files are uploaded, they are moved to the data lake by a cron job that runs on the same instance. The SFTP server is reachable on DNS sftp.example.com through the use of Amazon Route 53.  
What should a solutions architect do to improve the reliability and scalability of the SFTP solution?  

- A. Move the EC2 instance into an Auto Scaling group. Place the EC2 instance behind an Application Load Balancer (ALB). Update the DNS record sftp.example.com in Route 53 to point to the ALB.
- B. Migrate the SFTP server to AWS Transfer for SFTP. Update the DNS record sftp.example.com in Route 53 to point to the server endpoint hostname.
- C. Migrate the SFTP server to a file gateway in AWS Storage Gateway. Update the DNS record sftp.example.com in Route 53 to point to the file gateway endpoint.
- D. Place the EC2 instance behind a Network Load Balancer (NLB). Update the DNS record sftp.example.com in Route 53 to point to the NLB.

## Understanding the Problem

The finance company currently runs a self-managed SFTP server on a single **Amazon EC2** instance to receive nightly financial data from third parties and then moves the files to an **Amazon S3** data lake using a cron job. The SFTP server is exposed via **Amazon Route 53** DNS: `sftp.example.com`.

The current setup's major drawbacks are:

1. **Single point of failure:** The SFTP server is on a single EC2 instance, which lacks **reliability** and high **availability**.
    
2. **Scalability issues:** Scaling the self-managed SFTP solution (especially for the session state and storage access) is complex and requires significant operational overhead.
    
3. **Operational overhead:** Managing, patching, and maintaining the operating system, SFTP server software, and the cron job requires effort.
    

The goal is to improve the **reliability** and **scalability** of the SFTP solution with minimal operational overhead and disruption, leveraging AWS managed services where possible.

## Services Explanation

### AWS Transfer Family (Specifically for SFTP)

**AWS Transfer Family** is a fully managed service that enables the transfer of files into and out of **Amazon S3** or **Amazon EFS** using the **SFTP**, **FTPS**, and **FTP** protocols.

- **Reliability & Scalability:** It's a highly available, multi-AZ service by design, eliminating single points of failure. It automatically scales to handle concurrent transfers.
    
- **Operational Overhead:** It removes the need to manage the underlying infrastructure (EC2 instances, operating systems, SFTP software).
    
- **DNS Integration:** Provides a service endpoint hostname that can be pointed to by **Amazon Route 53**.
    
- **Integration with S3:** Directly and securely stores uploaded files into an S3 bucket, simplifying the process (no more cron job needed for file movement).
    

### Amazon EC2 & Auto Scaling Group (ASG)

**Amazon EC2** provides resizable compute capacity in the cloud. **Auto Scaling Groups** help ensure you have the correct number of EC2 instances available to handle the load.

- **Reliability:** ASG can replace unhealthy instances, improving availability.
    
- **Scalability:** ASG can launch more instances to handle increased load.
    
- **Limitation for SFTP:** Maintaining session state and shared storage access across multiple SFTP instances in an ASG is complex and adds significant architectural overhead (e.g., requires using shared file systems like EFS or complex synchronization, which is not as simple as using a managed service).
    

### Elastic Load Balancing (ELB) - Application Load Balancer (ALB) and Network Load Balancer (NLB)

- **Application Load Balancer (ALB):** Operates at Layer 7 (HTTP/HTTPS). It **cannot** be used to load balance the **SFTP** protocol, which typically runs over SSH on port 22.
    
- **Network Load Balancer (NLB):** Operates at Layer 4 (TCP/UDP). While an **NLB can load balance TCP on port 22** to multiple EC2 instances, it still leaves the complexity of managing the SFTP server software, maintaining session state, and ensuring synchronized access to the S3 data lake across multiple instances.
    

### AWS Storage Gateway - File Gateway

**AWS Storage Gateway** connects an on-premises software appliance with cloud storage to provide seamless and secure integration. A **File Gateway** presents an **NFS** or **SMB** file share interface to on-premises applications, storing the data as objects in S3.

- **SFTP:** **File Gateway does not natively provide an SFTP endpoint**. It provides NFS and SMB file shares. Therefore, it cannot be used to directly replace the existing SFTP server for third-party uploads over SFTP.

## Correct Option(s) Explanation

### B. Migrate the SFTP server to AWS Transfer for SFTP. Update the DNS record sftp.example.com in Route 53 to point to the server endpoint hostname.

This is the **correct** and most efficient solution.

- **Reliability & Scalability:** **AWS Transfer Family** is a fully managed, highly available, and auto-scaling service (as per the **AWS Transfer Family documentation**). This immediately resolves the single point of failure and scalability concerns of the single EC2 instance.
    
- **Operational Overhead:** It eliminates the need to manage, patch, and maintain the EC2 instance and SFTP software.
    
- **S3 Integration:** The uploaded files are directly and reliably stored in **Amazon S3** (the data lake), eliminating the need for the unreliable, self-managed cron job.
    
- **Minimal Disruption:** The **Route 53** update ensures third-party clients can continue to connect to `sftp.example.com` with minimal change to their configuration.

## Incorrect Options Explanation

### A. Move the EC2 instance into an Auto Scaling group. Place the EC2 instance behind an Application Load Balancer (ALB). Update the DNS record sftp.example.com in Route 53 to point to the ALB.

- **ALB cannot be used for SFTP:** **Application Load Balancer** operates at Layer 7 (HTTP/HTTPS) and **does not support the SFTP protocol** (TCP on port 22). This option is technically invalid.
    
- **Complexity:** Even if a different load balancer were used, this self-managed setup would still require significant effort to manage high-availability stateful SFTP servers in an ASG with shared storage access (likely EFS or a more complex S3 access mechanism).
    

### C. Migrate the SFTP server to a file gateway in AWS Storage Gateway. Update the DNS record sftp.example.com in Route 53 to point to the file gateway endpoint.

- **Storage Gateway does not support SFTP:** **AWS Storage Gateway File Gateway** primarily provides access using **NFS** and **SMB** protocols, not **SFTP**. Therefore, it cannot be used to replace an existing SFTP server that third parties connect to via SFTP.
    

### D. Place the EC2 instance behind a Network Load Balancer (NLB). Update the DNS record sftp.example.com in Route 53 to point to the NLB.

- **Incomplete Solution:** While an **NLB can load balance TCP traffic (like SFTP on port 22)**, this option only addresses load distribution, not the underlying SFTP server's high availability or shared state. You would still need to:
    
    1. Move the EC2 instance into an Auto Scaling Group (not mentioned).
        
    2. Architect a complex solution for shared, consistent file access to S3 across multiple SFTP instances (e.g., using EFS with the servers).
        
- **Operational Overhead:** The company is still responsible for managing the EC2 instances, the operating system, and the SFTP server software. This solution does not leverage a fully managed service to improve reliability and scalability as effectively as Option B and increases operational complexity. Option B is a far superior, fully managed approach.
