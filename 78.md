# AWS SAP-C02 Practice Question: Client-Side Encryption and IAM

---

## Analysis

### 1. Understanding the Problem
The core of this scenario is **Client-Side Encryption (CSE)** using an AWS KMS Customer Master Key (CMK)—now referred to as an AWS KMS key. 

In a Client-Side Encryption workflow using the AWS SDK:
1.  **Upload:** The client (the application/script running the SDK) is responsible for encrypting the data *locally* before it is sent to S3. To do this, the client requests a new Data Key from KMS.
2.  **Download:** The client downloads the encrypted object and the encrypted data key (stored as metadata), then asks KMS to decrypt the data key so the client can decrypt the object locally.

The provided IAM policy grants `kms:Encrypt` and `kms:Decrypt`. 
* **The Symptom:** Downloads work (because `kms:Decrypt` allows the client to decrypt the data key). Uploads fail (because the client cannot acquire the keys needed to encrypt the data).
* **The Missing Link:** The AWS SDK implementation of Client-Side Encryption utilizes **Envelope Encryption**. The client does not send the file to KMS to be encrypted (which would use `kms:Encrypt`). Instead, the client asks KMS to generate a unique Data Key.

### 2. Services Explanation
* **AWS KMS (Key Management Service):** A managed service that makes it easy to create and control the cryptographic keys used to protect your data. It uses hardware security modules (HSMs) to protect the security of your keys.
* **Amazon S3 Client-Side Encryption:** The act of encrypting data inside your environment (the client) before sending it to Amazon S3. The AWS SDK handles this process automatically if configured with a KMS Key ID.
* **Envelope Encryption:** The practice of encrypting plaintext data with a data key, and then encrypting the data key with a root key (KMS Key).

### 3. Correct Option Explanation
**A. kms:GenerateDataKey**
When using the AWS SDK for Client-Side encryption with a KMS key, the encryption flow for an upload (`PutObject`) is as follows:
1.  The SDK sends a `GenerateDataKey` request to AWS KMS specifying the CMK ID.
2.  KMS returns two versions of a specific data key:
    * **Plaintext Data Key:** Used by the client immediately to encrypt the object data in memory.
    * **Encrypted Data Key:** Uploaded to S3 alongside the encrypted object (stored as object metadata).
3.  The SDK wipes the plaintext key from memory.

Without the `kms:GenerateDataKey` permission, the SDK cannot request the keys required to perform the local encryption. Therefore, the upload fails before the data even reaches S3.

### 4. Incorrect Options Explanation
**B. kms:GetKeyPolicy**
This permission allows a principal to view the Key Policy document attached to a KMS key. It is a management/administrative permission, not a cryptographic data-plane permission required for encryption operations.

**C. kms:GetPublicKey**
This permission is used primarily for **Asymmetric Keys** (RSA or ECC). It allows a user to download the public portion of an asymmetric key pair to perform encryption locally (outside of AWS) or verify signatures. S3 Client-Side encryption typically uses Symmetric keys (AES), and even if asymmetric keys were used, the SDK abstraction primarily relies on the `GenerateDataKey` workflow for object encryption.

**D. kms:Sign**
This permission is used for creating a digital signature using an Asymmetric KMS key. It is used to ensure data integrity and authenticity, not for the confidentiality (encryption) of the object data.
