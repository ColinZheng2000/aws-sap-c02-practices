A company needs to migrate an on-premises SFTP site to AWS. The SFTP site currently runs on a Linux VM. Uploaded files are made available to downstream applications through an NFS share.

As part of the migration to AWS, a solutions architect must implement high availability. The solution must provide external vendors with a set of static public IP addresses that the vendors can allow. The company has set up an AWS Direct Connect connection between its on-premises data center and its VPC.

Which solution will meet these requirements with the LEAST operational overhead?

A. Create an AWS Transfer Family server. Configure an internet-facing VPC endpoint for the Transfer Family server. Specify an Elastic IP address for each subnet. Configure the Transfer Family server to place files into an Amazon Elastic File System (Amazon EFS) file system that is deployed across multiple Availability Zones. Modify the configuration on the downstream applications that access the existing NFS share to mount the EFS endpoint instead.
B. Create an AWS Transfer Family server. Configure a publicly accessible endpoint for the Transfer Family server. Configure the Transfer Family server to place files into an Amazon Elastic File System (Amazon EFS) file system that is deployed across multiple Availability Zones. Modify the configuration on the downstream applications that access the existing NFS share to mount the EFS endpoint instead.
C. Use AWS Application Migration Service to migrate the existing Linux VM to an Amazon EC2 instance. Assign an Elastic IP address to the EC2 instance. Mount an Amazon Elastic File System (Amazon EFS) file system to the EC2 instance. Configure the SFTP server to place files in the EFS file system. Modify the configuration on the downstream applications that access the existing NFS share to mount the EFS endpoint instead.
D. Use AWS Application Migration Service to migrate the existing Linux VM to an AWS Transfer Family server. Configure a publicly accessible endpoint for the Transfer Family server. Configure the Transfer Family server to place files into an Amazon FSx for Lustre file system that is deployed across multiple Availability Zones. Modify the configuration on the downstream applications that access the existing NFS share to mount the FSx for Lustre endpoint instead.


Suggest Answer: A

## 📊 Analysis

### 🔍 Understanding the Problem

A company needs to **migrate an on-premises SFTP site** running on a Linux VM to AWS. The current architecture:

- **SFTP server** on a Linux VM → receives file uploads from external vendors
- **NFS share** → makes uploaded files available to downstream applications

The migration requirements are:
1. **High availability** — multi-AZ deployment
2. **Static public IP addresses** — external vendors need fixed IPs to add to their firewall allow-lists
3. **NFS-compatible storage** — downstream applications currently mount an NFS share
4. **Least operational overhead** — prefer managed services
5. **Existing Direct Connect** — on-premises data center is already connected to the VPC

The core challenge: how do you get a managed SFTP service with static public IPs (for vendor allow-listing) that integrates with NFS for downstream consumption?

### ✅ Correct Option: A — AWS Transfer Family with Internet-Facing VPC Endpoint + Elastic IPs + EFS

**AWS Transfer Family** is a fully managed service that supports SFTP, FTPS, and FTP protocols. This option correctly implements the architecture:

**1. AWS Transfer Family Server**
- Fully managed SFTP service — no servers to patch, maintain, or monitor
- Native integration with S3 and EFS as storage backends
- Handles user authentication, key management, and protocol compliance

**2. Internet-Facing VPC Endpoint**
- Transfer Family offers three endpoint types:
  - **Public endpoint**: Accessible from the Internet but uses **dynamic** AWS-managed IP addresses (cannot be allow-listed)
  - **VPC endpoint (internal)**: Only accessible from within the VPC
  - **VPC endpoint (internet-facing)**: Accessible from the Internet with **static Elastic IP addresses** assigned to each subnet — THIS is the critical feature
- By specifying an Elastic IP for each subnet (multi-AZ), external vendors get a set of **fixed, unchanging** public IP addresses they can add to their firewall rules

**3. Amazon EFS (Elastic File System)**
- Fully managed NFS file system that can be mounted across **multiple Availability Zones**
- Provides the **same NFS protocol** that downstream applications already use — minimal migration changes
- Highly available by design — EFS stores data redundantly across AZs
- Downstream applications (connected via Direct Connect or within VPC) simply remount from the old NFS export to the EFS endpoint

**4. High Availability Architecture**:
```
External Vendors (Internet)
        │
        ▼
┌───────────────────────────────────┐
│  AWS Transfer Family (SFTP)       │
│  Internet-facing VPC Endpoint     │
│  ┌──────────┐  ┌──────────┐      │
│  │  AZ-1    │  │  AZ-2    │      │
│  │  EIP-1   │  │  EIP-2   │      │
│  └────┬─────┘  └────┬─────┘      │
└───────┼─────────────┼─────────────┘
        │             │
        ▼             ▼
┌───────────────────────────────────┐
│  Amazon EFS (Multi-AZ)            │
│  mount target AZ-1 │ AZ-2        │
└───────────────────┬───────────────┘
                    │
                    ▼
┌───────────────────────────────────┐
│  Downstream Applications          │
│  (via VPC or Direct Connect)      │
│  Mount EFS via NFS protocol       │
└───────────────────────────────────┘
```

### ❌ Incorrect Options

**B. AWS Transfer Family with publicly accessible endpoint + EFS**
- A **publicly accessible endpoint** for Transfer Family uses **dynamic IP addresses** managed by AWS
- These IP addresses **change over time** — AWS can rotate them for maintenance, scaling, or DDoS mitigation
- External vendors **cannot reliably add dynamic IPs** to their firewall allow-lists — the IP could change, breaking connectivity
- This fails the "static public IP addresses" requirement
- The distinction between "public endpoint" (dynamic IPs) and "internet-facing VPC endpoint" (static Elastic IPs) is critical and frequently tested

**C. AWS Application Migration Service to migrate the Linux VM to EC2 + Elastic IP + EFS**
- Uses **AWS Application Migration Service (MGN)** for lift-and-shift rehosting — this works but introduces significant operational overhead:
  - You now manage the EC2 instance (OS patching, security updates, monitoring)
  - The SFTP software on the VM must be maintained and configured
  - Auto Scaling and multi-AZ high availability require additional configuration (launch templates, Auto Scaling groups)
  - If the EC2 instance fails, you need to manually recover or build complex automation
- This is **NOT the "least operational overhead"** option — it's a lift-and-shift approach that preserves the management burden of the on-premises solution
- The question explicitly asks for LEAST operational overhead, which favors managed services

**D. AWS Application Migration Service to migrate Linux VM to AWS Transfer Family server + FSx for Lustre**
- **AWS Application Migration Service cannot migrate a VM to AWS Transfer Family** — MGN replicates servers to EC2 instances, not to managed services. Transfer Family is a managed service with no concept of "migrating a VM into it"
- **Amazon FSx for Lustre** is a high-performance parallel file system designed for **HPC and machine learning workloads**, not general-purpose NFS sharing
- FSx for Lustre is not natively compatible with the NFS protocol that downstream applications use — they would need to be reconfigured to use Lustre clients
- This option conflates unrelated services and shows fundamental misunderstanding of both the migration tool and the file system choice

### 🏭 Enterprise Hands-On Knowledge

1. **AWS Transfer Family Endpoint Types Decision Matrix**:
   | Endpoint Type | Static IPs | Internet Access | VPC Access | Use Case |
   |--------------|------------|-----------------|------------|----------|
   | Public | ❌ Dynamic | ✅ Yes | ❌ No | General SFTP, no IP allow-listing needed |
   | VPC (Internal) | ✅ Private IPs | ❌ No | ✅ Yes | Internal-only file transfers |
   | VPC (Internet-facing) | ✅ Elastic IPs | ✅ Yes | ✅ Yes | External vendors needing IP allow-listing |

2. **Transfer Family Authentication Options**:
   - **Service-managed users**: Username/SSH key pairs stored in Transfer Family
   - **AWS Directory Service (Microsoft AD)**: Integrate with existing corporate directory
   - **Custom identity provider**: Lambda-based authentication with API Gateway, Cognito, etc.
   - **AWS Secrets Manager**: Store SSH keys in Secrets Manager for automated rotation

3. **EFS Performance Modes**:
   - **General Purpose**: Default, low latency, suitable for most workloads
   - **Max I/O**: Higher throughput and IOPS, slightly higher latency
   - **Throughput modes**: Bursting (default, throughput scales with storage) or Provisioned (fixed throughput independent of storage size)
   - For SFTP workloads with many small file uploads, General Purpose + Bursting is usually sufficient

4. **Security Best Practices for External SFTP**:
   - Deploy Transfer Family in a **dedicated "DMZ" subnet** with restrictive NACLs and Security Groups
   - Use **S3 bucket policies or EFS IAM policies** to restrict which vendors can access which directories
   - Enable **CloudTrail logging** for all SFTP operations (file uploads, downloads, deletes)
   - Configure **Amazon EventBridge** rules to trigger downstream processing when files are uploaded
   - Implement **file retention policies** using EFS lifecycle management or S3 lifecycle rules

5. **EFS vs. FSx Decision Guide**:
   | Feature | Amazon EFS | Amazon FSx for Lustre | Amazon FSx for Windows | Amazon FSx for NetApp ONTAP |
   |---------|------------|----------------------|------------------------|------------------------------|
   | Protocol | NFSv4 | Lustre | SMB | NFS, SMB, iSCSI |
   | Use Case | General Linux workloads | HPC, ML, big data | Windows file shares | Enterprise NAS migration |
   | Multi-AZ | ✅ Yes | ❌ Single-AZ (scratch), Multi-AZ (persistent) | ✅ Yes | ✅ Yes |
   | NFS compatible | ✅ Native | ❌ Requires Lustre client | ❌ SMB only | ✅ Native NFS support |

6. **Migration Cutover Strategy**:
   - Phase 1: Set up Transfer Family + EFS in parallel with existing on-prem SFTP
   - Phase 2: Provide vendors with the new static IP addresses and SFTP endpoint
   - Phase 3: Vendors update their firewall rules and cut over to the new endpoint
   - Phase 4: Downstream applications remount from old NFS to EFS endpoint (via Direct Connect for hybrid)
   - Phase 5: Decommission the on-premises Linux VM after verifying all traffic has migrated

7. **Direct Connect Integration**: Since Direct Connect is already established, downstream applications can access the EFS mount target through the VPC via the Direct Connect private VIF. This provides consistent, low-latency access to files uploaded by vendors.
