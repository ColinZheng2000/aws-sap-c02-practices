A company is using Amazon OpenSearch Service to analyze data. The company loads data into an OpenSearch Service cluster with 10 data nodes from an Amazon S3 bucket that uses S3 Standard storage. The data resides in the cluster for 1 month for read-only analysis. After 1 month, the company deletes the index that contains the data from the cluster. For compliance purposes, the company must retain a copy of all input data.  
  
The company is concerned about ongoing costs and asks a solutions architect to recommend a new solution.  
  
Which solution will meet these requirements MOST cost-effectively?

- A. Replace all the data nodes with UltraWarm nodes to handle the expected capacity. Transition the input data from S3 Standard to S3 Glacier Deep Archive when the company loads the data into the cluster.
- B. Reduce the number of data nodes in the cluster to 2 Add UltraWarm nodes to handle the expected capacity. Configure the indexes to transition to UltraWarm when OpenSearch Service ingests the data. Transition the input data to S3 Glacier Deep Archive after 1 month by using an S3 Lifecycle policy.
- C. Reduce the number of data nodes in the cluster to 2. Add UltraWarm nodes to handle the expected capacity. Configure the indexes to transition to UltraWarm when OpenSearch Service ingests the data. Add cold storage nodes to the cluster Transition the indexes from UltraWarm to cold storage. Delete the input data from the S3 bucket after 1 month by using an S3 Lifecycle policy.
- D. Reduce the number of data nodes in the cluster to 2. Add instance-backed data nodes to handle the expected capacity. Transition the input data from S3 Standard to S3 Glacier Deep Archive when the company loads the data into the cluster.


## AWS OpenSearch Service Cost Optimization and Data Retention

Tags:  #OpenSearchService #S3 #LifecyclePolicy #UltraWarm #ColdStorage #CostOptimization 

---

### Understanding the Problem

The company is currently using a 10-node Amazon OpenSearch Service cluster for 1 month of **read-only analysis** and is concerned about **ongoing costs**. The input data is sourced from **Amazon S3 Standard**. The core requirements are:

1. **Reduce costs** for the OpenSearch Service cluster.
    
2. Maintain **1 month** of data in the cluster for read-only analysis.
    
3. **Retain a copy** of all input data for **compliance** purposes **after** the data is deleted from the cluster.
    

This scenario is a classic use case for multi-tier data storage within OpenSearch Service and cost-effective long-term archival in S3. The solution must integrate both service features to minimize cost while meeting retention and compliance needs.

---

### Services Explanation

|**Service**|**Feature/Concept**|**Description**|
|---|---|---|
|**Amazon OpenSearch Service**|**Hot Data Nodes**|High-performance, instance-backed storage for indexing, updates, and fast querying. **Highest cost tier.**|
|**Amazon OpenSearch Service**|**UltraWarm Nodes**|Uses a combination of nodes and **S3** for storage. Data is immutable, read-only, and queried less frequently. **Significantly lower cost** than hot nodes for read-only data.|
|**Amazon OpenSearch Service**|**Cold Storage**|The **lowest cost** tier in OpenSearch Service. Data is stored fully on **S3**, not locally on node instance storage. Data must be **attached** (moved to UltraWarm) before querying. **Lowest TCO for inactive data.**|
|**Amazon S3**|**S3 Standard**|Highly durable, available, and performant object storage for frequently accessed data.|
|**Amazon S3**|**S3 Glacier Deep Archive**|The **lowest-cost** storage class in S3 for long-term retention and digital preservation. Ideal for compliance archives. Retrieval time is in hours.|
|**Amazon S3**|**S3 Lifecycle Policy**|Automated rules to manage object storage based on age. Can transition data to cheaper storage classes (like Glacier Deep Archive) or expire (delete) objects.|

---

### Correct Option(s) Explanation

**The correct option is B.**

**B. Reduce the number of data nodes in the cluster to 2. Add UltraWarm nodes to handle the expected capacity. Configure the indexes to transition to UltraWarm when OpenSearch Service ingests the data. Transition the input data to S3 Glacier Deep Archive after 1 month by using an S3 Lifecycle policy.**

This solution addresses all requirements and is the **MOST cost-effective**:

1. **Cost Reduction in OpenSearch Service:** The cluster is reduced from 10 hot data nodes to **2 hot data nodes** (minimum needed for stability/indexing) and **UltraWarm nodes** for the bulk of the 1-month read-only data. OpenSearch's **UltraWarm** tier is designed precisely for data that is infrequently accessed and read-only, offering **significantly lower cost** per GB than hot nodes by leveraging S3 for storage. Since the data is only for read-only analysis, UltraWarm is the perfect fit. (Refer to the official AWS documentation on OpenSearch Service storage tiers).
    
2. **Compliance/Retention:** The input data in S3 (which the company must retain) is handled by an **S3 Lifecycle policy** to transition the data to **S3 Glacier Deep Archive** after **1 month**. This is the **most cost-effective** and standard way to meet long-term compliance retention requirements for source data, as S3 Glacier Deep Archive is AWS's lowest-cost archival storage.

### Incorrect Options Explanation

|**Option**|**Reason for Incorrectness**|
|---|---|
|**A. Replace all the data nodes with UltraWarm nodes... Transition the input data from S3 Standard to S3 Glacier Deep Archive when the company loads the data into the cluster.**|**OpenSearch Service Requirement:** UltraWarm nodes **cannot** handle initial data ingestion and indexing; this requires **hot data nodes**. You can't replace _all_ hot nodes. **Retention Timing:** Transitioning the input data to S3 Glacier Deep Archive _immediately_ is a misinterpretation of the requirement. The retention is for the _input data_, but the current practice of deleting the _index_ after 1 month implies the S3 data only needs to be archived _after_ the 1-month analysis window.|
|**C. Reduce the number of data nodes... Add UltraWarm nodes... Transition the indexes... Add cold storage nodes... Delete the input data from the S3 bucket after 1 month by using an S3 Lifecycle policy.**|**OpenSearch Service Cost/Complexity:** This introduces **Cold Storage** and a redundant transition (Hot -> UltraWarm -> Cold). While Cold Storage is cheaper than UltraWarm, the data only needs to be kept for **1 month** and is currently being **deleted** from the cluster afterward. Adding Cold Storage is unnecessarily complex and not the _MOST_ cost-effective solution compared to simply archiving the _source data_ in S3. **Compliance Failure:** Deleting the input data from the S3 bucket after 1 month using a Lifecycle policy **violates the compliance requirement** to _retain a copy_ of all input data.|
|**D. Reduce the number of data nodes in the cluster to 2. Add instance-backed data nodes to handle the expected capacity. Transition the input data from S3 Standard to S3 Glacier Deep Archive when the company loads the data into the cluster.**|**OpenSearch Service Cost:** Adding more **instance-backed data nodes** (Hot Nodes) defeats the primary goal of cost reduction. UltraWarm nodes are the cost-effective solution for read-only data. **Retention Timing:** Transitioning the input data to S3 Glacier Deep Archive _immediately_ is unnecessary and prematurely archives the data, similar to the issue in Option A.|
