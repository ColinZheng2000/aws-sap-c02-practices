A company has an on-premises data center and is using Kubernetes to develop a new solution on AWS. The company uses Amazon Elastic Kubernetes Service (Amazon EKS) clusters for its development and test environments.

The EKS control plane and data plane for production workloads must reside on premises. The company needs an AWS managed solution for Kubernetes management.

Which solution will meet these requirements with the LEAST operational overhead?

A. Install an AWS Outposts server in the on-premises data center. Deploy Amazon EKS by using a local cluster configuration on the Outposts server for the production workloads.
B. Install Amazon EKS Anywhere on the company's hardware in the on-premises data center. Deploy the production workloads on an EKS Anywhere cluster.
C. Install an AWS Outposts server in the on-premises data center. Deploy Amazon EKS by using an extended cluster configuration on the Outposts server for the production workloads.
D. Install an AWS Outposts server in the on-premises data center. Install Amazon EKS Anywhere on the Outposts server. Deploy the production workloads on an EKS Anywhere cluster.



# AWS SAP-C02 Question Analysis — Q355

## 📋 Analysis

### 1. Understanding the Problem

The company already uses Amazon EKS for development and testing in AWS, but production has stricter enterprise requirements:

- Kubernetes production workloads must stay on premises.
- Both:
  - Kubernetes control plane
  - Kubernetes worker nodes (data plane)
  must reside locally in the company data center.
- The company still wants an AWS-managed Kubernetes solution.
- The solution should minimize operational effort.

This question tests the architectural differences between:

- Amazon EKS
- Amazon EKS on AWS Outposts
- Amazon EKS Anywhere

The key exam keyword is:

- “least operational overhead”

Combined with:

- “control plane and data plane must reside on premises”

This immediately points toward EKS local clusters on Outposts.

- --

### 2. Options Analysis

#### ✅ Option A (CORRECT): Use Amazon EKS local clusters on AWS Outposts

This is the correct solution because Amazon EKS local clusters on Outposts provide:

- Local Kubernetes control plane on Outposts
- Local worker nodes on Outposts
- AWS-managed Kubernetes experience
- Reduced operational complexity compared to self-managed Kubernetes

Why this satisfies all requirements:

1. Both planes remain on premises
- EKS local clusters place:
  - Kubernetes API server
  - etcd
  - scheduler
  - worker nodes
  inside the Outposts environment.

2. AWS-managed operational model
- AWS manages:
  - Outposts hardware lifecycle
  - infrastructure integration
  - EKS service integration
- The customer avoids heavy Kubernetes infrastructure management.

3. Lowest operational overhead
- Compared with EKS Anywhere, this removes much of:
  - cluster lifecycle management
  - infrastructure maintenance
  - Kubernetes operational burden

Enterprise reality:
- Medical companies often require:
  - local processing
  - compliance boundaries
  - deterministic latency
  - operational continuity during WAN interruptions
- Industrial manufacturing environments frequently use:
  - edge analytics
  - factory automation
  - machine telemetry processing
  requiring local Kubernetes execution.

Important SAP-C02 distinction:

| Deployment Type | Control Plane | Worker Nodes |
| --- | --- | --- |
| EKS Regional | AWS Region | AWS Region |
| EKS Extended Cluster | AWS Region | Outposts |
| EKS Local Cluster | Outposts | Outposts |

Official AWS reference:
- Amazon EKS local clusters on AWS Outposts

https://docs.aws.amazon.com/eks/latest/userguide/eks-outposts-local-cluster-overview.html

- --

#### ❌ Option B (INCORRECT): Use Amazon EKS Anywhere on customer hardware

This option satisfies the “on-premises” requirement but fails the “least operational overhead” requirement.

Why:

EKS Anywhere is primarily customer-managed Kubernetes.

The company becomes responsible for:
- cluster provisioning
- patching
- upgrades
- lifecycle management
- operational troubleshooting
- infrastructure compatibility validation

AWS provides:
- tooling
- validated Kubernetes distributions
- integrations

But AWS does NOT fully manage the Kubernetes environment the same way as EKS local clusters on Outposts.

Key SAP-C02 concept:
- “AWS managed” usually favors native AWS-managed services over self-managed variants.

This option is operationally heavier than Option A.

Official AWS reference:
https://anywhere.eks.amazonaws.com/

- --

#### ❌ Option C (INCORRECT): Use EKS extended clusters on Outposts

This fails a core requirement.

In EKS extended clusters:
- Worker nodes run on Outposts
- Kubernetes control plane remains in the AWS Region

The question explicitly requires:
- control plane on premises
- data plane on premises

Therefore this architecture is invalid.

This is one of the most important EKS hybrid deployment distinctions for the SAP exam.

Exam memory shortcut:

- Extended cluster:
  - “extend workers”
- Local cluster:
  - “everything local”

Official AWS reference:
https://docs.aws.amazon.com/eks/latest/userguide/eks-outposts.html

- --

#### ❌ Option D (INCORRECT): Install EKS Anywhere on Outposts

This is unnecessarily complicated and increases operational burden.

Problems:

1. Duplicated architecture complexity
- Outposts already supports native EKS local clusters.
- Adding EKS Anywhere creates an unnecessary abstraction layer.

2. Higher operational responsibility
- EKS Anywhere still requires customer-managed Kubernetes operations.

3. Architectural anti-pattern
- Running customer-managed Kubernetes on AWS-managed Outposts hardware wastes the benefits of native EKS integration.

This directly violates:
- “least operational overhead”

In enterprise environments, simpler operational models are strongly preferred for:
- auditability
- supportability
- upgrade management
- incident response

### 3. Hands-On Knowledge

#### Real Enterprise Deployment Guidance

Use EKS Local Clusters on Outposts when:
- production systems must stay local
- regulatory requirements prohibit regional control planes
- manufacturing plants need ultra-low latency
- intermittent WAN connectivity exists
- edge computing is required

Typical industries:
- healthcare
- pharmaceutical manufacturing
- oil & gas
- automotive factories
- defense

- --

#### Operational Difference: EKS Anywhere vs EKS Local Clusters

| Feature | EKS Anywhere | EKS Local Cluster |
| --- | --- | --- |
| Hardware Ownership | Customer | AWS Outposts |
| Kubernetes Management | Customer-heavy | AWS-managed integration |
| Control Plane Location | On-prem | On-prem |
| Operational Overhead | Higher | Lower |
| SAP-C02 Preference | Sometimes | Usually preferred |

- --

#### Common Interview Gotcha

Many engineers mistakenly think:
- “EKS Anywhere” = fully managed EKS on premises

This is incorrect.

EKS Anywhere is:
- an AWS-supported Kubernetes distribution
- but operational ownership remains largely with the customer.

Meanwhile:
- EKS local clusters on Outposts provide a much more AWS-managed operational experience.

- --

#### Example EKS Anywhere Cluster Creation

```bash
eksctl anywhere create cluster -f cluster.yaml
```

This demonstrates:
- customer-driven cluster lifecycle management.

In contrast, EKS on Outposts integrates directly with AWS-managed infrastructure services.

- --

#### Architecture Design Insight

For regulated enterprises:

- Outposts is often selected when:
  - compliance teams require AWS operational accountability
  - standardized hardware support is important
  - managed infrastructure reduces audit complexity

- EKS Anywhere is more suitable when:
  - existing hardware investments must be preserved
  - AWS Outposts procurement is not feasible
  - disconnected operations are required.

* *面试官可能追问 (Possible Follow-up Questions)**：

1. "What is the difference between EKS local clusters and EKS extended clusters?"
   → 回答要点 (Key Answer Points)：EKS local clusters place both the Kubernetes control plane and worker nodes on Outposts, enabling fully local Kubernetes operations. EKS extended clusters keep the control plane in the AWS Region while only worker nodes run locally on Outposts.

2. "Why would a company choose EKS Anywhere instead of EKS on Outposts?"
   → 回答要点 (Key Answer Points)：EKS Anywhere is useful when companies want Kubernetes on their own hardware or require disconnected environments. However, they must accept higher operational responsibility for upgrades, patching, and lifecycle management.
