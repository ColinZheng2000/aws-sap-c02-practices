A company developed a pilot application by using AWS Elastic Beanstalk and Java. To save costs during development, the company's development team deployed the application into a single-instance environment. Recent tests indicate that the application consumes more CPU than expected. CPU utilization is regularly greater than 85%, which causes some performance bottlenecks.  
  
A solutions architect must mitigate the performance issues before the company launches the application to production.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Create a new Elastic Beanstalk application. Select a load-balanced environment type. Select all Availability Zones. Add a scale-out rule that will run if the maximum CPU utilization is over 85% for 5 minutes.
- B. Create a second Elastic Beanstalk environment. Apply the traffic-splitting deployment policy. Specify a percentage of incoming traffic to direct to the new environment in the average CPU utilization is over 85% for 5 minutes.
- C. Modify the existing environment’s capacity configuration to use a load-balanced environment type. Select all Availability Zones. Add a scale-out rule that will run if the average CPU utilization is over 85% for 5 minutes.
- D. Select the Rebuild environment action with the load balancing option. Select an Availability Zones. Add a scale-out rule that will run if the sum CPU utilization is over 85% for 5 minutes.

## Analysis

### 1. Understanding the Problem
The company is currently running a **Single-Instance Environment** in Elastic Beanstalk. This type of environment contains one EC2 instance with an Elastic IP address, but it lacks an Elastic Load Balancer (ELB) and an Auto Scaling Group configured for scaling out (adding more instances).

* **The Issue:** The CPU usage exceeds 85%, causing bottlenecks. The current single instance cannot handle the load.
* **The Goal:** Enable the application to scale horizontally (add more instances) to handle the load.
* **The Constraint:** The solution must have the **LEAST operational overhead**.



### 2. Services Involved

* **AWS Elastic Beanstalk:** A Platform as a Service (PaaS) that orchestrates the deployment of applications. It handles capacity provisioning, load balancing, and auto-scaling.
    * **Single-Instance Environment:** Designed for development/low cost. Contains 1 EC2, 1 EIP, no ELB.
    * **Load-Balanced Environment:** Designed for production. Contains an Auto Scaling Group (ASG) and an Elastic Load Balancer (ELB).
* **Auto Scaling Group (ASG):** Monitors application metrics (like CPU) and adjusts capacity by adding or removing EC2 instances.
* **Elastic Load Balancer (ELB):** Distributes incoming application traffic across multiple targets (EC2 instances) in different Availability Zones.

### 3. Correct Option Explanation

**C. Modify the existing environment’s capacity configuration to use a load-balanced environment type. Select all Availability Zones. Add a scale-out rule that will run if the average CPU utilization is over 85% for 5 minutes.**

This is the correct answer because Elastic Beanstalk allows you to change the configuration of an *existing* environment from "Single Instance" to "Load Balanced" directly within the console or via CLI without recreating the application or environment from scratch.

* **Operational Overhead:** This is the lowest. You simply navigate to the **Configuration** tab, select **Capacity**, and change the **Environment Type** from Single Instance to Load Balanced. Elastic Beanstalk automatically provisions the ELB and updates the ASG settings for you.
* **Scaling Logic:** Setting the trigger to Average CPU > 85% addresses the specific bottleneck mentioned in the scenario. Selecting all Availability Zones ensures High Availability (HA).

### 4. Incorrect Options Explanation

**A. Create a new Elastic Beanstalk application...**
* **Why it is incorrect:** Creating a completely new *Application* (the top-level container in Beanstalk) is unnecessary. You would have to migrate the code bundles, environment variables, and configuration settings to the new application. This incurs significantly higher operational overhead than modifying the existing one.

**B. Create a second Elastic Beanstalk environment. Apply the traffic-splitting deployment policy...**
* **Why it is incorrect:** 1.  **Traffic Splitting** in Beanstalk is primarily used for **Canary Deployments** (testing a new version of code with a small percentage of traffic), not for auto-scaling based on load. 
    2.  Managing two distinct environments to handle load manually (or via traffic splitting rules) creates complexity.
    3.  It does not solve the root cause: the environment needs to be able to scale out autonomously.

**D. Select the Rebuild environment action with the load balancing option...**
* **Why it is incorrect:** 1.  The **Rebuild Environment** action terminates and re-provisions resources, but it generally rebuilds based on the *saved configuration*. You cannot simply select "load balancing option" *during* the rebuild action button click. You must change the configuration first (which is Option C).
    2.  Using **Sum CPU** is generally less effective for this specific scaling scenario than **Average CPU**. If you have 1 instance at 90% CPU, the Sum is 90. If you scale to 2 instances at 50% each, the Sum is 100. Scaling based on "Sum" requires constant math adjustments based on the number of instances, whereas "Average" normalizes the metric regardless of the instance count.
