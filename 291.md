A company is running a workload that consists of thousands of Amazon EC2 instances. The workload is running in a VPC that contains several public subnets and private subnets. The public subnets have a route for 0.0.0.0/0 to an existing internet gateway. The private subnets have a route for 0.0.0.0/0 to an existing NAT gateway.
A solutions architect needs to migrate the entire fleet of EC2 instances to use IPv6. The EC2 instances that are in private subnets must not be accessible from the public internet.
What should the solutions architect do to meet these requirements?

A. Update the existing VPC, and associate a custom IPv6 CIDR block with the VPC and all subnets. Update all the VPC route tables, and add a route for ::/0 to the internet gateway.
B. Update the existing VPC, and associate an Amazon-provided IPv6 CIDR block with the VPC and all subnets. Update the VPC route tables for all private subnets, and add a route for ::/0 to the NAT gateway.
C. Update the existing VPC, and associate an Amazon-provided IPv6 CIDR block with the VPC and all subnets. Create an egress-only internet gateway. Update the VPC route tables for all private subnets, and add a route for ::/0 to the egress-only internet gateway.
D. Update the existing VPC, and associate a custom IPv6 CIDR block with the VPC and all subnets. Create a new NAT gateway, and enable IPv6 support. Update the VPC route tables for all private subnets, and add a route for ::/0 to the IPv6-enabled NAT gateway.

# AWS SAP-C02 Analysis — Migrating EC2 Fleet to IPv6 While Keeping Private Subnets Non-Public

## Understanding the Problem

The environment currently uses:

- Public subnets:
  - `0.0.0.0/0` → Internet Gateway (IGW)
- Private subnets:
  - `0.0.0.0/0` NAT Gateway

The company wants:

- Full IPv6 support for EC2 instances
- Private subnet instances must NOT be directly reachable from the internet
- Maintain outbound internet access for private instances

Key AWS networking concepts:

- IPv6 in AWS is globally routable
- NAT Gateway is primarily for IPv4 NAT
- IPv6 does not NAT in the same way as IPv4
- Private IPv6 outbound access requires an Egress-Only Internet Gateway

- --

# Correct Answer

## ✅ C. Use an Amazonvided IPv6 CIDR, associate it with subnets, create an egress-only internet gateway, and route ::/0 from private subnets to the egress-only IGW

### Why this is correct

This is the AWS-recommended architecture for private IPv6 outbound connectivity.

Steps involved:

1. Associate an Amazon-provided IPv6 CIDR block with the VPC
2. Associate IPv6 CIDRs with subnets
3. Create an Egress-Only Internet Gateway (EIGW)
4. Add IPv6 route:
   - `::/0 → Egress-Only Internet`
   - for private subnets only

Result:

- Private instances can initiate outbound IPv6 traffic
- Inbound unsolicited internet traffic is blocked
- Public subnets can still use standard Internet Gateway if needed

- --

## Key AWS Networking Behavior

### IPv4 Private Connectivity
Uses:
- NAT Gateway

Reason:
- IPv4 private addresses are not publicly routable

### IPv6 Private Connectivity
Uses:
- Egress-Only Internet Gateway

Reason:
- IPv addresses are globally unique and publicly routable
- AWS uses EIGW to block unsolicited inbound traffic

- --

# Why Other Options Are Incorrect

- --

## ❌ A. Custom IPv6 CIDR + route all ::/0 to IGW

### Problems

1. Internet Gateway allows bidirectional internet connectivity
   - Private subnet instances would become publicly reachable

2. Requirement explicitly states:
   - Private instances must NOT be accessible from public internet

3. Using a custom IPv6 CIDR is unnecessary complexity
   - Amazon-provided IPv6 CIDRs are standard best practice

### Important Concept

IGW for IPv6 behaves differently from NAT:
- Any globally routable IPv6 address can potentially receive inbound traffic if security groups/NACLs allow it

- --

## ❌ B. Route IPv6 traffic to NAT Gateway

### Problems

NAT Gateway does NOT provide traditional IPv6 NAT functionality.

For IPv6:
- AWS uses Egress-Only Internet Gateway
- NAT Gateway is not the intended architectural component

This option reflects misunderstanding of IPv4 vs IPv6 design.

### SAP-C02 Exam Trap

A common exam distractor:
- Assuming NAT Gateway works identically for IPv6

It does not.

- --

## ❌ D. Create IPv6-enabled NAT Gateway

### Problems

There is no standard architecture where NAT Gateway replaces EIGW for IPv6 private outbound internet access.

Even though AWS has introduced NAT64/DNS64 capabilities for some scenarios:
- That is for IPv6-to-IPv4 translation
- Not for general private IPv6 internet egress protection

The canonical AWS design remains:
- Egress-Only Internet Gateway

Additionally:
- "Enable IPv6 support on NAT Gateway" is misleading in this context

- --

| Requirement | AWS Component |
| --- | --- |
| Public IPv6 internet access | Internet Gateway |
| Private outbound-only IPv6 access | Egress-Only Internet Gateway |
| Private outbound IPv4 access | NAT Gateway |

# Enterprise Hands-On Knowledge

## Real-World IPv6 Migration Pattern

Large enterprises usually migrate in phases:

### Phase 1 — Dual Stack
- Keep IPv4
- Add6 alongside it

Reason:
- Many enterprise applications still depend on IPv4

- --

### Phase 2 — Public Tier
- AL/NLBs enabled for dual-stack
- Public subnets get IGW IPv6 routes

- --

### Phase 3 — Private Tier
- Private application instances use:
  - `::/0 → Egress-Only IGW`

This preserves:
- Outbound package downloads
- API calls
- Patch management

Without exposing:
- SSH
- RDP
- Internal services

- --

## Security Group Considerations

With IPv6:

- `0.0.0.0/0` becomes:
  - `::/0`

Security groups often accidentally become overly permissive during migration.

Common enterprise control:
- Separate IPv4 and IPv6 security policies
- AWS Config rules validating IPv6 exposure

- --

## Operational Considerations

### VPC Flow Logs
Ensure:
- IPv6 traffic logging enabled

Useful during:
- Security audits
- Migration troubleshooting

### IAM & Automation

Large fleets usually automate IPv6 migration using:
- CloudFormation
- Terraform
- AWS CDK

Because:
- Thousands of ENIs and route tables may require updates

# SAP-C02 Exam Takeaway

Memorize this mapping:

| Connectivity Type | Solution |
| --- | --- |
| Public IPv4 | IGW |
| Private IPv4 outbound | NAT Gateway |
| Public IPv6 | IGW |
| Private IPv6 outbound-only | Egress-Only IGW |

For SAP-C02, if the question says:

- "IPv6"
- "private subnet"
- "must not be publicly accessible"

The answer is almost always:
- Egress-Only Internet Gateway
