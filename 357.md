A company used AWS CloudFormation to create all new infrastructure in its AWS member accounts. The resources rarely change and are properly sized for the expected load. The monthly AWS bill is consistent.

Occasionally, a developer creates a new resource for testing and forgets to remove the resource when the test is complete. Most of these tests last a few days before the resources are no longer needed.

The company wants to automate the process of finding unused resources. A solutions architect needs to design a solution that determines whether the cost in the AWS bill is increasing. The solution must help identify resources that cause an increase in cost and must automatically notify the company's operations team.

Which solution will meet these requirements?

A. Turn on billing alerts. Use AWS Cost Explorer to determine the costs for the past month. Create an Amazon CloudWatch alarm for total estimated charges. Specify a cost threshold that is higher than the costs that Cost Explorer determined. Add a notification to alert the operations team if the alarm threshold is breached.
B. Turn on billing alerts. Use AWS Cost Explorer to determine the average monthly costs for the past 3 months. Create an Amazon CloudWatch alarm for total estimated charges. Specify a cost threshold that is higher than the costs that Cost Explorer determined. Add a notification to alert the operations team if the alarm threshold is breached.
C. Use AWS Cost Anomaly Detection to create a cost monitor that has a monitor type of Linked account. Create a subscription to send daily AWS cost summaries to the operations team. Specify a threshold for cost variance.
D. Use AWS Cost Anomaly Detection to create a cost monitor that has a monitor type of AWS services. Create a subscription to send daily AWS cost summaries to the operations team. Specify a threshold for cost variance.


# AWS SAP-C02 Question Analysis — Q357

## 📋 Analysis

### 1. Understanding the Problem

The environment is operationally stable because infrastructure is provisioned through AWS CloudFormation and resource sizing is already optimized. Monthly spending is therefore predictable.

The actual operational risk comes from:
- temporary developer-created resources
- forgotten test environments
- short-lived cost spikes
- manual cleanup failures

The company needs a solution that can:
- automatically detect unusual spending behavior
- identify what caused the increase
- notify the operations team
- work well in a stable baseline environment

This is an AWS FinOps and operational governance scenario. Since the normal bill is steady, anomaly detection is more suitable than static threshold monitoring.

- --

### 2. Options Analysis

#### ❌ Option A (INCORRECT): Single-month static billing threshold

This approach uses:
- Billing alerts
- AWS Cost Explorer historical review
- CloudWatch billing alarms

Why it is insufficient:

1. Static threshold limitation
- CloudWatch billing alarms only compare against a manually defined number.
- If developers create temporary resources that do not exceed the threshold significantly, the anomaly may go undetected.

2. Poor anomaly intelligence
- The solution cannot determine which AWS service caused the increase.
- Operations teams would still need manual investigation.

3. Weak baseline model
- Using only the previous month is operationally weak for enterprise trend analysis.

Enterprise anti-pattern:
- Static billing alarms become difficult to maintain as environments evolve.

AWS generally recommends Cost Anomaly Detection for intelligent cost variance monitoring.

- --

#### ❌ Option B (INCORRECT): Three-month average threshold

This option is slightly better than Option A because it uses a broader historical baseline.

However, the core issue remains:
- still static threshold monitoring
- still dependent on manually selected limits
- still lacks anomaly intelligence
- still cannot directly identify the service causing increased spend

Operational issue:
- Cloud costs are dynamic.
- Static alarms frequently generate either:
  - false positives
  - missed anomalies

This does not fully satisfy the requirement to help identify cost-causing resources/services automatically.

- --

#### ❌ Option C (INCORRECT): Cost anomaly detection with Linked account monitor

This option correctly introduces AWS Cost Anomaly Detection, which is the appropriate AWS-native service.

However, the monitor type is not optimal.

Why:

1. Linked account monitor scope
- Detects anomalies per AWS account
- Useful for multi-account governance

2. Reduced troubleshooting granularity
- The requirement focuses on identifying resources causing cost increases.
- Service-level visibility is more operationally useful than account-level visibility here.

Example:
- A linked account anomaly only indicates:
  "Account X has increased spend."
- It does not immediately indicate whether:
  - EC2
  - NAT Gateway
  - RDS
  - EBS
  caused the anomaly.

This slows incident investigation.

- --

#### ✅ Option D (CORRECT): AWS services monitor with anomaly subscription

This is the best solution because it directly matches all operational requirements.

Why this works:

1. AWS Cost Anomaly Detection uses ML-based analysis
- Learns historical spending behavior automatically
- Detects abnormal spend increases dynamically
- No need for manually maintained thresholds

2. AWS services monitor provides better operational visibility
- Detects anomalies grouped by AWS service
- Makes troubleshooting significantly faster

Examples:
- Unexpected EC2 instance usage
- Forgotten RDS databases
- Idle NAT Gateways
- Unattached EBS volumes
- Excessive data transfer charges

3. Subscription notifications automate operational response
- Daily summaries can be sent automatically
- Teams receive alerts without manual review
- Fits enterprise operational governance patterns

4. Variance thresholds reduce noise
- Prevents unnecessary alerts for tiny changes
- Supports practical FinOps operations

Why this is architecturally superior:
- adaptive monitoring
- lower operational overhead
- faster root-cause identification
- AWS-native managed solution

### 3. Hands-On Knowledge

In enterprise AWS environments, forgotten temporary resources commonly come from:

- EC2 test instances
- NAT Gateways
- Elastic Load Balancers
- RDS databases
- EBS snapshots
- Elastic IP addresses
- Public IPv4 allocations

A very common real-world issue:
- Developers terminate EC2 instances
- But associated EBS volumes remain attached or orphaned
- Storage costs continue silently

Typical enterprise implementation pattern:

1. Cost anomaly detected
- AWS Cost Anomaly Detection identifies unusual spend

2. SNS notification triggered
- Sent to:
  - Operations
  - FinOps team
  - Slack/Teams integrations

3. Automated investigation
- Lambda queries:
  - Cost Explorer API
  - Resource Groups Tagging API
  - AWS Config

4. Optional remediation
- Stop idle resources automatically
- Create ServiceNow/Jira tickets
- Notify resource owners

Example CLI:

```bash
aws ce get-anomalies \
  - -monitor-arn arn:aws:ce::123456789012:anomalymonitor/example-monitor
```

Best-practice governance enhancement:

Use mandatory tagging policies:
- Owner
- Project
- CostCenter
- ExpirationDate
- Environment

Combined with:
- AWS Organizations Tag Policies
- SCP guardrails
- AWS Config compliance rules

This enables automated cleanup of expired test resources.

Important SAP-C02 exam insight:

- CloudWatch billing alarms
  = static threshold monitoring

- AWS Cost Anomaly Detection
  = ML-based adaptive cost analysis

For modern enterprise environments, AWS prefers anomaly-based monitoring for operational cost governance.

* *面试官可能追问 (Possible Follow-up Questions)**：

1. "AWS Budgets 和 AWS Cost Anomaly Detection 的区别是什么？"
   → 回答要点 (Key Answer Points)：
   AWS Budgets is proactive governance using predefined thresholds. AWS Cost Anomaly Detection is reactive ML-based monitoring for unusual spending behavior. Enterprises commonly combine both for complete FinOps governance.

2. "为什么 Service monitor 比 Linked account monitor 更适合这个场景？"
   → 回答要点 (Key Answer Points)：
   Service-level monitoring provides better operational troubleshooting because it identifies which AWS service generated the abnormal spend. Linked account monitoring is better for account-level governance but lacks detailed operational visibility.
