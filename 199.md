A financial services company loaded millions of historical stock trades into an Amazon DynamoDB table. The table uses on-demand capacity mode. Once each day at midnight, a few million new records are loaded into the table. Application read activity against the table happens in bursts throughout the day, and a limited set of keys are repeatedly looked up. The company needs to reduce costs associated with DynamoDB.
Which strategy should a solutions architect recommend to meet this requirement?

A. Deploy an Amazon ElastiCache cluster in front of the DynamoDB table.
B. Deploy DynamoDB Accelerator (DAX). Configure DynamoDB auto scaling. Purchase Savings Plans in Cost Explorer.
C. Use provisioned capacity mode. Purchase Savings Plans in Cost Explorer.
D. Deploy DynamoDB Accelerator (DAX). Use provisioned capacity mode. Configure DynamoDB auto scaling.


Suggest Answer: D

---

## Analysis

### Understanding the Problem

A financial services company stores millions of historical stock trades in a DynamoDB table using **on-demand capacity mode**. Workload characteristics: (1) **predictable daily batch write** of a few million records at midnight, (2) **bursty read activity** throughout the day, and (3) a **limited set of keys** is repeatedly looked up (hot keys). The goal is to **reduce DynamoDB costs**. The key insight: on-demand mode charges per-request pricing (~$1.25/million writes, $0.25/million reads), which becomes expensive for sustained, predictable workloads. Switching to provisioned capacity + caching + auto scaling yields significant savings.

### Correct Option: D

**Deploy DynamoDB Accelerator (DAX). Use provisioned capacity mode. Configure DynamoDB auto scaling.**

This solution attacks costs from three angles:

- **DynamoDB Accelerator (DAX)**: DAX is a fully managed, in-memory cache cluster purpose-built for DynamoDB. It delivers **microsecond latency** for cached reads. Since the workload repeatedly looks up a limited set of keys, DAX will absorb the vast majority of read traffic, dramatically reducing the number of read request units (RRUs) consumed against the DynamoDB table. Key features: write-through caching (writes go to DynamoDB first, then DAX updates), support for both item-level and query cache, and seamless compatibility with existing DynamoDB API calls (no application code changes beyond pointing to the DAX cluster endpoint).

- **Provisioned Capacity Mode**: With a predictable daily batch write at midnight, provisioned mode is far more cost-effective than on-demand. You can provision the exact write capacity needed for the batch job (e.g., a few million writes over one hour ≈ 500-1000 WCUs). Provisioned write capacity costs $0.00065 per WCU-hour — significantly cheaper than on-demand writes at $1.25 per million write request units for sustained workloads.

- **DynamoDB Auto Scaling**: Manages the provisioned read/write capacity automatically based on utilization targets. During read bursts, auto scaling increases RCUs; during quiet periods, it scales down. This prevents over-provisioning (paying for unused capacity 24/7) while ensuring the table doesn't throttle during peaks. The combination of DAX (handling hot-key reads) + auto scaling (handling residual read/write bursts) provides both cost efficiency and performance.

### Incorrect Options

**A: ElastiCache cluster in front of DynamoDB** — Amazon ElastiCache (Redis or Memcached) **can** cache DynamoDB reads, but it's a **DIY approach** compared to DAX. You must implement cache logic in your application: check cache → miss → read DynamoDB → populate cache, plus handle cache invalidation and TTL logic. DAX handles all of this transparently (write-through, item cache, LRU eviction). ElastiCache also introduces additional operational overhead (cluster management, node type selection, parameter tuning) and does not benefit from the microsecond latency that DAX achieves through its purpose-built architecture. Critically, **this option does not switch from on-demand to provisioned mode**, missing the largest cost optimization lever.

**B: DAX + DynamoDB auto scaling + Savings Plans** — **Savings Plans do not apply to DynamoDB.** AWS Savings Plans cover EC2 instances, AWS Lambda, and AWS Fargate usage. For DynamoDB, the equivalent commitment-based discount is **DynamoDB Reserved Capacity**, purchased through the DynamoDB console (not Cost Explorer Savings Plans). This is a factual error that makes the option invalid regardless of its other merits.

**C: Provisioned capacity mode + Savings Plans** — Same Savings Plans incompatibility as Option B. Even if we substitute "Reserved Capacity" for "Savings Plans," this option still misses the most impactful cost optimization for the described workload: **caching frequently accessed hot keys**. Without DAX, every bursty read hits the DynamoDB table directly, consuming RCUs. You would need to provision sufficient RCUs to handle peak read bursts, which is expensive. DAX reduces the RCU requirement by potentially 90%+ for the hot-key read pattern — a far greater savings lever than reserved capacity alone.

### Hands-On Knowledge

1. **On-Demand vs Provisioned Cheat Sheet**:

| Workload Pattern | Recommended Mode |
|---|---|
| Unknown/unpredictable traffic | On-Demand |
| Predictable, steady-state traffic | Provisioned + Auto Scaling |
| Spiky traffic with >2x peak/trough ratio | On-Demand (unless you know peaks well) |
| Batch writes with known volume | Provisioned (exact WCU calculation) |

2. **DAX Cluster Sizing**: Start with a 2-node cluster (one primary + one read replica) for high availability. Instance types: `dax.r5.large` (for moderate workloads) to `dax.r5.8xlarge`. DAX nodes are deployed within a VPC and automatically perform automatic failover if the primary node fails.

3. **Cost Calculation Example**:
   - On-Demand: 10 million reads/day × $0.25/million RRU = $2.50/day = $75/month for reads alone
   - Provisioned without DAX: 100 RCUs (for bursts) × $0.00013/RCU-hour × 730 hours = $9.49/month
   - Provisioned with DAX (80% cache hit rate): 20 RCUs (for cache misses) × $0.00013 × 730 = $1.90/month + DAX cluster cost (~$0.10/hour × 730 = $73/month for r5.large) = ~$74.90/month... but for millions of requests, DAX becomes cost-effective at scale when it prevents provisioning hundreds or thousands of RCUs. Always calculate your break-even point.

4. **DynamoDB Reserved Capacity**: Commit to a minimum provisioned throughput level for a 1-year or 3-year term. Discounts: 1-year ~33%, 3-year ~67%. Priced per 100 RCUs or 100 WCUs per region. Unlike Savings Plans, you pay for the reserved capacity even if you don't use it — make sure your baseline is predictable.

5. **DAX Limitations to Know**: 
   - DAX does not support `Scan` operations natively (they bypass the cache).
   - DAX is VPC-only — not accessible via the public internet.
   - DAX is strongly consistent for `GetItem` but eventually consistent for `Query` results (by default).
   - Encryption at rest is supported, but it must be enabled at cluster creation time.
