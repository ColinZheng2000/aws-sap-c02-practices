A company hosts a community forum site using an Application Load Balancer (ALB) and a Docker application hosted in an Amazon ECS cluster. The site data is stored in Amazon RDS for MySQL and the container image is stored in ECR. The company needs to provide their customers with a disaster recovery SLA with an RTO of no more than 24 hours and RPO of no more than 8 hours.

Which of the following solutions is the MOST cost-effective way to meet the requirements?

A. Use AWS CloudFormation to deploy identical ALB, EC2, ECS and RDS resources in two regions. Schedule RDS snapshots every 8 hours. Use RDS multi-region replication to update the secondary region's copy of the database. In the event of a failure, restore from the latest snapshot, and use an Amazon Route 53 DNS failover policy to automatically redirect customers to the ALB in the secondary region.
B. Store the Docker image in ECR in two regions. Schedule RDS snapshots every 8 hours with snapshots copied to the secondary region. In the event of a failure, use AWS CloudFormation to deploy the ALB, EC2, ECS and RDS resources in the secondary region, restore from the latest snapshot, and update the DNS record to point to the ALB in the secondary region.
C. Use AWS CloudFormation to deploy identical ALB, EC2, ECS, and RDS resources in a secondary region. Schedule hourly RDS MySQL backups to Amazon S3 and use cross-region replication to replicate data to a bucket in the secondary region. In the event of a failure, import the latest Docker image to Amazon ECR in the secondary region, deploy to the EC2 instance, restore the latest MySQL backup, and update the DNS record to point to the ALB in the secondary region.
D. Deploy a pilot light environment in a secondary region with an ALB and a minimal resource EC2 deployment for Docker in an AWS Auto Scaling group with a scaling policy to increase instance size and number of nodes. Create a cross-region read replica of the RDS data. In the event of a failure, promote the replica to primary, and update the DNS record to point to the ALB in the secondary region.

## Understanding the Problem
You have a containerized community forum behind an **ALB**, running on **ECS**, storing images in **ECR**, and using **RDS for MySQL** for data. The business requires a Disaster Recovery (DR) solution with:

- **RTO ≤ 24 hours** (service must be back within a day)
- **RPO ≤ 8 hours** (data loss must be no more than 8 hours)

The key is to pick the **MOST cost-effective** design that still meets these DR objectives. In AWS DR strategy terms (per AWS guidance), you generally choose among:
- Backup & restore (cheapest, slowest recovery)
- Pilot light
 Warm standby
- Multi-site active/active (most expensive, fastest)

Given **RTO=24h** and **RPO=8h**, a **backup & restore** approach is typically sufficient and cheapest.

## Correct Option Explanation (Most Cost-Effective)

### ✅ Option B
* *Why it meets RPO (≤ 8 hours):**
- Taking **RDS snapshots every 8 hours** and **copying snapshots to a secondary region** supports an RPO bounded by the snapshot frequency (up to 8 hours of data loss).
- Cross-Region snapshot copy is a standard DR technique for RDS.

* *Why it meets RTO (≤ 24 hours):**
- In a regional failure, you:
 1. Deploy infrastructure in the DR region (via **CloudFormation**)
  2. Restore **RDS** from the latest copied snapshot
  3. Point DNS to the DR region’s ALB  
- Even if RDS restore + ECS/ALB provisioning takes hours, it is typically feasible within **24 hours**, especially if CloudFormation templates are pre-built and tested.

* *Why it is the MOST cost-effective:**
- You are **not** continuously running a full duplicate stack (compute/ECS/ALB/RDS) in the second region.
- You mainly pay for:
  - Snapshot storage
  - Snapshot copy data transfer/storage
  - Minimal ECR replication/storage (or dual-region images)
- This aligns with AWS’s lowest-cost DR model: **Backup & Restore**.

* *AWS official guidance to connect this to:**
- AWS DR patterns describe **Backup and restore** the lowest cost option for less aggressive RTO/RPO targets.  
  Ref: *AWS Prescriptive Guidance – Disaster recovery (DR) strategies on AWS*  
  https://docs.amazon.com/prescriptive-guidance/latest/disaster-recovery-aws/welcome.html

- RDS snapshot copy across regions is the standard mechanism for regional DR with backups.  
  Ref: *Amazon RDS User Guide – Copying a DB snapshot*  
  https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_CopySnapshot.html

- ECR supports cross-Region replication (recommended vs manual “store in two regions”).  
  Ref: *Amazon ECR User Guide – Private registry replication*  
  https://docs.aws.amazon.com/AmazonECR/latest/userguide/replication.html

## Incorrect Options Explanation

### ❌ Option A
- Deploying **identical full environments in two regions** is closer to **warm standby / multi-site**, which is **more expensive** than needed for RTO 24h RPO 8h.
- “Use RDS multi-region replication” is also a red flag: for **RDS for MySQL**, the common native cross-region approach is **cross-Region read replicas** (async) rather than a generic “multi-region replication” feature wording.
- Restoring from snapshot in failover still implies downtime; if you’re already paying for full duplicate infrastructure, this option is internally inconsistent and cost-inefficient.

### ❌ Option C
- “Hourly MySQL backups to S3” “restore from S3” is not the typical managed, reliable DR posture for **RDS MySQL** compared with **automated backups/snapshots** and cross-region snapshot copy.
- Operationally heavy: you must manage logical dumps/exports, consistency, retention, encryption, and restore procedures yourself.
- Also mentions “import the latest Docker image to ECR” during DR—this adds avoidable recovery time and complexity (ECR replication is the standard approach).

### ❌ Option D
- This is a **pilot light** style solution: minimal compute in DR region plus **cross-region read replica**.
- It likely meets R/RPO (often far better than required), but **is not the MOST cost-effective** because:
  - A **cross-region RDS read replica** runs continuously and can be a significant ongoing cost (instance hours + storage + cross-region replication).
  - You are also running some always-on infrastructure (ALB + baseline compute).
- Pilot light is typically chosen when you need **faster recovery** than backup/restore, not when RTO is as relaxed as 24 hours.

Refs:
- RDS read replicas and cross-region replication behavior (async):  
  https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReadRepl.html

## Enterprise Hands-on Knowledge (What I’d expect in a medical/industrial environment)

1. **Pre-build and test CloudFormation (or CDK/Terraform) regularly**
   - With a 24h RTO, the biggest real-world risk is “the template doesn’t work anymore” (AMI IDs, ECS capacity, IAM permissions, deprecated settings).
   - Run a **scheduled DR game day** (quarterly is common in regulated orgs).

2. **Use ECR Cross-Region Replication Rules**
   - Don’t rely on developers to push to two regions.
   - Ensure the DR region always has the needed image tags/digests.

3. **RDS snapshot + copy: validate restore time**
   - Snapshot restore duration varies with DB size, storage type, and workload.
   - Measure restore times to ensure it fits inside the 24h RTO with margin.

4 **Route 53 failover: choose manual vs automated carefully**
   - For backup/restore DR, failover is often **manual** because the DR region is not “hot.”
   - If you automate failover health checks to a region with no running stack, you could fail over prematurely. Many enterprises prefer:
     - **Manual failover runbook** + approval (especially in medical/industrial compliance contexts).

5. **Security/compliance**
   - Encrypt snapshots and ensure cross-region copies preserve encryption (KMS keys must exist/allowed in DR region).
   - Validate IAM roles, secrets, and parameter store/Secrets Manager replication strategy (often forgotten).
