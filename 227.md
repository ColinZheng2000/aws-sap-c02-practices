A company has deployed its database on an Amazon RDS for MySQL DB instance in the us-east-1 Region. The company needs to make its data available to customers in Europe. The customers in Europe must have access to the same data as customers in the United States (US) and will not tolerate high application latency or stale data. The customers in Europe and the customers in the US need to write to the database. Both groups of customers need to see updates from the other group in real time.

Which solution will meet these requirements?

A. Create an Amazon Aurora MySQL replica of the RDS for MySQL DB instance. Pause application writes to the RDS DB instance. Promote the Aurora Replica to a standalone DB cluster. Reconfigure the application to use the Aurora database and resume writes. Add eu-west-1 as a secondary Region to the DB cluster. Enable write forwarding on the DB cluster. Deploy the application in eu-west-1. Configure the application to use the Aurora MySQL endpoint in eu-west-1.
B. Add a cross-Region replica in eu-west-1 for the RDS for MySQL DB instance. Configure the replica to replicate write queries back to the primary DB instance. Deploy the application in eu-west-1. Configure the application to use the RDS for MySQL endpoint in eu-west-1.
C. Copy the most recent snapshot from the RDS for MySQL DB instance to eu-west-1. Create a new RDS for MySQL DB instance in eu-west-1 from the snapshot. Configure MySQL logical replication from us-east-1 to eu-west-1. Enable write forwarding on the DB cluster. Deploy the application in eu-wes&1. Configure the application to use the RDS for MySQL endpoint in eu-west-1.
D. Convert the RDS for MySQL DB instance to an Amazon Aurora MySQL DB cluster. Add eu-west-1 as a secondary Region to the DB cluster. Enable write forwarding on the DB cluster. Deploy the application in eu-west-1. Configure the application to use the Aurora MySQL endpoint in eu-west-1.


Suggest Answer: A

## 📊 Analysis

### 🔍 Understanding the Problem

A company has a **single-region Amazon RDS for MySQL** deployment in `us-east-1` (US). They need to expand to Europe (`eu-west-1`) with the following strict requirements:

1. **Read AND write access** from both regions (customers in both US and Europe write to the database)
2. **Real-time data visibility** — both groups must see each other's updates immediately
3. **No stale data** — eventual consistency is NOT acceptable
4. **Low latency** — European customers won't tolerate high latency

This is fundamentally a **multi-region, multi-writer** database problem. Standard RDS MySQL cross-region read replicas are **read-only** and can't solve this. The solution requires a database platform that supports **writes from multiple regions** while maintaining strong consistency.

### ✅ Correct Option: A — Migrate to Aurora MySQL Global Database with Write Forwarding

This option outlines a complete migration path from RDS MySQL to **Amazon Aurora MySQL Global Database** with **write forwarding**. Let's break down each step:

**Phase 1: Migrate RDS MySQL → Aurora MySQL**
1. **Create an Aurora MySQL Read Replica** from the existing RDS MySQL DB instance — this is the standard zero-downtime migration path from RDS MySQL to Aurora
2. **Pause application writes** to the RDS MySQL instance (brief maintenance window)
3. **Promote the Aurora Read Replica** to a standalone Aurora DB cluster — the cutover moment
4. **Reconfigure the application** to point to the Aurora cluster endpoint, then **resume writes**

**Phase 2: Extend to Multi-Region**
5. **Add `eu-west-1` as a secondary Region** to the Aurora DB cluster — this creates an **Aurora Global Database**
   - The primary region (`us-east-1`) handles all writes
   - The secondary region (`eu-west-1`) maintains a read-only copy with **sub-second replication latency** (typically < 1 second)
6. **Enable write forwarding** on the DB cluster — this is the critical feature that makes writes from Europe possible:
   - Applications in `eu-west-1` send write queries to the **secondary region's endpoint**
   - Aurora automatically **forwards** those writes to the primary region in `us-east-1`
   - The writes are executed on the primary, then replicated back to the secondary
   - Applications see the results as if they wrote locally
7. **Deploy the application in `eu-west-1`** using the local Aurora endpoint

**Why this architecture meets ALL requirements:**
- ✅ **Read AND write from Europe**: Write forwarding enables European customers to issue writes against the local endpoint
- ✅ **Real-time visibility**: Aurora Global Database uses dedicated infrastructure with sub-second replication
- ✅ **No stale data**: Aurora replication is based on the storage layer, ensuring consistency
- ✅ **Low latency for reads**: European customers read from the local Aurora cluster in `eu-west-1`
- ✅ **Low latency for writes**: Write forwarding routes writes to the primary, but the application code uses the local endpoint

### ❌ Incorrect Options

**B. Add a cross-Region replica in eu-west-1 for the RDS for MySQL DB instance. Configure the replica to replicate write queries back to the primary DB instance**
- **RDS MySQL cross-Region read replicas are READ-ONLY** — they cannot accept writes
- "Configure the replica to replicate write queries back" is **not a native RDS feature**. MySQL logical replication is one-way (primary → replica), not bidirectional
- Setting up two-way replication manually would require complex MySQL configuration (circular replication) which:
  - Risks **write conflicts** (split-brain) — two users updating the same row in different regions
  - Is NOT supported or managed by RDS
  - Violates the "least operational overhead" principle
- This fundamentally misunderstands RDS replication capabilities

**C. Copy the most recent snapshot from the RDS for MySQL DB instance to eu-west-1. Create a new RDS for MySQL DB instance in eu-west-1 from the snapshot. Configure MySQL logical replication from us-east-1 to eu-west-1. Enable write forwarding on the DB cluster**
- **Write forwarding is an Aurora feature**, NOT available for RDS MySQL
- MySQL logical replication from us-east-1 to eu-west-1 is **one-way only** (read-only in Europe)
- Snapshot copy + restore introduces **data staleness** — the snapshot is point-in-time and already outdated by the time you restore it
- Logical replication based on binary log (binlog) has higher latency than Aurora's storage-level replication
- This setup doesn't support writes from Europe at all

**D. Convert the RDS for MySQL DB instance to an Amazon Aurora MySQL DB cluster. Add eu-west-1 as a secondary Region to the DB cluster. Enable write forwarding**
- **You cannot "convert" an RDS MySQL instance to Aurora in-place.** There is no "convert" button or API
- The migration path is: RDS MySQL → create Aurora Read Replica → promote replica (as described in Option A)
- This option **skips the migration step entirely**, which means it's technically impossible as described
- Even if the database were already on Aurora, this option doesn't explain how to get there

### 🏭 Enterprise Hands-On Knowledge

1. **Aurora Global Database Architecture**:
   ```
   us-east-1 (Primary)                    eu-west-1 (Secondary)
   ┌─────────────────────┐               ┌─────────────────────┐
   │  Writer Instance    │               │  Reader Instance(s) │
   │  (Read/Write)       │               │  (Read + Forwarded  │
   │         │           │               │   Writes)           │
   │         ▼           │   Storage     │         │           │
   │  Shared Storage     │◄──────────────│  Shared Storage     │
   │  (6 copies, 3 AZs)  │  Replication  │  (6 copies, 3 AZs) │
   └─────────────────────┘               └─────────────────────┘
   ```
   - Storage-level replication (not binlog-based) → sub-second lag, typically < 1 second
   - Cross-region replication is based on the redo log, making it extremely efficient

2. **Write Forwarding Deep Dive**:
   - Write forwarding uses a **local writer endpoint** in the secondary region
   - The forwarded write goes through: Secondary instance → cross-region network → Primary writer → execution → replication back to secondary
   - **Compatible operations**: INSERT, UPDATE, DELETE, CREATE TABLE, etc.
   - **Limitations**: Not all SQL operations are forwarded (e.g., LOCK TABLES, user-defined functions)
   - **Latency add**: Expect ~100-200ms additional latency for forwarded writes (cross-region round trip)
   - **Monitoring**: Use `WriteForwardingLatency` CloudWatch metric

3. **RDS MySQL → Aurora Migration Checklist**:
   - ✅ Verify MySQL version compatibility (5.6, 5.7, 8.0)
   - ✅ Ensure binary logging is enabled on the source RDS MySQL (required for Aurora read replica)
   - ✅ Check for unsupported features (MyISAM tables → must convert to InnoDB)
   - ✅ Plan the cutover window (the promotion step requires brief write downtime)
   - ✅ Test the migration path in a non-production environment first
   - ✅ Update application connection strings to use Aurora endpoints

4. **Global Database vs. Other Multi-Region Strategies**:
   | Strategy | Writes from Secondary | Replication Lag | Complexity |
   |----------|----------------------|-----------------|------------|
   | Aurora Global DB + Write Forwarding | ✅ Yes (forwarded) | Sub-second | Medium |
   | RDS Cross-Region Read Replica | ❌ Read-only | Seconds-minutes | Low |
   | Multi-Primary (Aurora Multi-Master) | ✅ Yes (limited to 4 writers, same region only) | N/A | High |
   | Application-level multi-write | ⚠️ Possible | Application-dependent | Very High |
   | DynamoDB Global Tables | ✅ Multi-region writes | < 1 second | Low (but different DB engine) |

5. **Write Conflict Resolution**: With write forwarding, there is NO write conflict because ALL writes go to the single primary. Write forwarding is NOT multi-master — it's a convenience layer that routes secondary-region writes to the primary. This preserves strong consistency.

6. **Failover and Disaster Recovery**: With Aurora Global Database:
   - You can **promote a secondary region to primary** in case of primary region failure
   - RTO: Typically < 1 minute to promote
   - RPO: Typically < 1 second (storage-level replication minimizes data loss)
   - This provides both low-latency reads AND disaster recovery in one architecture
