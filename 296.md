A company has a website that serves many visitors. The company deploys a backend service for the website in a primary AWS Region and a disaster recovery (DR) Region.

A single Amazon CloudFront distribution is deployed for the website. The company creates an Amazon Route 53 record set with health checks and a failover routing policy for the primary Region’s backend service. The company configures the Route 53 record set as an origin for the CloudFront distribution. The company configures another record set that points to the backend service's endpoint in the DR Region as a secondary failover record type. The TTL for both record sets is 60 seconds.

Currently, failover takes more than 1 minute. A solutions architect must design a solution that will provide the fastest failover time.

Which solution will achieve this goal?

A. Deploy an additional CloudFront distribution. Create a new Route 53 failover record set with health checks for both CloudFront distributions.
B. Set the TTL to 4 second for the existing Route 53 record sets that are used for the backend service in each Region.
C. Create new record sets for the backend services by using a latency routing policy. Use the record sets as an origin in the CloudFront distribution.
D. Create a CloudFront origin group that includes two origins, one for each backend service Region. Configure origin failover as a cache behavior for the CloudFront distribution.



# AWS SAP-C02 Analysis

## Understanding the Problem

Current architecture:

```text
Users
   |
CloudFront
   |
Route 53 Failover Record
   |---- Primary Region backend
   |---- DR Region backend
```

The company already uses:
- Route 53 failover routing
- Health checks
- CloudFront distribution
- DR backend region

Problem:
- Failover takes more than 1 minute
- TTL is already 60 seconds
- Need the fastest failover

Critical concept:

CloudFront caches DNS resolutions for origins. Even if Route 53 failover changes DNS responses, CloudFront does not immediately re-resolve the origin DNS name.

Therefore:
- Reducing Route 53 TTL alone does not guarantee fast failover
- The failover mechanism should occur inside CloudFront itself

---

# Correct Option

## D. Use a CloudFront origin group with origin failover

### Why this is correct

CloudFront Origin Groups provide:
- Native failover inside CloudFront
- Fast failover without waiting for DNS propagation
- Automatic failover on HTTP status codes or connection failures

Architecture becomes:

```text
Users
   |
CloudFront
   |
Origin Group
   |---- Primary backend
   |---- Secondary DR backend
```

Front directly detects:
- 500 errors
- 502
- 503
- 504
- connection failures
- timeout failures

and immediately routes traffic to the secondary originThis is much faster than DNS failover.

---

## Why this provides the fastest failover

DNS failover depends on:
- DNS caching
- Resolver behavior
- TTL honoring
- CloudFront origin DNS refresh timing

CloudFront Origin Failover:
- Happens at the edge
- Does not rely on client DNS refresh
- Avoids DNS propagation delays- Is nearly immediate

This is the AWS-recommended architecture for high availability behind CloudFront.

---

# Incorrect Options

## A. Add another CloudFront distribution with Route 53 failover

### Why incorrect

This introduces:
- DNS-level failover between CloudFront distributions

Problems:
- Still dependent on Route 53 DNS TTL
- Browser DNS caching remains
- Resolver caching remains
- Additional complexity

This does not improve failover speed significantly.

---

## B. Reduce TTL to 4 seconds

### Why incorrect

This is a classic exam trap.

Lowering Route 53 TTL:
- Does not guarantee rapid failover
- Recursive resolvers may ignore low TTLs
- CloudFront caches origin DNS internally

Most importantly:
CloudFront itself may not immediately honor DNS changes for origin resolution.

Thus failover can still exceed expected timing.

---

## C. Use latency routing policy

### Why incorrect

Latency routing:
- Optimizes latency
- Does NOT provide disaster recovery failover

If the primary endpoint becomes unhealthy:
- Latency routing alone does not guarantee failover behavior

It solves a different problem:
- Best-performance routing
not
- Fast failure recovery

# Key SAP-C02 Concepts Tested

## . DNS failover vs application failover

### DNS failover limitations

DNS failover is affected by:
- TTL
- Resolver caching
- Client caching
- Browser behavior
- CDN internal caching

Failover is never truly immediate.

---

### Application-layer failover

CloudFront origin failover works at:
- HTTP request layer
- Edge location layer

This is much faster and more deterministic.

---

## 2. CloudFront Origin Groups

CloudFront supports:
- Primary origin
- Secondary origin
- Failover criteria

Supported failover triggers:
- HTTP 500
- HTTP 502
- HTTP 503
- HTTP 504
- Connection timeout

This is commonly used in:
- Multi-region web apps
- Active/passive DR
- Enterprise portals
- Healthcare systems
- Industrial dashboards

# Enterprise Hands-On Knowledge

## Real-world enterprise pattern

Common enterprise architecture:

```text
CloudFront
   |
Origin Group
   |----B Primary Region
   |---- ALB Secondary Region
```

Benefits:
- Near-instant failover
- Global edge resiliency
- Simpler operations
- Reduced DNS dependency

---

## Important operational detail

CloudFront origin failover:
- Only applies to GET/HEAD/OPTIONS requests by default
- Non-idempotent requests require careful design

For APIs:
- Multi-region active-active architectures may be better
