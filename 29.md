A company is hosting a three-tier web application in an on-premises environment. Due to a recent surge in traffic that resulted in downtime and a significant financial impact, company management has ordered that the application be moved to AWS. The application is written in .NET and has a dependency on a MySQL database. A solutions architect must design a scalable and highly available solution to meet the demand of 200,000 daily users.  
Which steps should the solutions architect take to design an appropriate solution?  

- A. Use AWS Elastic Beanstalk to create a new application with a web server environment and an Amazon RDS MySQL Multi-AZ DB instance. The environment should launch a Network Load Balancer (NLB) in front of an Amazon EC2 Auto Scaling group in multiple Availability Zones. Use an Amazon Route 53 alias record to route traffic from the company's domain to the NLB.
- B. Use AWS CloudFormation to launch a stack containing an Application Load Balancer (ALB) in front of an Amazon EC2 Auto Scaling group spanning three Availability Zones. The stack should launch a Multi-AZ deployment of an Amazon Aurora MySQL DB cluster with a Retain deletion policy. Use an Amazon Route 53 alias record to route traffic from the company's domain to the ALB.
- C. Use AWS Elastic Beanstalk to create an automatically scaling web server environment that spans two separate Regions with an Application Load Balancer (ALB) in each Region. Create a Multi-AZ deployment of an Amazon Aurora MySQL DB cluster with a cross-Region read replica. Use Amazon Route 53 with a geoproximity routing policy to route traffic between the two Regions.
- D. Use AWS CloudFormation to launch a stack containing an Application Load Balancer (ALB) in front of an Amazon ECS cluster of Spot instances spanning three Availability Zones. The stack should launch an Amazon RDS MySQL DB instance with a Snapshot deletion policy. Use an Amazon Route 53 alias record to route traffic from the company's domain to the ALB.

## Understanding the Problem

The company needs to migrate a **three-tier web application (.NET and MySQL)** from an on-premises environment to AWS. The key requirements are **scalability** to handle a surge in traffic (up to 200,000 daily users) and **high availability** to prevent future downtime and financial impact. The solutions architect must design an appropriate architecture.

**Key Requirements:**

1. **High Availability:** The solution must tolerate failures (e.g., in an Availability Zone).
    
2. **Scalability:** The architecture must automatically scale to meet the demand of 200,000 daily users.
    
3. **Application Stack:** .NET application and MySQL database.
    
4. **Database:** Must be highly available and suitable for a web application backend.

## Correct Option(s) Explanation

**Option B is the most appropriate and robust solution.**

- **Scalability and High Availability (Web Tier):** An **Application Load Balancer (ALB)** distributing traffic across an **Amazon EC2 Auto Scaling group** spanning **three Availability Zones** provides both horizontal scaling and high availability. ALB is the standard choice for HTTP/HTTPS web applications.
    
- **Database High Availability and Performance:** **Amazon Aurora MySQL DB cluster** is an excellent choice for a scalable, high-performance, and highly available MySQL-compatible database. A **Multi-AZ deployment** for Aurora is the standard architecture for high availability.
    
- **Infrastructure as Code:** **AWS CloudFormation** is a professional way to provision and manage the entire architecture, ensuring repeatability and consistency.
    
- **Data Durability and Disaster Recovery:** The **Retain deletion policy** on the database cluster is a critical best practice to prevent accidental data loss if the CloudFormation stack is ever deleted.
    
- **Traffic Routing:** An **Amazon Route 53 alias record** is the correct method to map the company's domain to the ALB, as it provides a robust, highly available, and free way to connect the domain to the Load Balancer endpoint. (Refer to the _AWS documentation on Route 53 Alias Records_).

## Incorrect Options Explanation

- **A. Use AWS Elastic Beanstalk... The environment should launch a Network Load Balancer (NLB)...**
    
    - **Issue:** While Elastic Beanstalk is a good choice for deployment, a **Network Load Balancer (NLB)** is generally overkill and less suitable than an ALB for a standard HTTP/HTTPS three-tier web application, which can benefit from the advanced routing features of an ALB. ALB is the more common and appropriate choice for this use case.
        
- **C. Use AWS Elastic Beanstalk to create an automatically scaling web server environment that spans two separate Regions... Use Amazon Route 53 with a geoproximity routing policy to route traffic between the two Regions.**
    
    - **Issue:** The problem statement only asks to design a scalable and highly available solution, which can be accomplished within a **single Region using multiple Availability Zones (Multi-AZ)**, which is a less complex and lower-cost solution. Spanning **two separate Regions** with **cross-Region read replicas** and **geoproximity routing** is a design for **Disaster Recovery (DR)**, not just high availability. This is over-engineering and adds unnecessary complexity and cost based on the stated requirements. High availability is achieved with Multi-AZ within a single region.
        
- **D. Use AWS CloudFormation... The stack should launch an Application Load Balancer (ALB) in front of an Amazon ECS cluster of Spot instances spanning three Availability Zones. The stack should launch an Amazon RDS MySQL DB instance with a Snapshot deletion policy...**
    
    - **Issue 1: Spot Instances in ECS:** While Spot Instances can be used for cost savings, using them for the **entire primary web tier** of a critical application with a demand of 200,000 daily users introduces a risk of service interruption, as Spot Instances can be terminated by AWS with a two-minute warning. On-Demand or a mix with an appropriate capacity strategy would be safer.
        
    - **Issue 2: RDS MySQL DB instance (single):** The option mentions a single **Amazon RDS MySQL DB instance**, which is typically a single-AZ deployment unless explicitly configured as Multi-AZ (which this option does not state, unlike the others). A single-AZ deployment is a **single point of failure** and fails the high-availability requirement.
        
    - **Issue 3: Snapshot deletion policy:** A **Snapshot deletion policy** on the DB instance in CloudFormation means that if the stack is deleted, a final snapshot is taken, but the DB instance itself is deleted. While better than `Delete`, the **Retain** policy (as in Option B) is often the safest choice for production databases to ensure the physical resource is never automatically deleted by an infrastructure change, aligning with best practices for critical data.
