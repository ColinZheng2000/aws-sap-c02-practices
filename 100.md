A delivery company needs to migrate its third-party route planning application to AWS. The third party supplies a supported Docker image from a public registry. The image can run in as many containers as required to generate the route map.  
  
The company has divided the delivery area into sections with supply hubs so that delivery drivers travel the shortest distance possible from the hubs to the customers. To reduce the time necessary to generate route maps, each section uses its own set of Docker containers with a custom configuration that processes orders only in the section's area.  
  
The company needs the ability to allocate resources cost-effectively based on the number of running containers.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Create an Amazon Elastic Kubernetes Service (Amazon EKS) cluster on Amazon EC2. Use the Amazon EKS CLI to launch the planning application in pods by using the --tags option to assign a custom tag to the pod.
- B. Create an Amazon Elastic Kubernetes Service (Amazon EKS) cluster on AWS Fargate. Use the Amazon EKS CLI to launch the planning application. Use the AWS CLI tag-resource API call to assign a custom tag to the pod.
- C. Create an Amazon Elastic Container Service (Amazon ECS) cluster on Amazon EC2. Use the AWS CLI with run-tasks set to true to launch the planning application by using the --tags option to assign a custom tag to the task.
- D. Create an Amazon Elastic Container Service (Amazon ECS) cluster on AWS Fargate. Use the AWS CLI run-task command and set enableECSManagedTags to true to launch the planning application. Use the --tags option to assign a custom tag to the task.

## Understanding the Problem
The core requirements of the delivery company’s migration project are as follows:
1. Migrate a third-party Docker-based route planning application to AWS, supporting scalable container deployment for different delivery sections.
2. Each delivery section uses a custom-configured container set to process only its regional orders, requiring the ability to assign custom tags to containerized workloads.
3. Allocate resources **cost-effectively** based on the number of running containers (i.e., pay only for active workloads, avoid idle resource costs).
4. Minimize **operational overhead** (reduce manual management of underlying infrastructure and orchestration tools).

The key trade-offs to evaluate are: container orchestration service (ECS vs. EKS), compute launch type (Fargate vs. EC2), tag management for workloads, and operational complexity/cost efficiency.

## Services Explanation
### 1. Amazon Elastic Container Service (ECS)
Per the [AWS ECS Official Documentation](https://docs.aws.amazon.com/ecs/latest/developerguide/what-is-ecs.html), ECS is a fully managed container orchestration service designed for **simplicity and low operational overhead**. It natively integrates with AWS services and supports two launch types:
- **EC2 launch type**: Requires users to provision, manage, patch, and scale EC2 instances (high operational overhead, potential idle resource costs).
- **Fargate launch type**: A serverless compute engine for ECS that eliminates the need to manage EC2 instances. Users pay only for vCPU and memory resources used by running containers (cost-effective, zero infrastructure management).

ECS supports [tagging for tasks, services, and clusters](https://docs.aws.amazon.com/ecs/latest/developerguide/ecs-tagging.html) via the `--tags` CLI option for custom tags, and the `enableECSManagedTags` parameter to auto-generate AWS-managed tags (e.g., cluster name, task definition ARN).

### 2. Amazon Elastic Kubernetes Service (EKS)
Per the [AWS EKS Official Documentation](https://docs.aws.amazon.com/eks/latest/userguide/what-is-eks.html), EKS is a fully managed Kubernetes service for users who need the Kubernetes ecosystem (e.g., multi-cloud portability, complex orchestration). It also supports EC2 and Fargate launch types, but:
- EKS introduces **Kubernetes-specific operational overhead** (e.g., managing kubectl configurations, Kubernetes API servers, node groups, and K8s-native resource scheduling).
- Kubernetes Pods are **K8s internal resources**, not top-level AWS resources. AWS’s `tag-resource` API (for AWS resource tagging) does **not** support Pods—Pod tags are limited to K8s internal labeling (not AWS resource tags).

### 3. AWS Fargate
Per the [AWS Fargate Official Documentation](https://docs.aws.amazon.com/fargate/latest/userguide/what-is-fargate.html), Fargate is a serverless compute engine for ECS/EKS that abstracts all underlying EC2 instance management (provisioning, patching, scaling). It bills by the second for active container resources, aligning with the “cost-effective resource allocation” requirement. In contrast, EC2 launch types require paying for instances even if containers are idle (poor cost efficiency).

## Correct Option (D) Explanation
Option D: *Create an Amazon ECS cluster on AWS Fargate. Use the AWS CLI run-task command and set enableECSManagedTags to true to launch the application. Use the --tags option to assign a custom tag to the task.*

This option meets **all requirements** with the least operational overhead for the following reasons:
1. **Minimal operational overhead**: ECS on Fargate eliminates EC2 instance management (Fargate’s serverless model) and ECS’s simpler orchestration model (no Kubernetes complexity) compared to EKS.
2. **Cost-effective resource allocation**: Fargate’s pay-as-you-go pricing (per second for vCPU/memory) ensures resources are allocated only based on running containers—no idle EC2 instance costs.
3. **Valid tag management**: Per [ECS Run-Task CLI Documentation](https://docs.aws.amazon.com/cli/latest/reference/ecs/run-task.html), the `--tags` option directly assigns custom tags to ECS tasks (AWS resources), and `enableECSManagedTags` auto-adds AWS-managed tags (e.g., cluster ID) for additional resource tracking.
4. **Scalability for regional container sets**: ECS supports launching multiple task instances with custom configurations for each delivery section, aligning with the regional processing requirement.

## Incorrect Options Explanation
### Option A
*Create an Amazon EKS cluster on Amazon EC2. Use the Amazon EKS CLI to launch the application in pods by using the --tags option to assign a custom tag to the pod.*

**Why incorrect**:
1. **High operational overhead**: EKS on EC2 requires managing both Kubernetes clusters (kubectl, API servers) and EC2 instances (patching, scaling, idle resources).
2. **Invalid tag management**: The `--tags` option for EKS Pods sets **K8s internal labels**, not AWS resource tags (unusable for AWS-level resource tracking).
3. **Poor cost efficiency**: EC2 launch type incurs idle instance costs, violating the “cost-effective resource allocation” requirement.

### Option B
*Create an Amazon EKS cluster on AWS Fargate. Use the Amazon EKS CLI to launch the application. Use the AWS CLI tag-resource API call to assign a custom tag to the pod.*

**Why incorrect**:
1. **Kubernetes operational overhead**: Even with Fargate, EKS requires managing Kubernetes-specific configurations (e.g., Pod scheduling, K8s RBAC) — higher overhead than ECS.
2. **Invalid tag management**: Per [AWS Tagging API Documentation](https://docs.aws.amazon.com/cli/latest/reference/resourcegroupstaggingapi/tag-resource.html), the `tag-resource` API only supports top-level AWS resources (e.g., EKS clusters, EC2 instances). Pods are K8s internal resources and cannot be tagged via this API.
3. **Unnecessary complexity**: The company has no stated need for Kubernetes, making EKS an over-engineered solution.

### Option C
*Create an Amazon ECS cluster on Amazon EC2. Use the AWS CLI with run-tasks set to true to launch the application by using the --tags option to assign a custom tag to the task.*

**Why incorrect**:
1. **High operational overhead**: ECS on EC2 requires manual management of EC2 instances (provisioning, patching, scaling) — violating the “least operational overhead” requirement.
2. **Poor cost efficiency**: EC2 launch type incurs hourly instance costs regardless of whether containers are running, leading to idle resource waste and failing the “cost-effective resource allocation” requirement.
3. While the `--tags` option for ECS tasks is valid, the EC2 launch type negates the core cost and operational goals.
