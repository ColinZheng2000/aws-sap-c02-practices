A company that provides image storage services wants to deploy a customer-facing solution to AWS. Millions of individual customers will use the solution. The solution will receive batches of large image files, resize the files, and store the files in an Amazon S3 bucket for up to 6 months.

The solution must handle significant variance in demand. The solution must also be reliable at enterprise scale and have the ability to rerun processing jobs in the event of failure.

Which solution will meet these requirements MOST cost-effectively?

A. Use AWS Step Functions to process the S3 event that occurs when a user stores an image. Run an AWS Lambda function that resizes the image in place and replaces the original file in the S3 bucket. Create an S3 Lifecycle expiration policy to expire all stored images after 6 months.
B. Use Amazon EventBridge to process the S3 event that occurs when a user uploads an image. Run an AWS Lambda function that resizes the image in place and replaces the original file in the S3 bucket. Create an S3 Lifecycle expiration policy to expire all stored images after 6 months.
C. Use S3 Event Notifications to invoke an AWS Lambda function when a user stores an image. Use the Lambda function to resize the image in place and to store the original file in the S3 bucket. Create an S3 Lifecycle policy to move all stored images to S3 Standard-Infrequent Access (S3 Standard-IA) after 6 months.
D. Use Amazon Simple Queue Service (Amazon SQS) to process the S3 event that occurs when a user stores an image. Run an AWS Lambda function that resizes the image and stores the resized file in an S3 bucket that uses S3 Standard-Infrequent Access (S3 Standard-IA). Create an S3 Lifecycle policy to move all stored images to S3 Glacier Deep Archive after 6 months.



# AWS SAP-C02 Analysis

## Understanding the Problem

Requirements:

- Customer-facing image storage platform
- Millions of users
- Large image uploads
- Resize images
- Store images for up to 6 months
- Handle highly variable demand
- Enterprise-scale reliability
- Ability to rerun failed processing jobs
- Most cost-effective solution

Key architectural concerns:

1. Massive scalability
2. Asynchronous processing
3. Failure durability and retry capability
4. Cost optimization for storage lifecycle
5. Event-driven serverless architecture

The critical phrase is:

> "ability to rerun processing jobs in the event of failure"

This strongly suggests the need for durable message buffering.

---

# Correct Option

## D. Use SQS + Lambda + S3 lifecycle to Deep Archive

### Why this is correct

Architecture:

```text
User Upload
    |
S3 Event
    |
Amazon SQS
    |
Lambda Image Processing
    |
Store resized image in S3
    |
Lifecycle -> Glacier Deep Archive after 6 months
```

This architecture provides:

- Massive scalability
- Decoupling
- Retry capability
- Durable message retention
- Failure recovery
- Cost optimization

---

## Key architectural advantage: SQS durability

SQS provides:
- Durable queueing
- Automatic retry
- Dead-letter queue (DLQ) support
- Decoupled processing
- Replay capability

This directly satisfies:

>ability to rerun processing jobs in the event of failure"

Compared to direct Lambda invocation:
- SQS is significantly more resilient

---

## Why Lambda is appropriate

Lambda works well because:
- Demand is highly variable
- Millions of customers
- Event-driven workload
- No idle infrastructure cost

Lambda auto-scales automatically.

---

## Why Glacier Deep Archive is cost-effective

Requirement:
- Store files for up to 6 months

Using lifecycle transitions:
- Reduces long-term storage cost significantly

S3 Glacier Deep Archive:
- Lowest S3 storage cost tier
- Excellent for infrequently accessed images

---

# Incorrect Options

## A. Step Functions + Lambda

### Why incorrect

Step Functions are powerful but unnecessary here.

Problems:
- More expensive
- Adds orchestration overhead
- No mention of complex workflows
- Does not inherently provide durable buffering like SQS

For simple image processing pipelines:
- SQS + Lambda is simpler and cheaper

---

## Additional issue

"Resize image in place and replace original file"

This is risky because:
- Original upload may be lost
- Recovery becomes difficult
- Reprocessing capability weakens

Enterprise systems usually:
- Preserve originals
- Generate derivative versions separately

---

## B. EventBridge + Lambda

### Why incorrect

EventBridge:
- Good for event routing
- Not ideal for durable work queues

Problem:
- EventBridge is not a processing buffer like SQS
- Less suitable for replay-heavy workloads
- Not optimized for large-scale worker processing pipelines

SQS is the AWS standard pattern for:
- Asynchronous durable workloads

---

## C. Direct S3 Event Notification to Lambda

### Why incorrect

This architecture lacks durable buffering.

Problem areas:
- Lambda invocation failures
- Burst traffic handling
- Retry limitations
- No easy replay capability

Direct S3 -> Lambda coupling is weaker for enterprise reliability.

---

## Additional issue

Lifecycle policy:
> Move to S3 Standard-IA after 6 months

But requirement says:
- Store for up to 6 months

This lifecycle policy would transition after the retention window instead of optimizing within it.

Also:
- Glacier Deep Archive is more cost-effective for archival retention.

---

# Key SAP-C02 Concepts Tested

## 1. Event-driven decoupled architectures

Best practice pattern:

```text
Producer -> Queue -> Consumer
```

Advantages:
- Scalability
- Reliability
- Fault isolation
- Replay capability

---

## 2. SQS + Lambda integration

SQS with Lambda provides:
- Automatic polling
- Batch processing
- Retry handling
- DLQ support
- Elastic concurrency

Very common SAP-C02 pattern.

---

## 3. Storage lifecycle optimization

S3 lifecycle policies are heavily tested.

Typical progression:

```text
S3 Standard
   -> Standard-IA
   -> Glacier Flexible Retrieval
   -> Glacier Deep Archive
```

Choose based on:
- Access frequency
- Retention duration
- Retrieval requirements

---

## 4. Serverless for variable workloads

Highly variable traffic:
- Favors Lambda
- Avoids EC2 overprovisioning
- Reduces operational overhead

# Enterprise Hands-On Knowledge

## Real-world image processing architecture

Very common enterprise design:

```text
S3 Upload Bucket
     |
S3 Event Notification
     |
SQS Queue
     |
Lambda / ECS Workers
     |
Processed Images Bucket
```

Benefits:
- Burst protection
- Retry safety
- Replay capability
- Independent scaling

---

## Production reliability best practices

### Use DLQs

Configure:
- SQS dead-letter queues

This enables:
- Investigation of failed images
- Replay after fixes

---

## Preserve originals

Best practice:
- Store original uploads separately
- Generate resized derivatives

Useful for:
- Future reprocessing
- New image formats
- AI/ML processing later

---

## Large image processing consideration

Large images may exceed Lambda limits:
- Memory
- Timeout
- /tmp storage

In real enterprise systems:
- ECS Fargate is often used for heavy image/video processing

But for this exam question:
- Lambda remains the intended answer.

---

## Cost optimization strategy

Recommended lifecycle example:

```text
Day 0: S3 Standard
Day 30: S3 Standard-IA
Day 90: Glacier Flexible Retrieval
Day 180: Expiration
```

For ultra-low-cost retention:
- Glacier Deep Archive is ideal
