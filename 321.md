A company is preparing to deploy an Amazon Elastic Kubernetes Service (Amazon EKS) cluster for a workload. The company expects the cluster to support an unpredictable number of stateless pods. Many of the pods will be created during a short time period as the workload automatically scales the number of replicas that the workload uses.

Which solution will MAXIMIZE node resilience?

A. Use a separate launch template to deploy the EKS control plane into a second cluster that is separate from the workload node groups.
B. Update the workload node groups. Use a smaller number of node groups and larger instances in the node groups.
C. Configure the Kubernetes Cluster Autoscaler to ensure that the compute capacity of the workload node groups stays underprovisioned.
D. Configure the workload to use topology spread constraints that are based on Availability Zone.


# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

This scenario focuses on improving resilience for worker nodes in an Amazon EKS environment that runs highly dynamic stateless workloads. The workload can rapidly scale and create many pods in a short period of time, which introduces scheduling pressure and potential infrastructure imbalance.

The key requirement is maximizing node resilience, not minimizing cost or simplifying operations. In Kubernetes and EKS, resilience is strongly tied to:
- Distribution of pods across failure domains
- Avoidance of single-node or single-AZ concentration
- Maintaining service availability during node or AZ failures

Because the workload is stateless, pod redistribution across nodes and Availability Zones is the preferred architectural strategy.

---

### 2. Options Analysis

#### ✅ Option D (CORRECT): Use topology spread constraints across Availability Zones

Topology spread constraints instruct Kubernetes to evenly distribute pods across defined topology domains such as:
- Availability Zones
- Nodes
- Hostnames

This directly improves resilience because:
- Failure of one AZ affects only a portion of replicas
- Failure of one node does not concentrate workload loss
- Rapid autoscaling events avoid creating “hotspot” nodes

For stateless workloads, AWS and Kubernetes best practices strongly recommend spreading replicas across multiple AZs to maintain high availability.

Enterprise relevance:
- Large-scale production EKS environments commonly use topology spread constraints together with:
  - Managed node groups
  - Multi-AZ subnets
  - Cluster Autoscaler or Karpenter
- This pattern is especially important in healthcare, pharmaceutical, and industrial systems where uptime and fault isolation are critical.

Example Kubernetes configuration:

```yaml
topologySpreadConstraints:
- maxSkew: 1
  topologyKey: topology.kubernetes.io/zone
  whenUnsatisfiable: DoNotSchedule
  labelSelector:
    matchLabels:
      app: api-service
```

This ensures pods are evenly distributed across AZs.

Official references:
- Kubernetes topology spread constraints:
  https://kubernetes.io/docs/concepts/scheduling-eviction/topology-spread-constraints/
- AWS EKS best practices:
  https://aws.github.io/aws-eks-best-practices/

#### ❌ Option A (INCORRECT): Separate launch template for EKS control plane

This option misunderstands EKS architecture.

In Amazon EKS:
- The control plane is fully managed by AWS
- Customers do not deploy the control plane using launch templates
- Node groups are separate from the managed control plane already

Why this fails:
- Does not improve worker node resilience
- Does not affect pod scheduling distribution
- Architecturally invalid for managed EKS control planes

This is common exam distractor targeting confusion between:
- Self-managed Kubernetes
- Managed EKS control plane architecture

Official reference:
- Amazon EKS architecture:
  https://docs.aws.amazon.com/eks/latest/userguide/what-is-eks.html

---

#### ❌ Option B (INCORRECT): Fewer node groups with larger instances

Using larger EC2 instances can improve operational simplicity or packing efficiency, but it reduces resilience.

Why this is problematic:
- Larger blast radius if a node fails
- More pods affected per node outage
- Increased scheduling disruption during scale events

In Kubernetes resilience design:
- More smaller nodes generally improve fault tolerance
- Failure domains should be minimized

Example:
- Losing one 64-vCPU node may impact hundreds of pods
- Losing one smaller node affects far fewer workloads

This option optimizes consolidation, not resilience.

---

#### ❌ Option C (INCORRECT): Keep node groups underprovisioned

Underprovisioning compute capacity directly harms resilience and scaling responsiveness.

Why this fails:
- Pods may remain Pending during sudden scaling events
- Increased scheduling contention
- Greater risk during node failures or AZ impairments

Cluster Autoscaler best practice is:
- Ensure enough capacity exists
- Scale proactively when workloads grow

Underprovisioned clusters are especially dangerous for burst workloads because:
- EC2 launch latency becomes a bottleneck
- Kubernetes scheduling delays impact application availability

This option conflicts with high-availability architecture principles.

Official reference:
- Cluster Autoscaler best practices:
  https://docs.aws.amazon.com/eks/latest/best-practices/cas.html

### 3. Hands-On Knowledge

In real enterprise EKS deployments, resilience usually combines several mechanisms together:

1. Multi-AZ Node Groups
- Always place worker nodes in at least 2–3 AZs
- Use managed node groups across private subnets
- Verify subnet tagging for Kubernetes load balancers

2. Topology Spread Constraints + Pod Anti-Affinity
- Topology spread distributes evenly
- Pod anti-affinity prevents same-app concentration

Example:

```yaml
podAntiAffinity:
  preferredDuringSchedulingIgnoredDuringExecution:
  - weight: 100
    podAffinityTerm:
      topologyKey: kubernetes.io/hostname
```

3. Cluster Autoscaler Best Practices
- Separate node groups by workload type
- Avoid oversized instances
- Maintain headroom for rapid burst scaling

4. Production Monitoring
Important metrics:
- Pending pods
- Node pressure
- Unschedulable pods
- AZ imbalance
- EC2 launch latency

Common enterprise observability stack:
- Amazon Managed Prometheus
- Amazon Managed Grafana
- CloudWatch Container Insights

5. Real-World Enterprise Gotcha
A frequent production issue occurs when:
- HPA scales rapidly
- Cluster Autoscaler reacts slowly
- Pods become concentrated in one AZ

Result:
- AZ outage causes massive simultaneous pod loss

Topology spread constraints significantly reduce this risk.

**面试官可能追问 (Possible Follow-up Questions)**：

1. " spread constraints 和 pod anti-affinity 有什么区别？"
   → 回答要点 (Key Answer Points)： spread constraints强调均匀分布，可控制AZ或节点之间的平衡；pod anti-affinity主要用于避免特定pod放在同一节点。生产环境通常两者组合使用。

2. "为什么 EKS 中通常建议使用更多小节点而不是少量大节点？"
   → 回答要点 (Key Answer Points)：小节点降低单点故障影响范围，提高调度灵活性，并减少节点失效时受影响pod数量。节点虽然提高资源利用率，但会扩大blast radius。
