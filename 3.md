A company uses AWS Organizations with a single OU named Production to manage multiple accounts. All accounts are members of the Production OU.  
Administrators use deny list SCPs in the root of the organization to manage access to restricted services.  
The company recently acquired a new business unit and invited the new unit's existing AWS account to the organization. Once onboarded, the administrators of the new business unit discovered that they are not able to update existing AWS Config rules to meet the company's policies.  
Which option will allow administrators to make changes and continue to enforce the current policies without introducing additional long-term maintenance?  

- A. Remove the organization's root SCPs that limit access to AWS Config. Create AWS Service Catalog products for the company's standard AWS Config rules and deploy them throughout the organization, including the new account.
- B. Create a temporary OU named Onboarding for the new account. Apply an SCP to the Onboarding OU to allow AWS Config actions. Move the new account to the Production OU when adjustments to AWS Config are complete.
- C. Convert the organization's root SCPs from deny list SCPs to allow list SCPs to allow the required services only. Temporally apply an SCP to the organization's root that allows AWS Config actions for principals only in the new account.
- D. Create a temporary OU named Onboarding for the new account. Apply an SCP to the Onboarding OU to allow AWS Config actions. Move the organization's root SCP to the Production OU. Move the new account to the Production OU when adjustments to AWS Config are complete.

# AWS Certified Solutions Architect - Professional (SAP-C02) Exam Question Analysis

## 🎯 Understanding the Problem

The company uses **AWS Organizations** to manage multiple accounts, with all production accounts residing in a single **Production OU**. They utilize **deny list Service Control Policies (SCPs)** attached to the **Root** of the organization to restrict access to certain services across **all** accounts. A newly acquired business unit's AWS account, after being onboarded, is unable to **update existing AWS Config rules**. The goal is to temporarily allow the new account to make necessary **AWS Config** adjustments to align with company policy, while ensuring:
1.  The existing organization-wide denial policies are **maintained**.
2.  The solution introduces **minimal additional long-term maintenance**.

The core conflict is the existing root-level deny SCP is blocking necessary configuration in the new account. A temporary deviation is needed, followed by re-enforcement of the standard policies.

***

## ⚙️ Services Explanation

| Service                          | Description                                                                                                                                                                                                                                     | Relevance to Question                                                                                                                                          |
| :------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **AWS Organizations**            | An account management service that lets you consolidate multiple AWS accounts into an organization that you create and centrally manage.                                                                                                        | The foundational service used by the company for account management, OUs, and SCPs.                                                                            |
| **Organizational Unit (OU)**     | A container within AWS Organizations that helps you group accounts and apply policies to them. OUs form a hierarchy that resembles an inverted tree.                                                                                            | OUs are the mechanism used to selectively apply different policies (SCPs) to groups of accounts.                                                               |
| **Service Control Policy (SCP)** | A type of policy that you can use to manage permissions in your organization. SCPs offer central control over the maximum available permissions for all accounts in your organization. **They are used to allow or deny services and actions.** | The core security mechanism causing the issue (deny list SCPs at the Root) and the tool needed for the temporary fix (a more permissive SCP at a specific OU). |
| **AWS Config**                   | A service that enables you to assess, audit, and evaluate the configurations of your AWS resources. **Rules** are used to define the desired configuration.                                                                                     | The specific service the administrators need to modify, which is currently being blocked by the root SCP.                                                      |
| **Deny List SCP**                | An SCP that explicitly **denies** access to specified services or actions. It's often the simpler way to enforce "blacklisting" in a large organization.                                                                                        | The existing policy type causing the block. **Explicit denies always override explicit allows.**                                                               |

***

## ✅ Correct Option(s) Explanation

The correct option is **D**.

**D. Create a temporary OU named Onboarding for the new account. Apply an SCP to the Onboarding OU to allow AWS Config actions. Move the organization's root SCP to the Production OU. Move the new account to the Production OU when adjustments to AWS Config are complete.**

This option is the most robust and introduces the least long-term maintenance by leveraging the **SCP inheritance model** and the power of **explicit denies**.

1.  **Create Temporary OU and Apply Allow SCP**: Creating an `Onboarding` OU and applying a *more permissive* SCP (that *allows* AWS Config actions) to it is the first step for isolation.
2.  **Move Root SCP to Production OU**: This is the **critical step**. The original deny list SCP was at the **Root**, meaning it affected *every* account, including the new `Onboarding` OU. By moving the original deny list SCP from the **Root** to the **Production OU**:
    * The deny policy *continues* to enforce company standards on all **existing Production accounts**.
    * The **Root** is now free of the specific deny policy, allowing the *less restrictive* `Onboarding` OU SCP to take effect for the new account. **(Reference: AWS SCP evaluation logic - a principal can only perform an action if it is explicitly allowed by all applicable SCPs in the inheritance path, and not explicitly denied by any applicable SCPs.)**
3.  **Move New Account to Production OU**: Once the configuration is complete, moving the account back to the `Production` OU ensures the account immediately falls under the scope of the **deny list SCP** (now applied to the `Production` OU), thereby re-enforcing the standard company policies.

This solution maintains the original policy set and only involves temporary OU and account movements, requiring **no modification** of the original policy document and minimal long-term cleanup (just deleting the temporary `Onboarding` OU and its SCP).

***

## ❌ Incorrect Options Explanation

| Option | Reason for Incorrectness |
| :--- | :--- |
| **A. Remove the organization's root SCPs that limit access to AWS Config. Create AWS Service Catalog products for the company's standard AWS Config rules and deploy them throughout the organization, including the new account.** | **Removing the root SCPs is a security risk** and immediately compromises the existing policy enforcement for *all* accounts. While **Service Catalog** is a good mechanism for deployment, it doesn't solve the immediate issue of the **blocked administrative access** to Config rules, nor does it address the security lapse of removing the root SCP. This introduces high risk and is not the minimal-maintenance solution. |
| **B. Create a temporary OU named Onboarding for the new account. Apply an SCP to the Onboarding OU to allow AWS Config actions. Move the new account to the Production OU when adjustments to AWS Config are complete.** | This solution fails because the **deny list SCP at the Root** would still be in effect. Since **explicit denies always override allows** in SCP evaluation, the *allow* SCP on the `Onboarding` OU would be overridden by the *deny* SCP at the `Root`, and the administrator would still be blocked. **(Reference: AWS SCP evaluation logic - an explicit deny anywhere in the hierarchy is final.)** |
| **C. Convert the organization's root SCPs from deny list SCPs to allow list SCPs to allow the required services only. Temporally apply an SCP to the organization's root that allows AWS Config actions for principals only in the new account.** | **Converting to allow list SCPs is a major, high-impact change** to the organization's security posture and increases long-term maintenance complexity (as every new service must be explicitly allowed). Furthermore, a temporary *allow* SCP at the Root would likely still be overridden by the general *deny* policy (if the conversion isn't fully comprehensive) and could be an over-permissive change that affects *all* accounts, introducing significant risk. It requires modifying a foundational security policy, which violates the "minimal additional long-term maintenance" constraint. |
