A company is running a web application in the AWS Cloud. The application consists of dynamic content that is created on a set of Amazon EC2 instances. The  
EC2 instances run in an Auto Scaling group that is configured as a target group for an Application Load Balancer (ALB).  
The company is using an Amazon CloudFront distribution to distribute the application globally. The CloudFront distribution uses the ALB as an origin. The company uses Amazon Route 53 for DNS and has created an A record of www.example.com for the CloudFront distribution.  
A solutions architect must configure the application so that itis highly available and fault tolerant.  
Which solution meets these requirements?  

- A. Provision a full, secondary application deployment in a different AWS Region. Update the Route 53 A record to be a failover record. Add both of the CloudFront distributions as values. Create Route 53 health checks.
- B. Provision an ALB, an Auto Scaling group, and EC2 instances in a different AWS Region. Update the CloudFront distribution, and create a second origin for the new ALB. Create an origin group for the two origins. Configure one origin as primary and one origin as secondary.
- C. Provision an Auto Scaling group and EC2 instances in a different AWS Region. Create a second target for the new Auto Scaling group in the ALB. Set up the failover routing algorithm on the ALB.
- D. Provision a full, secondary application deployment in a different AWS Region. Create a second CloudFront distribution, and add the new application setup as an origin. Create an AWS Global Accelerator accelerator. Add both of the CloudFront distributions as endpoints.

## Understanding the Problem

The current setup has a single application stack (ALB -> ASG -> EC2) deployed in _one_ AWS Region. This stack serves as the _origin_ for a global Amazon CloudFront distribution. The setup is highly available _within_ that single Region (thanks to the ALB and ASG spanning multiple Availability Zones), but it is _not_ fault-tolerant against a complete **Regional failure**.

If the entire AWS Region hosting the ALB and EC2 instances goes down, the CloudFront origin will be unreachable. CloudFront will serve cached content (if any) until its TTL expires, after which users will receive errors (e.g., 504 Gateway Timeout).

The goal is to implement a multi-Region, fault-tolerant solution so that if the primary application stack fails, traffic is automatically served by a secondary, healthy stack in another Region.

---

## Services Explanation

- **Amazon CloudFront:** A global Content Delivery Network (CDN) service. It caches content at "edge locations" close to users to reduce latency. It fetches content from a configured "origin" (like an ALB, S3 bucket, or EC2 instance).
    
- **Application Load Balancer (ALB):** A Layer 7 (HTTP/HTTPS) load balancer that routes traffic to targets (like EC2 instances) within a _single Region_. It operates across multiple Availability Zones for high availability.
    
- **Amazon EC2 Auto Scaling Group (ASG):** Automatically adjusts the number of EC2 instances in a group based on demand or a schedule. It also maintains a healthy fleet by replacing terminated or unhealthy instances, providing fault tolerance within a Region.
    
- **Amazon Route 53:** A highly available and scalable global Domain Name System (DNS) web service. It can be used to route users to infrastructure, including using health checks and "failover" routing policies to direct traffic away from unhealthy endpoints.
    
- **CloudFront Origin Group:** A feature within CloudFront designed for high availability. An origin group contains two origins: a **primary** and a **secondary**. CloudFront will always send requests to the primary origin. However, if the primary origin fails (e.g., returns specific HTTP 5xx error codes), CloudFront will _automatically_ fail over and send subsequent requests to the secondary origin.
    
- **AWS Global Accelerator:** A networking service that improves the availability and performance of applications for global users. It provides static anycast IP addresses that act as a fixed entry point and routes traffic to the nearest _healthy regional endpoint_ (e.g., ALB, NLB, EC2).

## Correct Option(s) Explanation

### B. Provision an ALB, an Auto Scaling group, and EC2 instances in a different AWS Region. Update the CloudFront distribution, and create a second origin for the new ALB. Create an origin group for the two origins. Configure one origin as primary and one origin as secondary.

This is the **correct** solution because it uses CloudFront's native high-availability feature, **Origin Groups**.

1. **New Regional Stack:** A second, independent application stack (ALB, ASG, EC2) is built in a different Region. This provides the "secondary" or "standby" deployment.
    
2. **Single CloudFront Distribution:** You continue to use your _single_ existing CloudFront distribution. This is a key benefit, as all users continue to access the same `www.example.com` endpoint (`distro.cloudfront.net`) without any changes.
    
3. **Origin Group:** Inside the CloudFront configuration, you create an "Origin Group."
    
    - **Primary Origin:** The original ALB (e.g., in `us-east-1`).
        
    - **Secondary Origin:** The new ALB (e.g., in `us-west-2`).
        
4. **Failover Mechanism:** You configure the cache behavior to use this _origin group_ instead of the single origin. When a user requests content, CloudFront sends the request to the primary origin (ALB in `us-east-1`). If that origin returns a 500, 502, 503, or 504 status code, CloudFront _immediately_ retries the request against the secondary origin (ALB in `us-west-2`).
    

This failover is transparent to the end-user and happens at the CDN level, making it much faster and more reliable than DNS-based failover.

**Reference:**

- AWS Documentation: [Optimizing high availability with CloudFront origin failover](https://www.google.com/search?q=https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/high-availability-origin-failover.html) - This document explicitly describes the solution in Option B.

## Incorrect Options Explanation

### A. Provision a full, secondary application deployment in a different AWS Region. Update the Route 53 A record to be a failover record. Add both of the CloudFront distributions as values. Create Route 53 health checks.

This architecture is a _valid_ multi-Region HA pattern, but it's _inferior_ to Option B for this scenario. It involves creating **two** separate CloudFront distributions and using **Route 53 DNS failover**.

- **Why it's not the best choice:** DNS failover is subject to DNS Time-to-Live (TTL) caching. When the primary origin fails, Route 53 will detect this via a health check and start returning the IP of the secondary distribution. However, clients and recursive DNS servers that have the _old_ DNS record cached will continue to hit the _failed_ primary distribution until their cache expires (which could be minutes or even hours). Option B's failover is nearly instantaneous.
    

### C. Provision an Auto Scaling group and EC2 instances in a different AWS Region. Create a second target for the new Auto Scaling group in the ALB. Set up the failover routing algorithm on the ALB.

This solution is **technically impossible**. An Application Load Balancer (ALB) is a **regional** service. It can only route traffic to target groups (like ASGs) that exist _within the same Region_. An ALB in `us-east-1` cannot have a target group in `eu-west-1`.

### D. Provision a full, secondary application deployment in a different AWS Region. Create a second CloudFront distribution, and add the new application setup as an origin. Create an AWS Global Accelerator accelerator. Add both of the CloudFront distributions as endpoints.

This is an **incorrect architectural pattern**. You would not put AWS Global Accelerator _in front of_ Amazon CloudFront. Both services act as a global entry point for your application, but they serve different purposes:

- **CloudFront** is for web content (static and dynamic) and benefits from a massive global network of edge caches.
    
- **Global Accelerator** is for accelerating traffic (both HTTP and non-HTTP) to regional endpoints, providing static Anycast IPs and routing to the healthiest, nearest backend (like an ALB or NLB).
    

Using GA to point to CloudFront is redundant and incorrect. The correct pattern is either `User -> CloudFront -> ALB` (for cached web content) or `User -> Global Accelerator -> ALB` (for non-cacheable or stateful applications).
