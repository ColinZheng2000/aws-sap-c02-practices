A health insurance company stores personally identifiable information (PII) in an Amazon S3 bucket. The company uses server-side encryption with S3 managed encryption keys (SSE-S3) to encrypt the objects. According to a new requirement, all current and future objects in the S3 bucket must be encrypted by keys that the company’s security team manages. The S3 bucket does not have versioning enabled.  
  
Which solution will meet these requirements?

- A. In the S3 bucket properties, change the default encryption to SSE-S3 with a customer managed key. Use the AWS CLI to re-upload all objects in the S3 bucket. Set an S3 bucket policy to deny unencrypted PutObject requests.
- B. In the S3 bucket properties, change the default encryption to server-side encryption with AWS KMS managed encryption keys (SSE-KMS). Set an S3 bucket policy to deny unencrypted PutObject requests. Use the AWS CLI to re-upload all objects in the S3 bucket.
- C. In the S3 bucket properties, change the default encryption to server-side encryption with AWS KMS managed encryption keys (SSE-KMS). Set an S3 bucket policy to automatically encrypt objects on GetObject and PutObject requests.
- D. In the S3 bucket properties, change the default encryption to AES-256 with a customer managed key. Attach a policy to deny unencrypted PutObject requests to any entities that access the S3 bucket. Use the AWS CLI to re-upload all objects in the S3 bucket.


---

## Understanding the Problem

The core of this problem is a data-at-rest encryption migration for an S3 bucket.

1.  **Current State:** Data (PII) is encrypted using **SSE-S3**. This means Amazon S3 manages the encryption keys (specifically, a unique key for each object, itself encrypted by a root key that S3 manages). The company has no control over these keys.
2.  **Required State:** The company's security team must *manage* the encryption keys. In AWS, this translates directly to using **AWS Key Management Service (KMS)** with a **customer-managed key (CMK)**. The service for this is **SSE-KMS**.
3.  **Scope:** The requirement applies to *all current* and *future* objects.
    * **Future Objects:** This can be solved by setting the S3 bucket's **default encryption** property to SSE-KMS and using a policy to *enforce* it.
    * **Current Objects:** This is the critical part. As official AWS documentation states, changing the default encryption *does not* re-encrypt existing objects. The existing objects are still encrypted with the old SSE-S3 keys. To re-encrypt them, they must be *rewritten* (e.g., copied over themselves).
4.  **Constraint:** The bucket does not have versioning, which simplifies the rewrite process—a copy-in-place will just overwrite the object.

## Services Explanation

* **Amazon S3 (Simple Storage Service):** A scalable object storage service. It provides various features for data-at-rest encryption.
* **Server-Side Encryption (SSE):** Encrypting data *after* it's received by S3, at the storage layer.
    * **SSE-S3:** Server-Side Encryption with S3-Managed Keys. S3 manages the entire key lifecycle. This is the *original* state in the question. It uses the AES-256 algorithm.
    * **SSE-KMS:** Server-Side Encryption with AWS KMS-Managed Keys. This uses AWS KMS to manage the encryption keys. This is the *target* state.
        * **AWS Managed Key (e.g., `aws/s3`):** A key in your account managed by AWS. You can't change its policy.
        * **Customer Managed Key (CMK):** A key you create and manage in KMS. You control its key policy, rotation, and lifecycle. This meets the requirement for the "security team to manage" the keys.
* **AWS KMS (Key Management Service):** A managed service to create and control encryption keys. It provides a highly available and auditable key management system.
* **S3 Bucket Policy:** A resource-based IAM policy attached to a bucket. It can be used to grant permissions or to *enforce* specific conditions, such as requiring encryption on all `PutObject` requests.
* **AWS CLI (Command Line Interface):** A tool to manage AWS services from the command line. A common way to re-encrypt existing objects is to run a recursive copy command, like `aws s3 cp s3://my-bucket/ s3://my-bucket/ --recursive`, which forces S3 to rewrite each object, applying the new default encryption settings.

## Correct Option(s) Explanation

### Option B
* **"In the S3 bucket properties, change the default encryption to server-side encryption with AWS KMS managed encryption keys (SSE-KMS)."**
    * **Analysis:** This is the correct first step for **future objects**. When setting this, the company would select their specific *customer-managed key* from KMS. Now, any *new* object uploaded without encryption headers will be automatically encrypted using this key.
* **"Set an S3 bucket policy to deny unencrypted PutObject requests."**
    * **Analysis:** This is a security best practice that helps enforce the encryption requirement for **future objects**. It prevents any user or service from accidentally uploading an unencrypted object. (Note: A stronger policy would deny any `PutObject` request that *doesn't* use `aws:kms`, preventing uploads using `SSE-S3` as well).
* **"Use the AWS CLI to re-upload all objects in the S3 bucket."**
    * **Analysis:** This is the *critical* step that addresses **current objects**. As stated in the [AWS S3 documentation](https://docs.aws.amazon.com/AmazonS3/latest/userguide/bucket-encryption.html), "Changing the default encryption configuration only applies to objects that are added to the bucket *after* the configuration is changed." To re-encrypt existing objects, they must be rewritten. Re-uploading (or copying them in-place) will cause S3 to apply the new default SSE-KMS configuration.

This option correctly addresses both *current* and *future* objects using the correct service (SSE-KMS).

---

## Incorrect Options Explanation

### Option A
* **"change the default encryption to SSE-S3 with a customer managed key."**
    * **Explanation:** This statement is technically contradictory. **SSE-S3** *by definition* uses S3-managed keys. **SSE-KMS** is the service that uses customer-managed keys. This terminology is incorrect, making the option invalid.

### Option C
* **"Set an S3 bucket policy to automatically encrypt objects on GetObject and PutObject requests."**
    * **Explanation:** This is incorrect for two major reasons.
    1.  It *fails to re-encrypt existing objects*. The *current* PII data would remain encrypted with SSE-S3, failing the primary requirement.
    2.  Encryption happens on write (`PutObject`), not on read (`GetObject`). You *decrypt* on `GetObject`. The statement is technically flawed.

### Option D
* **"change the default encryption to AES-256 with a customer managed key."**
    * **Explanation:** This has the same terminology flaw as Option A. In the S3 console and documentation, "AES-256" is the alias for **SSE-S3** (S3-managed keys). While SSE-KMS *also* uses the AES-256 algorithm, "AES-256" as an *encryption type* refers to SSE-S3. The correct *service* for using customer-managed keys is **SSE-KMS**, as stated in Option B.
