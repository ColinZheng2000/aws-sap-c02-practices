A company wants to migrate its on-premises application to AWS. The database for the application stores structured product data and temporary user session data. The company needs to decouple the product data from the user session data. The company also needs to implement replication in another AWS Region for disaster recovery.

Which solution will meet these requirements with the HIGHEST performance?

A. Create an Amazon RDS DB instance with separate schemas to host the product data and the user session data. Configure a read replica for the DB instance in another Region.
B. Create an Amazon RDS DB instance to host the product data. Configure a read replica for the DB instance in another Region. Create a global datastore in Amazon ElastiCache for Memcached to host the user session data.
C. Create two Amazon DynamoDB global tables. Use one global table to host the product data. Use the other global table to host the user session data. Use DynamoDB Accelerator (DAX) for caching.
D. Create an Amazon RDS DB instance to host the product data. Configure a read replica for the DB instance in another Region. Create an Amazon DynamoDB global table to host the user session data.

# AWS SAP-C02 Analysis — Decoupling Structured Product Data and Session Data with Cross-Region Replication

## Understanding the Problem

The requirements are:

- Migrate an on-premises application to AWS
- Store:
  - Structured product data
  - Temporary user session data
- Decouple these two workloads
- Replicate data to another AWS Region for disaster recovery
- Achieve the HIGHEST performance

The key architectural clue is:
- Product data and session data have very different access patterns

Typical enterprise design:
- Structured relational data → relational database
- High-throughput ephemeral session state → in-memory or NoSQL datastore

The phrase "highest performance" is extremely important.

- --

# Correct Option

## D. Amazon RDS for product data + cross-Region read replica + DynamoDB global table for session data

- --

## Why this is correct

This option properly separates workloads according to their characteristics.

### 1. Product data fits relational databases

Structured product data commonly requires:
- Relational integrity
- Complex queries
- Transactions
- Joins
- Referential constraints

Amazon RDS is appropriate for:
- Product catalogs
- ERP-style structured data
- Enterprise transactional systems

Cross-Region read replicas provide:
- Disaster recovery
- Read scalability
- Lower RPO

Official reference:
- https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_ReadRepl.XRgn.html

- --

### 2. User session data fits DynamoDB extremely well

Temporary session data requires:
- Very low latency
- Massive scalability
- Simple key-value access
- High throughput
- Multi-Region replication

DynamoDB global tables are excellent for:
- Session stores
- Shopping carts
- Authentication/session tokens
- Distributed user state

Benefits:
- Single-digit millisecond latency
- Multi-Region active-active replication
- Fully managed
- Extremely high scalability

This is generally higher performance than relational session storage.

Official reference:
- https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/GlobalTables.html

- --

### 3. Proper workload decoupling

This architecture decouples:
- Transactional relational workload
- Stateless session workload

Benefits:
- Independent scaling
- Better resilience
- Optimized performance
- Reduced database contention

This is a common enterprise modernization pattern.

- --

### 4. Best performance among all options

Why highest performance?

Because:
- RDS handles relational queries efficiently
- DynamoDB handles ultra-low-latency key-value session access
- Session traffic avoids relational database bottlenecks

This avoids:
- Connection exhaustion
- Lock contention
- Excessive relational writes

Especially important in:
- High-scale web applications
- Industrial customer portals
- Medical application session handling

# Incorrect Options

## A. Single RDS instance with separate schemas

### Why this is incorrect

This option does NOT properly decouple workloads.

Problems:
- Product data and session data share same DB resources
- Session workload can impact transactional workload
- Increased contention
- Reduced scalability
- Higher latency under heavy user activity

Session data in relational databases is a common anti-pattern at scale.

Although cross-Region replication exists, performance is inferior.

- --

## B. RDS + ElastiCache for Memcached global datastore

### Why this is incorrect

Main issue:
- Amazon ElastiCache for Memcached does NOT support global datastore

Global datastore is supported for:
- Amazon ElastiCache for Redis

NOT:
- Memcached

Therefore this option is technically invalid.

Official reference:
- https://docs.aws.amazon.com/AmazonElastiCache/latest/red-ug/Redis-Global-Datastores.html

- --

## C. Two DynamoDB global tables + DAX

### Why this is incorrect

This option delivers extremely high performance but it is not the best fit for structured product data.

### Problem 1 — Structured relational data

Product data is described as:
- Structured

This often implies:
- Relational querying
- Complex relationships
- SQL-style operations

DynamoDB is not ideal for:
- Complex relational workloads
- Ad hoc joins
- Transaction-heavy relational systems

- --

### Problem 2 —AX is unnecessary for session storage

DAX is:
- An in-memory cache for DynamoDB
- Useful for read-heavy workloads

Session data typically:
- Has frequent writes
- Is already low-latency in DynamoDB

DAX does not significantly improve session-write performance.

- --

### Problem 3 — Overengineering

This design:
- Increases architectural complexity
- Requires DynamoDB data modeling expertise
- May complicate relational migration

For enterprise migrations, retaining relational product data in RDS is often the most practical and performant approach.

# Hands-on Knowledge for SAP-C02

## Real Enterprise Architecture Pattern

Very common production architecture:

### Product Data
- Amazon Aurora or RDS
- Multi-AZ deployment
- Cross-Region replicas

### Session State
- DynamoDB global tables
OR
- ElastiCache Redis

This separation is common in:
- E-commerce
- Healthcare portals
- Industrial dashboards
- SaaS platforms

- --

## Why Session Data Should Be Decoupled

Session workloads are:
- Bursty
- High-volume
- Low-value
- Short-lived

Storing them in relational databases causes:
- Excessive writes
- Lock contention
- Increased IOPS costs
- Performance degradation

- --

## DynamoDB Global Tables for DR

Global tables provide:
- Active-active replication
- Automatic multi-Region synchronization
- Fast failover support

Excellent for:
- Multi-Region user applications
- Disaster recovery
- Global applications

- --

## Enterprise DR Consideration

RDS cross-Region replicas:
- Usually asynchronous
- May have replication lag

DynamoDB global tables:
- Replicate automatically
- Near real-time replication

For session data:
- Small inconsistencies are usually acceptable

- --

## SAP-C02 Exam Tip

When you see:
- Temporary session data
- Massive scale
- High performance
- Low latency

Think:
- DynamoDB
- ElastiCache Redis

When you see:
- Structured transactional data

Think:
- RDS
- Aurora

The exam frequently tests:
- Matching datastore type to workload pattern
