A solutions architect is planning to migrate critical Microsoft SQL Server databases to AWS. Because the databases are legacy systems, the solutions architect will move the databases to a modern data architecture. The solutions architect must migrate the databases with near-zero downtime.

Which solution will meet these requirements?

A. Use AWS Application Migration Service and the AWS Schema Conversion Tool (AWS SCT). Perform an in-place upgrade before the migration. Export the migrated data to Amazon Aurora Serverless after cutover. Repoint the applications to Amazon Aurora.
B. Use AWS Database Migration Service (AWS DMS) to rehost the database. Set Amazon S3 as a target. Set up change data capture (CDC) replication. When the source and destination are fully synchronized, load the data from Amazon S3 into an Amazon RDS for Microsoft SQL Server DB instance.
C. Use native database high availability tools. Connect the source system to an Amazon RDS for Microsoft SQL Server DB instance. Configure replication accordingly. When data replication is finished, transition the workload to an Amazon RDS for Microsoft SQL Server DB instance.
D. Use AWS Application Migration Service. Rehost the database server on Amazon EC2. When data replication is finished, detach the database and move the database to an Amazon RDS for Microsoft SQL Server DB instance. Reattach the database and then cut over all networking.


Suggest Answer: C

---

## Analysis

### Understanding the Problem

A solutions architect must migrate **critical Microsoft SQL Server databases** (legacy systems) to AWS with **near-zero downtime**. The migration must also transition to a "modern data architecture." The key tension: balancing minimal cutover downtime with architectural modernization, while respecting that SQL Server has specific migration tooling constraints on AWS.

### Correct Option: C

**Use native database high availability tools. Connect the source system to an Amazon RDS for Microsoft SQL Server DB instance. Configure replication accordingly. When data replication is finished, transition the workload to an Amazon RDS for Microsoft SQL Server DB instance.**

- **Native SQL Server HA tools** (Always On Availability Groups, log shipping, database mirroring, or transactional replication) are the gold standard for SQL Server migrations. They are battle-tested and offer the most control over replication latency and cutover timing.
- **Amazon RDS for SQL Server** natively supports backup/restore via S3 (`rds_backup_database` and `rds_restore_database` stored procedures) and can act as a replication target/subscriber. You can take a full backup, restore it to RDS, then configure ongoing log shipping or replication to keep the target synchronized.
- **Near-zero downtime** is achievable because the source database remains online during initial data seeding and ongoing replication. At cutover, you perform a brief application stop, apply any final transaction log backups, and repoint the connection strings to the RDS endpoint. Typical downtime: seconds to minutes.
- RDS for SQL Server eliminates the operational burden of managing the SQL Server engine (patching, backups, Multi-AZ failover), fulfilling the "modern data architecture" goal within the SQL Server ecosystem.

### Incorrect Options

**A: Application Migration Service + SCT → Aurora Serverless** — This option contains a **fatal technical incompatibility**: Amazon Aurora Serverless supports only **MySQL and PostgreSQL** engines. There is no Aurora flavor for Microsoft SQL Server. AWS Schema Conversion Tool (SCT) could theoretically convert the schema from SQL Server to PostgreSQL, but converting a critical legacy database's schema, stored procedures, functions, and data types is an extremely high-risk, months-long project — not a "near-zero downtime" migration. Application Migration Service (MGN) is designed for server-level (block) replication, not database-level migration.

**B: DMS → S3 → RDS SQL Server** — AWS Database Migration Service (DMS) can use S3 as an intermediate target, but this adds an **unnecessary hop**. DMS supports RDS for SQL Server as a direct target endpoint, eliminating the S3 intermediary. The extra load/unload step from S3 increases migration time and introduces a failure point. Additionally, DMS has limitations with SQL Server-specific features (triggers, CLR assemblies, certain data types) that may cause issues during migration.

**D: Application Migration Service → EC2 → detach DB → RDS** — MGN replicates at the **block level** (entire server). After replication, the idea of "detaching" a database from a SQL Server instance on EC2 and "reattaching" it to RDS is **not supported** by AWS. RDS for SQL Server is a managed service — you cannot attach arbitrary MDF/LDF database files. The only supported data import paths into RDS SQL Server are: native backup/restore via S3, DMS, or bulk copy (bcp). This option describes a technically impossible operation.

### Hands-On Knowledge

1. **Native Backup/Restore to RDS SQL Server**: 
   - Run `EXEC msdb.dbo.rds_backup_database @source_db_name='MyDB', @s3_arn='arn:aws:s3:::bucket/backup.bak';` on the source
   - Copy the backup to S3
   - On RDS: `EXEC msdb.dbo.rds_restore_database @restore_db_name='MyDB', @s3_arn='arn:aws:s3:::bucket/backup.bak';`
   - Monitor with: `EXEC msdb.dbo.rds_task_status;`

2. **SQL Server Always On with RDS**: RDS SQL Server Enterprise Edition (Multi-AZ) supports Always On Availability Groups within AWS. For hybrid migration, you can use log shipping from on-premises to RDS, then switch the application when the lag is near zero.

3. **DMS Limitations with SQL Server**: DMS does not replicate the following: system databases (master, msdb), logins/users (use `rds_restore_database` with `@restore_metadata_only=1` or manually script them), SQL Server Agent jobs, and certain data types (sql_variant, hierarchyid, geography). Always verify feature compatibility before choosing DMS.

4. **AWS Application Migration Service (MGN)**: MGN is ideal for "lift-and-shift" rehosting of entire servers to EC2. It replicates continuously at the block level. MGN is **not** a database migration tool — it moves the entire OS and application stack. Use it when you plan to keep SQL Server on EC2, not when moving to RDS.

5. **Real-world pattern for SQL Server modernization**: Many enterprises use a phased approach — Phase 1: native tools migration to RDS SQL Server (stabilize on managed service); Phase 2: evaluate SCT for conversion to Aurora PostgreSQL for specific application modules where modernization ROI justifies the effort.
