A company hosts a blog post application on AWS using Amazon API Gateway, Amazon DynamoDB, and AWS Lambda. The application currently does not use API keys to authorize requests. The API model is as follows:

GET /posts/{postId}: to get post details
GET /users/{userId}: to get user details
GET /comments/{commentId}: to get comments details

The company has noticed users are actively discussing topics in the comments section, and the company wants to increase user engagement by making the comments appear in real time.

Which design should be used to reduce comment latency and improve user experience?

A. Use edge-optimized API with Amazon CloudFront to cache API responses.
B. Modify the blog application code to request GET/comments/{commentId} every 10 seconds.
C. Use AWS AppSync and leverage WebSockets to deliver comments.
D. Change the concurrency limit of the Lambda functions to lower the API response time.


Suggest Answer: C

---

## 📊 Analysis

### 🔍 Understanding the Problem

A company hosts a blog application using **Amazon API Gateway** (REST API), **Amazon DynamoDB**, and **AWS Lambda**. The API endpoints are:
- `GET /posts/{postId}` — get post details
- `GET /users/{userId}` — get user details
- `GET /comments/{commentId}` — get comments details

Currently, the application uses a traditional **request-response (polling) model** — clients must explicitly request comment data to see updates. The company wants to **increase user engagement** by making comments appear in **real time**, reducing latency. The core challenge is: how do you transition from a REST-based polling architecture to a real-time push-based architecture?

### ✅ Correct Option: C — Use AWS AppSync and Leverage WebSockets to Deliver Comments

**AWS AppSync** is a fully managed GraphQL service that natively supports **real-time subscriptions** via **WebSockets** (using MQTT over WebSockets protocol). Here's why this is the optimal solution:

- **Real-Time Subscriptions**: AppSync allows clients to subscribe to specific data mutations. When a new comment is added (via a GraphQL mutation), AppSync automatically pushes the update to all subscribed clients — no polling needed.
- **WebSocket Protocol**: Unlike REST's request-response model, WebSockets maintain a persistent bidirectional connection, enabling the server to push data to clients instantly with minimal latency.
- **Managed Service**: AppSync handles connection management, scaling, and reconnection logic — zero operational overhead.
- **Integration with DynamoDB**: AppSync can connect directly to DynamoDB as a data source, making it seamless to integrate with the existing backend.
- **Granular Subscriptions**: Clients can subscribe to comments for a specific post (`comments/{postId}`), ensuring they only receive relevant updates.

The architecture flow would be:
1. Client opens a WebSocket connection via AppSync and subscribes to `onCreateComment` for a specific post
2. When a user posts a comment, a GraphQL mutation triggers
3. AppSync resolves the mutation (writes to DynamoDB) and publishes the new comment to all subscribers
4. All connected clients see the comment appear immediately

### ❌ Incorrect Options

**A. Use edge-optimized API with Amazon CloudFront to cache API responses**
- An **edge-optimized API** routes requests through CloudFront edge locations to reduce latency for globally distributed users, but it still operates on a **request-response (REST)** model
- CloudFront caching would actually **worsen** the real-time experience — cached responses would return stale data for a period defined by the TTL
- Caching is counterproductive for real-time data that changes frequently (like comments)
- This does NOT address the fundamental architecture change from polling to push

**B. Modify the blog application code to request GET/comments/{commentId} every 10 seconds**
- This is client-side **polling**, which is the exact opposite of what we want
- Polling introduces:
  - **10-second latency** at minimum — comments appear up to 10 seconds late
  - **Unnecessary network traffic** and API calls even when no new comments exist
  - **Increased backend load** — every active user hammers the API every 10 seconds
  - **Does not scale** — 10,000 concurrent users polling every 10 seconds = 1,000 requests/second continuously
- This is the "naive" solution that should be avoided

**D. Change the concurrency limit of the Lambda functions to lower the API response time**
- Lambda **concurrency limits** control how many concurrent executions are allowed, not how fast each execution runs
- Increasing concurrency might help with throughput under load but has **zero impact on individual request latency** for a well-functioning Lambda
- Lambda cold starts contribute to latency, but that's addressed by Provisioned Concurrency, not concurrency limits
- This fundamentally misdiagnoses the problem: the issue is the polling architecture, not Lambda performance

### 🏭 Enterprise Hands-On Knowledge

1. **AppSync vs. API Gateway Decision Framework**:
   | Requirement | AppSync (GraphQL) | API Gateway (REST/HTTP) |
   |-------------|-------------------|------------------------|
   | Real-time / WebSocket | ✅ Native subscriptions | ✅ WebSocket API (separate) |
   | Over-fetching control | ✅ Client specifies fields | ❌ Fixed response schema |
   | Multiple data sources in one request | ✅ Single query | ❌ Multiple API calls |
   | Caching | ✅ Server-side caching | ✅ CloudFront/API Gateway cache |
   | Learning curve | Medium (GraphQL) | Low (REST) |

2. **AppSync Subscription Patterns**:
   - **onCreate** subscriptions: Trigger for new records (new comments)
   - **onUpdate** subscriptions: Trigger for edits (edited comments)
   - **onDelete** subscriptions: Trigger for deletions (moderated comments)
   - Use **subscription arguments** to filter by `postId` so users only get comments for the post they're viewing

3. **WebSocket Connection Management**: AppSync handles reconnection automatically. The client SDK (Amplify) includes built-in retry logic. For enterprise apps, implement heartbeat mechanisms to detect stale connections.

4. **Alternative: API Gateway WebSocket API**: You could also use API Gateway's WebSocket API with Lambda and DynamoDB Streams to push updates. However, AppSync provides a more integrated experience for GraphQL-based applications and reduces boilerplate code.

5. **Cost Considerations**: AppSync charges for:
   - **Query and Mutation operations** (per million)
   - **Real-time subscription connection minutes** — this can add up for chatty applications
   - **Data transfer out**
   - For high-traffic comment sections, consider caching frequently-read data with AppSync's built-in caching

6. **Security**: AppSync supports multiple authorization modes:
   - **API_KEY**: For public read access (reading comments)
   - **AMAZON_COGNITO_USER_POOLS**: For authenticated users posting comments
   - **AWS_IAM**: For service-to-service communication
   - Combine modes so reading comments is open but posting requires authentication
