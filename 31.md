A company is hosting an image-processing service on AWS in a VPC. The VPC extends across two Availability Zones. Each Availability Zone contains one public subnet and one private subnet.  
The service runs on Amazon EC2 instances in the private subnets. An Application Load Balancer in the public subnets is in front of the service. The service needs to communicate with the internet and does so through two NAT gateways. The service uses Amazon S3 for image storage. The EC2 instances retrieve approximately 1 ׀¢׀’ of data from an S3 bucket each day.  
The company has promoted the service as highly secure. A solutions architect must reduce cloud expenditures as much as possible without compromising the service's security posture or increasing the time spent on ongoing operations.  
Which solution will meet these requirements?  

- A. Replace the NAT gateways with NAT instances. In the VPC route table, create a route from the private subnets to the NAT instances.
- B. Move the EC2 instances to the public subnets. Remove the NAT gateways.
- C. Set up an S3 gateway VPC endpoint in the VPC. Attach an endpoint policy to the endpoint to allow the required actions on the S3 bucket.
- D. Attach an Amazon Elastic File System (Amazon EFS) volume to the EC2 instances. Host the image on the EFS volume.

## Analysis

### Understanding the Problem

The company is hosting an image-processing service on **Amazon EC2 instances** in **private subnets** across two Availability Zones (AZs). An **Application Load Balancer (ALB)** is in the public subnets. The EC2 instances need **internet access** (provided by two **NAT Gateways**) and need to access **Amazon S3** for image storage, retrieving a significant volume of data ($\approx 1 \text{ TB}$ daily).

The primary goal is to **reduce cloud expenditures as much as possible** without:

1. Compromising the service's **security posture**.
    
2. Increasing the time spent on **ongoing operations**.
    

The current architecture has a significant cost driver: **Data transfer out (DTO) through the NAT Gateways**. Accessing S3 from a private subnet via a NAT Gateway incurs:

1. **NAT Gateway Hourly Charge** for the $1 \text{ TB}$ of data processed.
    
2. **Data Processing Charge** for the $1 \text{ TB}$ of data processed through the NAT Gateway.
    

The $1 \text{ TB}$ of daily S3 traffic through the NAT Gateways is a substantial recurring cost. Eliminating this cost while maintaining security is the key to solving this problem.

### Services Explanation

| **Service**                       | **Description**                                                                                                                                                                                                                                                                     | **Relevance to the Question**                                                                                                                                           |
| --------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **NAT Gateway**                   | A managed AWS service that allows instances in a private subnet to connect to the internet or other AWS services outside the VPC but prevents the internet from initiating a connection with those instances. **Cost is incurred for hourly uptime and per GB of data processed.**  | A major cost component that the solution aims to eliminate for S3 traffic.                                                                                              |
| **Amazon S3**                     | Object storage built to store and retrieve any amount of data from anywhere. Used here for image storage.                                                                                                                                                                           | The destination service for the high-volume data transfer.                                                                                                              |
| **VPC Gateway Endpoint (for S3)** | A $\text{VPC}$ endpoint that provides **reliable and secure connectivity** to **Amazon S3** from instances within a $\text{VPC}$ **without requiring an Internet Gateway, a NAT device, or a VPN connection.** Traffic uses the **AWS network backbone** and doesn't leave the VPC. | The specific service designed to provide private, cost-effective access to S3, bypassing NAT Gateways. **There is no charge for using $\text{VPC}$ Gateway Endpoints.** |
| **NAT Instance**                  | An $\text{EC2}$ instance configured to perform NAT. It's a self-managed service.                                                                                                                                                                                                    | A potential, but discouraged, alternative to NAT Gateway, mainly because it shifts operational burden and can be less reliable/scalable.                                |
| **Amazon EFS**                    | A scalable, elastic, cloud-native $\text{NFS}$ file system for $\text{EC2}$.                                                                                                                                                                                                        | An incorrect storage solution for an image-processing service where S3 is already in use and is the more appropriate object storage service.                            |

### Correct Option(s) Explanation

**C. Set up an S3 gateway VPC endpoint in the VPC. Attach an an endpoint policy to the endpoint to allow the required actions on the S3 bucket.**

- **Cost Reduction:** The primary cost in the current setup related to S3 access is the data processing fee through the **NAT Gateways**. $\text{VPC}$ **Gateway Endpoints** allow the $\text{EC2}$ instances in the private subnets to access **Amazon S3 directly** over the **AWS private network** (as documented in the [AWS $\text{VPC}$ documentation](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-endpoints-s3.html)). **There is no additional charge for using a $\text{VPC}$ Gateway Endpoint**, thus eliminating the NAT Gateway data processing charges for the $1 \text{ TB}$ of daily S3 data retrieval. This is a massive cost saving.
    
- **Security Posture:** Accessing S3 via a $\text{VPC}$ Endpoint is inherently **more secure** than using a NAT Gateway as the traffic never traverses the public internet and remains within the $\text{AWS}$ network. Furthermore, an **endpoint policy** can be attached to **restrict S3 access only to the necessary bucket and actions**, improving the security posture.
    
- **Operational Overhead:** Setting up a $\text{VPC}$ Gateway Endpoint is a one-time configuration change (creating the endpoint and updating route tables). It's a **managed service** and adds **no operational burden** (no maintenance, patching, or scaling concerns).
    

### Incorrect Options Explanation

**A. Replace the NAT gateways with NAT instances. In the VPC route table, create a route from the private subnets to the NAT instances.**

- **Operational Overhead:** Replacing a managed service ($\text{NAT}$ Gateway) with a self-managed solution ($\text{NAT}$ Instance) **significantly increases operational overhead** (patching, maintenance, scaling, high availability setup).
    
- **Cost/Complexity:** While $\text{NAT}$ Instances might have a lower _hourly_ cost than $\text{NAT}$ Gateways, the overall cost savings are typically not significant, and **they still incur data transfer costs** for the S3 traffic, failing to meet the "reduce cloud expenditures as much as possible" goal.
    

**B. Move the EC2 instances to the public subnets. Remove the NAT gateways.**

- **Security Posture:** Moving the $\text{EC2}$ instances (which contain the core service) to **public subnets** significantly **compromises the security posture**. The instances would be directly addressable from the internet, which is typically a major security violation for backend services. The requirement is to _not_ compromise security.
    

**D. Attach an Amazon Elastic File System (Amazon EFS) volume to the EC2 instances. Host the image on the EFS volume.**

- **Operational Overhead/Cost:** This option involves **changing the storage mechanism entirely** from $\text{S3}$ to $\text{EFS}$. This is a complex change, likely requiring application code modification and data migration, which significantly **increases operational time and cost**.
    
- **Cost/Suitability:** While $\text{EFS}$ might be suitable for some workloads, **S3 is a cheaper and highly scalable solution** for "image storage" and is already in use. Changing the storage system is not the most direct or efficient way to solve a NAT Gateway cost issue.
