A company is running a data-intensive application on AWS. The application runs on a cluster of hundreds of Amazon EC2 instances. A shared file system also runs on several EC2 instances that store 200 TB of data. The application reads and modifies the data on the shared file system and generates a report. The job runs once monthly, reads a subset of the files from the shared file system, and takes about 72 hours to complete. The compute instances scale in an Auto Scaling group, but the instances that host the shared the system run continuously. The compute and storage instances are all in the same AWS Region.  
A solutions architect needs to reduce costs by replacing the shared file system instances. The file system must provide high performance access to the needed data for the duration of the 72-hour run.  
Which solution will provide the LARGEST overall cost reduction while meeting these requirements?  

- A. Migrate the data from the existing shared file system to an Amazon S3 bucket that uses the S3 Intelligent-Tiering storage class. Before the job runs each month, use Amazon FSx for Lustre to create a new file system with the data from Amazon S3 by using lazy loading. Use the new file system as the shared storage for the duration of the job. Delete the file system when the job is complete.
- B. Migrate the data from the existing shared file system to a large Amazon Elastic Block Store (Amazon EBS) volume with Multi-Attach enabled. Attach the EBS volume to each of the instances by using a user data script in the Auto Scaling group launch template. Use the EBS volume as the shared storage for the duration of the job. Detach the EBS volume when the job is complete.
- C. Migrate the data from the existing shared file system to an Amazon S3 bucket that uses the S3 Standard storage class. Before the job runs each month, use Amazon FSx for Lustre to create a new file system with the data from Amazon S3 by using batch loading. Use the new file system as the shared storage for the duration of the job. Delete the file system when the job is complete.
- D. Migrate the data from the existing shared file system to an Amazon S3 bucket. Before the job runs each month, use AWS Storage Gateway to create a file gateway with the data from Amazon S3. Use the file gateway as the shared storage for the job. Delete the file gateway when the job is complete.



## Analysis

### Understanding the Problem

The company is running a data-intensive application that uses a **shared file system** hosted on continuously running Amazon EC2 instances to store **200 TB** of data. The application runs a compute-intensive job **once monthly** for about **72 hours** (3 days), reading and modifying a **subset of the files**. The goal is to **reduce costs significantly** by replacing the continuously running shared file system instances while maintaining **high-performance access** for the 72-hour job duration. The compute instances are in an Auto Scaling group, and both compute and storage are in the same AWS Region. The cost savings must be "the largest overall."

**Key Constraints/Requirements:**

1. **Shared File System:** The solution must provide shared file access for a cluster of hundreds of EC2 instances.
    
2. **High Performance:** Essential for the 72-hour job.
    
3. **Large Cost Reduction:** The primary objective.
    
4. **Data Size:** 200 TB.
    
5. **Usage Pattern:** Data is mostly _cold_ (only accessed for 3 days monthly); a _subset_ is read and modified.
    
6. **Duration:** Temporary usage (72 hours/month).

### Services Explanation

| Service                                | Description                                                                                                                                                                                                                | Relevance to the Problem                                                                                                                                                                                                                                                                                                         |
| -------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Amazon S3**                          | Object storage offering high durability, availability, and scalability. It is pay-as-you-go, making it very cost-effective for large amounts of rarely accessed data.                                                      | Ideal for storing the **200 TB** of data for the vast majority of the month (approx. 27-28 days) to achieve the largest cost reduction.                                                                                                                                                                                          |
| **S3 Intelligent-Tiering**             | S3 storage class that automatically moves data between two access tiers (Frequent Access and Infrequent Access) based on access patterns, optimizing storage costs without performance impact.                             | A **cost-effective** choice for data with **unknown or changing access patterns**, like the 200 TB that is mostly idle but briefly accessed monthly.                                                                                                                                                                             |
| **S3 Standard**                        | S3 storage class for frequently accessed data.                                                                                                                                                                             | Less cost-effective than S3 Intelligent-Tiering or S3 Glacier Flexible Retrieval for data that is mostly infrequently accessed, even if it is accessed for 72 hours monthly.                                                                                                                                                     |
| **Amazon FSx for Lustre**              | A high-performance file system optimized for fast processing of workloads such as machine learning, high-performance computing (HPC), and media processing. It can be linked to an S3 bucket to process data stored there. | Meets the **high-performance** and **shared file system** requirements, and its ability to link to S3 allows for an on-demand, temporary deployment. **"Lazy loading"** and **"Batch loading"** are methods for populating the file system from S3.                                                                              |
| **Lazy Loading (FSx for Lustre)**      | Data is only copied from S3 to FSx for Lustre when it is first accessed. This is faster to set up and ideal when only a **subset of data is needed**.                                                                      | Matches the requirement that only a **subset** of files is read, reducing the setup time and cost.                                                                                                                                                                                                                               |
| **Batch Loading (FSx for Lustre)**     | All data is pre-copied from S3 to FSx for Lustre before the job starts. This is slower to set up but ensures all data is immediately available at the start.                                                               | Less efficient if only a **subset** of data is needed, as it incurs the cost and time of copying all 200 TB.                                                                                                                                                                                                                     |
| **Amazon EBS Multi-Attach**            | Allows a single Provisioned IOPS SSD (io1 or io2) volume to be concurrently attached to up to 16 Nitro-based EC2 instances in the same Availability Zone.                                                                  | It **does not support shared file system access** across a cluster; it provides block-level access, and the operating system on the instances must manage concurrent access, which is very complex and risky for a general shared file system application. Also, the instances must be in the same AZ, not just the same Region. |
| **AWS Storage Gateway - File Gateway** | A set of hybrid cloud storage services that provides on-premises applications with access to AWS cloud storage. It acts as a cache/protocol translator for S3.                                                             | Primarily designed for **on-premises** to cloud integration. While it presents an NFS/SMB interface, running it on EC2 to access S3 _within_ the same Region is an **unnecessary and expensive overhead** compared to native integration like FSx for Lustre.                                                                    |

### Correct Option(s) Explanation

**A. Migrate the data from the existing shared file system to an Amazon S3 bucket that uses the S3 Intelligent-Tiering storage class. Before the job runs each month, use Amazon FSx for Lustre to create a new file system with the data from Amazon S3 by using lazy loading. Use the new file system as the shared storage for the duration of the job. Delete the file system when the job is complete.**

- **Cost Reduction:** Moving 200 TB of data from continuously running EC2 instances to **Amazon S3** (especially with S3 Intelligent-Tiering) provides the **largest cost reduction**. EC2 instance costs for 200 TB of storage are much higher than S3 costs. The **S3 Intelligent-Tiering** class further optimizes costs by automatically moving data between access tiers, which is ideal for data accessed infrequently (only 72 hours per month).
    
- **Performance and Shared Access:** **Amazon FSx for Lustre** is a **high-performance shared file system** that meets the requirement for the data-intensive application.
    
- **Temporary/On-Demand Usage:** FSx for Lustre is created **before the job** and **deleted when the job is complete**. This pay-as-you-go model for the high-performance layer (Lustre) only for the 72 hours per month is critical for cost savings compared to running a service 24/7.
    
- **Lazy Loading:** Since the job reads only a **subset of the files**, using **lazy loading** is the most efficient method. Data is only copied/read into the FSx for Lustre file system when explicitly requested, saving on data transfer costs, job setup time, and Lustre storage/I/O costs for the unused majority of the 200 TB.
    

---

### Incorrect Options Explanation

**B. Migrate the data from the existing shared file system to a large Amazon Elastic Block Store (Amazon EBS) volume with Multi-Attach enabled. Attach the EBS volume to each of the instances by using a user data script in the Auto Scaling group launch template. Use the EBS volume as the shared storage for the duration of the job. Detach the EBS volume when the job is complete.**

- **Shared Access Failure:** EBS Multi-Attach provides **block-level access**, not a shared file system. The operating system would need to manage concurrent access, which is highly complex, not supported by standard file systems (like XFS, ext4), and a major risk for data corruption. This violates the "shared file system" requirement.
    
- **Scalability/Availability Failure:** Multi-Attach is limited to a small number of instances (up to 16) and must be in the same Availability Zone, whereas the problem mentions a "cluster of hundreds of Amazon EC2 instances."
    

**C. Migrate the data from the existing shared file system to an Amazon S3 bucket that uses the S3 Standard storage class. Before the job runs each month, use Amazon FSx for Lustre to create a new file system with the data from Amazon S3 by using batch loading. Use the new file system as the shared storage for the duration of the job. Delete the file system when the job is complete.**

- **Not the Largest Cost Reduction:** This option uses **S3 Standard** (more expensive than S3 Intelligent-Tiering for infrequently accessed data) and **Batch Loading**. Batch loading copies the **entire 200 TB** from S3 to FSx for Lustre, which is expensive and unnecessary, as the job only needs a **subset** of the files. The combination makes it less cost-effective than Option A.
    

**D. Migrate the data from the existing shared file system to an Amazon S3 bucket. Before the job runs each month, use AWS Storage Gateway to create a file gateway with the data from Amazon S3. Use the file gateway as the shared storage for the job. Delete the file gateway when the job is complete.**

- **Performance/Cost Overhead:** AWS Storage Gateway (File Gateway) is primarily an **on-premises** solution for accessing cloud storage. Using it within an AWS Region introduces **unnecessary compute and caching overhead** (running the gateway EC2 instance) and **performance limitations** compared to the native, high-performance parallel file system of FSx for Lustre. It is a more complex and less performant solution than Option A for this specific cloud-native, high-performance, short-duration requirement, making the overall cost reduction smaller.
