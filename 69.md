A company plans to refactor a monolithic application into a modern application design deployed on AWS. The CI/CD pipeline needs to be upgraded to support the modern design for the application with the following requirements:  
✑ It should allow changes to be released several times every hour.  
✑ It should be able to roll back the changes as quickly as possible.  
Which design will meet these requirements?  

- A. Deploy a CI/CD pipeline that incorporates AMIs to contain the application and their configurations. Deploy the application by replacing Amazon EC2 instances.
- B. Specify AWS Elastic Beanstalk to stage in a secondary environment as the deployment target for the CI/CD pipeline of the application. To deploy, swap the staging and production environment URLs.
- C. Use AWS Systems Manager to re-provision the infrastructure for each deployment. Update the Amazon EC2 user data to pull the latest code artifact from Amazon S3 and use Amazon Route 53 weighted routing to point to the new environment.
- D. Roll out the application updates as part of an Auto Scaling event using prebuilt AMIs. Use new versions of the AMIs to add instances, and phase out all instances that use the previous AMI version with the configured termination policy during a deployment event.

---
### Understanding the Problem

The company is refactoring a monolithic application to a modern design on AWS and needs a **CI/CD pipeline** that supports:

1. **High frequency of releases:** Several times every hour (implying speed and minimal downtime).
    
2. **Fast rollback capability:** Ability to quickly revert to a previous working state.
    

This demands an **immutable deployment strategy** that offers low-risk, zero-downtime deployment and immediate rollback capabilities.

### Correct Option(s) Explanation

The correct option is **B. Specify AWS Elastic Beanstalk to stage in a secondary environment as the deployment target for the CI/CD pipeline of the application. To deploy, swap the staging and production environment URLs.**

- **Zero Downtime/High Frequency:** This describes a **Blue/Green Deployment** strategy implemented via Elastic Beanstalk's **environment swap** feature. The "staging" environment (Green) is provisioned with the new application version while the "production" environment (Blue) is still serving traffic. Once testing is complete, the URLs are instantly swapped. This offers a **near-instantaneous, zero-downtime deployment** ideal for releasing changes "several times every hour."
    
- **Fast Rollback:** If an issue is found, the **rollback is immediate** by simply performing a **reverse swap** of the environment URLs. This is the fastest rollback mechanism, as the old, working environment (Blue) is still healthy and available.
    

> _Reference:_ The AWS documentation on Elastic Beanstalk deployment policies details the "Swap Environment URLs" option, which is a form of blue/green deployment that satisfies the requirements for speed and quick rollback.

---

### Incorrect Options Explanation

- **A. Deploy a CI/CD pipeline that incorporates AMIs to contain the application and their configurations. Deploy the application by replacing Amazon EC2 instances.**
    
    - **Issue:** While using AMIs for immutable infrastructure is a great practice, "replacing Amazon EC2 instances" generally refers to an **Immutable Deployment** (like with Auto Scaling/CodeDeploy) or a **rolling update** that replaces one instance at a time. This process of creating a new AMI, spinning up new instances, waiting for health checks, and terminating old ones is **slower** than an environment URL swap and may introduce brief periods of mixed traffic or latency, making it less ideal for releases "several times every hour." The rollback is also slower, requiring another full deployment of the old AMI.
        
- **C. Use AWS Systems Manager to re-provision the infrastructure for each deployment. Update the Amazon EC2 user data to pull the latest code artifact from Amazon S3 and use Amazon Route 53 weighted routing to point to the new environment.**
    
    - **Issue:** Using Systems Manager to **re-provision infrastructure** for _every_ deployment is **slow and inefficient**. Furthermore, pulling the code at instance launch via **user data** is a **mutable deployment** approach (instances change state after launch), which is riskier and slower to roll back than an immutable or blue/green strategy. While Route 53 weighted routing can achieve a gradual traffic shift, it is a **slow rollout** strategy, which conflicts with the need for immediate, high-frequency releases and fast rollbacks.
        
- **D. Roll out the application updates as part of an Auto Scaling event using prebuilt AMIs. Use new versions of the AMIs to add instances, and phase out all instances that use the previous AMI version with the configured termination policy during a deployment event.**
    
    - **Issue:** This describes a standard **Immutable Deployment** via Auto Scaling. Similar to option A, while it ensures consistency, the process of launching new EC2 instances, warming them up, and then terminating old ones takes significant time (minutes) and is **not as fast** as an immediate URL swap. The rollback process requires another full deployment cycle of the old AMI, which is also **slower** than an instant environment swap.
