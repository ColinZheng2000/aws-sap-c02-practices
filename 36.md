A company hosts a Git repository in an on-premises data center. The company uses webhooks to invoke functionality that runs in the AWS Cloud. The company hosts the webhook logic on a set of Amazon EC2 instances in an Auto Scaling group that the company set as a target for an Application Load Balancer (ALB). The Git server calls the ALB for the configured webhooks. The company wants to move the solution to a serverless architecture.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. For each webhook, create and configure an AWS Lambda function URL. Update the Git servers to call the individual Lambda function URLs.
- B. Create an Amazon API Gateway HTTP API. Implement each webhook logic in a separate AWS Lambda function. Update the Git servers to call the API Gateway endpoint.
- C. Deploy the webhook logic to AWS App Runner. Create an ALB, and set App Runner as the target. Update the Git servers to call the ALB endpoint.
- D. Containerize the webhook logic. Create an Amazon Elastic Container Service (Amazon ECS) cluster, and run the webhook logic in AWS Fargate. Create an Amazon API Gateway REST API, and set Fargate as the target. Update the Git servers to call the API Gateway endpoint.
## Analysis

### Understanding the Problem

The company needs to migrate its existing webhook architecture from **Amazon EC2 instances behind an Application Load Balancer (ALB)** to a **serverless architecture** with the **LEAST operational overhead**. The solution must be able to receive HTTP requests (webhooks) from an on-premises Git server. Operational overhead is minimized by choosing fully managed services that require the least amount of infrastructure management, patching, scaling configuration, and maintenance.

### Services Explanation

|**Service**|**Description**|**Operational Overhead/Use Case**|
|---|---|---|
|**AWS Lambda**|A serverless compute service that lets you run code without provisioning or managing servers.|**Lowest overhead** (pay-per-use, automatic scaling, no server management). Ideal for event-driven functions like webhooks.|
|**Lambda Function URL**|A dedicated HTTP(S) endpoint for an AWS Lambda function, providing a direct, simple way to invoke the function.|**Minimal overhead** (no extra gateway/load balancer management). Suitable for simple, single-function endpoints.|
|**Amazon API Gateway**|A fully managed service that makes it easy for developers to create, publish, maintain, monitor, and secure APIs at any scale. Comes in **REST API** and **HTTP API** types.|**Low overhead** (no server management). HTTP API is generally lower cost and lower latency than REST API. Great for organizing multiple webhooks under a single domain.|
|**AWS App Runner**|A fully managed service that makes it easy to quickly deploy containerized web applications and API services.|**Low overhead** (manages infrastructure, scaling, and load balancing), but still manages a service environment, not just a function. Typically for persistent web applications.|
|**AWS Fargate** (with Amazon ECS)|A serverless compute engine for Amazon ECS that lets you run containers without having to manage servers or clusters.|**Low-to-medium overhead** (no EC2 management, but requires managing ECS cluster configuration, task definitions, etc.). Best for containerized apps that require more control or longer processing times than Lambda.|
|**Application Load Balancer (ALB)**|Distributes incoming application traffic across multiple targets, such as EC2 instances, containers, or Lambda functions (indirectly).|**Medium overhead** (requires configuration and maintenance, though less than EC2). Using it with a serverless service like App Runner adds complexity/cost over direct access.|

---

### Correct Option(s) Explanation

The correct option is **B**.

**B. Create an Amazon API Gateway HTTP API. Implement each webhook logic in a separate AWS Lambda function. Update the Git servers to call the API Gateway endpoint.**

1. **Serverless:** Both **Amazon API Gateway** and **AWS Lambda** are fully managed, serverless services. Lambda is the quintessential serverless compute offering.
    
2. **Least Operational Overhead:** This combination is a standard, battle-tested pattern for building scalable, high-performance, and **low-overhead** APIs/webhooks on AWS.
    
    - **Lambda** eliminates server management entirely.
        
    - **API Gateway (HTTP API)** provides a single, unified, and performant endpoint for _all_ webhooks, eliminating the need to update the Git server for every _new_ webhook, and offering features like security and throttling. The HTTP API type is specifically chosen for its lower latency and cost compared to the REST API, minimizing overhead.
        
3. **Scalability and Performance:** This setup scales automatically to handle any volume of webhook requests.

---

### Incorrect Options Explanation

**A. For each webhook, create and configure an AWS Lambda function URL. Update the Git servers to call the individual Lambda function URLs.**

- **Reason for being less optimal (Higher Overhead):** While Lambda Function URLs are serverless and have minimal setup, they are best for _single-function_ applications. Having a separate URL for _each_ webhook logic becomes an administrative burden (higher operational overhead) as the number of webhooks grows, especially for updating the on-premises Git server configurations. Option B provides a single, centralized endpoint.
    

**C. Deploy the webhook logic to AWS App Runner. Create an ALB, and set App Runner as the target. Update the Git servers to call the ALB endpoint.**

- **Reason for being incorrect (Higher Overhead):** Although App Runner is serverless compute for web apps, adding an **Application Load Balancer (ALB)** in front of it significantly _increases_ operational overhead, complexity, and cost. App Runner already includes a managed load balancer/endpoint. The ALB is redundant and violates the goal of **LEAST operational overhead**.
    

**D. Containerize the webhook logic. Create an Amazon Elastic Container Service (Amazon ECS) cluster, and run the webhook logic in AWS Fargate. Create an Amazon API Gateway REST API, and set Fargate as the target. Update the Git servers to call the API Gateway endpoint.**

- **Reason for being incorrect (Higher Overhead):**
    
    - **Containerization/ECS/Fargate:** While Fargate is serverless _compute_, it requires managing container images, task definitions, and cluster configurations, which involves significantly **higher operational overhead** than using raw Lambda functions.
        
    - **API Gateway REST API:** The question asks for the **least operational overhead**. The **HTTP API** (used in option B) is generally the lower-overhead, lower-cost, and lower-latency choice for a simple proxy to Lambda/webhooks compared to the older **REST API** type.
