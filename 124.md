A solutions architect must create a business case for migration of a company's on-premises data center to the AWS Cloud. The solutions architect will use a configuration management database (CMDB) export of all the company's servers to create the case.  
  
Which solution will meet these requirements MOST cost-effectively?

- A. Use AWS Well-Architected Tool to import the CMDB data to perform an analysis and generate recommendations.
- B. Use Migration Evaluator to perform an analysis. Use the data import template to upload the data from the CMDB export.
- C. Implement resource matching rules. Use the CMDB export and the AWS Price List Bulk API to query CMDB data against AWS services in bulk.
- D. Use AWS Application Discovery Service to import the CMDB data to perform an analysis.

# Analysis of Cost-Effective Migration Business Case Creation with CMDB Data
Tags: #AWS-SAP-C02 #AWS-Migration-Evaluator #AWS-Well-Architected-Tool #AWS-Application-Discovery-Service #AWS-Price-List-Bulk-API #Cloud-Migration #CMDB #TCO-Analysis

## 1. Understanding the Problem
The core requirements are:
- Create a **business case** for migrating an on-premises data center to AWS. A migration business case typically includes total cost of ownership (TCO) comparison (on-prem vs. AWS), resource mapping, and migration feasibility/recommendations.
- Use an existing **CMDB (Configuration Management Database) export** (containing all company servers) as the data source—no need for additional resource discovery.
- The solution must be **most cost-effective** (minimize manual effort, avoid custom development, leverage managed AWS tools).

Key priority: Use a purpose-built AWS tool to convert CMDB data into actionable migration insights (TCO, resource matches) without unnecessary complexity or custom work.

## 2. Services Explanation
### Core Services Involved
| Service | Purpose & Key Features | Official Reference |
|---------|------------------------|--------------------|
| **AWS Migration Evaluator** (formerly AWS TCO Calculator Advanced) | Purpose-built tool for migration planning and business case development. Accepts imports of on-premises resource data (e.g., CMDB exports) via pre-built templates. Automatically maps on-prem servers to equivalent AWS resources, calculates TCO savings, and generates migration recommendations. Fully managed—no custom code required. | [Migration Evaluator Data Import](https://docs.aws.amazon.com/migration-evaluator/latest/userguide/data-import.html) |
| **AWS Well-Architected Tool** | Designed to review existing AWS or on-premises architectures against the AWS Well-Architected Framework (pillars: Operational Excellence, Security, Reliability, Performance Efficiency, Cost Optimization, Sustainability). It does **not** specialize in migration TCO analysis or CMDB data import for business case creation. | [Well-Architected Tool Purpose](https://docs.aws.amazon.com/wellarchitected/latest/userguide/what-is-well-architected-tool.html) |
| **AWS Price List Bulk API** | Provides raw, machine-readable AWS pricing data (e.g., EC2 instance costs, storage prices). It requires custom development to: 1) Define resource matching rules (map on-prem servers to AWS services), 2) Query pricing data in bulk, 3) Calculate TCO. No built-in migration business case capabilities. | [Price List Bulk API Docs](https://docs.aws.amazon.com/awsaccountbilling/latest/aboutv2/price-list-api.html) |
| **AWS Application Discovery Service** | Discovers on-premises resources (servers, applications, dependencies) via agent-based or agentless methods to collect data for migration planning. It is designed for **resource discovery** (when CMDB data is incomplete or unavailable), not for importing existing CMDB exports to generate a business case. | [Application Discovery Service Use Cases](https://docs.aws.amazon.com/application-discovery/latest/ag/what-is-appdiscovery.html) |

### Critical Migration Business Case Context
Per AWS Migration Best Practices, a cost-effective business case relies on **managed tools that automate TCO calculation and resource mapping** (avoiding custom work). Migration Evaluator is explicitly positioned by AWS as the tool for this scenario—integrating CMDB data to streamline business case development.

## 3. Correct Option Explanation (Option B)
### Why Option B is Correct
1. **Purpose-Built for the Use Case**: Migration Evaluator is AWS’s official tool for creating migration business cases using on-premises data (including CMDB exports). Its pre-built data import template directly supports CMDB data ingestion, eliminating the need for custom data formatting or tooling.
2. **Cost-Effectiveness**: As a fully managed service, it automates the most labor-intensive steps:
   - Mapping on-prem servers (from CMDB) to equivalent AWS resources (e.g., on-prem physical server → EC2 instance type, storage → S3/EBS).
   - Calculating TCO (on-prem costs vs. AWS costs, including savings from AWS pricing models like Reserved Instances or Savings Plans).
   - Generating actionable recommendations for migration (e.g., resource right-sizing, migration sequence).
   This avoids the time and cost of building custom solutions (e.g., Option C’s resource matching rules).
3. **Business Case Alignment**: Migration Evaluator’s output directly feeds into a migration business case—providing quantitative (cost savings) and qualitative (feasibility, risk) insights required to justify the migration.

### Official AWS Guidance Alignment
According to AWS’s [Migration Evaluator User Guide](https://docs.aws.amazon.com/migration-evaluator/latest/userguide/what-is-migration-evaluator.html), the tool’s primary use case is "helping you build a business case for cloud migration" by importing on-premises resource data (including CMDB exports) to deliver TCO analysis and migration plans.

## 4. Incorrect Options Explanation
### Option A
- **Critical Flaw**: The AWS Well-Architected Tool is not designed for migration business cases or CMDB data import. Its focus is on auditing existing architectures for alignment with Well-Architected principles, not on TCO comparison or resource mapping for migration.
- **Additional Flaw**: It cannot generate the cost-related insights (e.g., on-prem vs. AWS TCO) that are core to a migration business case—making it irrelevant to the requirement.

### Option C
- **Critical Flaw**: Using the AWS Price List Bulk API requires significant custom development:
  1. Designing and maintaining resource matching rules (e.g., correlating on-prem CPU/RAM/storage to AWS services)—prone to error and time-consuming.
  2. Building a system to query pricing data in bulk and calculate TCO.
  This is not cost-effective, as it requires engineering effort and ongoing maintenance (e.g., updating rules when AWS services/pricing change).
- **Additional Flaw**: The Price List API only provides raw pricing data—not migration recommendations or business case structure—leaving gaps in the required output.

### Option D
- **Critical Flaw**: AWS Application Discovery Service is for **discovering** on-premises resources, not for importing existing CMDB data. It is used when a company lacks a complete CMDB (e.g., to fill gaps in resource inventory), but the question specifies the company already has a CMDB export—making this tool redundant.
- **Additional Flaw**: Application Discovery Service does not generate TCO analysis or business case elements. It only collects resource data, which would still require integration with other tools (e.g., Price List API) to build a business case—adding unnecessary complexity.
