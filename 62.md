Example Corp. has an on-premises data center and a VPC named VPC A in the Example Corp. AWS account. The on-premises network connects to VPC A through an AWS Site-To-Site VPN. The on-premises servers can properly access VPC A. Example Corp. just acquired AnyCompany, which has a VPC named VPC B. There is no IP address overlap among these networks. Example Corp. has peered VPC A and VPC B.  
  
Example Corp. wants to connect from its on-premise servers to VPC B. Example Corp. has properly set up the network ACL and security groups.  
  
Which solution will meet this requirement with the LEAST operational effort?

- A. Create a transit gateway. Attach the Site-to-Site VPN, VPC A, and VPC B to the transit gateway. Update the transit gateway route tables for all networks to add IP range routes for all other networks.
- B. Create a transit gateway. Create a Site-to-Site VPN connection between the on-premises network and VPC B, and connect the VPN connection to the transit gateway. Add a route to direct traffic to the peered VPCs, and add an authorization rule to give clients access to the VPCs A and B.
- C. Update the route tables for the Site-to-Site VPN and both VPCs for all three networks. Configure BGP propagation for all three networks. Wait for up to 5 minutes for BGP propagation to finish.
- D. Modify the Site-to-Site VPN’s virtual private gateway definition to include VPC A and VPC B. Split the two routers of the virtual private getaway between the two VPCs.

## Understanding the Problem

The core requirement is to enable connectivity between an **on-premises data center** and a newly acquired **VPC B** (AnyCompany's network) with the **least operational effort**.

### Current State:

1. **On-premises Data Center** $\leftrightarrow$ **VPC A** via **AWS Site-to-Site VPN**.
    
2. **VPC A** $\leftrightarrow$ **VPC B** via **VPC Peering**.
    
3. **No IP address overlap** among the networks (On-Prem, VPC A, VPC B).
    
4. Connectivity is currently: On-Prem $\leftrightarrow$ VPC A.
    

### Goal:

Establish connectivity: **On-Prem** $\leftrightarrow$ **VPC B**.

### Key Constraint:

**Least operational effort.**

The primary challenge lies in the nature of **VPC Peering**, which is **non-transitive**. This means traffic from the on-premises network cannot inherently route through VPC A to reach VPC B, even though VPC A is peered with VPC B. A direct connection or a central routing hub is required.

## Services Explanation

| **Service**                       | **Description**                                                                                                                                            | **Relevance to Problem**                                                                                                                                  |
| --------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **AWS Site-to-Site VPN**          | A secure connection between your on-premises data center and your AWS VPC, using a **Virtual Private Gateway (VGW)** or an **AWS Transit Gateway (TGW)**.  | Used for the existing On-Prem $\leftrightarrow$ VPC A connection. The connection to VPC B needs to be established.                                        |
| **VPC Peering**                   | A networking connection between two VPCs that enables you to route traffic between them privately.                                                         | Used for the existing VPC A $\leftrightarrow$ VPC B connection. **Crucially, it is not transitive.**                                                      |
| **AWS Transit Gateway (TGW)**     | A network transit hub that you can use to interconnect your VPCs and on-premises networks. It simplifies network management and scales easily.             | Designed specifically to solve non-transitive routing problems and centralize network connections, making it ideal for connecting multiple VPCs and VPNs. |
| **Virtual Private Gateway (VGW)** | A logical connection object attached to a single VPC, used as the termination point for VPN connections from the on-premises network to that specific VPC. | Used for the existing VPN connection to VPC A. **A single VGW can only be attached to one VPC.**                                                          |

## Correct Option(s) Explanation

The correct solution is **A**.

**A. Create a transit gateway. Attach the Site-to-Site VPN, VPC A, and VPC B to the transit gateway. Update the transit gateway route tables for all networks to add IP range routes for all other networks.**

- **Transit Gateway (TGW) as the Hub:** TGW is the most efficient and scalable solution for connecting multiple VPCs and on-premises networks, directly solving the non-transitivity issue of VPC Peering.
    
- **Minimal Effort:** By migrating the existing Site-to-Site VPN connection _from_ the Virtual Private Gateway (VGW) _to_ the Transit Gateway, and attaching both VPC A and VPC B to the TGW, a full mesh network is created via a single, central hub.
    
    - The existing VPN can be re-associated with the TGW (or recreated with the same Customer Gateway), and new routes are configured only on the TGW's route tables and the VPC route tables pointing to the TGW. This is significantly less complex than managing multiple point-to-point connections.
        
- **Connectivity:**
    
    - On-Prem $\leftrightarrow$ TGW $\leftrightarrow$ VPC A
        
    - On-Prem $\leftrightarrow$ TGW $\leftrightarrow$ VPC B
        
    - VPC A $\leftrightarrow$ TGW $\leftrightarrow$ VPC B
        
- This approach is the **AWS-recommended best practice** for this scenario, as documented in the **AWS Transit Gateway Documentation**, which emphasizes its ability to simplify complex many-to-many connectivity. It requires fewer individual connection setups and route table updates compared to creating a new separate VPN or modifying the existing VGW configuration for non-transitive routing.

## Incorrect Options Explanation

**B. Create a transit gateway. Create a Site-to-Site VPN connection between the on-premises network and VPC B, and connect the VPN connection to the transit gateway. Add a route to direct traffic to the peered VPCs, and add an authorization rule to give clients access to the VPCs A and B.**

- **Error:** This approach suggests creating a _second_ VPN connection directly between On-Prem and VPC B, then attaching it to a TGW, and _also_ mentions the old "peered VPCs" and "authorization rule."
    
    - If a TGW is used, **VPC Peering** is typically replaced or superseded by TGW attachments for transitive routing.
        
    - It creates a **second VPN connection** where Option A reuses/migrates the existing one and centralizes all traffic through the TGW, which is more operationally complex (more connections to manage).
        
    - The "authorization rule" terminology is incorrect for TGW networking. TGW uses route tables and association/propagation.
        

**C. Update the route tables for the Site-to-Site VPN and both VPCs for all three networks. Configure BGP propagation for all three networks. Wait for up to 5 minutes for BGP propagation to finish.**

- **Error: Non-transitive Peering:** This solution ignores the fundamental limitation of **VPC Peering**: it is **non-transitive**. Simply adding a route in VPC A's route table to direct On-Prem traffic to VPC B via the Peering Connection _will not work_. The traffic must be directly routed, or a central hub like TGW must be used.
    
    - Attempting to route traffic _through_ VPC A to VPC B is explicitly blocked by the VPC Peering design.
        
    - This would require complex and unsupported configurations and violates the "least operational effort" constraint.
        

**D. Modify the Site-to-Site VPN’s virtual private gateway definition to include VPC A and VPC B. Split the two routers of the virtual private getaway between the two VPCs.**

- **Error: VGW Limitation:** A **Virtual Private Gateway (VGW)** can only be attached to a **single VPC** at any given time.
    
    - It is physically and logically impossible to "modify the VGW definition to include VPC A and VPC B" or "split the two routers of the virtual private gateway between the two VPCs." This option describes a non-existent or unsupported AWS feature.
        
    - VGWs do not support transitive routing between attached networks, necessitating the use of **Transit Gateway** for multi-VPC/On-Prem connectivity.
