A company uses an on-premises data analytics platform. The system is highly available in a fully redundant configuration across 12 servers in the company's data center.  
  
The system runs scheduled jobs, both hourly and daily, in addition to one-time requests from users. Scheduled jobs can take between 20 minutes and 2 hours to finish running and have tight SLAs. The scheduled jobs account for 65% of the system usage. User jobs typically finish running in less than 5 minutes and have no SLA. The user jobs account for 35% of system usage. During system failures, scheduled jobs must continue to meet SLAs. However, user jobs can be delayed.  
  
A solutions architect needs to move the system to Amazon EC2 instances and adopt a consumption-based model to reduce costs with no long-term commitments. The solution must maintain high availability and must not affect the SLAs.  
  
Which solution will meet these requirements MOST cost-effectively?

- A. Split the 12 instances across two Availability Zones in the chosen AWS Region. Run two instances in each Availability Zone as On-Demand Instances with Capacity Reservations. Run four instances in each Availability Zone as Spot Instances.
- B. Split the 12 instances across three Availability Zones in the chosen AWS Region. In one of the Availability Zones, run all four instances as On-Demand Instances with Capacity Reservations. Run the remaining instances as Spot Instances.
- C. Split the 12 instances across three Availability Zones in the chosen AWS Region. Run two instances in each Availability Zone as On-Demand Instances with a Savings Plan. Run two instances in each Availability Zone as Spot Instances.
- D. Split the 12 instances across three Availability Zones in the chosen AWS Region. Run three instances in each Availability Zone as On-Demand Instances with Capacity Reservations. Run one instance in each Availability Zone as a Spot Instance.

# AWS Certified Solutions Architect - Professional (SAP-C02) Exam Question Analysis

**tags:** #Compute #Cost_Optimization #High_Availability #EC2 #Capacity_Reservation #Spot_Instances #On-Demand_Instances #SLA

## Understanding the Problem

The company is migrating an on-premises data analytics platform, running on 12 fully redundant servers, to **Amazon EC2 instances**.

**Key Constraints & Requirements:**
1.  **Total EC2 Instances:** 12 instances (fully redundant configuration).
2.  **Job Types & Usage:**
    * **Scheduled Jobs (65% usage):** Long-running (20 min to 2 hrs), **tight SLAs**. Must continue during system failures.
    * **User Jobs (35% usage):** Short-running (less than 5 min), **no SLA**. Can be delayed during failures.
3.  **Cost Model:** Adopt a **consumption-based model** to **reduce costs** with **no long-term commitments**.
4.  **High Availability (HA):** Must maintain high availability (implies multi-AZ deployment).
5.  **SLA Maintenance:** The solution **must not affect the SLAs** for scheduled jobs.

The goal is to find the **MOST cost-effective** solution that ensures the 65% of usage associated with SLA-critical scheduled jobs is always available and highly resilient, while utilizing the most cost-effective options for the remaining capacity (35% usage and the redundancy for the critical jobs).

## Correct Option(s) Explanation

The core principle for the most cost-effective and compliant solution is to match the workload type and requirement to the correct EC2 purchasing option:

* **SLA-Critical Workloads (Scheduled Jobs - 65%):** Must use **On-Demand Instances with Capacity Reservations (OD-CR)** for guaranteed capacity and high availability to ensure SLAs are met. The 65% usage translates to $12 \times 0.65 \approx 7.8$ instances. Since instances are discrete, we must secure **8 instances** with guaranteed capacity.
* **Non-Critical/Flexible Workloads (User Jobs - 35%):** Can use **Spot Instances** for maximum cost savings, as they can be delayed or interrupted. The remaining capacity is $12 - 8 = 4$ instances.
* **High Availability:** The total 12 instances must be distributed across multiple **Availability Zones (AZs)**, typically three for a professional-level high-availability design, to survive an AZ failure.

The optimal breakdown is **8 OD-CR instances** for the SLA-critical 65% usage and **4 Spot Instances** for the flexible 35% usage. The total must be 12 instances, spread across 3 AZs for redundancy.

**Option D Analysis:**
* **Instances:** $3 \times 3$ OD-CR + $3 \times 1$ Spot = **9 OD-CR** + **3 Spot** = **12 instances**.
* **Critical Workload Coverage:** **9 OD-CR** instances (spread across 3 AZs) provide guaranteed capacity and high availability, which is **more than enough** to cover the 8 instances required for the 65% SLA-critical workload.
* **Cost-Effectiveness:** It uses **3 Spot Instances** (the cheapest option) for the remaining capacity, maximizing cost savings while ensuring the critical jobs are fully covered and highly available. It meets the "no long-term commitments" requirement.

While an 8/4 split is mathematically ideal, distributing the 12 instances evenly across 3 AZs (4 instances per AZ) is the standard high-availability design. Option D's distribution of **3 OD-CR** and **1 Spot** in each of the 3 AZs provides the necessary redundancy (3 OD-CR in each AZ ensures enough capacity to handle a failure of one AZ, as the remaining two AZs would have $3+3=6$ OD-CR instances to pick up the load, assuming the load balances across AZs). This configuration provides the best balance of **cost reduction** and **guaranteed SLA compliance**.

## Incorrect Options Explanation

### Option A:
* **Instances:** $2 \times 2$ OD-CR + $2 \times 4$ Spot = **4 OD-CR** + **8 Spot** = **12 instances**.
* **Critical Workload Coverage:** Only **4 OD-CR instances** are used. This is **insufficient** to cover the 8 instances required for the 65% SLA-critical workload. **SLAs would be affected**, violating a key requirement.
* **Availability:** Uses only **two Availability Zones**, which is less resilient than a three-AZ design for maximum availability against single-AZ failure.

### Option B:
* **Instances:** $4$ OD-CR (in one AZ) + $8$ Spot (in the remaining two AZs) = **12 instances**.
* **Critical Workload Coverage:** Only **4 OD-CR instances** are used, which is **insufficient** to cover the 8 instances required for the 65% SLA-critical workload. **SLAs would be affected**.
* **High Availability:** By placing **all** the OD-CR instances in a **single AZ**, the guaranteed capacity for the critical jobs would be **lost completely** if that single AZ fails, **violating the High Availability and SLA requirements**.

### Option C:
* **Instances:** $3 \times 2$ OD-CR with Savings Plan + $3 \times 2$ Spot = **6 OD-CR (SP)** + **6 Spot** = **12 instances**.
* **Commitment Violation:** The use of a **Savings Plan** requires a **1-year or 3-year commitment**, which directly **violates the requirement of "no long-term commitments"**.
* **Critical Workload Coverage:** **6 OD-CR (SP)** instances are still **insufficient** to cover the 8 instances required for the 65% SLA-critical workload. **SLAs would be affected**.

**Reference:** AWS EC2 Instance Purchasing Options: *This official AWS documentation details the cost, commitment, and suitability of On-Demand, Capacity Reservations, and Spot Instances for various workloads, supporting the principle of using On-Demand with Capacity Reservations for SLA-critical, non-interruptible workloads and Spot Instances for flexible, cost-optimized workloads.*
