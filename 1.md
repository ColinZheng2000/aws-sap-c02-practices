A company needs to architect a hybrid DNS solution. This solution will use an Amazon Route 53 private hosted zone for the domain cloud.example.com for the resources stored within VPCs.  
✑ The company has the following DNS resolution requirements:  
✑ On-premises systems should be able to resolve and connect to cloud.example.com.  
All VPCs should be able to resolve cloud.example.com.  
There is already an AWS Direct Connect connection between the on-premises corporate network and AWS Transit Gateway.  
Which architecture should the company use to meet these requirements with the HIGHEST performance?  

- A. Associate the private hosted zone to all the VPCs. Create a Route 53 inbound resolver in the shared services VPC. Attach all VPCs to the transit gateway and create forwarding rules in the on-premises DNS server for cloud.example.com that point to the inbound resolver.
- B. Associate the private hosted zone to all the VPCs. Deploy an Amazon EC2 conditional forwarder in the shared services VPC. Attach all VPCs to the transit gateway and create forwarding rules in the on-premises DNS server for cloud.example.com that point to the conditional forwarder.
- C. Associate the private hosted zone to the shared services VPC. Create a Route 53 outbound resolver in the shared services VPC. Attach all VPCs to the transit gateway and create forwarding rules in the on-premises DNS server for cloud.example.com that point to the outbound resolver.
- D. Associate the private hosted zone to the shared services VPC. Create a Route 53 inbound resolver in the shared services VPC. Attach the shared services VPC to the transit gateway and create forwarding rules in the on-premises DNS server for cloud.example.com that point to the inbound resolver.

### Understanding the Problem

The core task is to design a high-performance hybrid DNS architecture. The company has a private domain, `cloud.example.com`, managed by a Route 53 private hosted zone (PHZ). The solution must satisfy two key requirements:
1.  **On-premises to AWS:** Systems in the on-premises data center must be able to resolve DNS names within `cloud.example.com`.
2.  **AWS to AWS:** All Virtual Private Clouds (VPCs) in AWS must also be able to resolve names in `cloud.example.com`.

The architecture must use an existing AWS Direct Connect and AWS Transit Gateway connection and prioritize the **highest performance**.

***

### Services Explanation

* **Amazon Route 53 Private Hosted Zone (PHZ)**: A PHZ is a container for DNS records (like A, CNAME, etc.) for a specific domain (e.g., `cloud.example.com`) that you want to manage within AWS. Unlike a public hosted zone, its records are only resolvable from within the VPCs you explicitly associate it with. Each VPC has a built-in DNS resolver (at the `.2` address of the VPC CIDR) that can resolve records in associated PHZs.

* **Amazon Route 53 Resolver:** This is a set of features that enables DNS resolution across hybrid environments. It is a highly available, managed service.
    * **Inbound Resolver Endpoint:** This allows DNS queries from your on-premises network (or any external network) to be forwarded to your VPCs. You place elastic network interfaces (ENIs) in a VPC, and your on-premises DNS server forwards queries for your private AWS domains to the IP addresses of these ENIs.
    * **Outbound Resolver Endpoint:** This allows DNS queries from within your VPCs to be forwarded to your on-premises network. This is used when your EC2 instances need to resolve names of on-premises servers (e.g., `server1.corp.example.com`).

* **AWS Transit Gateway:** This service acts as a cloud router and a central hub for connecting your VPCs and on-premises networks. It simplifies network management by operating on a hub-and-spoke model, avoiding complex VPC peering meshes.



***

### Analysis of the Correct Option

#### A. Associate the private hosted zone to all the VPCs. Create a Route 53 inbound resolver in the shared services VPC. Attach all VPCs to the transit gateway and create forwarding rules in the on-premises DNS server for cloud.example.com that point to the inbound resolver.

This option provides the most direct and highest-performing solution that meets all requirements:

* **Resolving from VPCs:** By **associating the private hosted zone with all VPCs**, each VPC's internal DNS resolver (`.2` resolver) can resolve `cloud.example.com` queries natively and with the lowest possible latency. This is the simplest and most performant method for intra-AWS resolution.
* **Resolving from On-Premises:** Creating a **Route 53 inbound resolver** is the purpose-built, managed AWS solution for this exact use case. The on-premises DNS server is configured with a conditional forwarding rule for `cloud.example.com` that points to the IP addresses of the inbound resolver's ENIs. Queries travel securely and efficiently over the Direct Connect and Transit Gateway to the inbound resolver, which then resolves the name using the private hosted zone.
* **Performance:** Using managed services like Route 53 Resolver endpoints offers high availability and scalability without the operational overhead or potential performance bottlenecks of a self-managed EC2 solution.

As stated in the AWS documentation, "To resolve DNS queries from a remote network to a private hosted zone, you must create a Route 53 Resolver inbound endpoint." This architecture perfectly aligns with AWS best practices for hybrid DNS.

***

### Analysis of the Incorrect Options

#### B. Associate the private hosted zone to all the VPCs. Deploy an Amazon EC2 conditional forwarder in the shared services VPC...

This architecture would work, but it fails on the **"HIGHEST performance"** requirement. A self-managed DNS forwarder on an EC2 instance introduces a single point of failure (unless you build a complex, highly available cluster) and becomes a potential performance bottleneck. You are responsible for patching, scaling, and managing the instance and the DNS software. The managed Route 53 Resolver service is designed for high performance and availability, making it the superior choice.

#### C. Associate the private hosted zone to the shared services VPC. Create a Route 53 outbound resolver in the shared services VPC...

This option is incorrect for two primary reasons:
1.  **Wrong Resolver Type:** It specifies an **outbound resolver**. Outbound resolvers are for sending DNS queries *from* a VPC *to* an on-premises network. The requirement is the opposite: to receive queries *from* on-premises. An inbound resolver is needed.
2.  **Inefficient PHZ Association:** Associating the PHZ with only the shared services VPC means other VPCs cannot resolve `cloud.example.com` directly. They would need complex forwarding rules to send their queries to the shared services VPC, adding latency and administrative overhead.

#### D. Associate the private hosted zone to the shared services VPC. Create a Route 53 inbound resolver in the shared services VPC...

This option correctly identifies the need for an **inbound resolver**. However, it fails on the VPC resolution requirement by **associating the PHZ only with the shared services VPC**. As explained for option C, this prevents other VPCs from resolving the domain natively. While it's possible to create Route 53 Resolver rules to forward queries from other VPCs to the shared services VPC, this adds an unnecessary network hop and configuration complexity. The solution in option A, which associates the PHZ directly with all VPCs, is far more direct and performant for DNS resolution within AWS.
