A global company has a mobile app that displays ticket barcodes. Customers use the tickets on the mobile app to attend live events. Event scanners read the ticket barcodes and call a backend API to validate the barcode data against data in a database. After the barcode is scanned, the backend logic writes to the database's single table to mark the barcode as used.

The company needs to deploy the app on AWS with a DNS name of api.example.com. The company will host the database in three AWS Regions around the world.

Which solution will meet these requirements with the LOWEST latency?

A. Host the database on Amazon Aurora global database clusters. Host the backend on three Amazon Elastic Container Service (Amazon ECS) clusters that are in the same Regions as the database. Create an accelerator in AWS Global Accelerator to route requests to the nearest ECS cluster. Create an Amazon Route 53 record that maps api.example.com to the accelerator endpoint
B. Host the database on Amazon Aurora global database clusters. Host the backend on three Amazon Elastic Kubernetes Service (Amazon EKS) clusters that are in the same Regions as the database. Create an Amazon CloudFront distribution with the three clusters as origins. Route requests to the nearest EKS cluster. Create an Amazon Route 53 record that maps api.example.com to the CloudFront distribution.
C. Host the database on Amazon DynamoDB global tables. Create an Amazon CloudFront distribution. Associate the CloudFront distribution with a CloudFront function that contains the backend logic to validate the barcodes. Create an Amazon Route 53 record that maps api.example.com to the CloudFront distribution.
D. Host the database on Amazon DynamoDB global tables. Create an Amazon CloudFront distribution. Associate the CloudFront distribution with a Lambda@Edge function that contains the backend logic to validate the barcodes. Create an Amazon Route 53 record that maps api.example.com to the CloudFront distribution.



# AWS SAP-C02 Question Analysis — Q342

## 📋 Analysis

### 1. Understanding the Problem

The company is building a globally distributed ticket validation system for live events. Barcode scanners around the world must validate tickets with extremely low response time.

The architecture requirements imply:

- Global users and geographically distributed traffic
- Very latency-sensitive API calls
- A write operation immediately after validation
- Multi-Region database deployment
- A single global DNS endpoint (`api.example.com`)
- High scalability during event spikes

The most important design objective is "LOWEST latency".

Therefore, the ideal solution should:
- Execute logic as close to users as possible
- Avoid cross-Region database writes
- Support active-active global data access
- Minimize infrastructure layers
- Reduce operational overhead

This is a classic edge-computing + globally replicated datastore scenario.

- --

### 2. Options Analysis

#### ✅ Option D (CORRECT): DynamoDB global tables with Lambda@Edge for edge-based validation

This is the best architecture because it combines:
- Edge compute
- Global active-active database replication
- Fully managed serverless services

Why this solution provides the lowest latency:

- DynamoDB global tables replicate data across multiple AWS Regions automatically.
  - Each Region supports local read and write operations
  - Barcode validation requests can use nearby regional replicas
  - Replication latency is typically very low

- Lambda@Edge executes backend validation logic at CloudFront edge locations.
  - Requests are processed closer to barcode scanners
  - Reduces internet round-trip latency significantly
  - Avoids sending every request to centralized regional compute clusters

- CloudFront provides AWS global edge network acceleration.
  - Traffic enters the AWS backbone quickly
  - Improves global responsiveness

- Route 53 maps the custom domain to CloudFront globally.

This design is operationally efficient:
- No container orchestration
- No EC2 management
- Automatic scaling
- Built-in high availability

Most importantly:
- Lambda@Edge can access DynamoDB using AWS SDK/API calls
- This enables real backend validation logic at the edge

This is the key difference compared with CloudFront Functions.

#### ❌ Option A (INCORRECT): ECS with Aurora Global Database and Global Accelerator

This is a strong architecture, but not the lowest-latency solution.

Why it falls short:

- ECS workloads are still Region-based.
  - Requests must travel from users to regional ECS clusters
  - Compute is not executed at edge locations

- Aurora Global Database is optimized mainly for:
  - Fast cross-Region reads
  - Disaster recovery
  - Read scaling

However:
- Writes occur primarily in one writer Region
- Global write latency becomes higher
- This is not ideal for real-time ticket redemption systems

Additionally:
- ECS introduces operational overhead:
  - Cluster scaling
  - Capacity planning
  - Container lifecycle management

AWS Global Accelerator improves network path selection, but:
- It does not provide edge compute
- The backend processing still occurs in Regions

This architecture is better suited for:
- Traditional APIs
- Stateful microservices
- Relational transactional systems

Not optimal for ultra-low-latency edge validation.

Official reference:
- Aurora Global Database:
  https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/aurora-global-database.html

- --

#### ❌ Option B (INCORRECT): EKS with CloudFront and Aurora Global Database

This option adds unnecessary complexity and still does not achieve the lowest latency.

Problems include:

- EKS clusters remain Region-based compute.
- Kubernetes operational complexity is high:
  - Node management
  - Cluster upgrades
  - Networking configuration
  - Autoscaling tuning

CloudFront does not magically turn EKS into edge compute.

Additional issue:
- Aurora still has centralized write behavior compared to DynamoDB global tables.

The workload itself is relatively simple:
- Validate ticket
- Mark ticket as used

This does not justify Kubernetes infrastructure.

This becomes an overengineered solution.

- --

#### ❌ Option C (INCORRECT): CloudFront Functions for backend validation

This is the most important trap in the question.

CloudFront Functions are intentionally lightweight.

They support:
- Header manipulation
- URL rewrites
- Request filtering
- Lightweight authentication logic

But they do NOT support:
- AWS SDK access
- Database connectivity
- Network calls to DynamoDB
- Complex backend processing

Therefore:
- Barcode validation against DynamoDB cannot be implemented properly using CloudFront Functions.

Lambda@Edge is specifically designed for advanced edge compute scenarios requiring backend integration.

This distinction is heavily tested in SAP-C02.

Official reference:
- CloudFront Functions limitations:
  https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/cloudfront-functions.html

### 3. Hands-On Knowledge

#### DynamoDB Global Tables Real-World Design

For ticketing systems, a common table design is:

```text
PK = EVENT#
SK = TICKET#
```

Benefits:
- Fast point lookups
- Good partition distribution
- Efficient scaling during large events

Avoid:
- Sequential IDs as partition keys
- Hot partitions during mass scanning

- --

#### Prevent Double Ticket Redemption

In real deployments, scanners may retry requests because of poor venue connectivity.

Use DynamoDB conditional writes:

```bash
ConditionExpression:
attribute_not_exists(redeemedAt)
```

This guarantees:
- Idempotency
- No duplicate redemption
- Safe concurrent processing

This is a common enterprise anti-fraud pattern.

Official reference:
- Conditional writes:
  https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Expressions.ConditionExpressions.html

#### Lambda@Edge Operational Constraints

Frequently asked in interviews:

- Lambda@Edge functions must be deployed in us-east-1
- Only published versions are supported
- No VPC attachment
- Deployment propagation takes time globally

Optimization tips:
- Keep deployment packages small
- Minimize cold starts
- Avoid heavy libraries

- --

#### Why DynamoDB Is Better Than Aurora Here

DynamoDB advantages:
- Single-digit millisecond latency
- Native multi-Region active-active
- Massive automatic scaling
- Serverless operations

Aurora advantages:
- SQL joins
- Complex relational queries
- Transaction-heavy applications

This workload is fundamentally:
- Key-value validation
- Simple write update
- Globally distributed

Therefore DynamoDB is the better fit.

1. "Why is Lambda@Edge more suitable than CloudFront Functions in this scenario?"
   → 回答要点 (Key Answer Points)：Lambda@Edge supports AWS SDK access, external service integration, and more advanced compute logic. CloudFront Functions are limited to lightweight request/response manipulation and cannot access DynamoDB directly.

2. "What are the trade-offs of DynamoDB global tables?"
   → 回答要点 (Key Answer Points)：DynamoDB global tables provide low-latency active-active replication, but cross-Region consistency is eventually consistent. Architects must handle replication conflicts and understand that simultaneous writes across Regions can create conflict resolution scenarios.
