A company has an organization that has many AWS accounts in AWS Organizations. A solutions architect must improve how the company manages common security group rules for the AWS accounts in the organization.  
The company has a common set of IP CIDR ranges in an allow list in each AWS account to allow access to and from the company's on-premises network.  
Developers within each account are responsible for adding new IP CIDR ranges to their security groups. The security team has its own AWS account. Currently, the security team notifies the owners of the other AWS accounts when changes are made to the allow list.  
The solutions architect must design a solution that distributes the common set of CIDR ranges across all accounts.  
Which solution meets these requirements with the LEAST amount of operational overhead?  

- A. Set up an Amazon Simple Notification Service (Amazon SNS) topic in the security team's AWS account. Deploy an AWS Lambda function in each AWS account. Configure the Lambda function to run every time an SNS topic receives a message. Configure the Lambda function to take an IP address as input and add it to a list of security groups in the account. Instruct the security team to distribute changes by publishing messages to its SNS topic.
- B. Create new customer-managed prefix lists in each AWS account within the organization. Populate the prefix lists in each account with all internal CIDR ranges. Notify the owner of each AWS account to allow the new customer-managed prefix list IDs in their accounts in their security groups. Instruct the security team to share updates with each AWS account owner.
- C. Create a new customer-managed prefix list in the security team's AWS account. Populate the customer-managed prefix list with all internal CIDR ranges. Share the customer-managed prefix list with the organization by using AWS Resource Access Manager. Notify the owner of each AWS account to allow the new customer-managed prefix list ID in their security groups.
- D. Create an IAM role in each account in the organization. Grant permissions to update security groups. Deploy an AWS Lambda function in the security team's AWS account. Configure the Lambda function to take a list of internal IP addresses as input, assume a role in each organization account, and add the list of IP addresses to the security groups in each account.

## The Question

**Scenario:**
A company uses AWS Organizations with many accounts. They need to allow access to/from on-premises networks using a common set of IP CIDR ranges across all accounts. Developers manage their own Security Groups (SGs), but the Security Team (in a separate account) currently notifies them manually when IPs change.

**Goal:**
Design a solution to distribute common CIDR ranges to all accounts with the **LEAST operational overhead**.

**Options:**
A. Set up SNS in the security account. Deploy Lambda in each account to listen to SNS and update SGs.
B. Create prefix lists in each account. Manually populate them. Notify owners to update SGs.
C. Create a customer-managed prefix list in the security account. Share it via AWS RAM. Notify owners to reference the shared list.
D. Create IAM roles in each account. Deploy a central Lambda in the security account to assume roles and update SGs remotely.

---

## Answer
**Correct Option:** **C**

---

## Analysis

### 1. Understanding the Problem
The core challenge here is **Distributed Configuration Management** in a multi-account environment.
* **The Requirement:** A dynamic list of IP addresses (allow list) needs to be synchronized across hundreds of Security Groups in different accounts.
* **The Constraint:** "Least operational overhead." This effectively rules out solutions that require maintaining custom code (Lambda), complex state management, or manual intervention for every update.
* **The Conflict:** Developers own the Security Groups, but Security owns the IP data. We need a way for Security to update the data without needing permission to touch the Developers' specific SG rules every time.

### 2. Services Explanation

* **Customer-managed Prefix Lists:**
    A managed resource that allows you to group multiple CIDR blocks into a single object. You can reference this Prefix List ID (e.g., `pl-12345678`) in Security Group rules and Route Tables just like you would a single IP address.
    

* **AWS Resource Access Manager (RAM):**
    A service that enables you to share AWS resources (like Subnets, Transit Gateways, and **Prefix Lists**) with other AWS accounts or your entire AWS Organization. When a resource is shared, it appears in the consumer account as if it were a local resource.

* **AWS Organizations:**
    Provides the hierarchy and boundaries for the accounts, allowing RAM to share resources with the entire organization ID rather than inviting accounts one by one.

### 3. Correct Option Explanation

**Option C: Create a new customer-managed prefix list in the security team's AWS account... Share... using AWS Resource Access Manager.**

This is the definitive "Cloud Native" solution for this pattern.

1.  **Central Management:** The Security Team creates the Prefix List *once* in their administrative account. They populate it with the on-premise CIDRs.
2.  **Zero-Touch Distribution:** By using AWS RAM, they share this Prefix List with the AWS Organization. It immediately becomes visible to all other accounts.
3.  **Decoupled Operations:** Developers simply add a **one-time** rule to their Security Groups allowing traffic from `pl-xxxxxxx`.
4.  **Updates:** When the Security Team adds a new IP to the Prefix List in the central account, **AWS automatically propagates this change** to every Security Group in the entire organization that references that list. No code runs, no notifications are needed, and propagation is near-instant.

This represents the lowest operational overhead because the "mechanism" of updating is handled entirely by the AWS platform, not by user-managed scripts.

### 4. Incorrect Options Explanation

**Option A (SNS + Lambda in every account):**
* **Why it fails:** This is an "Event-Driven" approach to a configuration problem. It requires deploying and maintaining a Lambda function in *every* current and future account. If the Lambda fails, security groups become out of sync. This creates a massive maintenance burden (high operational overhead).

**Option B (Prefix lists in each account + Manual sync):**
* **Why it fails:** While it uses Prefix Lists, it creates them locally in every account. This violates the "Don't Repeat Yourself" (DRY) principle. The Security Team still has to notify owners or manually update hundreds of lists whenever an IP changes. It does not solve the "distribution" problem.

**Option D (Central Lambda + IAM Roles):**
* **Why it fails:** This is the "Scripting" approach.
    * It requires setting up IAM Trust Relationships in every target account.
    * The Lambda logic is complex (looping through accounts, handling errors, avoiding API throttling).
    * **Infrastructure as Code (IaC) Drift:** If developers manage their SGs using Terraform or CloudFormation, an external Lambda modifying those SGs will cause state drift, leading to deployment failures or changes being reverted automatically.
