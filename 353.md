A company wants to migrate an Amazon Aurora MySQL DB cluster from an existing AWS account to a new AWS account in the same AWS Region. Both accounts are members of the same organization in AWS Organizations.

The company must minimize database service interruption before the company performs DNS cutover to the new database.

Which migration strategy will meet this requirement? (Choose two.)

A. Take a snapshot of the existing Aurora database. Share the snapshot with the new AWS account. Create an Aurora DB cluster in the new account from the snapshot.
B. Create an Aurora DB cluster in the new AWS account. Use AWS Database Migration Service (AWS DMS) to migrate data between the two Aurora DB clusters.
C. Use AWS Backup to share an Aurora database backup from the existing AWS account to the new AWS account. Create an Aurora DB cluster in the new AWS account from the snapshot.
D. Create an Aurora DB cluster in the new AWS account. Use AWS Application Migration Service to migrate data between the two Aurora DB clusters.




# AWS SAP-C02 Question Analysis — Q353

## 📋 Analysis

### 1. Understanding the Problem

The scenario focuses on migrating an Amazon Aurora MySQL cluster from one AWS account to another AWS account within the same AWS Region.

Key requirements:

- Cross-account migration
- Same AWS Region
- Both accounts belong to the same AWS Organization
- Minimize service interruption during DNS cutover
- Migration should keep the target database as current as possible before final switchover

The most important phrase is “minimize database service interruption.”  
This means the company should avoid a long offline restore-only migration strategy and instead use continuous replication/synchronization until cutover time.

This question tests:
- Aurora migration strategies
- Cross-account snapshot sharing
- AWS DMS continuous replication capability
- Difference between backup/restore and near-zero-downtime migration

---

### 2. Options Analysis

#### ✅ Option A (CORRECT): Use Aurora snapshot sharing across accounts
This option is valid because Aurora snapshots can be shared with another AWS account in the same Region.

Why this works:
- Provides an initial baseline copy of the database
- Native Aurora capability
- Efficient way to bootstrap the target cluster
- Common enterprise migration pattern for cross-account database movement

However:
- Snapshot restore alone does NOT minimize downtime sufficiently because the restored DB becomes stale immediately after snapshot creation.
- Therefore, snapshot sharing is usually combined with replication technology such as AWS DMS.

Why still correct:
- The question asks for TWO actions forming the migration strategy.
- Snapshot sharing is commonly used as the initial load stage.

Official reference:
- Amazon Aurora User Guide — Sharing DB cluster snapshots
- https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/USER_ShareSnapshot.html

---

#### ✅ Option B (CORRECT): Use AWS DMS for ongoing replication
AWS DMS is the critical service that minimizes interruption.

Why this is the best fit:
- Supports homogeneous migrations such as Aurora MySQL → Aurora MySQL
- Supports ongoing replication using Change Data Capture (CDC)
- Keeps source and target synchronized until cutover
- Enables near-minimal downtime migrations

Typical enterprise sequence:
1. Create target Aurora cluster
2. Load initial data
3. Enable CDC replication with DMS
4. Wait for replication lag near zero
5. Stop writes briefly
6. Final sync
7. DNS cutover

This is the standard AWS-recommended migration approach for low-downtime database migrations.

Official references:
- AWS DMS User Guide
- https://docs.aws.amazon.com/dms/latest/userguide/Welcome.html
- AWS Prescriptive Guidance for Aurora migrations

---

#### ❌ Option C (INCORRECT): AWS Backup sharing alone does not minimize downtime
AWS Backup can indeed share backups cross-account.

However:
- This is fundamentally still a backup-and-restore workflow
- No continuous replication mechanism exists in this option
- Restored database becomes outdated after backup creation
- Cutover downtime would increase because application writes continue on source DB

Important exam distinction:
- Backup/restore = migration with downtime
- DMS CDC = migration with minimal downtime

AWS Backup is more commonly associated with:
- Centralized governance
- Compliance retention
- Disaster recovery
- Backup administration across accounts

Not low-downtime database migration.

---

#### ❌ Option D (INCORRECT): AWS Application Migration Service is for servers, not databases
AWS Application Migration Service (MGN) replicates block-level server volumes for lift-and-shift server migration.

Why incorrect:
- Not designed for Aurora database replication
- Cannot perform logical database replication or CDC
- Not a database migration tool

This is a common SAP-C02 trap:
- MGN → server migration
- DMS → database migration

Official reference:
- AWS Application Migration Service documentation
- Designed for physical/virtual server migration into EC2

### 3. Hands-On Knowledge

In real enterprise Aurora migrations, the most common low-downtime pattern is:

#### Typical Enterprise Migration Flow

1. Create target Aurora cluster
2. Initialize data
   - Snapshot restore OR DMS full load
3. Configure AWS DMS CDC
4. Monitor replication lag
5. Freeze writes briefly during cutover
6. Update Route 53 / application connection strings
7. Validate application
8. Decommission old cluster later

---

#### Important DMS Design Considerations

##### Replication Instance Sizing
Under-sizing DMS replication instances causes:
- Replication lag
- CDC latency
- Task failures

For production:
- Use Multi-AZ replication instance
- Enable CloudWatch monitoring
- Monitor:
  - CDCLatencySource
  - CDCLatencyTarget

---

#### Aurora Snapshot Sharing Gotchas

Encrypted snapshots require:
- Shared KMS key permissions
- Target account access to CMK

Common migration failure:
- Snapshot shared successfully
- KMS permissions forgotten
- Restore operation fails

Cross-account KMS policy must allow:
```json
{
  "Effect": "Allow",
  "Principal": {
    "AWS": "arn:aws:iam::<target-account>:root"
  },
  "Action": [
    "kms:Decrypt",
    "kms:DescribeKey"
  ],
  "Resource": "*"
}
```

---

#### Useful AWS CLI Commands

Share Aurora snapshot:
```bash
aws rds modify-db-cluster-snapshot-attribute \
  --db-cluster-snapshot-identifier my-snapshot \
  --attribute-name restore \
  --values-to-add <target-account-id>
```

Create DMS replication task:
```bash
aws dms create-replication-task \
  --migration-type cdc
```

1. "How does AWS DMS minimize downtime during database migration?"
   → 回答要点 (Key Answer Points)：AWS DMS supports Change Data Capture (CDC), which continuously replicates ongoing database changes after the initial load. During cutover, only a short write freeze is required to synchronize final transactions.

   → 回答要点 (Key Answer Points)：The most common issues involve KMS permissions for encrypted snapshots, networking/security group connectivity for DMS, replication lag monitoring, and ensuring application compatibility after cutover.
