A company runs an application on a fleet of Amazon EC2 instances that are in private subnets behind an internet-facing Application Load Balancer (ALB). The ALB is the origin for an Amazon CloudFront distribution. An AWS WAF web ACL that contains various AWS managed rules is associated with the CloudFront distribution.  
  
The company needs a solution that will prevent internet traffic from directly accessing the ALB.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Create a new web ACL that contains the same rules that the existing web ACL contains. Associate the new web ACL with the ALB.
- B. Associate the existing web ACL with the ALB.
- C. Add a security group rule to the ALB to allow traffic from the AWS managed prefix list for CloudFront only.
- D. Add a security group rule to the ALB to allow only the various CloudFront IP address ranges.

# Analysis: Restrict Direct Internet Access to ALB Behind CloudFront
Tags: #AWS-Certified-Solutions-Architect-Professional #SAP-C02 #CloudFront #ALB #Security-Groups #Prefix-Lists

## 1. Understanding the Problem
The company has an architecture where:
- Amazon EC2 instances run in **private subnets** (no direct internet access).
- An **internet-facing Application Load Balancer (ALB)** fronts the EC2 instances.
- The ALB is the origin for an **Amazon CloudFront distribution** (CloudFront acts as the entry point for end users).
- An **AWS WAF web ACL** (with managed rules) is associated with CloudFront to filter malicious traffic.

The core requirement is to **prevent direct internet traffic from accessing the ALB**—only traffic routed through CloudFront should reach the ALB. Additionally, the solution must have the **least operational overhead**.

Key constraints:
- Avoid manual maintenance of IP lists.
- Minimize changes to existing WAF configurations (since WAF is already protecting CloudFront).
- Ensure the solution is scalable and aligned with AWS best practices.

## 2. Services Explanation
### 2.1 Amazon CloudFront
A content delivery network (CDN) that caches content at edge locations and routes user traffic to the nearest edge. For origins behind CloudFront, AWS provides a **managed prefix list** for CloudFront IP addresses. This prefix list is automatically updated by AWS as CloudFront’s IP ranges change (documented in [AWS CloudFront IP Address Ranges](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/LocationsOfEdgeServers.html)).

### 2.2 Application Load Balancer (ALB) Security Groups
Security groups act as virtual firewalls for ALBs, controlling inbound/outbound traffic. For internet-facing ALBs, security group rules define which sources can send traffic to the ALB. By default, an internet-facing ALB allows traffic from any source (0.0.0.0/0) unless restricted by security groups.

### 2.3 AWS WAF Web ACL
AWS WAF is a web application firewall that protects against common web exploits (e.g., SQL injection, XSS). Web ACLs can be associated with **CloudFront distributions** or **ALBs**. However, associating a WAF with an ALB does **not** block direct internet traffic—it only filters traffic that reaches the ALB (per [AWS WAF Association Documentation](https://docs.aws.amazon.com/waf/latest/developerguide/web-acl-associating-aws-resources.html)).

### 2.4 AWS Managed Prefix Lists
Prefix lists are collections of CIDR blocks that can be referenced in security group rules. AWS provides managed prefix lists for many services (e.g., CloudFront, S3) that are **automatically maintained by AWS**. This eliminates the need for manual updates when service IP ranges change (documented in [AWS Managed Prefix Lists](https://docs.aws.amazon.com/vpc/latest/userguide/managed-prefix-lists.html)).

## 3. Correct Option Explanation (Option C)
### Solution
Add a security group rule to the ALB that **allows traffic only from the AWS managed prefix list for CloudFront**.

### Why This Meets Requirements
1. **Blocks direct internet traffic**: The ALB’s security group will only accept traffic from CloudFront’s IP addresses (via the managed prefix list). Any direct traffic from non-CloudFront IPs will be rejected.
2. **Least operational overhead**:
   - The AWS managed prefix list is **automatically updated**—no manual effort to track CloudFront IP range changes (unlike maintaining static IP lists).
   - No changes to the existing WAF configuration (the WAF is already protecting CloudFront, and adding WAF to the ALB is redundant for this requirement).
3. **Aligned with AWS best practices**: AWS explicitly recommends using managed prefix lists for CloudFront to restrict origin access (per [Restricting Access to Amazon CloudFront Origins](https://docs.aws.amazon.com/AmazonCloudFront/latest/DeveloperGuide/restrict-access-to-origin.html)).

### Hands-On Knowledge
In practice, you would:
1. Navigate to the ALB’s security group in the AWS Console.
2. Edit the inbound rule (e.g., for HTTP/HTTPS ports 80/443).
3. Set the **source** to `com.amazonaws.global.cloudfront.origin-facing` (the managed prefix list for CloudFront origin-facing IPs).
4. Remove any existing inbound rules that allow 0.0.0.0/0 (to block direct internet access).

## 4. Incorrect Options Explanation
### Option A: Create a new web ACL and associate it with the ALB
- **Why incorrect**: AWS WAF filters traffic, but it does **not** block direct internet access to the ALB. Even with a web ACL, the ALB would still be reachable from any internet IP—WAF would only filter malicious requests from those IPs.
- **Operational overhead**: Creating a duplicate web ACL (with the same rules as the CloudFront-associated ACL) introduces redundant configuration, increases maintenance effort, and violates the "least overhead" requirement.

### Option B: Associate the existing web ACL with the ALB
- **Why incorrect**: Associating a single web ACL with multiple resources (CloudFront + ALB) is not supported for internet-facing ALBs (per [AWS WAF Resource Compatibility](https://docs.aws.amazon.com/waf/latest/developerguide/waf-resource-associations.html)). Even if it were supported, this would not block direct traffic—it would only apply WAF rules to traffic that reaches the ALB.
- **Operational overhead**: This approach provides no benefit for the core requirement and risks misconfiguring the WAF (e.g., rule conflicts between CloudFront and ALB traffic).

### Option D: Add a security group rule to allow only CloudFront IP address ranges
- **Why incorrect**: While this would technically block direct traffic, it requires **manual maintenance** of CloudFront’s IP ranges. CloudFront updates its IP ranges regularly, and you would need to manually download and update the security group rules (e.g., via scripts or scheduled jobs) to keep up with changes.
- **Operational overhead**: This is high compared to using a managed prefix list—manual IP updates are error-prone, time-consuming, and not scalable for enterprise environments. AWS explicitly discourages this approach in favor of managed prefix lists.
