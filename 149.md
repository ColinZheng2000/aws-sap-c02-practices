A global manufacturing company plans to migrate the majority of its applications to AWS. However, the company is concerned about applications that need to remain within a specific country or in the company's central on-premises data center because of data regulatory requirements or requirements for latency of single-digit milliseconds. The company also is concerned about the applications that it hosts in some of its factory sites, where limited network infrastructure exists.

The company wants a consistent developer experience so that its developers can build applications once and deploy on premises, in the cloud, or in a hybrid architecture. The developers must be able to use the same tools, APIs, and services that are familiar to them.

Which solution will provide a consistent hybrid experience to meet these requirements?

A. Migrate all applications to the closest AWS Region that is compliant. Set up an AWS Direct Connect connection between the central on-premises data center and AWS. Deploy a Direct Connect gateway.
B. Use AWS Snowball Edge Storage Optimized devices for the applications that have data regulatory requirements or requirements for latency of single-digit milliseconds. Retain the devices on premises. Deploy AWS Wavelength to host the workloads in the factory sites.
C. Install AWS Outposts for the applications that have data regulatory requirements or requirements for latency of single-digit milliseconds. Use AWS Snowball Edge Compute Optimized devices to host the workloads in the factory sites.
D. Migrate the applications that have data regulatory requirements or requirements for latency of single-digit milliseconds to an AWS Local Zone. Deploy AWS Wavelength to host the workloads in the factory sites.

Suggested Answer: C

---

## 📋 Analysis

### 1. Understanding the Problem

A **global manufacturing company** has a diverse application landscape with conflicting deployment requirements:

| Requirement | Constraint |
|-------------|-----------|
| Some applications | Must stay **in a specific country** (data residency/regulatory) |
| Some applications | Need **single-digit millisecond latency** (must be on-premises) |
| Factory sites | **Limited network infrastructure** (edge locations with poor connectivity) |
| Developer experience | **Consistent** — build once, deploy anywhere using the **same tools, APIs, and services** |
| Cloud migration | The majority of apps ARE moving to AWS |

The company needs a **hybrid solution** that spans the full spectrum: cloud (AWS Regions), on-premises (central data center), and edge (factory floors). The solution must provide a **consistent AWS experience** across all environments — this is the hallmark of AWS's hybrid strategy.

---

### 2. Options Analysis

#### ✅ **Option C (CORRECT): AWS Outposts + Snowball Edge Compute Optimized**

This addresses all three deployment targets with consistent AWS experience:

**AWS Outposts** for the central on-premises data center:
- Outposts brings **native AWS infrastructure** — racks of AWS-designed hardware running AWS compute, storage, database, and networking services — physically installed in the customer's data center.
- Developers use the **exact same AWS APIs, CLI, console, and tools** (EC2, ECS, EKS, RDS, S3 on Outposts, etc.) as they do in AWS Regions.
- Data stays **on-premises**, satisfying data regulatory requirements.
- **Single-digit millisecond latency** to on-premises applications — the Outposts rack is literally in the same building.
- AWS manages, monitors, and patches the Outposts hardware — the customer just consumes services.

**AWS Snowball Edge Compute Optimized** for factory sites:
- Small, rugged, portable devices that provide **AWS compute and storage at the edge**.
- Runs EC2 instances (sbe-c and sbe-g instances), Lambda functions (via IoT Greengrass), and selected AWS services locally.
- Designed for environments with **limited or intermittent network connectivity** — can operate disconnected from the internet.
- Ideal for factory floors, remote sites, and industrial environments where a full Outposts rack is impractical.

**Why this delivers consistent developer experience:**
- Both Outposts and Snowball Edge run AWS-native services. Developers use the **same CloudFormation templates, same IAM roles, same ECS task definitions, same Lambda code** — across Region, Outposts, and Snowball Edge.

#### ❌ **Option A (INCORRECT): Migrate ALL to AWS + Direct Connect**

- This option **migrates everything to the cloud**, which directly violates:
  - **Data regulatory requirements** to keep data in a specific country (some countries require data to physically stay within borders, and the nearest compliant Region may still be in a different country).
  - **Single-digit millisecond latency** — if the application and its consumers are on-premises, even the closest AWS Region with Direct Connect will have latency exceeding single-digit milliseconds (physical distance + network hops).
- Direct Connect alone does not solve data residency — the data still resides in an AWS Region, which may be in a different country.
- This option completely ignores the requirement for some applications to **remain on-premises**.

#### ❌ **Option B (INCORRECT): Snowball Edge Storage Optimized + AWS Wavelength**

- **Snowball Edge Storage Optimized** is designed for **data transfer and local storage** — not for running applications. It lacks the compute capabilities of the Compute Optimized variant. You can't run EC2 instances or containerized workloads on it.
- **AWS Wavelength** embeds AWS compute and storage at **5G telecom provider edge locations** (like Verizon, Vodafone, KDDI). It is designed for ultra-low-latency mobile applications served over 5G networks — NOT for factory environments with limited network infrastructure. Wavelength requires a 5G carrier partnership.
- Factory sites with limited network infrastructure are not 5G edge locations — they're industrial environments. Wavelength is the wrong edge solution for this use case.

#### ❌ **Option D (INCORRECT): AWS Local Zones + AWS Wavelength**

- **AWS Local Zones** are AWS-managed edge locations placed in metropolitan areas — they are NOT on-premises. They extend a parent AWS Region to a nearby city, providing single-digit millisecond latency for end-users in that metro area.
  - **Local Zones do NOT solve data residency** — data in a Local Zone is still within the parent AWS Region's jurisdiction. If the Region is in a different country, the Local Zone data is also in that country.
  - Local Zones also cannot provide the on-premises experience for the central data center.
- **AWS Wavelength**, as discussed above, is for 5G mobile edge, not factory/industrial edge.
- Neither Local Zones nor Wavelength address the requirement of keeping data within a specific country when the nearest Region is elsewhere, nor do they work for factory sites with limited network.

**Consistency in Practice:**
```bash
# Deploy the SAME container to Region, Outposts, or Snowball Edge
aws ecs create-service \
  --cluster manufacturing-cluster \
  --service-name factory-processor \
  --task-definition factory-processor:1 \
  --launch-type EC2 \
  --desired-count 3

# The only difference is which cluster/region you target —
# the task definition, IAM roles, and application code are identical.
```

**Important: Outposts Requirements & Limitations:**
- Requires a **dedicated, direct connection** to the parent AWS Region (via AWS Direct Connect, VPN, or public internet as a backup).
- Not available in all countries — check the Outposts rack availability by Region.
- Supported services: EC2, EBS, ECS, EKS, RDS, S3 on Outposts, EMR, SageMaker Edge Manager, and more (but not all AWS services).
- Minimum commitment term: 3 years.

**Important: Snowball Edge Compute Limitations:**
- Runs a subset of AWS services — primarily EC2 instances (sbe-c instance types) and Lambda functions via IoT Greengrass.
- Storage is limited (42 TB usable on Compute Optimized device).
- Devices are physically shipped by AWS; not permanently installed like Outposts.
- Primarily designed for edge processing, data collection, and periodic data transfer to AWS.
