A solutions architect works for a government agency that has strict disaster recovery requirements. All Amazon Elastic Block Store (Amazon EBS) snapshots are required to be saved in at least two additional AWS Regions. The agency also is required to maintain the lowest possible operational overhead.

Which solution meets these requirements?

A. Configure a policy in Amazon Data Lifecycle Manager (Amazon DLM) to run once daily to copy the EBS snapshots to the additional Regions.
B. Use Amazon EventBridge to schedule an AWS Lambda function to copy the EBS snapshots to the additional Regions.
C. Setup AWS Backup to create the EBS snapshots. Configure Amazon S3 Cross-Region Replication to copy the EBS snapshots to the additional Regions.
D. Schedule Amazon EC2 Image Builder to run once daily to create an AMI and copy the AMI to the additional Regions.

## Understanding the Problem
A government agency requires **disaster recovery** controls such that **every Amazon EBS snapshot** must exist in **at least two additional AWS Regions** (i.e., multi-Region copies). The agency also wants the **lowest possible operational overhead** (managed service, minimal custom code, minimal maintenance).

So we need a solution that:
- Automates **snapshot creation + cross-Region copy** (≥ 2)
- Minimizes custom orchestration (no Lambda/cron unless necessary)
- Is designed for EBS snapshot lifecycle and retention

- --

## Correct Option(s) Explanation
### ✅ A. Configure an Amazon Data Lifecycle Manager (Amazon DLM) policy to copy EBS snapshots to additional Regions
* *Why this meets the requirements**
- **Amazon DLM** is purpose-built to automate the lifecycle of **EBS snapshots** (creation, retention, deletion).
- DLM supports **cross-Region copy** of EBS snapshots as part of a lifecycle policy, which directly addresses the requirement to store snapshots in other Regions.
- It is **low operational overhead**: no servers, no custom code, no event rules to maintain—just a policy.

* *Key points**
- `[Amazon DLM]` manages EBS snapshot lifecycle natively
- `[Cross-Region copy]` can replicate snapshots into other Regions (configure multiple destination Regions when supported by the feature set/console at the time)
- `[Retention policies]` reduce compliance risk and manual cleanup

* *Official AWS references**
- Amazon Data Lifecycle Manager (DLM) for EBS snapshots:  
  https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/s-lifecycle.html  
- DLM copy snapshots across Regions (copy rules):  
  https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/snapshot-lifecycle.html#dlm-copy-snapshots

> Exam note: For “lowest operational overhead” and “EBS snapshots” specifically, **DLM** is usually the first-choice managed service.

## Incorrect Options Explanation
### ❌ B. EventBridge schedule + Lambda to copy snapshots
- This can work technically, but it increases **operational overhead**:
  - Custom code to maintain and secure (IAM, dependencies, testing)
  - Retries/error handling, idempotency, throttling, pagination, monitoring
  - Multi-Region copy orchestration complexity grows quickly
- Managed lifecycle policies (DLM / AWS Backup) are preferred for government/compliance environments.

* *Services involved**:  
`[Amazon EventBridge]` `[AWS Lambda]` `[EBS Snapshot Copy]`  
* *Reason incorrect**: higher ops burden than managed alternatives.

- --

### ❌ C. AWS Backup + S3 Cross-Region Replication (CRR)
- **EBS snapshots are not replicated by S3 CRR**. CRR applies to **S3 objects in buckets**, not EBS snapshots.
- While **AWS Backup** *does* support **cross-Region copy** of recovery points (including EBS), the option’s described mechanism (S3 CRR) is wrong.

* *Services involved**:  
`[AWS Backup]` `[Amazon S3 CRR]`  
* *Reason incorrect**: invalid replication mechanism for EBS snapshots.

> If the option had said “AWS Backup cross-Region copy to two Regions,” it could be a contender. But as written, it’s incorrect.

- --

### ❌ D. EC2 Image Builder to create/copy AMIs
- The requirement is specifically about **EBS snapshots** being saved in two additional Regions.
- AMIs can be copied cross-Region, but:
  - This changes the artifact from “snapshot compliance” to “AMI compliance”
  - It introduces extra components and doesn’t directly ensure **all EBS snapshots** are copied
- Also higher operational complexity than a snapshot lifecycle policy.

* *Services involved**:  
`[EC2 Image Builder]` `[AMI]`  
* *Reason incorrect**: doesn’t meet the “all EBS snapshots” requirement and is not the simplest DR control.

## Hands-on Knowledge (Enterprise / Gov DR Practices)
### Practical design recommendations
- **Use tag-based DLM policies** (e.g., `Backup=Gold`) to ensure all relevant volumes are included automatically as infrastructure changes.
- Define:
  - Snapshot schedule (e.g., hourly/daily) aligned with **RPO**
  - Retention count/days aligned with records policy
  - **Cross-Region copy** to **two destination Regions** (for regional DR + geopolitical separation)
- Add **encryption controls**:
  - Ensure snapshots are encrypted with **KMS keys** in source and destination Regions
  - Plan for KMS key policies, grants, and potential cross-account access if needed

### Operational controls you should implement
- `[AWS CloudTrail]` auditing for snapshot creation/copy API calls
- `[Amazon EventBridge]` + `[SNS]` alerts on DLM policy failures (DLM emits events you can monitor)
- Periodic DR testing: restore volumes from copied snapshots in each DR Region (compliance evidence)

### When you might prefer AWS Backup instead
If the agency later expands requirements beyond EBS (e.g., EFS, RDS, DynamoDB, Storage Gateway), **AWS Backup** can centralize governance across services. For “EBS snapshots only” and minimal overhead, **DLM** is typically the cleanest fit.
