A finance company is running its business-critical application on current-generation Linux EC2 instances. The application includes a self-managed MySQL database performing heavy I/O operations. The application is working fine to handle a moderate amount of traffic during the month. However, it slows down during the final three days of each month due to month-end reporting, even though the company is using Elastic Load Balancers and Auto Scaling within its infrastructure to meet the increased demand.  
  
Which of the following actions would allow the database to handle the month-end load with the LEAST impact on performance?

- A. Pre-warming Elastic Load Balancers, using a bigger instance type, changing all Amazon EBS volumes to GP2 volumes.
- B. Performing a one-time migration of the database cluster to Amazon RDS, and creating several additional read replicas to handle the load during end of month.
- C. Using Amazon CloudWatch with AWS Lambda to change the type, size, or IOPS of Amazon EBS volumes in the cluster based on a specific CloudWatch metric.
- D. Replacing all existing Amazon EBS volumes with new PIOPS volumes that have the maximum available storage size and I/O per second by taking snapshots before the end of the month and reverting back afterwards.
# AWS Scenario: Optimizing Database Performance for Month-End Reporting Spikes

**Exam Domain:** Domain 1: Design for Organizational Complexity / Domain 2: Design for New Solutions
**Tags:** #AWS/SAP-C02 #AWS/Database/RDS #AWS/Storage/EBS #AWS/Architecture/Performance

---

## 1. The Question

**Scenario:**
A finance company runs a business-critical application on current-generation Linux EC2 instances. The application uses a **self-managed MySQL database** performing **heavy I/O operations**.
* **Normal State:** Works fine during the month.
* **Problem:** Slows down during the final three days (month-end reporting).
* **Current Infra:** Uses Elastic Load Balancers (ELB) and Auto Scaling Groups (ASG).

**Goal:**
Which action allows the database to handle the month-end load with the **LEAST impact on performance**?

---

## 2. Analysis

### Understanding the Problem
The core issue is a **Database I/O Bottleneck** caused by specific "month-end reporting" activities.
1.  **Workload Type:** The scenario specifies "reporting." Reporting workloads are typically **Read-Heavy** (SELECT queries) that scan large amounts of data.
2.  **Current Architecture:** The application tier scales (ELB/ASG), but the database is a monolith (Self-managed MySQL on EC2).
3.  **Constraint:** We need the "least impact on performance" when the load hits.
4.  **The Trap:** Attempting to solve a database read-contention issue by just throwing raw storage speed (IOPS) at it, rather than architectural optimization (splitting reads/writes).

### Services Explanation

* **Self-Managed MySQL on EC2:** You are responsible for patching, scaling, backups, and high availability. Scaling usually involves vertical scaling (bigger instance) or manual sharding/replication setup.
* **Amazon RDS (Relational Database Service):** A managed service that automates patching, backups, and scaling. Crucially, it supports **Read Replicas**.
* **Read Replicas:** Asynchronous copies of the primary database. You can route read-only traffic (like reporting) to replicas, freeing up the primary database for write operations (business-critical heavy I/O).
* **EBS Elastic Volumes:** Features that allow you to increase volume size, change volume type, or adjust performance (IOPS) on live volumes. **Crucial Limitation:** You generally **cannot decrease** the size of an EBS volume without creating a new volume and migrating data.

---

### Correct Option Explanation

**Option B: Performing a one-time migration of the database cluster to Amazon RDS, and creating several additional read replicas to handle the load during end of month.**

* **Why it works:** The performance degradation is caused by reporting queries competing for I/O against business-critical transactional writes. By migrating to RDS, you gain the ability to spin up **Read Replicas** easily.
* **Handling the Load:** During month-end, the application (or the reporting tool) can point to the Read Replicas. This isolates the heavy reporting I/O from the primary instance. The primary instance continues to handle transactions without performance degradation.
* **"Least Impact":** While the migration itself is a one-time effort, the question asks how to handle the *month-end load* with the least impact. Offloading traffic to replicas ensures the main application is not impacted by the reporting jobs.

> **Key takeaway:** When you see "Reporting" or "Analytics" causing database slowness, immediately look for **Read Replicas** to segregate read traffic from write traffic.



---

### Incorrect Options Explanation

**A. Pre-warming ELB, bigger instance, changing to GP2.**
* **ELB Pre-warming:** ELBs scale automatically. Pre-warming is only for massive, sudden flash traffic (not a predictable month-end report). Furthermore, the bottleneck is the DB, not the load balancer.
* **GP2:** The scenario says the DB performs "heavy I/O." GP2 (General Purpose) is burstable but often inferior to Provisioned IOPS (io1/io2) for sustained heavy database I/O. Downgrading or switching to GP2 might make it worse.
* **Bigger Instance:** Requires a stop/start (downtime) and doesn't solve the architectural inefficiency of mixing reporting and transactional loads.

**C. Using CloudWatch + Lambda to change EBS volume type/size/IOPS.**
* **Performance Penalty:** While EBS Elastic Volumes allow live modification, the volume goes into an "Optimizing" state. During this state, volume performance is often degraded. Doing this *during* the high-load period is dangerous.
* **Irreversibility:** You can increase size/IOPS easily. However, you **cannot decrease** the size of an EBS volume directly. To "revert back" (shrink) after the month-end, you would have to create a new smaller volume, copy all data, and swap them, which is operationally complex and risky.

**D. Replacing volumes with PIOPS via snapshots and reverting back.**
* **Downtime:** Detaching the root or data volumes to swap them with new Snapshot-based volumes usually requires stopping the DB service or the Instance to ensure data consistency.
* **Complexity:** This requires a massive amount of manual or scripted orchestration every single month.
* **Risk:** "Reverting back" involves the same downtime and risk of data loss if the snapshot restoration isn't perfectly synchronized with new writes that occurred during the month-end.
