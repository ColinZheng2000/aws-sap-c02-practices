A solutions architect needs to assess a newly acquired company’s portfolio of applications and databases. The solutions architect must create a business case to migrate the portfolio to AWS. The newly acquired company runs applications in an on-premises data center. The data center is not well documented. The solutions architect cannot immediately determine how many applications and databases exist. Traffic for the applications is variable. Some applications are batch processes that run at the end of each month.

The solutions architect must gain a better understanding of the portfolio before a migration to AWS can begin.

Which solution will meet these requirements?

A. Use AWS Server Migration Service (AWS SMS) and AWS Database Migration Service (AWS DMS) to evaluate migration. Use AWS Service Catalog to understand application and database dependencies.
B. Use AWS Application Migration Service. Run agents on the on-premises infrastructure. Manage the agents by using AWS Migration Hub. Use AWS Storage Gateway to assess local storage needs and database dependencies.
C. Use Migration Evaluator to generate a list of servers. Build a report for a business case. Use AWS Migration Hub to view the portfolio. Use AWS Application Discovery Service to gain an understanding of application dependencies.
D. Use AWS Control Tower in the destination account to generate an application portfolio. Use AWS Server Migration Service (AWS SMS) to generate deeper reports and a business case. Use a landing zone for core accounts and resources.

SUggested Answer: C

---

## 📋 Analysis

### 1. Understanding the Problem

The solutions architect faces a **brownfield migration assessment** — a newly acquired company with a poorly documented on-premises data center. Before any migration can begin, the architect must:

- **Discover** what applications and databases exist (inventory).
- **Understand dependencies** between applications and databases.
- **Quantify** the portfolio to build a business case.
- Account for **variable traffic patterns** (including monthly batch jobs).

This is the **Assess phase** of the AWS migration journey. The tools selected must handle discovery, dependency mapping, and business case generation without requiring pre-existing documentation.

---

### 2. Options Analysis

#### ✅ **Option C (CORRECT): Migration Evaluator + Migration Hub + Application Discovery Service**

This option maps directly to the three phases of migration assessment:

1. **Migration Evaluator** (formerly TSO Logic): An agentless assessment tool that analyzes on-premises compute, storage, and licensing data to generate a **business case** with TCO projections, right-sizing recommendations, and licensing cost analysis. It ingests data from existing inventory sources (vCenter, CMDB, spreadsheets) — perfect for poorly documented environments.

2. **AWS Migration Hub**: A central place to **visualize the discovered portfolio** — servers, applications, and their migration status across the entire estate. It provides a unified dashboard.

3. **AWS Application Discovery Service**: Deploys agents (or uses agentless collectors) on on-premises servers to **map application dependencies** — which servers talk to which, on what ports, with what protocols. This reveals the unknown application topology.

**Why this is the correct combination:**
- Migration Evaluator → Business case (TCO, right-sizing, licensing)
- Application Discovery Service → Dependency mapping (what talks to what)
- Migration Hub → Portfolio visualization (single pane of glass)
- Each tool serves a distinct, non-overlapping purpose in the Assess phase.
- Application Discovery Service agents collect performance data over time, capturing variable traffic patterns and end-of-month batch spikes.

#### ❌ **Option A (INCORRECT): AWS SMS + AWS DMS + AWS Service Catalog**

- **AWS Server Migration Service (SMS)** is a migration execution tool (lift-and-shift), **not** an assessment/discovery tool. It replicates live servers to AWS, which is premature before understanding the portfolio.
- **AWS Database Migration Service (DMS)** is for database migration execution, not dependency mapping or discovery.
- **AWS Service Catalog** is for managing a catalog of approved IT services and products — it has nothing to do with understanding application dependencies in an on-premises environment.
- **None of these tools perform discovery or business case generation.**

#### ❌ **Option B (INCORRECT): AWS Application Migration Service + Migration Hub + Storage Gateway**

- **AWS Application Migration Service (MGN)** is the successor to CloudEndure — a **lift-and-shift migration execution** tool. While it uses agents and Migration Hub, it's designed for executing migrations, not discovering unknown portfolios.
- **AWS Storage Gateway** provides hybrid cloud storage (file, volume, tape gateways) — it does NOT assess database dependencies or application topology.
- Running MGN agents would start replicating servers without understanding what they do, their dependencies, or whether they're even needed. This is the "migrate first, ask questions later" anti-pattern.

#### ❌ **Option D (INCORRECT): AWS Control Tower + AWS SMS + Landing Zone**

- **AWS Control Tower** is for setting up and governing a **multi-account AWS environment** (landing zone) — it does NOT discover on-premises applications or generate an application portfolio.
- **AWS SMS** (as discussed above) is for migration execution, not discovery.
- **Landing zones** are about the destination AWS architecture (accounts, OUs, guardrails), not about understanding the source environment.
- This option confuses *destination environment setup* with *source environment discovery*. Control Tower is relevant later in the migration journey (Mobilize phase), not during initial discovery.

#### Tool Deep Dive

**Migration Evaluator:**
- Agentless: ingests existing inventory exports (CSV, vCenter, etc.).
- Outputs: TCO comparison (on-prem vs. AWS), right-sizing recommendations, Microsoft licensing optimization.
- Best for: Building the financial business case for C-level approval.

**AWS Application Discovery Service:**
- Two modes:
  - **Agent-based**: Install an agent on each server for detailed metrics (CPU, memory, network connections, running processes). Recommended for production workloads with variable traffic.
  - **Agentless**: Deploy a VMware collector appliance that uses vCenter APIs. Less granular but no agent installation needed.
- Outputs: Server utilization metrics, network dependency maps, process-level information.
- Data is stored in AWS Migration Hub and can be exported to Athena/QuickSight for custom analysis.

**AWS Migration Hub:**
- Central dashboard for tracking migration progress.
- Supports discovery data import from multiple sources.
- Integrates with MGN, DMS, SMS, and partner tools.
- Provides the "single pane of glass" for the migration team.

**Enterprise-Ready Assessment Workflow:**
```bash
# 1. Deploy Application Discovery Service agents on sample servers
# (Use AWS-provided installers for Windows/Linux)

# 2. After 2-4 weeks of data collection, export to Migration Hub
aws discovery describe-agents
aws discovery list-server-neighbors  # Shows dependency map

# 3. Run Migration Evaluator on the collected data
# (Web-based tool at https://migration-evaluator.aws)

# 4. Use Migration Hub to view and organize the portfolio
aws migration-hub list-application-states

# 5. Build business case with Migration Evaluator reports
```

**Critical Exam Distinction:**

| Service | Phase | Purpose |
|---------|-------|---------|
| **Migration Evaluator** | Assess | Business case, TCO, right-sizing |
| **Application Discovery Service** | Assess | Dependency mapping, performance data |
| **Migration Hub** | All phases | Central portfolio tracking |
| **Application Migration Service (MGN)** | Migrate | Block-level replication & cutover |
| **Server Migration Service (SMS)** | Migrate | Snapshot-based server migration |
| **Database Migration Service (DMS)** | Migrate | Database migration (homogeneous/heterogeneous) |

**Gotcha:** The exam often mixes Assess-phase and Migrate-phase tools in the same answer to confuse you. Always ask: "Is this the right phase?" before selecting a tool combination.
