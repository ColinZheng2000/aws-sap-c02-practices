An AWS partner company is building a service in AWS Organizations using its organization named org1. This service requires the partner company to have access to AWS resources in a customer account, which is in a separate organization named org2. The company must establish least privilege security access using an API or command line tool to the customer account.  
What is the MOST secure way to allow org1 to access resources in org2?  

- A. The customer should provide the partner company with their AWS account access keys to log in and perform the required tasks.
- B. The customer should create an IAM user and assign the required permissions to the IAM user. The customer should then provide the credentials to the partner company to log in and perform the required tasks.
- C. The customer should create an IAM role and assign the required permissions to the IAM role. The partner company should then use the IAM role's Amazon Resource Name (ARN) when requesting access to perform the required tasks.
- D. The customer should create an IAM role and assign the required permissions to the IAM role. The partner company should then use the IAM role's Amazon Resource Name (ARN), including the external ID in the IAM role's trust policy, when requesting access to perform the required tasks.

## 🧠 Analysis: Cross-Account Access for Partner Services

---

### Understanding the Problem

The core requirement is to establish a **secure**, **least privilege** access mechanism for a partner company's AWS environment ($\text{org1}$) to access resources in a customer's separate AWS account ($\text{org2}$). This access must be programmatic (via API or CLI). The solution must adhere to the **least privilege principle** and be the **MOST secure** method available. The context involves two different AWS Organizations, emphasizing the need for a robust cross-account, cross-organization solution.

---

### Services Explanation

- **AWS Identity and Access Management (IAM) Roles:** IAM Roles are the best practice and most secure method for granting **cross-account access** within AWS. Unlike users, roles don't have standard long-term credentials (password or access keys). Instead, an entity (like an IAM user or an EC2 instance in another account) assumes the role to obtain temporary, limited-time security credentials.
    
- **Trust Policy:** This is a JSON policy attached to an IAM role that defines **who** (the Principal) is allowed to assume the role. In a cross-account scenario, the Principal is typically the account ID of the other AWS account.
    
- **External ID:** A configuration option in an IAM role's trust policy. It is a piece of arbitrary string data that is required to be passed by the assuming entity. Its primary purpose is to **prevent the confused deputy problem**, particularly in scenarios where a third-party (like a partner company) is offering a service that involves accessing resources in their customer's accounts. It ensures that the role is only assumed by the entity that _intended_ the operation, adding a crucial layer of security verification.
    

---

### Correct Option(s) Explanation

**D. The customer should create an IAM role and assign the required permissions to the IAM role. The partner company should then use the IAM role's Amazon Resource Name (ARN), including the external ID in the IAM role's trust policy, when requesting access to perform the required tasks.**

- **IAM Role:** This is the foundation of secure cross-account access, adhering to the least privilege principle by providing only **temporary credentials** for specific, required tasks.
    
- **External ID:** This is what makes option D the **MOST secure** choice. According to the **AWS official documentation on IAM best practices** (specifically regarding "Preventing the Confused Deputy Problem with the External ID"), using an External ID is strongly recommended for scenarios where a third-party (the partner company) manages the access to your resources on your behalf. The partner company provides a unique, secret External ID during the `AssumeRole` API call, and the role's trust policy enforces its presence, mitigating the risk of the partner being tricked into performing actions for an unauthorized party (the **Confused Deputy Problem**).
    

---

### Incorrect Options Explanation

- **A. The customer should provide the partner company with their AWS account access keys to log in and perform the required tasks.**
    
    - **Incorrect:** Sharing the **root account access keys** is an extreme security risk and violates fundamental AWS security best practices. Root account access has unrestricted permissions across the entire account. This provides far more than least privilege access.
        
- **B. The customer should create an IAM user and assign the required permissions to the IAM user. The customer should then provide the credentials to the partner company to log in and perform the required tasks.**
    
    - **Incorrect:** While better than sharing root credentials, sharing **IAM user credentials (long-term access keys)** is still considered less secure than using IAM roles for cross-account access. The keys never expire unless manually rotated, increasing the risk profile in case of compromise. IAM Roles with temporary credentials are the established best practice for this scenario.
        
- **C. The customer should create an IAM role and assign the required permissions to the IAM role. The partner company should then use the IAM role's Amazon Resource Name (ARN) when requesting access to perform the required tasks.**
    
    - **Incorrect (Less Secure):** This uses the correct mechanism (**IAM Role**) but omits the crucial security enhancement of the **External ID**. While an IAM role is secure, in a multi-customer partner environment, omitting the External ID leaves the customer susceptible to the **Confused Deputy Problem**. Therefore, it is _less_ secure than option D, and thus not the **MOST secure** way.
