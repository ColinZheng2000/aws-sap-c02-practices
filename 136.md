A company needs to audit the security posture of a newly acquired AWS account. The company’s data security team requires a notification only when an Amazon S3 bucket becomes publicly exposed. The company has already established an Amazon Simple Notification Service (Amazon SNS) topic that has the data security team's email address subscribed.

Which solution will meet these requirements?

A. Create an S3 event notification on all S3 buckets for the isPublic event. Select the SNS topic as the target for the event notifications.
B. Create an analyzer in AWS Identity and Access Management Access Analyzer. Create an Amazon EventBridge rule for the event type “Access Analyzer Finding” with a filter for “isPublic: true.” Select the SNS topic as the EventBridge rule target.
C. Create an Amazon EventBridge rule for the event type “Bucket-Level API Call via CloudTrail” with a filter for “PutBucketPolicy.” Select the SNS topic as the EventBridge rule target.
D. Activate AWS Config and add the cloudtrail-s3-dataevents-enabled rule. Create an Amazon EventBridge rule for the event type “Config Rules Re-evaluation Status” with a filter for “NON_COMPLIANT.” Select the SNS topic as the EventBridge rule target.

Suggested Answer: B

---

## 📋 Analysis

### 1. Understanding the Problem

The company has **newly acquired an AWS account** and needs to **audit its security posture**, specifically detecting when any **S3 bucket becomes publicly exposed**. The data security team already has an SNS topic with email subscriptions configured, and they only want notifications — not automated remediation. The solution must identify both existing public buckets and newly exposed ones.

**Key insight:** This is a continuous monitoring problem, not a one-time audit. The solution needs to detect public exposure regardless of how it happened — via bucket policy, ACL, or access point policy.

---

### 2. Options Analysis

#### ✅ **Option B (CORRECT): IAM Access Analyzer + EventBridge + SNS**

This is the **purpose-built AWS solution** for detecting publicly accessible resources:

1. **AWS Identity and Access Management (IAM) Access Analyzer** continuously monitors resource-based policies (S3 bucket policies, S3 ACLs, KMS key policies, IAM roles, SQS queue policies, etc.) and generates **findings** when a resource becomes accessible from outside your account — including public access.

2. **Amazon EventBridge** natively integrates with Access Analyzer — all findings are emitted as EventBridge events with the detail type `Access Analyzer Finding`. You can filter for `"isPublic": true` to only trigger on public exposure events.

3. **Amazon SNS topic** is set as the target of the EventBridge rule, sending email notifications to the data security team.

**Why this is the best solution:**
- **Continuous monitoring** — not just point-in-time. New buckets or policy changes are detected.
- **Covers all exposure vectors** — bucket policies, ACLs, and access point policies.
- **Zero operational overhead** — fully managed, no custom code, no Lambda functions to maintain.
- **Precise filtering** — the `isPublic` field in the finding makes it trivial to filter for only public access issues.

#### ❌ **Option A (INCORRECT): S3 Event Notification for "isPublic" Event**

- **S3 event notifications do NOT have an "isPublic" event type.** S3 event types include `s3:ObjectCreated:*`, `s3:ObjectRemoved:*`, `s3:ObjectRestore:*`, `s3:Replication:*`, etc. There is no event for bucket-level access changes.
- S3 event notifications are object-level events, not bucket policy/ACL change events.
- Even if such an event existed, it would only fire on changes — not detect buckets that were already public before the notification was configured.

#### ❌ **Option C (INCORRECT): CloudTrail + EventBridge for "PutBucketPolicy"**

- This approach only detects **new policy changes** via `PutBucketPolicy` API calls. It misses:
  - Buckets that were already publicly accessible before the rule was created.
  - Public access granted via **ACL changes** (`PutBucketAcl`), not bucket policy.
  - Public access granted via **S3 Access Points** or **Multi-Region Access Points**.
- Additionally, this requires CloudTrail to be enabled and properly configured, adding setup overhead.
- The filter for `PutBucketPolicy` is too narrow — a comprehensive audit needs to cover ACLs, access points, and existing configurations.

#### ❌ **Option D (INCORRECT): AWS Config + cloudtrail-s3-dataevents-enabled Rule**

- The **`cloudtrail-s3-dataevents-enabled`** Config managed rule checks whether **CloudTrail data events are enabled for S3 buckets** — it does NOT check if S3 buckets are publicly accessible.
- This is a completely different security concern (logging/auditability) unrelated to public access detection.
- Even if a more appropriate Config rule were used (like `s3-bucket-public-read-prohibited` or `s3-bucket-public-write-prohibited`), the filtering described in the option — `Config Rules Re-evaluation Status` with `NON_COMPLIANT` — would trigger on ANY non-compliant Config rule, not just S3 public access rules, leading to noisy notifications.

### 3. Hands-On Knowledge

#### IAM Access Analyzer vs. Other S3 Public Access Detection Methods

| Feature | IAM Access Analyzer | AWS Config s3-bucket-public-* | CloudTrail PutBucketPolicy |
|---------|---------------------|------------------------------|---------------------------|
| Continuous monitoring | ✅ Yes | ✅ Periodic | ❌ Event-driven for changes only |
| Detects existing public buckets | ✅ Immediately | ✅ On next evaluation | ❌ Only new changes |
| Covers ACL-based exposure | ✅ Yes | ✅ Yes | ❌ No (PutBucketAcl separate) |
| Covers Access Point exposure | ✅ Yes | ❌ No | ❌ No |
| Covers cross-account grants | ✅ Yes | ✅ Partial | ❌ Only if via bucket policy |
| Zero setup beyond enablement | ✅ Yes (enable analyzer) | ❌ Requires Config recorder | ❌ Requires CloudTrail trail |
| Free tier | ✅ Free for S3 findings | ❌ Config rules cost | ❌ CloudTrail costs |

#### Setting Up the Correct Solution (Option B)

**Step 1: Create IAM Access Analyzer (Account Level)**
```bash
aws accessanalyzer create-analyzer \
  --analyzer-name s3-public-access-analyzer \
  --type ACCOUNT
```
> Once created, the analyzer automatically begins evaluating all resource-based policies in the account.

**Step 2: Create EventBridge Rule**
```json
{
  "source": ["aws.access-analyzer"],
  "detail-type": ["Access Analyzer Finding"],
  "detail": {
    "isPublic": [true],
    "resourceType": ["AWS::S3::Bucket"]
  }
}
```
```bash
aws events put-rule \
  --name s3-public-access-finding \
  --event-pattern '{"source":["aws.access-analyzer"],"detail-type":["Access Analyzer Finding"],"detail":{"isPublic":[true],"resourceType":["AWS::S3::Bucket"]}}'
```

**Step 3: Add SNS Topic as Target**
```bash
aws events put-targets \
  --rule s3-public-access-finding \
  --targets "Id"="1","Arn"="arn:aws:sns:region:account:security-team-topic"
```

**Important Exam Gotchas:**
- Access Analyzer must be created **per-Region, per-account** (or at the organization level via delegated administrator).
- Access Analyzer findings for S3 are **free** (included in the free tier). Findings for other resource types may incur costs.
- The `isPublic` field is distinct from `isShared` — `isPublic` means the entire internet, `isShared` means specific external accounts/organizations. Both may be relevant depending on your security posture.
- Access Analyzer does NOT remediate — it only identifies and reports. For automated remediation, combine with EventBridge → Lambda → auto-remediate.
- **IAM Access Analyzer requires no CloudTrail, no Config, no Lambda** — it's a standalone, fully managed service.
