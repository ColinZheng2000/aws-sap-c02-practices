A company is running several workloads in a single AWS account. A new company policy states that engineers can provision only approved resources and that engineers must use AWS CloudFormation to provision these resources. A solutions architect needs to create a solution to enforce the new restriction on the IAM role that the engineers use for access.  
  
What should the solutions architect do to create the solution?

- A. Upload AWS CloudFormation templates that contain approved resources to an Amazon S3 bucket. Update the IAM policy for the engineers’ IAM role to only allow access to Amazon S3 and AWS CloudFormation. Use AWS CloudFormation templates to provision resources.
- B. Update the IAM policy for the engineers’ IAM role with permissions to only allow provisioning of approved resources and AWS CloudFormation. Use AWS CloudFormation templates to create stacks with approved resources.
- C. Update the IAM policy for the engineers’ IAM role with permissions to only allow AWS CloudFormation actions. Create a new IAM policy with permission to provision approved resources, and assign the policy to a new IAM service role. Assign the IAM service role to AWS CloudFormation during stack creation.
- D. Provision resources in AWS CloudFormation stacks. Update the IAM policy for the engineers’ IAM role to only allow access to their own AWS CloudFormation stack.

### Understanding the Problem

The company is establishing a **governance policy** requiring that engineers can _only_ provision **approved AWS resources** and must use **AWS CloudFormation** for all provisioning. The goal is to enforce this restriction on the **IAM role** used by the engineers. The solution needs to ensure that the engineers can execute CloudFormation operations, but the underlying resources created by those stacks must be limited to a predefined, approved set.

This is a classic governance and security challenge, focusing on **preventing "shadow IT"** or unauthorized resource creation while enabling necessary developer/engineer workflow (Infrastructure as Code via CloudFormation).

### Correct Option(s) Explanation

**The correct option is C.**

**C. Update the IAM policy for the engineers’ IAM role with permissions to only allow AWS CloudFormation actions. Create a new IAM policy with permission to provision approved resources, and assign the policy to a new IAM service role. Assign the IAM service role to AWS CloudFormation during stack creation.**

This approach leverages the key security separation within the CloudFormation workflow:

1. **Engineer's IAM Role (User Role):** This role's policy should be restricted to only allow _actions on CloudFormation_ itself (e.g., `cloudformation:CreateStack`, `cloudformation:DeleteStack`, `cloudformation:UpdateStack`). This addresses the requirement that **engineers must use AWS CloudFormation**. It prevents engineers from directly calling API actions on unapproved services (like `ec2:RunInstances` or `s3:CreateBucket`).
    
2. **CloudFormation Service Role (Execution Role):** This is the crucial part for governance.
    
    - A **new IAM service role** (also called an _execution role_) is created.
        
    - An **IAM policy** is attached to this service role that **explicitly allows provisioning _only_ the approved resources** (e.g., `ec2:RunInstances` but perhaps only for a specific approved instance type, or `s3:CreateBucket` but with an enforced tag or logging configuration).
        
    - When the engineer creates a stack via CloudFormation, they specify this **approved service role**.
        
    
    **Official AWS Documentation Reference (CloudFormation Execution Role):**
    
    > When you create, update, or delete a stack, you can specify an IAM service role that CloudFormation can assume to perform the stack operations. ... The permissions of this role govern which resources you can provision.
    > 
    > (Reference: AWS CloudFormation User Guide - AWS Service Roles)
    

This dual-role setup perfectly enforces both requirements: the **engineer** can only execute the **CloudFormation process**, and the **CloudFormation process** can only provision the **approved resources**.

### Incorrect Options Explanation

|**Option**|**Reason for Incorrectness**|
|---|---|
|**A. Upload AWS CloudFormation templates that contain approved resources to an Amazon S3 bucket. Update the IAM policy for the engineers’ IAM role to only allow access to Amazon S3 and AWS CloudFormation. Use AWS CloudFormation templates to provision resources.**|This only controls **where the templates come from** (S3) and ensures the engineer can _run_ CloudFormation. It **does not restrict the underlying resources** that CloudFormation can create. The engineer could still reference a template that provisions unapproved resources, and the _execution role_ (which defaults to the engineer's permissions if not specified) would need broad permissions, defeating the purpose.|
|**B. Update the IAM policy for the engineers’ IAM role with permissions to only allow provisioning of approved resources and AWS CloudFormation. Use AWS CloudFormation templates to create stacks with approved resources.**|If the engineer's IAM role is also the role used by CloudFormation to _provision the resources_ (the default execution behavior), granting it permissions to approved resources _still_ grants the engineer the ability to **directly call the API actions** for those resources _outside_ of CloudFormation. This violates the policy that **engineers must use AWS CloudFormation** for provisioning, as they could bypass it.|
|**D. Provision resources in AWS CloudFormation stacks. Update the IAM policy for the engineers’ IAM role to only allow access to their own AWS CloudFormation stack.**|This restricts the engineer's access to **manage the stack object itself** (`cloudformation:DescribeStacks` etc.) but **does not restrict the resource types** that the stack can provision. The engineer could still launch a stack that provisions unauthorized resources, as the restriction is on the stack _object_, not the underlying service API calls made by the _execution role_.|
