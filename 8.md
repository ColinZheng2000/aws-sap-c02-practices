A company wants to use a third-party software-as-a-service (SaaS) application. The third-party SaaS application is consumed through several API calls. The third- party SaaS application also runs on AWS inside a VPC.  
The company will consume the third-party SaaS application from inside a VPC. The company has internal security policies that mandate the use of private connectivity that does not traverse the internet. No resources that run in the company VPC are allowed to be accessed from outside the company's VPC. All permissions must conform to the principles of least privilege.  
Which solution meets these requirements?  

- A. Create an AWS PrivateLink interface VPC endpoint. Connect this endpoint to the endpoint service that the third-party SaaS application provides. Create a security group to limit the access to the endpoint. Associate the security group with the endpoint.
- B. Create an AWS Site-to-Site VPN connection between the third-party SaaS application and the company VPC. Configure network ACLs to limit access across the VPN tunnels.
- C. Create a VPC peering connection between the third-party SaaS application and the company VPC. Update route tables by adding the needed routes for the peering connection.
- D. Create an AWS PrivateLink endpoint service. Ask the third-party SaaS provider to create an interface VPC endpoint for this endpoint service. Grant permissions for the endpoint service to the specific account of the third-party SaaS provider.


**Tags:** #AWS_Networking #AWS_PrivateLink #VPC_Endpoint #Security #Least_Privilege #SAP-C02

***

### Understanding the Problem

The company needs to establish a **private, non-internet-traversing connection** from its AWS VPC to a **third-party Software-as-a-Service (SaaS) application** that also runs within an AWS VPC.

**Key Requirements:**

1.  **Private Connectivity:** Connection must not traverse the internet.
2.  **Internal Access Only:** No resources in the company VPC should be accessible from outside the VPC (including from the SaaS provider, unless explicitly configured for the API access).
3.  **Least Privilege:** All permissions must adhere to the principle of least privilege.
4.  **Consumption Method:** The SaaS is consumed via **API calls**.

***

### Correct Option(s) Explanation

**A. Create an AWS PrivateLink interface VPC endpoint. Connect this endpoint to the endpoint service that the third-party SaaS application provides. Create a security group to limit the access to the endpoint. Associate the security group with the endpoint.**

This option is the **correct solution** because it directly meets all requirements:

1.  **Private Connectivity:** An **AWS PrivateLink interface VPC endpoint** establishes a **private connection** over the Amazon network, ensuring traffic does **not traverse the internet**. The connection is to an **Endpoint Service** hosted by the third-party provider.
2.  **Internal Access Only/Isolation:** PrivateLink is non-transitive. It creates a one-way connection where the consumer (company) can reach the provider's (SaaS) service, but the provider **cannot initiate a connection back** to the consumer's VPC. This offers the necessary isolation.
3.  **Least Privilege:** **Security Groups** associated with the interface VPC endpoint and the subnets hosting the application servers allow for granular control over the traffic (e.g., only port 443 from specific company subnets) thereby enforcing the **principle of least privilege**. The official AWS documentation on Interface VPC Endpoints and Endpoint Services confirms this architecture as the standard for secure, private SaaS consumption.

***

### Incorrect Options Explanation

**B. Create an AWS Site-to-Site VPN connection between the third-party SaaS application and the company VPC. Configure network ACLs to limit access across the VPN tunnels.**

* **Violates Security/SaaS Best Practice:** While VPNs can be private if integrated with AWS Direct Connect (not mentioned) or secure (encrypted) over the internet, establishing a direct VPN tunnel between two parties' VPCs (especially a third party) is not the standard, isolated, and scalable way to consume a SaaS API.
* **Lack of Isolation:** A VPN establishes a point-to-point network link, which, like VPC Peering, gives a greater degree of network access than desired for a typical SaaS API. It doesn't offer the inherent isolation of PrivateLink.

**C. Create a VPC peering connection between the third-party SaaS application and the company VPC. Update route tables by adding the needed routes for the peering connection.**

* **Violates Least Privilege/Isolation:** VPC Peering provides full, bidirectional, non-isolated network connectivity between the two VPCs (subject to route tables and security groups). It requires the company to trust the third-party provider's network security completely, as the third party could potentially access other resources in the company's VPC (subject to strict security group and NACL configuration, which is complex to maintain). This architecture violates the **principle of least privilege** and the isolation typically required when connecting to a third-party service. PrivateLink is designed to solve the security and isolation problems inherent in using VPC peering for SaaS.

**D. Create an AWS PrivateLink endpoint service. Ask the third-party SaaS provider to create an interface VPC endpoint for this endpoint service. Grant permissions for the endpoint service to the specific account of the third-party SaaS provider.**

* **Role Reversal:** This option describes the **reverse scenario**. The company is the *consumer* of the SaaS, not the *provider*. An **Endpoint Service** is what the *provider* (SaaS application) creates to *offer* their service. The **Interface VPC Endpoint** is what the *consumer* (company) creates to *consume* the service. Therefore, the steps are backwards for the company's role as the consumer.
