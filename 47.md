A company has a legacy monolithic application that is critical to the company's business. The company hosts the application on an Amazon EC2 instance that runs Amazon Linux 2. The company's application team receives a directive from the legal department to back up the data from the instance's encrypted Amazon  
Elastic Block Store (Amazon EBS) volume to an Amazon S3 bucket. The application team does not have the administrative SSH key pair for the instance. The application must continue to serve the users.  
Which solution will meet these requirements?  

- A. Attach a role to the instance with permission to write to Amazon S3. Use the AWS Systems Manager Session Manager option to gain access to the instance and run commands to copy data into Amazon S3.
- B. Create an image of the instance with the reboot option turned on. Launch a new EC2 instance from the image. Attach a role to the new instance with permission to write to Amazon S3. Run a command to copy data into Amazon S3.
- C. Take a snapshot of the EBS volume by using Amazon Data Lifecycle Manager (Amazon DLM). Copy the data to Amazon S3.
- D. Create an image of the instance. Launch a new EC2 instance from the image. Attach a role to the new instance with permission to write to Amazon S3. Run a command to copy data into Amazon S3.

## Analysis

### Understanding the Problem

The core requirement is to **back up data** from an existing, encrypted **Amazon EBS volume** (attached to a critical, running **Amazon EC2 instance**) to an **Amazon S3 bucket**.

Key constraints are:

1. **Critical Application:** The application **must continue to serve users** (zero downtime for the application is implied).
    
2. **Access Limitation:** The team **does not have the administrative SSH key pair** for the EC2 instance.
    
3. **Source/Destination:** Data is on an encrypted EBS volume, and the destination is S3.
    
4. **Security/Permission:** The solution must allow the instance to write data to S3.
    

The solution must address access without the SSH key, maintain application availability, and facilitate the data transfer.

### Correct Option(s) Explanation

**A. Attach a role to the instance with permission to write to Amazon S3. Use the AWS Systems Manager Session Manager option to gain access to the instance and run commands to copy data into Amazon S3.**

1. **Attach an IAM Role:** Attaching an **IAM Instance Profile (Role)** with an appropriate policy (e.g., allowing `s3:PutObject`) to the **existing, running EC2 instance** grants it the necessary permission to write data to the specified S3 bucket. This change is **non-disruptive** to the running application.
    
2. **AWS Systems Manager Session Manager:** The requirement states the team **does not have the SSH key pair**. **Session Manager** is the **official AWS recommended solution** for secure, **key-less access** to an EC2 instance, provided the **SSM Agent is running** (standard on Amazon Linux 2) and the **instance has an IAM role** allowing SSM operations. This enables the team to gain a shell/command line access.
    
3. **Execute the Copy Command:** Once connected via Session Manager, the team can use a tool like the **AWS CLI** (`aws s3 cp` command) to directly copy the files from the EBS volume (local filesystem) to the S3 bucket, **without disrupting the running application.**
    

This option meets _all_ the stated requirements:

- Application **continues to serve users** (non-disruptive operations).
    
- **Key-less access** is achieved via Session Manager.
    
- Data is moved from EBS volume to S3.
    

_(Reference: AWS documentation on Session Manager and its benefit of not needing key pairs or open ports; AWS documentation on IAM roles for EC2 instances to access other AWS services like S3.)_

### Incorrect Options Explanation

**B. Create an image of the instance with the reboot option turned on. Launch a new EC2 instance from the image. Attach a role to the new instance with permission to write to Amazon S3. Run a command to copy data into Amazon S3.**

- **Flaw:** Creating an AMI (image) with the **reboot option turned on** will cause the **running EC2 instance to reboot** during the image creation process. This violates the critical requirement that the **application must continue to serve the users** (zero downtime/disruption).
    

**C. Take a snapshot of the EBS volume by using Amazon Data Lifecycle Manager (Amazon DLM). Copy the data to Amazon S3.**

- **Flaw 1 (Snapshot vs. Data):** An EBS Snapshot is a **block-level backup** of the volume. While you can create a volume _from_ the snapshot, and _then_ copy the data from the new volume to S3, this option **does not describe a mechanism to directly copy the _data_ from the snapshot itself to S3.** Data copy requires the file system to be mounted on an instance.
    
- **Flaw 2 (Access/Transfer):** This option does not address how the data transfer _to S3_ will be performed, nor does it address the initial problem of the lack of administrative access to the _original_ instance to perform the copy _from the source_.
    

**D. Create an image of the instance. Launch a new EC2 instance from the image. Attach a role to the new instance with permission to write to Amazon S3. Run a command to copy data into Amazon S3.**

- **Flaw:** Creating an AMI (image) of a running Linux instance **without the reboot option** _may_ be non-disruptive, but the **data copy is performed on a _new_ instance**. This is unnecessarily complex, as it involves:
    
    1. Creating a new AMI (which is a full OS+data backup).
        
    2. Launching a new EC2 instance (new compute cost).
        
    3. Performing the copy.
        
    4. Terminating the new instance.
        
        The primary goal is simply to get the data from the running instance to S3. Option A is significantly more efficient and direct, utilizing the existing compute resource and solving the key-less access problem directly with Systems Manager. While technically possible, Option D is not the best solution in an AWS Operations/SAP context where efficiency and simplicity are prized. Option A directly addresses the constraints (no key) and the task (data copy) on the source system with minimal overhead.
        

_(Reference: AWS documentation on AMI creation process, noting the potential for I/O pauses or the need to reboot; General principle of using the least complex and most efficient non-disruptive method.)_

---

### Conclusion

Option A leverages core AWS operational best practices by using **IAM Roles** for secure authorization and **Systems Manager Session Manager** for secure, key-less administrative access, enabling the direct, non-disruptive data transfer to S3. This is the most operationally sound and cost-effective solution that meets all requirements.
