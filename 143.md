A company runs a processing engine in the AWS Cloud. The engine processes environmental data from logistics centers to calculate a sustainability index. The company has millions of devices in logistics centers that are spread across Europe. The devices send information to the processing engine through a RESTful API.

The API experiences unpredictable bursts of traffic. The company must implement a solution to process all data that the devices send to the processing engine. Data loss is unacceptable.

Which solution will meet these requirements?

A. Create an Application Load Balancer (ALB) for the RESTful API. Create an Amazon Simple Queue Service (Amazon SQS) queue. Create a listener and a target group for the ALB Add the SQS queue as the target. Use a container that runs in Amazon Elastic Container Service (Amazon ECS) with the Fargate launch type to process messages in the queue.
B. Create an Amazon API Gateway HTTP API that implements the RESTful API. Create an Amazon Simple Queue Service (Amazon SQS) queue. Create an API Gateway service integration with the SQS queue. Create an AWS Lambda function to process messages in the SQS queue.
C. Create an Amazon API Gateway REST API that implements the RESTful API. Create a fleet of Amazon EC2 instances in an Auto Scaling group. Create an API Gateway Auto Scaling group proxy integration. Use the EC2 instances to process incoming data.
D. Create an Amazon CloudFront distribution for the RESTful API. Create a data stream in Amazon Kinesis Data Streams. Set the data stream as the origin for the distribution. Create an AWS Lambda function to consume and process data in the data stream.

Suggested Answer: B

## 📋 Analysis

### 1. Understanding the Problem

The company operates a **global IoT data ingestion pipeline** with these characteristics:

- **Millions of devices** across Europe sending environmental data via a RESTful API.
- **Unpredictable traffic bursts** — the ingestion layer must absorb sudden spikes without throttling or rejecting requests.
- **Zero data loss** is the paramount requirement — every device message must eventually reach the processing engine.
- The processing engine is the downstream consumer — the ingestion architecture must decouple API receipt from data processing.

The core architectural pattern needed here is **message buffering/decoupling**: the API accepts data quickly, persists it in a buffer, and processing happens asynchronously.

---

### 2. Options Analysis

#### ✅ **Option B (CORRECT): API Gateway HTTP API → SQS Service Integration → Lambda**

This is the **optimal serverless ingestion pattern**:

1. **Amazon API Gateway HTTP API** provides a fully managed RESTful endpoint. API Gateway automatically scales to handle any traffic volume, is highly available, and supports throttling/burst controls.
2. **SQS service integration** — API Gateway HTTP API supports **direct AWS service integrations** (without Lambda). This means the API can write messages directly into an SQS queue using a native integration. No intermediate compute is needed at the ingestion point — reducing latency, cost, and failure points.
3. **Amazon SQS** acts as a **durable buffer**. Once a message is in SQS, it is persisted redundantly across multiple AZs. If downstream processing slows down, messages safely accumulate in the queue. There is no data loss.
4. **AWS Lambda** polls the SQS queue and processes messages. Lambda's SQS integration automatically scales the number of pollers based on the queue depth.

**Why this is the best solution:**
- **Durable decoupling:** API Gateway accepts the request, writes to SQS, returns success. Even if the processing engine is down, no data is lost — it's safely in SQS.
- **Direct service integration** eliminates a Lambda function at the API layer — reducing cost, latency, and failure points.
- **Automatic scaling at every layer:** API Gateway → SQS → Lambda all scale independently and elastically.
- **Unpredictable bursts are absorbed by SQS** — Lambda can process at its own pace.

#### ❌ **Option A (INCORRECT): ALB + SQS as ALB Target + ECS Fargate**

- **ALB does not support SQS as a target.** Application Load Balancer supports these target types: EC2 instances, IP addresses, Lambda functions, and (as of 2023) NLBs. SQS is NOT a supported ALB target.
- To send ALB traffic to SQS, you would need a Lambda function as an intermediary (ALB → Lambda → SQS), which is not described in this option.
- **Architectural flaw:** Even if it worked, placing an ALB in front of SQS is the wrong pattern — ALBs distribute traffic across targets; SQS is a queue, not a target pool.

#### ❌ **Option C (INCORRECT): API Gateway REST API + EC2 Auto Scaling Proxy Integration**

- **No message buffer:** API Gateway → EC2 directly. If the EC2 fleet cannot keep up with a burst, requests fail (HTTP 5xx errors from overwhelmed backend). This violates the "zero data loss" requirement.
- **EC2 scaling latency:** Auto Scaling takes minutes to react to traffic spikes. During unpredictable bursts, instances won't scale fast enough — data will be lost.
- **No decoupling:** The API is tightly coupled to the processing tier. Any processing slowdown directly impacts API availability.
- API Gateway REST API proxy integration is designed for synchronous request-response patterns, not asynchronous data ingestion at IoT scale.

#### ❌ **Option D (INCORRECT): CloudFront + Kinesis Data Streams as Origin**

- **CloudFront cannot use Kinesis Data Streams as an origin.** CloudFront origins are: S3 buckets, MediaStore containers, MediaPackage channels, Application Load Balancers, API Gateway endpoints, EC2 instances, and custom HTTP origins. Kinesis Data Streams is NOT a supported CloudFront origin.
- CloudFront is a CDN — optimized for caching and delivering content to users, not for ingesting data from devices. Using a CDN for data ingestion is an anti-pattern.
- Even if somehow connected, Kinesis Data Streams requires proper shard provisioning to handle bursts — without SQS as a buffer, write throttling (`WriteThroughputExceeded`) could cause data loss during bursts.

### 3. Hands-On Knowledge

#### Optimal IoT Data Ingestion Architecture on AWS

```
Millions of Devices (RESTful API)
        │
        ▼
┌───────────────────┐
│  API Gateway      │  ← Fully managed, auto-scaling endpoint
│  HTTP API         │
└───────┬───────────┘
        │ Direct AWS Service Integration (no Lambda!)
        ▼
┌───────────────────┐
│  SQS Standard     │  ← Durable buffer, ~unlimited throughput
│  Queue            │     4+ days retention, at-least-once delivery
└───────┬───────────┘
        │ Lambda polls SQS (event source mapping)
        ▼
┌───────────────────┐
│  Lambda           │  ← Auto-scaling processing
│  (Consumer)       │
└───────┬───────────┘
        ▼
┌───────────────────┐
│  Processing       │
│  Engine           │
└───────────────────┘
```

#### API Gateway HTTP API – SQS Service Integration

The key advantage is the **direct integration** (no Lambda needed for the API-to-SQS hop):

**HTTP API Route Configuration (AWS CLI):**
```bash
# 1. Create HTTP API
aws apigatewayv2 create-api \
  --name iot-ingestion-api \
  --protocol-type HTTP

# 2. Create route with SQS integration
aws apigatewayv2 create-integration \
  --api-id abc123 \
  --integration-type AWS_PROXY \
  --integration-subtype SQS-SendMessage \
  --payload-format-version 1.0 \
  --credentials-arn arn:aws:iam::account:role/api-gateway-sqs-role \
  --request-parameters '{
    "QueueUrl": "https://sqs.region.amazonaws.com/account/iot-ingestion-queue",
    "MessageBody": "$request.body"
  }'

aws apigatewayv2 create-route \
  --api-id abc123 \
  --route-key "POST /data" \
  --target "integrations/def456"
```

**Why this pattern guarantees zero data loss:**

| Layer | Data Durability Mechanism |
|-------|--------------------------|
| API Gateway | Returns 200 OK only after SQS accepts the message |
| SQS | Messages stored redundantly across ≥3 AZs. Retention up to 14 days. |
| Lambda | Automatic retries. Failed messages returned to queue or sent to DLQ. |
| SQS DLQ | Failed-after-retries messages moved to separate queue for review. |

**Standard Queue vs. FIFO Queue for IoT:**
- **SQS Standard** is preferred for this use case: nearly unlimited TPS, at-least-once delivery, best-effort ordering.
- **SQS FIFO** could be used for exactly-once processing but has a 300 TPS limit (or 3,000 with high-throughput mode), which may not handle millions of devices.
- For most IoT scenarios, Standard queues with idempotent processing (handling duplicates in Lambda) is the recommended pattern.

**Key Distinction: API Gateway REST API vs. HTTP API vs. ALB**

| Feature | REST API | HTTP API | ALB |
|---------|----------|----------|-----|
| SQS Service Integration | ✅ | ✅ | ❌ |
| Kinesis Service Integration | ✅ | ❌ | ❌ |
| Usage plans / API keys | ✅ | ❌ | ❌ |
| WAF integration | ✅ | ❌ | ✅ |
| Cost | Higher | Lower (~70% cheaper) | Per-hour |
| Latency | Higher | Lower | Lowest |
