A company wants to optimize AWS data-transfer costs and compute costs across developer accounts within the company's organization in AWS Organizations. Developers can configure VPCs and launch Amazon EC2 instances in a single AWS Region. The EC2 instances retrieve approximately 1 TB of data each day from Amazon S3.

The developer activity leads to excessive monthly data-transfer charges and NAT gateway processing charges between EC2 instances and S3 buckets, along with high compute costs. The company wants to proactively enforce approved architectural patterns for any EC2 instance and VPC infrastructure that developers deploy within the AWS accounts. The company does not want this enforcement to negatively affect the speed at which the developers can perform their tasks.

Which solution will meet these requirements MOST cost-effectively?

A. Create SCPs to prevent developers from launching unapproved EC2 instance types. Provide the developers with an AWS CloudFormation template to deploy an approved VPC configuration with S3 interface endpoints. Scope the developers' IAM permissions so that the developers can launch VPC resources only with CloudFormation.
B. Create a daily forecasted budget with AWS Budgets to monitor EC2 compute costs and S3 data-transfer costs across the developer accounts. When the forecasted cost is 75% of the actual budget cost, send an alert to the developer teams. If the actual budget cost is 100%, create a budget action to terminate the developers' EC2 instances and VPC infrastructure.
C. Create an AWS Service Catalog portfolio that users can use to create an approved VPC configuration with S3 gateway endpoints and approved EC2 instances. Share the portfolio with the developer accounts. Configure an AWS Service Catalog launch constraint to use an approved IAM role. Scope the developers' IAM permissions to allow access only to AWS Service Catalog.
D. Create and deploy AWS Config rules to monitor the compliance of EC2 and VPC resources in the developer AWS accounts. If developers launch unapproved EC2 instances or if developers create VPCs without S3 gateway endpoints, perform a remediation action to terminate the unapproved resources.


Suggest Answer: C

---

## Analysis

### Understanding the Problem

A company uses AWS Organizations with multiple developer accounts. Developers have broad freedom to configure VPCs and launch EC2 instances in a single AWS Region. The EC2 instances collectively retrieve approximately 1 TB of data per day from Amazon S3. This architecture has led to **three cost problems**: (1) excessive NAT gateway data processing charges (since traffic from private EC2 instances to S3 traverses the NAT Gateway at $0.045/GB), (2) excessive cross-AZ or inter-VPC data transfer charges, and (3) high compute costs from developers choosing unapproved (likely oversized) EC2 instance types. The company wants to **proactively enforce approved architectural patterns** — particularly ensuring that EC2 → S3 traffic uses cost-optimized paths — without slowing down developers. The key cost insight: $1 \text{ TB/day} \times 30 \text{ days} = 30 \text{ TB/month}$. NAT Gateway data processing at $0.045/GB = $\$1,382$/month just for S3 traffic that could be free via a VPC gateway endpoint.

### Correct Option: C — AWS Service Catalog + S3 Gateway Endpoints + Launch Constraint

**Why C is correct:**

- **AWS Service Catalog** enables organizations to create and manage curated catalogs of IT services (products) that are approved for use. By defining a portfolio containing an approved VPC configuration (with S3 gateway endpoints) and approved EC2 instance types, the company **proactively enforces** architectural standards — developers can only launch what's in the catalog.
- **S3 Gateway Endpoints** are the critical cost optimization: They provide a private, free path from VPC resources to Amazon S3 without traversing the public internet or a NAT Gateway. Data transfer through a gateway endpoint incurs **no additional data processing charges** (unlike NAT Gateway at $0.045/GB). For 1 TB/day, this saves ~$1,382/month in NAT Gateway data processing alone, plus the NAT Gateway hourly charges.
- **S3 Gateway Endpoint vs. Interface Endpoint**: Gateway endpoints are **free** and route traffic via AWS's backbone network using route table entries. Interface endpoints (AWS PrivateLink) cost $0.01/hour per endpoint + $0.01/GB data processed — for 1 TB/day, interface endpoints would still incur significant costs. The exam expects you to know that **S3 and DynamoDB use gateway endpoints**, which are the cost-optimized choice.
- **Launch constraint**: Configuring a launch constraint with an approved IAM role ensures that all resources provisioned through Service Catalog have the correct, least-privilege permissions. This prevents developers from inadvertently (or intentionally) creating resources with overly permissive roles.
- **IAM permissions scoped to Service Catalog only**: By restricting developers' IAM permissions to allow resource creation **only through Service Catalog**, the company enforces the approved patterns without preventing developers from self-serving. Developers can still provision the resources they need, but they must use the pre-approved templates.
- **Proactive, not reactive**: Unlike AWS Config rules (Option D) which detect and remediate violations after they occur, Service Catalog prevents non-compliant resources from being created in the first place.

**Official reference**: [AWS Service Catalog](https://docs.aws.amazon.com/servicecatalog/latest/adminguide/introduction.html) — "AWS Service Catalog allows organizations to create and manage catalogs of IT services that are approved for use on AWS. These IT services can include everything from virtual machine images, servers, software, and databases to complete multi-tier application architectures."

### Incorrect Options

**A — SCPs + CloudFormation template + IAM scoped to CloudFormation + S3 Interface Endpoints**

- **SCPs (Service Control Policies)** can prevent developers from launching unapproved EC2 instance types, but they are a **negative control** (deny list) — they stop bad behavior but don't guide developers toward the right patterns. Developers might still create VPCs without S3 endpoints and scratch their heads about why S3 access is slow or expensive.
- **S3 Interface Endpoints** (AWS PrivateLink) are the **wrong endpoint type for cost optimization**. Interface endpoints cost $0.01/hour/AZ + $0.01/GB processed. For 1 TB/day with a multi-AZ deployment: ~$21.90/month in hourly charges + ~$300/month in data processing = still substantial cost. The correct choice for S3 is a **gateway endpoint** (free).
- **CloudFormation + IAM scoping**: While restricting IAM to only allow CloudFormation (`cloudformation:CreateStack`) is a valid governance pattern, it's less user-friendly than Service Catalog. Developers need to write or locate the correct CloudFormation template, manage parameters, and understand stack lifecycle. Service Catalog provides a more curated, guided experience.
- The combination of SCPs (deny) + CloudFormation (DIY templates) provides weaker guardrails than Service Catalog's curated portfolio approach.

**B — AWS Budgets with alerts and termination actions**

- **Reactive, not proactive**: Budgets alert when costs exceed thresholds — they don't prevent the costly behavior from occurring. By the time the alert fires at 75%, the company has already incurred significant unnecessary charges.
- **Terminating resources at 100% budget** is destructive and would severely disrupt developer productivity. This contradicts the requirement "does not want this enforcement to negatively affect the speed at which the developers can perform their tasks."
- Budgets are a financial governance tool, not an architectural governance tool. They don't enforce VPC design patterns or S3 access paths — they just measure spend.
- For a company concerned about architectural patterns (S3 endpoints, instance types), monitoring spend alone doesn't address the root cause.

**D — AWS Config rules + remediation actions**

- **Reactive (detect-and-remediate)**: AWS Config evaluates resources against rules and can trigger remediation (e.g., terminate non-compliant EC2 instances or delete VPCs without S3 endpoints). However, the violation has already occurred — resources were created, and costs were incurred before remediation kicks in.
- **Negatively affects developer speed**: If a developer provisions an EC2 instance for testing and it gets terminated by a Config remediation action, this is highly disruptive. Developers may lose work, and trust in the platform erodes.
- Config rules have an evaluation latency (rules can take up to 15 minutes to evaluate after a resource change) — this creates a window where non-compliant resources exist and incur costs.
- While Config rules are valuable for **continuous compliance monitoring**, they are not the best tool for **proactive enforcement** — Service Catalog is purpose-built for that use case.

### Hands-On Enterprise Knowledge

1. **VPC Endpoint Type Selection Guide**:
   | Service | Endpoint Type | Cost | Use Case |
   |---|---|---|---|
   | Amazon S3 | Gateway | Free | Route traffic via route table; no PrivateLink |
   | Amazon DynamoDB | Gateway | Free | Route traffic via route table; no PrivateLink |
   | All other AWS services | Interface (PrivateLink) | $0.01/hr/AZ + $0.01/GB | Private connectivity to AWS APIs |

   **Exam tip**: S3 and DynamoDB are the **only two services** that support gateway endpoints. For all other services, you must use interface endpoints (PrivateLink).

2. **Enterprise multi-account governance strategy**: A mature AWS Organization typically implements a layered governance model:
   - **Layer 1 — SCPs**: Broad guardrails (e.g., deny ability to leave the organization, deny CloudTrail deletion)
   - **Layer 2 — Service Catalog**: Curated, approved products that developers can self-serve (proactive)
   - **Layer 3 — AWS Config**: Continuous compliance monitoring and drift detection (detective)
   - **Layer 4 — Budgets/Cost Explorer**: Financial governance and anomaly detection

3. **Service Catalog launch constraint deep dive**: The launch constraint defines an IAM role that Service Catalog assumes when provisioning a product. This role:
   - Must have permissions to create all resources defined in the CloudFormation template
   - Is assumed by Service Catalog, not the end user — so the end user doesn't need direct permissions to create EC2, VPCs, etc.
   - Enables the principle of **least privilege at scale** — developers only have `servicecatalog:ProvisionProduct` permission

4. **Cost calculation for S3 data transfer**: 
   - NAT Gateway → S3 (through public internet): $0.045/GB (NAT GW data processing) + $0.00/GB (S3 data transfer in is free) = $0.045/GB
   - S3 Gateway Endpoint: $0.00/GB (free)
   - S3 Interface Endpoint: $0.01/GB (data processed by the endpoint)
   - For 1 TB/day (30 TB/month): Gateway Endpoint saves $1,350/month vs NAT Gateway, and $300/month vs Interface Endpoint

5. **Real-world implementation sequence**:
   1. Design the approved VPC CloudFormation template (with S3 gateway endpoints, proper subnet design, routing)
   2. Create an AWS Service Catalog portfolio and add the product
   3. Configure a launch constraint role with appropriate permissions
   4. Share the portfolio with developer accounts (via Organizations or account-level sharing)
   5. Update developer IAM policies to restrict EC2/VPC creation to Service Catalog only
   6. Gradually migrate existing non-compliant resources to the new approved pattern
