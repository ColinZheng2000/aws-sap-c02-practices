A financial services company has an asset management product that thousands of customers use around the world. The customers provide feedback about the product through surveys. The company is building a new analytical solution that runs on Amazon EMR to analyze the data from these surveys. The following user personas need to access the analytical solution to perform different actions:

• Administrator: Provisions the EMR cluster for the analytics team based on the team’s requirements
• Data engineer: Runs ETL scripts to process, transform, and enrich the datasets
• Data analyst: Runs SQL and Hive queries on the data

A solutions architect must ensure that all the user personas have least privilege access to only the resources that they need. The user personas must be able to launch only applications that are approved and authorized. The solution also must ensure tagging for all resources that the user personas create.

Which solution will meet these requirements?

A. Create IAM roles for each user persona. Attach identity-based policies to define which actions the user who assumes the role can perform. Create an AWS Config rule to check for noncompliant resources. Configure the rule to notify the administrator to remediate the noncompliant resources.
B. Setup Kerberos-based authentication for EMR clusters upon launch. Specify a Kerberos security configuration along with cluster-specific Kerberos options.
C. Use AWS Service Catalog to control the Amazon EMR versions available for deployment, the cluster configuration, and the permissions for each user persona.
D. Launch the EMR cluster by using AWS CloudFormation, Attach resource-based policies to the EMR cluster during cluster creation. Create an AWS. Config rule to check for noncompliant clusters and noncompliant Amazon S3 buckets. Configure the rule to notify the administrator to remediate the noncompliant resources.



# SAP-C02 Analysis — AWS Service Catalog for Governance and Least Privilege EMR Provisioning

## Understanding the Problem

This question focuses on:

- Multi-persona access control
- Amazon EMR governance
- Least privilege access
- Restricting approved applications/configurations
- Mandatory tagging enforcement
- Enterprise operational governance

Three personas:

| Persona | Responsibility |
|---|---|
| Administrator | Provision EMR clusters |
| Data Engineer | Run ETL jobs |
| Data Analyst | Run SQL/Hive queries |

The key requirements are:

1. Least privilege access
2. Only approved applications/configurations allowed
3. Enforce tagging standards
4. Governance over what users can provision

This is not only an IAM question.

This is fundamentally:
# centralized governed provisioning

The AWS-native service designed for this is:
# AWS Service Catalog

---

# Correct Option

## C. Use AWS Service Catalog to control EMR versions, cluster configuration, and permissions

---

## Why This Is Correct

AWS Service Catalog is specifically designed for:

- standardized deployments
- approved infrastructure products
- governance
- access control
- mandatory tagging
- enterprise self-service provisioning

This directly satisfies ALL requirements.

---

# Why Service Catalog Fits Perfectly

## 1. Restricting Approved Applications

Requirement:
> users must be able to launch only approved and authorized applications

Service Catalog allows administrators to publish:
- approved EMR cluster templates
- approved EMR versions
- approved applications/configurations

Users can only deploy:
- catalog-approved products

This is a classic enterprise governance use case.

---

## 2. Least Privilege Access

Service Catalog integrates with:
- IAM
- portfolios
- launch constraints

You can define:
- which personas can launch which EMR products

Examples:

| Persona | Permissions |
|---|---|
| Admin | Full EMR provisioning |
| Data Engineer | ETL-capable clusters |
| Analyst | Query-only clusters |

---

## 3. Enforcing Tagging

Service Catalog supports:
- TagOptions
- mandatory resource tags
- standardized metadata

This directly addresses:
> ensure tagging for all resources

This is a major clue pointing to Service Catalog.

---

## 4. Standardized EMR Deployments

Service Catalog products can define:
- cluster size
- EMR release version
- networking
- encryption
- security configuration
- installed applications

This ensures:
- operational consistency
- security compliance
- version governance

# Incorrect Options

---

## A. IAM Roles + AWS Config

### Why Incorrect

This partially addresses:
- least privilege

But fails to fully address:
- restricting approved applications
- controlled provisioning
- governance at deployment time

AWS Config:
- detects noncompliance AFTER deployment

The requirement needs:
- preventative governance
not:
- reactive remediation

Key SAP exam distinction:

| Approach | Type |
|---|---|
| AWS Config | Detective |
| Service Catalog | Preventive governance |

Also:
- IAM alone does not standardize EMR configurations

---

## B. Kerberos Authentication

### Why Incorrect

Kerberos addresses:
- authentication within EMR clusters

Useful for:
- Hadoop ecosystem security
- Hadoop service authentication

But it does NOT solve:
- provisioning governance
- approved applications
- tagging enforcement
- deployment restrictions

This option distracts candidates using EMR-specific terminology.

---

## D. CloudFormation + Resource Policies + AWS Config

### Why Incorrect

CloudFormation helps with:
- infrastructure as code

But this still lacks:
- controlled self-service governance
- persona-based catalog access
- standardized approved products

Problems:

1. EMR primarily uses IAM identity-based access
   - not resource-based policies in the way implied

2. AWS Config remains reactive

3. Users could still deploy unauthorized stacks unless additional governance layers exist

Service Catalog is effectively:
- governed CloudFormation at enterprise scale

That is the architectural distinction.

# Important SAP-C02 Governance Concept

## Preventive vs Detective Controls

This question strongly tests:

| Control Type | AWS Service |
|---|---|
| Preventive | Service Catalog / SCPs |
| Detective | AWS Config |
| Corrective | Automation remediation |

The requirement:
> users can launch ONLY approved applications

means:
# preventive control required

Therefore:
- Config alone is insufficient
