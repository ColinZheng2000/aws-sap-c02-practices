A large mobile gaming company has successfully migrated all of its on-premises infrastructure to the AWS Cloud. A solutions architect is reviewing the environment to ensure that it was built according to the design and that it is running in alignment with the Well-Architected Framework.
While reviewing previous monthly costs in Cost Explorer, the solutions architect notices that the creation and subsequent termination of several large instance types account for a high proportion of the costs. The solutions architect finds out that the company's developers are launching new Amazon EC2 instances as part of their testing and that the developers are not using the appropriate instance types.
The solutions architect must implement a control mechanism to limit the instance types that only the developers can launch.
Which solution will meet these requirements?

A. Create a desired-instance-type managed rule in AWS Config. Configure the rule with the instance types that are allowed. Attach the rule to an event to run each time a new EC2 instance is launched.
B. In the EC2 console, create a launch template that specifies the instance types that are allowed. Assign the launch template to the developers' IAM accounts.
C. Create a new IAM policy. Specify the instance types that are allowed. Attach the policy to an IAM group that contains the IAM accounts for the developers
D. Use EC2 Image Builder to create an image pipeline for the developers and assist them in the creation of a golden image.

## 1. Understanding the Problem

- The company is fully on AWS and is now reviewing the environment against the **AWS Well-Architected Framework**.
- Cost Explorer shows **high costs** from:
  - Creation and termination of **large EC2 instance types** used by **developers for testing**.
- The specific requirement:
  - Implement a **control mechanism** to **limit which EC2 instance types developers can launch**.
  - The control should apply **only to developers**, not necessarily to other roles (e.g., production ops).

Key points:

- This is a **governance and access control** problem, not just monitoring.
- The solution must:
  - **Prevent** developers from launching disallowed instance types (proactive control).
  - Be **enforceable via AWS-native mechanisms** (IAM, policies, etc.).
  - Align with **Cost Optimization** and **Security / IAM** pillars of the Well-Architected Framework.

## 2. Services and Concepts Involved

### 2.1 AWS Identity and Access Management (IAM)

- Service to control **who can do what** on which resources.
- Uses **policies** (JSON documents) with `Effect`, `Action`, `Resource`, and `Condition`.
- You can constrain **EC2 instance types** using IAM policy conditions:
  - Example condition key: `ec2:InstanceType`
- Attach policies to:
  - **Users**
  - **Groups**
  - **Roles**

### 2.2 Amazon EC2

- Developers are launching **new EC2 instances**.
- Instance types vary in:

## 3. Why Option C Is Correct (Insightful Explanation)

> **C. Create a new IAM policy. Specify the instance types that are allowed. Attach the policy to an IAM group that contains the IAM accounts for the developers**

### Mechanism

You can define an IAM policy that:

- **Allows** developers to run EC2 instances as normal.
- **Denies** `RunInstances` if the requested instance type is not in an approved list.

A typical pattern is:

1. **Allow** EC2 usage generally (or more targeted permissions).
2. Add an explicit **Deny with conditions**:
   - Checks `ec2:InstanceType`
   - If the instance type is NOT one of the allowed types → deny.

Example IAM policy snippet (conceptual):

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowStandardEC2Actions",
      "Effect": "Allow",
      "Action": [
        "ec2:Describe*",
        "ec2:RunInstances",
        "ec2:TerminateInstances",
        "ec2:StartInstances",
        "ec2:StopInstances"
      ],
      "Resource": "*"
    },
    {
      "Sid": "DenyDisallowedInstanceTypes",
      "Effect": "Deny",
      "Action": "ec2:RunInstances",
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "ec2:InstanceType": [
            "t3.small",
            "t3.medium",
            "t3.large"
          ]
        }
      }
    }
  ]
}
```

- Any attempt to launch an EC2 instance with an instance type not in the list will fail with **AccessDenied**.
- Attach this policy to an **IAM group** like `DevEC2Restricted`, then add all developer IAM users/roles to that group.

### Why This Fits the Requirements

- **Directly limits instance types**: uses IAM condition key `ec2:InstanceType`.
- **Scoped only to developers**: by attaching the policy only to developer IAM group.
- **Proactive control**: Prevents non-compliant instances from ever launching.
- **Aligns with Well-Architected**:
  - **Cost Optimization**: prevents oversized instances for dev/test.
  - **Security / IAM**: least privilege and fine-grained access controls.
  - **Operational Excellence**: centralized, policy-driven governance.

- IAM policy evaluation and explicit deny:

## 4. Why Other Options Are Incorrect (Insightful Explanation)

### A. AWS Config managed rule with desired-instance-type

> **A. Create a desired-instance-type managed rule in AWS Config. Configure the rule with the instance types that are allowed. Attach the rule to an event to run each time a new EC2 instance is launched.**

- **Issue 1 – AWS Config is detective, not preventive**:
  - Config can detect non-compliant EC2 instances **after they are launched**, not before.
  - Developers can still launch any instance type, which may incur cost even if terminated soon after.
- **Issue 2 – Enforcement requires extra setup**:
  - To enforce, you would need an **automated remediation** (e.g., SSM Automation or Lambda) to stop/terminate non-compliant instances.
  - Even then, instances exist briefly and may still incur some cost and operational noise.
- **Does not satisfy the requirement of a direct “control mechanism” that limits what developers can launch**, in the sense of preventing incorrect launches upfront.

Conclusion: Good for **compliance and monitoring**, not ideal for **direct access control** of instance types.

Docs:  
- AWS Config rules and remediation:  
  https://docs.aws.amazon.com/config/latest/developerguide/evaluate-config_use-managed-rules.html  
  https://docs.aws.amazon.com/config/latest/developerguide/automatic-remediation.html

---

### B. Launch template assigned to developers’ IAM accounts

> **B. In the EC2 console, create a launch template that specifies the instance types that are allowed. Assign the launch template to the developers' IAM accounts.**

Several issues:

- **Launch templates do not by themselves restrict other launching methods**:
  - Developers can still use the console or CLI to click "Launch Instance" and bypass the template, unless **IAM policies** specifically force them to use only that template.
- **IAM does not “assign” a launch template to an IAM account**:
  - You can **authorize or deny** the `ec2:RunInstances` call when not using certain templates via IAM conditions, but that is not described here.
  - The answer does **not mention** any IAM policy enforcing usage of the launch template.
- This option alone is more about **standardization** than **enforcement**.

To truly enforce launch template usage, you would still need IAM conditions like:

```json
"Condition": {
  "StringEquals": {
    "ec2:LaunchTemplate": "lt-1234567890abcdef0"
  }
}
```

But the option does not include that; thus, it's incomplete and not sufficient to meet the requirement.

Docs:  
- Controlling EC2 launches using IAM and launch templates:  
  https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ExamplePolicies_EC2.html

---

### D. EC2 Image Builder golden image

> **D. Use EC2 Image Builder to create an image pipeline for the developers and assist them in the creation of a golden image.**

- EC2 Image Builder focuses on:
  - Building **standard AMIs** with specific OS, patches, and installed software.
  - Improving consistency and security baselines of images.
- It **does not limit instance type selection**:
  - A single AMI can be launched on many different instance types.
  - Developers could still choose `m5.24xlarge` or other large instances using the same golden image.
- This addresses **standardization** and **security/compliance of images**, not **cost control via instance-type restrictions**.

Conclusion: Useful for **image governance**; irrelevant to controlling which **instance types** are launched.

Docs:  
- EC2 Image Builder use cases:  
  https://docs.aws.amazon.com/imagebuilder/latest/userguide/what-is-image-builder.html

## 5. Summary

- The main problem: **Developers are launching overly large EC2 instance types**, causing **high costs**.
- The desired outcome: a **control** that **prevents** developers from launching disallowed instance types.
- The AWS-native, Well-Architected-aligned solution is:
  - Use **IAM policies** with `ec2:InstanceType` conditions.
  - Attach to an **IAM group** that includes all developers.

✅ **Correct option: C** – IAM policy limiting allowed instance types and attached to developer IAM group.  

This is **preventive**, **targeted to developers**, and directly supports **cost optimization** and **governance**.
