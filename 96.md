A company runs its application in the eu-west-1 Region and has one account for each of its environments: development, testing, and production. All the environments are running 24 hours a day, 7 days a week by using stateful Amazon EC2 instances and Amazon RDS for MySQL databases. The databases are between 500 GB and 800 GB in size.  
  
The development team and testing team work on business days during business hours, but the production environment operates 24 hours a day, 7 days a week. The company wants to reduce costs. All resources are tagged with an environment tag with either development, testing, or production as the key.  
  
What should a solutions architect do to reduce costs with the LEAST operational effort?

- A. Create an Amazon EventBridge rule that runs once every day. Configure the rule to invoke one AWS Lambda function that starts or slops instances based on me tag, day, and time.
- B. Create an Amazon EventBridge rule that runs every business day in the evening. Configure the rule to invoke an AWS Lambda function that stops instances based on the tag. Create a second EventBridge rule that runs every business day in the morning. Configure the second rule lo invoke another Lambda function that starts instances based on the tag.
- C. Create an Amazon EventBridge rule that runs every business day in the evening, Configure the rule to invoke an AWS Lambda function that terminates, instances based on the lag. Create a second EventBridge rule that runs every business day in the morning. Configure the second rule lo invoke another Lambda function that restores the instances from their last backup based on the tag.
- D. Create an Amazon EventBridge rule that runs every hour. Configure the rule to invoke one AWS Lambda function that terminates or restores instances from their last backup based on the tag. day, and time.

## Analysis

### 1. Understanding the Problem
The scenario presents a classic "cost optimization" challenge for non-production environments.
* **Current State:** Dev, Test, and Prod run 24/7.
* **Requirement:** Dev and Test only need to run during business hours (approx. 8-10 hours/day). Prod must remain 24/7.
* **Resources:** *Stateful* EC2 instances and large RDS databases (500GB - 800GB).
* **Goal:** Reduce costs with the *least operational effort*.
* **Key Constraints:** The solution must rely on the existing tags. The databases are large, meaning data restoration is time-consuming.

### 2. Services Involved

* **Amazon EventBridge (formerly CloudWatch Events):** A serverless event bus service. In this context, it is used as a **Scheduler** (cron job) to trigger actions at specific times (Morning/Evening).
    
* **AWS Lambda:** A compute service that runs code in response to events. Here, it is used to execute the AWS SDK API calls to modify the state of resources.
* **Amazon EC2 (State handling):** EC2 instances can be **Stopped** (shutdown but EBS volumes persist) and **Started** (booted up with the same EBS volumes/data).
* **Amazon RDS (State handling):** RDS instances can be **Stopped** for up to 7 days. While stopped, you are not charged for DB Instance hours, only for storage.

### 3. Correct Option Explanation

**Option B is the correct answer.**

* **Operational Efficiency:** This option separates the concerns into two distinct events: a "Stop" event in the evening and a "Start" event in the morning. This is the cleanest logic for a scheduler.
* **State Preservation:** By using the **Stop** and **Start** API actions, the solution preserves the data on the EBS volumes and the RDS storage without needing to perform backups or snapshots. The instance ID and IP configurations (if using Elastic IPs or private IPs in VPC) generally remain consistent or are easily managed.
* **Cost Savings:** Stopping instances (EC2 and RDS) when not in use stops the billing for compute hours, which is usually the largest portion of the bill.
* **Feasibility:** Lambda functions can easily query resources by Tag (`Environment: development`, `Environment: testing`) and issue `StopInstances` / `StopDBInstance` or `StartInstances` / `StartDBInstance` commands.

> **Ref:** **AWS Instance Scheduler** is a pre-built AWS Solution implementation that functions exactly like Option B. It uses EventBridge (or CloudWatch) rules to trigger Lambda functions to stop and start instances based on schedules.

### 4. Incorrect Options Explanation

* **Option A is incorrect.**
    * **Logic Flaw:** The option suggests running the rule *once every day*. If the rule runs at 8:00 AM to start instances, it cannot also run at 6:00 PM to stop them. You would need the rule to run *at least twice* a day (or hourly) for a single rule to handle both start and stop logic based on time checks.
    * **Complexity:** Even if the frequency were higher, putting "start" and "stop" logic into a single monolithic function triggered blindly increases code complexity compared to having dedicated "Start" and "Stop" triggers.

* **Option C is incorrect.**
    * **Operational Overhead & Risk:** This option suggests **Terminating** instances and **Restoring** from backup.
    * **Performance Impact:** Restoring an RDS database that is **500 GB to 800 GB** takes a significant amount of time (creating a new instance from a snapshot). The database would likely not be ready for the development team when they arrive in the morning.
    * **State Loss:** Terminating EC2 instances means losing the instance entirely. You would have to recreate the environment from an AMI every morning, which might drift from the previous day's stateful data if not managed perfectly.

* **Option D is incorrect.**
    * **Same issues as Option C:** It relies on **Terminating and Restoring** large datasets, which is inefficient and slow.
    * **Unnecessary Invocations:** Running the rule *every hour* adds unnecessary invocations (cost and log noise), whereas Option B runs exactly when needed.

---

## Key Takeaway for the Exam
When dealing with **Stateful** resources (EC2/RDS) for cost optimization in Dev/Test environments, always look for **Stop/Start** patterns. Avoid **Terminate/Restore** patterns unless the requirement explicitly states that environments must be immutable or rebuilt fresh daily. Terminating large databases is rarely a valid strategy for daily pausing.
