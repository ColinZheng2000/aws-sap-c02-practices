A company runs an ecommerce application in a single AWS Region. The application uses a five-node Amazon Aurora MySQL DB cluster to store information about customers and their recent orders. The DB cluster experiences a large number of write transactions throughout the day.

The company needs to replicate the data in the Aurora database to another Region to meet disaster recovery requirements. The company has an RPO of 1 hour.

Which solution will meet these requirements with the LOWEST cost?

A. Modify the Aurora database to be an Aurora global database. Create a second Aurora database in another Region.
B. Enable the Backtrack feature for the Aurora database. Create an AWS Lambda function that runs daily to copy the snapshots of the database to a backup Region.
C. Use AWS Database Migration Service (AWS DMS). Create a DMS change data capture (CDC) task that replicates the ongoing changes from the Aurora database to an Amazon S3 bucket in another Region.
D. Turn off automated Aurora backups. Configure Aurora backups with a backup frequency of 1 hour. Specify another Region as the destination Region. Select the Aurora database as the resource assignment.


Suggest Answer: C

---

## Analysis

### Understanding the Problem

An ecommerce company runs a single-Region application backed by a **five-node Amazon Aurora MySQL DB cluster** that handles a large volume of write transactions throughout the day. The database stores customer information and recent order data — both critical to business operations. The company requires a **disaster recovery (DR)** strategy to replicate this data to another AWS Region with:

- **RPO (Recovery Point Objective): 1 hour** — at most 1 hour of transactional data may be lost in a disaster.
- **Lowest possible cost** — the solution must be the most economical option among those that satisfy the RPO.

The core challenge is balancing the need for cross-region data protection with the **write-heavy workload**, which makes continuous replication solutions expensive. The 1-hour RPO window is moderate — not requiring near-real-time (seconds) replication, but also not tolerant of daily batch snapshots.

### Correct Option(s) Explanation

**Option C (Correct):** Use **AWS Database Migration Service (AWS DMS)** with a **change data capture (CDC)** task that replicates ongoing changes from the Aurora database to an **Amazon S3 bucket** in another Region.

**Why this works:**

- **AWS DMS CDC** continuously captures and replicates database changes (INSERT, UPDATE, DELETE) from the source Aurora cluster to a target. The target here is **Amazon S3** in the DR Region, which stores replicated data in formats like CSV or Parquet.
- **RPO of 1 hour is easily met:** DMS CDC operates continuously, with latency typically measured in seconds to minutes. This comfortably satisfies the 1-hour RPO (and actually exceeds it). The data arrives in the DR Region near-real-time.
- **Lowest cost among viable options:**
  - DMS requires a single **replication instance** (e.g., `dms.t3.medium` or `dms.c5.large`) — a fixed hourly cost. For moderate write throughput, a smaller instance suffices.
  - **S3 storage** is extremely cheap (~$0.023/GB/month for Standard tier), and you can use S3 Lifecycle policies to transition older change data to even cheaper tiers (e.g., S3 Infrequent Access or Glacier).
  - **No secondary Aurora cluster** running 24×7 — unlike Aurora Global Database, which requires a fully provisioned secondary cluster.
- **Restore process:** In a DR event, you restore data from the S3 change logs into a new Aurora cluster in the DR Region using `LOAD DATA FROM S3` or AWS Glue/ETL jobs, then redirect application traffic.

- **Reference:** [AWS DMS CDC for Aurora MySQL](https://docs.aws.amazon.com/dms/latest/userguide/CHAP_Source.MySQL.html) | [AWS DMS Target S3](https://docs.aws.amazon.com/dms/latest/userguide/CHAP_Target.S3.html) | [AWS DMS Pricing](https://aws.amazon.com/dms/pricing/)

### Incorrect Options Explanation

**Option A:** Modify the Aurora database to be an **Aurora Global Database** and create a second Aurora database in another Region.

- ❌ **Prohibitively expensive for 1-hour RPO:** Aurora Global Database provides sub-second replication with a fully provisioned secondary cluster (you pay for compute instances, storage, and cross-region replication I/O in the secondary Region). For a write-heavy five-node cluster, the secondary Region costs would be nearly equivalent to the primary — effectively **doubling the database cost**. This is extreme overkill when only a 1-hour RPO is required. Aurora Global Database is designed for RPOs measured in seconds and RTOs in minutes, not hours.
- ❌ **Cost comparison:** A five-node Aurora cluster (e.g., `db.r6g.xlarge` × 5) running 24×7 in a secondary Region costs thousands of dollars per month. DMS with a single replication instance costs a fraction of that.

**Option B:** Enable the **Backtrack** feature and create a Lambda function that runs **daily** to copy snapshots to a backup Region.

- ❌ **Two fatal flaws:**
  1. **Backtrack is not a DR tool:** Aurora Backtrack allows you to "rewind" a database cluster to a specific point in time *within the same cluster and Region* — it does not replicate data to another Region. It is designed for quick recovery from logical errors (e.g., a bad deployment or accidental DELETE without WHERE clause), not for cross-region disaster recovery.
  2. **Daily snapshots violate RPO:** Taking snapshots daily means the RPO would be up to **24 hours**, not 1 hour. In a disaster, up to a full day's worth of customer orders would be lost — far exceeding the 1-hour RPO requirement.
- ❌ Even if the Lambda function ran hourly, Backtrack is still irrelevant to cross-region DR and the custom-Lambda approach introduces operational overhead and reliability concerns.

**Option D:** Turn off automated Aurora backups. Configure Aurora backups with a backup frequency of 1 hour. Specify another Region as the destination Region.

- ❌ **Technical inaccuracy in the option:** Aurora's **automated backups** (the built-in RDS backup mechanism) operate on a **daily snapshot + continuous transaction log** model. There is **no "backup frequency of 1 hour" setting** for Aurora's native automated backup feature. You cannot configure Aurora native backups to run every hour — the snapshot component is daily, and the transaction logs provide PITR granularity down to 5 minutes *within the same Region*.
- ❌ **Cross-region snapshot copy limitation:** While you can manually or programmatically copy Aurora snapshots to another Region, doing this every hour would require custom scripting and management overhead. Each snapshot copy is a full copy operation (though incremental in storage billing), and managing this at scale with hourly frequency introduces complexity and failure modes that AWS DMS handles natively.
- ❌ **Turning off automated backups is dangerous:** Disabling automated backups removes point-in-time recovery (PITR) capability for the *primary* Region, eliminating a critical safety net for non-DR scenarios like accidental data deletion. This is not aligned with AWS operational best practices.

### Enterprise Hands-on Knowledge

1. **DMS CDC for Aurora MySQL Setup Checklist:**
   - Enable **binary logging** on the Aurora source cluster (set `binlog_format=ROW` in the DB cluster parameter group)
   - Create a DMS replication instance in the **source Region** (not the target) to minimize network latency
   - Configure the DMS source endpoint with the Aurora cluster **reader endpoint** (not the writer) to avoid impacting write performance
   - Select **S3** as the target endpoint in the DR Region, with `addColumnName=true` and `dataFormat=parquet` for efficient restores
   - Enable **CloudWatch monitoring** on the DMS task to track `CDCLatencySource` and `CDCLatencyTarget` metrics

2. **Cost Comparison (Real-World Estimates for Write-Heavy Workload):**

   | Solution | Estimated Monthly Cost | RPO | RTO |
   |---|---|---|---|
   | Aurora Global Database (5 nodes) | $8,000–15,000+ | < 1 sec | Minutes |
   | DMS CDC to S3 | $250–600 | Seconds | Hours |
   | Cross-Region Snapshot Copy (AWS Backup hourly) | $150–400 | ~1 hour | Hours |
   | Daily Manual Snapshots | $50–100 | 24 hours | Hours |

   DMS CDC to S3 provides the best balance of RPO compliance and cost for a 1-hour RPO requirement.

3. **S3 Data Lifecycle for DR Replication Data:**
   - Keep the latest 24 hours of CDC data in **S3 Standard** for immediate restore capability
   - Transition older CDC files to **S3 Infrequent Access** after 24 hours (saves ~40% on storage)
   - Archive to **S3 Glacier Instant Retrieval** after 7 days for audit and compliance
   - Use **S3 Object Lock** in compliance mode to protect DR data from deletion (ransomware protection)

4. **Restore Procedure from S3 CDC Data:**
   1. In the DR Region, launch a new Aurora cluster from the most recent full snapshot (taken periodically)
   2. Apply CDC changes stored in S3 sequentially to roll forward to the desired point in time
   3. This process should be scripted (e.g., using AWS Step Functions) and tested regularly as part of DR drills

5. **SAP-C02 Exam Pattern:** When you see:
   - RPO measured in hours (not minutes or seconds) → **exclude** Global Database / Global Tables
   - "Lowest cost" or "most cost-effective" → **exclude** fully-provisioned secondary infrastructure
   - Write-heavy workload → DMS CDC is a strong candidate due to its continuous, incremental replication model
   - The exam frequently tests the distinction between **Backtrack** (in-Region logical error recovery) and **cross-region DR** solutions
