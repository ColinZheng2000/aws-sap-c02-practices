A company owns a chain of travel agencies and is running an application in the AWS Cloud. Company employees use the application to search for information about travel destinations. Destination content is updated four times each year.
Two fixed Amazon EC2 instances serve the application. The company uses an Amazon Route 53 public hosted zone with a multivalue record of travel.example.com that returns the Elastic IP addresses for the EC2 instances. The application uses Amazon DynamoDB as its primary data store. The company uses a self-hosted Redis instance as a caching solution.
During content updates, the load on the EC2 instances and the caching solution increases drastically. This increased load has led to downtime on several occasions. A solutions architect must update the application so that the application is highly available and can handle the load that is generated by the content updates.

# AWS SAP-C02 Question Analysis — Q315

## 📋 Analysis

### 1. Understanding the Problem

The company currently has a fragile architecture:

- Two fixed EC2 instances directly exposed through Route 53 multivalue routing
- Self-managed Redis cache
- Heavy read traffic spikes during quarterly content updates
- Repeated downtime during those predictable peak events

The architecture problems are:

- No elastic compute scaling
- No true load balancer
- Operational burden from self-managed cache
- Predictable spikes are not proactively handled
- Single points of scaling bottlenecks

The solution must achieve:

- High availability
- Better traffic distribution
- Managed caching
- Automatic scaling during known traffic surges
- Reduced operational overhead

Because the backend database is DynamoDB, AWS-native acceleration technologies become highly relevant.

---

### 2. Options Analysis

#### ✅ Option A (CORRECT): DAX + ALB + Auto Scaling + scheduled scaling

This option fully addresses scalability, availability, and operational resilience.

Why this is correct:

- DynamoDB Accelerator (DAX)
  - Purpose-built in-memory cache for DynamoDB
  - Reduces read latency from milliseconds to microseconds
  - Offloads heavy read traffic from DynamoDB during content update spikes
  - Fully managed and Multi-AZ capable

- Auto Scaling group
  - Replaces fixed EC2 capacity
  - Enables horizontal scaling during traffic increases
  - Improves fault tolerance

- Application Load Balancer (ALB)
  - Provides true Layer 7 load balancing
  - Performs health checks
  - Routes traffic only to healthy instances
  - Integrates natively with Auto Scaling

- Route 53 alias to ALB
  - Standard AWS production architecture
  - Cleaner and more resilient than multivalue EC2 IP routing

- Scheduled scaling
  - Traffic spikes occur only four times yearly
  - Predictable events are ideal for proactive scaling
  - Prevents cold-start delays and reactive scaling lag

This is the most operationally mature architecture.

Official references:
- DAX:
  https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/DAX.html
- Scheduled scaling:
  https://docs.aws.amazon.com/autoscaling/ec2/userguide/ec2-auto-scaling-scheduled-scaling.html
- ALB:
  https://docs.aws.amazon.com/elasticloadbalancing/latest/application/introduction.html

---

#### ❌ Option B (INCORRECT): ElastiCache Redis + CloudFront + manual scaling

This option contains architectural flaws.

Problems:

- CloudFront is not the appropriate core traffic management component here.
  - The workload is primarily dynamic application traffic.
  - ALB is required for backend request balancing.

- Auto Scaling group cannot directly act as a CloudFront origin.
  - CloudFront origins are usually:
    - ALB
    - S3
    - EC2/custom DNS endpoint

- Manual scaling is operationally weak.
  - The traffic pattern is predictable.
  - Scheduled scaling should be used instead.

- Although ElastiCache Redis is valid technically, DAX is the better fit for DynamoDB-centric workloads.

Main failure:
- Incorrect traffic distribution architecture
- Missing ALB
- Manual operations instead of automation

---

#### ❌ Option C (INCORRECT): ElastiCache Memcached + ALB + scheduled scaling

This option improves compute scalability but selects the weaker cache solution.

Why this is not best:

- Memcached limitations:
  - No persistence
  - No replication
  - Weaker high availability capabilities
  - Less enterprise-grade resiliency

- DynamoDB backend strongly favors DAX:
  - Native integration
  - API compatibility
  - Optimized read-through caching behavior

The following parts are correct:
- ALB
- Auto Scaling
- Scheduled scaling

However, SAP-C02 focuses on choosing the best managed AWS-native service for the workload. DAX is purpose-built for DynamoDB acceleration.

Official comparison:
https://docs.aws.amazon.com/whitepapers/latest/database-caching-strategies-using-redis/redis-vs-memcached.html

---

#### ❌ Option D (INCORRECT): DAX + CloudFront + manual scaling

This option partially improves caching but still has major infrastructure weaknesses.

Problems:

- CloudFront is not a replacement for ALB
- ASG cannot directly function as CloudFront origin
- Manual scaling is inferior for predictable traffic spikes
- Missing Layer 7 balancing and health checks

Even though DAX is correct, the frontend traffic management design is not production-grade compared with Option A.

### 3. Hands-On Knowledge

#### Typical Enterprise Architecture

A production-grade architecture would look like:

```text
Users
  ↓
Route 53
  ↓
ALB
  ↓
EC2 Auto Scaling Group
  ↓
DAX Cluster
  ↓
DynamoDB
```

Key benefits:
- Multi-AZ resilience
- Elastic horizontal scaling
- Managed cache layer
- Reduced operational burden

---

#### Why DAX Is Powerful

DAX is extremely effective for:

- Read-heavy DynamoDB workloads
- Hot partition mitigation
- Frequently repeated queries
- Metadata lookup systems
- Catalog/search-heavy applications

Latency comparison:

- DynamoDB direct read:
  - Single-digit milliseconds
- DAX cached read:
  - Microseconds

Important operational note:
- DAX benefits mostly read-intensive workloads
- Applications must use DAX SDK/client endpoints

---

#### Scheduled Scaling Example

Because updates occur quarterly, scheduled scaling is ideal.

Example CLI:

```bash
aws autoscaling put-scheduled-update-group-action \
  --auto-scaling-group-name travel-app-asg \
  --scheduled-action-name quarterly-update-scaleout \
  --start-time "2026-09-01T07:00:00Z" \
  --min-size 6 \
  --max-size 12 \
  --desired-capacity 8
```

This ensures:
- Instances are already warm
- Cache is populated before demand surge
- Lower request latency during peak windows

---

#### Why ALB Is Better Than Route 53 Multivalue Routing

Route 53 multivalue routing:
- Only returns multiple IP addresses
- Not true request balancing
- Limited traffic management capability

ALB provides:
- Request-level balancing
- Health checks
- Sticky sessions
- TLS offloading
- Better observability
- Integration with Auto Scaling

This modernization pattern appears frequently in SAP-C02 scenarios.

---

#### Enterprise Operational Best Practices

In real production systems:

- Enable Multi-AZ DAX clusters
- Use EC2 instances across at least 2 AZs
- Configure CloudWatch alarms for:
  - ALB target latency
  - DAX CPU usage
  - Cache hit ratio
  - DynamoDB throttling
- Enable ALB access logs
- Consider Auto Scaling warm pools for faster launches

Additional enterprise enhancements:
- AWS WAF on ALB
- Systems Manager Patch Manager
- Blue/green deployments with CodeDeploy
- Infrastructure as Code using CloudFormation or Terraform

**面试官可能追问 (Possible Follow-up Questions)**：

1. "什么时候应该选择 DAX 而不是 ElastiCache Redis？"
   → 回答要点 (Key Answer Points)：DAX 是 DynamoDB 专用缓存，提供 API 级集成和微秒级读取延迟，适合 DynamoDB 的读密集型工作负载。Redis 更通用，适合复杂缓存逻辑、消息队列、Pub/Sub 等场景。

2. "为什么 predictable traffic spike 更适合 scheduled scaling？"
   → 回答要点 (Key Answer Points)：Scheduled scaling 可以在流量到达前提前准备容量，避免 reactive scaling 的冷启动问题。对于节假日、促销、季度更新等固定时间流量峰值，这是更稳定、更低风险的方案。
