A company has a new application that needs to run on five Amazon EC2 instances in a single AWS Region. The application requires high-throughput, low-latency network connections between all of the EC2 instances where the application will run. There is no requirement for the application to be fault tolerant.
Which solution will meet these requirements?

A. Launch five new EC2 instances into a cluster placement group. Ensure that the EC2 instance type supports enhanced networking.
B. Launch five new EC2 instances into an Auto Scaling group in the same Availability Zone. Attach an extra elastic network interface to each EC2 instance.
C. Launch five new EC2 instances into a partition placement group. Ensure that the EC2 instance type supports enhanced networking.
D. Launch five new EC2 instances into a spread placement group. Attach an extra elastic network interface to each EC2 instance.


Suggest Answer: A

---

## Analysis

### Understanding the Problem

A company needs to run a new application on exactly **five EC2 instances** in a single AWS Region. The defining requirement is **high-throughput, low-latency network connections between ALL instances**. There is explicitly **no fault tolerance requirement**, which frees the architect from spreading instances across Availability Zones or hardware. The question tests knowledge of EC2 placement group types and their networking characteristics.

### Correct Option: A

**Launch five new EC2 instances into a cluster placement group. Ensure that the EC2 instance type supports enhanced networking.**

- **Cluster Placement Group**: This is a logical grouping of instances within a **single Availability Zone**. AWS places instances in a cluster placement group on hardware that is physically close together — often within the same high-bisection-bandwidth network segment. This proximity delivers **low-latency, high-throughput** (up to 10 Gbps for a single flow, up to 100 Gbps aggregate) network performance between instances. This is purpose-built for the stated requirement.
- **Enhanced Networking**: Enabled via Elastic Network Adapter (ENA) or Intel 82599 VF, enhanced networking uses **SR-IOV (Single Root I/O Virtualization)** to bypass the hypervisor for network traffic. Benefits include: higher packets per second (PPS), lower inter-instance latency, and reduced jitter. Enabling enhanced networking ensures the instances can actually achieve the throughput the cluster placement group makes possible.
- The combination of cluster placement group + enhanced networking is the **AWS-recommended pattern** for high-performance computing (HPC), tightly coupled analytics, and low-latency distributed applications.

### Incorrect Options

**B: Auto Scaling group same AZ + extra ENIs** — Adding extra Elastic Network Interfaces (ENIs) does **not** improve inter-instance latency or throughput. Each ENI adds more network capacity (bandwidth) to an instance, but the physical distance between instances on different underlying hardware remains unchanged. An Auto Scaling group is irrelevant — the requirement is for a fixed set of five instances, not dynamic scaling. This option addresses neither the low-latency nor the throughput requirement.

**C: Partition placement group + enhanced networking** — A **partition placement group** divides instances into logical segments (partitions) where each partition runs on **separate racks** with independent power and networking. This design intentionally isolates partitions to limit the blast radius of hardware failures — great for large distributed workloads like HDFS, Cassandra, or Kafka. However, instances in different partitions do **not** benefit from the tight physical proximity and low-latency interconnects that a cluster placement group provides. Enhanced networking helps, but the physical separation undermines the primary requirement.

**D: Spread placement group + extra ENIs** — A **spread placement group** places each instance on **distinct underlying hardware** within an AZ, with a hard limit of **7 instances per group per AZ**. This is designed for small, critical workloads where you need to minimize correlated failures (e.g., each instance serves a different customer). Spreading instances across different hardware maximizes fault isolation — the exact opposite of what's needed for low-latency interconnects. Extra ENIs don't compensate for physical hardware dispersion.

### Hands-On Knowledge

1. **Cluster Placement Group Limitations**: 
   - All instances must be launched in a single AZ.
   - Not all instance types support placement groups. Compute-optimized (C5, C6g, etc.), memory-optimized (R5, X1, etc.), and accelerated-computing (P3, P4, etc.) types generally do.
   - To achieve maximum throughput, launch all instances at the same time (homogeneous instance type recommended). Adding instances later may cause capacity errors if the existing hardware rack is full.
   - Use `placement-group-strategy: cluster` in CloudFormation/Terraform.

2. **Enhanced Networking Requirements**:
   - For ENA (current generation): supported on most instance types. Requires the `ena` kernel module on Linux or the ENA driver on Windows.
   - For Intel 82599 VF (legacy): requires specific instance types (C3, R3, I2, etc.) and the `ixgbevf` driver.
   - Check with: `aws ec2 describe-instances --query 'Reservations[*].Instances[*].EnaSupport'`

3. **Network Performance Monitoring**: Use `iperf3` or `qperf` between instances to benchmark actual throughput. ENA metrics are available in CloudWatch (`NetworkPacketsIn`, `NetworkPacketsOut`, `EnaBwInAllowanceExceeded` — the latter indicates you're hitting bandwidth limits).

4. **Jumbo Frames**: Enable MTU 9001 (jumbo frames) within a cluster placement group VPC subnet to further reduce overhead and improve throughput for large data transfers. All instances in the group must support and be configured for jumbo frames.

5. **Decision Framework for Placement Groups**:

| Requirement | Placement Group Type |
|---|---|
| Low latency + high throughput between instances | **Cluster** |
| Hardware fault isolation for a few critical instances | **Spread** |
| Correlated failure protection for large distributed workloads | **Partition** |
