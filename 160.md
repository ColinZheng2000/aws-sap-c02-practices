A company is implementing a serverless architecture by using AWS Lambda functions that need to access a Microsoft SQL Server DB instance on Amazon RDS. The company has separate environments for development and production, including a clone of the database system.

The company's developers are allowed to access the credentials for the development database. However, the credentials for the production database must be encrypted with a key that only members of the IT security team's IAM user group can access. This key must be rotated on a regular basis.

What should a solutions architect do in the production environment to meet these requirements?

A. Store the database credentials in AWS Systems Manager Parameter Store by using a SecureString parameter that is encrypted by an AWS Key Management Service (AWS KMS) customer managed key. Attach a role to each Lambda function to provide access to the SecureString parameter. Restrict access to the SecureString parameter and the customer managed key so that only the IT security team can access the parameter and the key.
B. Encrypt the database credentials by using the AWS Key Management Service (AWS KMS) default Lambda key. Store the credentials in the environment variables of each Lambda function. Load the credentials from the environment variables in the Lambda code. Restrict access to the KMS key so that only the IT security team can access the key.
C. Store the database credentials in the environment variables of each Lambda function. Encrypt the environment variables by using an AWS Key Management Service (AWS KMS) customer managed key. Restrict access to the customer managed key so that only the IT security team can access the key.
D. Store the database credentials in AWS Secrets Manager as a secret that is associated with an AWS Key Management Service (AWS KMS) customer managed key. Attach a role to each Lambda function to provide access to the secret. Restrict access to the secret and the customer managed key so that only the IT security team can access the secret and the key.

Suggest Answer: D

---

## 📋 Analysis

### Understanding the Problem

A company runs a **serverless architecture** with AWS Lambda functions accessing an **Amazon RDS for SQL Server** database across separate development and production environments. The core security requirements are:

1. **Production database credentials must be encrypted** with a key accessible exclusively by the IT security team's IAM user group.
2. **The encryption key must be rotated regularly.**
3. Developers may freely access development credentials, but production must be strictly controlled.

The challenge is selecting a solution that provides **encrypted storage**, **fine-grained access control** to the encryption key, **automatic key rotation**, and **secure credential retrieval by Lambda functions** — all without exposing production secrets to developers.

---

### ✅ Correct Option: D

**Store credentials in AWS Secrets Manager, encrypted with a KMS customer managed key (CMK). Attach an IAM role to Lambda for access. Restrict access to both the secret and the CMK to the IT security team.**

**Why this is correct:**

| Requirement | How Option D Fulfills It |
|---|---|
| **Encrypted storage** | Secrets Manager encrypts secrets at rest using a KMS CMK that you control. |
| **IT security team exclusive access to key** | KMS key policies allow you to designate *key administrators* (the IT security team) who can manage, rotate, and control the key. Lambda receives only `kms:Decrypt` permission via its IAM role — it cannot administer the key. |
| **Regular key rotation** | KMS CMKs support **automatic annual key rotation** (enabled by default for new CMKs). Secrets Manager also supports **built-in secret rotation** for RDS databases via Lambda rotation functions. |
| **Lambda access** | Lambda assumes an IAM execution role with permissions to call `secretsmanager:GetSecretValue` and `kms:Decrypt` on the specific secret and CMK. |
| **Separation of duties** | The IT security team administers the key (via key policy), developers cannot access production secrets (via IAM policy on the secret), and Lambda functions have least-privilege access for runtime use. |

**AWS Best Practice Alignment:**
- The *AWS Well-Architected Framework — Security Pillar* explicitly recommends **AWS Secrets Manager** over Parameter Store for database credentials, API keys, and other sensitive secrets because of its native rotation support and tighter RDS integration.
- Reference: [AWS Secrets Manager — Best Practices](https://docs.aws.amazon.com/secretsmanager/latest/userguide/best-practices.html)

---

### ❌ Incorrect Options

| Option | Why It Fails |
|---|---|
| **A (SSM Parameter Store SecureString)** | While Parameter Store SecureString can be encrypted with a KMS CMK, it **lacks native secret rotation** for database credentials. More critically, restricting access to *both the parameter and the key* to *only* the IT security team means Lambda cannot read the parameter at runtime — a logical contradiction. Parameter Store is designed for configuration data, not for production database secrets. AWS recommends Secrets Manager for database credentials. |
| **B (KMS default Lambda key + env vars)** | The **AWS managed Lambda key** cannot have its key policy restricted to specific IAM principals — AWS managed keys grant broad access. Environment variables are not encrypted at rest in a secure vault; they are plaintext in the Lambda console and execution environment. This fails both encryption and access-control requirements. |
| **C (KMS CMK + encrypted env vars)** | Lambda environment variables are **not natively encrypted with a KMS CMK** — you would need to manually encrypt/decrypt them in code, adding complexity and risk. More importantly, environment variables are visible to anyone with `lambda:GetFunctionConfiguration` permission and are passed in plaintext to the execution environment at cold start. This is not a secure credential storage pattern. |

### 🏭 Hands-on Enterprise Knowledge

1. **KMS Key Rotation in Practice:**
   - When you enable automatic key rotation on a KMS CMK, AWS generates new *backing key material* annually while keeping the same key ARN and alias. All previously encrypted data remains decryptable because KMS stores multiple versions of the key material.
   - Rotation does **not** re-encrypt existing ciphertext — new encryption operations use the new backing key. This is why you must also rotate the *secret value* periodically (Secrets Manager handles this with its own rotation mechanism).

2. **Secrets Manager Rotation Architecture:**
   - For RDS SQL Server, Secrets Manager uses a **Lambda rotation function** that follows a multi-step process: `createSecret` → `setSecret` → `testSecret` → `finishSecret`. This is because SQL Server does not support the single-user rotation strategy natively (unlike MySQL/PostgreSQL which can use the master user for rotation).
   - Reference: [AWS Secrets Manager — Rotating AWS Secrets Manager secrets for Amazon RDS for Microsoft SQL Server](https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html)

3. **IAM Policy Boundaries for Lambda + Secrets:**
   ```json
   {
     "Effect": "Allow",
     "Action": [
       "secretsmanager:GetSecretValue",
       "kms:Decrypt"
     ],
     "Resource": [
       "arn:aws:secretsmanager:region:account:secret:prod/db-credential-*",
       "arn:aws:kms:region:account:key/your-cmk-id"
     ]
   }
   ```
   This grants Lambda exactly what it needs while the IT team retains administrative control via separate key policy and IAM policies.

4. **Why Not SSM Parameter Store for Database Credentials?**
   - **4 KB / 8 KB size limit** — most database credentials fit, but advanced configurations (connection strings with multiple hosts, certificates) may exceed it.
   - **No native RDS integration** — Secrets Manager can automatically update the RDS master password during rotation; Parameter Store cannot.
   - **No cross-account sharing** — Secrets Manager supports resource-based policies for cross-account access; Parameter Store requires IAM role assumption.
