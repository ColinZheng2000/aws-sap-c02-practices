# AWS SAP Exam Question Analysis: Replatforming with Least Overhead

## Question
A company is refactoring its on-premises order-processing platform in the AWS Cloud. The platform includes a web front end that is hosted on a fleet of VMs, RabbitMQ to connect the front end to the backend, and a Kubernetes cluster to run a containerized backend system to process the orders. The company does not want to make any major changes to the application.

Which solution will meet these requirements with the LEAST operational overhead?

**A. Create an AMI of the web server VM. Create an Amazon EC2 Auto Scaling group that uses the AMI and an Application Load Balancer. Set up Amazon MQ to replace the on-premises messaging queue. Configure Amazon Elastic Kubernetes Service (Amazon EKS) to host the order-processing backend.**

**B.** Create a custom AWS Lambda runtime to mimic the web server environment. Create an Amazon API Gateway API to replace the front-end web servers. Set up Amazon MQ to replace the on-premises messaging queue. Configure Amazon Elastic Kubernetes Service (Amazon EKS) to host the order-processing backend.

**C.** Create an AMI of the web server VM. Create an Amazon EC2 Auto Scaling group that uses the AMI and an Application Load Balancer. Set up Amazon MQ to replace the on-premises messaging queue. Install Kubernetes on a fleet of different EC2 instances to host the order-processing backend.

**D.** Create an AMI of the web server VM. Create an Amazon EC2 Auto Scaling group that uses the AMI and an Application Load Balancer. Set up an Amazon Simple Queue Service (Amazon SQS) queue to replace the on-premises messaging queue. Configure Amazon Elastic Kubernetes Service (Amazon EKS) to host the order-processing backend.

---
## Analysis

### 1. Understanding the Problem
The scenario presents a classic "Replatforming" (or "Lift and Reshape") migration case. We need to map three distinct on-premises components to AWS services while adhering to two strict constraints:
1.  **No major application changes:** We cannot rewrite code or change protocols significantly.
2.  **Least operational overhead:** We should use managed services wherever possible to reduce the burden of patching, scaling, and maintaining infrastructure.

**The Architecture Mapping:**
* **Front End:** VMs $\rightarrow$ ?
* **Messaging:** RabbitMQ $\rightarrow$ ?
* **Backend:** Kubernetes $\rightarrow$ ?

### 2. Services Explanation

* **Amazon MQ:** A managed message broker service for Apache ActiveMQ and **RabbitMQ**. It is designed specifically for migrating legacy applications that use standard protocols (like AMQP, MQTT, and STOMP) without rewriting code.
* **Amazon EKS (Elastic Kubernetes Service):** A managed service where AWS handles the Kubernetes **Control Plane** (API Server, etcd, scheduler). This significantly reduces operational overhead compared to managing your own Kubernetes cluster.
* **Amazon SQS (Simple Queue Service):** A cloud-native, highly scalable queue. However, it uses AWS proprietary APIs (HTTP based). Migrating from RabbitMQ to SQS usually requires refactoring application code.
* **EC2 Auto Scaling & ALB:** The standard AWS pattern for hosting stateless web applications running on Virtual Machines (EC2).

### 3. Correct Option Explanation

**Option A is correct because:**
* **Web Front End:** Migrating VMs to **AMIs** with **Auto Scaling** and an **ALB** is the standard "Replatform" approach. It provides high availability and elasticity without changing the application code.
* **Messaging (The Key Differentiator):** **Amazon MQ** supports RabbitMQ engine versions. Since the on-prem application uses RabbitMQ, moving to Amazon MQ allows the application to connect to the new broker with updated endpoints/credentials but **without code refactoring**. This meets the "no major changes" requirement.
* **Backend:** **Amazon EKS** is the managed solution for Kubernetes. It removes the heavy lifting of managing the Kubernetes control plane (which is required in Option C), fulfilling the "least operational overhead" requirement.

### 4. Incorrect Options Explanation

* **Option B (Incorrect - Violates Constraints):**
    * Moving from VMs to **AWS Lambda** (Serverless) is a "Refactor/Re-architect" strategy, not a Replatform. This would require significant changes to how the application runs and is packaged, violating the "no major changes" requirement.
    
* **Option C (Incorrect - High Overhead):**
    * Installing Kubernetes on **raw EC2 instances** is a self-managed approach. You would be responsible for the Control Plane, etcd backups, upgrades, and patching of the Kubernetes software itself. This has significantly higher **operational overhead** compared to using EKS.

* **Option D (Incorrect - Violates Constraints):**
    * While **Amazon SQS** is a great service with low overhead, it is not protocol-compatible with RabbitMQ (which typically uses AMQP). To use SQS, developers would have to rewrite the application's messaging logic to use the AWS SDK. This violates the "no major changes" requirement.
