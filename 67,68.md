A company has its cloud infrastructure on AWS. A solutions architect needs to define the infrastructure as code. The infrastructure is currently deployed in one  
AWS Region. The company's business expansion plan includes deployments in multiple Regions across multiple AWS accounts.  
What should the solutions architect do to meet these requirements?  

- A. Use AWS CloudFormation templates. Add IAM policies to control the various accounts. Deploy the templates across the multiple Regions.
- B. Use AWS Organizations. Deploy AWS CloudFormation templates from the management account. Use AWS Control Tower to manage deployments across accounts.
- C. Use AWS Organizations and AWS CloudFormation StackSets. Deploy a CloudFormation template from an account that has the necessary IAM permissions.
- D. Use nested stacks with AWS CloudFormation templates. Change the Region by using nested stacks.

## Understanding the Problem

The company needs to define its infrastructure as code (IaC) to support future expansion into **multiple AWS Regions** and **multiple AWS accounts**. The current infrastructure is only in one Region. The solutions architect must choose an IaC and deployment strategy that effectively manages resources consistently and centrally across this complex, distributed environment.

## Services Explanation

| **Service**                      | **Description**                                                                                                                                                                                                                 | **Use Case Relevance**                                                                                                                                                                                                                  |
| -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **AWS CloudFormation**           | An IaC service that allows you to model and provision AWS and third-party resources. A **template** is a JSON or YAML file that defines the desired resources.                                                                  | The base IaC tool for defining the infrastructure.                                                                                                                                                                                      |
| **AWS CloudFormation StackSets** | An extension of CloudFormation that lets you provision a single CloudFormation stack into **multiple target AWS accounts** and **multiple AWS Regions** with a single operation. It centralizes control from a primary account. | Crucial for the multi-Region, multi-account deployment requirement.                                                                                                                                                                     |
| **AWS Organizations**            | An account management service that allows you to consolidate multiple AWS accounts into an organization that you create and centrally manage. It's often used to structure billing, governance, and security across accounts.   | Essential for grouping and managing the multiple AWS accounts required for the business expansion.                                                                                                                                      |
| **AWS Control Tower**            | A service that sets up and governs a secure, multi-account AWS environment, following best practices. It uses AWS Organizations and StackSets internally.                                                                       | Manages the overall multi-account structure but isn't the primary tool for _deploying_ specific application infrastructure defined by an existing custom CloudFormation template across various accounts/Regions in this direct manner. |
| **IAM Policies**                 | Identity and Access Management policies that define permissions for an AWS entity (user, group, or role) to access AWS resources.                                                                                               | Necessary for cross-account access, but not a deployment _mechanism_ itself.                                                                                                                                                            |
| **Nested Stacks**                | CloudFormation stacks created as part of another stack (the parent stack). They help in reusing common templates and organizing resources.                                                                                      | Primarily for modularizing templates within _a single Region and account_, not natively for cross-account or cross-Region deployment.                                                                                                   |

## Correct Option(s) Explanation

### C. Use AWS Organizations and AWS CloudFormation StackSets. Deploy a CloudFormation template from an account that has the necessary IAM permissions.

- **AWS Organizations:** This is the recommended service for managing the "multiple AWS accounts." It provides the structure to define the target environments.
    
- **AWS CloudFormation StackSets:** This is the key service that specifically addresses deploying a single CloudFormation template (the infrastructure definition) across **multiple accounts** (managed by Organizations) and **multiple Regions**. The documentation for StackSets explicitly states its purpose is for "managing infrastructure across multiple AWS accounts and Regions."
    
- **Deployment Account and IAM:** StackSets are deployed from a central **administrator account** that is granted the necessary IAM permissions (via a service-managed or self-managed execution role in the target accounts/Regions) to provision resources in the target accounts. This centralized control meets the need for consistent, enterprise-wide IaC deployment.
    

This combination of services is the **AWS-recommended best practice** for the complex, distributed deployment requirement. (Refer to the AWS CloudFormation StackSets documentation for multi-account, multi-Region deployment).

## Incorrect Options Explanation

### A. Use AWS CloudFormation templates. Add IAM policies to control the various accounts. Deploy the templates across the multiple Regions.

- **Inefficient Deployment:** This approach requires manually executing the CloudFormation template once for _each_ Region and once for _each_ account (e.g., if there are 3 accounts and 4 regions, you'd deploy $3 \times 4 = 12$ times).
    
- **Missing Central Control:** CloudFormation _templates_ alone do not have a native mechanism for centralized, one-to-many deployment across multiple accounts and Regions. You'd need a custom script or pipeline, which is less efficient and robust than StackSets.
    

### B. Use AWS Organizations. Deploy AWS CloudFormation templates from the management account. Use AWS Control Tower to manage deployments across accounts.

- **Misapplication of Control Tower:** AWS Control Tower's primary role is setting up a secure baseline (Landing Zone) and governance. While it _uses_ StackSets internally for its own governance (known as **Guardrails**), it is **not** the standard tool for deploying _custom application infrastructure_ defined in a company's specific CloudFormation template across accounts/Regions. StackSets is the more direct and correct service for the _deployment_ task.
    
- **Missing Core Mechanism:** The option doesn't mention the necessary service, **CloudFormation StackSets**, which is the mechanism for the multi-account/multi-Region deployment.
    

### D. Use nested stacks with AWS CloudFormation templates. Change the Region by using nested stacks.

- **Incorrect Use of Nested Stacks:** Nested stacks are for **template modularity and reuse** _within_ a single deployment (typically within a single account and Region). They do **not** inherently solve the challenge of deploying across multiple AWS accounts or changing the target Region of the deployment. The target Region is determined when the top-level (root) stack is created.
