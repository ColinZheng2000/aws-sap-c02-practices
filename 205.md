A solutions architect needs to review the design of an Amazon EMR cluster that is using the EMR File System (EMRFS). The cluster performs tasks that are critical to business needs. The cluster is running Amazon EC2 On-Demand Instances at all times for all task, primary, and core nodes. The EMR tasks run each morning, starting at 1:00 AM. and take 6 hours to finish running. The amount of time to complete the processing is not a priority because the data is not referenced until late in the day.

The solutions architect must review the architecture and suggest a solution to minimize the compute costs.

Which solution should the solutions architect recommend to meet these requirements?

A. Launch all task, primary, and core nodes on Spot Instances in an instance fleet. Terminate the cluster, including all instances, when the processing is completed.
B. Launch the primary and core nodes on On-Demand Instances. Launch the task nodes on Spot Instances in an instance fleet. Terminate the cluster, including all instances, when the processing is completed. Purchase Compute Savings Plans to cover the On-Demand Instance usage.
C. Continue to launch all nodes on On-Demand Instances. Terminate the cluster, including all instances, when the processing is completed. Purchase Compute Savings Plans to cover the On-Demand Instance usage.
D. Launch the primary and core nodes on On-Demand Instances. Launch the task nodes on Spot Instances in an instance fleet. Terminate only the task node instances when the processing is completed. Purchase Compute Savings Plans to cover the On-Demand Instance usage.



Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company runs an Amazon EMR cluster using EMRFS (EMR File System, which stores data in Amazon S3 rather than HDFS on the cluster nodes). The cluster currently runs all nodes (task, primary, and core) on On-Demand EC2 instances 24/7. The EMR jobs execute daily starting at 1:00 AM and take 6 hours to complete. The critical insight is that **processing completion time is not a priority** — the data isn't referenced until late in the day. The goal is purely to **minimize compute costs** while maintaining the reliability required for business-critical tasks. Because the cluster uses EMRFS (S3-backed), data durability is handled by S3 independently of cluster node lifecycle — this enables flexible cost-saving strategies for the compute layer.

### Correct Option: D — On-Demand for Primary/Core + Spot for Task Nodes + Terminate Only Task Nodes + Savings Plans

**Why D is correct:**

- **Primary and core nodes on On-Demand Instances**: Primary nodes manage the cluster (resource management, job scheduling via YARN) and core nodes hold HDFS data (if used) and perform data processing. While EMRFS stores data in S3, core nodes still cache intermediate data and provide compute capacity. Keeping them on On-Demand ensures cluster stability and prevents job failures from Spot interruptions — critical for business-critical tasks.
- **Task nodes on Spot Instances in an instance fleet**: Task nodes are purely computational — they don't store data and can be added/removed without data loss. Spot Instances offer up to 90% discount compared to On-Demand, making them ideal for the 6-hour daily processing window. Instance fleets allow EMR to provision Spot capacity across multiple instance types and Availability Zones, improving Spot availability and reducing the chance of capacity shortages.
- **Terminate only task nodes after processing**: Since primary and core nodes remain running 24/7, **Compute Savings Plans** provide a significant discount (up to 66%) for the steady-state On-Demand usage. This is the financially optimal combination: Savings Plans for the 24/7 baseline (primary + core), Spot for the variable 6-hour workload (task nodes).
- **EMRFS decouples storage from compute**: Because data resides in S3 (not HDFS), terminating task nodes does not cause data loss. The cluster can add fresh task nodes the next day without rebuilding data.

**Official references**:
- [EMRFS and Spot Instances](https://docs.aws.amazon.com/emr/latest/ManagementGuide/emr-plan-instances-spot.html) — "Task nodes are a good candidate for Spot Instances because they only run tasks and do not store data."
- [Compute Savings Plans](https://docs.aws.amazon.com/savingsplans/latest/userguide/what-is-savings-plans.html) — Apply automatically to EC2 usage regardless of instance family, size, or region, providing flexibility for the primary and core node fleet.

### Incorrect Options

**A — All nodes (primary, core, task) on Spot Instances + terminate entire cluster**

- **Extremely risky for business-critical tasks**: Spot Instances can be reclaimed by AWS with only a 2-minute warning. If the primary node is interrupted, the entire EMR cluster terminates immediately, and all running jobs fail. If core nodes are interrupted, in-progress data processing may be lost (even with EMRFS, intermediate shuffle data on core nodes is lost).
- While EMRFS in S3 prevents permanent data loss, repeated job failures due to Spot interruptions are unacceptable for business-critical processing.
- AWS best practice explicitly recommends against running primary and core nodes on Spot for production/critical workloads.

**B — On-Demand for primary/core + Spot for task nodes + terminate entire cluster + Savings Plans**

- **Terminating the entire cluster daily** means the primary and core nodes run only 6 hours per day. **Compute Savings Plans require a 1-year or 3-year hourly commitment** — you'd be paying for 24 hours of coverage but only using 6 hours, effectively wasting 75% of your Savings Plan commitment. This is financially irrational.
- If you terminate the cluster daily and only run 6 hours, **On-Demand pricing without Savings Plans would actually be cheaper** than committing to a Savings Plan for unused hours. Alternatively, you could use Spot for primary/core too (but that brings back the reliability issue in Option A).
- The correct cost optimization for a 6-hour daily window is Spot (for flexible nodes) + On-Demand without long-term commitment, or Savings Plans only for 24/7 workloads.

**C — All nodes On-Demand + terminate entire cluster + Savings Plans**

- This combines the worst of both worlds: no Spot savings AND wasted Savings Plan commitment (paying for 24 hours, using 6 hours).
- Terminating the entire cluster daily with all On-Demand nodes loses the opportunity to use Spot Instances for the elastic task node layer.
- On-Demand pricing at full rate for all nodes, plus wasted Savings Plan commitment, is the least cost-optimized approach among the options.

### Hands-On Enterprise Knowledge

1. **EMR node roles**: Understanding the three EMR node types is essential for the SAP-C02 exam:
   - **Primary node**: Manages the cluster, runs YARN ResourceManager, HDFS NameNode (if using HDFS), and coordinates job execution. Single point of coordination — never use Spot.
   - **Core nodes**: Run YARN NodeManager and HDFS DataNode (if using HDFS). They process data AND store HDFS data. For EMRFS clusters, they can technically be Spot but it's not recommended for critical workloads due to shuffle data loss risk.
   - **Task nodes**: Run YARN NodeManager only — purely computational, no data storage. **Ideal for Spot Instances**. They can be added and removed freely.

2. **EMRFS vs. HDFS**: EMRFS allows data to persist in S3 independently of cluster lifecycle. This architectural choice enables cost-saving patterns like transient clusters (create → process → terminate). If the cluster used HDFS, terminating core nodes would cause permanent data loss, making transient clusters infeasible. The exam often tests this distinction.

3. **Instance fleets vs. uniform instance groups**: Instance fleets (mentioned in the correct options) allow you to specify target capacities across Spot and On-Demand, and EMR intelligently provisions the best mix across multiple instance types and Availability Zones. This improves Spot availability and resilience compared to uniform instance groups. For enterprise Spot usage, instance fleets are the recommended modern approach.

4. **Savings Plans selection strategy**:
   - **Compute Savings Plans**: Most flexible — applies to any EC2 usage (any family, size, region). Best when you have predictable 24/7 baseline but may change instance types over time.
   - **EC2 Instance Savings Plans**: Cheaper than Compute Savings Plans but locked to a specific instance family in a specific region. Best when you have stable, predictable instance type usage.
   - **Reserved Instances**: Less flexible than Savings Plans, but still useful for specific use cases. Generally, Savings Plans are preferred for new commitments.

5. **Enterprise cost optimization pattern for EMR**: Many enterprises run EMR clusters with a "persistent core, elastic task" pattern — primary + core nodes as RI/Savings Plans baseline (24/7), task nodes as Spot (scaled for each job run). For truly non-time-sensitive workloads, an even cheaper pattern is "transient cluster with Spot for all nodes" using EMRFS, but only when job retries on Spot interruption are acceptable.
