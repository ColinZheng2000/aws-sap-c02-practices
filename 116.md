A company wants to use AWS to create a business continuity solution in case the company's main on-premises application fails. The application runs on physical servers that also run other applications. The on-premises application that the company is planning to migrate uses a MySQL database as a data store. All the company's on-premises applications use operating systems that are compatible with Amazon EC2.

Which solution will achieve the company's goal with the LEAST operational overhead?

A. Install the AWS Replication Agent on the source servers, including the MySQL servers. Set up replication for all servers. Launch test instances for regular drills. Cut over to the test instances to fail over the workload in the case of a failure event.
B. Install the AWS Replication Agent on the source servers, including the MySQL servers. Initialize AWS Elastic Disaster Recovery in the target AWS Region. Define the launch settings. Frequently perform failover and fallback from the most recent point in time.
C. Create AWS Database Migration Service (AWS DMS) replication servers and a target Amazon Aurora MySQL DB cluster to host the database. Create a DMS replication task to copy the existing data to the target DB cluster. Create a local AWS Schema Conversion Tool (AWS SCT) change data capture (CDC) task to keep the data synchronized. Install the rest of the software on EC2 instances by starting with a compatible base AMI.
D. Deploy an AWS Storage Gateway Volume Gateway on premises. Mount volumes on all on-premises servers. Install the application and the MySQL database on the new volumes. Take regular snapshots. Install all the software on EC2 Instances by starting with a compatible base AMI. Launch a Volume Gateway on an EC2 instance. Restore the volumes from the latest snapshot. Mount the new volumes on the EC2 instances in the case of a failure event.

# SAP-C02 Practice Question – AWS Business Continuity / DR with Least Operational Overhead  
#sap-c02 #dr #elastic-disaster-recovery #dms #storage-gateway #ec2 #mysql #business-continuity

- --

## Understanding the Problem

- The company’s **primary application is on‑premises**, running on **physical servers**.
- Application uses a **MySQL database**.
- All on-prem apps use OSs compatible with **Amazon EC2**.
- Goal: **Business continuity** if the on-prem main application fails.
- Key requirement: Solution with the **LEAST operational overhead**.
- This is not a full “migration”; it’s about setting up an **AWS-based DR / failover** environment.

We need to balance:
- Automation and simplicity (low ops overhead),
- Coverage of **entire application stack** (app servers + MySQL),
- Suitability for physical servers and on-prem workloads.

- --

## Services Explanation (Relevant to the Options)

### 1. AWS Elastic Disaster Recovery (AWS DRS) – formerly CloudEndure Disaster Recovery
- **Purpose:** Minimize downtime and data loss for physical/virtual/cloud servers by replicating them to AWS.
- Works with:  
  - **Physical servers**, VMware, Hyper‑V, other clouds.
- How it works:
  - Install **AWS Replication Agent** on source servers.
  - Continuous block‑level replication to a **staging area** in AWS.
  - During DR, it launches fully provisioned **EC2 instances** based on **launch templates/settings**.
- Key features:
  - Orchestration and automation of failover / failback.
  - Point-in-time recovery.
  - Can be used for drills with minimal overhead.
- Official docs:  
  - [AWS Elastic Disaster Recovery – User Guide](https://docs.aws.amazon.com/drs/latest/userguide/)

### 2. AWS Database Migration Service (AWS DMS) + AWS Schema Conversion Tool (AWS SCT)
- **AWS DMS**:
  - Purpose: Continuous replication / one-time migration of **databases**.
  - Supports **MySQL** and replication to **Aurora MySQL**.
  - Provides ongoing **CDC (change data capture)**.
- **AWS SCT**:
  - Primarily for converting DB schemas between engines, can also do CDC in certain hybrid scenarios.
- But: They only handle **databases**, not entire application servers.
- You still must design and maintain **EC2 deployment, AMIs, app install/patching, config** → more operational work.
- Official docs:
  - [AWS DMS User Guide](https://docs.aws.amazon.com/dms/latest/userguide/Welcome.html)
  - [AWS SCT User Guide](https://docs.aws.amazon.com/SchemaConversionTool/latest/userguide/Welcome.html)

### 3. AWS Storage Gateway – Volume Gateway
- Provides hybrid cloud storage:
  - **Volume Gateway**: iSCSI volumes with data backed up to AWS.
  - Snapshots stored as **EBS snapshots** in AWS.
- For DR:
  - You can restore **volumes** as EBS volumes attached to EC2.
- But:
  - Only handles **storage layer** (disks/volumes).
  - Application configuration, OS, and services still must be manually installed/maintained on EC2.
- Official docs:
  - [AWS Storage Gateway – Volume Gateway](https://docs.aws.amazon.com/storagegateway/latest/vgw/WhatIsStorageGateway.html)

## Correct Option: **B**

> **B. Install the AWS Replication Agent on the source servers, including the MySQL servers. Initialize AWS Elastic Disaster Recovery in the target AWS Region. Define the launch settings. Frequently perform failover and fallback from the most recent point in time.**

### Why B is Correct (and Has the Least Operational Overhead)

1. **Single solution for entire stack (OS + app + DB)**  
   - AWS DRS is specifically designed for **DR of entire servers (physical or virtual)** including:
     - OS,
     - Application binaries,
     - Config,
     - MySQL data.
   - You don’t need to separately design DB migration + app reinstallation.

2. **Automatic provisioning and orchestration**  
   - Once the **Replication Agent** is installed and **launch settings** defined:
     - Failover is mostly **push‑button** (or via API).
     - AWS DRS will automatically:
       - Create EBS volumes from replicated data,
       - Launch the right EC2 instance types,
       - Attach volumes,
       - Apply networking settings, tags, etc.

3. **Optimized for physical server DR**  
   - The question states: **application runs on physical servers**.
   - AWS DRS is the recommended service for physical to AWS DR:
     - "AWS Elastic Disaster Recovery supports disaster recovery of physical servers to AWS."  
       (ref: AWS DRS user guide)

4. **Reduced operational overhead vs. custom solutions**  
   - You do **not** have to:
     - Manually keep AMIs up to date.
     - Manage DB replication at the app level.
     - Script snapshot restore / mount processes.
   - You just:
     - Deploy the replication agent,
     - Configure replication and launch templates,
     - Periodically test failovers (as recommended).

5. **Continuous replication with low RPO/RTO**  
   - DRS continuously replicates changes, supporting:
     - Near real-time RPO.
     - Rapid EC2 launch (RTO) at failover.

## Insightful Explanation of Incorrect Options

### Option A

> **A. Install the AWS Replication Agent on the source servers, including the MySQL servers. Set up replication for all servers. Launch test instances for regular drills. Cut over to the test instances to fail over the workload in the case of a failure event.**

At first glance, this looks similar to B and mentions the AWS Replication Agent. The subtle but important issue:

1. **Missing explicit AWS DRS workflow**  
   - It never mentions **AWS Elastic Disaster Recovery** or using its orchestration features.
   - “Set up replication for all servers” and “launch test instances” could be interpreted as **manual management** of replication and instance launches.
   - It sounds like using just the replication component without the full DRS service automation.

2. **Higher operational overhead**  
   - You would be responsible for:
     - Defining and managing how test instances are launched and kept in sync.
     - Manual cut-over procedure that may be ad‑hoc.
   - Compared with option B, which explicitly:
     - “Initialize AWS Elastic Disaster Recovery”
     - “Define the launch settings”
     - “Frequently perform failover and fallback from the most recent point in time”
   - B clearly uses DRS in the **intended, managed way**; A is vague and implies more manual handling.

For the exam, when two options use similar building blocks, choose the one that explicitly uses the **managed DR service** for orchestration and minimum overhead → **B over A**.

- --

### Option C

> **C. Create AWS Database Migration Service (AWS DMS) replication servers and a target Amazon Aurora MySQL DB cluster to host the database. Create a DMS replication task to copy the existing data to the target DB cluster. Create a local AWS Schema Conversion Tool (AWS SCT) change data capture (CDC) task to keep the data synchronized. Install the rest of the software on EC2 instances by starting with a compatible base AMI.**

Issues:

1. **Only database-level DR; not full stack**  
   - DMS + SCT covers **DB migration and replication** only.
   - You still must:
     - Build and maintain **EC2-based application servers**.
     - Manage OS patching, app deployment, scaling, and config.
   - This is significantly more operational work than using DRS to replicate existing physical servers.

2. **Migration pattern, not simplest DR pattern**  
   - This looks like a **partial re‑architecture**:
     - MySQL → **Aurora MySQL**.
     - Stand up new EC2 app layer.
   - Good for long-term modernization, but **not** minimal overhead for DR.

3. **Extra moving parts**  
   - DMS replication instance sizing and health.
   - SCT CDC configuration and maintenance.
   - Aurora cluster operations (backups, sizing, etc).

This is a valid design for **modernizing** into managed databases and cloud‑native architecture but does **not** satisfy "least operational overhead" for **DR of an existing on‑prem physical app**.

- --

### Option D

> **D. Deploy an AWS Storage Gateway Volume Gateway on premises. Mount volumes on all on-premises servers. Install the application and the MySQL database on the new volumes. Take regular snapshots. Install all the software on EC2 Instances by starting with a compatible base AMI. Launch a Volume Gateway on an EC2 instance. Restore the volumes from the latest snapshot. Mount the new volumes on the EC2 instances in the case of a failure event.**

Issues:

1. **Storage-level solution; not full DR automation**  
   - Volume Gateway provides **block storage integration** with AWS and snapshots to EBS.
   - For DR, you must:
     - Launch Volume Gateway on EC2,
     - Restore volumes from snapshots,
     - Mount them to EC2 instances.
   - Does **not** replicate:
     - OS state,
     - Application configuration,
     - System-level dependencies.

2. **High operational overhead**  
   - You need to:
     - Maintain AMIs and EC2 launch processes.
     - Ensure mount procedures and app start-up scripts are correct and tested.
     - Manage MySQL configuration consistency across on-prem and AWS.

3. **More complex operational playbook**  
   - Compared to “click failover in DRS console”:
     - Here you manage snapshots, launch gateways, restore volumes, attach to EC2, start apps, verify user connectivity.

This is not the simplest path for DR when a dedicated DR service (AWS DRS) exists.

- --

## Summary

- The question is about **business continuity (DR) for on-prem physical servers** with **least operational overhead**.
- AWS offers a **specialized service** for this: **AWS Elastic Disaster Recovery (DRS)**, using the **AWS Replication Agent**.
- Option **B** explicitly uses DRS in the intended manner and automates failover/fallback with minimal manual work.

**Correct Answer: B**
