A software company needs to create short-lived test environments to test pull requests as part of its development process. Each test environment consists of a single Amazon EC2 instance that is in an Auto Scaling group.

The test environments must be able to communicate with a central server to report test results. The central server is located in an on-premises data center. A solutions architect must implement a solution so that the company can create and delete test environments without any manual intervention. The company has created a transit gateway with a VPN attachment to the on-premises network.

Which solution will meet these requirements with the LEAST operational overhead?

A. Create an AWS CloudFormation template that contains a transit gateway attachment and related routing configurations. Create a CloudFormation stack set that includes this template. Use CloudFormation StackSets to deploy a new stack for each VPC in the account. Deploy a new VPC for each test environment.
B. Create a single VPC for the test environments. Include a transit gateway attachment and related routing configurations. Use AWS CloudFormation to deploy all test environments into the VPC.
C. Create a new OU in AWS Organizations for testing. Create an AWS CioudFormation template that contains a VPC, necessary networking resources, a transit gateway attachment, and related routing configurations. Create a CloudFormation stack set that includes this template. Use CloudFormation StackSets for deployments into each account under the testing OU. Create a new account for each test environment.
D. Convert the test environment EC2 instances into Docker images. Use AWS CloudFormation to configure an Amazon Elastic Kubernetes Service (Amazon EKS) cluster in a new VPC, create a transit gateway attachment, and create related routing configurations. Use Kubernetes to manage the deployment and lifecycle of the test environments.


Suggest Answer: B

---

## 📋 Analysis

### Understanding the Problem

A software company needs an automated way to spin up and tear down short-lived test environments. Each environment is an EC2 instance in an Auto Scaling group, and all must communicate back to an on-premises central server via an existing Transit Gateway with VPN attachment. The core challenge is designing a solution that minimizes **operational overhead** — meaning the fewest moving parts, least management burden, and simplest automation.

### ✅ Correct Option: B

**Create a single VPC for all test environments, include the transit gateway attachment and routing once, then use CloudFormation to deploy each test environment into that shared VPC.**

This is the correct answer because:

1. **Single VPC = Single Transit Gateway Attachment**: A Transit Gateway attachment is created per VPC. By using one shared VPC, you create the attachment and configure routing exactly once. This eliminates repetitive networking setup for every test environment.

2. **Least Operational Overhead**: CloudFormation stacks can be created and deleted on demand within the same VPC. Each stack provisions only the EC2 instance and Auto Scaling group — not the entire network stack. This makes creation and teardown fast, simple, and reliable.

3. **Resource Efficiency**: You avoid hitting VPC limits per Region (default 5), and you don't fragment your IP address space across dozens of VPCs. A single VPC with well-planned subnets can host many isolated test environments using security groups for segmentation.

4. **Centralized Routing**: All test environments share the same route tables pointing to the Transit Gateway, ensuring consistent connectivity to the on-premises central server without per-environment route management.

### ❌ Incorrect Options

**Option A — CloudFormation StackSets deploying a new VPC per test environment**

- Creating a new VPC for *every* test environment is the antithesis of "least overhead." Each VPC requires its own Transit Gateway attachment, route table configuration, subnet design, and NAT gateway (for internet access). This ballooning complexity wastes time and hits service quotas quickly.
- StackSets are designed for multi-account/multi-region deployments, not for repeatedly creating VPCs in a single account. Using them this way is an anti-pattern.

**Option C — New AWS account per test environment via Organizations + StackSets**

- This introduces **massive** operational overhead: account creation has API rate limits, each account needs its own VPC, Transit Gateway sharing (via AWS RAM), and routing. Account creation and teardown takes minutes, not seconds.
- AWS Organizations is powerful for multi-account governance but wildly disproportionate for ephemeral test environments lasting minutes or hours.

**Option D — Dockerize EC2 instances and deploy via EKS**

- This requires re-platforming from EC2-based testing to containerized workloads — a fundamental architectural change that introduces Kubernetes cluster management, container image pipelines, and EKS operational burden.
- The question explicitly states the environments are EC2-based. Adding EKS adds the Kubernetes control plane, worker nodes, and networking complexity — the opposite of "least operational overhead."

### 🔧 Hands-On Knowledge

- **Transit Gateway (TGW)** is a network transit hub that interconnects VPCs, VPN connections, and Direct Connect. Each VPC requires exactly one TGW attachment; all resources in that VPC share it.
- **Single VPC multi-environment pattern**: Use separate subnets per environment (or per team), with security groups providing isolation. Tag resources by environment for cost tracking.
- **CloudFormation for test environments**: Define a template that takes parameters like `EnvironmentName`, `SubnetId`, and `InstanceType`. Use `DeletionPolicy: Delete` so destroying the stack cleanly removes all resources.
- **Auto Scaling group of size 1**: For a single-instance test environment, set desired/min/max to 1. If the instance fails, ASG automatically replaces it — adding resilience even for test workloads.
- **Avoid TGW attachment churn**: Creating and deleting TGW attachments repeatedly can hit API rate limits and cause propagation delays. A shared VPC pattern avoids this entirely.
