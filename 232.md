A company uses an organization in AWS Organizations to manage the company's AWS accounts. The company uses AWS CloudFormation to deploy all infrastructure. A finance team wants to build a chargeback model. The finance team asked each business unit to tag resources by using a predefined list of project values.

When the finance team used the AWS Cost and Usage Report in AWS Cost Explorer and filtered based on project, the team noticed noncompliant project values. The company wants to enforce the use of project tags for new resources.

Which solution will meet these requirements with the LEAST effort?

A. Create a tag policy that contains the allowed project tag values in the organization's management account. Create an SCP that denies the cloudformation:CreateStack API operation unless a project tag is added. Attach the SCP to each OU.
B. Create a tag policy that contains the allowed project tag values in each OU. Create an SCP that denies the cloudformation:CreateStack API operation unless a project tag is added. Attach the SCP to each OU.
C. Create a tag policy that contains the allowed project tag values in the AWS management account. Create an IAM policy that denies the cloudformation:CreateStack API operation unless a project tag is added. Assign the policy to each user.
D. Use AWS Service Catalog to manage the CloudFormation stacks as products. Use a TagOptions library to control project tag values. Share the portfolio with all OUs that are in the organization.


Suggest Answer: A

---

## Analysis

### Understanding the Problem

A company managing multiple AWS accounts through AWS Organizations deploys all infrastructure via AWS CloudFormation. The finance team is building a **chargeback model** that relies on a `project` tag with predefined allowed values. However, the AWS Cost and Usage Report (CUR) reveals **noncompliant project tag values** — meaning some resources were tagged with values outside the approved list, or the tag was missing entirely. The goal is to **enforce the use of compliant project tags** on all new resources with the **least operational effort**.

Key requirements:
1. Enforce that a `project` tag is always present on newly created resources.
2. Ensure the tag value comes from a **predefined list** of allowed project values.
3. Achieve this with the **least effort** across a multi-account AWS Organization.

### Correct Option(s) Explanation

**Option A (Correct):** Create a **tag policy** in the organization's **management account** that defines the allowed `project` tag values. Then create a **Service Control Policy (SCP)** that denies the `cloudformation:CreateStack` API action unless a `project` tag is attached, and attach this SCP to each Organizational Unit (OU).

**Why this works:**

- **Tag Policies** (part of AWS Organizations) allow you to standardize tags across all accounts in your organization. When created in the management account, tag policies **flow down** to all member accounts within the OUs to which they are attached. You define:
  - Which tag keys are mandatory (`project`)
  - Allowed values for those keys (the predefined project list)
  - Compliance enforcement (block noncompliant tagging operations)

- **SCPs** (Service Control Policies) act as guardrails at the organization/OU level. They define the maximum available permissions for all IAM entities (users, roles) within the affected accounts. An SCP denying `cloudformation:CreateStack` without the required tag condition ensures that **no principal** can create a CloudFormation stack — and thus no new resources — without a compliant `project` tag.

- The combination of **tag policy + SCP** provides **two-layer enforcement**: the tag policy governs the tagging rules and reports compliance, while the SCP actively **blocks** noncompliant API calls. This is the **least effort** because both are managed centrally from the management account and apply organization-wide automatically.

- **Reference:** [AWS Tag Policies Documentation](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_tag-policies.html) | [AWS SCP Documentation](https://docs.aws.amazon.com/organizations/latest/userguide/orgs_manage_policies_scps.html)

### Incorrect Options Explanation

**Option B:** Create a tag policy in **each OU** instead of the management account.

- ❌ Tag policies are designed to be created in the **management account** and then attached to OUs or accounts. Creating a separate tag policy per OU is redundant, introduces inconsistency, and requires significantly more effort to maintain. When you update allowed values, you would need to update every OU-level tag policy individually. The management account is the correct central place.

**Option C:** Use an **IAM policy** (instead of an SCP) and assign it to **each user**.

- ❌ IAM policies assigned per-user do not scale in a multi-account organization. You would need to create and attach policies for every IAM user across every account, which is **high operational overhead**. Moreover, IAM policies can be overridden by other attached policies (unless explicit deny), whereas SCPs provide an **un-overridable boundary** that applies to all principals in the account. An IAM policy also cannot prevent a user with administrator access from removing the restriction. SCPs **cannot be overridden** by any principal within the account.

**Option D:** Use **AWS Service Catalog** with TagOptions to control project tag values.

- ❌ While Service Catalog's TagOptions can enforce tags on provisioned products, this approach requires:
  - Packaging every CloudFormation template as a Service Catalog **product**
  - Creating and managing **portfolios**
  - Sharing portfolios across OUs
  - Training teams to use Service Catalog instead of direct CloudFormation deployments
  
  This is a **heavyweight solution** that fundamentally changes the deployment workflow. It violates the "least effort" requirement and is disproportionate for the simple task of enforcing a tag value list.

### Enterprise Hands-on Knowledge

1. **Tag Policy Best Practices:** In production environments, tag policies should be defined centrally and attached to the **root** of the organization, with inheritance ensuring all OUs and accounts are covered. Use the `enforcedFor` setting to specify which resource types the policy applies to (e.g., `ec2:instance`, `s3:bucket`).

2. **SCP Condition for Tag Enforcement:** A typical SCP statement to enforce the `project` tag on CloudFormation looks like:
   ```json
   {
     "Effect": "Deny",
     "Action": "cloudformation:CreateStack",
     "Resource": "*",
     "Condition": {
       "Null": {
         "aws:RequestTag/project": "true"
       }
     }
   }
   ```
   To also enforce allowed values, use `StringNotEquals` with `aws:RequestTag/project` against the allowed list.

3. **Chargeback Model Foundation:** Tag enforcement is the first step. For a mature chargeback model, also consider:
   - **AWS Cost Categories** to group costs by project/business unit
   - **AWS Budgets** with alerts per tag
   - **CUR (Cost and Usage Report)** delivered to S3 and queried via Athena for detailed analysis

4. **Common Pitfall in SAP-C02:** Many candidates choose IAM policies (Option C) thinking they are "simpler." However, the exam tests the understanding that **SCPs are the correct guardrail mechanism at the organization level**, while IAM policies govern permissions within an account. SCPs establish the **maximum boundary**; IAM policies grant permissions within that boundary.

5. **Real-World Governance:** In enterprise AWS environments, combine tag policies with **AWS Config rules** (e.g., `required-tags`) to detect noncompliant resources that may have been created before enforcement was in place, and use **AWS Resource Groups** to organize resources by tag for operational visibility.
