A company is running applications on AWS in a multi-account environment. The company's sales team and marketing team use separate AWS accounts in AWS Organizations.  
  
The sales team stores petabytes of data in an Amazon S3 bucket. The marketing team uses Amazon QuickSight for data visualizations. The marketing team needs access to data that the sates team stores in the S3 bucket. The company has encrypted the S3 bucket with an AWS Key Management Service (AWS KMS) key. The marketing team has already created the IAM service role for QuickSight to provide QuickSight access in the marketing AWS account. The company needs a solution that will provide secure access to the data in the S3 bucket across AWS accounts.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Create a new S3 bucket in the marketing account. Create an S3 replication rule in the sales account to copy the objects to the new S3 bucket in the marketing account. Update the QuickSight permissions in the marketing account to grant access to the new S3 bucket.
- B. Create an SCP to grant access to the S3 bucket to the marketing account. Use AWS Resource Access Manager (AWS RAM) to share the KMS key from the sates account with the marketing account. Update the QuickSight permissions in the marketing account to grant access to the S3 bucket.
- C. Update the S3 bucket policy in the marketing account to grant access to the QuickSight role. Create a KMS grant for the encryption key that is used in the S3 bucket. Grant decrypt access to the QuickSight role. Update the QuickSight permissions in the marketing account to grant access to the S3 bucket.
- D. Create an IAM role in the sales account and grant access to the S3 bucket. From the marketing account, assume the IAM role in the sales account to access the S3 bucket. Update the QuickSight rote, to create a trust relationship with the new IAM role in the sales account.

## 1. Understanding the Problem

**Scenario:**
* **Source (Sales Account):** Stores petabytes of data in an **Amazon S3** bucket. The data is encrypted using **AWS KMS**.
* **Destination (Marketing Account):** Uses **Amazon QuickSight** for visualization.
* **Requirement:** The Marketing team's QuickSight needs to read data from the Sales team's S3 bucket.
* **Constraint:** The solution must have the **LEAST operational overhead**.

**Key Challenges:**
1.  **Cross-Account Access:** Resources are in different AWS accounts within an Organization.
2.  **Encryption:** The data is encrypted; access requires permissions for both the storage (S3) and the encryption key (KMS).
3.  **Identity:** QuickSight uses a specific IAM service role in the Marketing account.

---

## 2. Services Explanation

* **Amazon QuickSight:** A business intelligence service. It accesses AWS data sources using an IAM Service Role (often called the QuickSight execution role) configured in the account where QuickSight is running.
* **Amazon S3 (Simple Storage Service):** Object storage. Access is controlled via **IAM User/Role Policies** (Identity-based) and **S3 Bucket Policies** (Resource-based).
* **AWS KMS (Key Management Service):** Manages encryption keys. Access to encrypted data requires permission to use the key (specifically `kms:Decrypt`). This is controlled via **KMS Key Policies**.
* **IAM Roles & Trust Policies:** A mechanism to delegate access. Cross-account access often involves a resource in Account A trusting a principal (User/Role) in Account B.
* **SCP (Service Control Policy):** Used in AWS Organizations to manage the *maximum available permissions* for accounts. It does not grant permissions; it only filters/limits them.



---

## 3. Correct Option Analysis

**Correct Answer:** **D**

> **Option D:** Create an IAM role in the sales account and grant access to the S3 bucket. From the marketing account, assume the IAM role in the sales account to access the S3 bucket. Update the QuickSight role to create a trust relationship with the new IAM role in the sales account.

**Why this is the correct answer (in the context of this specific question):**

This is a classic "process of elimination" question found in the Professional exam. While there is a standard method (Bucket Policies) often used, we must select the *valid* option presented.

1.  **Validity:** This option utilizes the **Role Assumption** pattern (`sts:AssumeRole`).
    * You create a Role in the **Sales Account** (where the data lives) that has local permissions to read S3 and decrypt via KMS.
    * You configure the Trust Policy of this Sales Role to allow the **Marketing Account** (specifically the QuickSight role) to assume it.
    * You update the QuickSight Role in the Marketing account to allow it to call `sts:AssumeRole` on the Sales Role.
2.  **Feasibility:** This effectively bridges the account gap. The QuickSight service acts as the identity in Marketing, assumes the identity in Sales, and gains access to the data as a local principal in the Sales account.

**Important Note on "Least Overhead":**
Typically, the absolute least overhead for S3 cross-account access is modifying the **S3 Bucket Policy** and **KMS Key Policy** to trust the external QuickSight role directly (Resource-based policy). However, look closely at **Option C** (the competitor for this logic) in the section below to see why it is incorrect *based on the specific wording of the question*.

## 4. Incorrect Options Analysis

**A. Create a new S3 bucket in the marketing account. Create an S3 replication rule...**
* **Why Incorrect:** This violates "Least Operational Overhead." Replicating **petabytes** of data incurs massive data duplication costs, transfer costs, and management overhead (monitoring replication lag). It is an inefficient architectural choice for simply reading data.

**B. Create an SCP to grant access to the S3 bucket... Use AWS RAM...**
* **Why Incorrect:**
    1.  **SCP Misunderstanding:** SCPs (Service Control Policies) are **guardrails**. They can only *deny* access or allow what is already granted by IAM; they cannot *grant* permissions that don't exist.
    2.  **RAM Limitations:** AWS Resource Access Manager (RAM) is generally used for networking resources (Subnets, Transit Gateways) or specific resource types. You do not use RAM to share KMS keys; KMS keys are shared via Key Policies.

**C. Update the S3 bucket policy in the marketing account... Create a KMS grant...**
* **Why Incorrect (The Trick):** This option describes the standard/preferred architectural pattern (Resource-based policies) but contains a **fatal factual error**.
    * The option says: "Update the S3 bucket policy **in the marketing account**."
    * The S3 bucket is located in the **Sales account**.
    * You cannot create or update an S3 bucket policy in the Marketing account for a bucket that resides in the Sales account. The policy must be applied in the account where the resource exists.
    * Because of this location error, this option is technically impossible to execute as written.
