A company has 10 accounts that are part of an organization in AWS Organizations. AWS Config is configured in each account. All accounts belong to either the Prod OU or the NonProd OU.  
  
The company has set up an Amazon EventBridge rule in each AWS account to notify an Amazon Simple Notification Service (Amazon SNS) topic when an Amazon EC2 security group inbound rule is created with 0.0.0.0/0 as the source. The company’s security team is subscribed to the SNS topic.  
  
For all accounts in the NonProd OU, the security team needs to remove the ability to create a security group inbound rule that includes 0.0.0.0/0 as the source.  
  
Which solution will meet this requirement with the LEAST operational overhead?

- A. Modify the EventBridge rule to invoke an AWS Lambda function to remove the security group inbound rule and to publish to the SNS topic. Deploy the updated rule to the NonProd OU.
- B. Add the vpc-sg-open-only-to-authorized-ports AWS Config managed rule to the NonProd OU.
- C. Configure an SCP to allow the ec2:AuthorizeSecurityGroupIngress action when the value of the aws:SourceIp condition key is not 0.0.0.0/0. Apply the SCP to the NonProd OU.
- D. Configure an SCP to deny the ec2:AuthorizeSecurityGroupIngress action when the value of the aws:SourceIp condition key is 0.0.0.0/0. Apply the SCP to the NonProd OU.

### Understanding the Problem

The company uses **AWS Organizations** with multiple accounts divided into **Prod OU** and **NonProd OU**. **AWS Config** is active in all accounts. A reactive mechanism is in place using **Amazon EventBridge** and **Amazon SNS** to _notify_ the security team whenever an **Amazon EC2 security group inbound rule** with **0.0.0.0/0** (meaning open to the entire internet) is created.

The core requirement is to **prevent** (remove the ability to create) the $0.0.0.0/0$ inbound rule in **all accounts within the NonProd OU** with the **LEAST operational overhead**.

Prevention is a **proactive** security measure, which is generally more effective and results in lower operational overhead than reactive (detect and remediate) measures for common security policy enforcement.

### Correct Option(s) Explanation

The requirement is to **prevent** the action in the entire NonProd OU with the **LEAST operational overhead**.

#### **D. Configure an SCP to deny the ec2:AuthorizeSecurityGroupIngress action when the value of the aws:SourceIp condition key is 0.0.0.0/0. Apply the SCP to the NonProd OU.**

- **Mechanism:** This solution uses a **Service Control Policy (SCP)**, which is the **proactive guardrail** mechanism within AWS Organizations.
    
- **Proactive Prevention:** An SCP with a `Deny` effect applied to the $\text{ec2:AuthorizeSecurityGroupIngress}$ action specifically when the **condition** $\text{aws:SourceIp}$ is $0.0.0.0/0$ will **prevent** any account or user in the NonProd OU from successfully creating this insecure rule.
    
- **Low Operational Overhead:**
    
    - It's a **single-time configuration** and deployment to the NonProd OU.
        
    - It applies to _all_ current and future accounts in that OU automatically.
        
    - It **stops the creation** at the API call level, eliminating the need for any ongoing monitoring (like AWS Config) or remediation (like Lambda functions or the existing EventBridge/SNS setup).
        
    - This is the standard and most efficient way to enforce a high-level security policy across multiple accounts in an organization.

### Incorrect Options Explanation

#### **A. Modify the EventBridge rule to invoke an AWS Lambda function to remove the security group inbound rule and to publish to the SNS topic. Deploy the updated rule to the NonProd OU.**

- **Mechanism:** This is a **reactive** solution. It **detects** the insecure rule creation (via EventBridge) and then **remediates** it (via Lambda).
    
- **Operational Overhead:** This has **high operational overhead**.
    
    - Requires development, testing, and maintenance of a custom **Lambda function** (code maintenance).
        
    - The Lambda function requires $\text{iam:PassRole}$, $\text{ec2:RevokeSecurityGroupIngress}$, and $\text{sns:Publish}$ permissions, increasing the attack surface.
        
    - The rule is _created first_, then _deleted_, leading to a brief window of insecurity and unnecessary resource changes that generate noise in CloudTrail and Config.
        
    - It requires deploying the rule and Lambda to **each account** in the NonProd OU, or using a complex, centralized deployment mechanism.
        

#### **B. Add the vpc-sg-open-only-to-authorized-ports AWS Config managed rule to the NonProd OU.**

- **Mechanism:** This is a **detective** solution. The AWS Config managed rule will check for non-compliant $\text{0.0.0.0/0}$ rules.
    
- **Operational Overhead:** This only **detects** the issue; it does **not prevent** the rule from being created, nor does it automatically **remediate** it. A separate remediation mechanism (like an EventBridge rule reacting to the Config non-compliance event, which then triggers a Lambda function) would still be required, leading to high operational overhead (similar to Option A, but with an extra service/detection step). The existing EventBridge/SNS rule is already performing a similar detection/notification function, so this option doesn't solve the core _prevention_ requirement.
    

#### **C. Configure an SCP to allow the ec2:AuthorizeSecurityGroupIngress action when the value of the aws:SourceIp condition key is not 0.0.0.0/0. Apply the SCP to the NonProd OU.**

- **Mechanism:** This uses an SCP, but with an **`Allow`** effect.
    
- **Why it's wrong (SCP logic):** SCPs work on a principle of **Deny overrides Allow**. An $\text{Allow}$ SCP _only_ defines the maximum permissions. If a user's IAM policy doesn't explicitly grant $\text{ec2:AuthorizeSecurityGroupIngress}$, this SCP does nothing. Even if their IAM policy _does_ grant it, an $\text{Allow}$ SCP restricting the condition will only be effective if it's placed in a context where _all other_ explicit $\text{Deny}$ policies are absent. The proper way to enforce a _restriction_ like this is to use an **explicit `Deny`** statement, which will always take precedence over any $\text{Allow}$ statements in the IAM identity policy or other policies. An $\text{Allow}$ policy here would be ineffective as a guardrail. Furthermore, it shares the same technical flaw as Option D by incorrectly using $\text{aws:SourceIp}$ instead of $\text{ec2:CidrBlock}$.
