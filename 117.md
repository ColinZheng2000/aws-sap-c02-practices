A company is subject to regulatory audits of its financial information. External auditors who use a single AWS account need access to the company's AWS account. A solutions architect must provide the auditors with secure, read-only access to the company's AWS account. The solution must comply with AWS security best practices.

Which solution will meet these requirements?

A. In the company's AWS account, create resource policies for all resources in the account to grant access to the auditors' AWS account. Assign a unique external ID to the resource policy.
B. In the company's AWS account, create an IAM role that trusts the auditors' AWS account. Create an IAM policy that has the required permissions. Attach the policy to the role. Assign a unique external ID to the role's trust policy.
C. In the company's AWS account, create an IAM user. Attach the required IAM policies to the IAM user. Create API access keys for the IAM user. Share the access keys with the auditors.
D. In the company's AWS account, create an IAM group that has the required permissions. Create an IAM user in the company's account for each auditor. Add the IAM users to the IAM group.

# Cross‑Account Access for External Auditors  
#aws/sap-c02 #security #iam #cross-account-access #audit #multi-account

- --

## Understanding the Problem

- An external audit team has **their own single AWS account**.
- They need **secure, read-only** access to **your company’s AWS account**.
- Must follow **AWS security best practices**.
- Needs to be **cross-account access** (auditors in their own account, not as IAM users in your account).

Key themes:
- Cross-account IAM access
- Use of IAM roles vs IAM users
- Read-only permissions
- Use of **external ID** for third‑party access

We need to pick the option that:
1. Uses **cross-account IAM roles** (not long-term user credentials).
2. Grants **least privilege, read-only** permissions.
3. Uses **external IDs** for a third-party (auditors) to prevent the confused‑deputy problem.
4. Aligns with AWS security best practices.

- --

## Related AWS Services and Concepts

### AWS Identity and Access Management (IAM)
Core features relevant here:

1. **IAM Roles (Cross-Account Access)**  
   - An IAM role with a **trust policy** that allows principals from another AWS account to assume it.  
   - Used for temporary credentials via **STS:AssumeRole**.  
   - Best practice for external/no-human entities or cross-account access.  
   - Can include an **external ID** condition to prevent confused‑deputy attacks.  
   - Docs:  
     - [Using IAM roles](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles.html)  
     - [How to enable cross-account access](https://docs.aws.amazon.com/IAM/latest/UserGuide/tutorial_cross-account-with-roles.html)

2. **IAM Policies**
   - Define **permissions** (e.g., `ReadOnlyAccess`, custom read-only policy).
   - Attached to roles, users, or groups.
   - Managed vs inline policies.

3. **IAM Users and Groups**
   - IAM user = long-term credentials (password, access keys).
   - IAM group = collection of users, attaches policies at group level.
   - Best practice: **avoid creating IAM users for external third parties**; prefer roles and STS.

4. **External ID**
   - A random string that a third party must provide when assuming a role.  
   - Used in the role’s trust policy with a `Condition` on `sts:ExternalId`.  
   - Prevents a third party from being tricked into using its permissions to access another customer’s resources.  
   - Docs:  
     - [How to use an external ID](https://docs.aws.amazon.com/IAM/latest/UserGuide/id_roles_create_for-user_externalid.html)

5. **Resource-based Policies**
   - For some services (S3, KMS, SNS, SQS, etc.), you can put policies on the resource itself.  
   - They are often used for cross-account access, but **not generally for “all resources in an account,”** and not the typical pattern for third‑party read across many services.

## Correct Option and Insightful Explanation

### ✅ Correct Answer: **B**

> In the company's AWS account, create an IAM role that trusts the auditors' AWS account. Create an IAM policy that has the required permissions. Attach the policy to the role. Assign a unique external ID to the role's trust policy.

* *Why this is correct:**

1. **Cross-account IAM role**  
   - This is the standard pattern for giving a third-party AWS account access:  
     - You create an IAM role in your account.
     - The role’s **trust policy** allows the auditors’ AWS account (their root or a specific IAM role/user in that account) to `sts:AssumeRole`.
   - Reference: *"How to enable cross-account access to AWS resources"* in the IAM User Guide.

2. **Read-only, least-privilege permissions**  
   - You attach an IAM permissions policy to the role granting the **minimum read-only access** needed (e.g., `ReadOnlyAccess` or a custom restricted read-only policy).
   - Auditors assume this role and receive **temporary security credentials** with read-only rights.

3. **Security best practice – external ID**  
   - Since the auditors are a **third-party** (not your own organization), using an **external ID** is recommended to prevent the **confused-deputy problem**.  
   - The trust policy might include something like:

     ```json
     {
       "Version": "2012-10-17",
       "Statement": [
         {
           "Effect": "Allow",
           "Principal": { "AWS": "arn:aws:iam::<auditor-account-id>:root" },
           "Action": "sts:AssumeRole",
           "Condition": {
             "StringEquals": {
               "sts:ExternalId": "some-unique-string"
             }
           }
         }
       ]
     }
     ```

   - The auditors must supply `ExternalId` when calling `sts:AssumeRole`.

4. **No long-term access keys**  
   - STS session credentials are temporary, reduce risk if compromised.
   - Fully aligned with AWS best practices: *"Use roles instead of long-term access keys wherever possible."*

## Incorrect Options and Insightful Explanation

### ❌ Option A

> In the company's AWS account, create resource policies for all resources in the account to grant access to the auditors' AWS account. Assign a unique external ID to the resource policy.

Issues:

1. **Not scalable or realistic**  
   - You would need to create or modify resource policies on **every resource** (S3 buckets, KMS keys, SNS topics, SQS queues, etc.).
   - Many services **do not support resource-based policies**. IAM roles/users are needed for them.

2. **External ID doesn’t apply here**  
   - **External ID** is a feature of **STS AssumeRole trust policies**, not generic resource-based policies.
   - This option misuses the concept of external ID.

3. **Not the standard design for third-party account-wide read access**  
   - The standard, documented pattern is: **cross-account IAM role** (Option B), not mass resource-based policies.

Therefore, this approach is both impractical and incorrect in how it uses external IDs.

- --

### ❌ Option C

> In the company's AWS account, create an IAM user. Attach the required IAM policies to the IAM user. Create API access keys for the IAM user. Share the access keys with the auditors.

Issues:

1. **Long-term credentials shared externally**  
   - Sharing IAM user access keys with an external party is **strongly discouraged**.
   - If keys are leaked or not rotated, risk is high.

2. **Poor manageability and governance**  
   - You lose clear separation of responsibility; auditors are using *your* IAM identity.
   - Harder to manage expiration. Best practice is to use **temporary credentials via roles**.

3. **Not aligned with AWS best practices**  
   - IAM best practices:  
     - Prefer **roles and temporary credentials** over access keys.  
     - Do not share credentials across organizational boundaries.

So while it technically can give read-only access, it fails the “AWS security best practices” requirement.

- --

### ❌ Option D

> In the company's AWS account, create an IAM group that has the required permissions. Create an IAM user in the company's account for each auditor. Add the IAM users to the IAM group.

Issues:

1. **Creates users for external individuals in your account**  
   - This tightly couples external auditors to your IAM identity system.
   - Operational overhead: manage a user per auditor, handle password policies, MFA, lifecycle, etc.

2. **Long-term credentials**  
   - Again, this relies on long-lived credentials, contradicting the best practice of using roles with temporary credentials.

3. **Poor fit for the scenario of external organization**  
   - Auditors already have an AWS account; best practice is to let them assume a **role** from their own account instead of onboarding them as users in your account.

Even though this could be made “secure-ish” with MFA, limited policies, and time-bound credentials, it is **not** the recommended approach compared to cross-account roles.

- --

## Key Takeaways for the SAP‑C02 Exam

- For third-party access (partners, auditors, vendors) who have their own AWS accounts:
  - Use **cross-account IAM roles** with **trust policies** to their account.
  - Provide **read-only** or least-privilege permissions via attached IAM policies.
  - Use **STS AssumeRole** and **external IDs** to prevent confused‑deputy issues.
- Avoid:
  - Sharing IAM user access keys with external parties.
  - Creating IAM users for external organizations when they have their own AWS accounts.
  - Relying on resource-based policies for broad, multi-service access across an account.

**Answer: B**
