An online retail company is migrating its legacy on-premises .NET application to AWS. The application runs on load-balanced frontend web servers, load-balanced application servers, and a Microsoft SQL Server database.

The company wants to use AWS managed services where possible and does not want to rewrite the application. A solutions architect needs to implement a solution to resolve scaling issues and minimize licensing costs as the application scales.

Which solution will meet these requirements MOST cost-effectively?

A. Deploy Amazon EC2 instances in an Auto Scaling group behind an Application Load Balancer for the web tier and for the application tier. Use Amazon Aurora PostgreSQL with Babelfish turned on to replatform the SQL Server database.
B. Create images of all the servers by using AWS Database Migration Service (AWS DMS). Deploy Amazon EC2 instances that are based on the on-premises imports. Deploy the instances in an Auto Scaling group behind a Network Load Balancer for the web tier and for the application tier. Use Amazon DynamoDB as the database tier.
C. Containerize the web frontend tier and the application tier. Provision an Amazon Elastic Kubernetes Service (Amazon EKS) cluster. Create an Auto Scaling group behind a Network Load Balancer for the web tier and for the application tier. Use Amazon RDS for SQL Server to host the database.
D. Separate the application functions into AWS Lambda functions. Use Amazon API Gateway for the web frontend tier and the application tier. Migrate the data to Amazon S3. Use Amazon Athena to query the data.

Suggest Answer: A

---

## 📋 Analysis

### Understanding the Problem

An online retail company is migrating a **legacy on-premises .NET application** with a classic three-tier architecture (web frontend → application servers → Microsoft SQL Server database) to AWS. The constraints are:

1. **Use AWS managed services where possible** — reduce operational burden.
2. **Do not rewrite the application** — preserve the .NET codebase.
3. **Resolve scaling issues** — the current on-premises setup cannot scale elastically.
4. **Minimize licensing costs as the application scales** — SQL Server license costs grow linearly with vCPU count; this must be addressed.

The challenge is finding a cost-effective migration path that provides auto-scaling for compute tiers and eliminates SQL Server licensing without requiring a full application rewrite.

---

### ✅ Correct Option: A

**Deploy EC2 instances in Auto Scaling groups behind an Application Load Balancer for web and application tiers. Use Amazon Aurora PostgreSQL with Babelfish to replatform the SQL Server database.**

**Why this is correct:**

| Requirement | How Option A Fulfills It |
|---|---|
| **Managed services** | Amazon Aurora is a fully managed relational database service. EC2 Auto Scaling + ALB provide managed-like scaling for compute. |
| **No rewrite** | **Babelfish for Aurora PostgreSQL** provides T-SQL language support and wire-protocol compatibility (Tabular Data Stream / TDS) so that .NET applications using `System.Data.SqlClient` or similar SQL Server drivers can connect to Aurora **with minimal to no code changes**. |
| **Scaling** | Auto Scaling groups handle variable load for web and app tiers. Aurora provides up to 15 read replicas and automatic storage scaling up to 128 TiB. |
| **License cost elimination** | PostgreSQL (underlying Aurora engine) is open-source — no SQL Server licensing fees. Babelfish translates SQL Server proprietary features (T-SQL, MERGE, etc.) to PostgreSQL equivalents. |

**Babelfish Deep Dive:**
- Babelfish runs as an extension within Aurora PostgreSQL.
- It emulates the SQL Server wire protocol (TDS), so existing ADO.NET / ODBC / JDBC drivers work unchanged.
- Supported T-SQL features include: stored procedures, functions, triggers, views, `MERGE`, `CURSOR`, `TRY...CATCH`, and many built-in functions.
- Reference: [Babelfish for Aurora PostgreSQL — Architecture](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/babelfish-architecture.html)

### ❌ Incorrect Options

| Option | Why It Fails |
|---|---|
| **B (DMS server imaging + DynamoDB)** | **AWS DMS (Database Migration Service)** is for *data migration*, not server imaging — server replication is handled by **AWS Application Migration Service (MGN)** or **Server Migration Service (SMS)**. More critically, **DynamoDB is a NoSQL key-value/document database** that is fundamentally incompatible with a relational SQL Server schema and the .NET application's SQL queries. Moving from SQL Server to DynamoDB would require a **complete application rewrite**, violating the "do not rewrite" constraint. |
| **C (EKS + RDS for SQL Server)** | While this uses managed services (EKS, RDS), it keeps **RDS for SQL Server**, which still incurs SQL Server **license-included** or **BYOL** costs that scale with instance size. For a retail application that scales significantly, SQL Server licensing can become the dominant cost. The question explicitly asks to **minimize licensing costs**, which this option fails to address. Additionally, containerizing a legacy .NET application on EKS introduces operational complexity (pod management, service mesh, etc.) without the licensing benefit. |
| **D (Lambda + API Gateway + S3 + Athena)** | This represents a **complete architectural rewrite** from a monolithic .NET application to a serverless event-driven architecture. .NET on Lambda requires significant refactoring (Lambda function handlers, stateless design, 15-minute execution limit). **Amazon Athena** is an interactive query service for analytics over S3 data — it is **not an OLTP database** and cannot serve as the backend for a transactional retail application. This option violates both "no rewrite" and "appropriate database technology" constraints. |

### 🏭 Hands-on Enterprise Knowledge

1. **Babelfish Compatibility Assessment:**
   - Before committing to Babelfish, run the **Babelfish Compass Tool** (an open-source assessment tool) against your SQL Server schema and T-SQL code. It generates a compatibility report showing which features are fully supported, partially supported, or unsupported (e.g., SQL Server Agent jobs, certain system stored procedures).
   - Reference: [Babelfish Compass Tool on GitHub](https://github.com/babelfish-for-postgresql/babelfish_compass)

2. **Migration Pattern — Replatform (Lift-and-Reshape):**
   - This scenario exemplifies the **replatform** strategy in the **6 R's of migration** (Rehost, Replatform, Repurchase, Refactor, Retain, Retire).
   - Compute tier remains on EC2 (rehost), but the database is replatformed from SQL Server to Aurora PostgreSQL with Babelfish. This balances cost optimization with minimal application changes.

3. **SQL Server Licensing Economics:**
   - SQL Server Enterprise Edition on RDS costs approximately **$0.25–$1.50+ per vCPU-hour** in licensing alone (varies by region and edition). For a 16-vCPU instance running 24/7, that's roughly **$2,880–$17,280/month** in SQL Server license costs.
   - Aurora PostgreSQL pricing is based on I/O consumption + instance hours — **no separate license fee**. For read-heavy retail workloads, Aurora's auto-scaling read replicas provide significant savings compared to SQL Server Always On Availability Groups.

4. **.NET on EC2 Auto Scaling Considerations:**
   - Session state must be externalized (e.g., to **Amazon ElastiCache for Redis** or **DynamoDB**) — ASP.NET's default in-memory session state breaks with multiple instances.
   - Use **AWS Systems Manager Session Manager** instead of RDP for secure instance access.
   - Deploy via **AWS CodeDeploy** with in-place or blue/green deployment strategies.

5. **Why Not Windows Containers on EKS for .NET?**
   - Windows container support on EKS requires Windows worker nodes, which have **higher minimum resource requirements** and **slower startup times** than Linux containers.
   - Windows containers are significantly larger (multiple GB vs MB for Linux), increasing ECR storage and pull time costs.
   - For a "do not rewrite" migration, plain EC2 instances provide a simpler, more predictable path.
