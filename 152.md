A company is running an application in the AWS Cloud. The core business logic is running on a set of Amazon EC2 instances in an Auto Scaling group. An Application Load Balancer (ALB) distributes traffic to the EC2 instances. Amazon Route 53 record api.example.com is pointing to the ALB.

The company's development team makes major updates to the business logic. The company has a rule that when changes are deployed, only 10% of customers can receive the new logic during a testing window. A customer must use the same version of the business logic during the testing window.

How should the company deploy the updates to meet these requirements?

A. Create a second ALB, and deploy the new logic to a set of EC2 instances in a new Auto Scaling group. Configure the ALB to distribute traffic to the EC2 instances. Update the Route 53 record to use weighted routing, and point the record to both of the ALBs.
B. Create a second target group that is referenced by the ALDeploy the new logic to EC2 instances in this new target group. Update the ALB listener rule to use weighted target groups. Configure ALB target group stickiness.
C. Create a new launch configuration for the Auto Scaling group. Specify the launch configuration to use the AutoScalingRollingUpdate policy, and set the MaxBatchSize option to 10. Replace the launch configuration on the Auto Scaling group. Deploy the changes.
D. Create a second Auto Scaling group that is referenced by the ALB. Deploy the new logic on a set of EC2 instances in this new Auto Scaling group. Change the ALB routing algorithm to least outstanding requests (LOR). Configure ALB session stickiness.

Suggest Answer: B

## Analysis

### Understanding the Problem

A company has a production application running on EC2 instances behind an ALB with Route 53 DNS. When deploying major updates, the business rule requires:
1. **Only 10% of customers** receive the new business logic during a testing window (phased rollout / canary deployment).
2. **Session affinity (stickiness)**: A customer must consistently use the *same* version of the business logic throughout the testing window.

The core challenge is to implement a **weighted traffic split** between old and new application versions at a 90:10 ratio, while ensuring **session persistence** so users don't flip between versions mid-session.

### Correct Option: B

**Create a second target group, deploy new logic to EC2 instances in that target group, use ALB weighted target groups, and configure stickiness.**

- **ALB Weighted Target Groups**: ALB listener rules support forwarding traffic to multiple target groups with configurable weights. Setting a 90:10 weight distribution between the old and new target groups directly satisfies the 10% customer requirement.
- **Target Group Stickiness (Sticky Sessions)**: ALB supports stickiness at the target group level using cookies (application-based or duration-based). Once a customer is routed to either the old or new target group, stickiness ensures all subsequent requests from that customer go to the same target group — thus the same application version — for the duration of the testing window.
- **Single ALB Architecture**: Using the existing ALB (no need for a second ALB) minimizes architectural complexity. Weighted target group routing is a native ALB feature, making this the most elegant and operationally simple solution.
- **No Route 53 Changes**: The `api.example.com` record continues pointing to the same ALB. Traffic splitting happens at the ALB layer, not at the DNS layer — which is more precise (DNS-based weighting is probabilistic and cached by clients).

**Why this is the best approach:**
- Precise 10% traffic split (not approximate, as with Route 53 weighted routing)
- Native session stickiness at the target group level
- Single ALB = simpler architecture
- Easy rollback: simply adjust weights back to 100:0

### Incorrect Options

**A: Second ALB + Route 53 Weighted Routing**
- **Overly complex**: Adding a second ALB doubles the infrastructure and cost.
- **Route 53 weighted routing is probabilistic, not deterministic**: DNS-based weighting operates at the DNS resolution level. Clients and intermediate DNS resolvers cache results, meaning a single client could flip between versions across requests. It does NOT guarantee exactly 10% of customers see the new version throughout a session.
- **No session stickiness guarantee**: Route 53 has no concept of session affinity. A customer's DNS record could be cached pointing to ALB-1, but if their resolver re-queries, they could land on ALB-2. This violates the "same version during testing window" requirement.

**C: AutoScalingRollingUpdate with MaxBatchSize 10**
- **Rolling update is a deployment strategy, not a traffic-splitting strategy**: Rolling updates gradually replace instances in the *same* Auto Scaling group. This means existing users get terminated mid-session when their instance is replaced — violating session stickiness.
- **No traffic isolation**: All instances serve traffic during the rollout. Users are not segmented; they could hit either old or new instances on successive requests.
- **MaxBatchSize 10 = 10 instances (or 10%)**: This controls the *batch size* of instance replacement during rollout, not the percentage of *customers* routed to the new version. These are fundamentally different concepts.
- **Launch Configuration is deprecated**: AWS now recommends Launch Templates instead of Launch Configurations.

**D: Second Auto Scaling Group + Least Outstanding Requests (LOR)**
- **LOR is not a traffic-splitting mechanism**: Least Outstanding Requests is a load-balancing algorithm that routes to the instance with the fewest pending requests. It does not provide weighted distribution between old/new versions. New instances might receive all or most traffic, not 10%.
- **Session stickiness alone is insufficient without weighted routing**: While stickiness would keep a user on the same version, without weighted target groups, the initial routing decision is not controlled at a 10% level.
- **No 10% guarantee**: The LOR algorithm combined with stickiness does not mathematically guarantee 10% of customers see the new version.

### Hands-On Enterprise Knowledge

**Implementing Weighted Target Groups (AWS CLI / Console):**

1. Create two target groups: `tg-old` (current version) and `tg-new` (new version).
2. Register EC2 instances running the respective application versions.
3. Edit the ALB listener rule's forward action to include both target groups with weights:
   - `tg-old`: weight 90
   - `tg-new`: weight 10
4. Enable stickiness on both target groups (duration-based cookie, e.g., 1 day):
   ```bash
   aws elbv2 modify-target-group-attributes \
     --target-group-arn arn:aws:elasticloadbalancing:...:targetgroup/tg-new/... \
     --attributes Key=stickiness.enabled,Value=true \
                  Key=stickiness.type,Value=lb_cookie \
                  Key=stickiness.lb_cookie.duration_seconds,Value=86400
   ```

**Production Considerations:**
- **Gradual ramp-up**: Start at 10%, monitor CloudWatch metrics, then use the ALB API to adjust weights incrementally (e.g., 10→25→50→100).
- **Automated rollback**: Configure CloudWatch Alarms on the new target group's 5xx errors or latency. Use AWS Lambda or CodeDeploy to automatically reset weights to 100:0.
- **Health checks**: ALB performs health checks on each target group independently. If the new target group fails health checks, ALB automatically stops routing traffic to it — providing a built-in safety net.
- **Stickiness cookie naming**: Use application-controlled cookies for more predictable behavior across different client types.
