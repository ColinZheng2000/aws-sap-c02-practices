A retail company is mounting IoT sensors in all of its stores worldwide. During the manufacturing of each sensor, the company’s private certificate authority (CA) issues an X.509 certificate that contains a unique serial number. The company then deploys each certificate to its respective sensor.

A solutions architect needs to give the sensors the ability to send data to AWS after they are installed. Sensors must not be able to send data to AWS until they are installed.

Which solution will meet these requirements?

A. Create an AWS Lambda function that can validate the serial number. Create an AWS IoT Core provisioning template. Include the SerialNumber parameter in the Parameters section. Add the Lambda function as a pre-provisioning hook. During manufacturing, call the RegisterThing API operation and specify the template and parameters.
B. Create an AWS Step Functions state machine that can validate the serial number. Create an AWS IoT Core provisioning template. Include the SerialNumber parameter in the Parameters section. Specify the Step Functions state machine to validate parameters. Call the StartThingRegistrationTask API operation during installation.
C. Create an AWS Lambda function that can validate the serial number. Create an AWS IoT Core provisioning template. Include the SerialNumber parameter in the Parameters section. Add the Lambda function as a pre-provisioning hook. Register the CA with AWS IoT Core, specify the provisioning template, and set the allow-auto-registration parameter.
D. Create an AWS IoT Core provisioning template. Include the SerialNumber parameter in the Parameters section. Include parameter validation in the template. Provision a claim certificate and a private key for each device that uses the CA. Grant AWS IoT Core service permissions to update AWS IoT things during provisioning.




# AWS SAP-C02 Question Analysis — Q330

## 📋 Analysis

### 1. Understanding the Problem

This question focuses on secure onboarding of IoT devices using AWS IoT Core.

Key requirements:

- Sensors are manufactured with:
  - X.509 certificates
  - certificates signed by the company’s private CA
  - unique serial numbers
- Devices are deployed globally
- Sensors must NOT be able to send data before installation
- Devices should become operational only after installation

The architecture challenge is:
- how to securely delay AWS activation until deployment time

This is a classic AWS IoT onboarding pattern involving:
✅ CA registration  
✅ automatic certificate registration  
✅ provisioning templates  
✅ pre-provisioning validation

---

## 2. Core AWS Concepts

### X.509 Device Authentication

AWS IoT Core authenticates devices using:
- mutual TLS (mTLS)
- X.509 certificates

Each device certificate uniquely identifies the sensor.

Because the certificates are already issued during manufacturing:
- identity already exists
- authorization must be delayed

This distinction is critical.

---

### Fleet Provisioning and JITP

AWS IoT supports:
- Just-in-Time Provisioning (JITP)
- Fleet Provisioning

These allow:
- automatic onboarding when devices first connect
- large-scale manufacturing workflows
- delayed activation

Very common enterprise IoT architecture.

---

### Pre-Provisioning Hooks

AWS IoT Core supports:
✅ Lambda pre-provisioning hooks

These hooks can:
- validate serial numbers
- check installation status
- confirm store assignment
- reject unauthorized devices

This exactly matches the requirement:
"must not send data until installed"

---

### CA Auto-Registration

AWS IoT can trust a private CA.

When:
- the CA is registered
- auto-registration is enabled

Then:
- any device signed by that CA can begin provisioning dynamically

This enables scalable manufacturing.

---

## 3. Option Analysis

---

## ❌ Option A (INCORRECT)

> During manufacturing, call RegisterThing API.

This fails the timing requirement.

### Why it is wrong

Provisioning occurs:
- during manufacturing

But the requirement states:
- devices must not send data before installation

If devices are registered early:
- they may already authenticate and communicate

This creates premature activation.

The architecture must defer provisioning until deployment time.

---

## ❌ Option B (INCORRECT)

> Use Step Functions to validate serial numbers.

Several problems exist.

### Problem 1 — Unsupported integration

AWS IoT provisioning supports:
✅ Lambda pre-provisioning hooks

It does NOT support:
❌ Step Functions as provisioning validators

---

### Problem 2 — Wrong API

`StartThingRegistrationTask`
is intended for:
- bulk registration workflows

Not dynamic installation-time onboarding.

This is not the correct AWS-native provisioning pattern.

---

## ✅ Option C (CORRECT)

> Use Lambda validation hooks, a provisioning template, register the CA, and allow auto-registration.

This fully satisfies the requirements.

---

### Why this works

#### Manufacturing stage

Each sensor receives:
- a certificate
- signed by the company CA
- serial number embedded

But:
- device is not yet provisioned in AWS IoT

Therefore:
✅ no operational AWS access yet

---

#### AWS IoT setup

The company:
- registers the CA certificate
- enables auto-registration
- configures a provisioning template
- attaches a Lambda pre-provisioning hook

---

#### Installation stage

When device first connects:

1. AWS IoT validates the certificate chain
2. Auto-registration starts
3. Provisioning template executes
4. Lambda hook validates:
   - serial number
   - installation status
   - inventory authorization
5. Device becomes active only if approved

This perfectly satisfies:
✅ delayed activation  
✅ secure onboarding  
✅ scalable provisioning

---

## ❌ Option D (INCORRECT)

This option incorrectly mixes fleet provisioning concepts.

### Problems

#### Claim certificates are unnecessary

The devices already have:
- manufacturer-issued certificates

Claim certificates are typically used when:
- devices lack permanent identity certificates initially

---

#### Template validation is insufficient

Provisioning templates alone cannot perform:
- external authorization checks
- installation validation logic
- dynamic approval workflows

The missing requirement is:
✅ Lambda-based validation

---

## 4. Enterprise IoT Architecture Perspective

This design reflects a common enterprise manufacturing pattern.

### Manufacturing responsibility
- Create device identity
- Install certificates

### Installation responsibility
- Activate operational access
- Validate deployment legitimacy

Benefits:
- Prevents stolen device activation
- Enables zero-touch deployment
- Scales globally
- Supports secure supply chains

Very common in:
- retail IoT
- industrial IoT
- smart buildings
- healthcare devices

## 5. Hands-On Knowledge

### Register CA with auto-registration

```bash
aws iot register-ca-certificate \
  --ca-certificate file://ca.pem \
  --verification-cert file://verification.pem \
  --set-as-active \
  --allow-auto-registration
```

---

  "Parameters": {
    "SerialNumber": {

    }
  }
}

# ✅ Final Answer

# ✅ C. Create a Lambda validation hook, create an IoT provisioning template, register the CA with AWS IoT Core, and enable auto-registration.

Reason:
- Devices receive certificates during manufacturing
- Devices are not activated until installation
- Lambda pre-provisioning hooks validate serial numbers
- CA auto-registration supports scalable onboarding

Acts as:
✅ provisioning-time security gate
