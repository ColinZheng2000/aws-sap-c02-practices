A company has introduced a new policy that allows employees to work remotely from their homes if they connect by using a VPN. The company is hosting internal applications with VPCs in multiple AWS accounts. Currently, the applications are accessible from the company's on-premises office network through an AWS Site- to-Site VPN connection. The VPC in the company's main AWS account has peering connections established with VPCs in other AWS accounts.  
A solutions architect must design a scalable AWS Client VPN solution for employees to use while they work from home.  
What is the MOST cost-effective solution that meets these requirements?  

- A. Create a Client VPN endpoint in each AWS account. Configure required routing that allows access to internal applications.
- B. Create a Client VPN endpoint in the main AWS account. Configure required routing that allows access to internal applications.
- C. Create a Client VPN endpoint in the main AWS account. Provision a transit gateway that is connected to each AWS account. Configure required routing that allows access to internal applications.
- D. Create a Client VPN endpoint in the main AWS account. Establish connectivity between the Client VPN endpoint and the AWS Site-to-Site VPN.

# AWS SAP-C02: Cost-Effective Client VPN Design

**Tags:** #AWS_SAP_C02 #Network_Connectivity #Cost_Optimization #Client_VPN #VPC_Peering #Transit_Gateway

## Analysis

### 1. Understanding the Problem
The scenario presents a "Hub and Spoke" architecture using **VPC Peering**.
* **Current State:** A "Main" VPC acts as the hub. It is connected to on-premises via Site-to-Site VPN. It is connected to other "Spoke" VPCs (in different accounts) via **existing** VPC Peering connections.
* **Requirement:** Enable remote workers (Client VPN) to access applications in *all* VPCs (Main and Spokes).
* **Constraint:** The solution must be the **MOST cost-effective**.



### 2. Services Involved

* **AWS Client VPN:** A managed client-based VPN service that enables you to securely access your AWS resources and on-premises network. It utilizes OpenVPN clients. It places Elastic Network Interfaces (ENIs) into your target VPC subnets.
* **VPC Peering:** A networking connection between two VPCs that enables you to route traffic between them using private IPv4 or IPv6 addresses. Notably, VPC Peering is **non-transitive** (VPC A connected to B, and B to C, does not mean A connects to C). However, resources *inside* VPC B can access both A and C.
* **AWS Transit Gateway (TGW):** A network transit hub that connects VPCs and on-premises networks. While highly scalable, it incurs hourly attachment costs and data processing fees.

### 3. Explanation of Correct Option (B)

**Why B is correct:**
Option B leverages the **existing infrastructure** to minimize cost. The company already has VPC Peering set up between the Main VPC and the Spoke VPCs.

When you deploy a Client VPN Endpoint into the Main VPC, the VPN clients receive an IP address from the Client VPN CIDR. The traffic enters the Main VPC through the Client VPN ENIs. From the perspective of the network, the traffic is originating from inside the Main VPC.

Because the Main VPC is peered with the Spoke VPCs, you can simply update the route tables to allow the Client VPN traffic to traverse the peering connection to reach the Spoke VPCs.

**Cost Analysis:**
* **Infrastructure Cost:** You only pay for **one** Client VPN Endpoint association.
* **Data Cost:** You pay for data transfer across the VPC Peering (which is generally lower than TGW processing fees).
* **Savings:** You avoid the hourly attachment fees and data processing fees associated with setting up a new Transit Gateway (Option C), and you avoid the multiple hourly association fees of setting up VPNs in every account (Option A).

### 4. Explanation of Incorrect Options

* **Option A (Create Client VPN in each account):**
    * **Why it fails:** This is the *least* cost-effective and least manageable solution. You would have to pay the hourly endpoint association fee for *every single AWS account*. It also increases administrative overhead (managing certificates and configs for multiple endpoints).
* **Option C (Provision a Transit Gateway):**
    * **Why it fails:** While Transit Gateway is a valid solution for connecting multiple VPCs, the question asks for the *most cost-effective* solution. The scenario explicitly states that VPC Peering is **already established**. Replacing free/cheap peering connections with a Transit Gateway would incur:
        1.  TGW hourly attachment fees for every VPC.
        2.  TGW Data processing fees ($ per GB).
    * Since peering exists and supports the requirement, moving to TGW adds unnecessary cost.
* **Option D (Connect Client VPN to Site-to-Site VPN):**
    * **Why it fails:** This describes a "hairpin" scenario or connecting back to on-premises. The Site-to-Site VPN connects the Main VPC to the *On-Premises office*. It does not help the Client VPN users access the *other AWS accounts* (Spoke VPCs). This option does not address the core requirement of accessing internal applications across multiple AWS accounts.

To do this, you must add a route to the Client VPN endpoint's route table that targets the peered VPC."* — [AWS Client VPN Administrator Guide](https://docs.aws.amazon.com/vpn/latest/clientvpn-admin/scenario-peered.html)
