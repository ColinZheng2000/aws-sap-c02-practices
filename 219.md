A company has a Windows-based desktop application that is packaged and deployed to the users' Windows machines. The company recently acquired another company that has employees who primarily use machines with a Linux operating system. The acquiring company has decided to migrate and rehost the Windows-based desktop application to AWS.

All employees must be authenticated before they use the application. The acquiring company uses Active Directory on premises but wants a simplified way to manage access to the application on AWS for all the employees.

Which solution will rehost the application on AWS with the LEAST development effort?

A. Set up and provision an Amazon Workspaces virtual desktop for every employee. Implement authentication by using Amazon Cognito identity pools. Instruct employees to run the application from their provisioned Workspaces virtual desktops.
B. Create an Auto Scaling group of Windows-based Amazon EC2 instances. Join each EC2 instance to the company’s Active Directory domain. Implement authentication by using the Active Directory that is running on premises. Instruct employees to run the application by using a Windows remote desktop.
C. Use an Amazon AppStream 2.0 image builder to create an image that includes the application and the required configurations. Provision an AppStream 2.0 On-Demand fleet with dynamic Fleet Auto Scaling policies for running the image. Implement authentication by using AppStream 2.0 user pools. Instruct the employees to access the application by starting browser-based AppStream 2.0 streaming sessions.
D. Refactor and containerize the application to run as a web-based application. Run the application in Amazon Elastic Container Service (Amazon ECS) on AWS Fargate with step scaling policies. Implement authentication by using Amazon Cognito user pools. Instruct the employees to run the application from their browsers.


Suggest Answer: C

---

## 📊 Analysis

### Understanding the Problem

A company has a **Windows-based desktop application** deployed to users' Windows machines. After acquiring a company whose employees use **Linux**, the acquiring company needs to:

- **Rehost** (not refactor) the Windows desktop application to AWS
- Authenticate **all employees** (both Windows and Linux users)
- Use the existing **on-premises Active Directory** for authentication
- Minimize **development effort** (rehost, don't rebuild)

This is a classic **application streaming / virtual desktop** scenario. The keyword "rehost" means "lift and shift" without rewriting the application code. The Linux users can't natively run a Windows desktop app, so the application must be delivered to them via streaming or virtual desktop.

### ✅ Correct Option: C — Amazon AppStream 2.0

| Component | Purpose |
|---|---|
| **AppStream 2.0 Image Builder** | Creates a custom image containing the Windows application and all required configurations (runtime, dependencies, settings) |
| **AppStream 2.0 On-Demand Fleet** | Runs the application on Windows-based streaming instances — users connect via browser from ANY OS (Windows, Linux, macOS) |
| **Dynamic Fleet Auto Scaling** | Scales the fleet based on demand — adds/removes instances as users connect/disconnect; minimizes cost |
| **AppStream 2.0 User Pools** | Built-in user directory for AppStream — manages user authentication without needing external identity provider for basic scenarios |

**Why this is the LEAST development effort (SAP-C02 lens):**

- **No code changes required** — the Windows application runs as-is on AppStream 2.0 Windows instances
- Linux users access a **fully functional Windows desktop app** through their browser — no OS compatibility issues
- AppStream 2.0 provides **native user authentication** via user pools (or can integrate with SAML 2.0, Active Directory via domain join)
- The application is streamed as **pixels** to the browser — the client machine only needs a modern browser
- Fleet Auto Scaling ensures you pay only for active streaming sessions
- **"Rehost" is explicitly mentioned** — AppStream 2.0 is the rehost solution for Windows desktop applications

**Key Integration with Active Directory:** AppStream 2.0 fleets can be **joined to an Active Directory domain** (including AWS Managed Microsoft AD or on-premises AD via AD Connector/trust relationship). While the option mentions "user pools," in production you would integrate with the existing AD. The option is still correct because user pools provide "a simplified way to manage access" as the question states.

### ❌ Incorrect Options

**A. Amazon WorkSpaces + Cognito Identity Pools**

- **WorkSpaces** provisions a **full Windows desktop** for every employee — overkill when you only need to deliver one application
- WorkSpaces are billed **monthly** (or hourly with minimum commitment) — significantly more expensive than AppStream 2.0's per-second streaming pricing
- Every employee needs a dedicated Workspace → high cost, resource waste for occasional app users
- **Cognito Identity Pools** provides **AWS credentials** for accessing AWS services — it does NOT authenticate users into a Windows desktop or Active Directory. This is the wrong authentication mechanism
- ❌ WorkSpaces = full desktop (overkill), Cognito identity pools = AWS credential broker (wrong auth mechanism for AD)

**B. Auto Scaling group of Windows EC2 + on-premises AD + Windows Remote Desktop**

- Requires every employee to use **Windows Remote Desktop (RDP)** — Linux users need an RDP client, creating friction
- Managing an Auto Scaling group of EC2 instances with AD domain join requires significant operational effort (domain join automation, user profile management, patching, etc.)
- **Remote Desktop** is NOT designed for multi-user application delivery — it either gives a full desktop (like Workspaces) or requires complex RemoteApp configuration
- Significant **development/operational effort** to build the automation for domain join, user session management, and scaling
- ❌ More operational overhead, less user-friendly for Linux users

**D. Refactor + Containerize + ECS Fargate + Cognito User Pools**

- **"Refactor and containerize"** is the opposite of "rehost with LEAST development effort"
- Converting a Windows desktop application to a web-based containerized app requires **complete code rewrite** — months of development effort
- Windows containers on ECS Fargate have limited support and specific requirements
- **Cognito User Pools** manages user registration and authentication for web/mobile apps — it does NOT integrate with on-premises Active Directory natively (requires federation setup)
- ❌ Maximum development effort — contradicts the "least development effort" and "rehost" requirements entirely

### 🏭 Enterprise Hands-On Knowledge

1. **AppStream 2.0 vs. WorkSpaces — Decision Framework:**

| Criterion | AppStream 2.0 | WorkSpaces |
|---|---|---|
| Use Case | Deliver 1–5 applications | Full desktop experience |
| Billing | Per-second (streaming) | Monthly or hourly |
| User Data Persistence | S3-backed Home Folders | Persistent D: drive |
| Best For | Occasional app access, training, demos | Full-time remote workers |
| Multi-OS Client | Browser (any OS) | Native client (Win/Mac/Linux) |

2. **AppStream 2.0 + Active Directory Integration in Practice:**

- Create an AD Connector or use AWS Managed Microsoft AD with a trust to on-premises AD
- Join the AppStream 2.0 fleet to the AD domain
- Users authenticate with their existing AD credentials via the AppStream 2.0 SAML 2.0 integration
- Group Policy Objects (GPOs) apply to streaming instances (with limitations)

3. **Cost Optimization with AppStream 2.0:**

- **On-Demand**: Pay per second with no upfront commitment
- **Always-On**: Provisioned immediately, billed at a lower hourly rate — for frequently used applications
- Use **Dynamic Auto Scaling** with schedule-based policies (scale up before business hours, scale down after) + demand-based policies (buffer of available instances)

4. **M&A Integration Pattern (This Scenario):** When acquiring a company, you often have a mixed OS environment temporarily. AppStream 2.0 bridges the gap without forcing OS migration. Over time, as the application is modernized (refactored to web-based), you can migrate users off AppStream 2.0. This is a proven **incremental modernization** strategy.
