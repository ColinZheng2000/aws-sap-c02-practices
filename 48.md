A company built an application based on AWS Lambda deployed in an AWS CloudFormation stack. The last production release of the web application introduced an issue that resulted in an outage lasting several minutes. A solutions architect must adjust the deployment process to support a canary release.  
Which solution will meet these requirements?  

- A. Create an alias for every new deployed version of the Lambda function. Use the AWS CLI update-alias command with the routing-config parameter to distribute the load.
- B. Deploy the application into a new CloudFormation stack. Use an Amazon Route 53 weighted routing policy to distribute the load.
- C. Create a version for every new deployed Lambda function. Use the AWS CLI update-function-configuration command with the routing-config parameter to distribute the load.
- D. Configure AWS CodeDeploy and use CodeDeployDefault.OneAtATime in the Deployment configuration to distribute the load.

## Understanding the Problem

The core requirement is to implement a **canary release** deployment strategy for an existing **AWS Lambda** function, which is currently deployed via **AWS CloudFormation**. The goal is to avoid major outages by gradually shifting traffic to the new version, allowing for testing and quick rollback if issues arise. A canary release is a deployment pattern where a small subset of users receives the new version (the "canary") before a full rollout.

## Correct Option(s) Explanation

**A. Create an alias for every new deployed version of the Lambda function. Use the AWS CLI `update-alias` command with the `routing-config` parameter to distribute the load.**

* **Explanation of A:** This is the *precise mechanism* for a canary release on Lambda.
    1.  A new Lambda **Version** is created for the new code.
    2.  The existing **Alias** is updated to point to *both* the old version (e.g., V1) and the new version (e.g., V2) using the `routing-config` parameter.
    3.  The `routing-config` is used to specify a weighted distribution (e.g., 90% to V1, 10% to V2 - the "canary").
    4.  Over time, the weights are shifted (e.g., 50/50, then 0/100) to complete the rollout.
    5.  This can be done via the AWS CLI, as stated, or automated via **CodeDeploy**, which performs these steps. Since A explicitly calls out the correct Lambda feature (`alias` and `routing-config`), it is technically correct.

## Incorrect Options Explanation

| Option | Reason for Incorrectness |
| :--- | :--- |
| **B. Deploy the application into a new CloudFormation stack. Use an Amazon Route 53 weighted routing policy to distribute the load.** | This is an **overkill and incorrect service** for version control on a single Lambda function. Route 53 operates at the DNS level, routing external traffic to different *endpoints* (e.g., different API Gateways or load balancers). It cannot manage traffic between two *versions* of the same Lambda function accessed via a single endpoint (like API Gateway or a direct invocation). The native Lambda Alias feature is the correct, simpler, and intended solution. |
| **C. Create a version for every new deployed Lambda function. Use the AWS CLI `update-function-configuration` command with the `routing-config` parameter to distribute the load.** | **Incorrect command and resource.** Lambda **Versions** are immutable and *cannot* be updated to configure routing. **Aliases** are the mutable resource that uses the `routing-config` parameter. The command to configure traffic splitting is `update-alias`, not `update-function-configuration`. |
| **D. Configure AWS CodeDeploy and use `CodeDeployDefault.OneAtATime` in the Deployment configuration to distribute the load.** | **Incorrect specific deployment configuration.** While **AWS CodeDeploy** is the *recommended service* for automated safe deployments (which manage the mechanism in A), the configuration `CodeDeployDefault.OneAtATime` is a deployment type for **EC2/On-Premises** deployments, not for a Canary/Linear traffic shift on Lambda. The correct Lambda safe deployment strategies are `LambdaCanary...` or `LambdaLinear...`. |
