A company has deployed an Amazon Connect contact center. Contact center agents are reporting large numbers of computer-generated calls. The company is concerned about the cost and productivity effects of these calls. The company wants a solution that will allow agents to flag the call as spam and automatically block the numbers from going to an agent in the future.

What is the MOST operationally efficient solution to meet these requirements?

A. Customize the Contact Control Panel (CCP) by adding a flag call button that will invoke an AWS Lambda function that calls the UpdateContactAttributes API. Use an Amazon DynamoDB table to store the spam numbers. Modify the contact flows to look for the updated attribute and to use a Lambda function to read and write to the DynamoDB table.
B. Use a Contact Lens for Amazon Connect rule that will look for spam calls. Use an Amazon DynamoDB table to store the spam numbers. Modify the contact flows to look for the rule and to invoke an AWS Lambda function to read and write to the DynamoDB table.
C. Use an Amazon DynamoDB table to store the spam numbers. Create a quick connect that the agents can transfer the spam call to from the Contact Control Panel (CCP). Modify the quick connect contact flow to invoke an AWS Lambda function to write to the DynamoDB table.
D. Modify the initial contact flow to ask for caller input. If the agent does not receive input, the agent should mark the caller as spam. Use an Amazon DynamoDB table to store the spam numbers. Use an AWS Lambda function to read and write to the DynamoDB table.


# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

The company has:

- Multiple branch offices
- Multiple AWS VPCs
- Need for:
  - Secure communication
  - Simplified management
  - Scalable architecture

This is a classic:
> Hybrid multi-network connectivity problem.

The key design goal is:
- Centralized networking
- Reduced operational complexity
- Scalable interconnection model

SAP-C02 heavily tests:
- Transit Gateway vs VPN mesh architectures

---

# 2. Core AWS Networking Concepts

## AWS Transit Gateway (TGW)

Transit Gateway acts as:
- A central cloud router

It simplifies connectivity between:
- VPCs
- VPNs
- Direct Connect
- On-premises networks

Instead of:
```text
Many point-to-point connections
```

You get:
```text
Hub-and-spoke architecture
```

This is one of the most important SAP-C02 networking services.

---

## Site-to-Site VPN

Used for:
- Secure encrypted tunnels
- Between branch office routers and AWS

Ideal for:
- Hybrid connectivity

---

# 3. Typical Exam Pattern

When you see:

- Many VPCs
- Many branch offices
- Scalability concerns
- Simplified management

The answer is very often:
✅ Transit Gateway

NOT:
- Full mesh VPNs
- VPC peering explosion
- Manual routing complexity

---

# 4. Best Architecture

Recommended design:

```text
Branch Offices
      ↓
Site-to-Site VPN
      ↓
Transit Gateway
      ↓
Multiple VPCs
```

Benefits:
- Centralized routing
- Simplified management
- Scalable growth
- Reduced connection count

---

# 5. Why Transit Gateway Is Superior

Without TGW:

For:
- N branch offices
- M VPCs

You may need:
```text
N × M VPN connections
```

This becomes operationally painful.

With TGW:

```text
Each network connects once to TGW
```

This dramatically simplifies:
- Route management
- Security controls
- Network expansion

---

# 6. Common Incorrect Alternatives

---

## ❌ VPC Peering

Problems:

- Does not scale well
- Full mesh complexity
- No transitive routing
- Route table explosion

SAP-C02 often tests:
> TGW replaces large-scale peering meshes.

---

## ❌ Direct Connect Only

Direct Connect:
- Provides dedicated connectivity
- But not centralized routing architecture

Still commonly paired WITH TGW.

Also:
- Expensive
- Slower to deploy
- Overkill unless high bandwidth/private circuits required

---

## ❌ Client VPN

Client VPN:
- Remote user connectivity
- Not branch office networking

Remember:

| Requirement | Service |
|---|---|
| Remote users | Client VPN |
| Branch networks | Site-to-Site VPN |
| Central routing | Transit Gateway |

---

# 7. Hands-On AWS Knowledge

## Transit Gateway Attachments

TGW supports:

- VPC attachments
- VPN attachments
- Direct Connect gateway attachments

This enables:
- Centralized hybrid networking

---

## Routing Example

```text
Branch A
   ↓
VPN
   ↓
TGW
   ↓
VPC A / VPC B / VPC C
```

All traffic routes through:
- One centralized transit layer

## High Availability

Site-to-Site VPN:
- Uses two tunnels by default
- Supports resilient hybrid connectivity

Transit Gateway:
- Regional managed service
- Highly scalable

# 8. Cost and Operational Efficiency

Transit Gateway improves:

- Network scalability
- Route management
- Security consistency
- Change management

Compared to:
- Hundreds of VPN tunnels
- Manual peering maintenance

This directly aligns with:
- AWS operational excellence
- Enterprise network architecture best practices

---

# 9. Exam Trap Recognition

## Trap 1:
Using VPC peering for large environments

Rule of thumb:

- Small/simple → Peering
- Large enterprise → TGW

---

## Trap 2:
Confusing Client VPN with Site-to-Site VPN

Remember:

- Client VPN → users
- Site-to-Site VPN → networks

---

## Trap 3:
Thinking Direct Connect replaces TGW

Reality:

- Direct Connect = transport
- Transit Gateway = routing hub

They solve different problems.

---

# 10. Real Enterprise Architecture

Large enterprises commonly deploy:

```text
Branches
    ↓
Site-to-Site VPN / Direct Connect
    ↓
Transit Gateway
    ↓
Shared Services VPC
Application VPCs
Security VPCs
Inspection VPCs
```

Advanced integrations:
- AWS Network Firewall
- Centralized egress
- Security inspection VPCs
- SD-WAN integration

TGW is foundational in modern AWS enterprise networking.

## 面试官可能追问

### 1. “Transit Gateway 和 VPC Peering 的区别是什么？”

回答要点：

VPC Peering：
- 点对点
- 不支持 transitive routing
- 大规模难维护

Transit Gateway：
- Hub-and-spoke
- 支持集中路由
- 更适合企业级网络

---

### 2. “什么时候使用 Direct Connect？”

回答要点：

适用于：
- 稳定高带宽需求
- 低延迟
- 合规要求
- 长期混合云架构

通常与：
- Transit Gateway
- DX Gateway
一起使用。

---

### 3. “为什么 Site-to-Site VPN 更适合 branch office？”

回答要点：

- 基于企业路由器
- 网络级连接
- 自动加密
- AWS 原生支持
- 比 Client VPN 更适合办公室互联
