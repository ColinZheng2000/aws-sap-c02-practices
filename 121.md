A global media company is planning a multi-Region deployment of an application. Amazon DynamoDB global tables will back the deployment to keep the user experience consistent across the two continents where users are concentrated. Each deployment will have a public Application Load Balancer (ALB). The company manages public DNS internally. The company wants to make the application available through an apex domain.

Which solution will meet these requirements with the LEAST effort?

A. Migrate public DNS to Amazon Route 53. Create CNAME records for the apex domain to point to the ALB. Use a geolocation routing policy to route traffic based on user location.
B. Place a Network Load Balancer (NLB) in front of the ALMigrate public DNS to Amazon Route 53. Create a CNAME record for the apex domain to point to the NLB’s static IP address. Use a geolocation routing policy to route traffic based on user location.
C. Create an AWS Global Accelerator accelerator with multiple endpoint groups that target endpoints in appropriate AWS Regions. Use the accelerator’s static IP address to create a record in public DNS for the apex domain.
D. Create an Amazon API Gateway API that is backed by AWS Lambda in one of the AWS Regions. Configure a Lambda function to route traffic to application deployments by using the round robin method. Create CNAME records for the apex domain to point to the API's URL.

# Analysis of Multi-Region Application Deployment with Apex Domain Requirement
Tags: #AWS-SAP-C02 #AWS-DynamoDB-Global-Tables #AWS-ALB #AWS-Route53 #AWS-Global-Accelerator #AWS-NLB #AWS-API-Gateway

## 1. Understanding the Problem
The core requirements are:
- Deploy an application across two AWS Regions, backed by **DynamoDB Global Tables** for consistent user experience.
- Each Region has a public **Application Load Balancer (ALB)**.
- The company uses an **internal public DNS** (no need to migrate it).
- Make the application accessible via an **apex domain** (e.g., `example.com`, not a subdomain like `app.example.com`).
- The solution must require the **least effort**.

Key constraints to note:
- Apex domains (also called "naked domains") **cannot use CNAME records** according to DNS standards. A CNAME maps a domain name to another domain name, and apex domains are reserved for SOA/NS/A/AAAA records (which map to IP addresses directly). This is a critical DNS rule that eliminates options relying on CNAME for apex domains.
- No requirement to migrate the existing internal DNS, so solutions that work with the existing DNS are preferred.

### Critical DNS Concept: Apex Domain & CNAME Limitation
Per [IETF DNS standards](https://www.rfc-editor.org/rfc/rfc1035), a CNAME record cannot coexist with other record types (e.g., SOA, NS) at the apex domain. This means you **cannot create a CNAME for `example.com`**—only A/AAAA records (which map to IP addresses) are allowed for apex domains.

## 3. Correct Option Explanation (Option C)
### Why Option C is Correct
1. **Apex Domain Compatibility**: AWS Global Accelerator provides **static anycast IP addresses** that never change. The company can create an **A record** (not CNAME) in its existing internal public DNS for the apex domain, pointing to these static IPs. This complies with DNS standards for apex domains and requires no migration of the existing DNS.
2. **Multi-Region Routing with Least Effort**: Global Accelerator automatically routes traffic to the nearest healthy endpoint group (targeting the ALBs in each Region) using the AWS global network. There is no need to configure complex routing policies (e.g., geolocation in Route 53) or manage additional load balancers (e.g., NLB in front of ALB).
3. **No Dependency on Route 53 Migration**: The company uses internal DNS, so migrating to Route 53 (as required in Options A and B) is avoided—this reduces effort significantly.
4. **Integration with ALB**: Global Accelerator natively supports ALBs as endpoints. It health-checks the ALBs and routes traffic only to healthy Regions, aligning with the high-availability requirement of the media application.

### Official AWS Guidance Alignment
According to the [AWS Global Accelerator Use Cases](https://docs.aws.amazon.com/global-accelerator/latest/dg/use-cases.html), one primary use case is **simplifying apex domain management** for multi-Region applications by using static IPs, avoiding the CNAME limitation and Route 53 migration.

## 4. Incorrect Options Explanation
### Option A
- **Critical Flaw**: It proposes creating a **CNAME record for the apex domain**, which violates DNS standards. Apex domains cannot use CNAME records.
- **Additional Flaw**: Requires migrating public DNS to Route 53, which adds effort (the company wants to use its existing internal DNS).
- **Irrelevant Feature**: Geolocation routing in Route 53 is useful, but the CNAME and DNS migration issues make this option invalid.

### Option B
- **Critical Flaw 1**: Still uses a **CNAME record for the apex domain**, which is not allowed by DNS standards. Even if NLB has static IPs, a CNAME maps to a domain name (not an IP), so this is a DNS violation.
- **Critical Flaw 2**: NLBs are Layer 4 load balancers, and placing an NLB in front of an ALB adds unnecessary complexity (extra load balancer to manage, increased cost, more configuration steps). This contradicts the "least effort" requirement.
- **Additional Flaw**: Requires migrating DNS to Route 53, which is not needed.

### Option D
- **Critical Flaw 1**: Uses a **CNAME record for the apex domain**, which is invalid per DNS rules.
- **Critical Flaw 2**: Overly complex architecture. Using API Gateway + Lambda for traffic routing adds unnecessary components (Lambda function to manage round-robin routing, API Gateway configuration) when a purpose-built service like Global Accelerator exists.
- **Additional Flaw**: Round-robin routing does not optimize for user location (unlike Global Accelerator’s latency-based routing), which harms the consistent user experience requirement across continents.
