A company recently deployed an application on AWS. The application uses Amazon DynamoDB. The company measured the application load and configured the RCUs and WCUs on the DynamoDB table to match the expected peak load. The peak load occurs once a week for a 4-hour period and is double the average load. The application load is close to the average load for the rest of the week. The access pattern includes many more writes to the table than reads of the table.  
  
A solutions architect needs to implement a solution to minimize the cost of the table.  
  
Which solution will meet these requirements?

- A. Use AWS Application Auto Scaling to increase capacity during the peak period. Purchase reserved RCUs and WCUs to match the average load.
- B. Configure on-demand capacity mode for the table.
- C. Configure DynamoDB Accelerator (DAX) in front of the table. Reduce the provisioned read capacity to match the new peak load on the table.
- D. Configure DynamoDB Accelerator (DAX) in front of the table. Configure on-demand capacity mode for the table.

## Analysis

### Understanding the Problem

The company is running an application on AWS using **Amazon DynamoDB**. The current configuration uses **provisioned capacity** (RCUs and WCUs) set to match the **peak load**, which is **double the average load** and occurs for only **4 hours once a week**. The access pattern has **many more writes (WCUs) than reads (RCUs)**.

The objective is to **minimize the cost** of the DynamoDB table, given the highly variable and predictable weekly load pattern.

* **Load Profile:** Highly variable and predictable (weekly peak).
* **Goal:** Minimize cost.
* **Constraint:** Must handle the peak load, which is double the average.

### Services Explanation

| AWS Service | Description | Relevance to the Problem |
| :--- | :--- | :--- |
| **Amazon DynamoDB** | A fast, flexible NoSQL database service for single-digit millisecond performance at any scale. Offers two capacity modes: **Provisioned** and **On-Demand**. | The core service being used. The solution must optimize its capacity mode and pricing. |
| **Provisioned Capacity Mode** | Requires specifying the number of **Read Capacity Units (RCUs)** and **Write Capacity Units (WCUs)** needed. You're charged for the capacity you provision, regardless of usage. Provides cost predictability and lower cost for predictable, sustained workloads. | Relevant because the current setup uses it and the load is mostly predictable. Reserved capacity is an option for cost saving. |
| **On-Demand Capacity Mode** | Charges for the data reads and writes the application performs. Ideal for unpredictable workloads. Capacity scales instantly. Higher cost per operation than provisioned capacity. | Relevant for highly variable loads, but might be more expensive overall than optimized provisioned capacity. |
| **Reserved Capacity** | A pricing option for **Provisioned Capacity Mode** where you pay a one-time upfront fee and commit to a minimum capacity level (RCUs/WCUs) for a 1-year or 3-year term, resulting in significant savings (up to 77%) compared to standard provisioned capacity pricing. | Directly addresses cost minimization for a predictable minimum load. |
| **AWS Application Auto Scaling** | A service that allows you to automatically scale provisioned resources for various AWS services, including DynamoDB tables. It can scale capacity up or down in response to actual usage or based on scheduled times. | Directly addresses the need to handle a predictable peak load without over-provisioning for the entire week. |
| **DynamoDB Accelerator (DAX)** | An in-memory cache for DynamoDB that delivers microsecond latency for eventually consistent reads. It only helps with **read (RCU)** scaling and performance. | Only relevant for the read pattern. The problem states writes are much greater than reads, and the core issue is minimizing **total** cost (both RCU and WCU). |

### Correct Option(s) Explanation

**A. Use AWS Application Auto Scaling to increase capacity during the peak period. Purchase reserved RCUs and WCUs to match the average load.**

* **Cost Minimization:** Purchasing **Reserved Capacity** (RCUs and WCUs) to match the **average load** provides the maximum discount on the vast majority of the weekly capacity (the 164 hours a week when the load is average). The average load is the predictable, sustained baseline.
* **Handling Peak Load:** **AWS Application Auto Scaling** can be configured with a **scheduled scaling action** to increase the provisioned capacity (RCUs and WCUs) to the peak level (double the average) just before the 4-hour peak period starts and scale it back down afterward. This ensures the application can handle the peak without paying the high price of running peak capacity 24/7.
* **AWS Official Documentation Reference:** The AWS DynamoDB documentation explicitly recommends using **Reserved Capacity** for workloads with "predictable, sustained capacity requirements" and **Auto Scaling** or **scheduled scaling** for "predictable traffic changes." This combination is the most cost-effective solution for a predictable, base-load plus peak-load profile.

### Incorrect Options Explanation

**B. Configure on-demand capacity mode for the table.**

* **Reasoning:** While On-Demand mode is excellent for *unpredictable* spikes and avoids capacity planning, it is generally **more expensive** than optimized Provisioned Capacity, especially when the load profile is highly **predictable** (like the weekly peak described). By using Reserved Capacity for the average load and Auto Scaling for the scheduled peak, the company achieves a lower overall cost than simply using On-Demand for the entire week.

**C. Configure DynamoDB Accelerator (DAX) in front of the table. Reduce the provisioned read capacity to match the new peak load on the table.**

* **Reasoning:**
    * **DAX Impact:** DAX only benefits **reads (RCUs)** by caching them, thereby reducing the *required* RCUs. It does **nothing** for **writes (WCUs)**.
    * **Problem Context:** The problem explicitly states that the access pattern includes **many more writes** than reads. The bulk of the cost/capacity issue lies with the WCUs, which DAX doesn't address.
    * **Cost:** This option doesn't leverage the significant cost savings of **Reserved Capacity** or the precise timing of **Auto Scaling** for the WCUs, making it more expensive than option A.

**D. Configure DynamoDB Accelerator (DAX) in front of the table. Configure on-demand capacity mode for the table.**

* **Reasoning:** This is a combination of two incorrect cost-optimization strategies for this scenario:
    * **On-Demand:** As explained in option B, On-Demand is generally more expensive for this predictable, base-plus-peak workload.
    * **DAX:** As explained in option C, DAX only addresses reads, and the main concern is writes, making it an ineffective primary cost optimization tool here. This combination is likely the most expensive of the four options for this specific workload profile.
