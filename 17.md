DynamoDB table. The application is showing an increase in the number of errors during PUT requests. Most of the PUT calls come from a small number of clients that are authenticated with specific API keys.  
A solutions architect has identified that a large number of the PUT requests originate from one client. The API is noncritical, and clients can tolerate retries of unsuccessful calls. However, the errors are displayed to customers and are causing damage to the API's reputation.  
What should the solutions architect recommend to improve the customer experience?  

- A. Implement retry logic with exponential backoff and irregular variation in the client application. Ensure that the errors are caught and handled with descriptive error messages.
- B. Implement API throttling through a usage plan at the API Gateway level. Ensure that the client application handles code 429 replies without error.
- C. Turn on API caching to enhance responsiveness for the production stage. Run 10-minute load tests. Verify that the cache capacity is appropriate for the workload.
- D. Implement reserved concurrency at the Lambda function level to provide the resources that are needed during sudden increases in traffic.

# AWS SAP Exam Question Analysis: API Gateway Throttling

## Understanding the Problem

The software company's application, built with **Amazon API Gateway**, **AWS Lambda**, and **Amazon DynamoDB**, is experiencing an increase in errors, specifically during **PUT requests**. A **small number of clients** authenticated with **specific API keys** are generating most of these PUT calls, with one client being the primary source of the high request volume. The API is **noncritical** and clients can **tolerate retries** of unsuccessful calls. However, the errors are visible to customers, which is **damaging the API's reputation**. The goal is to recommend a solution that will **improve the customer experience** by reducing the visible errors, even if it means unsuccessful calls require retries.

Key constraints and observations:
* High error rate during **PUT requests** from a **few authenticated clients**.
* Single client is the main source of the load.
* Client can tolerate retries.
* Errors are visible and damaging reputation.
* Goal: Improve customer experience (reduce visible errors).

This scenario points towards a need to **manage and control the traffic volume** from specific high-volume clients to prevent them from overwhelming the API and causing errors for themselves and potentially others, while ensuring a graceful failure/retry mechanism.

## Services Explanation

| Service | Description | Role in the Solution |
| :--- | :--- | :--- |
| **Amazon API Gateway** | A fully managed service that makes it easy for developers to create, publish, maintain, monitor, and secure APIs at any scale. | Provides the mechanism for implementing API **throttling** and **usage plans** to manage request traffic from clients, especially those authenticated with API keys. |
| **AWS Lambda** | A serverless compute service that lets you run code without provisioning or managing servers. | Executes the backend logic for the API calls. Throttling applied here manages *compute* capacity, not API *request* volume by client key. |
| **Usage Plans (API Gateway)** | API Gateway feature that defines who can access your deployed stages and methods and optionally how much they can access. Usage plans specify **throttling** and **quota limits** that API Gateway enforces on individual client API keys. | The tool to enforce limits on specific clients (identified by API key) causing the high load. This directly addresses the problem's source. |
| **Throttling (API Gateway)** | A mechanism within API Gateway (often part of a usage plan) to limit the rate at which requests are handled to prevent API backend services from being overwhelmed. When limits are exceeded, API Gateway returns a **HTTP 429 Too Many Requests** error. | The action taken by API Gateway to cap the request rate for a client, protecting the backend and providing a standard, retryable error code. |
| **API Caching (API Gateway)** | Caches responses from the API's endpoint to reduce the number of calls made to the backend and improve latency. | Primarily for improving read performance (GET requests) and reducing backend load. Not directly relevant to mitigating excessive **PUT requests** from a single source. |
| **Reserved Concurrency (AWS Lambda)** | Guarantees that the specified maximum number of concurrent instances of a function will always be available. Prevents other functions from using that capacity. | Manages capacity at the compute layer. While it prevents Lambda-side throttling, it doesn't limit the *request rate* from a *specific API key client* and won't improve the customer experience if the issue is *upstream* (API Gateway, DynamoDB, or simply too many requests overall). |

## Correct Option(s) Explanation

The correct option is **B**.

**B. Implement API throttling through a usage plan at the API Gateway level. Ensure that the client application handles code 429 replies without error.**

* **Implement API throttling through a usage plan at the API Gateway level:** This is the most effective solution. The problem states that the high volume and errors originate from a **small number of clients authenticated with specific API keys**. **API Gateway Usage Plans** are explicitly designed to associate specific **API keys** with **throttling limits** (request rate and burst) and **quotas**. Throttling prevents the high-volume client from overwhelming the API and backend services (Lambda/DynamoDB), which is the likely cause of the errors that are damaging the API's reputation. This protects the service and the experience of other clients.

* **Ensure that the client application handles code 429 replies without error:** When a client exceeds the limits defined in the usage plan, API Gateway returns an **HTTP 429 Too Many Requests** error. The problem states that clients **can tolerate retries** and the goal is to **improve the customer experience** (i.e., prevent the *visible errors* that are currently happening). By throttling the traffic and returning a 429, the client application receives a standard, clear signal that it needs to back off and retry, rather than receiving a more generic or internal error (which might be the current issue). A well-designed client application will handle the 429 gracefully, potentially with a message like "Please try again in a moment," or, ideally, handle the retry logic internally without displaying an error to the user, thus **improving the customer experience**.

* *Reference:* Official AWS documentation on [API Gateway Usage Plans and Throttling](https://docs.aws.amazon.com/apigateway/latest/developerguide/api-gateway-control-access-with-usage-plans.html) confirms that this feature is used to limit requests from clients identified by API keys.

## Incorrect Options Explanation

**A. Implement retry logic with exponential backoff and irregular variation in the client application. Ensure that the errors are caught and handled with descriptive error messages.**

* This is **client-side mitigation** and a **best practice**, but it doesn't solve the core problem of the server being overwhelmed (or nearing it) by one client's aggressive behavior, which is causing *reputational damage* due to visible errors. The issue is described as errors *during* PUT requests, which implies the requests are failing on the server side (Lambda or DynamoDB). Asking the client to implement a solution is difficult to enforce and relies on client compliance, which may not be immediate or possible for all existing clients. The most direct and immediate fix is to implement **server-side throttling** (Option B). Furthermore, if the client is already receiving an error code that signifies a server-side failure other than 429, implementing retries without server-side throttling could just perpetuate the problem by causing more load.

**C. Turn on API caching to enhance responsiveness for the production stage. Run 10-minute load tests. Verify that the cache capacity is appropriate for the workload.**

* API Caching is designed for **GET requests** (data retrieval) to reduce latency and backend load. The errors are occurring during **PUT requests** (data modification). Caching is not applicable or effective for requests that modify data, making this recommendation irrelevant to solving the problem of excessive modification requests.

**D. Implement reserved concurrency at the Lambda function level to provide the resources that are needed during sudden increases in traffic.**

* **Reserved Concurrency** ensures that the Lambda function has enough *compute capacity* to handle spikes, preventing the *Lambda service* from throttling the function (a 429 from Lambda). However, this only shifts the bottleneck. The root cause is an **overload from a specific client identified by an API key**. Implementing Reserved Concurrency does not limit the request rate from that specific client. The load could simply shift the error to another service (e.g., overwhelming the **DynamoDB table's Write Capacity Units**, which causes errors that still damage the API's reputation). The correct approach is to directly control the source of the traffic using **API Gateway Usage Plans and Throttling**, as described in Option B.
