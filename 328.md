A solutions architect is preparing to deploy a new security tool into several previously unused AWS Regions. The solutions architect will deploy the tool by using an AWS CloudFormation stack set. The stack set's template contains an IAM role that has a custom name. Upon creation of the stack set, no stack instances are created successfully.

What should the solutions architect do to deploy the stacks successfully?

A. Enable the new Regions in all relevant accounts. Specify the CAPABILITY_NAMED_IAM capability during the creation of the stack set.
B. Use the Service Quotas console to request a quota increase for the number of CloudFormation stacks in each new Region in all relevant accounts. Specify the CAPABILITY_IAM capability during the creation of the stack set.
C. Specify the CAPABILITY_NAMED_IAM capability and the SELF_MANAGED permissions model during the creation of the stack set.
D. Specify an administration role ARN and the CAPABILITY_IAM capability during the creation of the stack set.



# AWS SAP-C02 Question Analysis — Q328

## 📋 Analysis

### 1. Understanding the Problem

This question combines several AWS governance and deployment concepts:

- AWS CloudFormation StackSets
- Multi-account / multi-Region deployment
- IAM resource creation
- Custom-named IAM roles
- Newly unused AWS Regions

The critical clues are:

1. "Several previously unused AWS Regions"
2. "Template contains an IAM role that has a custom name"
3. "No stack instances are created successfully"

This means there are actually two deployment prerequisites involved:
- The target Regions must be enabled
- CloudFormation must be explicitly authorized to create named IAM resources

SAP-C02 often tests whether candidates recognize multiple simultaneous deployment blockers.

- --

## 2. Key AWS Concepts

### Named IAM Resources Require Special Capability

Whenever a CloudFormation template creates:
- IAM roles
- IAM users
- IAM policies

with explicitly defined names such as:

```yaml
RoleName: SecurityAuditRole
```

CloudFormation requires:

```text
CAPABILITY_NAMED_IAM
```

Without this capability:
- Stack creation fails immediately
- CloudFormation refuses deployment for security reasons

Important distinction:

| Capability | Purpose |
| --- | --- |
| CAPABILITY_IAM | Generic IAM creation |
| CAPABILITY_NAMED_IAM | IAM resources with custom names |

Custom names always require:
`CAPABILITY_NAMED_IAM`

- --

### New AWS Regions Must Be Enabled

The question explicitly says:
"previously unused AWS Regions"

In AWS Organizations environments:
- New Regions may be disabled by default
- Accounts must opt in to Region usage
- StackSets cannot deploy into disabled Regions

Therefore:
- Relevant accounts must enable those Regions first

This is the second hidden requirement.

- --

## 3. Options Analysis

### ✅ Option A (CORRECT)

> Enable the new Regions in all relevant accounts. Specify the CAPABILITY_NAMED_IAM capability during the creation of the stack set.

This correctly addresses BOTH root causes.

#### Why this works

##### Region enablement
The target Regions are previously unused, so:
- They may not be activated
- Stack instances cannot launch there until enabled

##### Named IAM capability
The template creates a custom-named IAM role, requiring:
- `CAPABILITY_NAMED_IAM`

This is exactly the required capability.

Therefore this option fully resolves the deployment issue.

- --

### ❌ Option B (INCORRECT)

> Use the Service Quotas console to request a quota increase for the number of CloudFormation stacks...

This is a distractor.

Problems:
- Nothing indicates stack quota exhaustion
- New Regions are not failing because of quotas
- CAPABILITY_IAM is insufficient for named IAM resources

The key mistake:
- Using `CAPABILITY_IAM` instead of `CAPABILITY_NAMED_IAM`

This alone makes the answer invalid.

- --

### ❌ Option C (INCORRECT)

> Specify the CAPABILITY_NAMED_IAM capability and the SELF_MANAGED permissions model...

Partial correctness only.

#### Correct part
- `CAPABILITY_NAMED_IAM` is required

#### Incorrect part
- SELF_MANAGED permissions model is unrelated to the failure
- Does not address disabled Regions

The question asks what is needed for successful deployment.

If Regions are disabled:
- Deployment still fails

Therefore incomplete.

- --

### ❌ Option D (INCORRECT)

> Specify an administration role ARN and the CAPABILITY_IAM capability...

Problems:
- Wrong capability
- `CAPABILITY_IAM` does not support custom IAM names
- Admin role ARN is not the core issue

Again, the missing requirement is:
`CAPABILITY_NAMED_IAM`

- --

## 4. Why AWS Requires CAPABILITY_NAMED_IAM

AWS treats named IAM resources as higher risk because:
- They can overwrite existing resources
- They may conflict with organizational standards
- Naming often implies elevated enterprise integration

Therefore CloudFormation requires explicit acknowledgment.

Example:

```yaml
Resources:
  SecurityRole:
    Type: AWS::IAM::Role
    Properties:
      RoleName: EnterpriseSecurityRole
```

This requires:

```bash
- -capabilities CAPABILITY_NAMED_IAM
```

Without it:
- Stack creation fails immediately

## 5. StackSets Deployment Architecture

Typical enterprise StackSets workflow:

### Administrator account
Hosts:
- StackSet definition
- Deployment orchestration

### Target accounts
Receive:
- Stack instances
- Regional deployments

### Common deployment blockers
SAP-C02 frequently tests these:

| Failure Cause | Result |
| --- | --- |
| Region disabled | Stack instance creation fails |
| Missing IAM capability | Template validation failure |
| Missing execution role | StackSet operation failure |
| SCP restrictions | Resource creation denied |
| Existing resource name conflict | Rollback/failure |

Understanding these operational failure modes is important for real enterprise deployments.

### 面试官可能追问 (Possible Follow-up Questions)

#### 1. CAPABILITY_IAM 和 CAPABILITY_NAMED_IAM 有什么区别？

回答要点：
- `CAPABILITY_IAM` allows CloudFormation to create IAM resources
- `CAPABILITY_NAMED_IAM` is specifically required when IAM resources have explicit custom names
- Named resources pose greater overwrite/conflict risk

- --

#### 2. StackSets 有哪两种权限模型？

回答要点：

##### Service-managed
- Integrated with AWS Organizations
- AWS automatically manages permissions

##### Self-managed
- Requires administrator and execution roles
- More manual setup
- Used for non-Organizations scenarios

- --

#### 3. 为什么新 Region 默认可能无法使用？

回答要点：
- Some AWS Regions are opt-in Regions
- Enterprises may disable Regions for:
  - Compliance
  - Cost control
  - Governance
  - Data residency requirements
