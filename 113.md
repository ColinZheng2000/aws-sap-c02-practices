An international delivery company hosts a delivery management system on AWS. Drivers use the system to upload confirmation of delivery. Confirmation includes the recipient’s signature or a photo of the package with the recipient. The driver’s handheld device uploads signatures and photos through FTP to a single Amazon EC2 instance. Each handheld device saves a file in a directory based on the signed-in user, and the file name matches the delivery number. The EC2 instance then adds metadata to the file after querying a central database to pull delivery information. The file is then placed in Amazon S3 for archiving.  
  
As the company expands, drivers report that the system is rejecting connections. The FTP server is having problems because of dropped connections and memory issues in response to these problems, a system engineer schedules a cron task to reboot the EC2 instance every 30 minutes. The billing team reports that files are not always in the archive and that the central system is not always updated.  
  
A solutions architect needs to design a solution that maximizes scalability to ensure that the archive always receives the files and that systems are always updated. The handheld devices cannot be modified, so the company cannot deploy a new application.  
  
Which solution will meet these requirements?

- A. Create an AMI of the existing EC2 instance. Create an Auto Scaling group of EC2 instances behind an Application Load Balancer. Configure the Auto Scaling group to have a minimum of three instances.
- B. Use AWS Transfer Family to create an FTP server that places the files in Amazon Elastic File System (Amazon EFS). Mount the EFS volume to the existing EC2 instance. Point the EC2 instance to the new path for file processing.
- C. Use AWS Transfer Family to create an FTP server that places the files in Amazon S3. Use an S3 event notification through Amazon Simple Notification Service (Amazon SNS) to invoke an AWS Lambda function. Configure the Lambda function to add the metadata and update the delivery system.
- D. Update the handheld devices to place the files directly in Amazon S3. Use an S3 event notification through Amazon Simple Queue Service (Amazon SQS) to invoke an AWS Lambda function. Configure the Lambda function to add the metadata and update the delivery system.

# Analysis of Scalable FTP File Upload & Processing Solution
## Understanding the Problem
The core requirements are:
1. **Maintain existing upload method**: Handheld devices cannot be modified—must retain FTP as the upload protocol.
2. **Solve scalability issues**: Resolve rejected/dropped connections and memory problems from the single EC2 FTP server.
3. **Ensure data integrity**: Guarantee all files are archived in S3 and the central delivery system is updated with metadata.
4. **Maximize scalability**: Accommodate company expansion without recurring performance or availability issues.

Current state: Single EC2 instance runs an FTP server, processes files (adds metadata via DB query), and moves files to S3. Issues include: FTP server overload (connections dropped, memory constraints), missing archived files, and incomplete central system updates. The cron reboot is a temporary band-aid, not a solution.

## Services Explanation
### Key AWS Services Involved
1. **AWS Transfer Family** (AWS Docs: [What Is AWS Transfer Family](https://docs.aws.amazon.com/transfer/latest/userguide/what-is-aws-transfer-family.html)):
   - Managed FTP/SFTP/FTPS service that eliminates the need to operate and scale EC2-based FTP servers.
   - Natively integrates with S3 or EFS for storage, supporting the existing FTP upload method (critical for unmodifiable handheld devices).
   - Scales automatically to handle thousands of concurrent connections—solves the core scalability issue of the single EC2 FTP server.
2. **Amazon S3** (AWS Docs: [Amazon S3 Features](https://aws.amazon.com/s3/features/)):
   - Durable, highly scalable object storage for archiving files. Eliminates file loss risks associated with the single EC2’s unreliability.
   - Supports event notifications to trigger downstream processing (e.g., metadata addition, system updates).
3. **S3 Event Notifications** (AWS Docs: [Configuring S3 Event Notifications](https://docs.aws.amazon.com/AmazonS3/latest/userguide/NotificationHowTo.html)):
   - Triggers actions (e.g., SNS, Lambda) when files are uploaded to S3. Ensures immediate processing of new files, preventing gaps in archiving or system updates.
4. **Amazon SNS** (AWS Docs: [What Is Amazon SNS](https://docs.aws.amazon.com/sns/latest/dg/welcome.html)):
   - Managed messaging service that forwards S3 event notifications to Lambda. Enables reliable decoupling between file upload and processing.
5. **AWS Lambda** (AWS Docs: [AWS Lambda Features](https://aws.amazon.com/lambda/features/)):
   - Serverless compute service that runs code to add metadata (via DB queries) and update the central delivery system.
   - Scales automatically with incoming events—no memory constraints or manual reboots required. Eliminates the single EC2’s processing bottleneck.
6. **Other Relevant Services (Incorrect Options)** :
   - **EC2 Auto Scaling + ALB**: ALB is designed for HTTP/HTTPS (not stateful FTP), which relies on multiple ports and connection state—load balancing FTP with ALB is unreliable. EC2-based FTP servers still require management and suffer from scalability limits.
   - **Amazon EFS**: Network file system that adds storage scalability but does not solve the processing bottleneck (still dependent on the existing EC2 instance).
   - **Amazon SQS**: Valid for event queuing, but Option D is invalid due to device modification constraints (not SQS itself).

## Correct Option Explanation (Option C)
This solution aligns with all constraints and requirements, leveraging AWS managed services for scalability and reliability:
1. **Retains FTP Upload**: AWS Transfer Family acts as a managed FTP server, so handheld devices (unmodifiable) can continue uploading via FTP—no changes to device behavior.
2. **Eliminates FTP Server Overhead**: Transfer Family scales automatically to handle concurrent connections, resolving dropped connections and memory issues from the single EC2.
3. **Durable Archiving**: Files are directly uploaded to S3 via Transfer Family—S3’s 99.999999999% (11 9’s) durability ensures no files are lost.
4. **Reliable Processing**: 
   - S3 event notifications trigger an SNS topic when a file is uploaded.
   - SNS invokes a Lambda function to add metadata (via DB queries) and update the central system.
   - Lambda’s serverless architecture scales with demand, eliminating processing bottlenecks and ensuring the central system is always updated.
5. **Minimal Operational Effort**: Managed services (Transfer Family, S3, SNS, Lambda) reduce overhead—no need for EC2 management, cron reboots, or manual scaling.

## Incorrect Options Explanation
### Option A (EC2 Auto Scaling + ALB)
- **Why Incorrect**: 
  1. **FTP Incompatibility with ALB**: ALB is optimized for HTTP/HTTPS (stateless protocols). FTP is stateful (uses control/data ports) and requires session persistence, which ALB does not natively support—this leads to dropped connections and unreliable transfers.
  2. **Continued EC2 Overhead**: Still relies on EC2 instances to run the FTP server, requiring management (patching, scaling, monitoring) and suffering from the same memory constraints as the single EC2 (just scaled horizontally).
  3. **No Processing Scalability**: The EC2 instances still handle file processing (metadata addition, system updates), which remains a bottleneck as the company expands.

### Option B (Transfer Family + EFS + Existing EC2)
- **Why Incorrect**: 
  1. **Processing Bottleneck Remains**: The existing EC2 instance (which had memory issues) is still used for file processing. Mounting EFS solves storage scalability but not the EC2’s processing limits—files may still fail to process if the EC2 crashes or is overloaded.
  2. **Unnecessary Complexity**: EFS adds network file system overhead vs. direct S3 integration. Transfer Family can write directly to S3, making EFS redundant and increasing operational complexity.
  3. **Single Point of Failure**: The EC2 instance remains a single point of failure for processing—if it goes down, files in EFS are not processed, leading to missing updates and archived files.

### Option D (Update Handheld Devices + S3 + SQS + Lambda)
- **Why Incorrect**: 
  1. **Violates Device Constraint**: The problem explicitly states handheld devices cannot be modified. This option requires updating devices to upload directly to S3 (bypassing FTP), which is not allowed.
  2. **Irrelevant of Other Components**: Even though SQS + Lambda is a valid processing pattern, the core constraint (unmodifiable devices) invalidates the entire solution.
