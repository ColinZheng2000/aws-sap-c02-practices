A company has AWS accounts that are in an organization in AWS Organizations. The company wants to track Amazon EC2 usage as a metric. The company’s architecture team must receive a daily alert if the EC2 usage is more than 10% higher the average EC2 usage from the last 30 days.

Which solution will meet these requirements?

A. Configure AWS Budgets in the organization's management account. Specify a usage type of EC2 running hours. Specify a daily period. Set the budget amount to be 10% more than the reported average usage for the last 30 days from AWS Cost Explorer. Configure an alert to notify the architecture team if the usage threshold is met
B. Configure AWS Cost Anomaly Detection in the organization's management account. Configure a monitor type of AWS Service. Apply a filter of Amazon EC2. Configure an alert subscription to notify the architecture team if the usage is 10% more than the average usage for the last 30 days.
C. Enable AWS Trusted Advisor in the organization's management account. Configure a cost optimization advisory alert to notify the architecture team if the EC2 usage is 10% more than the reported average usage for the last 30 days.
D. Configure Amazon Detective in the organization's management account. Configure an EC2 usage anomaly alert to notify the architecture team if Detective identifies a usage anomaly of more than 10%.


# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

The requirement is about centralized monitoring of Amazon EC2 usage across multiple AWS accounts managed by AWS Organizations.

The key technical requirements are:

- Organization-level monitoring
- Daily alerting
- Historical trend comparison
- Automatic anomaly detection
- Alert when EC2 usage exceeds normal baseline by 10%
- Minimal manual operational overhead

The phrase “10% higher than the average usage from the last 30 days” is the most important clue. This implies dynamic behavioral analysis instead of static thresholds.

This is a classic FinOps and governance scenario frequently tested in SAP-C02.

---

### 2. Options Analysis

#### ✅ Option B (CORRECT): Use AWS Cost Anomaly Detection for EC2 monitoring

AWS Cost Anomaly Detection is purpose-built for identifying unusual cost or usage behavior using machine learning models trained on historical trends.

Why this satisfies the requirements:

- Supports AWS Organizations centralized monitoring
- Can monitor AWS services individually
- Amazon EC2 can be specifically filtered
- Uses historical baseline analysis automatically
- Generates anomaly alerts
- Eliminates the need for manually recalculating thresholds
- Works well for daily anomaly monitoring

The service continuously evaluates current usage against learned historical patterns, which directly matches the requirement of comparing against the previous 30 days.

Enterprise perspective:

Large organizations often experience normal workload fluctuations caused by:
- Batch jobs
- Month-end processing
- Auto Scaling events
- Development spikes

Static limits create excessive false positives. Cost Anomaly Detection reduces operational noise by understanding historical behavior patterns automatically.

#### ❌ Option A (INCORRECT): AWS Budgets with manual threshold configuration

AWS Budgets can monitor usage and send alerts, but it does not provide adaptive anomaly detection.

Why this is incorrect:

- Requires manually defining thresholds
- Does not automatically calculate rolling 30-day averages
- Cannot intelligently adapt to workload trends
- Introduces operational maintenance burden
- Static thresholds are less accurate for anomaly detection

Important exam distinction:

- AWS Budgets = governance and fixed spending controls
- AWS Cost Anomaly Detection = intelligent anomaly detection using ML

Real-world issue:

In enterprise environments, manually updating budget thresholds becomes impractical as infrastructure scales across multiple accounts and teams.

Official reference:
https://docs.aws.amazon.com/cost-management/latest/userguide/budgets-managing-costs.html

---

#### ❌ Option C (INCORRECT): AWS Trusted Advisor cost optimization alerts

Trusted Advisor is not designed for usage anomaly detection.

Why this fails:

- Provides best-practice recommendations
- Focuses on optimization checks
- Does not compare current EC2 usage against rolling historical averages
- No built-in anomaly monitoring engine for EC2 usage behavior

Trusted Advisor is commonly used for:
- Idle resources
- RI optimization
- Security recommendations
- Fault tolerance checks

It is not a FinOps anomaly analytics platform.

Official reference:
https://docs.aws.amazon.com/awssupport/latest/user/trusted-advisor.html

---

#### ❌ Option D (INCORRECT): Amazon Detective anomaly alerting

Amazon Detective is a security investigation service.

Why this is incorrect:

- Focused on security analytics and threat investigation
- Analyzes findings from services like GuardDuty
- Does not monitor AWS usage trends or billing anomalies
- No capability for EC2 usage anomaly detection

This option intentionally mixes security operations with financial governance.

Official reference:
https://docs.aws.amazon.com/detective/latest/userguide/what-is-detective.html

### 3. Hands-On Knowledge

In real enterprise environments, AWS Cost Anomaly Detection is commonly integrated into centralized FinOps operations.

Typical architecture:

1. AWS Organizations enabled
2. Consolidated billing configured
3. Cost Anomaly Detection created in management account
4. Service monitor configured for Amazon EC2
5. SNS notifications integrated with:
   - Email
   - Slack
   - Microsoft Teams
   - PagerDuty
   - ServiceNow

Common operational patterns:

- Separate monitors by:
  - Production
  - Development
  - Business unit
  - Shared platform account

- Combine with:
  - Cost Categories
  - CUR (Cost and Usage Report)
  - Athena analytics
  - QuickSight dashboards

Important operational gotchas:

- Initial anomaly learning requires historical billing data
- Sudden legitimate scaling events may trigger alerts
- Proper tagging strategy significantly improves investigation speed
- Daily summary alerts are usually better than immediate alerts for large enterprises

Useful CLI example:

Create anomaly monitor:
```bash
aws ce create-anomaly-monitor \
  --anomaly-monitor '{
    "MonitorName":"EC2-Monitor",
    "MonitorType":"DIMENSIONAL",
    "MonitorDimension":"SERVICE"
  }'
```

Create alert subscription:
```bash
aws ce create-anomaly-subscription \
  --anomaly-subscription '{
    "SubscriptionName":"EC2-Anomaly-Alerts",
    "Frequency":"DAILY",
    "Threshold":10
  }'
```

SAP-C02 exam tip:

When the question mentions:
- historical baseline
- unusual spending
- anomaly
- machine learning
- unexpected increase

The answer is very often AWS Cost Anomaly Detection rather than Budgets.
