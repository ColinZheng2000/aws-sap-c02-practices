A financial services company runs a complex, multi-tier application on Amazon EC2 instances and AWS Lambda functions. The application stores temporary data in Amazon S3. The S3 objects are valid for only 45 minutes and are deleted after 24 hours.

The company deploys each version of the application by launching an AWS CloudFormation stack. The stack creates all resources that are required to run the application. When the company deploys and validates a new application version, the company deletes the CloudFormation stack of the old version.

The company recently tried to delete the CloudFormation stack of an old application version, but the operation failed. An analysis shows that CloudFormation failed to delete an existing S3 bucket. A solutions architect needs to resolve this issue without making major changes to the application's architecture.

Which solution meets these requirements?

A. Implement a Lambda function that deletes all files from a given S3 bucket. Integrate this Lambda function as a custom resource into the CloudFormation stack. Ensure that the custom resource has a DependsOn attribute that points to the S3 bucket's resource.
B. Modify the CloudFormation template to provision an Amazon Elastic File System (Amazon EFS) file system to store the temporary files there instead of in Amazon S3. Configure the Lambda functions to run in the same VPC as the file system. Mount the file system to the EC2 instances and Lambda functions.
C. Modify the CloudFormation stack to create an S3 Lifecycle rule that expires all objects 45 minutes after creation. Add a DependsOn attribute that points to the S3 bucket’s resource.
D. Modify the CloudFormation stack to attach a DeletionPolicy attribute with a value of Delete to the S3 bucket.


# CloudFormation Fails to Delete S3 Bucket

## Understanding the Problem

### Architecture Flow

```text
CloudFormation Stack
       
        ├── EC2 Instances
        ├── Lambda Functions
        └── S3 Bucket (temporary objects)
                    │
                    └── Objects remain up to 24 hours
```

### What Happened?

```text
Stack Deletion Started
        │
        └── CloudFormation tries deleting S3 bucket
                    │
                    └── Bucket still contains objects
                                │
                                └── Deletion fails
```

### Core AWS Knowledge

Amazon S3 buckets:
- MUST be empty before deletion
- CloudFormation does NOT automatically delete bucket objects

---

# Option Analysis
| Option | Result | Key Reason |
|---|---|---|
| A | ✅ Correct | Deletes bucket objects before bucket deletion |
| B | ❌ Incorrect | Major architecture change |
| C | ❌ Incorrect | Lifecycle expiration is asynchronous |
| D | ❌ Incorrect | `DeletionPolicy: Delete` does not empty bucket |

---

# Correct Option
## A. Lambda-backed Custom Resource
### Solution Route
```text
CloudFormation Delete
        │
        ├── Invoke Custom Resource
                │
        │         └── Lambda deletes all S3 objects
        │
        └── CloudFormation deletes empty bucket successfully
```

### This Works

CloudFormation can delete:
- empty S3 buckets

CloudFormation cannot delete:
- non-empty S3 buckets

The Lambda custom resource performs the missing cleanup step.

### Why `DependsOn` Matters

`DependsOn` ensures:

```text
Cleanup Lambda runs
        ↓
Objects deleted
        ↓
Bucket deletion starts
```

Without ordering control:
- bucket deletion may occur too early

---

# Incorrect Options

## B. Replace S3 with EFS

### Why Wrong

| Issue | Explanation |
|---|---|
| Major redesign | Violates requirement |
| More complexity | VPC + mount targets required |
| Lambda VPC overhead | start impact |
| Wrong storage type | EFS not ideal for temporary object storage |

---

## C. S3 Lifecycle Rule

### Why Wrong

### Lifecycle Timing

```text
Object reaches expiration time
        │
        └── S3 removes object later (async process)
```

AWS does NOT guarantee immediate deletion.

Result:
- bucket may still contain objects during stack deletion

### Important Exam Point

Lifecycle policies:
- optimize storage management
- are NOT deterministic cleanup mechanisms

---

## D. DeletionPolicy: Delete

### Why Wrong

`DeletionPolicy: Delete` means:

```text
CloudFormation should attempt deletion
```

It does NOT mean:

```text
Automatically empty the bucket
```

The bucket still fails deletion if objects exist.

---

# Important AWS Service Behaviors

| Service | Important Behavior |
|---|---|
| Amazon S3 | Bucket must be empty before deletion |
| CloudFormation | Does not purge S3 objects automatically |
| Lambda Custom Resource | Common cleanup automation pattern |
| S3 Lifecycle | Asynchronous expiration process |

# Enterprise Hands-on Knowledge

## Common Production Pattern

Many enterprises use:

```text
CloudFormation
    + Custom Cleanup Lambda
```

for:
- temporary environments
- blue/green deployments
- CI/CD ephemeral
- test environments

---

## Versioned Bucket Consideration

If S3 Versioning is enabled:

The cleanup Lambda must delete:
- object versions
- delete markers

Otherwise deletion still fails.

### Real-world Operational Issue

Many engineers only delete:
- current objects

But forget:
- previous versions

Result:
- stack deletion still fails

---

## CDK Equivalent

AWS CDK internally solves this using:
```typescript
autoDeleteObjects: true
```

This actually deploy:
- a hidden custom cleanup Lambda

Useful SAP-C02 insight:
- many AWS abstractions internally use Lambda custom resources
