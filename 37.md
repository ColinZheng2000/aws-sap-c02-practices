A company is planning to migrate 1,000 on-premises servers to AWS. The servers run on several VMware clusters in the company’s data center. As part of the migration plan, the company wants to gather server metrics such as CPU details, RAM usage, operating system information, and running processes. The company then wants to query and analyze the data.  
  
Which solution will meet these requirements?

- A. Deploy and configure the AWS Agentless Discovery Connector virtual appliance on the on-premises hosts. Configure Data Exploration in AWS Migration Hub. Use AWS Glue to perform an ETL job against the data. Query the data by using Amazon S3 Select.
- B. Export only the VM performance information from the on-premises hosts. Directly import the required data into AWS Migration Hub. Update any missing information in Migration Hub. Query the data by using Amazon QuickSight.
- C. Create a script to automatically gather the server information from the on-premises hosts. Use the AWS CLI to run the put-resource-attributes command to store the detailed server data in AWS Migration Hub. Query the data directly in the Migration Hub console.
- D. Deploy the AWS Application Discovery Agent to each on-premises server. Configure Data Exploration in AWS Migration Hub. Use Amazon Athena to run predefined queries against the data in Amazon S3.

## 💡 Analysis

### Understanding the Problem

The company is migrating 1,000 VMware-based on-premises servers to AWS. The requirement is to:

1. **Gather detailed server metrics** from the on-premises hosts, including:
    
    - CPU details
        
    - RAM usage
        
    - Operating system (OS) information
        
    - Running processes
        
2. **Query and analyze** this collected data.
    

The solution must be scalable for 1,000 servers and fit within the typical AWS migration process.

---

### Services Explanation

| **Service**                               | **Description**                                                                                                                                                                                                                | **Relevance to the Problem**                                                                                                                    |
| ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **AWS Application Discovery Service**     | Helps plan migration projects by gathering information about on-premises data centers, including server utilization, configuration, and network dependencies. It has two methods: Agentless (for VMware only) and Agent-based. | The primary tool for **gathering server metrics and details** for migration planning.                                                           |
| **AWS Application Discovery Agent**       | Software installed directly on on-premises servers (OS-level) that collects detailed system performance and configuration data, including running processes and OS information.                                                | Specifically designed to meet the requirement for **detailed, real-time metrics** and **running processes**.                                    |
| **AWS Agentless Discovery Connector**     | A virtual appliance deployed on VMware vCenter that collects configuration data, performance history, and network dependency mapping **without installing an agent on the OS**.                                                | Collects high-level information but **lacks the ability to gather detailed, per-process metrics** which is a key requirement.                   |
| **AWS Migration Hub**                     | Provides a single location to track the progress of application migrations across multiple AWS and partner solutions. It can use data collected by the Discovery Service.                                                      | Central repository for migration data. **Data Exploration** feature is key for analysis.                                                        |
| **Data Exploration in AWS Migration Hub** | A feature that enables querying and analyzing the collected server data (from Application Discovery Service) using Amazon Athena against a dataset stored in Amazon S3.                                                        | The mechanism for **querying and analyzing** the collected data.                                                                                |
| **Amazon Athena**                         | An interactive query service that makes it easy to analyze data directly in Amazon S3 using standard SQL.                                                                                                                      | The tool used by **Data Exploration** to **query the server data** stored in S3.                                                                |
| **AWS Glue**                              | A serverless data integration service that makes it easy to discover, prepare, and combine data for analytics, machine learning, and application development (often used for ETL).                                             | Too complex and unnecessary for simply querying pre-collected server data. Data Exploration/Athena handles the analysis directly.               |
| **Amazon S3 Select**                      | Allows for retrieving a subset of data from an S3 object by using simple SQL expressions.                                                                                                                                      | Used for simple retrieval from a single S3 object, not for complex, cross-file analysis of a structured dataset like server discovery data.     |
| **Amazon QuickSight**                     | A scalable, serverless, embeddable, machine learning-powered business intelligence (BI) service.                                                                                                                               | Used for visualization and reporting, but not the primary tool for executing _predefined queries_ on raw discovery data. Athena is more direct. |

### Correct Option(s) Explanation

**D. Deploy the AWS Application Discovery Agent to each on-premises server. Configure Data Exploration in AWS Migration Hub. Use Amazon Athena to run predefined queries against the data in Amazon S3.**

- **Deploy the AWS Application Discovery Agent to each on-premises server:** This is the correct choice because the **agent** collects the **detailed operational metrics** required, specifically **running processes**, CPU details, and RAM usage, by operating at the OS level. The agentless connector generally doesn't collect running processes.
    
- **Configure Data Exploration in AWS Migration Hub:** This feature automatically sets up the data pipeline, moving the agent-collected data into an S3 bucket and configuring the necessary metadata (via AWS Glue Data Catalog) to make the data queryable.
    
- **Use Amazon Athena to run predefined queries against the data in Amazon S3:** This is the native mechanism by which **Data Exploration** in Migration Hub allows users to analyze the collected server inventory and performance data. Athena, being a standard SQL query engine over S3, is perfect for the "query and analyze" requirement.
    

---

### Incorrect Options Explanation

**A. Deploy and configure the AWS Agentless Discovery Connector virtual appliance on the on-premises hosts. Configure Data Exploration in AWS Migration Hub. Use AWS Glue to perform an ETL job against the data. Query the data by using Amazon S3 Select.**

- **Incorrect Service/Data:** The **Agentless Discovery Connector** does _not_ provide the level of detailed operational data required, particularly **running processes**. It focuses more on configuration and basic performance.
    
- **Incorrect Analysis Toolchain:** Using **AWS Glue to perform an ETL job** and then **Amazon S3 Select** is an overly complex and incorrect approach. The native solution for querying this data, as part of Migration Hub's Data Exploration, is **Amazon Athena**. S3 Select is not suitable for complex, cross-file analysis of a structured dataset.
    

**B. Export only the VM performance information from the on-premises hosts. Directly import the required data into AWS Migration Hub. Update any missing information in Migration Hub. Query the data by using Amazon QuickSight.**

- **Incorrect Data Ingestion:** Manually exporting, importing, and updating data for **1,000 servers** is not a scalable or professional solution. The process is prone to errors, requires custom development, and loses the ongoing, automated collection capabilities of the Discovery Service.
    
- **Incorrect Analysis Tool:** While QuickSight can visualize the data, Athena is the more direct and appropriate tool for running _predefined queries_ for analysis on the raw Discovery Service data, especially when using the Data Exploration feature.
    

**C. Create a script to automatically gather the server information from the on-premises hosts. Use the AWS CLI to run the `put-resource-attributes` command to store the detailed server data in AWS Migration Hub. Query the data directly in the Migration Hub console.**

- **Incorrect Data Ingestion:** Creating a custom script and using the `put-resource-attributes` command is meant for **updating** specific resource details, not for scalable, continuous ingestion of high-volume, time-series performance and process data from 1,000 servers. This is not the intended use case for bulk data collection.
    
- **Incorrect Analysis Tool:** The Migration Hub console allows basic viewing, but it does **not** allow for running complex SQL queries (the "query and analyze" requirement) directly against the raw performance data. This analysis requires the Data Exploration/Athena setup.
