A company wants to migrate its data analytics environment from on premises to AWS. The environment consists of two simple Node.js applications. One of the applications collects sensor data and loads it into a MySQL database. The other application aggregates the data into reports. When the aggregation jobs run, some of the load jobs fail to run correctly.  
The company must resolve the data loading issue. The company also needs the migration to occur without interruptions or changes for the company's customers.  
What should a solutions architect do to meet these requirements?  

- A. Set up an Amazon Aurora MySQL database as a replication target for the on-premises database. Create an Aurora Replica for the Aurora MySQL database, and move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as AWS Lambda functions behind a Network Load Balancer (NLB), and use Amazon RDS Proxy to write to the Aurora MySQL database. When the databases are synced, disable the replication job and restart the Aurora Replica as the primary instance. Point the collector DNS record to the NLB.
- B. Set up an Amazon Aurora MySQL database. Use AWS Database Migration Service (AWS DMS) to perform continuous data replication from the on-premises database to Aurora. Move the aggregation jobs to run against the Aurora MySQL database. Set up collection endpoints behind an Application Load Balancer (ALB) as Amazon EC2 instances in an Auto Scaling group. When the databases are synced, point the collector DNS record to the ALB. Disable the AWS DMS sync task after the cutover from on premises to AWS.
- C. Set up an Amazon Aurora MySQL database. Use AWS Database Migration Service (AWS DMS) to perform continuous data replication from the on-premises database to Aurora. Create an Aurora Replica for the Aurora MySQL database, and move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as AWS Lambda functions behind an Application Load Balancer (ALB), and use Amazon RDS Proxy to write to the Aurora MySQL database. When the databases are synced, point the collector DNS record to the ALB. Disable the AWS DMS sync task after the cutover from on premises to AWS.
- D. Set up an Amazon Aurora MySQL database. Create an Aurora Replica for the Aurora MySQL database, and move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as an Amazon Kinesis data stream. Use Amazon Kinesis Data Firehose to replicate the data to the Aurora MySQL database. When the databases are synced, disable the replication job and restart the Aurora Replica as the primary instance. Point the collector DNS record to the Kinesis data stream.

## Understanding the Problem

The core of the problem has three parts:
1.  **Migration:** The company needs to move an on-premises MySQL database and two Node.js applications (one for collection, one for reporting) to AWS.
2.  **Performance Issue:** There is a critical resource contention problem. The "aggregation" (read-heavy) jobs are causing the "collection" (write-heavy) jobs to fail. This must be resolved.
3.  **Constraint:** The migration must happen with "no interruptions" for customers. This implies a continuous replication and cutover strategy, not a "stop and copy" (cold) migration.

A successful solution *must* address all three points: a valid migration path, a fix for the read/write contention, and a "hot" cutover plan.

## Services Explanation

* **Amazon Aurora (MySQL-compatible):** A fully managed, relational database service that combines the performance and availability of high-end commercial databases with the simplicity and cost-effectiveness of open-source databases. Its key feature relevant here is the separation of storage and compute, which allows for fast, low-latency **Read Replicas** (up to 15) that share the same underlying storage volume.
    * *Official Doc:* [What is Amazon Aurora?](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Overview.html)

* **AWS Database Migration Service (DMS):** The standard AWS service for migrating databases to AWS with minimal downtime. It can perform a one-time full load and then continuously replicate ongoing changes (Change Data Capture - CDC) from the source (on-premises MySQL) to the target (Aurora).
    * *Official Doc:* [What is AWS Database Migration Service?](https://docs.aws.amazon.com/dms/latest/userguide/Welcome.html)

* **Aurora Read Replica:** A read-only copy of the primary database instance. Directing read-heavy traffic (like the aggregation/reporting jobs) to a Read Replica is the standard pattern for alleviating read/write contention on the primary instance.
    * *Official Doc:* [Amazon Aurora Replicas](https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/Aurora.Replication.html)

* **AWS Lambda:** A serverless, event-driven compute service. It's ideal for running code for "collector" applications without managing servers. It scales automatically but can create a high number of concurrent, short-lived database connections.

* **Application Load Balancer (ALB):** A Layer 7 (HTTP/S) load balancer. It's perfectly suited for routing web traffic to targets like EC2 instances, containers, and, importantly, **Lambda functions**.

* **Network Load Balancer (NLB):** A Layer 4 (TCP) load balancer. It's designed for ultra-high performance and static IP addresses. While it *can* have Lambda targets, ALB is generally a more natural fit for HTTP-based applications.

* **Amazon RDS Proxy:** A fully managed, highly available database proxy. Its primary job is to **pool and share** database connections. This is *critical* when using services like Lambda, which can open thousands of concurrent connections and easily exhaust the database's connection limit.
    * *Official Doc:* [Using Amazon RDS Proxy](https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/rds-proxy.html)

## Correct Option(s) Explanation

### ✅ C. Set up an Amazon Aurora MySQL database. Use AWS Database Migration Service (AWS DMS) to perform continuous data replication... Create an Aurora Replica... move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as AWS Lambda functions behind an Application Load Balancer (ALB), and use Amazon RDS Proxy... When the databases are synced, point the collector DNS record to the ALB. Disable the AWS DMS sync task...

This option is the only one that correctly addresses all three requirements of the problem:

1.  **Migration & No Downtime:** It uses **AWS DMS** for "continuous data replication." This is the best-practice method for a minimal-downtime database migration, satisfying the "no interruptions" requirement.
2.  **Performance Fix:** It creates an **Aurora Replica** and moves the "aggregation jobs" (the heavy read workload) to it. This directly solves the contention problem. Writes from the collection app go to the primary instance, and reads from the reporting app go to the replica.
3.  **Application Architecture:** It proposes a modern, scalable, and robust serverless architecture for the collector app:
    * **ALB:** Provides the public-facing HTTP/S endpoint for the collectors.
    * **Lambda:** Runs the Node.js collector code without server management.
    * **RDS Proxy:** Solves the classic "Lambda-to-RDS" connection-storm problem by pooling connections. This is a key design pattern.
4.  **Cutover:** The cutover plan is sound. Once DMS has the Aurora database in sync with the on-premises one, the application (collector) DNS is pointed to the new AWS endpoint (the ALB). After traffic is fully migrated, the DMS sync can be safely disabled.

## Incorrect Options Explanation

### ❌ A. Set up an Amazon Aurora MySQL database as a replication target... Create an Aurora Replica... move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as AWS Lambda functions behind a Network Load Balancer (NLB), and use Amazon RDS Proxy...

* **Vague/Incorrect Migration:** "Set up... as a replication target" is vague. It doesn't specify AWS DMS. While native MySQL replication is possible, DMS is the purpose-built AWS service for this, especially for minimal-downtime cutovers.
* **Flawed Cutover:** The instruction to "restart the Aurora Replica as the primary instance" is operationally incorrect for this scenario. In a DMS migration, the *target primary instance* (the one DMS writes to) becomes the new primary. You don't "restart" the read replica to make it primary; you *promote* it, but that's not the standard DMS cutover flow.
* **Sub-optimal Service:** Using an NLB (Layer 4) for a Lambda-based collector app is less ideal than an ALB (Layer 7), which is designed for HTTP/S traffic and has richer routing features.

### ❌ B. Set up an Amazon Aurora MySQL database. Use AWS Database Migration Service (AWS DMS)... Move the aggregation jobs to run against the Aurora MySQL database. Set up collection endpoints behind an Application Load Balancer (ALB) as Amazon EC2 instances...

* **Critical Failure:** This option **does not solve the contention problem**. It migrates the applications and database, but it points the aggregation jobs to the *primary* ("the Aurora MySQL database"). This simply moves the exact same read/write contention problem from on-premises to AWS. The collection app's writes will *still* fail when the aggregation jobs run. This violates a primary requirement.

### ❌ D. Set up an Amazon Aurora MySQL database. Create an Aurora Replica... move the aggregation jobs to run against the Aurora Replica. Set up collection endpoints as an Amazon Kinesis data stream. Use Amazon Kinesis Data Firehose to replicate the data...

* **Wrong Migration Tool:** This option confuses data *streaming* with database *replication*. **Kinesis is not a database migration tool.** You cannot use Kinesis Data Firehose to "replicate" an existing MySQL database (with its schema, existing data, and tables) to Aurora. Kinesis is for ingesting new, real-time data streams. This makes the entire migration strategy invalid.
* **Flawed Cutover:** Like option A, it describes an incorrect cutover process ("restart the Aurora Replica as the primary instance").
