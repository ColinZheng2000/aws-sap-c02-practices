A company’s public API runs as tasks on Amazon Elastic Container Service (Amazon ECS). The tasks run on AWS Fargate behind an Application Load Balancer (ALB) and are configured with Service Auto Scaling for the tasks based on CPU utilization. This service has been running well for several months.

Recently, API performance slowed down and made the application unusable. The company discovered that a significant number of SQL injection attacks had occurred against the API and that the API service had scaled to its maximum amount.

A solutions architect needs to implement a solution that prevents SQL injection attacks from reaching the ECS API service. The solution must allow legitimate traffic through and must maximize operational efficiency.

Which solution meets these requirements?

A. Create a new AWS WAF web ACL to monitor the HTTP requests and HTTPS requests that are forwarded to the ALB in front of the ECS tasks.
B. Create a new AWS WAF Bot Control implementation. Add a rule in the AWS WAF Bot Control managed rule group to monitor traffic and allow only legitimate traffic to the ALB in front of the ECS tasks.
C. Create a new AWS WAF web ACL. Add a new rule that blocks requests that match the SQL database rule group. Set the web ACL to allow all other traffic that does not match those rules. Attach the web ACL to the ALB in front of the ECS tasks.
D. Create a new AWS WAF web ACL. Create a new empty IP set in AWS WAF. Add a new rule to the web ACL to block requests that originate from IP addresses in the new IP set. Create an AWS Lambda function that scrapes the API logs for IP addresses that send SQL injection attacks, and add those IP addresses to the IP set. Attach the web ACL to the ALB in front of the ECS tasks.

Suggested Answer:  C

---

## 📋 Analysis

### 1. Understanding the Problem

The company's **public API** (ECS/Fargate behind ALB) is under **SQL injection attack**. Key symptoms:

- Service Auto Scaling **scaled to its maximum** — the attack is consuming all provisioned compute resources, making the API unusable for legitimate users.
- The company **already knows** these are SQL injection attacks.
- The solution must **prevent** SQL injection attacks from ever reaching the ECS API service, while **allowing legitimate traffic** through.
- **Operational efficiency** is required — meaning minimal manual intervention, no custom code to maintain.

The issue is at the **application layer (Layer 7)** — a Web Application Firewall is needed, and it must be configured to **block** malicious requests, not just observe them.

---

### 2. Options Analysis

#### ✅ **Option C (CORRECT): AWS WAF Web ACL + SQL Database Managed Rule Group + Block**

This is the **purpose-built, turnkey solution**:

1. **AWS WAF** is a Layer 7 web application firewall that inspects HTTP(S) requests before they reach the ALB.
2. **SQL database managed rule group** is an **AWS-managed rule group** that contains rules specifically designed to detect and block SQL injection patterns (e.g., `' OR 1=1 --`, `UNION SELECT`, `DROP TABLE`, etc.).
3. **Block action** on matching rules — requests matching SQL injection signatures are immediately rejected (HTTP 403 Forbidden), never reaching the ECS tasks.
4. **Default action: Allow** — all non-matching traffic passes through to the ALB.
5. The web ACL is attached to the **ALB** — WAF natively integrates with ALB, API Gateway, CloudFront, and AppSync.

**Why this is the optimal solution:**
- **Preventive, not reactive:** Malicious requests are blocked at the edge before consuming any ECS compute capacity.
- **Zero operational overhead:** The SQL database managed rule group is automatically maintained and updated by AWS Threat Research — no custom rule writing or signature maintenance.
- **Maximizes operational efficiency:** The ECS service no longer wastes resources processing malicious payloads; Auto Scaling behaves correctly for legitimate traffic only.
- **Immediate protection:** Once the web ACL is created and associated with the ALB, protection is active within seconds.

#### ❌ **Option A (INCORRECT): WAF Web ACL to "Monitor" Only**

- The option says **"monitor"** the HTTP requests, not **block** them.
- Monitoring (using WAF in **Count** mode) is useful for initial testing and baselining, but it does NOT prevent attacks from reaching the API.
- The company already knows SQL injection is happening — they need blocking, not more monitoring.
- **Key exam tip:** In WAF, the same rule can be configured with different actions — `Block`, `Allow`, `Count`, or `CAPTCHA`. The wording "monitor" implies `Count` mode, which does not prevent attacks.

#### ❌ **Option B (INCORRECT): AWS WAF Bot Control**

- **AWS WAF Bot Control** is a managed rule group designed to detect and manage **bot traffic** (scrapers, scanners, crawlers, automated clients). It categorizes bots as verified, benign, or undesirable.
- Bot Control does NOT detect SQL injection patterns — it detects bot behavior (signature-based and behavior-based). SQL injection is a code injection attack, not necessarily bot-related.
- A legitimate user with a slightly malformed query parameter could trigger a SQL injection attempt without being a bot. Conversely, a bot could send perfectly legitimate requests.
- Bot Control and SQL injection protection solve **different problems** — don't confuse them.

#### ❌ **Option D (INCORRECT): WAF + Empty IP Set + Lambda Scraping Logs + Block by IP**

- This is a **reactive, manual, and ultimately ineffective approach:**
  - **Reactive:** Attacks must first succeed and appear in logs before IPs are blocked. Meanwhile, the API remains vulnerable.
  - **Whack-a-mole:** Attackers can rotate IP addresses, use botnets, or distribute attacks across thousands of proxies. An IP blocklist quickly becomes a maintenance nightmare.
  - **Operational overhead:** A custom Lambda function to parse logs, extract IPs, and update the IP set is custom code that must be maintained, monitored, and debugged.
  - **Doesn't maximize operational efficiency** — it's the opposite: a fragile custom pipeline.
- **IP-based blocking is a blunt instrument** — it can't distinguish between a malicious SQL injection request and a legitimate request from the same IP that happens to contain SQL-like syntax in a search field or comment box.

### 3. Hands-On Knowledge

#### AWS WAF Managed Rule Groups for Application Security

AWS WAF provides several **AWS-managed rule groups** for common threats. The most relevant to the SAP-C02 exam:

| Managed Rule Group | Protects Against | Use Case |
|-------------------|------------------|----------|
| **SQL database** | SQL injection attempts (e.g., `' OR 1=1`) | APIs, web apps with database backends |
| **Core rule set (CRS)** | OWASP Top 10 (XSS, LFI, RFI, etc.) | General web application protection |
| **Known bad inputs** | Exploit attempts, malware signatures | Broad threat intelligence |
| **PHP / WordPress / Linux / Windows** | Platform-specific vulnerabilities | Specific platform protection |
| **Bot Control** | Automated bot traffic | Scraping, credential stuffing, inventory hoarding |
| **Account takeover prevention** | Credential stuffing, brute-force login | Login/authentication endpoints |
| **ATP (Advanced Threat Protection)** | Advanced request analysis with ML | Sophisticated, evolving threats |

#### Implementing the Correct Solution (Option C)

**Step 1: Create WAF Web ACL**
```bash
aws wafv2 create-web-acl \
  --name sql-injection-protection \
  --scope REGIONAL \
  --default-action '{"Allow": {}}' \
  --visibility-config '{
    "SampledRequestsEnabled": true,
    "CloudWatchMetricsEnabled": true,
    "MetricName": "sql-injection-web-acl"
  }' \
  --region eu-west-1
```

**Step 2: Add SQL Database Managed Rule Group**
```bash
aws wafv2 update-web-acl \
  --name sql-injection-protection \
  --scope REGIONAL \
  --id <web-acl-id> \
  --lock-token <lock-token> \
  --rules '[
    {
      "Name": "AWS-AWSManagedRulesSQLiRuleSet",
      "Priority": 1,
      "Statement": {
        "ManagedRuleGroupStatement": {
          "VendorName": "AWS",
          "Name": "AWSManagedRulesSQLiRuleSet"
        }
      },
      "OverrideAction": {"None": {}},
      "VisibilityConfig": {
        "SampledRequestsEnabled": true,
        "CloudWatchMetricsEnabled": true,
        "MetricName": "sql-injection-rule-group"
      }
    }
  ]' \
  --region eu-west-1
```

**Step 3: Associate Web ACL with ALB**
```bash
aws wafv2 associate-web-acl \
  --web-acl-arn arn:aws:wafv2:region:account:regional/webacl/sql-injection-protection/xxx \
  --resource-arn arn:aws:elasticloadbalancing:region:account:loadbalancer/app/api-alb/xxx
```

**Best Practices for WAF in Production:**

1. **Start with Count mode** when deploying new managed rule groups. Let them run for a period (1-2 weeks) to identify false positives:
   ```bash
   # Set OverrideAction to Count for initial deployment
   "OverrideAction": {"Count": {}}
   ```

2. **Review WAF logs** (sent to CloudWatch Logs, S3, or Kinesis Data Firehose) to identify false positives from legitimate traffic.

3. **Create exception rules** (custom rules with Allow action, higher priority) for known safe patterns that the managed rules might flag.

4. **After baselining, switch to Block mode** for production protection:
   ```bash
   "OverrideAction": {"None": {}}  # Default managed rule action = Block
   ```

5. **Combine multiple managed rule groups** — SQL injection + Core Rule Set + Known Bad Inputs for defense-in-depth.

**WAF Actions Deep Dive:**

| Action | Behavior | When to Use |
|--------|----------|-------------|
| **Allow** | Pass request to origin | Default for all non-malicious traffic |
| **Block** | Return HTTP 403 | Active protection against known attacks |
| **Count** | Log only, pass anyway | Testing, baselining, detecting false positives |
| **CAPTCHA** | Challenge the user | Bot mitigation, suspicious-but-not-certain requests |
| **Challenge** | JavaScript challenge | Lightweight bot detection |

**Operational Efficiency Consideration:** Option D scores very low on operational efficiency because it requires:
- Writing, testing, and maintaining a Lambda function
- Configuring CloudWatch Logs subscription or S3 event triggers
- Managing an IP set that grows unboundedly
- Handling the race condition between attack detection and IP blocking

Option C requires none of this — it's a managed service with AWS-maintained rules.
