An enterprise company wants to allow its developers to purchase third-party software through AWS Marketplace. The company uses an AWS Organizations account structure with full features enabled, and has a shared services account in each organizational unit (OU) that will be used by procurement managers. The procurement team's policy indicates that developers should be able to obtain third-party software from an approved list only and use Private Marketplace in AWS  
Marketplace to achieve this requirement. The procurement team wants administration of Private Marketplace to be restricted to a role named procurement- manager-role, which could be assumed by procurement managers. Other IAM users, groups, roles, and account administrators in the company should be denied  
Private Marketplace administrative access.  
What is the MOST efficient way to design an architecture to meet these requirements?  

- A. Create an IAM role named procurement-manager-role in all AWS accounts in the organization. Add the PowerUserAccess managed policy to the role. Apply an inline policy to all IAM users and roles in every AWS account to deny permissions on the AWSPrivateMarketplaceAdminFullAccess managed policy.
- B. Create an IAM role named procurement-manager-role in all AWS accounts in the organization. Add the AdministratorAccess managed policy to the role. Define a permissions boundary with the AWSPrivateMarketplaceAdminFullAccess managed policy and attach it to all the developer roles.
- C. Create an IAM role named procurement-manager-role in all the shared services accounts in the organization. Add the AWSPrivateMarketplaceAdminFullAccess managed policy to the role. Create an organization root-level SCP to deny permissions to administer Private Marketplace to everyone except the role named procurement-manager-role. Create another organization root-level SCP to deny permissions to create an IAM role named procurement-manager-role to everyone in the organization.
- D. Create an IAM role named procurement-manager-role in all AWS accounts that will be used by developers. Add the AWSPrivateMarketplaceAdminFullAccess managed policy to the role. Create an SCP in Organizations to deny permissions to administer Private Marketplace to everyone except the role named procurement-manager-role. Apply the SCP to all the shared services accounts in the organization.

# Analysis for AWS Certified Solutions Architect - Professional (SAP-C02) Exam Question

Tags: #SAP-C02 #AWS_Organizations #AWS_Marketplace #Private_Marketplace #IAM #SCP #Security #Governance

## Understanding the Problem

The enterprise needs to centralize the administration of **AWS Private Marketplace** to a specific set of users, the procurement managers, who will use a dedicated role named **`procurement-manager-role`** in **shared services accounts**.

**Key Requirements:**

1. **Software Procurement:** Developers must purchase third-party software _only_ from an approved list using **AWS Private Marketplace**.
    
2. **Organization Structure:** The company uses **AWS Organizations** with full features enabled.
    
3. **Administration Location:** Administration must happen from a **shared services account** in each Organizational Unit (OU).
    
4. **Administrative Role:** A role named **`procurement-manager-role`** must be the _only_ entity allowed to administer Private Marketplace.
    
5. **Denial of Access:** All other IAM users, groups, roles, and account administrators in the entire organization must be **denied** Private Marketplace administrative access.
    
6. **Efficiency:** The solution must be the **MOST efficient** way to design the architecture.

## Services Explanation

|Service|Description|Role in Solution|
|---|---|---|

|**Service Control Policies (SCPs)**|Policies that offer central control over the maximum available permissions for all accounts in an organization. SCPs do **not** grant permissions; they are maximum permission guardrails.|Used as the **most efficient** way to enforce a denial of administrative access across the _entire organization_ except for the single approved role.|

## Correct Option(s) Explanation

### Correct Option: C

**C. Create an IAM role named procurement-manager-role in all the shared services accounts in the organization. Add the AWSPrivateMarketplaceAdminFullAccess managed policy to the role. Create an organization root-level SCP to deny permissions to administer Private Marketplace to everyone except the role named procurement-manager-role. Create another organization root-level SCP to deny permissions to create an IAM role named procurement-manager-role to everyone in the organization.**

This is the **MOST efficient** and architecturally sound solution because:

1. **Role Placement and Policy:** It correctly places the **`procurement-manager-role`** in the specified **shared services accounts** and grants it the necessary administrative permission (`AWSPrivateMarketplaceAdminFullAccess`).
    
2. **Organization-wide Denial (Efficiency and Security):** It leverages an **Organization root-level SCP** to enforce a **DENY** condition on Private Marketplace administration for _everyone_ by default. An SCP is the **most efficient** way to set a mandatory, non-overrideable guardrail across all current and future accounts in the organization.
    
3. **Exception to Denial:** The SCP uses a `NotPrincipal` element (or similar condition based on the role's ARN) to **EXCLUDE** the specific **`procurement-manager-role`** from the denial. This allows the procurement role to administer the service, while all other principals are denied.
    
4. **Preventing Policy Evasion (Security):** The second SCP, which denies the creation of the same-named role (`procurement-manager-role`), is a critical security step. Without it, a malicious user could create an identically named role in a different account, assume it, and bypass the first SCP's denial, as the SCP is written to allow _any_ principal with that name. **This two-pronged SCP approach is standard for securing administrative access at the Organization level.**
    

This solution efficiently meets all requirements, especially the need for organization-wide denial and restriction to a specific role, utilizing the top-down governance capabilities of AWS Organizations.

## Incorrect Options Explanation

### A. Create an IAM role named procurement-manager-role in all AWS accounts in the organization. Add the PowerUserAccess managed policy to the role. Apply an inline policy to all IAM users and roles in every AWS account to deny permissions on the AWSPrivateMarketplaceAdminFullAccess managed policy.

- **Inefficient/Impractical:** Applying an inline policy to _all IAM users and roles in every AWS account_ is manually intensive, error-prone, and cannot be centrally managed. It does not scale and would require constant updates for new users/roles/accounts.
    
- **Wrong Policy:** `PowerUserAccess` doesn't specifically grant the required administrative access to Private Marketplace, and it grants much more than needed. The correct policy is `AWSPrivateMarketplaceAdminFullAccess`.
    
- **Role Placement:** The question specifies the role should be in _shared services accounts_, not _all_ accounts.
    

### B. Create an IAM role named procurement-manager-role in all AWS accounts in the organization. Add the AdministratorAccess managed policy to the role. Define a permissions boundary with the AWSPrivateMarketplaceAdminFullAccess managed policy and attach it to all the developer roles.

- **Wrong Scope/Service:** A **Permissions Boundary** is a _secondary policy_ that sets the _maximum_ permissions for an IAM entity (user/role). It **does not deny** administrative access across the entire organization to _account administrators_ or _other roles/users_ who don't have a boundary applied. An account administrator with `AdministratorAccess` could still administer Private Marketplace.
    
- **Ineffective Denial:** This method only restricts the _developer roles_ and does not enforce the organization-wide denial required for _all_ other users, groups, roles, and administrators.
    
- **Wrong Role Policy:** `AdministratorAccess` grants full access, which is too broad, and the role placement is wrong (should only be in shared services accounts).
    

### D. Create an IAM role named procurement-manager-role in all AWS accounts that will be used by developers. Add the AWSPrivateMarketplaceAdminFullAccess managed policy to the role. Create an SCP in Organizations to deny permissions to administer Private Marketplace to everyone except the role named procurement-manager-role. Apply the SCP to all the shared services accounts in the organization.

- **Wrong Role Placement:** The question explicitly states the role should be in **shared services accounts**, but this option places it in _developer accounts_.
    
- **Wrong SCP Target:** The SCP is meant to enforce denial _organization-wide_, or at least across all accounts including developer, production, and other environments. Applying the SCP _only_ to the **shared services accounts** would mean that administrators in _developer_ accounts (or other OUs) are _not_ denied Private Marketplace administration, directly violating the requirement that "Other IAM users, groups, roles, and account administrators in the company should be **denied** Private Marketplace administrative access." The SCP must be applied higher up (e.g., at the Root) for maximum efficiency and coverage.
