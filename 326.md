A company is using AWS Control Tower to manage AWS accounts in an organization in AWS Organizations. The company has an OU that contains accounts. The company must prevent any new or existing Amazon EC2 instances in the OU's accounts from gaining a public IP address.

Which solution will meet these requirements?

A. Configure all instances in each account in the OU to use AWS Systems Manager. Use a Systems Manager Automation runbook to prevent public IP addresses from being attached to the instances.
B. Implement the AWS Control Tower proactive control to check whether instances in the OU's accounts have a public IP address. Set the AssociatePublicIpAddress property to False. Attach the proactive control to the OU.
C. Create an SCP that prevents the launch of instances that have a public IP address. Additionally, configure the SCP to prevent the attachment of a public IP address to existing instances. Attach the SCP to the OU.
D. Create an AWS Config custom rule that detects instances that have a public IP address. Configure a remediation action that uses an AWS Lambda function to detach the public IP addresses from the instances.



# AWS SAP-C02 Question Analysis

## 📋 Analysis

### 1. Understanding the Problem

The scenario focuses on centralized governance in a multi-account AWS environment managed by AWS Control Tower and AWS Organizations.

The core requirement is:
- Prevent new EC2 instances from receiving public IP addresses
- Prevent existing EC2 instances from later obtaining public IP addresses
- Apply the control consistently across all accounts inside an OU

The keyword here is "prevent". In AWS governance architecture, preventive controls are usually implemented with:
- Service Control Policies (SCPs)
- IAM explicit deny mechanisms
- Organization-level governance controls

The requirement is not asking for:
- Detection only
- Notification only
- Post-deployment remediation

This distinction is extremely important in SAP-C02 questions.

- --

### 2. Options Analysis

#### ❌ Option A (INCORRECT): Systems Manager Automation is reactive operational remediation

AWS Systems Manager Automation can execute remediation workflows, but this solution has several architectural problems.

Why it does not fully satisfy the requirement:
- It reacts after the resource already exists
- Public IP exposure may already occur before automation runs
- Existing EC2 instances could temporarily become publicly reachable
- Requires Systems Manager operational setup in every account

This is operational automation, not preventive governance.

From an enterprise security perspective:
- Regulated environments prefer blocking noncompliant actions before deployment
- Security baselines should not depend on remediation timing

AWS Systems Manager is excellent for:
- Patch management
- Operational workflows
- Fleet maintenance
- Incident remediation

But not ideal as the primary enforcement mechanism for organization-wide preventive network governance.

- --

#### ❌ Option B (INCORRECT): Control Tower proactive controls are not sufficient for complete prevention

AWS Control Tower proactive controls evaluate resources before provisioning, primarily through supported IaC workflows such as CloudFormation.

However, this option has several limitations:
- Existing instances are not comprehensively protected
- Direct API or CLI actions may not always be fully governed
- Proactive controls are narrower than SCP enforcement
- Setting `AssociatePublicIpAddress=False` alone does not prevent all future public exposure methods

The requirement explicitly includes:
- Existing resources
- Future modification prevention

SAP-C02 frequently tests this distinction:
- Proactive controls help enforce standards
- SCPs provide broader organization-wide preventive enforcement

Therefore, this option is incomplete.

- --

#### ✅ Option C (CORRECT): SCPs provide organization-wide preventive governance

This is the correct answer because Service Control Policies operate at the AWS Organizations level and enforce preventive restrictions across all member accounts in the OU.

An SCP can:
- Deny EC2 launches with public IP assignment
- Deny Elastic IP association actions
- Deny API calls that would expose instances publicly

This directly satisfies all requirements:
- Prevention of new public IP assignments
- Prevention of changes to existing instances
- Centralized governance across accounts

Why this is the best enterprise solution:
- Preventive rather than detective
- Centrally managed
- Consistent across accounts
- No dependency on agents or remediation execution
- Strong compliance alignment for regulated workloads

Typical APIs restricted by SCPs include:
- `ec2:RunInstances`
- `ec2:AssociateAddress`
- `ec2:ModifyNetworkInterfaceAttribute`

Real-world governance pattern:
- SCP = preventive enforcement
- AWS Config = compliance visibility
- Security Hub = centralized findings aggregation
- Control Tower = landing zone governance

Official AWS references:
- AWS Organizations SCP documentation
- AWS Control Tower controls documentation
- AWS Security Reference Architecture

- --

#### ❌ Option D (INCORRECT): AWS Config + Lambda is detective/remediative

AWS Config custom rules can detect EC2 instances with public IP addresses, and Lambda remediation can remove them afterward.

The issue is timing.

Why it fails the requirement:
- Public exposure already exists before remediation
- Detection is asynchronous
- Lambda execution may be delayed
- There is a compliance exposure window

This approach is suitable for:
- Drift detection
- Compliance reporting
- Operational remediation

But it is not a strict preventive security control.

SAP-C02 exam pattern:
- "Prevent" → SCP / IAM / preventive controls
- "Detect" → Config / Security Hub / CloudTrail
- "Remediate" → Lambda / Systems Manager Automation

Understanding these control categories is a high-frequency exam topic.

### 3. Hands-On Knowledge

Example SCP denying EC2 launches with public IP assignment:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyPublicIPLaunch",
      "Effect": "Deny",
      "Action": "ec2:RunInstances",
      "Resource": "*",
      "Condition": {
        "Bool": {
          "ec2:AssociatePublicIpAddress": "true"
        }
      }
    }
  ]
}
```

Enterprise architecture best practices:
- Deploy EC2 instances only in private subnets
- Use NAT Gateway for outbound internet access
- Use bastion alternatives such as Systems Manager Session Manager
- Centralize internet egress through Transit Gateway inspection VPCs

Common security architecture in medical and industrial environments:
- No public IPs on workloads
- PrivateLink for service connectivity
- VPC endpoints for AWS service access
- Centralized inspection using Network Firewall

Useful operational commands:

Check EC2 public IP assignments:
```bash
aws ec2 describe-instances \
  - -query "Reservations[*].Instances[*].[InstanceId,PublicIpAddress]"
```

List SCPs attached to an OU:
```bash
aws organizations list-policies-for-target \
  - -target-id ou-xxxx-example \
  - -filter SERVICE_CONTROL_POLICY
```

Important production caveat:
- SCP explicit deny affects even administrator roles inside member accounts
- Always validate SCPs in sandbox OUs before production rollout
- Incorrect SCP design can unintentionally block automation pipelines

**面试官可能追问 (Possible Follow-up Questions)**：

1. "SCP 和 IAM Policy 有什么区别？"
   → 回答要点 (Key Answer Points)：IAM policies grant permissions to identities within an account, while SCPs define the maximum allowed permissions for accounts in AWS Organizations. SCPs do not grant permissions themselves and explicit deny always overrides IAM allow.

2. "为什么很多企业禁止 EC2 使用公网 IP？"
   → 回答要点 (Key Answer Points)：Public IPs increase attack surface and internet exposure risk. Enterprises usually enforce private subnet architectures with centralized egress, NAT, VPN, Direct Connect, or Zero Trust access models for compliance and security control.
