A company needs to implement a patching process for its servers. The on-premises servers and Amazon EC2 instances use a variety of tools to perform patching.  
Management requires a single report showing the patch status of all the servers and instances.  
Which set of actions should a solutions architect take to meet these requirements?  

- A. Use AWS Systems Manager to manage patches on the on-premises servers and EC2 instances. Use Systems Manager to generate patch compliance reports
- B. Use AWS OpsWorks to manage patches on the on-premises servers and EC2 instances. Use Amazon QuickSight integration with OpsWorks to generate patch compliance reports.
- C. Use an Amazon EventBridge (Amazon CloudWatch Events) rule to apply patches by scheduling an AWS Systems Manager patch remediation job. Use Amazon Inspector to generate patch compliance reports.
- D. Use AWS OpsWorks to manage patches on the on-premises servers and EC2 instances. Use AWS X-Ray to post the patch status to AWS Systems Manager OpsCenter to generate patch compliance reports.

# AWS SAP-C02 Exam Question Analysis: Patching and Compliance Reporting

## Understanding the Problem

The company requires a **unified patching process** and a **single, consolidated report** for the patch status across its **Amazon EC2 instances** and **on-premises servers** (a hybrid environment). The current environment uses a variety of tools, which is inefficient. The solution must streamline patching and provide centralized compliance reporting.

---

## Services Explanation

### AWS Systems Manager (SSM)

**AWS Systems Manager (SSM)** provides a unified interface for operational tasks, including automation, compliance, and configuration management across AWS resources and on-premises servers.

* **SSM Patch Manager**: A feature of Systems Manager that automates the process of patching managed instances with security and other types of updates. It ensures instances are compliant with a patch baseline.
    * **Hybrid Environment Support**: SSM uses the **SSM Agent** (installed on EC2 and on-premises servers) and **AWS Systems Manager Activations** to manage on-premises servers, treating them as **managed nodes**.
    * **Compliance Reporting**: Patch Manager automatically generates **patch compliance data** that can be viewed directly in the Systems Manager console, providing a consolidated view of compliance across all managed nodes (EC2 and on-premises).

### AWS OpsWorks

**AWS OpsWorks** is a configuration management service that provides managed instances of Chef and Puppet. It allows users to use Chef and Puppet to automate how servers are configured, deployed, and managed, including patching. While it can handle patching, its primary focus is configuration management and application deployment, not holistic, single-pane-of-glass compliance reporting for hybrid environments like SSM Patch Manager.

### Amazon QuickSight

**Amazon QuickSight** is a cloud-native, serverless business intelligence (BI) service. It can be used to analyze data and create visualizations and reports, including reports for AWS services. However, it's a reporting *tool* and not the native compliance reporting *source* for OpsWorks patching.

### Amazon EventBridge (formerly CloudWatch Events)

**Amazon EventBridge** is a serverless event bus service that makes it easy to connect applications using data from your own applications, integrated Software-as-a-Service (SaaS) applications, and AWS services. It is often used to trigger actions based on schedules or events, such as triggering an SSM Run Command or Patch Manager operation on a schedule. It is part of the solution for *scheduling* patching but doesn't handle the reporting aspect.

### Amazon Inspector

**Amazon Inspector** is an automated security assessment service that helps improve the security and compliance of applications deployed on AWS. It assesses EC2 instances for vulnerabilities and deviations from best practices. **It focuses on vulnerability assessment and security best practices, not specifically on the compliance status of patches applied by tools like Systems Manager Patch Manager.**

### AWS X-Ray

**AWS X-Ray** is a service that helps developers analyze and debug distributed applications, such as those built using microservices. It provides an end-to-end view of requests as they travel through the application, showing component performance. It has no direct role in patch management or compliance reporting.

### AWS Systems Manager OpsCenter

**AWS Systems Manager OpsCenter** provides a central location where operations engineers and IT professionals can view, investigate, and resolve operational issues (OpsItems) related to AWS resources. While patch compliance issues could turn into OpsItems, it is not the primary service for generating consolidated patch compliance *reports*.

---

## Correct Option(s) Explanation

### A. Use AWS Systems Manager to manage patches on the on-premises servers and EC2 instances. Use Systems Manager to generate patch compliance reports.

This option is the **correct** solution because it directly addresses all the requirements using the purpose-built AWS service:

1.  **Single Patch Management Tool (AWS Systems Manager)**: Systems Manager, specifically **Patch Manager**, is designed to centrally manage patching for both **Amazon EC2 instances** and **on-premises servers** (managed nodes). This eliminates the need for a variety of tools.
2.  **Single Compliance Report (Systems Manager)**: Patch Manager automatically collects and stores patch compliance data for all managed nodes and presents it in a **consolidated view** within the Systems Manager console. This fulfills the requirement for a single report.

This approach aligns with AWS best practices for hybrid environment operations and is extensively documented in the [AWS Systems Manager documentation](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-patch.html) regarding **Patch Manager** and [Managing On-Premises Servers](https://docs.aws.amazon.com/systems-manager/latest/userguide/systems-manager-managed-instances-on-premises.html).

## Incorrect Options Explanation

### B. Use AWS OpsWorks to manage patches on the on-premises servers and EC2 instances. Use Amazon QuickSight integration with OpsWorks to generate patch compliance reports.

* **OpsWorks** *can* manage patching using Chef/Puppet, but it's not the primary, native, or most straightforward AWS service for *unified patch compliance reporting* in a hybrid environment compared to SSM Patch Manager.
* **QuickSight** is a BI tool. While it *could* be used, generating reports via QuickSight requires more setup (data source configuration, dashboard creation) than the native, built-in compliance reporting provided by Systems Manager Patch Manager, making it overly complex and not the optimal solution.

### C. Use an Amazon EventBridge (Amazon CloudWatch Events) rule to apply patches by scheduling an AWS Systems Manager patch remediation job. Use Amazon Inspector to generate patch compliance reports.

* The first part of the action, **EventBridge scheduling a Systems Manager job**, is a valid and common architectural pattern for *scheduling* the patching process.
* The second part is incorrect: **Amazon Inspector** focuses on **vulnerability assessment and security best practices**, not on generating **patch compliance reports** that show the status of patches applied by Systems Manager Patch Manager. Systems Manager itself generates these reports.

### D. Use AWS OpsWorks to manage patches on the on-premises servers and EC2 instances. Use AWS X-Ray to post the patch status to AWS Systems Manager OpsCenter to generate patch compliance reports.

* **OpsWorks** is sub-optimal for unified compliance reporting compared to SSM (as noted in B).
* **AWS X-Ray** is a tracing and debugging service and has no role in managing or reporting patch compliance status.
* **OpsCenter** is for managing operational issues (OpsItems), not generating comprehensive patch compliance reports. This is a mix of services used outside their intended scope for this requirement.
