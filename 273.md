A North American company with headquarters on the East Coast is deploying a new web application running on Amazon EC2 in the us-east-1 Region. The application should dynamically scale to meet user demand and maintain resiliency. Additionally, the application must have disaster recovery capabilities in an active-passive configuration with the us-west-1 Region.

## Understanding the Problem
You have a web application on **Amazon EC2** in **us-east-1** that must:
- **Dynamically scale** with demand → needs **Auto Scaling** behind a **load balancer** across **multiple AZs**.
- Maintain **resiliency** within the primary Region →-AZ design.
- Provide **disaster recovery (DR) active-passive** with **us-west-1** → a secondary Region that is *not serving traffic normally* but take over if the primary fails.
- Provide **traffic failover** between Regions → typically **Amazon Route 53 failover routing** with **health checks** to the primary endpoint.

In an active-passive multi-Region pattern, you generally deploy the *same stack* in both Regions, but Route 53 routes traffic to the only when health checks indicate the primary is unhealthy.

## Correct Option(s) Explanation
### ✅ **B is correct**
It describes the standard, AWS-recommended building blocks for **multi-AZ + multi-Region active-passive**:

1. **ALB across multiple AZs** in **us-east-1**  
   - This gives **in-Region high availability** and spreads traffic across AZs.
   - ALB is a **Regional** service and is designed to span **multiple AZs within one**.

2. **EC2 Auto Scaling group across multiple AZs** behind the ALB in **us-east-1**  
   - Auto Scaling provides **dynamic** and instance replacement.

3. **Deploy the same solution in us-west-1**  
   - This creates the **passive** DR stack (often warm standby or pilot light depending on capacity).

4. **Route 53 failover routing policy + health checks**  
   - Route 53 can be configured with **PRIMARY** (us-east-1 ALB) and **SECONDARY** (us-west-1 ALB) records.
   - If Route 53 health checks fail for the primary endpoint, Route 53 returns the secondary record.
   - This matches the requirement: **active-passive DR**.

* *AWS docs to anchor this:**
- Route 53 failover routing: *Amazon Route 53 Developer Guide Routing policies (Failover)*  
- ALB scope: *Elastic Load Balancing User Guide – ALB is Regional and supports multiple AZs*
- Auto Scaling multi-AZ: *Amazon EC2 Auto Scaling User Guide – Auto Scaling groups and Availability Zones*

## Incorrect Options Explanation

### ❌ **A is incorrect**
Key issues:
- **Auto Scaling group spanning both VPCs and Regions** is not supported.  
  EC2 Auto Scaling groups are **Regional** (they can span multiple AZs *within the same Region*, not multiple Regions).
- **VPC peering** is unnecessary for this DR goal and doesn’t solve cross-Region failover routing.
- Active-passive DR is typically handled at **DNS (Route 53)** or via **Global Accelerator**, not by trying to stretch compute groups across Regions.

### ❌ **C is incorrect**
Key issues:
- **An ALB cannot span multiple VPCs and cannot span Regions.**  
  An ALB is deployed **into subnets in one VPC** and is **Regional**.
- Inter-Region VPC peering does not enable a “single ALB across both VPCs.” That architecture is not supported.

### ❌ **D is incorrect**
It’s close, but it fails to clearly meet the explicit requirement:
- It says “use Route 53 health checks to provide high availability across both Regions,” but does **not specify a failover routing policy** (active-passive).  
- “Separate Route 53 records” could imply other routing behaviors (weighted, latency-based, etc.). **active-passive**, the expected explicit design is **Route 53 Failover policy** with primary/secondary endpoints.

In SAP-C02, when the requirement explicitly calls out **active-passive**, AWS expects the **failover routing policy** wording/design.

## Hands-on Enterprise Knowledge (Medical / Industrial Readiness)
For regulated or mission-critical environments, the compute layer is only part of DR. In real implementations you also need:

1. **Deterministic failover behavior**
   - Route 53 failover relies on **DNS caching/TTL**. Set an appropriate TTL (often 30–60 for critical apps), but balance against resolver behavior and query volume.
   - Health checks should test **application health**, not just TCP/HTTP 200 from the load balancer. Consider `/health` endpoints and dependency checks.

2. **Data layer DR planning (often the real hard part)**
   - Ensure stateful components have cross-Region (database, object storage, secrets/config).
   - Runbooks for **data consistency** and **RPO/RTO** targets should be documented and tested.

3. **Operational discipline**
   - Use infrastructure-as-code so east/west stacks remain consistent.
   - Regular DR game days: validate Route 53 failover, scale-up behavior in the passive Region, and rollback procedures.
   - Audit evidence: change management, DR test results, and monitoring alarms (typical in medical/industrial compliance).

4. **Capacity strategy for passive Region**
   - If “active-passive” means warm standby, keep minimal instances running and scale out on failover (Auto Scaling desired capacity adjustments can be automated).
   - Make sure quotas (EC2, EIP, ALB, etc.) are pre-approved in both Regions.
