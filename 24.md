A company is in the process of implementing AWS Organizations to constrain its developers to use only Amazon EC2, Amazon S3, and Amazon DynamoDB. The  
Developers account resides in a dedicated organizational unit (OU). The Solutions Architect has implemented the following SCP on the Developers account:  
![](https://www.examtopics.com/assets/media/exam-media/04241/0037900001.png)  
When this policy is deployed, IAM users in the Developers account are still able to use AWS services that are not listed in the policy.  
What should the Solutions Architect do to eliminate the Developers' ability to use services outside the scope of this policy?  

- A. Create an explicit deny statement for each AWS service that should be constrained.
- B. Remove the FullAWSAccess SCP from the Developer account's OU.
- C. Modify the FullAWSAccess SCP to explicitly deny all services.
- D. Add an explicit deny statement using a wildcard to the end of the SCP.

An AWS Certified Solutions Architect - Professional (SAP-C02) candidate should be able to solve this problem by applying their knowledge of **Service Control Policies (SCPs)** within **AWS Organizations** and understanding the default behavior and inheritance of these policies.

The correct answer is **B. Remove the FullAWSAccess SCP from the Developer account's OU.**

---

## Analysis

### Understanding the Problem

The company wants to restrict IAM users in the **Developers account** (which is within a dedicated **Organizational Unit (OU)**) to only be able to use **Amazon EC2**, **Amazon S3**, and **Amazon DynamoDB**. A **Service Control Policy (SCP)** has been created with `Allow` statements for these three services. However, despite this SCP, developers can still use other AWS services.

The core issue lies in the **inheritance and evaluation logic of SCPs** in AWS Organizations. By default, when an OU or account is created, it inherits the `FullAWSAccess` SCP, which explicitly allows all services. An `Allow` in an SCP can never override an `Explicit Deny`, but an `Allow` in an SCP **can be overridden by a more restrictive `Allow` or a missing `Allow`** at a higher level in the hierarchy, _or_ by an `Explicit Deny` at any level.

Crucially, SCPs only filter the **maximum available permissions**; they do not grant permissions. The IAM user must still have an IAM policy that grants the permission. In this case, the custom SCP is an "allow-list" that allows only the desired services. To make this "allow-list" effective, all other services must be implicitly denied, which happens only if the **default `FullAWSAccess` SCP is removed or replaced**.

**_Official AWS Documentation Reference:_**

- _AWS Organizations Documentation: Understanding Policy Inheritance and SCPs_ states: "**SCPs do not grant permissions.** For an IAM user or role to do anything, they must still have an IAM policy that grants them the permission. **SCPs act as a filter** that specifies the maximum permissions for all users and roles in affected accounts."
    
- The default `FullAWSAccess` SCP is:
    
    JSON
    
    ```
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": "*",
          "Resource": "*"
        }
      ]
    }
    ```
    
    This policy, when active, allows all services, overriding the implicit deny of the custom SCP.
    

### Correct Option(s) Explanation

- **B. Remove the FullAWSAccess SCP from the Developer account's OU.**
    
    - **Reasoning:** Since the company's custom SCP only has `Allow` statements for EC2, S3, and DynamoDB, it _implicitly denies_ all other services. However, this implicit deny is currently being overridden by the **default `FullAWSAccess` SCP** which explicitly allows all actions (`"Action": "*"`). SCP evaluation is an **"AND"** operation across the hierarchy (Root → OU → Account). For an action to be allowed, it must be allowed by the IAM policy _and_ every applicable SCP in the hierarchy.
        
    - By **removing the permissive `FullAWSAccess` SCP** (or detaching it from the OU), the custom SCP with only the desired `Allow` statements becomes the effective policy. All services **not** listed in the custom SCP will then be **implicitly denied** because they are not explicitly allowed by _any_ attached SCP. This satisfies the requirement to constrain the developers to only the listed services.
        

### Incorrect Options Explanation

- **A. Create an explicit deny statement for each AWS service that should be constrained.**
    
    - **Reasoning:** While an explicit deny is always an effective constraint (Explicit Deny → Explicit Allow → Implicit Deny), creating an explicit deny for _every_ other AWS service is operationally infeasible and error-prone. The goal is to enforce an "allow-list," and the proper AWS Organizations mechanism for an allow-list is to remove the overarching `Allow` (i.e., `FullAWSAccess`) and use a custom SCP with only the desired `Allow` actions.
        
- **C. Modify the FullAWSAccess SCP to explicitly deny all services.**
    
    - **Reasoning:** SCPs are inherited. If the `FullAWSAccess` SCP is modified to deny all services, it would affect **all** accounts and OUs in the entire organization (unless it's attached only to the Developer OU, which is unlikely for the default policy). This would be an extreme, organization-wide change that would likely break _all_ other accounts and is not the targeted solution for a single account/OU. The correct action is to _remove_ the policy, not drastically modify it.
        
- **D. Add an explicit deny statement using a wildcard to the end of the SCP.**
    
    - **Reasoning:** Adding a statement like `"Effect": "Deny", "Action": "*", "Resource": "*"` to the _end_ of the custom SCP would **explicitly deny all actions for all services**. Since an explicit deny _always_ takes precedence, this statement would override the preceding `Allow` statements for EC2, S3, and DynamoDB, thereby preventing the developers from using **any** service, including the ones they are supposed to use. This is overly restrictive and defeats the purpose of the custom SCP. The custom SCP itself is correct for an allow-list; the issue is the presence of the default, permissive SCP.
