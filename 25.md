A company is hosting a monolithic REST-based API for a mobile app on five Amazon EC2 instances in public subnets of a VPC. Mobile clients connect to the API by using a domain name that is hosted on Amazon Route 53. The company has created a Route 53 multivalue answer routing policy with the IP addresses of all the EC2 instances. Recently, the app has been overwhelmed by large and sudden increases to traffic. The app has not been able to keep up with the traffic.  
A solutions architect needs to implement a solution so that the app can handle the new and varying load.  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Separate the API into individual AWS Lambda functions. Configure an Amazon API Gateway REST API with Lambda integration for the backend. Update the Route 53 record to point to the API Gateway API.
- B. Containerize the API logic. Create an Amazon Elastic Kubernetes Service (Amazon EKS) cluster. Run the containers in the cluster by using Amazon EC2. Create a Kubernetes ingress. Update the Route 53 record to point to the Kubernetes ingress.
- C. Create an Auto Scaling group. Place all the EC2 instances in the Auto Scaling group. Configure the Auto Scaling group to perform scaling actions that are based on CPU utilization. Create an AWS Lambda function that reacts to Auto Scaling group changes and updates the Route 53 record.
- D. Create an Application Load Balancer (ALB) in front of the API. Move the EC2 instances to private subnets in the VPC. Add the EC2 instances as targets for the ALB. Update the Route 53 record to point to the ALB.

### ✅ Correct Option

**D. Create an Application Load Balancer (ALB) in front of the API. Move the EC2 instances to private subnets in the VPC. Add the EC2 instances as targets for the ALB. Update the Route 53 record to point to the ALB.**

### 🧠 Analysis for Learning

#### Understanding the Problem

The company is experiencing performance issues and is unable to handle sudden, large increases in traffic to its monolithic REST-based API, currently hosted on five **Amazon EC2 instances**. The setup uses **Amazon Route 53** with a **multivalue answer routing policy** pointing directly to the EC2 instance IPs. This setup has **no built-in mechanism for automatic scaling, health checks, or efficient load distribution**, leading to the current overload.

The requirement is to implement a solution that allows the app to handle **new and varying load** (i.e., **scalability**) with the **LEAST operational overhead**.

#### Correct Option Explanation

**D. Create an Application Load Balancer (ALB) in front of the API. Move the EC2 instances to private subnets in the VPC. Add the EC2 instances as targets for the ALB. Update the Route 53 record to point to the ALB.**

1. **Application Load Balancer (ALB):** The ALB will distribute incoming traffic efficiently across the EC2 instances (targets). It performs **automatic health checks** to ensure traffic only goes to healthy instances, providing high availability. The ALB itself is **highly scalable and resilient** by design (managed service).
    
2. **Move EC2 to Private Subnets:** This is a security best practice. The instances hosting the application logic should not be directly accessible from the internet. The **ALB acts as the single public entry point**.
    
3. **Update Route 53:** Changing the Route 53 record to point to the ALB (using an **Alias record** which is highly recommended for AWS resources like ALB) removes the need for the problematic multivalue answer policy.
    

**Operational Overhead:** This solution requires the least operational overhead for a few reasons:

- It utilizes the **existing EC2-based monolithic application** with **minimal architectural change** (no code rewrite, no containerization).
    
- **ALB is a managed service**; AWS handles its scaling, availability, and patching.
    
- While not explicitly mentioned, an **Auto Scaling Group** is the natural next step, and integrating an ASG with an ALB is a **standard, low-overhead pattern** that completes the scalability requirement. The ALB/ASG combination is the most straightforward, low-effort path to horizontal scaling for existing EC2 applications.
    

#### Incorrect Options Explanation

- **A. Separate the API into individual AWS Lambda functions. Configure an Amazon API Gateway REST API with Lambda integration for the backend. Update the Route 53 record to point to the API Gateway API.**
    
    - **Reason for Incorrectness (Operational Overhead):** This requires a complete **re-architecting and rewriting of the monolithic API** into serverless functions (**Lambda**). This is a massive effort with high development overhead, even though the resulting operational overhead would be low _after_ deployment. The prompt specifies a solution with the **LEAST operational overhead**, which implies minimal change to the existing application.
        
- **B. Containerize the API logic. Create an Amazon Elastic Kubernetes Service (Amazon EKS) cluster. Run the containers in the cluster by using Amazon EC2. Create a Kubernetes ingress. Update the Route 53 record to point to the Kubernetes ingress.**
    
    - **Reason for Incorrectness (Operational Overhead):** This requires **containerization** of the application (moderate effort) and the setup and **ongoing management of an Amazon EKS cluster** (high operational overhead). EKS is a complex orchestration platform that introduces significant overhead (managing control plane, worker nodes, networking, ingress) compared to a simple ALB/ASG setup for a monolithic REST API.
        
- **C. Create an Auto Scaling group. Place all the EC2 instances in the Auto Scaling group. Configure the Auto Scaling group to perform scaling actions that are based on CPU utilization. Create an AWS Lambda function that reacts to Auto Scaling group changes and updates the Route 53 record.**
    
    - **Reason for Incorrectness (Complexity/Inefficiency):** While an **Auto Scaling Group (ASG)** is essential for the scaling requirement, this option proposes a highly inefficient and high-overhead method of load balancing.
        
        - **Custom Lambda for Route 53:** Relying on a custom **Lambda function to manually update the Route 53 multivalue record** when instances scale in/out is a complex, error-prone, and high-overhead operational task.
            
        - **No Centralized Load Balancing:** Using Route 53 multivalue answer routing policy for load balancing is poor practice as it **lacks health checks (beyond basic DNS checks)** and **doesn't distribute load evenly or based on instance load** like a proper load balancer.
            
        - The **standard, low-overhead pattern** is to use an **ALB** in front of the ASG, not a custom Lambda function and a problematic Route 53 policy.
