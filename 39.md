A company is planning to host a web application on AWS and wants to load balance the traffic across a group of Amazon EC2 instances. One of the security requirements is to enable end-to-end encryption in transit between the client and the web server.  
  
Which solution will meet this requirement?

- A. Place the EC2 instances behind an Application Load Balancer (ALB). Provision an SSL certificate using AWS Certificate Manager (ACM), and associate the SSL certificate with the ALB. Export the SSL certificate and install it on each EC2 instance. Configure the ALB to listen on port 443 and to forward traffic to port 443 on the instances.
- B. Associate the EC2 instances with a target group. Provision an SSL certificate using AWS Certificate Manager (ACM). Create an Amazon CloudFront distribution and configure it to use the SSL certificate. Set CloudFront to use the target group as the origin server.
- C. Place the EC2 instances behind an Application Load Balancer (ALB) Provision an SSL certificate using AWS Certificate Manager (ACM), and associate the SSL certificate with the ALB. Provision a third-party SSL certificate and install it on each EC2 instance. Configure the ALB to listen on port 443 and to forward traffic to port 443 on the instances.
- D. Place the EC2 instances behind a Network Load Balancer (NLB). Provision a third-party SSL certificate and install it on the NLB and on each EC2 instance. Configure the NLB to listen on port 443 and to forward traffic to port 443 on the instances.

---

## Analysis

### Understanding the Problem

The requirement is to host a web application on AWS using Amazon EC2 instances, load balance the traffic, and ensure **end-to-end encryption in transit** between the client and the web server. End-to-end encryption means the traffic remains encrypted from the client to the load balancer, and then is _re-encrypted_ or _kept encrypted_ from the load balancer to the backend web server (EC2 instance). This typically involves using **HTTPS** (SSL/TLS) at both the frontend (client to load balancer) and the backend (load balancer to server).

### Services Explanation

|**Service**|**Description**|**Role in Solution**|
|---|---|---|
|**Application Load Balancer (ALB)**|Operates at the application layer (Layer 7 of the OSI model). Supports path-based, host-based, and query-string-based routing. Crucially, it supports **SSL/TLS termination** (decrypting client traffic) and **re-encryption** (establishing a new encrypted connection to the target).|Used for load balancing and handling the SSL/TLS offload and re-encryption to meet the end-to-end encryption requirement.|
|**AWS Certificate Manager (ACM)**|A service that lets you easily provision, manage, and deploy public and private SSL/TLS certificates for use with AWS services (like ALB, CloudFront).|Used to provision the SSL/TLS certificate for the frontend connection (client to ALB).|
|**Amazon EC2 (Elastic Compute Cloud)**|Provides resizable compute capacity in the cloud.|Used to host the web application. Requires an installed SSL/TLS certificate to terminate the encrypted connection from the ALB.|
|**Amazon CloudFront**|A fast content delivery network (CDN) service that securely delivers data, videos, applications, and APIs globally with low latency.|Primarily for caching and performance, but can also terminate SSL/TLS.|
|**Network Load Balancer (NLB)**|Operates at the transport layer (Layer 4 of the OSI model). It can handle millions of requests per second and is typically used for extreme performance or static IP needs. **NLB handles encryption/decryption differently**, often performing **passthrough** unless specifically configured for termination.|While it can be configured for TLS termination, the standard and most cost-effective solution for end-to-end encryption with L7 features is the ALB.|

---

### Correct Option(s) Explanation

**C. Place the EC2 instances behind an Application Load Balancer (ALB) Provision an SSL certificate using AWS Certificate Manager (ACM), and associate the SSL certificate with the ALB. Provision a third-party SSL certificate and install it on each EC2 instance. Configure the ALB to listen on port 443 and to forward traffic to port 443 on the instances.**

1. **ALB and ACM:** Using an **ALB** and an **ACM-provisioned certificate** on the ALB's listener on port 443 handles the first half of the encryption: **Client $\rightarrow$ ALB (Encryption)**. This is a standard, low-maintenance, and cost-effective method on AWS.
    
2. **End-to-End Encryption:** The key to end-to-end encryption in this pattern is that the ALB must forward the traffic encrypted to the EC2 instances. By configuring the ALB to **forward traffic to port 443 on the instances**, the ALB establishes a new, secure connection to the backend server. The configuration would use the **HTTPS protocol** for the target group (target protocol set to HTTPS).
    
3. **Backend Certificate:** For the ALB's connection to the instance to be secure (HTTPS), the EC2 instance must have an **installed SSL/TLS certificate** to terminate the connection. While ACM certificates can be used with some integrated services (like ALB, CloudFront), they **cannot be directly downloaded** and installed on an EC2 instance (a reason why Option A is incorrect). Therefore, a **third-party or self-signed certificate** must be installed on the EC2 instances.
    

This configuration meets the requirement for end-to-end encryption (Client → ALB encrypted; ALB → EC2 encrypted).

(Reference: AWS Documentation - Application Load Balancer - HTTPS listeners and Target Group configuration.)

### Incorrect Options Explanation

**A. Place the EC2 instances behind an Application Load Balancer (ALB). Provision an SSL certificate using AWS Certificate Manager (ACM), and associate the SSL certificate with the ALB. Export the SSL certificate and install it on each EC2 instance. Configure the ALB to listen on port 443 and to forward traffic to port 443 on the instances.**

- **Incorrect Point:** **You cannot export (download) certificates provisioned by AWS Certificate Manager (ACM)** for use outside of integrated AWS services like ALB or CloudFront. ACM certificates are managed and deployed only within the AWS environment. The backend EC2 instances require a downloadable certificate.
    

**B. Associate the EC2 instances with a target group. Provision an SSL certificate using AWS Certificate Manager (ACM). Create an Amazon CloudFront distribution and configure it to use the SSL certificate. Set CloudFront to use the target group as the origin server.**

- **Incorrect Point:** This solution focuses on using CloudFront, which is usually for content delivery and acceleration. More importantly, it **does not explicitly mention setting up encryption between the CloudFront origin (the target group/ALB) and the EC2 instances.** The default setup between CloudFront and a non-S3 origin is often HTTP unless specified otherwise, and the configuration here is incomplete for end-to-end encryption (client $\rightarrow$ CloudFront $\rightarrow$ Load Balancer/Target $\rightarrow$ EC2). Even if CloudFront is used, an ALB is typically needed behind it for routing to EC2, making this an overly complex and potentially incomplete solution for the core problem.
    

**D. Place the EC2 instances behind a Network Load Balancer (NLB). Provision a third-party SSL certificate and install it on the NLB and on each EC2 instance. Configure the NLB to listen on port 443 and to forward traffic to port 443 on the instances.**

- **Incorrect Point:** While the NLB _can_ be configured for TLS termination (installing a certificate on the NLB and listening on 443), the question implies a standard web application which is best suited for an **Application Load Balancer (ALB)** due to its Layer 7 (application) features like path-based routing. Crucially, the preferred and most cost-effective method for handling SSL on the frontend is using **ACM with an ALB**, not installing a third-party certificate directly on the NLB (though possible). Furthermore, if the NLB is configured for **TLS Passthrough** (the default high-performance mode for NLB when targets are on 443), the NLB **does not decrypt the traffic**, and the certificate would only be needed on the EC2 instance, not the NLB itself. This option uses a non-optimal load balancer type and an unnecessary configuration step (installing a certificate on the NLB when ALB/ACM is better).
