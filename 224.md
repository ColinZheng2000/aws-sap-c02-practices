A company manages hundreds of AWS accounts centrally in an organization in AWS Organizations. The company recently started to allow product teams to create and manage their own S3 access points in their accounts. The S3 access points can be accessed only within VPCs, not on the Internet.
What is the MOST operationally efficient way to enforce this requirement?

A. Set the S3 access point resource policy to deny the s3:CreateAccessPoint action unless the s3:AccessPointNetworkOrigin condition key evaluates to VPC.
B. Create an SCP at the root level in the organization to deny the s3:CreateAccessPoint action unless the s3:AccessPointNetworkOrigin condition key evaluates to VPC.
C. Use AWS CloudFormation StackSets to create a new IAM policy in each AWS account that allows the s3:CreateAccessPoint action only if the s3:AccessPointNetworkOrigin condition key evaluates to VPC.
D. Set the S3 bucket policy to deny the s3:CreateAccessPoint action unless the s3:AccessPointNetworkOrigin condition key evaluates to VPC.


Suggest Answer: B

---

## 📊 Analysis

### 🔍 Understanding the Problem

A company manages **hundreds of AWS accounts** centrally using **AWS Organizations**. Product teams are now allowed to create and manage their own **S3 Access Points** in their respective accounts. However, there is a security requirement: S3 Access Points must be accessible **only within VPCs** — never from the public Internet. The key question is: what is the **MOST operationally efficient** way to enforce this restriction across an organization with hundreds of accounts?

The "most operationally efficient" qualifier points toward centralized governance mechanisms that scale automatically across all accounts without per-account configuration.

### ✅ Correct Option: B — Create an SCP at the Root Level to Deny s3:CreateAccessPoint Unless the s3:AccessPointNetworkOrigin Condition Key Evaluates to VPC

**Service Control Policies (SCPs)** are the central governance mechanism in AWS Organizations. They act as **preventive guardrails** that define the maximum permissions for accounts in the organization. Here's why this is the best answer:

- **Centralized Governance**: An SCP attached to the **root** of the organization automatically applies to ALL existing accounts and ANY future accounts added to the organization. One policy, universal coverage.
- **Preventive Control**: The SCP denies the `s3:CreateAccessPoint` action **before it happens** unless the `s3:AccessPointNetworkOrigin` condition key is set to `VPC`. This means non-compliant access points cannot even be created — the most effective form of enforcement.
- **Cannot Be Overridden**: Even if an account administrator has full IAM permissions (`s3:*`), the SCP acts as a boundary. No IAM principal in any child account can perform a denied action, period.
- **Zero Operational Overhead**: After creating the SCP once, there is nothing to manage, update, or patch per account. New accounts automatically inherit the guardrail.
- **Immutable to Account Administrators**: Product teams cannot modify or delete the SCP, ensuring persistent compliance.

The SCP would look conceptually like:
```json
{
  "Effect": "Deny",
  "Action": "s3:CreateAccessPoint",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "s3:AccessPointNetworkOrigin": "VPC"
    }
  }
}
```

### ❌ Incorrect Options

**A. Set the S3 access point resource policy to deny s3:CreateAccessPoint unless the s3:AccessPointNetworkOrigin condition key evaluates to VPC**
- **Architectural misunderstanding**: An **S3 Access Point resource policy** controls access to the access point AFTER it is created — it does NOT control the `s3:CreateAccessPoint` API action itself
- Even if you could use an access point policy to restrict creation (which you cannot), you would need to set this on **every single access point in every single account** — this is the opposite of "operationally efficient"
- Each product team would need to remember to set this policy correctly every time they create an access point, which is error-prone and not enforceable

**C. Use AWS CloudFormation StackSets to create a new IAM policy in each AWS account that allows s3:CreateAccessPoint only if the s3:AccessPointNetworkOrigin condition key evaluates to VPC**
- **IAM policies are not enforceable boundaries**: An IAM policy that "allows" an action with conditions can be circumvented if a user has another policy (or a more permissive policy) that allows `s3:CreateAccessPoint` without the condition. IAM evaluates policies with an "allow if any policy allows" model
- **IAM policies are not preventive**: Unlike SCPs, IAM policies are evaluated at the principal level and can be bypassed by creating additional IAM policies or roles
- **Operational complexity**: Managing StackSets across hundreds of accounts introduces maintenance overhead — you must ensure StackSets are deployed, updated, and haven't drifted
- **Product teams could create additional IAM policies** that grant `s3:CreateAccessPoint` without the VPC restriction

**D. Set the S3 bucket policy to deny s3:CreateAccessPoint unless the s3:AccessPointNetworkOrigin condition key evaluates to VPC**
- **Scope issue**: An S3 **bucket policy** governs access to objects within a specific bucket, not the creation of access points
- `s3:CreateAccessPoint` is a **control plane API** that operates at the S3 service level, not on a specific bucket. You cannot use a bucket policy to control who can create access points
- Even conceptually, you would need to apply this to every bucket across hundreds of accounts — not operationally efficient

### 🏭 Enterprise Hands-On Knowledge

1. **SCP Hierarchy Strategy**: The enterprise best practice is to organize OUs hierarchically:
   - **Root SCP**: Broad guardrails (like this VPC-only access point rule)
   - **OU-level SCPs**: Department-specific restrictions
   - **Account-level SCPs**: Rare — only for exceptions
   - Never attach SCPs directly to individual accounts for general policies

2. **SCP Evaluation Logic**: SCPs do NOT grant permissions — they only **remove** permissions. The effective permissions for a principal = (IAM policies ALLOW) **intersected with** (SCPs do NOT deny). This is why SCPs are superior to IAM policies for governance — they cannot be "allowed" around.

3. **S3 Access Points Network Origin Options**:
   - `VPC`: Access point is only accessible from within the specified VPC
   - `Internet`: Access point is accessible from the Internet (and VPCs with proper routing)
   - Always configure `Block Public Access` settings on access points for defense-in-depth

4. **Combined Governance Approach**: For defense in depth in a large enterprise:
   - **SCP** (Layer 1): Prevent creation of non-compliant access points organization-wide
   - **AWS Config Rule** (Layer 2): Detect any access points created before the SCP was applied, or any drift
   - **VPC Endpoint Policies** (Layer 3): Control which access points can be accessed from specific VPCs
   - **CloudTrail** (Layer 4): Audit all `CreateAccessPoint` API calls for compliance monitoring

5. **Testing SCPs**: Before applying an SCP to the root, test it on a **non-production OU** first. Use AWS Organizations' **policy simulation** or IAM Policy Simulator to verify the SCP behaves as expected. SCPs can take several minutes to propagate to all accounts and regions.

6. **S3 Access Points and VPC**: When creating a VPC-only access point, you must specify the VPC ID. The access point then gets a dedicated hostname that resolves to a private IP within the VPC. Traffic never traverses the public Internet. Ensure the VPC has the necessary **Gateway VPC Endpoint** or **Interface VPC Endpoint** for S3.
