A company recently completed the migration from an on-premises data center to the AWS Cloud by using a replatforming strategy. One of the migrated servers is running a legacy Simple Mail Transfer Protocol (SMTP) service that a critical application relies upon. The application sends outbound email messages to the company’s customers. The legacy SMTP server does not support TLS encryption and uses TCP port 25. The application can use SMTP only.  
  
The company decides to use Amazon Simple Email Service (Amazon SES) and to decommission the legacy SMTP server. The company has created and validated the SES domain. The company has lifted the SES limits.  
  
What should the company do to modify the application to send email messages from Amazon SES?

- A. Configure the application to connect to Amazon SES by using TLS Wrapper. Create an IAM role that has ses:SendEmail and ses:SendRawEmail permissions. Attach the IAM role to an Amazon EC2 instance.
- B. Configure the application to connect to Amazon SES by using STARTTLS. Obtain Amazon SES SMTP credentials. Use the credentials to authenticate with Amazon SES.
- C. Configure the application to use the SES API to send email messages. Create an IAM role that has ses:SendEmail and ses:SendRawEmail permissions. Use the IAM role as a service role for Amazon SES.
- D. Configure the application to use AWS SDKs to send email messages. Create an IAM user for Amazon SES. Generate API access keys. Use the access keys to authenticate with Amazon SES.

## 📚 Analysis

### Understanding the Problem

The company is migrating a critical application that relies on a **legacy SMTP server** for outbound customer emails. The application _must_ continue using **SMTP** and the legacy server **does not support TLS** and uses **TCP port 25**. The company wants to **decommission the legacy SMTP server** and use **Amazon Simple Email Service (SES)**. The goal is to modify the application to send emails via Amazon SES while respecting the constraint that the application _can only use SMTP_. This implies connecting to SES via an SMTP interface.

### Services Explanation

|**Service**|**Description**|**Relevant Details for this Problem**|
|---|---|---|
|**Amazon Simple Email Service (SES)**|A cost-effective, flexible, and scalable email platform that enables developers to send mail from within any application.|Supports sending email via **SMTP interface**, **SES API** (HTTPS), or **AWS SDKs**. It's the target replacement for the legacy SMTP server.|
|**Simple Mail Transfer Protocol (SMTP)**|The standard protocol for sending email.|The application _must_ use this protocol. SES provides an SMTP endpoint.|
|**TLS (Transport Layer Security)**|A cryptographic protocol designed to provide communication security over a computer network.|Standard security practice for email, but the legacy app doesn't support it. SES _requires_ secure connections (TLS Wrapper or STARTTLS).|
|**TCP Port 25**|The standard port for SMTP traffic.|Often throttled or blocked by ISPs and AWS (e.g., on EC2 for outbound traffic) to prevent spam. SES uses ports **25**, **465** (TLS Wrapper), or **587** (STARTTLS).|
|**IAM (Identity and Access Management)**|AWS service for securely managing access to AWS services and resources.|Used to authenticate and authorize email sending. SES SMTP credentials are a special type of IAM user credential.|
|**STARTTLS**|An extension to plain text communication protocols, that tells the protocol to switch to an encrypted (TLS or SSL) connection instead of using a separate port for encrypted communication.|One method to secure an SMTP connection to SES (on ports 25 or 587).|
|**TLS Wrapper (SMTPS)**|A method of securing a connection where the client immediately connects using TLS on a dedicated port.|The other method to secure an SMTP connection to SES (on port 465).|
|**IAM Role**|An AWS identity with permission policies that determine what the identity can and cannot do.|Used for applications running on AWS services (like EC2) to access other services _via the API_, but not typically for SMTP authentication.|
|**IAM User**|An identity within an AWS account that represents a person or application.|SES SMTP credentials are derived from an IAM user/role, but SES provides a specific set of **SMTP credentials** for authentication.|
|**AWS SDKs**|Software Development Kits for various programming languages to interact with AWS services.|Used to interact with the **SES API** (HTTPS), an alternative to the SMTP protocol.|

### Correct Option(s) Explanation

The correct option is **B**.

**B. Configure the application to connect to Amazon SES by using STARTTLS. Obtain Amazon SES SMTP credentials. Use the credentials to authenticate with Amazon SES.**

1. **Constraint Fulfillment (SMTP Required):** The application _must_ use SMTP. Amazon SES offers an **SMTP interface**, which is the required method.
    
2. **Authentication:** When connecting to the SES SMTP endpoint, the application must authenticate. Amazon SES uses standard SMTP authentication, but the username and password must be replaced with **SES SMTP credentials**. These are specifically generated from an IAM user or role and are used _only_ for SMTP access.
    
3. **Security:** SES _requires_ a secure connection. Since the legacy application uses TCP port 25 but **does not support TLS encryption**, it's likely a simple SMTP client. However, to connect to SES, a secure method must be used.
    
    - **STARTTLS** is a method where the connection _starts_ unencrypted (often on port 587 or 25, if unblocked) and then is _upgraded_ to TLS.
        
    - The prompt states the legacy server **does not support TLS**. This is a key constraint. However, if the application _must_ use the SMTP protocol to connect to SES, it _must_ use a secure method (STARTTLS or TLS Wrapper) because **SES requires a secure connection**.
        
    - The most practical solution that sticks to the SMTP protocol is to use the **SES SMTP endpoint** and its required **STARTTLS** (port 587 or 25) or **TLS Wrapper** (port 465). Given the need to decommission the legacy, the application **must be minimally modified** to support the security that SES demands. The combination of SES SMTP credentials and a standard SMTP security method like STARTTLS (which is widely compatible) is the prescribed AWS best practice for using the SES SMTP endpoint.
        
    - Referencing the official SES documentation, "Amazon SES requires that you encrypt your SMTP connection using **TLS** or **STARTTLS**" (AWS SES Developer Guide).

### Incorrect Options Explanation

|**Option**|**Reason for Incorrectness**|
|---|---|
|**A. Configure the application to connect to Amazon SES by using TLS Wrapper. Create an IAM role that has ses:SendEmail and ses:SendRawEmail permissions. Attach the IAM role to an Amazon EC2 instance.**|**Incorrect Authentication and Access Method:** Using an **IAM Role attached to an EC2 instance** is the correct method for applications to access AWS services _via the API/SDKs_ (e.g., using `ses:SendEmail`). However, this is **not** the authentication method for the SES **SMTP endpoint**. SMTP requires **SES SMTP Credentials** (username/password).|
|**C. Configure the application to use the SES API to send email messages. Create an IAM role that has ses:SendEmail and ses:SendRawEmail permissions. Use the IAM role as a service role for Amazon SES.**|**Incorrect Protocol:** The application _can use SMTP only_. Using the **SES API** (via HTTPS) requires modifying the application to use the SES REST API, which violates the constraint. Also, the IAM role would be used by the application/EC2 instance, not "as a service role for Amazon SES."|
|**D. Configure the application to use AWS SDKs to send email messages. Create an IAM user for Amazon SES. Generate API access keys. Use the access keys to authenticate with Amazon SES.**|**Incorrect Protocol and Suboptimal Authentication:** Similar to C, using **AWS SDKs** uses the SES API (HTTPS), violating the _SMTP only_ constraint. While IAM user API keys _can_ be used to access the SES API, the entire approach ignores the need to use SMTP. (Note: **SES SMTP credentials** are _derived_ from IAM user/role credentials but are specifically formatted for SMTP).|
