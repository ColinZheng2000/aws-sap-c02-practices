A company has an organization in AWS Organizations. The company is using AWS Control Tower to deploy a landing zone for the organization. The company wants to implement governance and policy enforcement. The company must implement a policy that will detect Amazon RDS DB instances that are not encrypted at rest in the company’s production OU.  
  
Which solution will meet this requirement?

- A. Turn on mandatory guardrails in AWS Control Tower. Apply the mandatory guardrails to the production OU.
- B. Enable the appropriate guardrail from the list of strongly recommended guardrails in AWS Control Tower. Apply the guardrail to the production OU.
- C. Use AWS Config to create a new mandatory guardrail. Apply the rule to all accounts in the production OU.
- D. Create a custom SCP in AWS Control Tower. Apply the SCP to the production OU.
---

## Understanding the Problem

The core requirement is to *detect* a specific configuration issue: Amazon RDS instances that are not encrypted at rest. This detection must be scoped only to the "production" Organizational Unit (OU) within an AWS Organization that is managed by AWS Control Tower. The solution must use the governance and policy features provided by Control Tower.

The keyword here is **"detect"**. This implies a monitoring and reporting mechanism, not a mechanism that *prevents* the action from happening in the first place.

---

## Services Explanation

* **AWS Control Tower:** This service provides the easiest way to set up and govern a new, secure, multi-account AWS environment (a "landing zone"). It automates the setup using AWS Organizations, AWS IAM Identity Center (formerly SSO), AWS Service Catalog, and AWS Config.
* **Guardrails:** This is the primary governance feature of AWS Control Tower. Guardrails are high-level rules that provide ongoing governance for your overall AWS environment. They come in two main types:
    * **Preventive Guardrails:** These are implemented using **Service Control Policies (SCPs)**. They *prevent* actions from occurring, ensuring accounts stay compliant with policies (e.g., "Disallow creation of public S3 buckets").
    * **Detective Guardrails:** These are implemented using **AWS Config rules**. They *detect* non-compliant resources or actions *after* they have occurred and report on them in the Control Tower dashboard (e.g., "Detect if public read access is enabled on S3 buckets").
* **Guardrail Categories:** Control Tower groups guardrails into categories based on how they are applied:
    * **Mandatory:** These are enabled by default when you set up your landing zone. They are foundational, cannot be disabled, and apply to the *entire* organization.
    * **Strongly Recommended:** These enforce common best practices. They are optional and can be enabled on specific OUs.
    * **Elective:** These are additional guardrails for tracking or locking down specific resource types. They are also optional and can be enabled on specific OUs.
* **AWS Config:** A service that enables you to assess, audit, and evaluate the configurations of your AWS resources. It's the underlying engine for all *detective* guardrails in Control Tower.
* **Service Control Policies (SCPs):** A feature of AWS Organizations used to manage permissions. SCPs define the *maximum* permissions available to IAM users and roles in an account. They are the underlying engine for all *preventive* guardrails in Control Tower.

---

## Correct Option(s) Explanation

**B. Enable the appropriate guardrail from the list of strongly recommended guardrails in AWS Control Tower. Apply the guardrail to the production OU.**

This is the correct solution because it perfectly aligns with the requirement and the functionality of AWS Control Tower.

1.  **"Detect" vs. "Prevent":** The question asks to *detect* non-compliance. This maps directly to a **detective guardrail**, which uses AWS Config.
2.  **Built-in Guardrail:** AWS Control Tower provides a specific, pre-built guardrail for this exact purpose: `Detect whether encryption is enabled for Amazon RDS database instances` (Control ID: `AWS-GR_RDS_ENCRYPTION_ENABLED`).
3.  **Scope (Production OU):** This guardrail is not *Mandatory*. It falls under the *Strongly Recommended* or *Elective* categories. This is key, as optional guardrails (unlike mandatory ones) can be selectively applied to specific OUs. The administrator can go into the Control Tower dashboard, find this guardrail, and choose to enable it *only* for the "production" OU.

This approach uses the native, high-level functionality of Control Tower as intended, providing automated detection and reporting on the compliance dashboard.

## Incorrect Options Explanation

* **A. Turn on mandatory guardrails in AWS Control Tower. Apply the mandatory guardrails to the production OU.**
    * This is incorrect for two reasons. First, **mandatory guardrails are enabled by default** when the landing zone is set up; you don't "turn them on" afterward. Second, **mandatory guardrails apply to the entire organization (root OU)**, not specific OUs. You cannot selectively apply them.

* **C. Use AWS Config to create a new mandatory guardrail. Apply the rule to all accounts in the production OU.**
    * This option has incorrect terminology. You **cannot "create a new mandatory guardrail."** Mandatory guardrails are defined *by AWS* and are part of the Control Tower service itself. While you *can* create custom AWS Config rules (which is what detective guardrails are), you don't do this by creating a "mandatory guardrail." Option B is the correct way to enable the *existing* guardrail that already does this.

* **D. Create a custom SCP in AWS Control Tower. Apply the SCP to the production OU.**
    * This is the most common distractor. An **SCP is a *preventive* control**. An SCP-based solution would *block* users from creating unencrypted RDS instances. The question explicitly asks to *detect* non-compliance, not *prevent* it. This makes an SCP the wrong tool for this specific requirement. The correct tool for *detection* is a detective guardrail, which is based on AWS Config.
