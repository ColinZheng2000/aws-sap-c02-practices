A solutions architect is auditing the security setup of an AWS Lambda function for a company. The Lambda function retrieves the latest changes from an Amazon Aurora database. The Lambda function and the database run in the same VPC. Lambda environment variables are providing the database credentials to the Lambda function.  
  
The Lambda function aggregates data and makes the data available in an Amazon S3 bucket that is configured for server-side encryption with AWS KMS managed encryption keys (SSE-KMS). The data must not travel across the internet. If any database credentials become compromised, the company needs a solution that minimizes the impact of the compromise.  
  
What should the solutions architect recommend to meet these requirements?

- A. Enable IAM database authentication on the Aurora DB cluster. Change the IAM role for the Lambda function to allow the function to access the database by using IAM database authentication. Deploy a gateway VPC endpoint for Amazon S3 in the VPC.
- B. Enable IAM database authentication on the Aurora DB cluster. Change the IAM role for the Lambda function to allow the function to access the database by using IAM database authentication. Enforce HTTPS on the connection to Amazon S3 during data transfers.
- C. Save the database credentials in AWS Systems Manager Parameter Store. Set up password rotation on the credentials in Parameter Store. Change the IAM role for the Lambda function to allow the function to access Parameter Store. Modify the Lambda function to retrieve the credentials from Parameter Store. Deploy a gateway VPC endpoint for Amazon S3 in the VPC.
- D. Save the database credentials in AWS Secrets Manager. Set up password rotation on the credentials in Secrets Manager. Change the IAM role for the Lambda function to allow the function to access Secrets Manager. Modify the Lambda function to retrieve the credentials Om Secrets Manager. Enforce HTTPS on the connection to Amazon S3 during data transfers.

# AWS SAP-C02 Question Analysis: Lambda-Aurora-S3 Security Setup
tags: #AWS-SAP-C02 #Lambda #Aurora #S3 #VPC #IAM #Security #VPC-Endpoint #IAM-DB-Auth

## 1. Understanding the Problem
The scenario involves three core components:
- An AWS Lambda function (in a VPC) that retrieves data from an Amazon Aurora DB cluster (same VPC), currently using **Lambda environment variables** for database credentials (insecure, no rotation).
- Lambda aggregates data and writes to an Amazon S3 bucket (SSE-KMS encrypted).
- Key requirements:
  1. **No internet for data transfer**: Data between Lambda (VPC) and S3 must not traverse the public internet.
  2. **Minimize credential compromise impact**: If database credentials are exposed, the solution must limit damage (env vars are high-risk as they are static and unrotated).

## 2. Services Explanation
### Core AWS Services Involved
| Service | Purpose & Key Details (with Official Docs) |
|---------|---------------------------------------------|
| Lambda Environment Variables | Stores configuration in plaintext (by default) or encrypted with KMS. **No built-in rotation** – credentials remain static, increasing compromise risk (https://docs.aws.amazon.com/lambda/latest/dg/configuration-envvars.html). |
| Amazon Aurora IAM Database Authentication | Replaces static passwords with **temporary authentication tokens** (valid for 15 mins by default). Uses IAM roles for authentication, eliminating static credentials. Ideal for minimizing compromise impact (https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/UsingWithRDS.IAMDBAuth.html). |
| AWS Secrets Manager | Purpose-built for secret storage (credentials, API keys) with **native auto-rotation** for Aurora/RDS (no custom code). Rotation frequency is configurable (e.g., 1–30 days) (https://docs.aws.amazon.com/secretsmanager/latest/userguide/rotating-secrets.html). |
| AWS Systems Manager Parameter Store | Stores secure strings (encrypted with KMS) but requires **custom Lambda functions for database credential rotation** (no native integration for Aurora). Better for non-secret configs (https://docs.aws.amazon.com/systems-manager/latest/userguide/sysman-paramstore-rotate.html). |
| S3 Gateway VPC Endpoint | Enables private connectivity between a VPC and S3 via the AWS backbone (no internet gateway/NAT required). Critical for meeting the "no internet" requirement (https://docs.aws.amazon.com/vpc/latest/privatelink/vpc-endpoints-s3.html). |
| HTTPS for S3 | Encrypts data in transit but **does not prevent internet traversal**. Lambda in a VPC without an S3 endpoint will still route traffic over the public internet (via NAT/internet gateway) (https://docs.aws.amazon.com/AmazonS3/latest/userguide/UsingHTTPS.html). |

## 3. Correct Option Insightful Explanation (Option A)
### Why Option A Meets All Requirements
Option A: *Enable IAM database authentication on the Aurora DB cluster. Change the IAM role for the Lambda function to allow the function to access the database by using IAM database authentication. Deploy a gateway VPC endpoint for Amazon S3 in the VPC.*

1. **Minimizes Credential Compromise Impact**:
   - IAM database authentication eliminates static credentials (replacing risky env vars). Lambda requests short-lived tokens (15 mins) from AWS STS instead of storing passwords. Even if a token is compromised, its short lifespan limits attacker access – the most effective way to minimize impact (per AWS security best practices: https://docs.aws.amazon.com/AmazonRDS/latest/AuroraUserGuide/rds-security-best-practices.html).
   - Lambda’s IAM role is granted permissions to generate these tokens, ensuring least-privilege access.

2. **No Internet for Data Transfer**:
   - A gateway VPC endpoint for S3 routes Lambda-to-S3 traffic through the AWS backbone, bypassing the public internet. This directly satisfies the "data must not travel across the internet" requirement (AWS confirms this is the only way to privately access S3 from a VPC: https://docs.aws.amazon.com/vpc/latest/privatelink/concepts.html#gateway-endpoints).

3. **Alignment with SAP-C02 Exam Objectives**:
   - Prioritizes "zero static credentials" (IAM DB auth) over stored secrets (Secrets Manager/Parameter Store) for maximum security – a key focus of AWS enterprise security architecture.

## 4. Incorrect Options Insightful Explanation
### Option B
*Enable IAM database authentication on the Aurora DB cluster. Change the IAM role for the Lambda function to allow the function to access the database by using IAM database authentication. Enforce HTTPS on the connection to Amazon S3 during data transfers.*

- **Failure**: Enforcing HTTPS only encrypts data in transit but does not prevent internet traversal. Lambda in a VPC without an S3 endpoint will still send traffic over the public internet (via NAT/internet gateway), violating the "no internet" requirement. HTTPS is a security control for confidentiality, not network path isolation.

### Option C
*Save the database credentials in AWS Systems Manager Parameter Store. Set up password rotation on the credentials in Parameter Store. Change the IAM role for the Lambda function to allow the function to access Parameter Store. Modify the Lambda function to retrieve the credentials from Parameter Store. Deploy a gateway VPC endpoint for Amazon S3 in the VPC.*

- **Failure**: Parameter Store lacks native auto-rotation for Aurora credentials. Rotation requires custom Lambda functions, which increases operational overhead and risk of misconfiguration (e.g., broken rotation workflows). Even with rotation, static credentials are still stored (and valid until rotation), which is less secure than IAM DB auth’s short-lived tokens. AWS recommends Secrets Manager (not Parameter Store) for database credentials requiring rotation (https://docs.aws.amazon.com/secretsmanager/latest/userguide/intro.html#compare-with-parameter-store).

### Option D
*Save the database credentials in AWS Secrets Manager. Set up password rotation on the credentials in Secrets Manager. Change the IAM role for the Lambda function to allow the function to access Secrets Manager. Modify the Lambda function to retrieve the credentials from Secrets Manager. Enforce HTTPS on the connection to Amazon S3 during data transfers.*

- **Dual Failure**:
  1. **Internet Traversal**: Enforcing HTTPS does not prevent Lambda-to-S3 traffic from using the public internet (same flaw as Option B).
  2. **Inferior Credential Security**: While Secrets Manager’s native rotation is better than Parameter Store, it still relies on static credentials (rotated periodically, e.g., 30 days) – a higher risk than IAM DB auth’s 15-minute tokens. For minimizing compromise impact, "no static credentials" (IAM DB auth) is superior to "rotated static credentials" (Secrets Manager).
