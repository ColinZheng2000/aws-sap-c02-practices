A company is running an application in the AWS Cloud. Recent application metrics show inconsistent response times and a significant increase in error rates. Calls to third-party services are causing the delays. Currently, the application calls third-party services synchronously by directly invoking an AWS Lambda function.  
  
A solutions architect needs to decouple the third-party service calls and ensure that all the calls are eventually completed.  
  
Which solution will meet these requirements?

- A. Use an Amazon Simple Queue Service (Amazon SQS) queue to store events and invoke the Lambda function.
- B. Use an AWS Step Functions state machine to pass events to the Lambda function.
- C. Use an Amazon EventBridge rule to pass events to the Lambda function.
- D. Use an Amazon Simple Notification Service (Amazon SNS) topic to store events and Invoke the Lambda function.

### Analysis

#### 1. Understanding the Problem
The core issues identified in the scenario are:
* **Latency & Errors:** The application is suffering from inconsistent response times (latency) and errors due to *synchronous* calls to a third-party service.
* **Coupling:** The current architecture is tightly coupled; if the third-party service hangs, the Lambda hangs, and the main application waits or fails.
* **Requirement:** We must **decouple** the process (fire and forget from the application's perspective) and ensure **eventual completion** (reliability/retries).

#### 2. Services Explanation

* **Amazon SQS (Simple Queue Service):** A fully managed message queuing service that enables you to decouple and scale microservices, distributed systems, and serverless applications. It offers a buffer between producers and consumers.
* **AWS Step Functions:** A serverless orchestration service that lets you combine AWS Lambda functions and other services into business workflows.
* **Amazon EventBridge:** A serverless event bus that makes it easier to build event-driven applications at scale using events generated from your applications, integrated SaaS applications, and AWS services.
* **Amazon SNS (Simple Notification Service):** A fully managed Pub/Sub messaging service. It pushes messages to subscribers immediately.

#### 3. Correct Option Explanation: Option A (Amazon SQS)
Using **Amazon SQS** is the standard architectural pattern to solve this specific set of problems:

* **Decoupling & Buffering:** SQS acts as a buffer. The application writes the request to the queue and immediately returns success to the user (solving the response time issue). The application no longer waits for the third party.
* **Handling Delays (Backpressure):** If the third-party service is slow, messages simply accumulate in the SQS queue. The Lambda function processing the queue can pick them up at a sustainable rate without overwhelming the downstream service or timing out the upstream application.
* **Eventual Completion (Reliability):** SQS integrates natively with Lambda.
    * If the Lambda fails (due to third-party error), SQS will retry the message after the *Visibility Timeout* expires.
    * You can configure a **Dead Letter Queue (DLQ)** to catch messages that fail processing after a maximum number of attempts, ensuring no data is lost and allowing for manual investigation/re-driving.

**Reference:**
> "SQS offers a reliable, highly scalable, hosted queue for storing messages as they travel between computers. By using SQS, you can move data between distributed application components and perform different tasks without losing messages or requiring each component to be always available."
> — [Amazon SQS Documentation](https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/welcome.html)

#### 4. Incorrect Options Explanation

* **Option B (AWS Step Functions):**
    * *Why it's incorrect:* While Step Functions *can* handle retries and error handling (and is a valid way to orchestrate calls), SQS is the more direct and cost-effective solution specifically for **buffering** and **load flattening**. Step Functions is generally used for complex workflows (state management), whereas SQS is used for high-throughput ingestion and decoupling. SQS provides the specific "queueing" mechanism needed to absorb the latency spikes mentioned in the prompt.
* **Option C (Amazon EventBridge):**
    * *Why it's incorrect:* EventBridge is primarily an event router (bus). While it supports asynchronous invocation and retries, it does not provide the same level of persistent buffering and flow control (throttling the consumer) as SQS. If the third-party service is down for an extended period, SQS is better suited to hold the backlog of messages.
* **Option D (Amazon SNS):**
    * *Why it's incorrect:* SNS follows a "Push" model. When the application publishes to SNS, SNS immediately attempts to push that message to the Lambda. If the Lambda is overwhelmed or the third party causes the Lambda to time out, SNS will retry, but it does not provide a queue to "hold" messages during downtime or latency spikes. It lacks the flow control required to stabilize the application response times effectively.
