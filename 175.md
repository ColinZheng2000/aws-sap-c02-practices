A company is running a containerized application in the AWS Cloud. The application is running by using Amazon Elastic Container Service (Amazon ECS) on a set of Amazon EC2 instances. The EC2 instances run in an Auto Scaling group.
The company uses Amazon Elastic Container Registry (Amazon ECR) to store its container images. When a new image version is uploaded, the new image version receives a unique tag.
The company needs a solution that inspects new image versions for common vulnerabilities and exposures. The solution must automatically delete new image tags that have Critical or High severity findings. The solution also must notify the development team when such a deletion occurs.
Which solution meets these requirements?

A. Configure scan on push on the repository. Use Amazon EventBridge (Amazon CloudWatch Events) to invoke an AWS Step Functions state machine when a scan is complete for images that have Critical or High severity findings. Use the Step Functions state machine to delete the image tag for those images and to notify the development team through Amazon Simple Notification Service (Amazon SNS).
B. Configure scan on push on the repository. Configure scan results to be pushed to an Amazon Simple Queue Service (Amazon SQS) queue. Invoke an AWS Lambda function when a new message is added to the SQS queue. Use the Lambda function to delete the image tag for images that have Critical or High severity findings. Notify the development team by using Amazon Simple Email Service (Amazon SES).
C. Schedule an AWS Lambda function to start a manual image scan every hour. Configure Amazon EventBridge (Amazon CloudWatch Events) to invoke another Lambda function when a scan is complete. Use the second Lambda function to delete the image tag for images that have Critical or High severity findings. Notify the development team by using Amazon Simple Notification Service (Amazon SNS).
D. Configure periodic image scan on the repository. Configure scan results to be added to an Amazon Simple Queue Service (Amazon SQS) queue. Invoke an AWS Step Functions state machine when a new message is added to the SQS queue. Use the Step Functions state machine to delete the image tag for images that have Critical or High severity findings. Notify the development team by using Amazon Simple Email Service (Amazon SES).


Suggest Answer: A

## Analysis

### Understanding the Problem

A company runs a containerized application on Amazon ECS (EC2 launch type) with images stored in Amazon ECR. Each new image version receives a unique tag. The solution must: (1) **automatically inspect** new image versions for vulnerabilities, (2) **automatically delete** image tags that have Critical or High severity findings, and (3) **notify** the development team upon deletion. The trigger must be immediate upon new image upload.

### Correct Option: A

**Configure scan on push on the repository. Use Amazon EventBridge to invoke an AWS Step Functions state machine when a scan is complete for images that have Critical or High severity findings. Use the Step Functions state machine to delete the image tag for those images and to notify the development team through Amazon Simple Notification Service (Amazon SNS).**

- **Scan on push** is an Amazon ECR feature that automatically triggers an image scan immediately after an image is pushed, leveraging the Common Vulnerabilities and Exposures (CVE) database from Clair. No custom scheduling or manual invocation is needed — the scan fires at exactly the right moment.
- **Amazon EventBridge** integrates natively with ECR. When a scan completes, ECR emits an event (`ECR Image Scan Completed`) to the default event bus. EventBridge rules can filter on finding severity (`CRITICAL` or `HIGH`) before invoking downstream targets.
- **AWS Step Functions** is the ideal orchestration engine for this multi-step workflow: evaluate scan findings → call `batch-delete-image` API to remove the offending tag → publish to an SNS topic. Step Functions provides built-in retry logic, error handling, and a visual execution history — critical for operational auditing of automated deletion decisions.
- **Amazon SNS** delivers the notification to the development team via email, SMS, or other subscribed endpoints.

### Incorrect Options

**B: Scan on push → SQS → Lambda → SES** — Amazon ECR does **not** natively push scan results to an SQS queue. You would need EventBridge as an intermediary to route events to SQS. More importantly, using Lambda for multi-step logic (check findings → delete tag → notify) lacks the visual workflow, retry semantics, and error handling that Step Functions provides out of the box. SES is a valid notification channel, but the architectural pattern is suboptimal.

**C: Scheduled Lambda hourly scan + EventBridge + Lambda → SNS** — A scheduled Lambda function that calls `start-image-scan` every hour introduces up to a 60-minute delay before vulnerabilities are detected. For Critical CVEs in production images, this latency is unacceptable. This also incurs unnecessary Lambda invocation costs even when no new images were pushed.

**D: Periodic scan + SQS + Step Functions → SES** — "Periodic image scan" (the `MANUAL` scan type triggered on a schedule) introduces the same latency issue as option C. Additionally, ECR does not natively send scan results to SQS without EventBridge routing. SES alone is fine for notifications but SNS offers broader protocol support (SMS, Lambda, HTTP/S, etc.).

### Hands-On Knowledge

1. **ECR Scan Types**: "Scan on push" triggers on every `PutImage` API call. "Continuous scan" (enhanced scanning powered by Amazon Inspector) provides ongoing monitoring. Basic scanning (Clair-based) is the default.
2. **EventBridge ECR Pattern**: The event `aws.ecr` with detail-type `ECR Image Scan` contains a `finding-severity-counts` field with counts for `CRITICAL`, `HIGH`, `MEDIUM`, etc. Use this to filter events before invoking automation.
3. **Step Functions for remediation**: Design the state machine with a `Choice` state to check severity, a `Task` state calling `ecr:batchDeleteImage`, and a `Task` state publishing to SNS. Add a `Fail` state with a meaningful error message for troubleshooting.
4. **Image tag immutability**: Consider enabling tag immutability on production ECR repositories. This prevents overwriting tags and ensures each tag uniquely identifies a specific image version.
5. **Real enterprise pattern**: Many organizations combine ECR scan on push → EventBridge → Step Functions → SNS with an additional Lambda that opens a Jira/ServiceNow ticket for tracking remediation progress.
