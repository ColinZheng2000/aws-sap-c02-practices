A company recently acquired several other companies. Each company has a separate AWS account with a different billing and reporting method. The acquiring company has consolidated all the accounts into one organization in AWS Organizations. However, the acquiring company has found it difficult to generate a cost report that contains meaningful groups for all the teams.  
  
The acquiring company’s finance team needs a solution to report on costs for all the companies through a self-managed application.  
  
Which solution will meet these requirements?

- A. Create an AWS Cost and Usage Report for the organization. Define tags and cost categories in the report. Create a table in Amazon Athena. Create an Amazon QuickSight dataset based on the Athena table. Share the dataset with the finance team.
- B. Create an AWS Cost and Usage Report for the organization. Define tags and cost categories in the report. Create a specialized template in AWS Cost Explorer that the finance department will use to build reports.
- C. Create an Amazon QuickSight dataset that receives spending information from the AWS Price List Query API. Share the dataset with the finance team.
- D. Use the AWS Price List Query API to collect account spending information. Create a specialized template in AWS Cost Explorer that the finance department will use to build reports.

### Understanding the Problem

The core challenge is to **consolidate cost and usage data** from multiple AWS accounts, now unified under one **AWS Organization**, and make it available for **detailed reporting** by a **self-managed application** used by the finance team.

Key requirements:

1. Generate a **cost report** for the entire organization (consolidated billing).
    
2. The report must support **meaningful groups** (implying the need for **tags** and/or **cost categories**).
    
3. The solution must facilitate reporting through a **self-managed application** (requiring programmatic access to the data, not just a console-based tool).

### Correct Option(s) Explanation

**A. Create an AWS Cost and Usage Report for the organization. Define tags and cost categories in the report. Create a table in Amazon Athena. Create an Amazon QuickSight dataset based on the Athena table. Share the dataset with the finance team.**

- **Create an AWS Cost and Usage Report (CUR) for the organization:** This is the essential first step. **CUR** is the _only_ service that provides the **granular, raw, detailed usage and cost data** for all accounts in the Organization, which is critical for complex, custom reporting.
    
- **Define tags and cost categories in the report:** This addresses the requirement for **meaningful groups**, as CUR includes these dimensions for slicing the cost data.
    
- **Create a table in Amazon Athena:** Athena is the standard, serverless, and highly scalable way to query the massive files generated by CUR stored in S3, making the data accessible via standard SQL.
    
- **Create an Amazon QuickSight dataset based on the Athena table. Share the dataset with the finance team:** This final step makes the consolidated and queryable data accessible for reporting. While the prompt mentions a "self-managed application," using the Athena/QuickSight stack (often referred to as the **Cloud Intelligence Dashboards** approach) is the most robust, scalable, and common practice for centralizing cost data. The finance team can either use QuickSight directly or connect their self-managed application to the data layer (Athena/S3). This option offers the necessary **data granularity and flexibility** required by a custom application.
    

---

### Incorrect Options Explanation

**B. Create an AWS Cost and Usage Report for the organization. Define tags and cost categories in the report. Create a specialized template in AWS Cost Explorer that the finance department will use to build reports.**

- **Reason:** While starting correctly with the **CUR** to get the granular data, relying on **AWS Cost Explorer** to fulfill the requirement for a "self-managed application" is a mismatch. Cost Explorer is a console-based visualization tool for _ad hoc_ analysis; it is not the correct mechanism for a **self-managed application** to access the detailed, underlying data for custom reporting.
    

**C. Create an Amazon QuickSight dataset that receives spending information from the AWS Price List Query API. Share the dataset with the finance team.**

- **Reason:** This is fundamentally incorrect. The **AWS Price List Query API** only provides **pricing information** (list prices or effective prices), not actual **spending or usage data** for the accounts. You cannot generate cost reports based on actual consumption from this API.
    

**D. Use the AWS Price List Query API to collect account spending information. Create a specialized template in AWS Cost Explorer that the finance department will use to build reports.**

- **Reason:** This option fails for two reasons:
    
    1. It incorrectly uses the **AWS Price List Query API** to collect **spending information**. This API provides prices, not spending/usage.
        
    2. It incorrectly relies on **AWS Cost Explorer** as the reporting source for a "self-managed application."
