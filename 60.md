A solutions architect is designing the data storage and retrieval architecture for a new application that a company will be launching soon. The application is designed to ingest millions of small records per minute from devices all around the world. Each record is less than 4 KB in size and needs to be stored in a durable location where it can be retrieved with low latency. The data is ephemeral and the company is required to store the data for 120 days only, after which the data can be deleted.  
The solutions architect calculates that, during the course of a year, the storage requirements would be about 10-15 TB.  
Which storage strategy is the MOST cost-effective and meets the design requirements?  

- A. Design the application to store each incoming record as a single .csv file in an Amazon S3 bucket to allow for indexed retrieval. Configure a lifecycle policy to delete data older than 120 days.
- B. Design the application to store each incoming record in an Amazon DynamoDB table properly configured for the scale. Configure the DynamoDB Time to Live (TTL) feature to delete records older than 120 days.
- C. Design the application to store each incoming record in a single table in an Amazon RDS MySQL database. Run a nightly cron job that executes a query to delete any records older than 120 days.
- D. Design the application to batch incoming records before writing them to an Amazon S3 bucket. Update the metadata for the object to contain the list of records in the batch and use the Amazon S3 metadata search feature to retrieve the data. Configure a lifecycle policy to delete the data after 120 days.

---

## Understanding the Problem

The core problem is to design a storage solution for a very high-throughput (millions of records/minute) workload of small records (< 4 KB). The solution must provide:
1.  **High Durability:** The data must be stored reliably.
2.  **Low Latency Retrieval:** Individual records must be read back quickly.
3.  **Automatic Deletion:** Data must be deleted after 120 days.
4.  **Cost-Effectiveness:** It must be the *most* cost-effective solution that meets all the above criteria.
5.  **Scale:** The total storage is relatively small (10-15 TB/year), but the *ingestion rate* is the main challenge.

---

## Services Explanation

* **Amazon S3 (Simple Storage Service):** A highly durable (11 nines) and scalable object storage service. It's extremely cost-effective for storing data ("storage at rest"). However, it's billed not just on storage but also on requests (like `PUT`, `GET`, `LIST`). While S3 can handle high request rates, millions of *individual small file* writes per minute can become expensive due to the per-request cost and can introduce "small file" performance challenges.
    * **S3 Lifecycle Policy:** A feature that automates the transition of objects to different storage classes or deletes them after a specified time. This directly addresses the 120-day deletion requirement.
    * **Official Docs:** [What is Amazon S3?](https://docs.aws.amazon.com/AmazonS3/latest/userguide/Welcome.html), [S3 Lifecycle Management](https://docs.aws.amazon.com/AmazonS3/latest/userguide/object-lifecycle-mgmt.html)

* **Amazon DynamoDB:** A fully managed, serverless, NoSQL key-value and document database. Its primary feature is providing **single-digit millisecond latency at any scale**. It's designed for high-traffic applications that need consistent, fast performance. You configure read and write capacity (or use On-Demand mode), and it handles the underlying partitioning and scaling.
    * **Time to Live (TTL):** A feature that allows you to define a per-item timestamp to determine when an item is no longer needed. DynamoDB automatically deletes expired items, typically within 48 hours of expiration, at no extra cost (you don't pay for the delete operation itself, just the storage until deletion).
    * **Official Docs:** [What is Amazon DynamoDB?](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/Introduction.html), [DynamoDB Time to Live (TTL)](https://docs.aws.amazon.com/amazondynamodb/latest/developerguide/TTL.html)

* **Amazon RDS (Relational Database Service):** A managed service for relational databases like MySQL, PostgreSQL, etc. It's designed for structured data with complex queries, transactions, and joins. It is *not* designed for the ingestion scale of millions of writes per minute; it would quickly become a bottleneck due to connection limits, locking, and I/O constraints.

---

## Correct Option Explanation

**B. Design the application to store each incoming record in an Amazon DynamoDB table properly configured for the scale. Configure the DynamoDB Time to Live (TTL) feature to delete records older than 120 days.**

This is the best solution because it directly addresses all requirements:

* **High Ingestion:** DynamoDB is built for this. It can scale to handle millions of writes per second (which is more than millions per minute) by configuring sufficient Write Capacity Units (WCUs) or using On-Demand mode.
* **Small Records (< 4 KB):** This is ideal for DynamoDB, as its item size limit is **400** KB.
* **Low Latency Retrieval:** This is DynamoDB's core value proposition. Retrieving a record using its primary key (`GetItem`) provides single-digit millisecond latency.
* **120-Day Deletion:** The **DynamoDB TTL** feature is the perfect, managed, and cost-effective way to handle this. You simply set an expiration attribute on each item, and DynamoDB handles the deletion automatically.
* **Cost-Effectiveness:** While S3 *storage* is cheaper per GB, the *total cost* of other solutions would be higher. Option A would incur massive S3 `PUT` request costs. Option C would require an enormous, expensive RDS instance that would still fail. Option B is the most cost-effective solution *that actually works* and meets the low-latency retrieval requirement.

## Incorrect Options Explanation

**A. Design the application to store each incoming record as a single .csv file in an Amazon S3 bucket...**
* **Fails on Ingestion/Cost:** This is a classic S3 anti-pattern. Writing millions of tiny 4 KB files per minute as individual objects will result in an enormous bill for `PUT` requests. The cost of the *requests* would likely far outweigh the storage cost.
* **Fails on Retrieval:** S3 does not have a native "indexed retrieval" for the *contents* of a CSV file. To retrieve a single record, you'd have to know the *exact S3 object key (filename)*. If you need to search, you'd have to use a service like Amazon Athena, which is *not* low latency.

**C. Design the application to store each incoming record in a single table in an Amazon RDS MySQL database...**
* **Fails on Ingestion:** A relational database like MySQL cannot handle millions of `INSERT` statements per minute on a single table. It would collapse under the load due to locking, connection overhead, and I/O bottlenecks.
* **Fails on Deletion:** Running a nightly `DELETE` query on a table with billions of rows (millions/min for 120 days) is operationally catastrophic. This query would run for hours, consume massive resources, and likely lock the table, crippling application performance.

**D. Design the application to batch incoming records before writing them to an Amazon S3 bucket...**
* **Fails on Retrieval:** This option correctly solves the ingestion problem of Option A by batching. However, it fails the "low latency retrieval" requirement. The proposal to "use the Amazon S3 metadata search feature" is not a valid, low-latency solution.
    1.  S3 object metadata is limited to 2 KB, so it cannot "contain the list of records" for any meaningful batch.
    2.  There is no native S3 "metadata search" that allows you to find and retrieve a specific 4 KB record *from within a larger batch file* with low latency. You would need to use S3 Select or Athena, which are for analytics and are not single-digit millisecond retrieval tools.
