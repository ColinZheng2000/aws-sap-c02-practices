A company runs a highly available data collection application on Amazon EC2 in the eu-north-1 Region. The application collects data from end-user devices and writes records to an Amazon Kinesis data stream and a set of AWS Lambda functions that process the records. The company persists the output of the record processing to an Amazon S3 bucket in eu-north-1. The company uses the data in the S3 bucket as a data source for Amazon Athena.

The company wants to increase its global presence. A solutions architect must launch the data collection capabilities in the sa-east-1 and ap-northeast-1 Regions. The solutions architect deploys the application, the Kinesis data stream, and the Lambda functions in the two new Regions. The solutions architect keeps the S3 bucket in eu-north-1 to meet a requirement to centralize the data analysis.

During testing of the new setup, the solutions architect notices a significant lag on the arrival of data from the new Regions to the S3 bucket.

Which solution will improve this lag time the MOST?

A. In each of the two new Regions, set up the Lambda functions to run in a VPC. Set up an S3 gateway endpoint in that VPC.
B. Turn on S3 Transfer Acceleration on the S3 bucket in eu-north-1. Change the application to use the new S3 accelerated endpoint when the application uploads data to the S3 bucket.
C. Create an S3 bucket in each of the two new Regions. Set the application in each new Region to upload to its respective S3 bucket. Set up S3 Cross-Region Replication to replicate data to the S3 bucket in eu-north-1.
D. Increase the memory requirements of the Lambda functions to ensure that they have multiple cores available. Use the multipart upload feature when the application uploads data to Amazon S3 from Lambda.




# AWS SAP-C02 Question Analysis — Q346

## 📋 Analysis

### 1. Understanding the Problem

The company originally operated entirely in a single AWS Region, so the ingestion pipeline had low latency because all components were geographically close together.

After expanding globally, the application stack now exists in:
- eu-north-1
- sa-east-1
- ap-northeast-1

However, the architecture still writes all processed data directly into a centralized Amazon S3 bucket in eu-north-1.

The observed lag is caused by long-distance cross-Region uploads from South America and Asia to Northern Europe.

The key architectural challenge is:

- Keep centralized analytics in eu-north-1
- Reduce ingestion latency from distant Regions
- Improve overall throughput and responsiveness

This is a classic AWS global ingestion architecture optimization scenario.

Core AWS principle:
- “Write locally, replicate globally.”

- --

### 2. Options Analysis

#### ✅ Option C (CORRECT): Use regional S3 buckets with S3 Cross-Region Replication

This is the best solution because it removes synchronous long-distance uploads from the ingestion path.

How it works:

1. Each Region writes data into a local S3 bucket:
   - sa-east-1 → local S3 bucket
   - ap-northeast-1 → local S3 bucket

2. S3 Cross-Region Replication (CRR) asynchronously replicates objects into the centralized eu-north-1 bucket.

3. Athena continues querying the centralized bucket.

Why this improves lag the MOST:

- Local S3 uploads are extremely fast because traffic stays inside the same Region.
- Users and Lambda functions no longer wait for intercontinental network transfers.
- Replication occurs asynchronously in the background using AWS-managed infrastructure.

Enterprise architecture reasoning:

This is the standard large-scale data lake ingestion pattern used by:
- Medical telemetry systems
- Industrial IoT platforms
- Global SaaS analytics pipelines
- Manufacturing sensor platforms

Benefits:
- Lower ingestion latency
- Better resiliency
- Reduced cross-Region bottlenecks
- Independent regional operation
- Better scalability

Important SAP-C02 insight:

The problem is architectural distance, not compute performance.

#### ❌ Option A (INCORRECT): Run Lambda in a VPC with S3 gateway endpoints

This option misunderstands the root cause.

Why it fails:

- S3 Gateway Endpoints provide private routing between a VPC and S3.
- They do NOT reduce physical cross-Region distance.
- The data still must travel from:
  - sa-east-1 → eu-north-1
  - ap-northeast-1 → eu-north-1

Additional problems:

- Lambda does not require VPC attachment to access S3.
- Putting Lambda into a VPC introduces:
  - ENI management overhead
  - More operational complexity
  - Potential cold-start impact

Key lesson:

VPC endpoints improve:
- Security
- Private routing
- Compliance

They do not solve global latency architecture problems.

AWS official reference:

- Gateway endpoints:
  https://docs.aws.amazon.com/vpc/latest/privatelink/vpc-endpoints-s3.html

- --

#### ❌ Option B (INCORRECT): Enable S3 Transfer Acceleration

This is a tempting answer, but it is not the BEST answer.

Why:

S3 Transfer Acceleration is primarily designed for:
- Internet clients
- Mobile apps
- Browser uploads
- Remote users uploading over public internet

In this scenario:
- Uploads originate from AWS Lambda within AWS infrastructure.
- AWS internal backbone networking is already optimized.

Transfer Acceleration may slightly help, but:
- The architecture still performs synchronous cross-continent uploads.
- The fundamental bottleneck remains.

Why CRR is superior:

CRR changes the architecture itself:
- local ingestion first
- asynchronous replication later

Transfer Acceleration only optimizes the transfer path.

SAP-C02 exam pattern:

Architectural redesign beats micro-optimization.

AWS official reference:

- S3 Transfer Acceleration:
  https://docs.aws.amazon.com/AmazonS3/latest/userguide/transfer-acceleration.html

#### ❌ Option D (INCORRECT): Increase Lambda memory and use multipart upload

This option focuses on compute tuning rather than network architecture.

Why it fails:

Increasing Lambda memory:
- Improves CPU allocation
- Improves networking throughput
- Helps execution speed

Multipart upload:
- Helps large-object parallel upload efficiency

However:
- The dominant issue is geographical distance and cross-Region latency.
- Faster uploads still traverse thousands of kilometers synchronously.

Additional practical point:

Kinesis + Lambda pipelines commonly generate:
- small batches
- event payloads
- streaming objects

Multipart upload provides limited value for small payloads.

Key SAP-C02 takeaway:

When latency is caused by geography:
- solve with distributed architecture
NOT
- vertical scaling

### 3. Hands-On Knowledge

#### Real Enterprise Multi-Region Pattern

Common production architecture:

```text
Regional Devices
      ↓
Regional API / EC2
      ↓
Regional Kinesis Data Streams
      ↓
Regional Lambda Processing
      ↓
Regional S3 Landing Bucket
      ↓
S3 Cross-Region Replication
      ↓
Centralized Data Lake
      ↓
Athena / Glue / EMR
```

This pattern is heavily used in:
- Smart factories
- Healthcare telemetry
- Vehicle telemetry
- Retail analytics
- Energy platforms

- --

#### Important CRR Operational Requirements

##### 1. Versioning Must Be Enabled

Both buckets require versioning.

Example:

```bash
aws s3api put-bucket-versioning \
  - -bucket regional-bucket \
  - -versioning-configuration Status=Enabled
```

- --

##### 2. Replication Is Asynchronous

CRR is not instant.

However:
- It is usually much faster overall than forcing every producer to upload remotely.

For stricter SLAs:
- Use S3 Replication Time Control (RTC).

- --

##### 3. Cost Considerations

CRR introduces:
- Inter-Region transfer cost
- Replication request charges
- Additional storage charges

But operationally it is often worth it because:
- ingestion performance improves dramatically
- user experience improves
- systems scale more predictably

1. "S3 Cross-Region Replication 和 Same-Region Replication 有什么区别？"
   → 回答要点 (Key Answer Points)：CRR 用于跨 Region 数据复制，通常用于灾备、全球分析和集中化数据湖；SRR 用于同 Region 内的隔离、日志聚合或合规需求。两者都依赖 S3 Versioning。

2. "为什么在全球架构中通常推荐本地写入再集中分析？"
   → 回答要点 (Key Answer Points)：本地写入能够显著降低用户侧延迟，并减少跨洲网络依赖。之后通过异步复制实现集中分析，可以获得更好的可扩展性和容错能力。
