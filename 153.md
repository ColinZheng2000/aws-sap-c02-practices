A large education company recently introduced Amazon Workspaces to provide access to internal applications across multiple universities. The company is storing user profiles on an Amazon FSx for Windows File Server file system. The file system is configured with a DNS alias and is connected to a self-managed Active Directory. As more users begin to use the Workspaces, login time increases to unacceptable levels.

An investigation reveals a degradation in performance of the file system. The company created the file system on HDD storage with a throughput of 16 MBps. A solutions architect must improve the performance of the file system during a defined maintenance window.

What should the solutions architect do to meet these requirements with the LEAST administrative effort?

A. Use AWS Backup to create a point-in-time backup of the file system. Restore the backup to a new FSx for Windows File Server file system. Select SSD as the storage type. Select 32 MBps as the throughput capacity. When the backup and restore process is completed, adjust the DNS alias accordingly. Delete the original file system.
B. Disconnect users from the file system. In the Amazon FSx console, update the throughput capacity to 32 MBps. Update the storage type to SSD. Reconnect users to the file system.
C. Deploy an AWS DataSync agent onto a new Amazon EC2 instance. Create a task. Configure the existing file system as the source location. Configure a new FSx for Windows File Server file system with SSD storage and 32 MBps of throughput as the target location. Schedule the task. When the task is completed, adjust the DNS alias accordingly. Delete the original file system.
D. Enable shadow copies on the existing file system by using a Windows PowerShell command. Schedule the shadow copy job to create a point-in-time backup of the file system. Choose to restore previous versions. Create a new FSx for Windows File Server file system with SSD storage and 32 MBps of throughput. When the copy job is completed, adjust the DNS alias. Delete the original file system.

Suggest Answer: B

## Analysis

### Understanding the Problem

A large education company uses Amazon WorkSpaces with user profiles stored on **Amazon FSx for Windows File Server**. The file system is:
- Configured with **HDD storage** (high latency, optimized for throughput-intensive workloads, not small random I/O)
- Provisioned with **throughput of 16 MBps** (low throughput capacity)
- Connected to a **self-managed Active Directory**
- Accessed via a **DNS alias**

As WorkSpaces adoption grows, login times become unacceptable because user profiles (which involve many small, random read/write operations — roaming profiles, NTUSER.DAT, etc.) suffer from both:
1. **HDD's higher latency** compared to SSD for random I/O patterns
2. **16 MBps throughput bottleneck** causing I/O queuing under concurrent user load

The task: improve performance during a defined maintenance window with the **LEAST administrative effort**.

**Key technical constraint**: For Amazon FSx for Windows File Server, **storage type (HDD vs. SSD) is immutable after file system creation**. You cannot convert HDD to SSD in place. Throughput capacity, however, *can* be modified.

### Correct Option: A

**Use AWS Backup to create a point-in-time backup → restore to a new FSx file system with SSD storage and 32 MBps throughput → adjust DNS alias → delete the original file system.**

- **AWS Backup natively supports FSx for Windows File Server**: AWS Backup provides fully managed, automated backup and restore for FSx. The backup captures the entire file system (data, metadata, file permissions, ACLs, and Active Directory integration) consistently.
- **Restore creates a new file system**: During restore, you specify the new storage type (SSD) and throughput capacity (32 MBps). This is the *only* way to change storage type — via migration to a new file system.
- **SSD storage is purpose-built for user profile workloads**: User profiles consist of many small files and registry-like databases (NTUSER.DAT). SSD provides the low-latency, high-IOPS random I/O performance needed for fast profile loading during login.
- **Doubling throughput to 32 MBps**: Increases the burst and sustained throughput ceiling, allowing more concurrent users to load profiles without throttling.
- **DNS alias cutover**: Since the company already uses a DNS alias (not direct file system DNS name), updating the alias to point to the new file system is a single, atomic change. Users reconnect seamlessly after DNS propagation (typically seconds to minutes within a VPC).
- **Least administrative effort among feasible options**: AWS Backup handles the heavy lifting of copying data. No need to deploy additional infrastructure (unlike DataSync with an EC2 agent). The process is: backup → restore with desired specs → DNS cutover → delete old.

### Incorrect Options

**B: Update throughput and storage type in the FSx console**
- **FATAL FLAW: Storage type cannot be changed after creation**. Amazon FSx for Windows File Server locks the storage type (HDD or SSD) at creation time. You *can* increase throughput capacity online (even without downtime), but the HDD→SSD conversion is impossible in place. This option describes an operation that simply does not exist in the AWS console or API.
- **Throughput can be increased, but not sufficient alone**: Even if you could increase throughput to 32 MBps (which you *can* do), staying on HDD storage would not fully resolve the random I/O latency problem inherent to user profile workloads.
- **"Disconnect users" is unnecessary for throughput modification**: Throughput capacity scaling on FSx for Windows does not require disconnecting users — it's an online operation.

**C: AWS DataSync with EC2 agent**
- **Unnecessary complexity**: DataSync requires deploying a DataSync agent on an EC2 instance, configuring source/destination locations, creating and scheduling a task, and monitoring execution. This is significantly more administrative effort than a simple AWS Backup and restore.
- **DataSync is designed for ongoing/hybrid data transfer**: DataSync excels at recurring data transfers between on-premises and AWS, or between AWS storage services. For a one-time migration within the same AWS region, it's over-engineered.
- **Agent management overhead**: The EC2-based DataSync agent must be patched, monitored, and eventually decommissioned — adding operational burden.

**D: Shadow copies with Windows PowerShell**
- **Shadow copies are a Windows Volume Shadow Copy Service (VSS) feature**: They provide point-in-time snapshots for file version recovery *within the same file system*. They are NOT a backup or migration mechanism.
- **Cannot be used to create a new file system**: Shadow copies exist within the context of the original volume. You cannot "restore" a shadow copy to a brand-new FSx file system with different specifications.
- **Not a recognized FSx migration pattern**: This approach is not documented or supported by AWS as a migration method for FSx.
- **Highest complexity, lowest reliability**: Orchestrating shadow copies via PowerShell for a production migration introduces unnecessary scripting, error handling, and validation requirements.

### Hands-On Enterprise Knowledge

**AWS Backup for FSx Migration — Step-by-Step Approach:**

1. **Create an on-demand backup** in AWS Backup (or use a scheduled backup plan):
   ```bash
   aws backup start-backup-job \
     --backup-vault-name fsx-vault \
     --resource-arn arn:aws:fsx:us-east-1:123456789012:file-system/fs-abc123 \
     --iam-role-arn arn:aws:iam::123456789012:role/AWSBackupRole \
     --lifecycle DeleteAfterDays=30
   ```

2. **Restore to a new file system** with SSD and 32 MBps:
   - In the AWS Backup console, select the recovery point → Restore
   - Specify: Storage Type = SSD, Throughput Capacity = 32 MBps
   - The restore preserves all file permissions, ACLs, and AD integration

3. **Update the DNS alias** (CNAME record) to point to the new file system's DNS name:
   ```
   New FSx DNS: amznfsxabc123.example.com → fs-xyz789.fsx.us-east-1.amazonaws.com
   ```

4. **Validate**: Have a pilot group of users log in to WorkSpaces and verify profile load times.

5. **Decommission**: After confirming successful migration, delete the original file system.

**FSx for Windows Performance Tuning Tips:**
- **Choose SSD for user profile workloads (FSLogix, roaming profiles)**: SSD provides sub-millisecond latency for random I/O. HDD is designed for throughput-intensive sequential workloads like media processing, not for profiles.
- **Throughput capacity formula**: Throughput capacity determines both sustained and burst I/O. For N users with profiles, budget approximately 0.5–1 MBps per concurrent user logging in.
- **DNS alias best practice**: Always use a DNS alias (CNAME) for FSx file systems. This allows for transparent migration, disaster recovery failover, and file system replacement without touching each WorkSpace or application configuration.
- **Maintenance window planning**: The restore time depends on data size. For large file systems, schedule the backup in advance (pre-window) and perform the restore and DNS cutover during the window.
- **Active Directory consideration**: Since the company uses self-managed AD, ensure the new FSx file system is joined to the same AD domain. AWS Backup restore handles this, but validate AD connectivity in the VPC.
