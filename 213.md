A company hosts a web application on AWS in the us-east-1 Region. The application servers are distributed across three Availability Zones behind an Application Load Balancer. The database is hosted in a MySQL database on an Amazon EC2 instance. A solutions architect needs to design a cross-Region data recovery solution using AWS services with an RTO of less than 5 minutes and an RPO of less than 1 minute. The solutions architect is deploying application servers in us-west-2, and has configured Amazon Route 53 health checks and DNS failover to us-west-2.

Which additional step should the solutions architect take?

A. Migrate the database to an Amazon RDS for MySQL instance with a cross-Region read replica in us-west-2.
B. Migrate the database to an Amazon Aurora global database with the primary in us-east-1 and the secondary in us-west-2.
C. Migrate the database to an Amazon RDS for MySQL instance with a Multi-AZ deployment.
D. Create a MySQL standby database on an Amazon EC2 instance in us-west-2.


Suggest Answer: B

---

## 📊 Analysis

### Understanding the Problem

A company has a web application in `us-east-1` with EC2 instances across three AZs behind an ALB, and a MySQL database on a self-managed EC2 instance. The solutions architect has already set up:

- Application servers deployed in `us-west-2` (warm standby)
- Route 53 health checks + DNS failover to `us-west-2`

The remaining challenge is the **database layer**: the MySQL database on EC2 needs a cross-Region disaster recovery solution with:

| Metric | Requirement |
|---|---|
| **RTO** (Recovery Time Objective) | < 5 minutes |
| **RPO** (Recovery Point Objective) | < 1 minute |

The solutions architect needs to choose the right AWS database service and configuration to meet these aggressive DR targets.

### ✅ Correct Option: B — Amazon Aurora Global Database

| Aspect | How Aurora Global Database Meets It |
|---|---|
| **RTO < 5 min** | Aurora Global Database can promote the secondary Region to primary in **under 1 minute** — well within the 5-minute RTO |
| **RPO < 1 min** | Aurora Global Database uses **storage-level replication** with typical cross-Region replication lag of **< 1 second**; the guaranteed RPO is < 1 minute |
| **Cross-Region** | Purpose-built for cross-Region disaster recovery — a single Aurora global database spans multiple Regions with one primary and up to 5 secondary Regions |
| **Managed** | Fully managed — no need to manage replication yourself; no EC2 patching or MySQL binary log configuration |

**Why this is the only viable option (SAP-C02 lens):**

Aurora Global Database is the **only AWS service** that can deliver sub-minute RPO across Regions with managed failover. It replicates at the **storage layer** (below the database engine) using dedicated infrastructure — this is fundamentally faster than logical replication (binlog-based) used by RDS read replicas. At SAP-C02 level, you must know that:

- Aurora Global DB failover time: typically **< 1 minute** (RTO)
- Aurora Global DB replication lag: typically **< 1 second** (RPO measured in ms)
- RDS cross-Region read replica replication lag: typically **minutes**, can spike to hours under load

### ❌ Incorrect Options

**A. Amazon RDS for MySQL with cross-Region read replica**

- RDS cross-Region read replicas use **asynchronous logical replication** (MySQL binlog replication)
- Replication lag is **not guaranteed** — can be minutes to hours depending on write volume and network conditions
- To fail over, you must **manually promote** the read replica to primary — this process takes **minutes** (breaking the < 5 min RTO)
- RPO is **unbounded** — data in transit during a disaster is lost; you can only measure it as "seconds to minutes behind" with no SLA
- ❌ Cannot meet RPO < 1 minute consistently

**C. Amazon RDS for MySQL with Multi-AZ deployment**

- Multi-AZ provides **synchronous replication within the SAME Region** (to a standby in a different AZ)
- This solves AZ-level failures, **NOT cross-Region** disasters
- Multi-AZ does NOT replicate data to `us-west-2` at all
- If `us-east-1` experiences a Region-wide outage, the Multi-AZ standby is also unavailable
- ❌ Does not address the cross-Region requirement at all — fundamental misunderstanding of Multi-AZ scope

**D. MySQL standby database on an Amazon EC2 instance in us-west-2**

- This is basically the **same self-managed approach** they already have — just in another Region
- The solutions architect must manually configure and maintain MySQL replication
- RTO requires manual intervention to promote standby → exceeds 5 minutes
- RPO depends on replication lag you manage yourself — no SLA
- Self-managed EC2 database adds **operational burden** (patching, backups, monitoring, failover scripting)
- ❌ Contradicts the direction of reducing operational complexity; can't guarantee RPO/RTO targets

### 🏭 Enterprise Hands-On Knowledge

1. **Aurora Global DB vs. RDS Cross-Region Read Replica — When to Choose Which:**
   - **Aurora Global DB**: When RPO < 1 min and RTO < 5 min are required (this exam scenario). Also when you need managed, headless cross-Region failover.
   - **RDS Read Replica**: When RPO of minutes-to-hours is acceptable and cost is the primary concern. RDS read replicas cost ~$0.02/GB for cross-Region data transfer.

2. **Aurora Global DB Limitations to Know for the Exam:**
   - Maximum 5 secondary Regions
   - Only supports Aurora MySQL (5.7+) and Aurora PostgreSQL (11.9+)
   - Secondary Regions are **read-only** — writes only go to the primary
   - You can't stop a global cluster's replication without removing the secondary

3. **Failover Process in Production:**
   - Use **RDS console, CLI, or API** to initiate a managed cross-Region failover
   - Typical failover time: **under 1 minute** for Aurora MySQL, under 2 minutes for Aurora PostgreSQL
   - After failover, the old primary becomes a secondary (if still reachable)
   - DNS endpoint remains the same — applications connect to the cluster endpoint, not a Region-specific endpoint

4. **Route 53 DNS Failover Integration:** The "health check + DNS failover" already configured for the app tier should point to the Aurora cluster endpoint. Combined with Aurora Global DB managed failover, you get a fully automated DR solution. However, in practice, you may want to control database failover separately from application failover to prevent split-brain scenarios.
