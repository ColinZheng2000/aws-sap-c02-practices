A company has an organization in AWS Organizations that has a large number of AWS accounts. One of the AWS accounts is designated as a transit account and has a transit gateway that is shared with all of the other AWS accounts. AWS Site-to-Site VPN connections are configured between all of the company's global offices and the transit account. The company has AWS Config enabled on all of its accounts.  
The company's networking team needs to centrally manage a list of internal IP address ranges that belong to the global offices. Developers will reference this list to gain access to their applications securely.  
Which solution meets these requirements with the LEAST amount of operational overhead?  

- A. Create a JSON file that is hosted in Amazon S3 and that lists all of the internal IP address ranges. Configure an Amazon Simple Notification Service (Amazon SNS) topic in each of the accounts that can be invoked when the JSON file is updated. Subscribe an AWS Lambda function to the SNS topic to update all relevant security group rules with the updated IP address ranges.
- B. Create a new AWS Config managed rule that contains all of the internal IP address ranges. Use the rule to check the security groups in each of the accounts to ensure compliance with the list of IP address ranges. Configure the rule to automatically remediate any noncompliant security group that is detected.
- C. In the transit account, create a VPC prefix list with all of the internal IP address ranges. Use AWS Resource Access Manager to share the prefix list with all of the other accounts. Use the shared prefix list to configure security group rules in the other accounts.
- D. In the transit account, create a security group with all of the internal IP address ranges. Configure the security groups in the other accounts to reference the transit account's security group by using a nested security group reference of "transit-account-id/sg-1a2b3c4d".

## Analysis

### Understanding the Problem

The core requirement is to find a **centrally managed** and **low-overhead** method for maintaining a list of on-premises IP address ranges. This list must be **usable by developers** in many different AWS accounts to configure their **security group rules**.

The environment is a multi-account setup using AWS Organizations, with a central "transit account" handling connectivity to on-premises offices via VPN and a shared Transit Gateway. This setup is a standard "hub-and-spoke" network architecture. The key constraint is achieving this with the **least operational overhead**, which strongly implies using an AWS managed service designed for this purpose rather than building a custom solution.

### Services Explanation

* **VPC Managed Prefix Lists:** A VPC prefix list is a managed object that represents a collection of one or more CIDR blocks. You can create a prefix list of IP addresses that you frequently use and reference it as a single unit in security group rules and route tables. This greatly simplifies rule management. When you need to update the IP ranges (e.g., a new office comes online), you modify the *prefix list* only. All security groups and route tables that reference this list are updated automatically.
* **AWS Resource Access Manager (RAM):** This service allows you to securely share specific AWS resources with any AWS account or within your AWS Organization. This is the standard mechanism for sharing resources like Transit Gateways, Subnets, and—critically for this question—VPC Managed Prefix Lists from a central account to other member accounts.
* **AWS Organizations:** A service for centrally managing and governing multiple AWS accounts. It's the umbrella under which RAM operates to share resources easily.
* **Amazon S3, SNS, and Lambda (Option A):** This combination represents a custom, event-driven automation solution. S3 stores a file, SNS broadcasts a change, and Lambda executes code (e.g., to update security groups).
* **AWS Config (Option B):** This is a service for assessing, auditing, and evaluating the configurations of your AWS resources. It primarily handles *compliance checking* (is this resource configured correctly?) and can be set up for *auto-remediation* (fix this non-compliant resource). It is not designed to be a configuration-distribution system.

### Correct Option(s) Explanation

**C. In the transit account, create a VPC prefix list with all of the internal IP address ranges. Use AWS Resource Access Manager to share the prefix list with all of the other accounts. Use the shared prefix list to configure security group rules in the other accounts.**

This is the correct solution because it was **purpose-built for this exact use case** and represents the **least operational overhead**.

1.  **Central Management:** You create and manage *one* prefix list in the central transit account. This is the single source of truth for all on-premises IP ranges.
2.  **Low Operational Overhead:** When a new office is added or an IP range changes, the networking team updates *only* this single prefix list. There is no code to write, no pipelines to run, and no services to manage in each member account. The change propagates automatically.
3.  **Easy for Developers:** Developers in member accounts don't need to know the 50 different IP ranges. They simply reference the shared prefix list ID (e.g., `pl-012345abcdef`) in their security group rules. This also helps with security group rule limits, as one prefix list reference counts as only *one* rule, even if it contains 100 CIDR blocks.
4.  **Uses AWS RAM:** Sharing the prefix list via AWS RAM is the standard, secure way to make a resource from one account available to all other accounts in an AWS Organization.

As the official AWS documentation states: "You can share your managed prefix lists with other AWS accounts using AWS Resource Access Manager (RAM). ... After you share a prefix list, participants can reference the prefix list in their resources, suchas security group rules and route tables."

* **Reference:** [Share a prefix list - Amazon VPC User Guide](https://docs.aws.amazon.com/vpc/latest/userguide/managed-prefix-lists-sharing.html)

### Incorrect Options Explanation

* **A. Create a JSON file that is hosted in Amazon S3...**
    This is a custom-built, complex solution that creates **significant operational overhead**. It requires deploying and maintaining an SNS topic and a Lambda function in *every single account*. This is brittle, hard to manage at scale, and requires custom code. It directly violates the "least amount of operational overhead" requirement when a managed service (Prefix Lists) exists.

* **B. Create a new AWS Config managed rule...**
    This misunderstands the purpose of AWS Config. Config is for *auditing compliance* (e.g., "Alert me if any security group *doesn't* have these IPs"), not for *distributing the list* for developers to use in the first place. This solution is reactive (fix-on-detect) rather than proactive (provide the correct configuration). It also doesn't solve the problem of *how* the developer references the list; they would still need to get the IPs from somewhere, making this an incomplete solution.

* **D. In the transit account, create a security group with all of the internal IP address ranges...**
    This solution is **technically flawed**. When you reference another security group ID in a rule (e.g., `<account-id>/sg-12345`), you are allowing traffic from **resources** (like EC2 instances or ENIs) that are *associated with* that security group. You are **not** inheriting the IP-based rules *from* that security group. The traffic from the on-premises offices originates from the *on-premises IP addresses*, not from an ENI in the transit account that has this new security group attached. Therefore, this rule would not work to allow traffic from the offices.
