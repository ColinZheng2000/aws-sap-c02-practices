A company has Linux-based Amazon EC2 instances. Users must access the instances by using SSH with EC2 SSH key pairs. Each machine requires a unique EC2 key pair.

The company wants to implement a key rotation policy that will, upon request, automatically rotate all the EC2 key pairs and keep the keys in a securely encrypted place. The company will accept less than 1 minute of downtime during key rotation.

Which solution will meet these requirements?

A. Store all the keys in AWS Secrets Manager. Define a Secrets Manager rotation schedule to invoke an AWS Lambda function to generate new key pairs. Replace public keys on EC2 instances. Update the private keys in Secrets Manager.
B. Store all the keys in Parameter Store, a capability of AWS Systems Manager, as a string. Define a Systems Manager maintenance window to invoke an AWS Lambda function to generate new key pairs. Replace public keys on EC2 instances. Update the private keys in Parameter Store.
C. Import the EC2 key pairs into AWS Key Management Service (AWS KMS). Configure automatic key rotation for these key pairs. Create an Amazon EventBridge scheduled rule to invoke an AWS Lambda function to initiate the key rotation in AWS KMS.
D. Add all the EC2 instances to Fleet Manager, a capability of AWS Systems Manager. Define a Systems Manager maintenance window to issue a Systems Manager Run Command document to generate new key pairs and to rotate public keys to all the instances in Fleet Manager.

## Understanding the Problem
### a) Paraphrase what is being asked
You need an automated, on-demand way to rotate the SSH credentials used to access many Linux EC2 instances (each instance has its own key pair), store the private keys securely encrypted, and perform the rotation with <1 minute disruption.

### b) Key requirements, constraints, and optimizations
- **Automated rotation across fleet** (triggerable “upon request”).
- **Secure encrypted storage** of private keys (centralized, auditable).
- **Less than 1 minute downtime** during rotation (ideally no downtime; at most brief access interruption).
- **Unique key pair per instance**.
- Operational excellence: scalable fleet operations, minimal manual steps, strong audit trail.

## Services Explanation
### a) Services involved in the options
- **AWS Secrets Manager**: managed secret storage with KMS encryption, fine-grained IAM, rotation framework (Lambda-based), audit via CloudTrail. Good for credentials that need rotation workflows.
- **AWS Systems Manager Parameter Store**: configuration/secret storage; **SecureString** parameters are encrypted with KMS. No native “rotation framework” like Secrets Manager, but can be automated with SSM Automation/Maintenance Windows/Lambda.
- **Systems Manager Maintenance Windows**: schedule/trigger operational tasks (Run Command/Automation/Lambda) across fleets with targeting (tags, resource groups).
- **Systems Manager Run Command**: executes commands/documents on managed instances without SSH; great for changing `~/.ssh/authorized_keys` safely at scale.
- **Systems Manager Fleet Manager**: console experience for managing instances; not required for automation; it’s an interface layered on SSM.
- **AWS KMS**: key management for encryption keys. It does **not** store/rotate arbitrary SSH key pairs; KMS rotation refers to **KMS keys**, not EC2 SSH keys.
- **Amazon EventBridge**: scheduler/event bus; can trigger Lambda/SSM/Automation.

### b) Real-world usage (regulated domains)
- Preferred pattern is to **avoid long-lived SSH keys** and instead use SSM Session Manager / IAM-based access. But the requirement explicitly mandates SSH key pairs, so we implement controlled rotation + encrypted storage + auditability.
- Regulated environments commonly require: **KMS encryption**, **least privilege IAM**, **CloudTrail logs**, and operational automation via **SSM** to reduce human access and prove change control.

### c) Relevant features
- Secrets Manager: encryption with KMS CMKs, rotation Lambda, version staging (`AWSCURRENT`, `AWSPREVIOUS`).
- Parameter Store: must use **SecureString** (not plain String) for encrypted storage.
- SSM Run Command can update `authorized_keys` with essentially **zero downtime** (no reboot), especially if you temporarily allow both old and new keys during cutover.

## Correct Option(s) Explanation
### a) Correct option(s)  **A**

### b) Why this is the best answer
- **Secrets Manager** is the most purpose-built service here for “store sensitive material + rotate automatically” with an established rotation workflow.
- A Lambda-driven rotation can:
  1) generate a new SSH key pair per instance,
  2) push the new **public** key to each instance (via SSM Run Command/SSH—SSM preferred),
  3) update the stored **private** key atomically in Secrets Manager,
  4) optionally keep the previous key briefly to meet the <1 minute disruption requirement (dual-key period).
- This meets “securely encrypted place” (Secrets Manager + KMS) and minimizes operational burden compared to building your own secret lifecycle.

### c) Pillar connections
i. **High availability/fault tolerance**: rotation can be done in batches with retries; dual-key strategy avoids lockouts.  
ii. **Cost optimization**: Secrets Manager costs more than Parameter Store, but reduces engineering/ops overhead and rotation risk (often the larger enterprise cost).  
iii. **Security/compliance/governance**: KMS-encrypted secrets, IAM access controls, CloudTrail auditing; clear rotation events.  
iv. **Operational excellence/automation**: native rotation framework + Lambda; can be invoked on-demand (API/console) rather than only scheduled.

### d) AWS references
- **AWS Secrets Manager User Guide – Rotate secrets**: describes Lambda-based rotation schedules and versioning.
- **AWS Systems Manager User Guide – Run Command / Maintenance Windows**: fleet operations execution model.
- **AWS Key Management Service Developer Guide – Key rotation**: clarifies rotation applies to KMS keys, not arbitrary SSH keys.

## Incorrect Options Explanation
### B
- Parameter Store as a **plain string** is not acceptable for secure encrypted storage; it must be **SecureString** with KMS.
- Even with SecureString, Parameter Store lacks Secrets Manager’s rotation framework/version staging; you’d be building more logic yourself (higher operational risk).
- Could be viable if rewritten as: SecureString + SSM Automation/Run Command + careful versioning, but it’s not as clean or “MOST” fitting for rotation.

### C
- KMS cannot “import EC2 key pairs” and rotate them as SSH key pairs. KMS manages **encryption keys**, not SSH identity keys used by OpenSSH/EC2 key pairs.
- Automatic rotation in KMS rotates KMS key material, not your stored SSH private keys.

### D
- Fleet Manager is not the mechanism that performs rotation; it’s mainly a UI on top of SSM. Also, generating/storing **private keys** securely is not addressed.
- Run Command can rotate **public keys** on instances, but you still need a secure encrypted repository + versioning/access controls for private keys (Secrets Manager/Parameter Store SecureString).
- To make viable: pair SSM Run Command with Secrets Manager (which essentially becomes option A’s pattern).
