A company’s solutions architect is reviewing a web application that runs on AWS. The application references static assets in an Amazon S3 bucket in the us-east-1 Region. The company needs resiliency across multiple AWS Regions. The company already has created an S3 bucket in a second Region.  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Configure the application to write each object to both S3 buckets. Set up an Amazon Route 53 public hosted zone with a record set by using a weighted routing policy for each S3 bucket. Configure the application to reference the objects by using the Route 53 DNS name.
- B. Create an AWS Lambda function to copy objects from the S3 bucket in us-east-1 to the S3 bucket in the second Region. Invoke the Lambda function each time an object is written to the S3 bucket in us-east-1. Set up an Amazon CloudFront distribution with an origin group that contains the two S3 buckets as origins.
- C. Configure replication on the S3 bucket in us-east-1 to replicate objects to the S3 bucket in the second Region. Set up an Amazon CloudFront distribution with an origin group that contains the two S3 buckets as origins.
- D. Configure replication on the S3 bucket in us-east-1 to replicate objects to the S3 bucket in the second Region. If failover is required, update the application code to load S3 objects from the S3 bucket in the second Region.

## Analysis

### Understanding the Problem

The core requirement is to achieve **cross-Region resiliency** for **static assets** stored in **Amazon S3** for a web application, while maintaining the **LEAST operational overhead**.

- **Current State:** Static assets are in an S3 bucket in _us-east-1_. A second S3 bucket exists in a different Region.
    
- **Goal:** The application must be able to access the assets even if one AWS Region becomes unavailable (resiliency).
    
- **Constraint:** The solution must minimize operational overhead (i.e., prefer managed services and declarative configurations over custom code/manual steps).
    

This problem involves two main aspects:

1. **Replicating** the static assets to the second S3 bucket.
    
2. Providing a **single, resilient endpoint** for the application to access the assets.

### Correct Option(s) Explanation

**The correct option is C.**

**C. Configure replication on the S3 bucket in us-east-1 to replicate objects to the S3 bucket in the second Region. Set up an Amazon CloudFront distribution with an origin group that contains the two S3 buckets as origins.**

1. **Replication:** **S3 Cross-Region Replication (CRR)** is the AWS-native, fully managed, declarative way to copy objects between Regions. This meets the requirement for data redundancy **with the LEAST operational overhead**, as it requires only configuration (a replication rule on the source bucket) and no custom code or compute management.
    
    - _Reference:_ AWS S3 Documentation states that CRR is a fully managed feature.
        
2. **Resilient Endpoint:** **Amazon CloudFront with an Origin Group** is the most efficient and low-overhead way to provide a single, highly available endpoint that can fail over between the two S3 buckets (Regions). The **Origin Group** feature is specifically designed to perform **origin failover** by automatically switching from the primary origin (e.g., S3 us-east-1) to the secondary origin (e/g., S3 second Region) if it detects a failure.
    
    - _Reference:_ AWS CloudFront Documentation on Origin Groups confirms this functionality.
        

This combined solution uses two managed services configured for automatic operation, resulting in the **least operational overhead** for both data synchronization and high-availability access.

### Incorrect Options Explanation

|Option|Reason for being Incorrect|
|---|---|
|**A. Configure the application to write each object to both S3 buckets. Set up an Amazon Route 53 public hosted zone with a record set by using a weighted routing policy for each S3 bucket. Configure the application to reference the objects by using the Route 53 DNS name.**|**High Operational Overhead (Replication):** Writing custom application logic to handle dual-writes for replication significantly increases operational overhead. This includes managing potential failures in one write, object consistency checks, and retries. **Incorrect Resiliency Mechanism:** A **Weighted Routing Policy** is used for load balancing or A/B testing, not for immediate failover based on service availability. While Route 53 does have a _Failover Routing Policy_ that can check health, it is less immediate for asset delivery than CloudFront's Origin Group failover and would not provide the CDN benefits (caching, performance) of CloudFront.|
|**B. Create an AWS Lambda function to copy objects from the S3 bucket in us-east-1 to the S3 bucket in the second Region. Invoke the Lambda function each time an object is written to the S3 bucket in us-east-1. Set up an Amazon CloudFront distribution with an origin group that contains the two S3 buckets as origins.**|**High Operational Overhead (Replication):** Using an **AWS Lambda function** for replication introduces custom code to write, test, maintain, and monitor. This is significantly higher operational overhead compared to the built-in **S3 Cross-Region Replication (CRR)** feature. While the CloudFront part is correct for the resilient endpoint, the custom replication solution violates the "LEAST operational overhead" constraint.|
|**D. Configure replication on the S3 bucket in us-east-1 to replicate objects to the S3 bucket in the second Region. If failover is required, update the application code to load S3 objects from the S3 bucket in the second Region.**|**High Operational Overhead (Failover):** This solution correctly uses S3 Replication (low overhead) for data synchronization. However, requiring a **manual update of the application code** to change the asset source _if failover is required_ is a time-consuming, manual step with high operational risk and overhead during a disaster. It prevents automatic, transparent failover, which is required for a highly resilient architecture. A single, resilient endpoint (like CloudFront or Route 53 Failover DNS) is necessary to avoid this manual intervention.|
