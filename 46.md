A company with several AWS accounts is using AWS Organizations and service control policies (SCPs). An Administrator created the following SCP and has attached it to an organizational unit (OU) that contains AWS account 1111-1111-1111:  
![](https://www.examtopics.com/assets/media/exam-media/04241/0029700001.png)  
Developers working in account 1111-1111-1111 complain that they cannot create Amazon S3 buckets. How should the Administrator address this problem?  

- A. Add s3:CreateBucket with "Allow" effect to the SCP.
- B. Remove the account from the OU, and attach the SCP directly to account 1111-1111-1111.
- C. Instruct the Developers to add Amazon S3 permissions to their IAM entities.
- D. Remove the SCP from account 1111-1111-1111.

### Understanding the Problem

The company is using **AWS Organizations** and **Service Control Policies (SCPs)** to manage permissions across multiple AWS accounts. An SCP is attached to an **Organizational Unit (OU)**, which contains account **1111-1111-1111**. This SCP includes an **explicit deny** for all `cloudtrail:*` actions and an **explicit allow** for all other actions (`Action: "*"`)—which is effectively redundant because SCPs default to an **implicit allow** of all actions unless explicitly denied.

**The core issue:** Developers in account 1111-1111-1111 are **unable to create Amazon S3 buckets**, even though the SCP appears to allow all actions _except_ those related to CloudTrail.

The goal is to determine the correct administrative action to resolve the developers' inability to create S3 buckets.

### Correct Option(s) Explanation

**C. Instruct the Developers to add Amazon S3 permissions to their IAM entities.**

- **SCPs do not grant permissions.** This is the crucial concept. SCPs only define a **permission boundary**—the _maximum_ set of actions an account's IAM users/roles can perform.
    
- The provided SCP allows all actions (`Action: "*"`) except `cloudtrail:*`. This means the permission boundary _does_ include `s3:CreateBucket`.
    
- However, for a developer to actually perform the `s3:CreateBucket` action, their specific **IAM user or role must be granted that permission via an IAM policy** (either attached directly, or via a group/role).
    
- Since the developers are complaining they cannot create S3 buckets, and the SCP is not blocking it (it explicitly allows it via `"*"`), the missing piece is the actual **IAM policy grant**.
    

**AWS Official Documentation Reference:**

> "SCPs are not applicable to service-linked roles. SCPs do not grant permissions. An SCP only filters permissions."
> 
> "For an action to be allowed, it must be allowed by both the SCP and the IAM policy."

The IAM entity **must** have an **explicit Allow** for `s3:CreateBucket` in their IAM policy.

---

### Incorrect Options Explanation

**A. Add `s3:CreateBucket` with `Allow` effect to the SCP.**

- **SCPs do not grant permissions.** Adding an explicit `Allow` for `s3:CreateBucket` to the SCP is unnecessary and will not solve the problem. The existing statement `"Action": "*"` already provides the maximum boundary for S3 actions (i.e., it doesn't deny them). The missing component is the actual IAM grant.
    

**B. Remove the account from the OU, and attach the SCP directly to account 1111-1111-1111.**

- This is an unnecessary administrative step. SCPs apply the same way whether attached to an **OU** or directly to an **Account**. The _evaluation logic_ is unaffected by the attachment method. The problem is a missing IAM grant, not a flawed attachment method.
    

**D. Remove the SCP from account 1111-1111-1111.**

- Removing the SCP would essentially set the permission boundary back to the AWS defaults (all actions allowed). While this would still allow the `s3:CreateBucket` action (since the default allows everything), it would also eliminate the necessary **explicit Deny** for CloudTrail, which the administrator _intended_ to implement. More importantly, it still doesn't _grant_ the missing `s3:CreateBucket` permission, which the IAM policy must provide. The correct approach is to check the IAM policies, not remove the organizational boundary control.
