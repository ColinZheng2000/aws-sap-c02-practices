A retail company is hosting an ecommerce website on AWS across multiple AWS Regions. The company wants the website to be operational at all times for online purchases. The website stores data in an Amazon RDS for MySQL DB instance.  
Which solution will provide the HIGHEST availability for the database?  

- A. Configure automated backups on Amazon RDS. In the case of disruption, promote an automated backup to be a standalone DB instance. Direct database traffic to the promoted DB instance. Create a replacement read replica that has the promoted DB instance as its source.
- B. Configure global tables and read replicas on Amazon RDS. Activate the cross-Region scope. In the case of disruption, use AWS Lambda to copy the read replicas from one Region to another Region.
- C. Configure global tables and automated backups on Amazon RDS. In the case of disruption, use AWS Lambda to copy the read replicas from one Region to another Region.
- D. Configure read replicas on Amazon RDS. In the case of disruption, promote a cross-Region and read replica to be a standalone DB instance. Direct database traffic to the promoted DB instance. Create a replacement read replica that has the promoted DB instance as its source.

## Understanding the Problem

The retail company is running a critical e-commerce website on AWS across **multiple AWS Regions** and requires the **HIGHEST availability** for its **Amazon RDS for MySQL DB instance** to ensure continuous online purchasing capability. The core requirement is a solution that provides the fastest and most robust recovery mechanism for the database in the event of a disruption, leveraging the multi-Region deployment for maximum resilience.

## Services Explanation

|**Service**|**Description**|**Official AWS Document/Concept**|
|---|---|---|
|**Amazon RDS (Relational Database Service)**|A managed service that makes it easy to set up, operate, and scale a relational database in the cloud. It manages administrative tasks like patching, backup, recovery, and scaling.|[Amazon RDS Documentation](https://aws.amazon.com/rds/)|

## Correct Option(s) Explanation

The correct option is **D**.

**D. Configure read replicas on Amazon RDS. In the case of disruption, promote a cross-Region and read replica to be a standalone DB instance. Direct database traffic to the promoted DB instance. Create a replacement read replica that has the promoted DB instance as its source.**

This solution offers the **HIGHEST availability** and aligns perfectly with the standard **Disaster Recovery (DR) strategy for Amazon RDS across multiple Regions**:

1. **Cross-Region Read Replicas:** The best and most common way to maintain a continuously updated, warm standby database in a different Region is by using a **cross-Region Read Replica**. This provides a significantly lower Recovery Point Objective (RPO) and Recovery Time Objective (RTO) than restoring from a backup, as the data is constantly being replicated.
    
2. **Promotion:** Upon disruption of the primary DB instance, the cross-Region Read Replica can be **promoted** to a new standalone primary DB instance. This promotion is a native, fast, and reliable RDS operation, minimizing downtime (low RTO).
    
3. **Restoring Architecture:** The final step of creating a replacement Read Replica ensures the multi-Region, highly available architecture is restored, making the new primary resilient again.
    

This mechanism is explicitly designed in AWS for high-availability and disaster recovery and is typically the recommended approach over backup restores for critical workloads.

## Incorrect Options Explanation

|**Option**|**Reason for Incorrectness**|
|---|---|
|**A. Configure automated backups on Amazon RDS. In the case of disruption, promote an automated backup to be a standalone DB instance. Direct database traffic to the promoted DB instance. Create a replacement read replica that has the promoted DB instance as its source.**|**Automated backups are insufficient for the "HIGHEST availability" multi-Region requirement.** Restoring from a backup (even a cross-Region one) takes significantly longer (higher **RTO**) than promoting a continuously replicated Read Replica. Also, restoring an automated backup is done by creating a _new_ DB instance, not by _promoting_ the backup itself.|
|**B. Configure global tables and read replicas on Amazon RDS. Activate the cross-Region scope. In the case of disruption, use AWS Lambda to copy the read replicas from one Region to another Region.**|**Amazon RDS does not use "Global Tables" or "cross-Region scope" in the way described; this terminology is for Amazon DynamoDB.** Furthermore, using **AWS Lambda to copy Read Replicas** from one Region to another is incorrect. Read Replicas are already an RDS feature and are replicated natively, not via a manual process orchestrated by Lambda.|
|**C. Configure global tables and automated backups on Amazon RDS. In the case of disruption, use AWS Lambda to copy the read replicas from one Region to another Region.**|This option is incorrect for the same reasons as B. It incorrectly mixes terminology (DynamoDB "Global Tables") and proposes a completely non-standard, manual, and inefficient method for cross-Region data management (**Lambda to copy Read Replicas**). Automated backups alone also offer a lower level of availability compared to continuously replicated Read Replicas.|
