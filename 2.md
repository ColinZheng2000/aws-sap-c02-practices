A company is providing weather data over a REST-based API to several customers. The API is hosted by Amazon API Gateway and is integrated with different AWS Lambda functions for each API operation. The company uses Amazon Route 53 for DNS and has created a resource record of weather.example.com. The company stores data for the API in Amazon DynamoDB tables. The company needs a solution that will give the API the ability to fail over to a different AWS Region.  
Which solution will meet these requirements?

- A. Deploy a new set of Lambda functions in a new Region. Update the API Gateway API to use an edge-optimized API endpoint with Lambda functions from both Regions as targets. Convert the DynamoDB tables to global tables.
- B. Deploy a new API Gateway API and Lambda functions in another Region. Change the Route 53 DNS record to a multivalue answer. Add both API Gateway APIs to the answer. Enable target health monitoring. Convert the DynamoDB tables to global tables.
- C. Deploy a new API Gateway API and Lambda functions in another Region. Change the Route 53 DNS record to a failover record. Enable target health monitoring. Convert the DynamoDB tables to global tables.
- D. Deploy a new API Gateway API in a new Region. Change the Lambda functions to global functions. Change the Route 53 DNS record to a multivalue answer. Add both API Gateway APIs to the answer. Enable target health monitoring. Convert the DynamoDB tables to global tables.
## Understanding the Problem

The company needs to implement a **multi-Region disaster recovery solution** for its weather data API. The current architecture consists of an API hosted on **Amazon API Gateway**, backed by **AWS Lambda functions**, and using **Amazon DynamoDB** for data storage. **Amazon Route 53** manages the DNS record. The key requirement is for the API to have the **ability to fail over to a different AWS Region** in case of an outage in the primary Region. This requires not only deploying the application stack (API Gateway and Lambda) in a second Region but also ensuring **data synchronization** (DynamoDB) and **traffic routing/failover** (Route 53).

## Services Explanation

### 1. Amazon Route 53

Route 53 is a highly available and scalable cloud Domain Name System (DNS) web service.

- **Failover Routing Policy:** Allows you to route traffic to a resource in a secondary Region when the resource in the primary Region is unavailable (unhealthy). This relies on health checks.
    
- **Multivalue Answer Routing Policy:** Lets you return multiple IP addresses for the same resource, optionally combined with health checks. It's used for load balancing or distributing traffic, but it isn't designed for a deterministic active-passive failover scenario.
    

### 2. Amazon API Gateway

API Gateway is a fully managed service that makes it easy for developers to create, publish, maintain, monitor, and secure APIs at any scale.

- **Regional API Endpoint:** The best choice for a multi-Region setup where you manage DNS failover externally with Route 53.
    
- **Edge-Optimized API Endpoint:** Uses Amazon CloudFront to improve client access latency from geographically diverse locations. It's an internal-to-AWS solution for performance optimization, not a primary mechanism for active-passive disaster recovery failover across distinct AWS Regions.
    

### 3. AWS Lambda

Lambda lets you run code without provisioning or managing servers. Functions are **Region-specific** and must be deployed in each Region where the API will operate.

### 4. Amazon DynamoDB

DynamoDB is a fast and flexible NoSQL database service.

- **Global Tables:** A feature that provides a fully managed, multi-Region, active-active database. It automatically replicates data between the specified AWS Regions, which is essential for a disaster recovery solution where the secondary Region needs up-to-date data. **(Reference: AWS Documentation - Amazon DynamoDB Global Tables)**

## Correct Option Explanation

The correct solution is **C. Deploy a new API Gateway API and Lambda functions in another Region. Change the Route 53 DNS record to a failover record. Enable target health monitoring. Convert the DynamoDB tables to global tables.**

1. **Deploy New Stack (API Gateway and Lambda):** The application must be deployed in the secondary Region to provide the failover endpoint. API Gateway endpoints and Lambda functions are **Regional resources**, so a new, complete stack must be set up.
    
2. **Convert DynamoDB to Global Tables:** This is the correct and necessary step to ensure that the data is automatically replicated and available in the secondary Region with low latency for the failover API stack. **(Reference: AWS Documentation - Best practices for implementing Amazon DynamoDB Global Tables)**
    
3. **Route 53 Failover Record and Health Monitoring:** The **Failover routing policy** in Route 53 is the definitive mechanism for an **Active-Passive disaster recovery** scenario.
    
    - The **Primary record** points to the API Gateway in the primary Region.
        
    - The **Secondary record** points to the API Gateway in the secondary Region.
        
    - **Health checks** are enabled on the primary endpoint. If the health check fails, Route 53 automatically routes all traffic to the secondary endpoint, meeting the explicit requirement for API failover. **(Reference: AWS Documentation - Configuring failover in Amazon Route 53)**

## Incorrect Options Explanation

### A. Deploy a new set of Lambda functions in a new Region. Update the API Gateway API to use an edge-optimized API endpoint with Lambda functions from both Regions as targets. Convert the DynamoDB tables to global tables.

- **Incorrect API Gateway Setup:** **Edge-optimized API endpoints** cannot natively integrate a single API endpoint with backend Lambda functions from **multiple, distinct AWS Regions** as part of a failover mechanism. The standard approach for a multi-Region, Active-Passive failover is to deploy two separate **Regional API endpoints** and use **Route 53 Failover routing**.
    

### B. Deploy a new API Gateway API and Lambda functions in another Region. Change the Route 53 DNS record to a multivalue answer. Add both API Gateway APIs to the answer. Enable target health monitoring. Convert the DynamoDB tables to global tables.

- **Incorrect Route 53 Policy:** The **Multivalue answer routing policy** is primarily for returning multiple healthy resources (often for client-side load distribution) and does not provide a deterministic, ordered **Active-Passive failover** mechanism. It randomly returns one or more healthy addresses and is not the standard pattern for a controlled disaster recovery switchover. The **Failover routing policy** is the correct choice for this requirement.
    

### D. Deploy a new API Gateway API in a new Region. Change the Lambda functions to global functions. Change the Route 53 DNS record to a multivalue answer. Add both API Gateway APIs to the answer. Enable target health monitoring. Convert the DynamoDB tables to global tables.

- **Incorrect Lambda concept:** There is **no such service as "global functions"** for AWS Lambda. Lambda functions are inherently **Regional resources** and must be deployed separately in each AWS Region.
    
- **Incorrect Route 53 Policy:** As explained in option B, the **Multivalue answer routing policy** is inappropriate for a controlled Active-Passive failover scenario.
