A company has a project that is launching Amazon EC2 instances that are larger than required. The project's account cannot be part of the company's organization in AWS Organizations due to policy restrictions to keep this activity outside of corporate IT. The company wants to allow only the launch of t3.small EC2 instances by developers in the project's account. These EC2 instances must be restricted to the us-east-2 Region.

What should a solutions architect do to meet these requirements?

A. Create a new developer account. Move all EC2 instances, users, and assets into us-east-2. Add the account to the company's organization in AWS Organizations. Enforce a tagging policy that denotes Region affinity.
B. Create an SCP that denies the launch of all EC2 instances except t3.small EC2 instances in us-east-2. Attach the SCP to the project's account.
C. Create and purchase a t3.small EC2 Reserved Instance for each developer in us-east-2. Assign each developer a specific EC2 instance with their name as the tag.
D. Create an IAM policy than allows the launch of only t3.small EC2 instances in us-east-2. Attach the policy to the roles and groups that the developers use in the project's account.

## Understanding the Problem
A standalone AWS account (cannot join the company’s AWS Organizations) is launching **oversized EC2 instances**. The company wants to:
- Allow developers to launch **only `t3.small`**
- Restrict launches to **Region `us-east-2`**
- Do this **within that single account**, without using Organizations controls.

Key constraint: **No AWS Organizations membership**, so **no SCPs** can be used. Therefore, the only enforceable guardrail available is **IAM-based controls** (identity policies, permission boundaries, session policies), plus optional detective controls.

- --

## Correct Option(s) Explanation
### ✅ D. Create an IAM policy that allows launching only `t3.small` in `us-east-` and attach it to the developer roles/groups
This is the only option that:
- Works **without AWS Organizations**
- Can enforce **instance type restriction** using IAM condition keys
- Can enforce **Region restriction** using IAM global condition keys

* *How it works (policy mechanics)**
- Restrict Region with `aws:RequestedRegion = us-east-2`
- Restrict instance type with `ec2:InstanceType = t3.small`
- Ensure the developer also has permission for required dependent actions (`ec2:RunInstances` also requires permissions for related resources such as `ec2:CreateTags`, `ec2:Describe*`, and sometimes `iam:PassRole` if using instance profiles)

* *Typical enforcement pattern**
- Use **explicit Deny** guardrails (stronger than “Allow-only”) because other policies could grant broader allows.
- Apply via:
  - Identity-based policy on the developer role/group, and/or
  - **Permissions boundary** on developer roles (strong enterprise control inside a single account)

* *Official AWS references**
- IAM policy evaluation logic (why explicit Deny is the strongest control):  
  https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_evaluation-logic.html
- EC2 condition keys (including instance type):  
  https://docs.aws.amazon.com/service-authorization/latest/reference/list_amazonec2.html
- Global condition key for Region (`aws:RequestedRegion`):  
  https://docs.aws.amazon.com/IAM/latest/UserGuide/reference_policies_condition-keys.html#condition-keys-requestedregion

> Exam framing: when SCPs are not possible, the next best preventive control is **IAM policies/permission boundaries**.

## Incorrect Options Explanation
### ❌ A. Add the account to Organizations and enforce tagging policy
- Directly violates the stated constraint: the account **cannot be part of the organization**.
- Tag policies do **not** enforce runtime restrictions like instance type/Region for EC2 launches (tag policies standardize tags; they don’t block API calls).

### ❌ B. Create an SCP and attach it to the project account
- **SCPs only apply to accounts that are members of AWS Organizations**.
- Since the project account cannot join the org, this is impossible.

### ❌ C. Buy Reserved Instances for each developer and assign tagged instances
- Reserved Instances are a **billing discount**, not a control mechanism.
- They do not prevent launching larger instance types or launching in other Regions.

## Hands-on Knowledge (Enterprise-grade Implementation)
### Recommended IAM guardrail (use Deny)
In practice, implement **explicit Deny** for anything outside the guardrails, because “Allow only” can be bypassed if another policy grants broader access.

* *Key controls to include**
- Deny `ec2:RunInstances` if:
  - `aws:RequestedRegion` is not `us-east-2`
  - `ec2:InstanceType` is not `t3.small`

* *Also consider**
- Restrict **Spot/Fleet/Auto Scaling paths if devs can provision compute indirectly:
  - `ec2:RequestSpotInstances`, `ec2:CreateFleet`, `autoscaling:CreateAutoScalingGroup`, `eks:Nodegroup` etc. (depending on environment)
- Control `iam:PassRole` so developers can only pass an approved instance profile role (prevents privilege escalation).

### Use permission boundaries for strong, scalable enforcement
If developers create their own roles/users, attach a **permissions boundary** that enforces:
- Region restriction
- Instance type restriction
This prevents privilege creep inside the account with low operational overhead.

### Add detective controls (optional but common in gov/regulated work)
- `[AWS CloudTrail]` + `[Amazon EventBridge]` rules to alert on noncompliant `RunInstances` attempts
- `[AWS Config]` rules (e.g., instance type allowed list, Region-based controls via aggregator patterns are harder in single account but still useful for compliance reporting)
