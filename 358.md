A company is deploying a new web-based application and needs a storage solution for the Linux application servers. The company wants to create a single location for updates to application data for all instances. The active dataset will be up to 100 GB in size. A solutions architect has determined that peak operations will occur for 3 hours daily and will require a total of 225 MiBps of read throughput.

The solutions architect must design a Multi-AZ solution that makes a copy of the data available in another AWS Region for disaster recovery (DR). The DR copy has an RPO of less than 1 hour.

Which solution will meet these requirements?

A. Deploy a new Amazon Elastic File System (Amazon EFS) Multi-AZ file system. Configure the file system for 75 MiBps of provisioned throughput. Implement replication to a file system in the DR Region.
B. Deploy a new Amazon FSx for Lustre file system. Configure Bursting Throughput mode for the file system. Use AWS Backup to back up the file system to the DR Region.
C. Deploy a General Purpose SSD (gp3) Amazon Elastic Block Store (Amazon EBS) volume with 225 MiBps of throughput. Enable Multi-Attach for the EBS volume. Use AWS Elastic Disaster Recovery to replicate the EBS volume to the DR Region.
D. Deploy an Amazon FSx for OpenZFS file system in both the production Region and the DR Region. Create an AWS DataSync scheduled task to replicate the data from the production file system to the DR file system every 10 minutes.




# AWS SAP-C02 Question Analysis — Q358

## 📋 Analysis

### 1. Understanding the Problem

The workload requires a shared storage platform for multiple Linux application servers. Several requirements immediately narrow the architecture choice:

- Multiple application instances must read/write the same dataset
- Storage must be highly available across multiple AZs
- Peak throughput reaches 225 MiBps for several hours daily
- Cross-Region disaster recovery is mandatory
- RPO must be less than 1 hour
- Operational simplicity is preferred

The most important keyword is effectively “single shared storage location for all Linux instances.” This eliminates many block-storage-centric designs because Linux shared-write access requires a distributed/shared file system.

The solution therefore needs:
- Managed shared filesystem
- Multi-AZ resilience
- Native or near-real-time replication
- Predictable throughput scaling

---

### 2. Options Analysis

#### ✅ Option A (CORRECT): Use Amazon EFS Multi-AZ with provisioned throughput and cross-Region replication

This option satisfies every technical and operational requirement.

Why it works:

- Amazon EFS is a fully managed shared NFS filesystem designed for concurrent Linux client access.
- EFS Standard storage is Regional and automatically stores data redundantly across multiple AZs.
- Provisioned Throughput allows throughput configuration independent of dataset size.
- Native EFS Replication provides asynchronous cross-Region replication with typical lag measured in minutes.
- RPO less than 1 hour is easily achievable.
- No custom synchronization tooling or clustered filesystem management is required.

Throughput reasoning:

- The workload needs 225 MiBps read throughput during peak windows.
- EFS throughput accounting allows reads to be met efficiently because read operations are metered differently from writes.
- Provisioned throughput is appropriate when:
  - the dataset is relatively small
  - throughput demand is high
  - workloads are predictable

Why this is architecturally strong:

- Minimal operational overhead
- Fully managed HA storage
- Native DR capability
- Excellent fit for web application shared content

Relevant AWS documentation:
- https://docs.aws.amazon.com/efs/latest/ug/efs-replication.html
- https://docs.aws.amazon.com/efs/latest/ug/performance.html

---

#### ❌ Option B (INCORRECT): Use FSx for Lustre with AWS Backup

This option does not properly satisfy the DR and workload profile requirements.

Why it fails:

- FSx for Lustre is optimized for HPC, analytics, rendering, and ML workloads.
- It is not the standard solution for shared enterprise web application storage.
- AWS Backup provides periodic backups, not continuous replication.
- Backup-and-restore workflows generally cannot guarantee aggressive RPO targets consistently.
- Recovery operations are slower and operationally more complex than native replication.

Important exam clue:

- Low RPO requirements usually favor replication technologies instead of scheduled backup solutions.

Enterprise reality:

FSx for Lustre is commonly selected for:
- genomics pipelines
- AI/ML training clusters
- financial simulations
- high-performance analytics

It is rarely the first recommendation for shared Linux application content storage.

Relevant documentation:
- https://docs.aws.amazon.com/fsx/latest/LustreGuide/what-is.html
- https://docs.aws.amazon.com/aws-backup/latest/devguide/whatisbackup.html

---

#### ❌ Option C (INCORRECT): Use gp3 EBS Multi-Attach with Elastic Disaster Recovery

This option contains both technical incompatibilities and architectural issues.

Primary failure points:

- gp3 does NOT support Multi-Attach.
- Multi-Attach is only supported on io1/io2 volumes.
- EBS is block storage, not shared filesystem storage.
- Multiple Linux servers sharing block devices requires clustered filesystem software.
- Standard filesystems like ext4/xfs are unsafe for concurrent multi-instance mounting.

Even if io2 Multi-Attach had been used:

- The operational complexity would still be high.
- Cluster-aware filesystem management would be required.
- This is inappropriate for a standard web application shared-data pattern.

Why SAP-C02 cares:

AWS strongly differentiates:
- shared file storage → EFS / FSx
- block storage → EBS

Relevant AWS documentation:
- https://docs.aws.amazon.com/ebs/latest/userguide/ebs-volumes-multi.html

Enterprise insight:

EBS Multi-Attach is generally reserved for:
- clustered databases
- SAP clustered applications
- quorum/shared-disk enterprise software

Not for ordinary web content sharing.

---

#### ❌ Option D (INCORRECT): Use FSx for OpenZFS with DataSync every 10 minutes

This solution partially addresses replication but fails the HA architecture requirement.

Why it fails:

- FSx for OpenZFS is not the best fit for Multi-AZ shared application storage.
- The architecture introduces unnecessary operational complexity through scheduled DataSync tasks.
- Native replication services are preferred over custom synchronization scheduling.
- DataSync synchronization intervals create additional operational monitoring requirements.

Important exam reasoning:

When AWS offers:
- managed HA
- native replication
- lower operational overhead

that choice is usually preferred over custom orchestration approaches.

Additional concern:

- DataSync is excellent for:
  - migrations
  - periodic synchronization
  - hybrid transfer workflows

But not typically the primary DR replication mechanism for cloud-native shared application storage.

Relevant AWS documentation:
- https://docs.aws.amazon.com/fsx/latest/OpenZFSGuide/what-is-fsx-openzfs.html
- https://docs.aws.amazon.com/datasync/latest/userguide/what-is-datasync.html

### 3. Hands-On Knowledge

#### EFS Performance Modes in Real Production

EFS supports multiple throughput models:

1. Bursting Throughput
   - Throughput scales with storage size
   - Small datasets may not sustain large throughput spikes

2. Provisioned Throughput
   - Fixed guaranteed throughput
   - Best for predictable enterprise workloads

3. Elastic Throughput
   - Auto-scales dynamically
   - Often cost-efficient for variable workloads

For SAP-C02:
- predictable throughput requirement + small dataset size → Provisioned Throughput is commonly the expected answer.

---

#### Common Enterprise EFS Architecture

Typical production deployment:

```text
Application EC2 Instances
        │
Auto Scaling Group
        │
EFS Mount Targets (Multi-AZ)
        │
Amazon EFS Regional File System
        │
EFS Replication
        │
Disaster Recovery Region
```

Benefits:
- Shared configuration consistency
- Simplified deployments
- Stateless compute layer
- Easier blue/green deployments
- Simplified DR orchestration

---

#### Useful CLI Examples

Create EFS filesystem:

```bash
aws efs create-file-system \
  --performance-mode generalPurpose \
  --throughput-mode provisioned \
  --provisioned-throughput-in-mibps 75
```

Mount EFS securely:

```bash
sudo mount -t efs -o tls fs-12345678:/ /shared-data
```

Check replication status:

```bash
aws efs describe-replication-configurations
```

---

#### DR Operational Reality

Cross-Region EFS replication characteristics:

- Replication is asynchronous
- DR filesystem remains read-only until failover
- Typical lag is measured in minutes
- Route 53 failover is commonly integrated
- Recovery runbooks should include:
  - DNS failover
  - mount target validation
  - application remount testing

Medical and industrial companies especially care about:
- deterministic recovery procedures
- auditability
- operational simplicity
- compliance-ready HA architectures

EFS aligns very well with those enterprise requirements.

**面试官可能追问 (Possible Follow-up Questions)**：

1. "为什么 EFS 比 EBS 更适合作为共享应用存储？"
   → 回答要点 (Key Answer Points)：EFS 是托管 NFS 文件系统，支持多个 Linux 实例同时安全挂载。EBS 是块存储，即使支持 Multi-Attach，也需要集群文件系统配合，否则会造成文件系统损坏。

2. "EFS Replication 与 AWS Backup 的核心差异是什么？"
   → 回答要点 (Key Answer Points)：EFS Replication 是持续异步复制，RPO 通常为分钟级；AWS Backup 是周期性快照/恢复模式，更适合长期保护与合规，不适合低 RPO 实时灾备场景。
