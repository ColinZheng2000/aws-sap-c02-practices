A company uses AWS Organizations for a multi-account setup in the AWS Cloud. The company's finance team has a data processing application that uses AWS Lambda and Amazon DynamoDB. The company's marketing team wants to access the data that is stored in the DynamoDB table.

The DynamoDB table contains confidential data. The marketing team can have access to only specific attributes of data in the DynamoDB table. The finance team and the marketing team have separate AWS accounts.

What should a solutions architect do to provide the marketing team with the appropriate access to the DynamoDB table?

A. Create an SCP to grant the marketing team's AWS account access to the specific attributes of the DynamoDB table. Attach the SCP to the OU of the finance team.
B. Create an IAM role in the finance team's account by using IAM policy conditions for specific DynamoDB attributes (fine-grained access control). Establish trust with the marketing team's account. In the marketing team's account, create an IAM role that has permissions to assume the IAM role in the finance team's account.
C. Create a resource-based IAM policy that includes conditions for specific DynamoDB attributes (fine-grained access control). Attach the policy to the DynamoDB table. In the marketing team's account, create an IAM role that has permissions to access the DynamoDB table in the finance team's account.
D. Create an IAM role in the finance team's account to access the DynamoDB table. Use an IAM permissions boundary to limit the access to the specific attributes. In the marketing team's account, create an IAM role that has permissions to assume the IAM role in the finance team's account.

Suggest Answer: B

---

## 📋 Analysis

### 1. Understanding the Problem

This is a **cross-account, fine-grained data access** scenario under AWS Organizations:

- **Finance team account:** Owns a DynamoDB table with **confidential data**.
- **Marketing team account:** Needs access to only **specific attributes** of items in that table — not the entire table.
- The two teams are in **separate AWS accounts**.

The core challenge is implementing **attribute-level access control** (also called fine-grained access control) for DynamoDB across account boundaries. The solution must follow the **principle of least privilege** — marketing users should only see the columns/attributes they are authorized to view.

---

### 2. Options Analysis

#### ✅ **Option B (CORRECT): Cross-Account IAM Role + DynamoDB Fine-Grained Policy Conditions**

This implements the standard AWS pattern for cross-account access with attribute-level control:

1. **Finance account (resource owner):**
   - Creates an **IAM role** with an IAM policy that includes **DynamoDB fine-grained access control conditions** — specifically the `dynamodb:Attributes` condition key to restrict which attributes can be accessed (e.g., only allow `GetItem`/`Query` on attributes `product_name`, `price`, but NOT `cost_price` or `profit_margin`).
   - Establishes a **trust relationship** (trust policy) that allows the marketing account's IAM role to assume this role.

2. **Marketing account (consumer):**
   - Creates an **IAM role** that has the `sts:AssumeRole` permission to assume the finance account's role.
   - Marketing team users/applications assume this role, which then assumes the finance account role, gaining temporary credentials with limited attribute access.

3. **DynamoDB Fine-Grained Access Control** is implemented via IAM policy condition keys:
   - `dynamodb:Attributes` — restricts which specific attribute names can be returned in results
   - `dynamodb:LeadingKeys` — restricts access to specific partition key values
   - `dynamodb:Select` — restricts the type of operation (e.g., only `SPECIFIC_ATTRIBUTES`)

**Example IAM Policy (Finance Account Role):**
```json
{
  "Effect": "Allow",
  "Action": ["dynamodb:GetItem", "dynamodb:Query"],
  "Resource": "arn:aws:dynamodb:region:account:table/ConfidentialTable",
  "Condition": {
    "ForAllValues:StringEquals": {
      "dynamodb:Attributes": ["product_name", "product_category", "msrp"]
    }
  }
}
```
This policy allows only `product_name`, `product_category`, and `msrp` attributes to be read — all other attributes (e.g., `cost_price`, `supplier`) are invisible to the marketing team.

#### ❌ **Option A (INCORRECT): SCP to Grant Access**

- **Service Control Policies (SCPs) are restrictive guardrails, not permissive grants.** SCPs define the **maximum available permissions** for accounts and OUs. They do NOT grant permissions — they only limit what can be done.
- Even if a permissive SCP were attached, it cannot implement **attribute-level DynamoDB access control** — SCPs operate at the AWS API level, not at the DynamoDB data-plane attribute level.
- Attaching an SCP to the **finance team's OU** to grant the marketing team access makes no architectural sense — SCPs affect the accounts within the OU they're attached to, not external accounts.
- **Key principle:** SCP + IAM policy together determine effective permissions, where SCP is the upper boundary. An SCP alone never grants access.

#### ❌ **Option C (INCORRECT): Resource-Based IAM Policy on DynamoDB Table**

- **DynamoDB does NOT support resource-based policies.** Unlike S3 (bucket policies), SQS (queue policies), SNS (topic policies), Lambda (resource-based policies), and KMS (key policies), DynamoDB only supports **identity-based IAM policies**.
- There is no way to attach a policy directly to a DynamoDB table. All access control for DynamoDB must be managed through IAM users, groups, or roles (identity-based policies).
- **Exam trap:** Many candidates assume all AWS services support resource-based policies. Know which ones do and don't:
  - ✅ Resource-based policies: S3, SQS, SNS, Lambda, KMS, Secrets Manager, ECR, API Gateway
  - ❌ No resource-based policies: DynamoDB, EC2, ECS, RDS, ElastiCache, Redshift

#### ❌ **Option D (INCORRECT): IAM Permissions Boundary for Fine-Grained Access**

- **Permissions boundaries** are a **governance mechanism** that defines the maximum permissions an IAM entity (user or role) can ever have — they set an upper limit.
- Permissions boundaries do NOT grant access — they only restrict. You still need an identity-based policy to actually grant the specific DynamoDB permissions.
- Most importantly, permissions boundaries cannot implement **attribute-level conditions** for DynamoDB fine-grained access — that requires specific condition keys (`dynamodb:Attributes`) in the identity-based policy attached to the role.
- Permissions boundaries are typically used for delegated administration (e.g., allowing a team to create IAM roles but limiting what those roles can do). They are not the right tool for this use case.

#### Step-by-Step Implementation

**Step 1: Finance Account — Create IAM Role with Fine-Grained Policy**
```bash
# Trust policy (trust-policy.json)
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {
      "AWS": "arn:aws:iam::MARKETING_ACCOUNT_ID:role/MarketingAccessRole"
    },
    "Action": "sts:AssumeRole"
  }]
}
```

```bash
aws iam create-role \
  --role-name DynamoDBAccessRole \
  --assume-role-policy-document file://trust-policy.json
```

**Step 2: Attach Fine-Grained Access Policy to the Role**
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["dynamodb:GetItem", "dynamodb:Query", "dynamodb:BatchGetItem"],
    "Resource": "arn:aws:dynamodb:region:FINANCE_ACCOUNT_ID:table/ConfidentialData",
    "Condition": {
      "ForAllValues:StringEquals": {
        "dynamodb:Attributes": ["product_name", "product_category", "msrp"]
      }
    }
  }]
}
```

**Step 3: Marketing Account — Create Role to Assume Finance Role**
```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": "sts:AssumeRole",
    "Resource": "arn:aws:iam::FINANCE_ACCOUNT_ID:role/DynamoDBAccessRole"
  }]
}
```

**Step 4: Marketing Account — Access DynamoDB via Assumed Role (Python/boto3)**
```python
import boto3

# Assume the finance account role
sts = boto3.client('sts')
assumed_role = sts.assume_role(
    RoleArn='arn:aws:iam::FINANCE_ACCOUNT_ID:role/DynamoDBAccessRole',
    RoleSessionName='MarketingSession'
)

# Use temporary credentials to access DynamoDB
dynamodb = boto3.client(
    'dynamodb',
    aws_access_key_id=assumed_role['Credentials']['AccessKeyId'],
    aws_secret_access_key=assumed_role['Credentials']['SecretAccessKey'],
    aws_session_token=assumed_role['Credentials']['SessionToken']
)

# This will succeed — only allowed attributes are returned
response = dynamodb.get_item(
    TableName='ConfidentialData',
    Key={'product_id': {'S': '12345'}},
    ProjectionExpression='product_name, product_category, msrp'  # Must match allowed list
)
```

#### DynamoDB Fine-Grained Access Control Condition Keys

| Condition Key | What It Controls | Example |
|--------------|------------------|---------|
| `dynamodb:Attributes` | Which attribute names can be accessed | Limit read to specific columns |
| `dynamodb:LeadingKeys` | Which partition key values can be accessed | Tenant isolation (user sees only their data) |
| `dynamodb:Select` | Type of read operation | Only allow `SPECIFIC_ATTRIBUTES` (not `ALL_ATTRIBUTES`) |
| `dynamodb:ReturnValues` | Which return values are allowed | Restrict `UpdateItem` return values |
| `dynamodb:ReturnConsumedCapacity` | Whether consumed capacity is returned | Reduce information leakage |

**Important Note — `ProjectionExpression` must match allowed `Attributes`:**
The client application MUST use `ProjectionExpression` that only requests the allowed attributes. If the request tries to access `ALL_ATTRIBUTES` or includes a restricted attribute in `ProjectionExpression`, DynamoDB will deny the request. This is enforced server-side.

**Cross-Account Access Patterns Comparison:**

| Pattern | Use Case | DynamoDB Support |
|---------|----------|-----------------|
| **Cross-account IAM role assumption** | Access from another AWS account | ✅ Yes |
| Resource-based policy | Grant access directly on resource | ❌ DynamoDB does NOT support |
| SCP | Limit maximum permissions in an OU | ✅ (but doesn't grant access) |
| Permissions boundary | Govern role creation permissions | ✅ (but doesn't grant access) |
| AWS Resource Access Manager (RAM) | Share resources across accounts | ❌ DynamoDB not supported |
