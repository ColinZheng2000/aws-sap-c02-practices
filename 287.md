A software as a service (SaaS) company uses AWS to host a service that is powered by AWS PrivateLink. The service consists of proprietary software that runs on three Amazon EC2 instances behind a Network Load Balancer (NLB). The instances are in private subnets in multiple Availability Zones in the eu-west-2 Region. All the company's customers are in eu-west-2.

However, the company now acquires a new customer in the us-east-1 Region. The company creates a new VPC and new subnets in us-east-1. The company establishes inter-Region VPC peering between the VPCs in the two Regions.

The company wants to give the new customer access to the SaaS service, but the company does not want to immediately deploy new EC2 resources in us-east-1.

Which solution will meet these requirements?

A. Configure a PrivateLink endpoint service in us-east-1 to use the existing NLB that is in eu-west-2. Grant specific AWS accounts access to connect to the SaaS service.
B. Create an NLB in us-east-1. Create an IP target group that uses the IP addresses of the company's instances in eu-west-2 that host the SaaS service. Configure a PrivateLink endpoint service that uses the NLB that is in us-east-1. Grant specific AWS accounts access to connect to the SaaS service.
C. Create an Application Load Balancer (ALB) in front of the EC2 instances in eu-west-2. Create an NLB in us-east-1. Associate the NLB that is in us-east-1 with an ALB target group that uses the ALB that is in eu-west-2. Configure a PrivateLink endpoint service that uses the NLB that is in us-east-1. Grant specific AWS accounts access to connect to the SaaS service.
D. Use AWS Resource Access Manager (AWS RAM) to share the EC2 instances that are in eu-west-2. In us-east-1, create an NLB and an instance target group that includes the shared EC2 instances from eu-west-2. Configure a PrivateLink endpoint service that uses the NLB that is in us-east-1. Grant specific AWS accounts access to connect to the SaaS service.


# SAP-C02 Analysis — Cross-Region AWS PrivateLink Access Using NLB IP Targets

## Understanding the Problem

The scenario involves:

- SaaS application exposed through AWS PrivateLink
- Existing infrastructure:
  - EC2 instances
  - behind Network Load Balancer (NLB)
  - deployed in eu-west-2
- New customer located in us-east-1
- Company wants:
  - customer access from us-east-1
  - without deploying new EC2 instances in us-east-1

The company already created:
- a VPC in us-east-1
- inter-Region VPC peering

The architectural challenge:
# AWS PrivateLink is regional

This is the critical exam concept.

PrivateLink endpoint services:
- cannot directly reference NLBs in another Region

Therefore:
- some regional frontend must exist in us-east-1

But:
- backend compute should remain in eu-west-2

The solution:
- use an NLB in us-east-1
- route over inter-Region connectivity
- use IP-based targets pointing to eu-west-2 instances

---

# Correct Option

## B. Create an NLB in us-east-1 with IP targets pointing to eu-west-2 instances

---

# Why This Is Correct

This architecture works because:

1. PrivateLink requires:
   - endpoint service and NLB in same Region
2. NLB supports:
   - IP target groups
3. Inter-Region VPC peering allows:
   - private IP connectivity between Regions

Therefore:

```text
Customer VPC (us-east-1)
        ↓
PrivateLink Endpoint
        ↓
Endpoint Service (us-east-1)
        ↓
NLB (us-east-1)
        ↓
IP Target Group
        ↓
EC2 Private IPs in eu-west-2
```

This avoids:
- deploying new application EC2 instances
- duplicating infrastructure

while still satisfying:
- regional PrivateLink requirement

---

# Critical AWS Concept

## AWS PrivateLink Is Regional

You cannot:
- create a PrivateLink endpoint service in us-east-1
and directly attach:
- an NLB in eu-west-2

This is the central trap in the question.

---

# Why IP Target Groups Work

NLB IP target groups support:
- RFC1918 private IPs
- targets reachable through routing

Since:
- inter-Region VPC peering exists

the us-east-1 NLB can route traffic to:
- private IPs in eu-west-2

This is a known AWS architecture pattern.

---

# Incorrect Options

---

## A. Configure PrivateLink in us-east-1 using existing eu-west-2 NLB

### Why Incorrect

PrivateLink endpoint services are strictly regional.

An endpoint service in:
- us-east-1

cannot use:
- an NLB in eu-west-2

This violates AWS PrivateLink architecture constraints.

Important SAP-C02 memory point:

| Service | Regional? |
|---|---|
| PrivateLink | Yes |
| Endpoint Service | Regional |
| Interface Endpoint | Regional |

---

## C. ALB + NLB association

### Why

Several problems exist here.

---

## Problem 1 — Cross-Region ALB Targeting Unsupported

NLB target groups that reference ALBs:
- require local/regional association
- not cross-Region ALB forwarding

You cannot:
- associate a us-east-1 NLB
with:
- an ALB in eu-west-2

---

## Problem 2 — Unnecessary Complexity

The architecture already works with:
- direct IP targets

Introducing:
- ALB
adds:
- extra hops
- unnecessary operational complexity

---

## Problem 3 — PrivateLink Uses NLB

PrivateLink endpoint services fundamentally integrate with:
- NLBs

The ALB is irrelevant here.

---

## D. Use AWS RAM to share EC2 instances

### Why Incorrect

AWS RAM does NOT support:
- sharing EC2 instances across accounts/Regions for NLB targeting

RAM is used for:
- subnets
- Transit Gateway
- Route53 Resolver rules
- License Manager
- resource sharing services

Not:
- EC2 instance sharing into NLB target groups

Also:
- instance target groups require local-region instances

This option combines unsupported features.

# Enterprise Hands-On Knowledge

## Real SaaS Multi-Region Expansion Pattern

Many SaaS providers initially:
- deploy application stack in one Region
- expand customer access gradually

A temporary expansion strategy:

```text
Regional Front Door
        ↓
Cross-Region Private Connectivity
        ↓
Centralized Compute Region
```

Benefits:
- lower operational cost
- avoids duplicate deployments
- enables phased regional rollout

Later- providers typically deploy regional compute
for:
- latency
- resilience
- compliance

---

# Important Latency Consideration

This architecture works technically, but:

Traffic path:

```text
us-east-1 customer
    ↓
us-east-1 endpoint
    ↓
cross-region peering
    ↓
eu-west-2 backend
```

This introduces:
- transatlantic latency

In financial or medical systems:
- acceptable only as transitional architecture

SAP-C02 often tests:
- technically valid vs strategically ideal

The question asks:
> without immediately deploying new EC2 resources

Therefore:
- temporary cross-region forwarding is acceptable

# Networking Constraints to Remember

## Inter-Region VPC Peering Supports

- private IP routing
- security group references (same-region only limitation)
- low-latency AWS backbone transport

Useful for:
- IP target group architectures

---

# Important NLB Capability

NLB supports target types:

| Target Type | Supported |
|---|---|
| Instance | Yes |
| IP | Yes |
| Lambda | No |

The key here is:
# IP target

Without IP targets:
- cross-region backend routing would not work
