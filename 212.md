A company wants to refactor its retail ordering web application that currently has a load-balanced Amazon EC2 instance fleet for web hosting, database API services, and business logic. The company needs to create a decoupled, scalable architecture with a mechanism for retaining failed orders while also minimizing operational costs.
Which solution will meet these requirements?

A. Use Amazon S3 for web hosting with Amazon API Gateway for database API services. Use Amazon Simple Queue Service (Amazon SQS) for order queuing. Use Amazon Elastic Container Service (Amazon ECS) for business logic with Amazon SQS long polling for retaining failed orders.
B. Use AWS Elastic Beanstalk for web hosting with Amazon API Gateway for database API services. Use Amazon MQ for order queuing. Use AWS Step Functions for business logic with Amazon S3 Glacier Deep Archive for retaining failed orders.
C. Use Amazon S3 for web hosting with AWS AppSync for database API services. Use Amazon Simple Queue Service (Amazon SQS) for order queuing. Use AWS Lambda for business logic with an Amazon SQS dead-letter queue for retaining failed orders.
D. Use Amazon Lightsail for web hosting with AWS AppSync for database API services. Use Amazon Simple Email Service (Amazon SES) for order queuing. Use Amazon Elastic Kubernetes Service (Amazon EKS) for business logic with Amazon Elasticsearch Service (Amazon ES) for retaining failed orders.



Suggest Answer: C

---

## 📊 Analysis

### Understanding the Problem

A retail company has a monolithic web application running on EC2 instances behind a load balancer — handling web hosting, database API services, and business logic all in one tier. The goal is to **decouple** these components into a **scalable architecture** while retaining **failed orders** and **minimizing operational costs**. This is a classic Serverless/Managed-Services refactoring scenario tested heavily in SAP-C02.

Key requirements:
- ✅ Decoupled architecture (separate web, API, and business logic)
- ✅ Mechanism to retain failed orders (dead-letter queue pattern)
- ✅ Minimize operational costs (prefer serverless/managed services)

### ✅ Correct Option: C — S3 + AppSync + SQS + Lambda + SQS DLQ

| Component | Service | Why It Fits |
|---|---|---|
| Web Hosting | **Amazon S3** (static website hosting) | Serverless, zero operational overhead, highly scalable, cost-effective for front-end assets |
| Database API | **AWS AppSync** | Managed GraphQL service — ideal for decoupled API layer connecting to databases; supports real-time and offline; fully serverless |
| Order Queuing | **Amazon SQS** | Fully managed message queuing; decouples front-end from business logic; scales infinitely |
| Business Logic | **AWS Lambda** | Serverless compute — pay-per-invocation, zero idle cost; perfect for order processing logic |
| Failed Order Retention | **Amazon SQS Dead-Letter Queue (DLQ)** | Native SQS feature — automatically captures messages that fail processing after a configured number of retries; no additional cost beyond SQS itself |

**Why this is the best solution (SAP-C02 lens):**
- Each component is fully managed and serverless → **minimizes operational costs**
- SQS + Lambda follows the **decoupled microservices pattern** with automatic retry
- SQS DLQ is the **canonical AWS pattern for retaining/preserving failed messages** for later analysis and reprocessing
- S3 + AppSync + Lambda → no servers to manage, aligns with the Well-Architected Framework's cost optimization pillar

### ❌ Incorrect Options

**A. S3 + API Gateway + SQS + ECS + SQS Long Polling**

- **API Gateway** is designed for REST/HTTP APIs, not database API services — AppSync is purpose-built for data APIs
- **Amazon ECS** running business logic adds operational complexity (container management, scaling, patching) vs. Lambda's serverless model → higher operational cost
- **SQS long polling** is a *message retrieval optimization* (reduces empty responses and cost), **NOT** a mechanism for retaining failed orders. Long polling doesn't persist failed messages — it just waits longer for messages to arrive. This reveals a fundamental misunderstanding of SQS features.

**B. Elastic Beanstalk + API Gateway + Amazon MQ + Step Functions + S3 Glacier Deep Archive**

- **Elastic Beanstalk** manages EC2 instances → operational overhead, contradicts "minimize costs"
- **Amazon MQ** is a managed message broker for *ActiveMQ/RabbitMQ* — overkill and more expensive than SQS when you don't need advanced broker features (message groups, transaction support)
- **Step Functions** is a *workflow orchestration* service, not a compute service for running business logic; you'd still need Lambda/ECS behind it
- **S3 Glacier Deep Archive** for failed orders is absurd — retrieval time is 12–48 hours; failed orders need to be accessed for analysis/reprocessing within minutes, not days

**D. Lightsail + AppSync + SES + EKS + Elasticsearch Service**

- **Lightsail** is for simple, low-traffic workloads — it's not designed for scalable production hosting
- **Amazon SES (Simple Email Service)** is an **email sending/receiving** service — it is NOT a message queue. Using SES for order queuing shows a fundamental misunderstanding of AWS services
- **EKS** adds significant Kubernetes operational overhead (control plane management, node group scaling, version upgrades) → highest operational cost of all options
- **Elasticsearch Service** is a **search and analytics engine**, not a dead-letter queue — it can't natively capture failed SQS messages

### 🏭 Enterprise Hands-On Knowledge

1. **SQS DLQ in Practice:** In production, configure `maxReceiveCount` on the main queue (typically 3–5). After that many failed processing attempts, SQS automatically moves the message to the DLQ. Set up a CloudWatch Alarm on `ApproximateNumberOfMessagesVisible` for the DLQ to alert operations when failed orders accumulate. Then use a separate Lambda to reprocess DLQ messages or move them back to the source queue via redrive.

2. **S3 Static Hosting + CloudFront:** For production, always front S3 with CloudFront for HTTPS (S3 static website endpoints only support HTTP), custom domain with ACM certificate, and edge caching. Use Origin Access Identity (OAI) or Origin Access Control (OAC) to restrict S3 bucket access to CloudFront only.

3. **AppSync vs API Gateway Decision Matrix:** Use AppSync when you need GraphQL, real-time subscriptions, or offline data access. Use API Gateway when you need REST APIs, WebSocket APIs, or direct AWS service integrations. For database API services, AppSync's resolver mapping templates directly connect to DynamoDB, RDS, or Lambda.

4. **Cost Comparison Reality:** Lambda + SQS + S3 typically costs **80-90% less** than an equivalent EC2/ECS/EKS solution for variable workloads. The break-even point shifts toward containers only at sustained high throughput (thousands of TPS continuously).
