A company has an asynchronous HTTP application that is hosted as an AWS Lambda function. A public Amazon API Gateway endpoint invokes the Lambda function. The Lambda function and the API Gateway endpoint reside in the us-east-1 Region. A solutions architect needs to redesign the application to support failover to another AWS Region.  
  
Which solution will meet these requirements?

- A. Create an API Gateway endpoint in the us-west-2 Region to direct traffic to the Lambda function in us-east-1. Configure Amazon Route 53 to use a failover routing policy to route traffic for the two API Gateway endpoints.
- B. Create an Amazon Simple Queue Service (Amazon SQS) queue. Configure API Gateway to direct traffic to the SQS queue instead of to the Lambda function. Configure the Lambda function to pull messages from the queue for processing.
- C. Deploy the Lambda function to the us-west-2 Region. Create an API Gateway endpoint in us-west-2 10 direct traffic to the Lambda function in us-west-2. Configure AWS Global Accelerator and an Application Load Balancer to manage traffic across the two API Gateway endpoints.
- D. Deploy the Lambda function and an API Gateway endpoint to the us-west-2 Region. Configure Amazon Route 53 to use a failover routing policy to route traffic for the two API Gateway endpoints.

## AWS SAP-C02 Exam Question Analysis

**Tags:** #Route_53 #API_Gateway #Lambda #Multi-Region_Architecture #Disaster_Recovery #SAP-C02

### Understanding the Problem

The company currently hosts an **asynchronous HTTP application** using an **AWS Lambda function** invoked via a **public Amazon API Gateway endpoint** in the **us-east-1 Region**. The primary requirement is to **redesign the application to support failover to another AWS Region** (e.g., **us-west-2**). This means the solution must establish a redundant setup in a secondary region and use a mechanism to automatically direct traffic to the secondary region if the primary region fails.

### Services Explanation

---

- **AWS Lambda:** A serverless, event-driven compute service that lets you run code without provisioning or managing servers. In this scenario, it hosts the application logic.
    
- **Amazon API Gateway:** A fully managed service that makes it easy for developers to create, publish, maintain, monitor, and secure APIs at any scale. It acts as the public-facing HTTP endpoint for the Lambda function.
    
- **Amazon Route 53:** A highly available and scalable cloud Domain Name System (DNS) web service. It can be used to manage traffic with various routing policies, including **Failover Routing**.
    
- **AWS Global Accelerator:** A networking service that improves the availability and performance of your applications with local or global users. It uses the AWS global network to direct traffic to the optimal regional endpoint.
    
- **Application Load Balancer (ALB):** A layer 7 load balancer that routes traffic to targets within a single region. It is not designed to manage cross-region failover directly in the way Global Accelerator or Route 53 with Failover Routing is.
    
- **Amazon Simple Queue Service (SQS):** A fully managed message queuing service that enables you to decouple and scale microservices, distributed systems, and serverless applications. While it supports decoupling and asynchronous processing, it does not inherently provide a mechanism for **cross-region HTTP endpoint failover** for a public-facing API.
    

### Correct Option(s) Explanation

---

The correct solution is **D**.

**D. Deploy the Lambda function and an API Gateway endpoint to the us-west-2 Region. Configure Amazon Route 53 to use a failover routing policy to route traffic for the two API Gateway endpoints.**

1. **Deployment:** A copy of the entire application stack (**Lambda function** and **API Gateway endpoint**) is deployed in the secondary **us-west-2** Region. This creates a fully redundant setup.
    
2. **Routing:** **Amazon Route 53** is used to manage the DNS for the application.
    
    - An **A** record (or **CNAME/ALIAS** record) is created that points to the primary API Gateway endpoint in **us-east-1** (Primary Resource Record Set).
        
    - A second **A** record is created that points to the secondary API Gateway endpoint in **us-west-2** (Secondary Resource Record Set).
        
    - Both records are configured with a **Failover Routing Policy**.
        
3. **Failover Logic:** Route 53 performs **health checks** on the primary endpoint. If the primary API Gateway (or the application behind it) fails the health checks, Route 53 automatically routes all traffic to the secondary endpoint in **us-west-2**, meeting the requirement for **regional failover**.
    

This approach is the standard and most cost-effective method for simple active/passive regional failover using DNS.

_Official AWS Documentation Reference: Amazon Route 53 Developer Guide - [Configuring failover](https://docs.aws.amazon.com/Route53/latest/DeveloperGuide/dns-failover.html)_

### Incorrect Options Explanation

---

**A. Create an API Gateway endpoint in the us-west-2 Region to direct traffic to the Lambda function in us-east-1. Configure Amazon Route 53 to use a failover routing policy to route traffic for the two API Gateway endpoints.**

- **Problem:** The API Gateway in **us-west-2** is configured to invoke the **Lambda function in us-east-1**. If the **us-east-1** Region fails, the Lambda function is unavailable, and the entire application fails, regardless of where the traffic lands. A complete regional failover requires the entire stack (API Gateway and Lambda) to be deployed in the secondary region.
    

**B. Create an Amazon Simple Queue Service (Amazon SQS) queue. Configure API Gateway to direct traffic to the SQS queue instead of to the Lambda function. Configure the Lambda function to pull messages from the queue for processing.**

- **Problem:** This option addresses **decoupling** (making the application truly asynchronous) but does **not** address **cross-region failover**. The API Gateway and SQS queue are still assumed to be in the primary region (us-east-1). If the primary region fails, the public API Gateway endpoint fails, and the application becomes unavailable. This is an excellent design choice for decoupling but fails to meet the core requirement of regional failover for the public HTTP endpoint.
    

**C. Deploy the Lambda function to the us-west-2 Region. Create an API Gateway endpoint in us-west-2 to direct traffic to the Lambda function in us-west-2. Configure AWS Global Accelerator and an Application Load Balancer to manage traffic across the two API Gateway endpoints.**

- **Problem:** **AWS Global Accelerator** is a valid tool for multi-region failover, often preferred for improved performance and instant failover. However, **API Gateway endpoints are not directly supported as standard endpoints for Global Accelerator**. Global Accelerator typically works with **Network Load Balancers (NLB)**, **Application Load Balancers (ALB)**, or **Elastic IP addresses**. While you could potentially use an ALB in front of a **Regional API Gateway (via an NLB, or by using a custom domain and pointing the ALB to it)**, the most straightforward and standard AWS solution for DNS-based failover of API Gateway endpoints is **Amazon Route 53**, as described in Option D. The use of an **ALB** in this context adds unnecessary complexity and is incorrect, as an ALB is not a standard component for routing directly to an API Gateway for cross-region failover. The complexity of routing Global Accelerator to a Regional API Gateway makes this an inferior choice to the simpler, direct Route 53 Failover in Option D.
