A company plans to deploy a new private intranet service on Amazon EC2 instances inside a VPC. An AWS Site-to-Site VPN connects the VPC to the company's on-premises network. The new service must communicate with existing on-premises services. The on-premises services are accessible through the use of hostnames that reside in the company.example DNS zone. This DNS zone is wholly hosted on premises and is available only on the company's private network.
A solutions architect must ensure that the new service can resolve hostnames on the company example domain to integrate with existing services.
Which solution meets these requirements?

A. Create an empty private zone in Amazon Route 53 for company example. Add an additional NS record to the company's on-premises company.example zone that points to the authoritative name servers for the new private zone in Route 53.
B. Turn on DNS hostnames for the VPC. Configure a new outbound endpoint with Amazon Route 53 Resolver. Create a Resolver rule to forward requests for company.example to the on-premises name servers.
C. Turn on DNS hostnames for the VPC. Configure a new inbound resolver endpoint with Amazon Route 53 Resolver. Configure the on-premises DNS server to forward requests for company.example to the new resolver.
D. Use AWS Systems Manager to configure a run document that will install a hosts file that contains any required hostnames. Use an Amazon EventBridge (Amazon CloudWatch Events) rule to run the document when an instance is entering the running state.



Suggest Answer: B

---

## 📊 Analysis

### Understanding the Problem

A company is deploying a new private intranet service on EC2 instances inside a VPC. The VPC connects to the on-premises network via **AWS Site-to-Site VPN**. The new EC2 service needs to communicate with existing on-premises services that are accessed via **hostnames** in the `company.example` DNS zone. This DNS zone is **wholly hosted on-premises** and is only available on the private network (not public internet).

The core challenge: **EC2 instances in the VPC need to resolve on-premises private DNS names** (`*.company.example`) through the VPN tunnel to the on-premises DNS servers. This is a **hybrid DNS resolution** scenario.

### ✅ Correct Option: B — Route 53 Resolver Outbound Endpoint + Forwarding Rule

| Component | Purpose |
|---|---|
| **Enable DNS hostnames for VPC** | Required prerequisite — allows EC2 instances in the VPC to receive public DNS hostnames and enables the VPC to use the Route 53 Resolver |
| **Route 53 Resolver Outbound Endpoint** | Creates a bridge from the VPC to on-premises DNS servers through the VPN/Direct Connect. DNS queries for specified domains are forwarded out of the VPC to on-premises resolvers |
| **Resolver Rule to forward `company.example`** | A conditional forwarding rule: "When an EC2 instance queries `*.company.example`, forward the query to the on-premises DNS servers (specified by IP address)" |

**How it works:**
```
EC2 instance → queries app.company.example
    → VPC+2 DNS resolver (Route 53 Resolver)
    → Resolver Rule matches "company.example"
    → Outbound Endpoint forwards query via VPN
    → On-premises DNS server resolves → returns IP
```

**Why this is the correct solution (SAP-C02 lens):**
- Route 53 Resolver Endpoints are **purpose-built** for hybrid DNS resolution between VPC and on-premises
- **Outbound** endpoint = VPC → on-premises (the direction needed here)
- **Inbound** endpoint = on-premises → VPC (opposite direction)
- The forwarding rule is a **conditional forwarder** — only queries for `company.example` go to on-premises; all other DNS queries (e.g., AWS services, internet) resolve normally within AWS
- This is the **standard AWS hybrid DNS architecture** per the Well-Architected Framework

### ❌ Incorrect Options

**A. Create empty private zone in Route 53 for `company.example` + point NS record on-premises to Route 53**

- This does the **opposite** of what's needed — it tells on-premises DNS to forward queries to Route 53
- The `company.example` zone is **wholly hosted on-premises** and only accessible on the private network — Route 53 can't reach those DNS servers
- An empty Route 53 private zone would give **NXDOMAIN** (non-existent domain) responses for all queries, breaking name resolution entirely
- The NS delegation direction is wrong: the problem needs EC2 → on-premises DNS, not on-premises → Route 53
- ❌ Fundamental misunderstanding of DNS delegation direction

**C. Inbound Resolver Endpoint + configure on-premises DNS to forward to Route 53**

- **Inbound** endpoint receives DNS queries **from on-premises** into the VPC — this is the reverse of what's needed
- The scenario requires EC2 instances (in AWS) to resolve on-premises hostnames, which requires an **outbound** endpoint
- Configuring on-premises DNS to forward to Route 53 would help on-premises services resolve VPC hostnames — not what the question asks
- ❌ Confuses inbound vs outbound resolver direction — a classic SAP-C02 trap

**D. Systems Manager Run Document to install hosts file + EventBridge rule on instance start**

- This is a **manual/static hostname management** approach — unscalable and error-prone
- Every time an on-premises hostname is added, changed, or removed, the hosts file must be updated across all EC2 instances
- Does not handle IP address changes dynamically (typical in enterprise environments)
- EventBridge can trigger on EC2 state changes, but the Run Document would need to be updated and re-executed for every DNS change
- Violates operational excellence — DNS should be managed through DNS infrastructure, not hosts files
- ❌ This is a "works technically but is terrible practice" distractor common in SAP-C02

### 🏭 Enterprise Hands-On Knowledge

1. **Route 53 Resolver Endpoints Architecture:**
   - **Outbound Endpoint**: Deployed in your VPC across multiple AZs (at least 2 for high availability). You provide the on-premises DNS server IPs as targets.
   - **Inbound Endpoint**: Deployed in your VPC — on-premises DNS servers forward queries to these IPs to resolve AWS private hosted zone names.
   - **Resolver Rules**: Can be associated with specific VPCs or shared across multiple VPCs via AWS Resource Access Manager (RAM).

2. **Common Enterprise Hybrid DNS Pattern:**
   - On-premises DNS servers resolve `corp.example` (on-premises resources)
   - Route 53 Resolver Outbound Endpoint + forwarding rule sends `corp.example` queries to on-premises
   - Route 53 Private Hosted Zone resolves `aws.example` (AWS resources)
   - Route 53 Resolver Inbound Endpoint allows on-premises to resolve `aws.example`
   - This creates **bidirectional DNS resolution** in a hybrid environment

3. **Prerequisites Checklist:**
   - ✅ VPC DNS hostnames enabled (`enableDnsHostnames: true`)
   - ✅ VPC DNS resolution enabled (`enableDnsSupport: true`)
   - ✅ VPN/Direct Connect connectivity to on-premises DNS servers
   - ✅ Security groups on the Resolver endpoint allow UDP/TCP port 53
   - ✅ On-premises firewall allows DNS queries from the Resolver endpoint IPs

4. **Cost Consideration:** Route 53 Resolver Endpoints incur hourly charges per endpoint ENI (~$0.125/hour per ENI in us-east-1). For production, deploy at least 2 ENIs across 2 AZs. Each endpoint handles ~10,000 DNS queries per second per ENI.
