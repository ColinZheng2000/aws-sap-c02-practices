A company is developing a new service that will be accessed using TCP on a static port. A solutions architect must ensure that the service is highly available, has redundancy across Availability Zones, and is accessible using the DNS name my.service.com, which is publicly accessible. The service must use fixed address assignments so other companies can add the addresses to their allow lists.  
Assuming that resources are deployed in multiple Availability Zones in a single Region, which solution will meet these requirements?  

- A. Create Amazon EC2 instances with an Elastic IP address for each instance. Create a Network Load Balancer (NLB) and expose the static TCP port. Register EC2 instances with the NLB. Create a new name server record set named my.service.com, and assign the Elastic IP addresses of the EC2 instances to the record set. Provide the Elastic IP addresses of the EC2 instances to the other companies to add to their allow lists.
- B. Create an Amazon ECS cluster and a service definition for the application. Create and assign public IP addresses for the ECS cluster. Create a Network Load Balancer (NLB) and expose the TCP port. Create a target group and assign the ECS cluster name to the NLB. Create a new A record set named my.service.com, and assign the public IP addresses of the ECS cluster to the record set. Provide the public IP addresses of the ECS cluster to the other companies to add to their allow lists.
- C. Create Amazon EC2 instances for the service. Create one Elastic IP address for each Availability Zone. Create a Network Load Balancer (NLB) and expose the assigned TCP port. Assign the Elastic IP addresses to the NLB for each Availability Zone. Create a target group and register the EC2 instances with the NLB. Create a new A (alias) record set named my.service.com, and assign the NLB DNS name to the record set.
- D. Create an Amazon ECS cluster and a service definition for the application. Create and assign public IP address for each host in the cluster. Create an Application Load Balancer (ALB) and expose the static TCP port. Create a target group and assign the ECS service definition name to the ALB. Create a new CNAME record set and associate the public IP addresses to the record set. Provide the Elastic IP addresses of the Amazon EC2 instances to the other companies to add to their allow lists.

## Understanding the Problem

The company is developing a new service with the following key requirements:

1.  **Protocol and Port:** Access must use **TCP** on a **static port**.
2.  **High Availability & Redundancy:** The service must be **highly available** and have **redundancy across Availability Zones (AZs)**.
3.  **Public Accessibility and DNS:** Accessible using the public DNS name **`my.service.com`**.
4.  **Fixed Address Assignments:** The service must use **fixed public IP addresses** so other companies can add them to their **allow lists**.
5.  **Deployment:** Resources are deployed in **multiple Availability Zones** in a **single Region**.

The solution must combine high availability (requiring multiple resources and load balancing) with the need for a **set of fixed, public IP addresses**.

| :----------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Network Load Balancer (NLB)**            | A high-performance, Layer 4 (TCP/UDP) load balancer. It can handle millions of requests per second and is designed for low latency. **Crucially, it supports assigning a single Elastic IP address per Availability Zone** for its public-facing interface. | Meets the **TCP static port** and **high availability/redundancy** requirements, and is essential for providing **fixed IP addresses** for the allow list.      |

| **Application Load Balancer (ALB)**        | A Layer 7 (HTTP/HTTPS) load balancer.                                                                                                                                                                                                                       | Not suitable, as the requirement specifies **TCP** access, not HTTP/HTTPS.                                                                                      |

## Correct Option(s) Explanation

### Correct Answer: C

**C. Create Amazon EC2 instances for the service. Create one Elastic IP address for each Availability Zone. Create a Network Load Balancer (NLB) and expose the assigned TCP port. Assign the Elastic IP addresses to the NLB for each Availability Zone. Create a target group and register the EC2 instances with the NLB. Create a new A (alias) record set named my.service.com, and assign the NLB DNS name to the record set.**

This solution perfectly meets all requirements:

1.  **High Availability and Redundancy:** Creating EC2 instances in multiple AZs and placing a **Network Load Balancer (NLB)** in front of them ensures high availability and cross-AZ redundancy.
2.  **TCP Static Port:** The **NLB** operates at Layer 4 and can expose the required **static TCP port**.
3.  **Fixed Address Assignments (Allow List):** Assigning **one Elastic IP address for each Availability Zone to the NLB** (the "Client-facing interface") is the **standard and correct way** to provide **fixed, static public IP addresses** for an NLB, as per the [AWS Network Load Balancer documentation](https://docs.aws.amazon.com/elasticloadbalancing/latest/network/network-load-balancers-tutorial.html). These EIPs are the addresses to be shared for the allow list.
4.  **Public DNS (`my.service.com`):** Creating an **A (alias) record set** in Route 53 that points to the **NLB's DNS name** provides the single, public-facing DNS name. Alias records are preferred over CNAMEs for pointing to AWS resources like NLBs, as they can be created at the zone apex (e.g., `service.com`) and integrate natively with the NLB's health checks and IP addresses.

***

## Incorrect Options Explanation

### A. Incorrect

**A. Create Amazon EC2 instances with an Elastic IP address for each instance. ... Assign the Elastic IP addresses of the EC2 instances to the record set.**

* **Failure Point:** While it provides fixed EIPs, it completely **bypasses the load balancer for DNS resolution**. Assigning multiple EC2 EIPs to a single DNS name (`my.service.com`) without a load balancer in front results in basic **DNS round-robin**, which **is not a highly available or redundant solution** in the same way an NLB is. There is no automatic health check-based failover; if an instance or EIP fails, the DNS record still exists, and a user might be directed to a non-functional address. The NLB is necessary for robust health checks and seamless traffic distribution/failover.

### B. Incorrect

**B. Create an Amazon ECS cluster... Create and assign public IP addresses for the ECS cluster. ... Create a new A record set named my.service.com, and assign the public IP addresses of the ECS cluster to the record set. ...**

* **Failure Point 1 (IP Assignment):** You don't assign "public IP addresses for the ECS cluster." If using ECS with Fargate, the tasks themselves get IP addresses. If using EC2, the hosts get IPs. The key requirement is to assign **fixed, static public IP addresses** to the load balancer's interface, which is done using **Elastic IPs** assigned to the **NLB**, not the cluster or its components directly in the context of the load balancing requirement.
* **Failure Point 2 (DNS Mapping):** The DNS record is pointed to the **public IP addresses of the ECS cluster** (which are dynamic/temporary if not EIPs), **bypassing the NLB** for the public DNS name, similar to option A's flaw. The NLB's primary function is to provide the single, highly available endpoint.

### D. Incorrect

**D. Create an Amazon ECS cluster... Create an Application Load Balancer (ALB) and expose the static TCP port. ...**

* **Failure Point 1 (Load Balancer Type):** Uses an **Application Load Balancer (ALB)**. An ALB is a Layer 7 (HTTP/HTTPS) load balancer. The requirement is for **TCP** on a static port. While ALBs can accept TCP and forward to HTTP on the target, they are **not the correct choice for a pure TCP service** and do not offer the low latency and transparency of the NLB. **Crucially, an ALB does not natively support assigning Elastic IPs for its public-facing IP addresses to meet the fixed address requirement**; it uses AWS-managed IPs that can change. Only the **NLB** supports the association of reserved Elastic IP addresses.
* **Failure Point 2 (IP Addresses and DNS):** The solution's final step suggests providing the **Elastic IP addresses of the Amazon EC2 instances** for the allow list, but the solution describes an ECS cluster. This is a contradictory and mixed-up instruction. The required EIPs must be assigned to the **NLB**.

The only option that correctly leverages the **Network Load Balancer (NLB)**'s unique feature of associating **Elastic IP addresses (EIPs) per Availability Zone** to provide the required **fixed, highly available public IP endpoints** is **Option C**.
