A company is building a solution in the AWS Cloud. Thousands or devices will connect to the solution and send data. Each device needs to be able to send and receive data in real time over the MQTT protocol. Each device must authenticate by using a unique X.509 certificate.  
  
Which solution will meet these requirements with the LEAST operational overhead?

- A. Set up AWS IoT Core. For each device, create a corresponding Amazon MQ queue and provision a certificate. Connect each device to Amazon MQ.
- B. Create a Network Load Balancer (NLB) and configure it with an AWS Lambda authorizer. Run an MQTT broker on Amazon EC2 instances in an Auto Scaling group. Set the Auto Scaling group as the target for the NLConnect each device to the NLB.
- C. Set up AWS IoT Core. For each device, create a corresponding AWS IoT thing and provision a certificate. Connect each device to AWS IoT Core.
- D. Set up an Amazon API Gateway HTTP API and a Network Load Balancer (NLB). Create integration between API Gateway and the NLB. Configure a mutual TLS certificate authorizer on the HTTP API. Run an MQTT broker on an Amazon EC2 instance that the NLB targets. Connect each device to the NLB.


---

## Analysis

### Understanding the Problem

The company needs a cloud solution to handle **thousands of devices** that will **connect and communicate in real time** using the **MQTT protocol**. A critical requirement is that each device must **authenticate using a unique X.509 certificate**. The solution must also have the **LEAST operational overhead**.

| **Requirement**                  | **Key Service Feature**                          |
| -------------------------------- | ------------------------------------------------ |
| Thousands of devices             | Scalability, high availability                   |
| Real-time data                   | Low latency, persistent connection               |
| MQTT protocol                    | Managed MQTT broker/endpoint                     |
| X.509 certificate authentication | Managed device identity and certificate handling |
| LEAST operational overhead       | Fully managed service                            |

### Correct Option(s) Explanation

✅ **C. Set up AWS IoT Core. For each device, create a corresponding AWS IoT thing and provision a certificate. Connect each device to AWS IoT Core.**

- **AWS IoT Core** is the purpose-built service for this exact scenario.
    
- **MQTT Support:** It natively supports the **MQTT protocol** for real-time, bi-directional communication.
    
- **X.509 Authentication:** It natively handles device identity using **X.509 certificates** (or other methods). Devices are represented as **AWS IoT things**, and their certificates are registered with IoT Core. This simplifies the critical security and authentication requirement.
    
- **Least Operational Overhead:** As a **fully managed service**, AWS manages the heavy lifting of the MQTT broker, scaling for thousands of devices, high availability, and the certificate validation process, drastically minimizing the customer's operational burden. This directly addresses the "LEAST operational overhead" requirement.
    

---

### Incorrect Options Explanation

❌ **A. Set up AWS IoT Core. For each device, create a corresponding Amazon MQ queue and provision a certificate. Connect each device to Amazon MQ.**

- **Mixed Services:** While both services support MQTT, this option proposes using **Amazon MQ** as the device endpoint instead of the purpose-built **AWS IoT Core** device gateway.
    
- **Operational Overhead:** Amazon MQ, while managed, is a traditional message broker service that requires selecting and operating broker types (e.g., ActiveMQ or RabbitMQ) and scaling them. It is **not as operationally lean** or as deeply integrated for large-scale device identity management (X.509) as IoT Core.
    
- **Incorrect Design Pattern:** Devices should connect directly to the AWS IoT Core endpoint, which then manages message routing (including potentially routing to an MQ queue later, but not as the initial connection point for thousands of devices).
    

❌ **B. Create a Network Load Balancer (NLB) and configure it with an AWS Lambda authorizer. Run an MQTT broker on Amazon EC2 instances in an Auto Scaling group. Set the Auto Scaling group as the target for the NLB. Connect each device to the NLB.**

- **High Operational Overhead:** This solution requires the customer to **self-manage** the MQTT broker software, including installation, patching, scaling, security hardening, and high availability setup (with ASG and NLB). The requirement for the **LEAST operational overhead** is violated.
    
- **Technical Implausibility:** An AWS Lambda authorizer is a feature for **API Gateway**, not for the NLB or directly for an MQTT broker on EC2. Implementing X.509 client certificate authentication and validation on the custom EC2 broker and integrating it with AWS services for authorization would be complex and completely defeats the "LEAST operational overhead" goal.
    

❌ **D. Set up an Amazon API Gateway HTTP API and a Network Load Balancer (NLB). Create integration between API Gateway and the NLB. Configure a mutual TLS certificate authorizer on the HTTP API. Run an MQTT broker on an Amazon EC2 instance that the NLB targets. Connect each device to the NLB.**

- **Incorrect Protocol/High Overhead:** This solution uses **API Gateway** for an **HTTP API** (and mutual TLS), but the requirement is for the **MQTT protocol**. MQTT connections are persistent and typically run over TCP on port 8883 (secure). While API Gateway has a WebSocket API option, this is not a standard MQTT endpoint.
    
- **Extreme Operational Overhead:** This combines a self-managed MQTT broker on EC2 with an NLB, _plus_ the added complexity of trying to shoehorn API Gateway and mTLS into the flow. This is exponentially **more complex and has higher operational overhead** than the self-managed option B, and vastly more than using the fully managed AWS IoT Core.

### Official Documentation References

- **AWS IoT Core Documentation:** "The AWS IoT Device Gateway enables devices to securely and efficiently communicate with AWS IoT Core over long-lived, bi-directional connections using **MQTT**, WebSockets, and HTTPS." (Confirms MQTT and secure connection handling).
    
- **AWS IoT Core Security:** "AWS IoT Core ensures secure connection between devices and the cloud by providing **X.509 certificate-based authentication** and encryption." (Confirms native X.509 support).
    
- **AWS Well-Architected Framework (Operational Excellence Pillar):** Fully managed, purpose-built services (like AWS IoT Core) are preferred over self-managed solutions (like EC2/NLB/ASG) to reduce operational burden and increase reliability.
