An adventure company has launched a new feature on its mobile app. Users can use the feature to upload their hiking and rafting photos and videos anytime. The photos and videos are stored in Amazon S3 Standard storage in an S3 bucket and are served through Amazon CloudFront.  
  
The company needs to optimize the cost of the storage. A solutions architect discovers that most of the uploaded photos and videos are accessed infrequently after 30 days. However, some of the uploaded photos and videos are accessed frequently after 30 days. The solutions architect needs to implement a solution that maintains millisecond retrieval availability of the photos and videos at the lowest possible cost.  
  
Which solution will meet these requirements?

- A. Configure S3 Intelligent-Tiering on the S3 bucket.
- B. Configure an S3 Lifecycle policy to transition image objects and video objects from S3 Standard to S3 Glacier Deep Archive after 30 days.
- C. Replace Amazon S3 with an Amazon Elastic File System (Amazon EFS) file system that is mounted on Amazon EC2 instances.
- D. Add a Cache-Control: max-age header to the S3 image objects and S3 video objects. Set the header to 30 days.

### Understanding the Problem

The adventure company stores user-uploaded **photos and videos** in **Amazon S3 Standard**, which are served via **Amazon CloudFront**. The primary goal is to **optimize storage cost** while maintaining **millisecond retrieval availability**.

The access pattern is key:

- Most objects are accessed **infrequently after 30 days**.
    
- **Some** objects are accessed **frequently after 30 days**.
    

The solution must be able to handle this mixed and unpredictable access pattern to reduce cost for the infrequently accessed data without compromising the availability or fast retrieval speed for the frequently accessed data.

---

### Services Explanation

|**Service**|**Description**|**Relevance to Problem**|
|---|---|---|
|**Amazon S3 (Simple Storage Service)**|An object storage service offering industry-leading scalability, data availability, security, and performance. **S3 Standard** is for general-purpose storage of frequently accessed data.|The current storage location and the core service for the solution.|
|**Amazon S3 Intelligent-Tiering**|An S3 storage class that automatically moves data between two access tiers (**Frequent Access** and **Infrequent Access**) based on access patterns. It's designed for data with unknown or changing access patterns.|A strong candidate for cost optimization for data with unpredictable access patterns. **Maintains millisecond retrieval.**|
|**Amazon S3 Glacier Deep Archive**|The lowest-cost S3 storage class, ideal for long-term retention. Retrieval times are measured in hours (12 hours standard, or minutes/hours expedited).|Too slow for the millisecond retrieval requirement.|
|**Amazon S3 Lifecycle Policies**|Rules that define actions for S3 objects during their lifetime, such as transitioning them to a different storage class or expiring them.|Can automate transitions but is static and cannot handle the required _some are frequent/most are infrequent_ dynamic pattern efficiently without retrieval fees for the frequently accessed ones.|
|**Amazon EFS (Elastic File System)**|A scalable, elastic, cloud-native file storage for Linux-based workloads for use with AWS Cloud services and on-premises resources. Designed as _file_ storage, not _object_ storage.|Not a cost-effective or practical replacement for S3 for web-served media content. Typically more expensive per GB than S3.|
|**Cache-Control Header**|An HTTP header that provides directions for caching mechanisms (like CloudFront or browsers). The `max-age` directive specifies the maximum amount of time a cached resource is considered fresh.|Controls **CloudFront/browser caching**, not **S3 storage class cost optimization**.|

---

### Correct Option(s) Explanation

**A. Configure S3 Intelligent-Tiering on the S3 bucket.**

- **S3 Intelligent-Tiering** is explicitly designed for data with **unknown or changing access patterns**, which perfectly matches the scenario where _most_ objects become infrequently accessed but _some_ are accessed frequently.
    
- It automatically moves objects that haven't been accessed for **30 consecutive days** from the **Frequent Access tier** to the **Infrequent Access tier** (at a lower cost) and then back if they are accessed again.
    
- Crucially, **retrieval is always in milliseconds** and there are **no retrieval fees** when data moves between the two access tiers. This satisfies the strict requirement of maintaining millisecond retrieval availability at the lowest possible cost.
    
- _Reference:_ AWS S3 Intelligent-Tiering documentation states it is ideal for data with unknown or changing access patterns and provides the same low-latency performance as S3 Standard.
    

---

### Incorrect Options Explanation

**B. Configure an S3 Lifecycle policy to transition image objects and video objects from S3 Standard to S3 Glacier Deep Archive after 30 days.**

- **Glacier Deep Archive** is the lowest-cost storage class but has **long retrieval times**, typically **12 hours (Standard)**, which does **not** meet the requirement for **millisecond retrieval availability**.
    
- Furthermore, transitioning _all_ objects based on a static rule does not account for the _some frequently accessed_ data. Accessing data in Glacier Deep Archive would incur **retrieval fees**, increasing the overall cost for the frequently accessed subset.
    

**C. Replace Amazon S3 with an Amazon Elastic File System (Amazon EFS) file system that is mounted on Amazon EC2 instances.**

- **EFS** is a **file system**, not an object storage service suitable for a large-scale mobile application's photo/video upload feature served by CloudFront. S3 is the standard solution for this use case.
    
- EFS is generally **more expensive** per GB than S3, making it a poor choice for **cost optimization** for this type of archival use case. It would also require significant re-architecture of the solution (EC2 instances for serving, etc.).
    

**D. Add a Cache-Control: max-age header to the S3 image objects and S3 video objects. Set the header to 30 days.**

- This header controls the **caching behavior of CloudFront and browsers**. While optimizing CloudFront cache hit ratio is good for performance and CloudFront cost, it has **no impact on the underlying S3 storage class cost**.
    
- The primary goal is to **optimize the cost of the storage** in S3, which is achieved by transitioning data to lower-cost S3 storage classes, not by setting HTTP headers.
