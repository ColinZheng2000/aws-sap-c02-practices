A financial services company in North America plans to release a new online web application to its customers on AWS. The company will launch the application in the us-east-1 Region on Amazon EC2 instances. The application must be highly available and must dynamically scale to meet user traffic. The company also wants to implement a disaster recovery environment for the application in the us-west-1 Region by using active-passive failover.  
Which solution will meet these requirements?  

- A. Create a VPC in us-east-1 and a VPC in us-west-1. Configure VPC peering. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in both VPCs. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in both VPCs. Place the Auto Scaling group behind the ALB.
- B. Create a VPC in us-east-1 and a VPC in us-west-1. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in that VPC. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in the us-east-1 VPC. Place the Auto Scaling group behind the ALB. Set up the same configuration in the us-west-1 VPC. Create an Amazon Route 53 hosted zone. Create separate records for each ALB. Enable health checks to ensure high availability between Regions.
- C. Create a VPC in us-east-1 and a VPC in us-west-1. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in that VPC. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in the us-east-1 VPC. Place the Auto Scaling group behind the ALB. Set up the same configuration in the us-west-1 VPC. Create an Amazon Route 53 hosted zone. Create separate records for each ALB. Enable health checks and configure a failover routing policy for each record.
- D. Create a VPC in us-east-1 and a VPC in us-west-1. Configure VPC peering. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in both VPCs. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in both VPCs. Place the Auto Scaling group behind the ALB. Create an Amazon Route 53 hosted zone. Create a record for the ALB.

# Analysis

## Understanding the Problem

The financial services company needs to deploy a **highly available** and **dynamically scalable** web application on **Amazon EC2** in the **us-east-1** Region. It also requires a **disaster recovery (DR)** environment in the **us-west-1** Region using an **active-passive failover** strategy.

Key requirements:

1. **Primary Region:** us-east-1 (Active)
    
2. **DR Region:** us-west-1 (Passive)
    
3. **High Availability (within Region):** Across multiple Availability Zones (AZs) in us-east-1.
    
4. **Dynamic Scaling:** Auto Scaling based on user traffic.
    
5. **Disaster Recovery Strategy:** Active-Passive failover between Regions.
    

## Services Explanation

|**AWS Service**|**Description**|**Use in Solution**|
|---|---|---|
|**Amazon VPC**|Amazon Virtual Private Cloud lets you provision a logically isolated section of the AWS Cloud where you can launch AWS resources in a virtual network that you define.|Creates isolated network environments in us-east-1 and us-west-1.|
|**Amazon EC2**|Amazon Elastic Compute Cloud provides resizable compute capacity in the cloud.|Runs the web application.|
|**Application Load Balancer (ALB)**|Distributes incoming application traffic across multiple targets, such as EC2 instances, in multiple Availability Zones.|Provides regional high availability and distributes traffic to EC2 instances.|
|**Auto Scaling Group (ASG)**|Manages a group of EC2 instances and automatically adjusts capacity to maintain performance and cost-effectiveness.|Ensures dynamic scaling and maintains instance fleet health.|
|**Amazon Route 53**|A highly available and scalable cloud Domain Name System (DNS) web service.|Used for global traffic management and implementing the active-passive failover strategy across Regions.|
|**Route 53 Failover Routing Policy**|A routing policy that allows you to route traffic to a primary resource and automatically failover to a secondary (backup) resource when the primary is unhealthy.|Crucial for the required active-passive disaster recovery setup.|

## Correct Option(s) Explanation

### C. Create a VPC in us-east-1 and a VPC in us-west-1. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in that VPC. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in the us-east-1 VPC. Place the Auto Scaling group behind the ALB. Set up the same configuration in the us-west-1 VPC. Create an Amazon Route 53 hosted zone. Create separate records for each ALB. Enable health checks and configure a **failover routing policy** for each record.

- **Regional High Availability and Scaling:** Creating the ALB and ASG to deploy EC2 instances across multiple AZs **within each VPC** (us-east-1 and us-west-1) satisfies the requirement for high availability and dynamic scaling in both the primary and DR environments (AWS documentation: _High Availability using multiple AZs_).
    
- **Active-Passive DR:** The key component is the use of **Amazon Route 53** with a **failover routing policy**. This policy is explicitly designed for the active-passive DR setup.
    
    - The **us-east-1 ALB** is configured as the **Primary record**.
        
    - The **us-west-1 ALB** is configured as the **Secondary (Failover) record**.
        
    - Route 53 **health checks** monitor the primary ALB's health (which reflects the health of the entire us-east-1 application stack). If the primary environment fails the health check, Route 53 automatically routes all traffic to the secondary (passive) environment in us-west-1 (AWS documentation: _Route 53 Failover Routing_). This perfectly meets the _active-passive failover_ requirement.

## Incorrect Options Explanation

### A. Create a VPC in us-east-1 and a VPC in us-west-1. Configure VPC peering. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in **both VPCs**. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in **both VPCs**. Place the Auto Scaling group behind the ALB.

- **Regional Limitation:** An **Application Load Balancer (ALB)** is a **regional service** and _cannot_ extend across regions (i.e., it cannot have targets in both us-east-1 and us-west-1). Targets for an ALB must be in subnets within the same VPC. While this setup _could_ be achieved with a **Gateway Load Balancer** or more complex networking, it fundamentally violates the design principle of the ALB.
    
- **VPC Peering/Cross-Region AZs:** An Auto Scaling Group deploying instances in AZs of two peered, different-region VPCs behind a single ALB is not a standard or supported architecture for an ALB. This approach is better suited for a multi-region **active-active** design using a service like **Global Accelerator** or **Route 53**, not a single-ALB, cross-region setup.
    

### B. Create a VPC in us-east-1 and a VPC in us-west-1. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in that VPC. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in the us-east-1 VPC. Place the Auto Scaling group behind the ALB. Set up the same configuration in the us-west-1 VPC. Create an Amazon Route 53 hosted zone. Create separate records for each ALB. Enable health checks to ensure **high availability between Regions**.

- **Missing Failover Policy:** While this option sets up the correct infrastructure (ALB/ASG in both Regions) and creates Route 53 records with health checks, it fails to specify the **failover routing policy**. Creating only separate records and enabling health checks defaults to a **Simple** or **Weighted** routing policy, which would not guarantee the required **active-passive** behavior. To achieve active-passive DR, an explicit **Failover routing policy** must be configured in Route 53 (AWS documentation: _Choosing a Routing Policy_).
    

### D. Create a VPC in us-east-1 and a VPC in us-west-1. Configure VPC peering. In the us-east-1 VPC, create an Application Load Balancer (ALB) that extends across multiple Availability Zones in **both VPCs**. Create an Auto Scaling group that deploys the EC2 instances across the multiple Availability Zones in **both VPCs**. Place the Auto Scaling group behind the ALB. Create an Amazon Route 53 hosted zone. Create a record for the ALB.

- **Regional Limitation (Same as A):** This option incorrectly attempts to configure a single, regional **Application Load Balancer (ALB)** to span across two different AWS Regions (us-east-1 and us-west-1) and target instances in both. As explained for option A, the ALB is a regional service and cannot do this.
    
- **DR Strategy Missing:** This option also completely fails to define an active-passive disaster recovery strategy, only creating a single Route 53 record for the (incorrectly configured) multi-region ALB.
