A startup company hosts a fleet of Amazon EC2 instances in private subnets using the latest Amazon Linux 2 AMI. The company's engineers rely heavily on SSH access to the instances for troubleshooting.  
The company's existing architecture includes the following:  
✑ A VPC with private and public subnets, and a NAT gateway  
✑ Site-to-Site VPN for connectivity with the on-premises environment  
✑ EC2 security groups with direct SSH access from the on-premises environment  
The company needs to increase security controls around SSH access and provide auditing of commands run by the engineers.  
Which strategy should a solutions architect use?  

- A. Install and configure EC2 Instance Connect on the fleet of EC2 instances. Remove all security group rules attached to EC2 instances that allow inbound TCP on port 22. Advise the engineers to remotely access the instances by using the EC2 Instance Connect CLI.
- B. Update the EC2 security groups to only allow inbound TCP on port 22 to the IP addresses of the engineer's devices. Install the Amazon CloudWatch agent on all EC2 instances and send operating system audit logs to CloudWatch Logs.
- C. Update the EC2 security groups to only allow inbound TCP on port 22 to the IP addresses of the engineer's devices. Enable AWS Config for EC2 security group resource changes. Enable AWS Firewall Manager and apply a security group policy that automatically remediates changes to rules.
- D. Create an IAM role with the AmazonSSMManagedInstanceCore managed policy attached. Attach the IAM role to all the EC2 instances. Remove all security group rules attached to the EC2 instances that allow inbound TCP on port 22. Have the engineers install the AWS Systems Manager Session Manager plugin for their devices and remotely access the instances by using the start-session API call from Systems Manager.

---

### Analysis

#### Understanding the Problem

The core requirements are twofold:

1. **Increase Security:** The current model allows direct SSH (port 22) access from on-premises over a VPN. While private, this still exposes an attack surface (port 22) on every instance. The goal is to tighten this, ideally by removing direct port access.
    
2. **Audit Commands:** The company needs a log of the _specific commands_ engineers are running during their troubleshooting sessions, not just a log of _who connected_.
    

---

#### Services Explanation

- **AWS Systems Manager (SSM) Session Manager:** This is a fully managed AWS service that provides secure and auditable instance management. It allows you to manage EC2 instances, on-premises instances, and virtual machines through an interactive one-click browser-based shell or through the AWS CLI.
    
    - **Security:** Session Manager works by having the **SSM Agent** (pre-installed on Amazon Linux 2) make an _outbound_ connection to the SSM service endpoints. This means you **do not need to open any inbound ports** (like SSH port 22 or RDP port 3389) on your security groups, which significantly improves your security posture.
        
    - **Auditing:** It provides built-in logging and auditing capabilities. You can configure Session Manager to log all session data, including the commands run and their output, to **Amazon CloudWatch Logs** or **Amazon S3**.
        
    - **Access Control:** Access is controlled entirely through **IAM policies**, allowing for granular control over which users can start sessions on which instances.
        
- **Amazon EC2 Instance Connect (EIC):** This service provides a simple and secure way to connect to your EC2 instances using SSH. It works by pushing a one-time-use SSH public key to the instance, which remains for 60 seconds.
    
    - **Security:** It uses IAM to _authorize_ who can push a key to an instance. However, it **still uses the SSH protocol and requires port 22 to be open** in the security group (though it can be restricted to the EIC service IP range).
        
    - **Auditing:** EIC _does not_ audit the commands run _within_ the SSH session. It only audits the API call (`SendSSHPublicKey`) in **AWS CloudTrail**, which just tells you _who_ initiated a connection, not _what_ they did.
        
- **Amazon CloudWatch Agent:** This is software you install on an instance to collect custom metrics and, more importantly, log files. It can be configured to monitor OS-level logs (like `/var/log/secure` or `auditd` logs) and send them to CloudWatch Logs for central storage and analysis.
    
- **AWS Config & AWS Firewall Manager:**
    
    - **AWS Config:** A service that enables you to assess, audit, and evaluate the _configurations_ of your AWS resources. It can track changes to security groups.
        
    - **AWS Firewall Manager:** A security management service that allows you to centrally configure and manage firewall rules (including Security Groups, AWS WAF, etc.) across your accounts and applications in AWS Organization.
        

---

#### Correct Option Explanation

- **D. Create an IAM role with the AmazonSSMManagedInstanceCore managed policy attached...**
    
    - **Why it's correct:** This solution perfectly meets both requirements.
        
    - **Security:** By attaching the IAM role with the `AmazonSSMManagedInstanceCore` policy, the EC2 instances can communicate with the SSM service. This allows you to **completely remove the inbound SSH (port 22) rule** from the security groups. Access is now brokered by the SSM service via an outbound connection from the instance agent, which is a much more secure "zero-trust" model.
        
    - **Auditing:** Engineers use the `start-session` API (via the CLI with the Session Manager plugin). Session Manager can be configured to log every single command and its output to CloudWatch Logs or S3. This provides a detailed, immutable audit trail of all actions taken, directly fulfilling the audit requirement.
        
    - **Reference:** As stated in the official AWS Systems Manager documentation, "Session Manager provides secure and auditable instance management without the need to open inbound ports, manage SSH keys, or use bastion hosts." Furthermore, under "Logging and monitoring," it details how to "log session data to Amazon S3" or "log session data to Amazon CloudWatch Logs."
        

---

#### Incorrect Options Explanation

- **A. Install and configure EC2 Instance Connect...**
    
    - **Why it's incorrect:** This fails the **auditing requirement**. As explained above, EC2 Instance Connect only logs the connection _event_ to CloudTrail. It does not provide any mechanism to log the _commands_ executed inside the resulting SSH session. It's also less secure than Session Manager because it still relies on port 22 being accessible.
        
- **B. Update the EC2 security groups to only allow inbound TCP on port 22 to the IP addresses of the engineer's devices...**
    
    - **Why it's incorrect:** This is a "do-it-yourself" and inferior solution.
        
        1. **Security:** It _doesn't_ significantly increase security; it just restricts the existing open port. Port 22 is still open, leaving a vector for attack. Managing a list of (potentially dynamic) engineer IPs is operationally complex and brittle.
            
        2. **Auditing:** While you _could_ configure the CloudWatch agent to ship OS audit logs (`auditd`), it's complex to set up, manage, and parse compared to the native, out-of-the-box command logging provided by Session Manager. Option D provides a much cleaner, more secure, and more robust solution.
            
- **C. Update the EC2 security groups to only allow inbound TCP on port 22...**
    
    - **Why it's incorrect:** This option completely misses the main requirement. It focuses on **auditing _configuration changes_ to the security groups** (using AWS Config and Firewall Manager) rather than **auditing the _commands run by engineers_**. It fails the command auditing requirement entirely.
