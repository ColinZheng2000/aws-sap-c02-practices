A company is migrating its infrastructure to the AWS Cloud. The company must comply with a variety of regulatory standards for different projects. The company needs a multi-account environment.

A solutions architect needs to prepare the baseline infrastructure. The solution must provide a consistent baseline of management and security, but it must allow flexibility for different compliance requirements within various AWS accounts. The solution also needs to integrate with the existing on-premises Active Directory Federation Services (AD FS) server.

Which solution meets these requirements with the LEAST amount of operational overhead?

A. Create an organization in AWS Organizations. Create a single SCP for least privilege access across all accounts. Create a single OU for all accounts. Configure an IAM identity provider for federation with the on-premises AD FS server. Configure a central logging account with a defined process for log generating services to send log events to the central account. Enable AWS Config in the central account with conformance packs for all accounts.
B. Create an organization in AWS Organizations. Enable AWS Control Tower on the organization. Review included controls (guardrails) for SCPs. Check AWS Config for areas that require additions. Add OUs as necessary. Connect AWS IAM Identity Center (AWS Single Sign-On) to the on-premises AD FS server.
C. Create an organization in AWS Organizations. Create SCPs for least privilege access. Create an OU structure, and use it to group AWS accounts. Connect AWS IAM Identity Center (AWS Single Sign-On) to the on-premises AD FS server. Configure a central logging account with a defined process for log generating services to send log events to the central account. Enable AWS Config in the central account with aggregators and conformance packs.
D. Create an organization in AWS Organizations. Enable AWS Control Tower on the organization. Review included controls (guardrails) for SCPs. Check AWS Config for areas that require additions. Configure an IAM identity provider for federation with the on-premises AD FS server.

## Understanding the Problem
You need a **multi-account** AWS environment that:
- Enforces a **consistent baseline** of management and security across accounts
- Still allows **flexibility** for different compliance needs per project/account (e.g., different controls by OU/account)
- Integrates with an existing on-prem **AD FS** for federation
- Has the **least operational overhead** (key differentiator)

The AWS-native service designed to up and continuously govern a multi-account landing zone with minimal ongoing effort is **AWS Control Tower**. For centralized workforce access with federation, AWS’s preferred pattern is **AWS IAM Identity Center** (successor of AWS SSO), which can integrate with external identity providers using **SAML 2.0** (AD FS supports SAML).

## Correct Option Explanation

### ✅ Option B
This option best meets the requirements with the least operational overhead:

1. **Baseline multi-account governance with minimal ops**
   - **AWS Control Tower** automates creation and governance of a landing zone, including:
     - Multi-account structure (management + shared accounts)
     - Preventive/detective guardrails (implemented largely via **SCPs** and **AWS Config** rules)
     - Centralized governance patterns without you hand-building them account-by-account

2. **Flexibility for different compliance requirements**
   - You can create **multiple OUs** and apply different Control Tower **guardrails** per OU.
   - This maps well to “different regulatory standards for different projects.”

3. **Federation with on-prem AD FS with low overhead**
   - **AWS IAM Identity Center** supports connecting to external identity providers via **SAML 2.0**, which is the common way to integrate with **AD FS**.
   - Identity Center also simplifies assigning permission sets across multiple AWS accounts compared to managing IAM roles and federation plumbing per account.

## Incorrect Options Explanation

### ❌ Option A
- Uses **Organizations + one SCP + single OU**:
  - A single OU and single SCP conflicts with the requirement for **flexibility** across regulatory standards. You usually need **multiple OUs** and different SCP sets.
- Federation via a standalone **IAM identity provider** can work, but:
  - It tends to create more **per-account/per-role** operational work than IAM Identity Center in a multi-account setup.
- Central logging described as a “defined process for log generating services” implies **manual integration work** and ongoing upkeep.
- “Enable AWS Config in the central account with conformance packs for all accounts” is not how AWS Config works operationally at scale; you typically enable Config in each account/region and aggregate centrally (or use Control Tower to manage the baseline).

### ❌ Option C
- This is a “build-your-own landing zone” approach:
  - Custom SCP design, OU structure, centralized logging process, Config aggregators/conformance packs.
- While it can meet the technical requirements, it has **more operational overhead** than using **AWS Control Tower**, which automates and standardizes these baselines.

### ❌ Option D
- Uses **Control Tower** (good), but then configures an **IAM identity provider** directly for AD FS federation instead of using **IAM Identity Center**.
- In Control Tower environments, Identity Center is the common, low-ops mechanism for managing multi-account access (permission sets, assignments, lifecycle).
- Using IAM IdP federation directly often increases operational burden (role management, mappings, per-account consistency) and misses Identity Center’s centralized access management advantages.

## Hands-on Knowledge (Enterprise / regulated environments)
- **Model compliance by OU**: Create OUs like `Med-Workloads`, `Industrial-IoT`, `PCI`, `Shared-Services`, and apply different Control Tower guardrails and SCP layers to each.
- **Don’t rely on one “least privilege SCP”**: In practice, you’ll use:
  - A baseline “deny risky services/regions” SCP
  - OU-specific SCPs for regulatory controls
  - Temporary exception process (time-bound) with change management
- **Identity Center + AD FS**:
  - Prefer Identity Center permission sets for consistent access across accounts.
  - Standardize groups in AD (e.g., `AWS-PCI-Admins`, `AWS-Med-ReadOnly`) mapped to permission sets.
- **Logging & config drift**:
  - Control Tower helps establish a baseline, but regulated environments typically extend it with:
    - Organization-wide CloudTrail strategy
    - SIEM ingestion
    - AWS Config aggregation and evidence collection workflows
- **Operational reality**: The biggest “low overhead” win is reducing bespoke scripts and one-off account setup. Control Tower is designed to keep account vending and baseline controls consistent over time.
