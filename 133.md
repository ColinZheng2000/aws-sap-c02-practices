A company is running a critical stateful web application on two Linux Amazon EC2 instances behind an Application Load Balancer (ALB) with an Amazon RDS for MySQL database. The company hosts the DNS records for the application in Amazon Route 53. A solutions architect must recommend a solution to improve the resiliency of the application.

The solution must meet the following objectives:

• Application tier: RPO of 2 minutes. RTO of 30 minutes
• Database tier: RPO of 5 minutes. RTO of 30 minutes

The company does not want to make significant changes to the existing application architecture. The company must ensure optimal latency after a failover.

Which solution will meet these requirements?

A. Configure the EC2 instances to use AWS Elastic Disaster Recovery. Create a cross-Region read replica for the RDS DB instance. Create an ALB in a second AWS Region. Create an AWS Global Accelerator endpoint, and associate the endpoint with the ALBs. Update DNS records to point to the Global Accelerator endpoint.
B. Configure the EC2 instances to use Amazon Data Lifecycle Manager (Amazon DLM) to take snapshots of the EBS volumes. Configure RDS automated backups. Configure backup replication to a second AWS Region. Create an ALB in the second Region. Create an AWS Global Accelerator endpoint, and associate the endpoint with the ALBs. Update DNS records to point to the Global Accelerator endpoint.
C. Create a backup plan in AWS Backup for the EC2 instances and RDS DB instance. Configure backup replication to a second AWS Region. Create an ALB in the second Region. Configure an Amazon CloudFront distribution in front of the ALB. Update DNS records to point to CloudFront.
D. Configure the EC2 instances to use Amazon Data Lifecycle Manager (Amazon DLM) to take snapshots of the EBS volumes. Create a cross-Region read replica for the RDS DB instance. Create an ALB in a second AWS Region. Create an AWS Global Accelerator endpoint, and associate the endpoint with the ALBs.

# a) Understanding the Problem  

i. The scenario:  
A stateful web app runs on two Linux EC2 instances behind an ALB, using RDS MySQL. DNS is in Route 53. You must improve resiliency (cross‑Region DR) without big architecture changes, while meeting:

- App tier: RPO 2 minutes, RTO 30 minutes.
- DB tier: RPO 5 minutes, RTO 30 minutes.
- Latency should be optimal after failover.
- Architecture should remain mostly as-is (still EC2 + ALB + RDS).

ii. Key requirements / constraints:

- **Stateful EC2 app** ⇒ need near‑continuous replication or very frequent snapshots to get RPO = 2 minutes.
- **RDS MySQL** ⇒ must achieve RPO = 5 minutes ⇒ cross‑Region read replica or equivalent.
- **RTO 30 minutes** ⇒ ability to bring up environment in second Region within ~30 min.
- **Minimal architecture changes** ⇒ avoid re-platforming (e.g., to containers, serverless).
- **Optimal latency after failover** ⇒ traffic should be routed to the active Region efficiently; Global Accelerator is a strong candidate.
- Solutions relying only on periodic snapshots/backup will likely miss the tight RPO, especially 2 minutes for EC2.

For exam thinking, watch “RPO in minutes” + “stateful EC2” ⇒ think **AWS Elastic Disaster Recovery (DRS)** rather than plain snapshots or AWS Backup.

# b) Services Explanation  

i–ii. Key services mentioned:

1. **Amazon EC2 + EBS**  
   - Compute instances with attached EBS volumes.  
   - For DR, you can:
     - Use **AWS Elastic Disaster Recovery (DRS)** for continuous replication and quick launch in another Region.
     - Use **EBS snapshots via DLM** for periodic backups (lower RPO, higher RTO).

2. **Application Load Balancer (ALB)**  
   - L7 load balancer in each Region in front of EC2 instances.  
   - In DR, you typically recreate ALB in the failover Region and point traffic there (via Global Accelerator, Route 53 failover, etc.).

3. **Amazon RDS for MySQL**  
   - Managed relational DB.  
   - For cross‑Region resiliency:
     - **Cross-Region read replica** (asynchronous replication, minute‑level RPO).
     - **Automated backups with cross‑Region copy** (good for backup but not for low RPO DR).
   - Multi-AZ protects within a Region; cross‑Region replica protects against Region failure.

4. **Route 53**  
   - Currently hosts DNS.  
   - For DR: can use failover routing or latency routing, but Global Accelerator provides better IP‑based, low‑latency routing.

5. **AWS Elastic Disaster Recovery (DRS)**  
   - Agent-based, *continuous block-level replication* from source servers to a staging area in the DR Region.  
   - RPO often within seconds to a couple of minutes, depending on write rate and network.  
   - Supports launching **drill or actual** EC2 instances from replicated data.  
   - Very useful for **stateful** workloads with tight RPO; minimal architecture change (app stays on EC2).

   In regulated domains (healthcare, industrial), this is common to meet strict DR SLAs for critical systems without rewriting apps.

6. **Amazon Data Lifecycle Manager (DLM)**  
   - Automates creation of EBS **snapshots** on schedules (e.g., every hour).  
   - Snapshot-based ⇒ RPO tied to schedule; cannot guarantee 2‑min RPO.  
   - Good for backups, not for very low RPO DR.

7. **AWS Backup**  
   - Centralized backup management for RDS, EBS, etc., and backup **replication** cross‑Region.  
   - Typically offers **hourly or longer** backup frequencies; not designed for minute‑level RPO, especially 2 minutes.  
   - More for compliance / backup than near‑real‑time DR.

8. **Cross-Region read replica for RDSSQL**  
   - Asynchronous replication to another Region; RPO typically in seconds/minutes.  
   - You can promote the replica to standalone master after failover.  
   - Meets RPO=5 minutes requirement in most designs.

9. **AWS Global Accelerator**  
   - Provides **static anycast IPs** at the AWS edge.  
   - Routes user traffic to the closest healthy endpoint (ALBs in different Regions).  
   - Ideal for **optimal latency** and quick failover, unlike DNS-based failover which depends on TTL.  
   - Common for critical apps needing cross‑Region HA/DR.

10. **Amazon CloudFront**  
    - CDN / HTTP distribution; DNS‑based endpoints, not static IPs.  
    - Designed for HTTP caching, not DR for stateful origins.  
    - Adds complexity and is not needed for DR RPO/RTO targets.

iii. Relevant features for regulated environments:

- **DRS continuous replication** = strong fit for RPO of few minutes.  
- **RDS cross‑Region replica** = standard HA/DR pattern.  
- **Global Accelerator** = improved failover time and latency.  
- **CloudWatch + CloudTrail** = monitor, audit failover operations.  
- IaC with CloudFormation/Terraform to codify DR infrastructure.

# c) Correct Option(s) Explanation  

i. **Correct answer: A**

ii. Why A best fits the requirements:

- **Application tier RPO = 2 minutes, RTO = 30 minutes**  
  - Using **AWS Elastic Disaster Recovery** on EC2 instances provides continuous replication, typically achieving RPO within seconds to a couple minutes, well aligned to 2‑minute RPO.  
  - When Region fails, you can launch the replicated servers in the DR Region and attach them to an ALB there; this is designed for RTO around tens of minutes.  
  - No big application changes: you still run Linux EC2 instances with the same architecture.

- **Database tier RPO = 5 minutes, RTO = 30 minutes**  
  - **Cross-Region read replica** for RDS MySQL gives asynchronous replication with minute‑level lag, meeting RPO 5 minutes.  
  - On failover, you promote the replica (RTO within 30 minutes is realistic).

- **Optimal latency after failover**  
  - **ALB in second Region** handles traffic locally.  
  - **Global Accelerator endpoint associated with both ALBs** provides:
    - Static edge IPs.
    - Automatic traffic routing to the healthy Region, minimizing DNS cache issues.
    - Low latency by always sending clients to the nearest healthy Region.
  - Better than DNS failover via Route 53 alone (which is TTL‑based).

- **Minimal architecture change**  
  - You keep the same core architecture: EC2 + ALB + RDS MySQL.  
  - You add DRS + RDS read replica + ALB + Global Accelerator—standard DR enhancements.

iii. Alignment with AWS best practices:

1. **High availability and fault tolerance**  
   - Multi-Region failover capability for both app and DB tiers.  
   - Using Global Accelerator for quick failover and routing is consistent with recommendations for critical workloads.  
   - “AWS Well-Architected Framework – Reliability Pillar” recommends multi-Region architectures and automated failover for critical apps.

2. **Cost optimization**  
   - DRS uses a low-cost staging area; you only pay for small staging resources until failover or drills.  
   - Cross-Region read replica is always-on but necessary for DB RPO.  
   - Compared with full active/active rewrite, this is cost-efficient.

3. **Security, compliance, governance**  
   - DRS traffic encrypted; RDS replicas can use KMS.  
   - All operations audited with CloudTrail.  
   - Cross-Region DR meets many regulatory DR/BCP requirements (HIPAA, ISO 27001, etc.).

4. **Operational excellence and automation**  
   - DRS supports **non-disruptive drills**, which are important in medical/industrial audits.  
   - Infrastructure can be codified via CloudFormation/Terraform.  
   - Monitoring via CloudWatch, CloudWatch Alarms on replication lag, DRS status, and RDS replica lag.

References:
- *AWS Elastic Disaster Recovery User Guide* – continuous block-level replication, RPO/RTO guidance.  
- *Amazon RDS User Guide – Working with read replicas* – cross-Region read replica behavior and promotion.  
- *AWS Global Accelerator Developer Guide* – global anycast IPs and multi-Region failover.

# d) Incorrect Options Explanation  

**B. DLM snapshots for EC2 + RDS automated backups with cross-Region copy + Global Accelerator**

- For the **app tier**, DLM snapshots are periodic. Even if you took snapshots every 15 minutes, you cannot meet RPO = 2 minutes. Snapshot creation itself is not continuous; data can be lost since last snapshot.  
- RDS backups with replication are good for backups, not low RPO DR; backup frequency is usually hourly or more.  
- RPO requirements (2 and 5 minutes) are not realistically met with snapshot-based schedules.  
- Operationally simpler than some, but fails key RPO objective.

**C. AWS Backup for EC2 & RDS with cross-Region replication + CloudFront**

- AWS Backup is built for **backup/restore**, not continuous DR. Backup intervals are typically >=1 hour.  
- Cannot provide RPO = 2 min for EC2 or 5 min for DB.  
- CloudFront in front of an ALB does not solve database or stateful app DR; it is a CDN for HTTP(S), not an app-tier replication solution.  
- Also, CloudFront adds complexity and doesn’t help with RTO=30 for dynamic stateful components.  
- Violates the low-RPO objective for both tiers.

**D. DLM snapshots for EC2 + cross-Region RDS read replica + ALB + Global Accelerator**

- DB side: **OK** – cross-Region read replica meets DB RPO 5 minutes.  
- App side: same problem as B; DLM snapshots cannot guarantee RPO of 2 minutes. Even very frequent snapshots are not designed for sub-5-minute RPO for stateful workloads.  
- To make it viable, you’d need extremely frequent snapshots, complex restore and re-attachment choreography, and you’d still risk missing the 2‑minute RPO.  
- This extra operational complexity still does not beat DRS, which is explicitly designed for such low RPO.

Conclusion: only **Option A** aligns with both the tight RPO/RTO constraints and the “minimal architecture changes” requirement, while still ensuring optimal latency after failover.
