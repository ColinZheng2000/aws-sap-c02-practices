A company has a serverless application comprised of Amazon CloudFront, Amazon API Gateway, and AWS Lambda functions. The current deployment process of the application code is to create a new version number of the Lambda function and run an AWS CLI script to update. If the new function version has errors, another CLI script reverts by deploying the previous working version of the function. The company would like to decrease the time to deploy new versions of the application logic provided by the Lambda functions, and also reduce the time to detect and revert when errors are identified.  
How can this be accomplished?  

- A. Create and deploy nested AWS CloudFormation stacks with the parent stack consisting of the AWS CloudFront distribution and API Gateway, and the child stack containing the Lambda function. For changes to Lambda, create an AWS CloudFormation change set and deploy; if errors are triggered, revert the AWS CloudFormation change set to the previous version.
- B. Use AWS SAM and built-in AWS CodeDeploy to deploy the new Lambda version, gradually shift traffic to the new version, and use pre-traffic and post-traffic test functions to verify code. Rollback if Amazon CloudWatch alarms are triggered.
- C. Refactor the AWS CLI scripts into a single script that deploys the new Lambda version. When deployment is completed, the script tests execute. If errors are detected, revert to the previous Lambda version.
- D. Create and deploy an AWS CloudFormation stack that consists of a new API Gateway endpoint that references the new Lambda version. Change the CloudFront origin to the new API Gateway endpoint, monitor errors and if detected, change the AWS CloudFront origin to the previous API Gateway endpoint.

## Solution Analysis

### Understanding the Problem

The company is using a serverless application consisting of **Amazon CloudFront**, **Amazon API Gateway**, and **AWS Lambda** functions. The current deployment process is **manual** and relies on **AWS CLI scripts** to:

1. Create a **new Lambda version**.
    
2. Update the function's configuration.
    
3. Execute a **separate CLI script for rollback** if errors occur.
    

The objectives are to:

1. **Decrease the time to deploy** new versions of the Lambda function logic.
    
2. **Reduce the time to detect and revert** errors when they are identified (i.e., faster and safer rollbacks).
    

This points toward the need for an **automated, safe, and integrated deployment strategy** that supports **canary/safe deployment** and **automated rollback** based on monitoring.

### Correct Option(s) Explanation

The correct option is **B**.

**B. Use AWS SAM and built-in AWS CodeDeploy to deploy the new Lambda version, gradually shift traffic to the new version, and use pre-traffic and post-traffic test functions to verify code. Rollback if Amazon CloudWatch alarms are triggered.**

This approach directly addresses both requirements:

1. **Decrease the time to deploy:** **AWS SAM** provides a streamlined way to define and package the serverless application, simplifying the deployment process compared to manual CLI scripts or complex CloudFormation management.
    
2. **Reduce the time to detect and revert errors:**
    
    - **AWS CodeDeploy** integrated with SAM is the **standard, modern, and most robust solution for safe Lambda deployments**. It enables **canary or linear deployments** (gradually shifting traffic), which reduces the blast radius of a bad deployment.
        
    - **Pre-traffic and post-traffic test functions** (known as **hooks**) allow for immediate, automated verification before shifting traffic and before completing the deployment, providing fast error detection.
        
    - **Automated Rollback:** CodeDeploy is configured to monitor **Amazon CloudWatch alarms** (e.g., high error rate, high latency). If an alarm is triggered during the traffic shifting phase, CodeDeploy **automatically and rapidly reverts** all traffic to the previous, known-good version of the function, achieving the fastest possible rollback.
        
    - _Reference:_ The **AWS Lambda Deployment Preferences** section in the **AWS SAM Developer Guide** details the use of CodeDeploy for automated traffic shifting and rollbacks.

### Incorrect Options Explanation

|Option|Reason for Incorrectness|Why it Fails to Meet Requirements|
|---|---|---|
|**A. Create and deploy nested AWS CloudFormation stacks... For changes to Lambda, create an AWS CloudFormation change set and deploy; if errors are triggered, revert the AWS CloudFormation change set to the previous version.**|**CloudFormation change sets** are a mechanism to preview changes, but reverting a deployed stack is not as fast or safe as CodeDeploy's automated traffic shifting. A full CloudFormation stack rollback can take time, resulting in **increased detection and revert time**. It also does not natively support **gradual traffic shifting** (canary deployment) for Lambda.|Doesn't provide the fastest revert time or safe, gradual deployment.|
|**C. Refactor the AWS CLI scripts into a single script that deploys the new Lambda version. When deployment is completed, the script tests execute. If errors are detected, revert to the previous Lambda version.**|This is essentially an **improved manual deployment** but still lacks the inherent safety and automation of a purpose-built deployment service. The rollback is entirely dependent on the script's logic and the _detection time_ is only _after_ the deployment is complete, meaning 100% of traffic hits the bad version until the script detects the error and executes the rollback. This does **not reduce the time to detect and revert** safely.|Fails to reduce the blast radius and relies on a slower, less robust, scripted rollback mechanism.|
|**D. Create and deploy an AWS CloudFormation stack that consists of a new API Gateway endpoint... Change the CloudFront origin to the new API Gateway endpoint, monitor errors and if detected, change the AWS CloudFront origin to the previous API Gateway endpoint.**|This strategy involves duplicating the API Gateway and manually (or via script) changing the **CloudFront Origin**. This is an **over-engineered, complex, and slow** way to achieve a simple rollback. It introduces unnecessary resources (a duplicate API Gateway) and the change to the CloudFront origin can take several minutes to propagate globally, resulting in **slow detection and revert time** compared to CodeDeploy's instant traffic-shift at the Lambda alias level.|Fails to reduce the time to deploy and revert due to reliance on slow CloudFront propagation and resource duplication.|
