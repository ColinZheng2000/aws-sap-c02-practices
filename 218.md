A company uses AWS CloudFormation to deploy applications within multiple VPCs that are all attached to a transit gateway. Each VPC that sends traffic to the public internet must send the traffic through a shared services VPC. Each subnet within a VPC uses the default VPC route table, and the traffic is routed to the transit gateway. The transit gateway uses its default route table for any VPC attachment.

A security audit reveals that an Amazon EC2 instance that is deployed within a VPC can communicate with an EC2 instance that is deployed in any of the company's other VPCs. A solutions architect needs to limit the traffic between the VPCs. Each VPC must be able to communicate only with a predefined, limited set of authorized VPCs.

What should the solutions architect do to meet these requirements?

A. Update the network ACL of each subnet within a VPC to allow outbound traffic only to the authorized VPCs. Remove all deny rules except the default deny rule.
B. Update all the security groups that are used within a VPC to deny outbound traffic to security groups that are used within the unauthorized VPCs.
C. Create a dedicated transit gateway route table for each VPC attachment. Route traffic only to the authorized VPCs.
D. Update the main route table of each VPC to route traffic only to the authorized VPCs through the transit gateway.

Suggest Answer: C

---

## 📊 Analysis

### Understanding the Problem

A company deploys applications across **multiple VPCs** all attached to a **Transit Gateway (TGW)**. The current configuration has a security problem:

- All VPCs route traffic to the TGW via their default route tables
- The TGW uses its **default route table** for all VPC attachments
- Result: Any EC2 instance in any VPC can communicate with EC2 instances in any other VPC — **no network segmentation**

The requirement: Each VPC must communicate **only** with a predefined, limited set of authorized VPCs — effectively **VPC-level micro-segmentation** via the Transit Gateway.

This is a **Transit Gateway route table isolation** scenario, one of the core networking patterns tested in SAP-C02.

### ✅ Correct Option: C — Dedicated Transit Gateway Route Table per VPC Attachment

| Mechanism | How It Works |
|---|---|
| **Transit Gateway Route Tables** | A TGW can have multiple route tables (not just the default one). Each VPC attachment can be associated with a specific route table |
| **Per-VPC Route Table** | Create a dedicated TGW route table for each VPC attachment |
| **Selective Route Propagation** | In each route table, only add static routes or propagate routes from the authorized VPC attachments |

**How this achieves isolation:**
```
TGW Route Table "VPC-A-RT":
  - Associated with VPC A attachment
  - Routes: 10.1.0.0/16 (VPC A - local), 10.2.0.0/16 (VPC B - authorized)
  
TGW Route Table "VPC-B-RT":
  - Associated with VPC B attachment
  - Routes: 10.2.0.0/16 (VPC B - local), 10.1.0.0/16 (VPC A - authorized), 10.3.0.0/16 (VPC C - authorized)
  
TGW Route Table "VPC-C-RT":
  - Associated with VPC C attachment
  - Routes: 10.3.0.0/16 (VPC C - local), 10.2.0.0/16 (VPC B - authorized)
  
Result: VPC A ↔ VPC B ✅ | VPC B ↔ VPC C ✅ | VPC A ↔ VPC C ❌ (no route in either direction)
```

**Why this is the correct solution (SAP-C02 lens):**
- Transit Gateway is a **Layer 3** network transit hub — routing decisions are made via TGW route tables
- Each VPC attachment can be associated with a **different TGW route table**, enabling granular traffic control
- This is the **prescribed AWS pattern** for implementing network segmentation between multiple VPCs connected via TGW
- The solution scales: adding a new VPC just requires updating the relevant route tables, not changing every VPC's own routing

### ❌ Incorrect Options

**A. Network ACL per subnet to allow outbound only to authorized VPCs**

- NACLs operate at the **subnet boundary** within a VPC — they filter traffic entering/leaving a subnet
- NACLs are **stateless** — you'd need to manage both outbound AND inbound rules (return traffic requires explicit allow)
- NACL rules are limited to **20 rules per NACL** (can be increased to 40) — not scalable for many VPCs
- NACLs cannot reference CIDR blocks dynamically — every authorized VPC CIDR must be hardcoded
- Managing NACLs across dozens of subnets in multiple VPCs is an **operational nightmare**
- ❌ NACLs are the wrong layer of abstraction — VPC-to-VPC isolation through TGW should be done at the TGW route table level, not the subnet level

**B. Security groups to deny outbound to security groups in unauthorized VPCs**

- Security groups are **stateful** and only support **ALLOW rules** — there are NO deny rules in security groups
- You cannot create a security group rule that says "deny traffic to security group X"
- Even if you could, security groups are per-VPC resources — you can't reference a security group from another VPC in a security group rule
- Cross-VPC traffic through TGW is controlled at the **routing layer**, not the security group layer
- ❌ This option contains two factual errors about security groups (support for deny rules, cross-VPC referencing)

**D. Update main route table of each VPC to route only to authorized VPCs through TGW**

- VPC route tables direct traffic **to** the TGW — they send all non-local traffic to the TGW attachment
- VPC route tables can only point to the TGW as a **single target** (`destination: 0.0.0.0/0 → target: tgw-xxx`)
- You **cannot** specify per-destination-VPC routes at the VPC route table level — "10.2.0.0/16 → TGW" vs "10.3.0.0/16 → blackhole" won't work because both routes point to the same TGW
- Once traffic reaches the TGW, the **TGW route table** decides where to forward it — the VPC route table has no per-destination VPC control
- ❌ VPC route tables lack the granularity needed — TGW route tables are the right control point

### 🏭 Enterprise Hands-On Knowledge

1. **TGW Route Table Design Patterns:**
   - **Hub-and-Spoke Isolation**: Each spoke VPC gets its own TGW route table with routes only to shared services VPC (hub). Spokes cannot talk to each other.
   - **Segmented Mesh**: Groups of VPCs (e.g., "Prod", "Non-Prod", "PCI") each get a TGW route table. VPCs within the same group can route to each other; cross-group traffic is blocked.
   - **Centralized Egress**: A dedicated "Egress VPC" has routes to all other VPCs, but spoke VPCs only route to the egress VPC and 0.0.0.0/0.

2. **TGW Route Table Key Facts for SAP-C02:**
   - Each VPC attachment can be associated with **exactly one** TGW route table at a time
   - Routes can be added **statically** or via **dynamic propagation** from VPC attachments
   - A single TGW supports multiple route tables (soft limit: 20, adjustable)
   - Route evaluation: **Most specific route wins** (longest prefix match), just like VPC route tables

3. **Combining TGW Route Tables with Other Controls (Defense in Depth):**
   - TGW route tables = **network segmentation** (Layer 3)
   - Security groups = **instance-level firewall** (Layer 4)
   - NACLs = **subnet-level stateless filtering** (Layer 4)
   - AWS Network Firewall + TGW = **deep packet inspection** (Layer 7)
   - For true zero-trust, combine TGW route isolation with security groups AND AWS Network Firewall

4. **Common Gotcha:** If you associate a VPC attachment with a non-default TGW route table, the default route table no longer applies to that attachment. Make sure the new route table has routes back to the VPC itself (the attachment automatically provides this) and to any shared services VPCs (e.g., for DNS, logging, authentication).
