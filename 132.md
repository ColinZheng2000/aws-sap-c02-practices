A company is running a web application in a VPC. The web application runs on a group of Amazon EC2 instances behind an Application Load Balancer (ALB). The ALB is using AWS WAF.

An external customer needs to connect to the web application. The company must provide IP addresses to all external customers.

Which solution will meet these requirements with the LEAST operational overhead?

A. Replace the ALB with a Network Load Balancer (NLB). Assign an Elastic IP address to the NLB.
B. Allocate an Elastic IP address. Assign the Elastic IP address to the ALProvide the Elastic IP address to the customer.
C. Create an AWS Global Accelerator standard accelerator. Specify the ALB as the accelerator's endpoint. Provide the accelerator's IP addresses to the customer.
D. Configure an Amazon CloudFront distribution. Set the ALB as the origin. Ping the distribution's DNS name to determine the distribution's public IP address. Provide the IP address to the customer.

# **a) Understanding the Problem**

i. The scenario describes a web app running on EC2 instances behind an Application Load Balancer that is already protected by AWS WAF. An external customer wants to connect, but for their firewall rules or whitelisting they require *fixed IP addresses* to access the application. You must choose an approach that allows the customer to use IP addresses (not DNS names) while keeping the current ALB+WAF setup, and you must minimize operational overhead.

ii. Key requirements and constraints:

- Existing architecture: ALB in front of EC2, ALB integrated with AWS WAF.
- Customer requires static IPs they can whitelist.
- Must continue to expose the existing web application (ALB origin).
- Minimize operational overhead (no complex management, IP changes, manual updates).
- In regulated domains (medical/industrial):
  - High availability and resilience are important.
  - Predictable, static IP addresses often needed for partner systems and vendor connectivity.
  - Security and compliance (network whitelisting, WAF, logging) must be preserved.

Optimization goals:
- Keep WAF protection if possible.
- Provide stable IPs across AZs/regions if feasible.
- Minimal management effort and complexity.
- Avoid unnecessary architectural change when not required.

# **b) Services Explanation**

i–ii. Services in the options and how they’re used, especially in regulated environments:

1. **Application Load Balancer (ALB)**  
   - Layer 7 (HTTP/HTTPS) load balancer. Supports path-based and host-based routing, sticky sessions, etc.
   - Integrates natively with **AWS WAF**, which is critical for web app firewall protections in healthcare/industrial systems.
   - Is addressed by DNS name, not by fixed IP addresses. Under the hood, ALB IPs can change over time.

2. **Network Load Balancer (NLB)**  
   - Layer 4 load balancer that can be assigned **Elastic IPs** per Availability Zone.
   - Designed for high performance and static IPs.
   - Does not directly integrate with AWS WAF at L7. You’d need separate architectures for WAF (e.g., CloudFront+WAF, ALB+WAF behind/alongside NLB), increasing complexity.
   - Common in OT/industrial environments where TCP-level traffic or static ingress IPs are required, but for HTTP(s) apps you often prefer ALB+WAF.

3. **Elastic IP address (EIP)**  
   - Static public IPv4 address that you can attach to network interfaces or certain resources (EC2, NAT Gateway, NLB).  
   - **Cannot be attached directly to an ALB**. ALBs are DNS-only frontends with dynamic IPs.
   - Used to give fixed endpoints for firewalls, VPNs, industrial connectivity.

4. **AWS Global Accelerator**  
   - Provides a pair of **static anycast IP addresses** that are advertised from AWS edge locations globally.
   - You attach **endpoints** such as ALBs, NLBs, EC2 instances, or Elastic IPs.
   - Provides intelligent routing to the closest healthy endpoint, latency improvements, and built-in fault tolerance.
   - You keep the ALB and WAF architecture intact; the customer whitelists the Global Accelerator IPs.
   - Very useful in regulated sectors for partner connectivity requiring static IPs with global performance and availability.

5. **Amazon CloudFront**  
   - Global CDN service, fronting origins such as S3 buckets or ALBs.
   - Integrates with AWS WAF (very common pattern in healthcare/industrial portals).
   - CloudFront distributions are addressed by DNS names. Edge IPs can change; AWS explicitly states that you **must not** rely on static CloudFront IPs but instead use AWS managed IP ranges (e.g., via `ip-ranges.json`) if needed.
   - There is no single static IP per distribution; IP sets are large, change over time, and are not suitable if a customer wants a small list of fixed IPs.

iii. Relevant features:

- **WAF on ALB or CloudFront**: For OWASP protections, rate limiting, and compliance.
- **High availability**: ALB and Global Accelerator both support multi-AZ, automatic failover.
- **Static IPs**: Only NLB with EIP or Global Accelerator provide truly stable static IPs in this scenario.
- **Operational overhead**:  
  - NLB + ALB + WAF layering is more complex to design and operate.  
  - CloudFront IP management is complex and not truly static.  
  - Global Accelerator is designed specifically for *static IP fronting ALB/NLB* with minimal configuration.

# **c) Correct Option(s) Explanation**

i. **Correct option: C – Use AWS Global Accelerator with the ALB as the endpoint, and provide the accelerator’s static IP addresses to the customer.**

ii. Why this is the best answer:

- You retain the existing **ALB + AWS WAF** architecture unchanged. This is important operationally (no re-architecture) and for security/compliance since WAF rules remain in place and you don’t have to re-implement them.
- Global Accelerator provides **two static anycast IP addresses** per standard accelerator. Those IPs are fixed and can be safely given to the external customer for firewall whitelisting.
- Traffic is routed to the ALB endpoint, so your web application stack remains as is.
- Operational overhead is minimal: you create the accelerator once, configure the ALB as endpoint, and then maintain it via IaC (CloudFormation/Terraform) if desired.
- In medical/industrial contexts where partner systems might only allow connections to pre-approved IPs (e.g., hospital networks, manufacturing plant firewalls), Global Accelerator is a very common pattern to front ALBs without losing WAF functionality.

iii. Connecting to key architecture themes:

1. **High availability and fault tolerance**  
   - Global Accelerator automatically routes traffic to healthy endpoints within a region and can spread across multiple regions if you add additional endpoints.  
   - ALB already provides high availability across multiple AZs.  
   - Reference: *AWS Global Accelerator Developer Guide – “High availability”* and *AWS Well-Architected Framework – Reliability Pillar*.

2. **Cost optimization**  
   - You add Global Accelerator cost, but you avoid rearchitecting with additional load balancers and WAF stacks.  
   - In an exam context, “LEAST operational overhead” often wins over minimal cost when security and correctness are at stake. For enterprise/regulated domains, operational simplicity and correctness often outweigh small cost differences.

3. **Security, compliance, governance**  
   - Keeps **WAF** in front of the ALB as-is. You preserve web ACLs for SQLi/XSS protection, IP reputation lists, etc.  
   - You can log and monitor via CloudWatch, AWS WAF logs, ALB access logs, and centralize in SIEM (e.g., using Kinesis Firehose or S3).  
   - Reference: *AWS WAF Developer Guide* and *AWS Security Best Practices whitepaper* for layering protections.

4. **Operational excellence and automation**  
   - Easy to manage via CloudFormation, CDK, or Terraform for standard accelerators.  
   - Tagging Global Accelerator resources with application/owner/environment tags helps operations and cost visibility.  
   - Very little ongoing operational work after initial setup.

# **d) Incorrect Options Explanation**

**A. Replace the ALB with an NLB and assign an Elastic IP**

- You lose ALB L7 features: path-based routing, host-based routing, HTTP headers awareness, etc.
- More importantly, **AWS WAF does not attach directly to NLB**, so you would lose the current WAF integration unless you build an additional layer (e.g., CloudFront+WAF, or ALB+WAF behind NLB which increases complexity and contradicts “least operational overhead.”
- This is a full re-architecture, which is inappropriate for a “simple static IP” requirement.
- Reference: *Elastic Load Balancing User Guide – Differences between ALB and NLB.*

**B. Allocate an Elastic IP address and assign it to the ALB**

- Technically **impossible**: an ALB cannot have an Elastic IP assigned. ALBs only expose DNS names and their IPs are dynamic.
- This option is invalid from a service capabilities standpoint.
- Reference: *Application Load Balancer documentation – “ALBs do not support static IP addresses or Elastic IPs.”*

**D. Use CloudFront with ALB as origin and discover the distribution IP**

- CloudFront does not give you a single, stable IP for a distribution. CloudFront edge locations use large and **changing IP ranges**.
- AWS publishes IP ranges in `ip-ranges.json`, and customers can whitelist those ranges, but:
  - The IP ranges are huge and shared across many customers.
  - The ranges change over time. This is high operational overhead and not “fixed IPs.”
- “Pinging” the distribution DNS name to find an IP is not valid: DNS resolution may rotate between many IPs and may change at any time.
- This violates the “must provide IP addresses” constraint in a reliable way and clearly fails “least operational overhead” as the customer would need to constantly track changing IPs or ranges.
- Reference: *Amazon CloudFront Developer Guide – “Recommended: Use DNS names, not IP addresses”* and AWS IP address ranges documentation.
