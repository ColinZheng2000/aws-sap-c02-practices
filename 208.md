A company has an application that runs on Amazon EC2 instances in an Amazon EC2 Auto Scaling group. The company uses AWS CodePipeline to deploy the application. The instances that run in the Auto Scaling group are constantly changing because of scaling events.

When the company deploys new application code versions, the company installs the AWS CodeDeploy agent on any new target EC2 instances and associates the instances with the CodeDeploy deployment group. The application is set to go live within the next 24 hours.

What should a solutions architect recommend to automate the application deployment process with the LEAST amount of operational overhead?

A. Configure Amazon EventBridge to invoke an AWS Lambda function when a new EC2 instance is launched into the Auto Scaling group. Code the Lambda function to associate the EC2 instances with the CodeDeploy deployment group.
B. Write a script to suspend Amazon EC2 Auto Scaling operations before the deployment of new code. When the deployment is complete, create a new AMI and configure the Auto Scaling group's launch template to use the new AMI for new launches. Resume Amazon EC2 Auto Scaling operations.
C. Create a new AWS CodeBuild project that creates a new AMI that contains the new code. Configure CodeBuild to update the Auto Scaling group’s launch template to the new AMI. Run an Amazon EC2 Auto Scaling instance refresh operation.
D. Create a new AMI that has the CodeDeploy agent installed. Configure the Auto Scaling group’s launch template to use the new AMI. Associate the CodeDeploy deployment group with the Auto Scaling group instead of the EC2 instances.


Suggest Answer: D

---

## Analysis

### Understanding the Problem

A company uses AWS CodePipeline to deploy an application running on EC2 instances within an Auto Scaling group (ASG). The ASG instances are ephemeral — they constantly change due to scaling events. The current manual process installs the CodeDeploy agent on every new instance and then associates that instance with the CodeDeploy deployment group. This is operationally unsustainable, especially with the application going live within 24 hours. The requirement is to **automate deployment with the least operational overhead**, meaning the solution should leverage built-in AWS integrations rather than custom scripting or Lambda functions. The key insight is that CodeDeploy has **native integration with Auto Scaling groups** — when you associate a deployment group with an ASG (rather than individual instances), CodeDeploy automatically deploys to all current and future instances in that ASG.

### Correct Option: D — AMI with CodeDeploy Agent + ASG Launch Template + Associate Deployment Group with ASG

**Why D is correct:**

- **Pre-baked AMI with CodeDeploy agent**: By creating a golden AMI with the CodeDeploy agent pre-installed and configuring the ASG launch template to use this AMI, every new instance launched by the ASG (via scaling events or instance replacements) automatically has the agent ready. No post-launch agent installation is needed.
- **Associate CodeDeploy deployment group with the ASG (not individual instances)**: This is the critical architectural change. When a CodeDeploy deployment group is associated with an Auto Scaling group, CodeDeploy automatically:
  - Deploys the application revision to all existing EC2 instances in the ASG
  - Automatically deploys to any **new instances** launched by the ASG (via scaling events, instance refresh, or EC2 Auto Scaling health checks)
  - Handles the lifecycle seamlessly — instances that are terminated are removed from the deployment, and new instances automatically receive the latest application revision
- **Least operational overhead**: This uses built-in AWS integrations with zero custom code (no Lambda, no EventBridge rules, no scripts). It's a one-time configuration change.
- **Aligns with CI/CD best practices**: Immutable infrastructure with pre-baked AMIs ensures consistency — every instance starts from the same known-good state.

**Official reference**: [Integrating CodeDeploy with Amazon EC2 Auto Scaling](https://docs.aws.amazon.com/codedeploy/latest/userguide/integrations-aws-auto-scaling.html) — "You can associate an Auto Scaling group with a CodeDeploy deployment group. When you create a deployment, CodeDeploy deploys the application revision to the current instances in the Auto Scaling group. When Amazon EC2 Auto Scaling launches new instances as part of a scale-out event, CodeDeploy deploys the application revision to the new instances."

### Incorrect Options

**A — EventBridge + Lambda to associate instances**

- Adds **unnecessary operational overhead**: This requires writing, testing, and maintaining a Lambda function and EventBridge rule. While it would work, it's a custom solution when a built-in integration already exists.
- **Timing race condition**: When a new EC2 instance launches, the CodeDeploy agent may not be installed yet when the Lambda function attempts to associate the instance with the deployment group. This creates a fragile dependency.
- Lambda cold starts and EventBridge event delivery guarantees (at-least-once) add complexity that the native ASG-deployment group integration avoids entirely.
- For the SAP-C02 exam, when AWS provides a native integration, custom code (Lambda/EventBridge) is rarely the "LEAST operational overhead" answer.

**B — Script to suspend ASG → create AMI → update launch template → resume ASG**

- **Extremely high operational overhead**: This is a manual, multi-step scripted process. Suspending ASG operations means the ASG won't scale in or out during the deployment window, potentially causing capacity issues.
- The script must handle error states (what if the AMI creation fails? What if the launch template update fails?). Manual scripts are fragile in production.
- Creating a new AMI for every deployment is time-consuming (AMI creation + instance launch can take 10-15 minutes) and doesn't scale well for frequent deployments.
- This approach also doesn't integrate with CodePipeline/CodeDeploy natively — you're bypassing the purpose-built deployment tools.

**C — CodeBuild to create AMI + update launch template + instance refresh**

- **Over-engineered**: CodeBuild is a build service for compiling source code, running tests, and producing artifacts. Using it to create AMIs and update launch templates is a misuse of the service — these are infrastructure operations, not build steps.
- An **instance refresh** replaces all instances in the ASG with new instances. While valid for periodic AMI updates, it's not the right mechanism for application code deployments through CodePipeline/CodeDeploy. Instance refreshes are for rolling out new AMI versions (OS patches, agent updates), not for deploying application code revisions.
- CodeDeploy provides blue/green deployment, canary, and all-at-once deployment strategies specifically designed for application code delivery — these are lost when using instance refresh.

### Hands-On Enterprise Knowledge

1. **Golden AMI pipeline pattern**: In enterprise environments, a common CI/CD pattern is:
   - **Pipeline 1 (Infrastructure)**: Packer or EC2 Image Builder creates a golden AMI with the OS, security patches, monitoring agents (CloudWatch agent, CodeDeploy agent, etc.), and runtime dependencies. This pipeline runs weekly or on security patch events.
   - **Pipeline 2 (Application)**: CodePipeline → CodeBuild (build/test) → CodeDeploy deploys application code onto the golden AMI instances. This pipeline runs on every code commit.
   
   This separation ensures fast application deployments (you're not rebuilding AMIs for every code change) while maintaining infrastructure consistency.

2. **CodeDeploy ASG integration details**:
   - The deployment group is associated with the ASG, not with individual instance IDs
   - During a deployment, CodeDeploy uses the ASG's lifecycle hooks to manage instance transitions
   - For **blue/green deployments** with ECS or Lambda, CodeDeploy can manage traffic shifting. For EC2/ASG, CodeDeploy performs **in-place deployments** by default (updating instances already in service)
   - When a scale-out event occurs **during** a deployment, CodeDeploy deploys the **current revision** (the one being deployed) to the new instance, maintaining consistency

3. **Common enterprise pitfall**: Many teams initially configure CodeDeploy to target individual EC2 instances (by instance ID or tags). This works initially but fails as soon as Auto Scaling replaces instances. The correct configuration from day one is to associate the deployment group with the ASG. This is a frequently tested concept on the SAP-C02 exam.

4. **CloudWatch agent considerations**: Just like the CodeDeploy agent, the CloudWatch agent (for custom metrics, logs) and Systems Manager agent (SSM) should also be pre-baked into the golden AMI. This is part of a comprehensive AMI hardening strategy.

5. **CodeDeploy agent auto-update**: The CodeDeploy agent can be configured to auto-update when new versions are released by AWS. This is enabled by default on new installations. In the golden AMI, consider whether to pin the agent version or enable auto-update based on your change management policies.
