A company is running an event ticketing platform on AWS and wants to optimize the platform's cost-effectiveness. The platform is deployed on Amazon Elastic Kubernetes Service (Amazon EKS) with Amazon EC2 and is backed by an Amazon RDS for MySQL DB instance. The company is developing new application features to run on Amazon EKS with AWS Fargate.

The platform experiences infrequent high peaks in demand. The surges in demand depend on event dates.

Which solution will provide the MOST cost-effective setup for the platform?

A. Purchase Standard Reserved Instances for the EC2 instances that the EKS cluster uses in its baseline load. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet predicted peak load for the year.
B. Purchase Compute Savings Plans for the predicted medium load of the EKS cluster. Scale the cluster with On-Demand Capacity Reservations based on event dates for peaks. Purchase 1-year No Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale out database read replicas during peaks.
C. Purchase EC2 Instance Savings Plans for the predicted base load of the EKS cluster. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale up the DB instance manually during peaks.
D. Purchase Compute Savings Plans for the predicted base load of the EKS cluster. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale up the DB instance manually during peaks.

# Cost Optimization for EKS + RDS Workload with Spiky Demand  
#aws/sap-c02 #cost-optimization #eks #rds #savings-plans #spot-instances

## Correct Answer  
* *D. Purchase Compute Savings Plans for the predicted base load of the EKS cluster. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale up the DB instance manually during peaks.**

- --

## Understanding the Problem

Key points:

- Event ticketing platform on **Amazon EKS with EC2**, plus **RDS for MySQL**.
- New features are being developed to run on **EKS with Fargate**.
- **Infrequent but high peaks** in demand, tied to known event dates.
- Goal: **MOST cost-effective** setup.

We must consider:

- How to price-optimize:
  - EKS compute on **EC2** (baseline + peaks).
  - EKS compute on **Fargate** (new features).
  - RDS database (baseline + peaks).
- Choose between:
  - **Reserved Instances vs Savings Plans (Compute vs EC2)**.
  - **On-Demand vs Spot vs Capacity Reservations**.
  - **Base load vs peak load coverage**.

- --

## Services and Pricing Model Explanation

### 1. Amazon EKS on EC2 & Fargate

- EKS control plane is managed by AWS; you pay for:
  - Control plane (per-cluster fee).
  - Worker nodes capacity: **EC2** or **Fargate**.

AWS Docs:  
- EKS pricing: https://docs.aws.amazon.com/eks/latest/userguide/eks-pricing.html

### 2. Amazon RDS for MySQL

- Managed relational database.
- Pricing drivers: instance size, storage, I/O, backups.
- Supports **Reserved Instances (RDS RIs)** for cost savings vs On-Demand.

AWS Docs:  
- RDS pricing: https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/Concepts.RDS_Fees.html

### 3. EC2 Pricing Options

#### Spot Instances
- Up to ~90% cheaper than On-Demand.
- **Interruption possible**; good for flexible or stateless workloads.
- Not suitable for critical single-instance DBs, but very good for bursty app servers.

Docs:  
- Spot: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/using-spot-instances.html

#### Savings Plans
- Commit to a **$ per hour** usage for 1 or 3 years in exchange for lower prices.

Two main types:  
- **Compute Savings Plans**  
  - Apply to **EC2 (all instance families/regions)**, **Fargate**, and **Lambda**.  
  - Most flexible (applies to EKS on EC2, EKS on Fargate).
- **EC2 Instance Savings Plans**  
  - Limited to a specific **instance family and region** (e.g., m5 in us-east-1).
  - Cheaper rate than Compute SP, but less flexible.
  
Docs:  
- Savings Plans: https://docs.aws.amazon.com/savingsplans/latest/userguide/what-is-savings-plans.html

#### Reserved Instances vs Savings Plans (for RDS & EC2)

- **RDS**: uses **Reserved Instances** (not Savings Plans) for discounts.
- **EC2**: can use either EC2 RIs or Savings Plans.
- RDS RIs can be 1-year or 3-year, and All Upfront / Partial Upfront / No Upfront.

Docs:  
- RDS RIs: https://docs.aws.amazon.com/AmazonRDS/latest/UserGuide/USER_WorkingWithReservedDBInstances.html

#### Capacity Reservations

- **On-Demand Capacity Reservations** guarantee capacity, but do **not** reduce cost by themselves.
- They can be combined with Savings Plans or RIs, but they are about availability, not cost reduction.

Docs:  
- Capacity reservations: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-capacity-reservations.html

## Why Option D Is Correct (Insightful Explanation)

**EKS Compute:**

- **Base load**:  
  - Workload is always-on (core ticketing functions).  
  - Using **Compute Savings Plans** covers:
    - EC2 instances running EKS worker nodes.
    - Fargate tasks (new features).
  - This gives flexibility to:
    - Change instance families,
    - Migrate some workloads to Fargate,
    - Possibly use other services in future, all still covered by the plan.
- **Peaks**:  
  - Peaks are **infrequent and predictable**.  
  - Using **Spot Instances** for additional EKS worker nodes during peaks:
    - Provides large cost savings vs On-Demand.
    - EKS supports mixed-instance ASG + Spot; can be configured for resilience.
  - This fits the pattern: **baseline = discounted, stable capacity; peaks = Spot**.

AWS docs on mixed instances/Spot in EKS:  
- https://docs.aws.amazon.com/eks/latest/userguide/managed-node-groups.html#managed-node-group-capacity-types

**RDS Database:**

- DB must be **highly available and stable**. You generally do NOT use Spot-like interruption.
- **Base load**:  
  - Use **1-year All Upfront Reserved Instances** for RDS:
    - Large cost saving over On-Demand.
    - Base load is predictable; good candidate for RI.
- **Peak load handling**:
  - The option suggests **manually scaling up the DB instance** during known peaks.
  - Because peaks are infrequent and predictable (by event dates), this is practical:
    - Scale up just before the event window.
    - Scale down after the load subsides.
  - You still cover the **base load** with RDS RI; the delta for peak time is a small On-Demand overhead for a short period.  
  - This tends to be more cost-effective than permanently overprovisioning to peak capacity.

This overall structure aligns well with AWS Cost Optimization best practices:  
- Optimize steady-state usage with **Savings Plans/RIs**.  
- Handle burst capacity with **Spot** (application tier) and **temporary scaling** (database).

## Why the Other Options Are Incorrect (Insightful Explanation)

### Option A

> Purchase Standard Reserved Instances for the EC2 instances that the EKS cluster uses in its baseline load. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet predicted peak load for the year.

Issues:

1. **EC2 RIs vs Compute Savings Plans**:  
   - EC2 RIs are **less flexible** than **Compute Savings Plans**, especially when you also have **Fargate**.  
   - RIs do not apply to Fargate; Compute Savings Plans do.
2. **DB RI sized for peak load**:  
   - Buying RIs for the **peak** level for the entire year wastes money, because peaks are infrequent.
   - The most cost-effective approach is typically to RI the **base**, then scale up temporarily.

Therefore, this overprovisions RDS all year and misses flexibility for Fargate.

### Option B

> Purchase Compute Savings Plans for the predicted medium load of the EKS cluster. Scale the cluster with On-Demand Capacity Reservations based on event dates for peaks. Purchase 1-year No Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale out database read replicas during peaks.

Issues:

1. **On-Demand Capacity Reservations for peaks**:
   - They guarantee capacity, but do **not** reduce compute cost.
   - For peaks that are infrequent and can tolerate scale-out across multiple nodes, **Spot** is much cheaper.
2. **Database scaling via read replicas**:
   - For **RDS MySQL**, read replicas help read-heavy workloads.
   - Ticketing platforms often have heavy **write** activity during ticket sales; scaling only reads may not be sufficient.
   - Also, creating/destroying read replicas incurs replication lag and operational overhead.
3. **No Upfront RDS RI**:
   - No Upfront has slightly higher cost than All Upfront. If the question is about **most cost-effective**, All Upfront gives max discount, assuming the company can afford the cash flow.

So B is not the most cost-effective configuration.

### Option C

> Purchase EC2 Instance Savings Plans for the predicted base load of the EKS cluster. Scale the cluster with Spot Instances to handle peaks. Purchase 1-year All Upfront Reserved Instances for the database to meet the predicted base load. Temporarily scale up the DB instance manually during peaks.

Issues:

1. **EC2 Instance Savings Plans vs Compute Savings Plans**:
   - EC2 Instance SP only apply to a specific **instance family** in a given region (e.g., m5).
   - They **do not** cover Fargate.
   - Given the scenario explicitly mentions **EKS with Fargate** for new features, you want coverage that also applies to Fargate.
   - **Compute Savings Plans** are more flexible, applying to EC2 and Fargate.
2. Everything else in C (Spot for peaks, RDS RI for base, manual DB scale for peaks) is sound, but the Savings Plan type is suboptimal for a mixed EC2 + Fargate environment.

Thus D improves on C by using **Compute Savings Plans**, matching the architecture.

## Summary

- Use **Compute Savings Plans** for **EKS base load** because:
  - They cover **EC2 and Fargate**.
  - Provide flexibility across instance families and future architecture changes.
- Use **Spot Instances** for **EKS peak load** to cheaply handle infrequent surges.
- Use **RDS Reserved Instances (All Upfront)** for **database base load** to maximize discount.
- **Temporarily scale up** the RDS DB instance for peak events rather than sizing RIs to the peak.

This combination in **Option D** matches AWS cost-optimization best practices and is the most cost-effective for the described workload.
